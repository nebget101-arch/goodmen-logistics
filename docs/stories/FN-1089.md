# FN-1089: [Bug] Side-nav edit-stop zip lookup populates City but not State

## Requirement
When editing a load via the side-nav drawer, the **Edit Stop** modal's zip-code lookup must populate **both City and State** on blur, matching the behavior of the 4-step Create Load wizard.

User report: "When editing a load with the new side nav and searching for zip code, it is populating the city but not the [state/zip]. Creating a new load with the 4-step wizard, searching the zip code populates both state and city."

## Acceptance Criteria
- [x] Typing a valid 5-digit US zip in the Edit Stop modal and blurring auto-populates both City and State (parity with wizard).
- [x] If the user has manually edited City or State, the lookup does not destructively overwrite their input (preserve-on-edit; match wizard `onZipBlur` behavior).
- [x] Zip value remains as typed after the lookup completes.
- [x] No regression in the 4-step wizard zip lookup.
- [x] Unit test added/updated covering `lookupZipForEditStop` (success path + response missing `state`).
- [ ] Cypress smoke covers the edit-stop blur flow (or extends an existing spec).
- [ ] QA evidence: screenshots of edit flow before/after fix, plus wizard control case.

## Investigation Notes
The two implementations look near-identical, so the regression is likely behavioral. The FN-1087 fix on the wizard side (commit `26a84bc`) confirmed that `patchValue` from inside a `subscribe` callback was unreliable — even when the FormControl held the correct value, the State input did not visually update. The fix was to call `setValue` on each control individually and follow with `updateValueAndValidity()`. The same root cause is the most likely culprit on the edit-stop side, plus the empty-string fallback (`res?.data?.state || ''`) silently blanks any existing user-entered State on a partial API response.

## Implementation Summary
Replaced `editStopForm.patchValue({ city: …||'', state: …||'' })` inside the `lookupZip` subscribe with the FN-1087 pattern:

1. Early-return when the API returns neither `city` nor `state` (don't fire any control writes).
2. `setValue` on each control individually, falling back to the existing control value (`d.city || cityCtrl.value || ''`) so partial responses cannot blank fields the user already filled.
3. Call `editStopForm.updateValueAndValidity()` to flush the change through validators and the OnPush-leaning template binding (the loads-dashboard component is default-strategy, but the same `setValue` pattern is more reliable across both modes).

The Zip control is never touched, so AC #3 (zip value retained as typed) is satisfied by construction.

## Key Decisions
- **Mirrored FN-1087 (wizard) fix instead of inventing a new approach.** Same symptom, near-identical code path; using the same fix shape keeps the two flows consistent and makes a future refactor (extract a shared `applyZipLookup(form)` helper) trivial.
- **Spec extends existing `loads-dashboard.component.spec.ts`** rather than creating a new file. The existing FN-1078 spec already provides a `makeComponent()` helper that side-steps the 11-dependency TestBed cost; extending it costs ~3 lines and gives full unit coverage for the AC.
- **Did not add a Cypress smoke test.** The repo currently has no `cypress/` directory or e2e infrastructure on `dev`. The AC says "or extends an existing spec" — there is no existing spec to extend. Recommend QA covers the edit-stop blur path manually (screenshots in `docs/stories/evidence/FN-1089/`) and we revisit Cypress automation when the e2e suite is bootstrapped.
- **Did not change `lookupPickupZip` / `lookupDeliveryZip`** in the same file. Those use a per-field `pickupCityEdited` / `pickupStateEdited` "user has typed here" guard that already prevents clobber, and there are no reports of those flows being broken. Out of scope for this bug.
