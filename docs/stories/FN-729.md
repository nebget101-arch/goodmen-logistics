# FN-729: Email-to-Load — Forward Rate Confirmation Email for Auto-Creation

## Requirement
Each tenant gets a unique inbound email address. Forward a rate-con email to that
address → FleetNeuron parses the email body + PDF attachments via AI → creates a
DRAFT load. Dispatchers are notified in-app so they can review and approve.

## Acceptance Criteria
- [ ] Inbound email provider configured (SendGrid Inbound Parse / SES)
- [ ] Each tenant gets a unique `loads-<slug>@inbound.fleetneuron.ai` address
- [ ] Webhook creates a DRAFT load from a forwarded rate-con PDF attachment
- [ ] Text-only forwarded email uses body fallback (draft with subject/body → notes)
- [ ] Whitelist rejects non-authorized senders (FN-761)
- [ ] Rate limit enforced — 100 emails/tenant/day (FN-761)
- [ ] Settings page shows the tenant address + recent email log (FN-762)

## Implementation Summary

### FN-759 (subtask) — Inbound email schema (branch: `database/FN-759/inbound-email-schema`)

**Migration 1:** `20260419100000_add_inbound_email_to_tenants.js`
- Adds `tenants.inbound_email_address TEXT` (nullable)
- `CREATE UNIQUE INDEX uq_tenants_inbound_email_address ON tenants (inbound_email_address) WHERE inbound_email_address IS NOT NULL` — partial unique index so multiple NULLs are allowed while non-null values stay unique
- Backfills existing tenants with `loads-<slug>-<idsuffix>@inbound.fleetneuron.ai`, where `<slug>` is the tenant name normalized (lowercased, diacritics stripped, non-alphanumerics collapsed to `-`, max 40 chars) and `<idsuffix>` is the first 8 hex chars of the tenant UUID (guarantees uniqueness even when two tenant names slugify identically)
- Idempotent: `hasColumn` / `IF NOT EXISTS` guards; backfill only touches rows where `inbound_email_address IS NULL`

**Migration 2:** `20260419100100_create_inbound_emails_log.js`
- Creates `inbound_emails` table matching the FN-759 spec:
  - `id UUID PK`, `tenant_id UUID NOT NULL` FK → `tenants.id` ON DELETE CASCADE
  - `from_email`, `subject`, `body_text`, `body_html` (all nullable TEXT)
  - `received_at TIMESTAMPTZ NOT NULL DEFAULT now()`
  - `load_id UUID` FK → `loads.id` ON DELETE SET NULL (nullable — set only when a DRAFT load is created)
  - `processing_status TEXT NOT NULL DEFAULT 'pending'` with CHECK constraint (`pending | success | failed | rejected`)
  - `error_message TEXT`
- Index `idx_inbound_emails_tenant_received_at ON (tenant_id, received_at DESC)` for per-tenant timeline queries

### FN-760 (subtask) — Backend webhook + pipeline (branch: `backend/FN-760/inbound-email-webhook`)

**Scope delivered:**
1. `POST /api/webhooks/email-inbound` in integrations-service, accepting SendGrid/SES multipart payloads.
2. Shared-secret auth via `INBOUND_EMAIL_WEBHOOK_SECRET` header check (configurable; skipped when unset for dev).
3. Tenant resolution from destination address against `tenants.inbound_email_address`.
4. Payload logging in `inbound_emails` (defensive — skipped cleanly if FN-759 migration has not yet run).
5. PDF pipeline: each PDF attachment → `extractLoadFromPdf` → insert DRAFT load (`source='email'` when column present) + `load_stops` rows + `load_attachments` row with R2-uploaded original.
6. Text fallback: when no PDF is attached, insert a DRAFT load with subject/body captured in `notes` (no stops).
7. In-app dispatcher notification via `sendInAppNotificationsToUsers` (target roles: `admin`, `dispatch`, `dispatcher`).
8. Tenant-facing `GET /api/tenants/me/inbound-email` and `GET /api/tenants/me/inbound-email/logs` routes in `goodmen-shared/routes/inbound-email.js`, mounted behind `authMiddleware` + `tenantContextMiddleware`.
9. Gateway rewrites: `/api/webhooks/email-inbound` and `/api/tenants/me/inbound-email` → `INTEGRATIONS_SERVICE_URL`.
10. Webhook always returns 200 once the payload is accepted (tenant mismatch, pipeline failure included) so the provider does not retry-storm; failure details are logged to `inbound_emails.processing_status` / `error_message`.

**Out of scope (deferred):**
- Whitelist enforcement + rate limiting + virus scan → **FN-761**
- Tenant unique-address provisioning/backfill + schema migration → **FN-759**
- Inbound provider + DNS configuration → **FN-758**
- Settings UI + loads-list email badge → **FN-762**
- `loads.source` column migration: not owned by FN-760; the code probes with `hasColumn('loads','source')` and sets the field only when the column exists. Recommend widening FN-759 (or a follow-up database subtask) to add `loads.source TEXT NULLABLE` + index before FN-762 implements the "Source: Email" filter.

### FN-761 (subtask) — Sender whitelist + rate limit + virus scan (branch: `backend/FN-761/inbound-email-security`)

**Scope delivered:**
1. New migration `20260419100000_create_inbound_email_whitelist.js` → table `inbound_email_whitelist (id, tenant_id, pattern, is_domain, created_by_user_id, created_at)` with unique `(tenant_id, LOWER(pattern))` and tenant index.
2. Security service `services/inbound-email-security.js` with three checks applied in order after tenant resolution:
    - **Rate limit** — counts `inbound_emails` rows for the tenant since UTC start-of-day and rejects at `INBOUND_EMAIL_DAILY_LIMIT` (default `100`). Includes prior rejections in the count so flooders can't bypass by forcing failures.
    - **Whitelist** — if any rows exist for the tenant, sender must match either an exact address or an `@domain` row. Empty whitelist = open inbox (no enforcement) to keep migration day-0 frictionless.
    - **Virus scan** — ClamAV INSTREAM over TCP (no new npm dep; uses built-in `net`). Env: `CLAMAV_HOST`, `CLAMAV_PORT` (default 3310), `CLAMAV_TIMEOUT_MS`. Mode via `INBOUND_EMAIL_VIRUS_SCAN`:
        - `optional` (default) — scan if configured; errors treated as clean so ClamAV outages don't lock out inbound mail.
        - `required` — scan must succeed; connection/timeout errors become `virus_scan_error` rejections.
        - `disabled` — skip entirely.
3. Hook in `processInboundEmail` (after log-row insert, before pipeline): on rejection, updates the audit row with `processing_status='rejected'` and `error_message='<reason>: <detail>'`, emits `inbound_email_rejected` warning log, and returns `200 { status: 'rejected', reason, detail }`.
4. Shared tenant routes extended with whitelist CRUD:
    - `GET /api/tenants/me/inbound-email/whitelist`
    - `POST /api/tenants/me/inbound-email/whitelist` (body: `{ pattern: 'ops@broker.com' | '@broker.com' }`, 409 on duplicate)
    - `DELETE /api/tenants/me/inbound-email/whitelist/:id`
5. Unit tests (`services/inbound-email-security.test.js`) — 5 new tests (23 total in the service) covering exact/domain matching, case-insensitivity, empty inputs, blank pattern rows.

**Out of scope:**
- Audit trail UI for whitelist changes (FN-762).
- Admin-wide bypass / role-based overrides — the whitelist is strictly tenant-scoped by design.
- Quota dashboards / usage charts — basic count is exposed via `GET /logs`; visualizations are FN-762's call.

## Key Decisions

### FN-759
1. **Partial unique index** on `inbound_email_address` instead of `UNIQUE` column constraint — allows multiple tenants to remain un-provisioned (NULL) while still rejecting duplicate non-null addresses.
2. **`<slug>-<id8>` address pattern** — pure `slug` would collide when tenant names overlap or slugify to an empty string (non-ASCII names). Appending 8 hex chars of the UUID guarantees uniqueness and keeps the address short and human-recognizable.
3. **`load_id` ON DELETE SET NULL** — preserve the audit log even if the linked load is later deleted.
4. **`tenant_id` ON DELETE CASCADE** — a deleted tenant should not leave orphaned inbound email records.
5. **CHECK constraint** on `processing_status` — cheap guard against bad writes at the DB layer in addition to app-level validation.

### FN-760
- **Defensive DB probing** — `hasTable`/`hasColumn` before using `tenants.inbound_email_address`, `inbound_emails`, `loads.source`. Lets integrations-service boot and run the health check when FN-759 / `loads.source` migrations are not yet applied.
- **Shared-secret auth instead of SendGrid HMAC** — SendGrid Inbound Parse does not sign payloads; SES signs SNS envelopes. A single `x-webhook-secret` header is sufficient for both providers and keeps the code small. Dev mode (env unset) logs a warning instead of rejecting.
- **Always 200** — webhook acknowledges the payload even on tenant mismatch / pipeline failure. Otherwise SendGrid retries aggressively. Failure detail is captured in the `inbound_emails` audit row so operators can triage without provider-side retry noise.
- **Reuse, don't refactor, loads.js** — copied the minimum load-creation logic into the processor rather than exporting `processSingleRateConfirmation` to avoid rippling test churn across the loads module.
- **Notification target roles** — `admin`, `dispatch`, and the newer `dispatcher` code. Matches the idle-truck monitor pattern.

### FN-761
- **Reuse `inbound_emails` for rate-limit counting** — accurate & already indexed by `(tenant_id, received_at DESC)`. Avoids a dedicated counter table.
- **ClamAV TCP INSTREAM by hand** — no extra npm dep; clean env toggles (`optional` / `required` / `disabled`).
- **Empty whitelist = open inbox** — tenants adopt the feature with zero config; enforcement kicks in only after they add at least one row.
- **Whitelist rejections become `processing_status='rejected'`** so the operator can always see *why* an email was dropped via the existing logs endpoint.
- **Domain prefix convention `@domain.com`** in whitelist rows (mirrors how most MTAs and users reason about allow-listing). `is_domain` is derived from the pattern plus stored explicitly so future UI can render it without re-parsing.
