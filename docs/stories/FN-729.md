# FN-729: Email-to-Load — Forward Rate Confirmation Email for Auto-Creation

## Goal
Each tenant gets a unique inbound email address (`loads-<tenant-slug>@inbound.fleetneuron.ai`).
Forwarded rate-confirmation emails → AI parses email body + PDF attachments → creates a DRAFT load
that a dispatcher reviews and approves.

### API contract assumed (from story description — implemented by FN-760 / FN-761)
- `GET /api/tenants/me/inbound-email` → `{ success, data: { address, is_active } }`
- `GET /api/tenants/me/inbound-email/logs?limit=&offset=` → `{ success, data: InboundEmailLog[], meta }`
- `POST /api/tenants/me/inbound-email/test` → `{ success, message? }`
- `GET /api/tenants/me/inbound-email/whitelist` → `{ success, data: WhitelistEntry[] }`
- `POST /api/tenants/me/inbound-email/whitelist` body `{ sender_email }` → `{ success, data }`
- `DELETE /api/tenants/me/inbound-email/whitelist/:id` → `{ success }`
- `GET /api/loads` list items gain optional `source?: string` (e.g. `'email'`, `'manual'`, `'ai'`)
- `GET /api/loads/:id` detail gains optional `inbound_email?: { from_email, subject, received_at, body_text, body_html }`
- List endpoint accepts `?source=email` filter param

> If the backend chooses different field names, the frontend adjustments are localized to
> `frontend/src/app/services/inbound-email.service.ts` and the `LoadListItem` / `LoadDetail`
> interfaces in `frontend/src/app/models/load-dashboard.model.ts`.

### Key decisions
- **Reused existing admin-page shell** (`ai-admin-shell`, `ai-header`, `panel ai-panel`) and table styles
  to match the look of `trial-requests-admin` / `multi-mc-admin`. Consistent with FleetNeuron admin UX.
- **Permission model**: the new route uses the same `anyPermission` set as other admin pages
  (`ROLES_MANAGE`, `ACCESS_ADMIN`, `USERS_EDIT`). Can be tightened later if the backend adds a
  dedicated `inbound_email.manage` permission.
- **Nav placement**: linked from the existing user-menu admin cluster (alongside "Locations Admin"),
  rather than introducing a new sidebar section. Inbound Email is a tenant-level settings page and
  matches the "admin settings" bucket users already know.
- **Source filter plumbing**: extended the existing `filters` + `activeFilterChips` pattern that
  FN-746 introduced for `needsReview`, rather than adding a parallel filter mechanism.
- **Whitelist UX**: inline add form + confirm-then-delete for each row; optimistic update on success.
  Duplicate-email and malformed-email are caught client-side before hitting the API.
- **Clipboard fallback**: follows the existing `trial-requests-admin` pattern — `navigator.clipboard`
  first, then `textarea + execCommand('copy')`, then `window.prompt` as last resort.
