## Implementation Summary

### FN-759 (subtask) â€” Inbound email schema (branch: `database/FN-759/inbound-email-schema`)

**Migration 1:** `20260419100000_add_inbound_email_to_tenants.js`
- Adds `tenants.inbound_email_address TEXT` (nullable)
- `CREATE UNIQUE INDEX uq_tenants_inbound_email_address ON tenants (inbound_email_address) WHERE inbound_email_address IS NOT NULL` â€” partial unique index so multiple NULLs are allowed while non-null values stay unique
- Backfills existing tenants with `loads-<slug>-<idsuffix>@inbound.fleetneuron.ai`, where `<slug>` is the tenant name normalized (lowercased, diacritics stripped, non-alphanumerics collapsed to `-`, max 40 chars) and `<idsuffix>` is the first 8 hex chars of the tenant UUID (guarantees uniqueness even when two tenant names slugify identically)
- Idempotent: `hasColumn` / `IF NOT EXISTS` guards; backfill only touches rows where `inbound_email_address IS NULL`

**Migration 2:** `20260419100100_create_inbound_emails_log.js`
- Creates `inbound_emails` table matching the FN-759 spec:
  - `id UUID PK`, `tenant_id UUID NOT NULL` FK â†’ `tenants.id` ON DELETE CASCADE
  - `from_email`, `subject`, `body_text`, `body_html` (all nullable TEXT)
  - `received_at TIMESTAMPTZ NOT NULL DEFAULT now()`
  - `load_id UUID` FK â†’ `loads.id` ON DELETE SET NULL (nullable â€” set only when a DRAFT load is created)
  - `processing_status TEXT NOT NULL DEFAULT 'pending'` with CHECK constraint (`pending | success | failed | rejected`)
  - `error_message TEXT`
- Index `idx_inbound_emails_tenant_received_at ON (tenant_id, received_at DESC)` for per-tenant timeline queries

## Key Decisions

### FN-759
1. **Partial unique index** on `inbound_email_address` instead of `UNIQUE` column constraint â€” allows multiple tenants to remain un-provisioned (NULL) while still rejecting duplicate non-null addresses.
2. **`<slug>-<id8>` address pattern** â€” pure `slug` would collide when tenant names overlap or slugify to an empty string (non-ASCII names). Appending 8 hex chars of the UUID guarantees uniqueness and keeps the address short and human-recognizable.
3. **`load_id` ON DELETE SET NULL** â€” preserve the audit log even if the linked load is later deleted.
4. **`tenant_id` ON DELETE CASCADE** â€” a deleted tenant should not leave orphaned inbound email records.
5. **CHECK constraint** on `processing_status` â€” cheap guard against bad writes at the DB layer in addition to app-level validation.

### FN-760
- **Defensive DB probing** â€” `hasTable`/`hasColumn` before using `tenants.inbound_email_address`, `inbound_emails`, `loads.source`. Lets integrations-service boot and run the health check when FN-759 / `loads.source` migrations are not yet applied.
- **Shared-secret auth instead of SendGrid HMAC** â€” SendGrid Inbound Parse does not sign payloads; SES signs SNS envelopes. A single `x-webhook-secret` header is sufficient for both providers and keeps the code small. Dev mode (env unset) logs a warning instead of rejecting.
- **Always 200** â€” webhook acknowledges the payload even on tenant mismatch / pipeline failure. Otherwise SendGrid retries aggressively. Failure detail is captured in the `inbound_emails` audit row so operators can triage without provider-side retry noise.
- **Reuse, don't refactor, loads.js** â€” copied the minimum load-creation logic into the processor rather than exporting `processSingleRateConfirmation` to avoid rippling test churn across the loads module.
- **Notification target roles** â€” `admin`, `dispatch`, and the newer `dispatcher` code. Matches the idle-truck monitor pattern.
