# FN-1560: [Warehouse Receiving] Quick-add receives at $0 cost; need editable unit cost + part default_cost reconcile

## Requirement

On the Warehouse Receiving page, adding a part via Quick add posts the receive line with `unit_cost = 0.00` whenever `parts.default_cost` is zero (which it is for many SKUs). There is no way to type the actual cost on the quick-add row, no way to edit it on the receive-lines table after the fact, and no way to push a corrected cost back into the part record so future receives default to the new figure.

Reproduction (from screenshot `Screenshot 2026-05-08 at 12.41.25 AM.png`): Allstate Garland — Main Warehouse, ticket `RCV-E8B7-1778200920738-0958`, search "tran" → click `+ Add` for `TRN-442 Transmission Sensor` qty 5 → row appears with UNIT COST `0.00`.

## Acceptance Criteria

- [ ] Quick-add panel rows show editable unit-cost input seeded from `default_cost`; `+ Add` posts the user-entered cost.
- [ ] Receive-lines UNIT COST cell is inline-editable on DRAFT tickets; blur/Enter persists via `PATCH /receiving/:ticketId/lines/:lineId`.
- [ ] Backend route `PATCH /receiving/:ticketId/lines/:lineId` validates and updates `unit_cost` (and optionally `qty_received`, `bin_location_override`); rejects edits on POSTED tickets; role-gated.
- [ ] When entered cost differs from `parts.default_cost` by > $0.01, UI shows non-blocking confirmation: "Update {sku} default cost from $A to $B?" with Update / Skip / Skip all this session.
- [ ] On Update, `PATCH /api/parts/:id` writes `default_cost` (and `default_retail_price` if user opts in via second checkbox tied to current markup); audit logged via `dtLogger`.
- [ ] Existing barcode-scan flow keeps working with no regression.

## Implementation Summary

### FN-1566 (subtask) — backend cost-patch routes

Added two PATCH endpoints and the supporting service + validator layer that the frontend reconcile flow (FN-1562) consumes:

- `PATCH /api/receiving/:ticketId/lines/:lineId` — updates any subset of `{ unit_cost, qty_received, bin_location_override }` on a DRAFT ticket. Returns `200` with the updated row, `400` on validation/POSTED-ticket errors, `403` for non-`parts_manager`/`admin` roles, `404` when the ticket or line is missing or the line is on a different ticket.
- `PATCH /api/parts/:id` — updates `default_cost` and/or `default_retail_price` only. Emits a `parts_cost_changed` audit log via `dtLogger` (old/new values, partId, performedBy from `req.user.id`, optional `ticketId` from request body) per the parent AC's "audit logged via dtLogger" requirement. The existing `PUT /api/parts/:id` route is untouched.

Validation rules (shared between both routes via `utils/cost-validators.js`): cost values must be finite, ≥ 0, and have ≤ 2 decimal places; numeric strings are coerced. `qty_received` must be a positive integer. `bin_location_override` is trimmed, blank → `null`, capped at 64 chars to fit the column.

Both routes use the existing in-file `requireRole(['admin', 'parts_manager'])` middleware. No DB migration was added — `parts.default_cost`, `parts.default_retail_price`, and `receiving_ticket_lines.unit_cost` already exist (`20260216_create_inventory_schema.js`).

## Key Decisions

- **No new audit table.** Per the Jira description's explicit "Do NOT add a new migration here", parts-cost changes are emitted as a structured `parts_cost_changed` log line with `{ partId, sku, performedBy, ticketId, changes: { field: { from, to } } }`. The Open Items section already tracks the dedicated audit-table follow-up decision.
- **Wrote `default_cost`, not `unit_cost`.** The receiving flow already reads `default_cost` (e.g. `routes/receiving.js:911` falls back to `part.default_cost` when no override is supplied). The reconcile prompt's purpose is to keep that fallback truthful, so PATCH `/api/parts/:id` targets `default_cost` / `default_retail_price` rather than the legacy `unit_cost` / `unit_price` columns the bulk PUT route writes. A future cleanup story can converge the two column families; out of scope here.
- **Service holds DB + validation, route is thin.** Per the Jira description's "do not duplicate validation between route and service" instruction. The route handlers only translate typed-error `statusCode` into the HTTP response and never re-check rules.
- **Validators in a standalone util** so the same rules are unit-testable without an Express + knex harness, and so a future PATCH route can re-use them.
- **Audit log records `from`/`to` as numbers**, coercing the prior `decimal(10,2)` value via `Number()`. Keeps the log machine-grep-able and avoids string/number drift if/when we add the dedicated audit table later.
