# FN-780 — Inbound email test returns 200 but no row appears in `inbound_emails` / Recent emails list

**Type:** Bug (parent)
**Status:** In Progress
**Parent story:** FN-729 (inbound-email pipeline)

## Problem

Clicking **Send test email** on `/admin/inbound-email` returns `{ success: true }` from `POST /api/tenants/me/inbound-email/test`, but no row appears in `inbound_emails` within 15 seconds. Three independent failure points can produce this symptom (DNS/SendGrid Inbound Parse unconfigured, webhook auth mismatch, or tenant-lookup miss) and the existing endpoint cannot distinguish between them.

### FN-782 (subtask)

**Scope delivered:**

- `POST /api/tenants/me/inbound-email/test` now:
  - Resolves MX records for the tenant's inbound email domain (`dns.resolveMx`).
  - Inserts a placeholder `inbound_emails` row (`processing_status='test_pending'`) *before* attempting the outbound send, guaranteeing the Recent emails table is never silent.
  - Short-circuits with `success: false` + actionable diagnostic when MX does not resolve (no misleading 200).
  - Calls `sendEmail` and surfaces the SendGrid `x-message-id` in the response.
  - Returns diagnostic JSON: `{ mxRecords, mxResolves, outboundProviderResponse, logRowId, instructions }`.
  - Marks the placeholder row `rejected` with an error message when the outbound send fails.
- `GET /api/tenants/me/inbound-email/health` returns `{ configured, address, mxOk, mxRecords, lastWebhookReceiptAt, lastSuccessAt, lastRejectionReason }` for the frontend to poll.
- `integrations-service/services/inbound-email-service.js`:
  - Reconciliation: when `processInboundEmail` fires and a matching `test_pending` row exists for the tenant within 5 minutes (subject-normalized match), the existing row is updated in place rather than a new one inserted. UI sees a single row transition from `test_pending → success/rejected`.
  - `inbound_email_tenant_not_found` logs at `error` (was `warn`) so it surfaces on dashboards — insertion into `inbound_emails` with `tenant_id=null` is impossible (FK NOT NULL), so the log is the only audit trail for mis-routed mail.
- `sendEmail` in `notification-service.js` now returns `messageId` (from SendGrid's `x-message-id` response header) when available.
- `matchesTestPendingRow` extracted as a pure helper in `inbound-email-helpers.js` and unit-tested.

**Migration:** `20260419200000_add_test_pending_to_inbound_emails.js` — extends the `inbound_emails_processing_status_check` CHECK constraint to allow `test_pending`. The original enum (`pending|success|failed|rejected`) would have blocked the placeholder insert.

## Key decisions (FN-782)

- **Placeholder is inserted before outbound send, not after.** Guarantees the row exists even if SendGrid is misconfigured, the node process dies after send but before logging, or the tenant later hits the `/test` endpoint from a closed tab. Reconciliation flips the status when/if the webhook round-trips.
- **Pure decision helper (`matchesTestPendingRow`) lives in `inbound-email-helpers.js`** — not in `inbound-email-service.js` — so tests can exercise it without importing the knex/SendGrid/R2 module graph. Follows the existing helper pattern.
- **Reconciliation window is 5 minutes, subject-exact match (case/whitespace insensitive).** Conservative: we'd rather miss reconciliation and insert a duplicate row than stomp an unrelated inbound email that happens to arrive during a `test_pending` window.
- **`tenant_not_found` stays at log-level (not table-level).** FK NOT NULL on `inbound_emails.tenant_id` means we cannot persist unrouted mail; promoting the log severity is the pragmatic escalation path.
- **`test_pending` added to the CHECK constraint via migration**, not dropped — future enum additions should follow the same pattern.
