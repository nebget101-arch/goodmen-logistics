# FN-867: S7 — Wire Edit flow into wizard (getLoad prefill → updateLoad; fold draft-review PDF into Step 4)

## Requirement
Wire `<app-load-wizard>` into edit mode. Given `[mode]="'edit'"` + `[loadId]`, fetch the load via `LoadsService.getLoad`, prefill all four step FormGroups, allow the user to edit any step, and submit via `updateLoad`. When the load is a draft with a source PDF, fold the PDF viewer into Step 4 and render a trip-timeline summary panel on Step 2.

## Implementation Summary

### FN-883 (subtask)
Extended `LoadWizardComponent` to support `mode='edit'` end-to-end:

1. **Edit-mode prefill** — On `ngOnInit`, when `mode === 'edit'` (or `view`) and `loadId` is set, the wizard either uses a pre-supplied `[loadDetail]` input (callers that already have the detail) or fetches one itself via `LoadsService.getLoad(loadId)`. A new `applyLoadDetail(detail)` helper maps each field onto the corresponding step FormGroup / FormArray:
   - `basics` gets `load_number`, `status`, `billing_status`, `broker_id`, `po_number`, `rate`, `dispatcher_user_id`, `notes`.
   - `stops` is rebuilt from the server list (sorted by `sequence`). Each row preserves the server-side `id` in a new hidden `id` control so updates can diff against the original.
   - `driverEquipment` gets `driver_id`, `truck_id`, `trailer_id`, and sets `showAllTrucks=true` so a pre-selected truck that isn't on the driver's default assignment remains visible in the dropdown.
   - `existingAttachments` is populated from `detail.attachments`.
   - `sourcePdfUrl` is computed from the first `RATE_CONFIRMATION` attachment using `DomSanitizer.bypassSecurityTrustResourceUrl`.

2. **updateLoad submit** — `onSubmit` branches: `create` / `ai-extract` → `submitCreate()` (existing path), `edit` → new `submitUpdate()` which calls `LoadsService.updateLoad(loadId, payload)`. The payload shape mirrors the legacy dashboard `updateLoad` body and now includes each existing stop's `id`, letting the backend diff inserts / updates / deletes. On success the wizard emits `updated` with the fresh `LoadDetail`; parents listen and refresh the dashboard row.

3. **Attachment diff** — No explicit diff is needed at submit time because the attachments step (FN-881) already operates in immediate-upload mode for edit/ai-extract: new files upload immediately via `uploadAttachmentWithProgress` (emits `attachmentUploaded`), and existing files delete immediately via `deleteAttachment` (emits `existingDeleted`). The wizard now also updates `sourcePdfUrl` in both callbacks so adding or removing a rate-con PDF toggles the Step 4 viewer live.

4. **PDF viewer on Step 4** — When `showSourcePdf` (currentStep === 'attachments' && sourcePdfUrl), Step 4 renders a two-column split layout with the sanitized iframe on the left and the attachments UI on the right. The panel widens to `max-width: 1180px` so both panes have room. A media query at 900px collapses back to a single column with the PDF stacked above the UI.

5. **Trip timeline summary on Step 2** — A new summary panel renders above the Stops list, showing each stop's sequence number, type (Pickup / Delivery), and city/state. Clicking a timeline item calls `onTimelineStopClick(i)`, which scrolls the target form row (`[data-wizard-stop-row="<i>"]` — added to the Stops step template) into view and applies a 1.2s pulse-highlight class defined in the Stops SCSS.

6. **Loading state** — While the initial `getLoad` is in flight, the wizard shell is hidden and a lightweight "Loading load details…" status is rendered in its place.

## Key Decisions
- **Kept attachment sync immediate (no queued diff at submit).** FN-881 already gave the attachments step an immediate-upload/delete path in edit mode, so a separate diff on submit would be redundant. This is documented inline and verified by the spec.
- **Preserved the 2-stop minimum invariant during prefill.** If a server load returned fewer than 2 stops, we top up with default placeholder stops so the existing `canProceed` gating and shell UX keep working.
- **Added `id` to stop FormGroups for both create and edit paths** so server-side row matching on update is unambiguous; create-mode stops just leave it `null`.
- **Kept the timeline summary + PDF viewer in the wizard template** rather than introducing new standalone sub-components. The summary is ~40 lines of declarative markup driven straight off the stops FormArray, and the PDF viewer is a single `<iframe>` tied to a sanitized `SafeResourceUrl` — the extra component surface wasn't worth the ceremony.
- **Made `loadDetail` an optional input.** Callers that already fetched the detail (e.g. dashboard row drawer) can pass it in and skip the wizard's own `getLoad` call. Callers that only have an `id` (e.g. direct deep link) get the round-trip for free.
- **Used `data-wizard-stop-row` (data attribute + `querySelector`) rather than a ViewChild relay.** The Stops component owns the DOM rows; piping a ViewChild array up to the wizard would have coupled the two components. A scoped data-attribute + a pulse class inside the Stops SCSS keeps the concern local.
