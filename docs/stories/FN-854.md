# FN-854: Loads grid inline Status/Billing dropdown clipped and not rendering

## Requirement
As a dispatcher, when I click the Status or Billing badge on a loads-dashboard row, I need to see the inline dropdown of valid transitions so I can update the load without opening the detail panel. The FN-787 implementation added the dropdowns but they are not visible to the user — either clipped by a parent container or positioned off-screen — blocking the core value of that story.

## Implementation Summary
Root cause: the inline Status/Billing menu was rendered as a `position: absolute` div inside `<td>` inside `cdk-virtual-scroll-viewport`. The viewport's internal `overflow: auto` (structural for virtualization) clipped the menu. The trigger pill opened correctly but the options panel was cut off by the viewport edge — invisible to the user.

Fix: port both dropdowns to a CDK Overlay portal attached to `document.body` via `cdkOverlayOrigin` + `cdkConnectedOverlay`. This escapes the viewport's overflow entirely and the menu floats above all scrollers. Kept the existing state model (`statusMenuLoadId` / `billingMenuLoadId`) and PATCH flow unchanged — only the rendering strategy moved.

Supporting changes:
- Added `OverlayModule` import to the lazy-loaded feature module.
- `ScrollStrategyOptions.close()` strategy so the menu dismisses if the virtual-scroll viewport scrolls (which could recycle the trigger row).
- Fallback position: `below-start` with auto-flip to `above-start` for rows near the viewport bottom.
- `document:keydown.escape` `HostListener` — the portaled menu is outside the component's DOM subtree, so component-scoped keydown bindings would not fire.
- Added `closeStatusMenu()` / `closeBillingMenu()` methods used by the overlay's `(overlayOutsideClick)` and `(detach)` handlers.
- Opening Status now also closes Billing (mutual exclusion was only enforced in the other direction).

## Key Decisions
- **CDK Overlay over CSS-only fix.** An `overflow: visible` chain cannot be applied to `cdk-virtual-scroll-viewport` — that breaks virtualization. Raising z-index alone doesn't escape an `overflow: hidden` ancestor. The Overlay portal is the correct structural fix and matches the ticket's recommended direction (a).
- **`ScrollStrategyOptions.close()`** instead of `reposition()`. In a virtualized list, scrolling can recycle the row that owns the trigger — leaving the menu dangling is worse than simply closing it.
- **Global class names** (`.loads-inline-menu*`) not BEM-scoped to `.status-inline`. The portal lives outside the component, so scoped SCSS can't reach it. Using distinct names (rather than re-exporting `.status-inline__menu` globally) avoids collisions and makes the portal styling intentional.
- **Kept `openStatusMenu` toggling semantics** (click again to close). The CDK Overlay `(overlayOutsideClick)` handles click-outside dismissal; combined with the existing `document:click` handler, both paths work regardless of whether the user clicks the trigger again or anywhere else.
- **Did not refactor `getValidStatusTransitions(load)` in `*ngFor`** despite it being a per-CD method call. Out of scope for this bug; the arrays are tiny (≤5 items) and change detection already runs under the same conditions as before.
