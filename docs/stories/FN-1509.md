# FN-1509: [inventory-service] Render deploy crashes — missing @aws-sdk/client-s3 dependency

## Requirement
Fix the production-blocking deploy failure on `fleetneuron-inventory-service`. After FN-1480 added an R2 upload path to `goodmen-shared/routes/receiving.js`, the service no longer boots on Render because its `package.json` doesn't declare `@aws-sdk/client-s3` (transitive deps from a `file:` linked package don't reliably resolve on Render).

## Implementation Summary

### FN-1511 (subtask)

Added the two AWS SDK packages that `goodmen-shared/storage/r2-storage.js` requires transitively:

```json
"@aws-sdk/client-s3": "^3.967.0",
"@aws-sdk/s3-request-presigner": "^3.967.0",
```

Placed directly after `@goodmen/shared` in `inventory-service/package.json` `dependencies`, matching the exact ordering already used in `vehicles-maintenance-service/package.json`. Versions match `goodmen-shared`'s own declaration and the four other consuming microservices (`drivers-compliance`, `vehicles-maintenance`, `integrations`, `logistics`), keeping the AWS SDK in lockstep across all microservices.

`npm install` from `backend/microservices/inventory-service/` regenerated `package-lock.json` cleanly (added 0 new top-level packages — both SDK packages were already present transitively via other deps; the lockfile now records them as explicit dependencies of `inventory-service` so Render's `npm ci` materializes them under `node_modules/@aws-sdk/`).

**Local boot caveat (does not affect Render):** running `node server.js` locally fails to resolve `express`/`dotenv` because npm symlinks the `file:` dep `@goodmen/shared`, and Node 18's default symlink resolution looks up `node_modules` from the *real* path of `goodmen-shared` (which has none of its own). On Render, `npm ci` installs against the inventory-service's own `package.json` / `package-lock.json`, populating `inventory-service/node_modules/` with everything the service and its mounted shared routes require — which is why the four other microservices that follow this same pattern boot fine on Render. The local boot failure mode is identical for `vehicles-maintenance-service`, `drivers-compliance-service`, etc., and is unrelated to this fix.

## Key Decisions

- **Out-of-scope services left untouched.** `auth-users-service`, `reporting-service`, and `ai-service` also lack `@aws-sdk/*` declarations but currently mount no shared route that requires r2-storage, so they aren't broken. Hardening them was explicitly excluded from this fix per the parent bug's "Latent risk" section. File a separate ticket if proactive hardening is desired.
- **Match existing pattern, not strict alphabetical.** `vehicles-maintenance-service` places `@aws-sdk/*` after `@goodmen/shared` (not in pure ASCII order). The ticket called out that pattern explicitly, so this PR matches it for consistency rather than re-sorting.
- **Did not modify `goodmen-shared/package.json` or `r2-storage.js`.** The shared package already declares both SDK packages correctly — the bug is purely Render's resolution of transitive `file:` deps, which is fixed by re-declaring at the consuming microservice level.
