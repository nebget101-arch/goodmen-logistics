# FN-1109: Smart defaults + live duplicate detection in Add form

## Requirement
Catch duplicates as the user types in the Add Part form (similarity ≥ 0.85). Provide an auto-SKU generator with collision-safe uniqueness.

## Acceptance Criteria
- [ ] `GET /api/parts/duplicate-check?name=&sku=&manufacturer=` (pg_trgm fuzzy, threshold 0.85, max 5).
- [ ] Debounced (350ms) call from form fields → inline "Possible duplicates" warning with one-click "Edit existing".
- [ ] No false-positive spam — warning hides on empty results; user can dismiss for the session.
- [ ] Auto-SKU button: `<MFG>-<CAT>-<NNNN>`, collision-checked, editable.
- [ ] QA: typo variants, no-spam test, auto-SKU uniqueness at scale.

## Implementation Summary

### FN-1110 (subtask)

Backend duplicate-check endpoint that powers the FE inline "Possible duplicates" warning (FN-1111) and the QA validation (FN-1112). Uses PostgreSQL `pg_trgm` for fuzzy matching with a weighted blended similarity (name 0.5 + sku 0.3 + manufacturer 0.2) and a per-component threshold of 0.85.

- New Knex migration enables `pg_trgm` and adds GIN/trigram indexes on `parts.name`, `parts.sku`, `parts.manufacturer`.
- New service method `partsService.findDuplicateCandidates({ name, sku, manufacturer, limit })` returns rows shaped as `[{ id, name, sku, manufacturer, similarity }]`, sorted by blended similarity desc, capped at 25 results (default 5).
- New route `GET /api/parts/duplicate-check?name=&sku=&manufacturer=&limit=5` requires `authMiddleware` only (matches existing `GET /api/parts` read-policy — any authenticated user can call it). Returns 400 if all three of name/sku/manufacturer are empty.
- Tests: `backend/packages/goodmen-shared/routes/parts-duplicate-check.test.js` covers typo-variant ("Fleetguard"/"FleetGuard"), exact match, no match, empty query, limit clamp, and similarity rounding.

### FN-1111 (subtask)

Frontend live duplicate detection + auto-SKU generator inside the Add Part modal of the Parts Catalog page.

- New presentational component `app-duplicate-warning` renders the "Possible duplicates" banner with up to 5 candidates, similarity %, an "Edit existing" button per row, and a single "Ignore — this is a new part" dismiss action. The component is dumb: parent owns the data + dismissed state.
- `PartsCatalogComponent` wires a `combineLatest`-of-(name, sku, manufacturer) `valueChanges` pipeline with `debounceTime(350)` + `distinctUntilChanged` and calls `apiService.duplicateCheckParts({ name, sku, manufacturer, limit: 5 })`. The pipeline is gated to (a) Add mode (`editingPartId === null`), (b) `!duplicateWarningDismissed`, and (c) at least one non-empty field — the BE returns 400 for an all-empty query, so we never send one. Errors are swallowed so a transient failure does not poison the stream.
- `duplicateCandidates` resets to `[]` whenever the BE returns empty, so the warning auto-hides as the user types past a duplicate.
- "Edit existing" calls `getPartById(candidate.id)` and reopens the modal in Edit mode; falls back to the candidate payload if the lookup fails.
- "Ignore — this is a new part" sets `duplicateWarningDismissed = true` for the rest of the modal session (cleared by `closeForm` / next `openForm`).
- "Generate SKU" button (visible only in Add mode) builds `<MFG>-<CAT>-<NNNN>` from the manufacturer + category fields' first 3 alphanumeric letters (uppercased; falls back to "GEN" when no letters), then picks a 4-digit counter via 10 random tries against the in-memory `parts` list and falls back to a deterministic 0000–9999 scan if the random retries collide. The field stays editable.
- `api.service.ts` got a new `duplicateCheckParts({ name, sku, manufacturer, limit })` method that builds the query string and trims empties.
- `DuplicateWarningComponent` registered in `app.module.ts` declarations.

Acceptance evidence:
- Debounce: `parts-catalog.component.spec.ts` — three rapid `patchValue` keystrokes coalesce into one `duplicateCheckParts` call after 350ms.
- Warning render: candidates flow into the component → banner element appears in the DOM.
- Dismiss: clicking dismiss flips `duplicateWarningDismissed`, hides the warning, and prevents subsequent typing from re-firing the call.
- Auto-hide on empty: empty `data` clears `duplicateCandidates`.
- No-false-positive: BE returns `[]` for "FleetGuard FF5052" → no warning.
- Edit mode: typing in the Edit form does NOT call `duplicateCheckParts`.
- Generate-SKU collision retry: when 9 999 of 10 000 candidates are taken, the function still returns the only free slot.
- "Edit existing": loads full record via `getPartById`, transitions modal to Edit mode for the chosen part.
- Editable: control is `enabled` after generate, user can override the value.

## Key Decisions

### FN-1110 (subtask)

- **Per-component threshold instead of blended threshold.** The 0.85 cutoff is applied to `GREATEST(name_sim, sku_sim, mfg_sim)` rather than the blended score, so a strong single-field match (e.g. just a near-identical name) surfaces a row. The blended score is still used for ranking. This avoids the surprise where a great name match plus an empty sku/manufacturer would fall under 0.85 because of the 0.5/0.3/0.2 weighting.
- **Empty fields contribute ~0, weights NOT renormalized.** A single-field `?name=` query caps the blended score at 0.5; that's intentional — it preserves the relative weighting of name vs. sku vs. manufacturer when the FE later passes more than one field.
- **Threshold-on-best plus rank-by-blend keeps the SQL one query.** A subquery (`scored` CTE) computes per-field similarities once; the outer SELECT uses `GREATEST(...)` for the WHERE filter and the linear blend for ORDER BY. No duplicate similarity computation, no second roundtrip.
- **No tenant scoping.** Followed existing `GET /api/parts` and the bulk-upload SKU dedup query — neither tenant-scopes today, so duplicate-check is consistent. If tenant scoping is added later, this endpoint will pick it up via the same shared filter.
- **Auth = authMiddleware only (no role gate).** The acceptance criterion says "consistent with other parts read endpoints"; `GET /api/parts`, `/categories`, `/manufacturers` all use plain `authMiddleware` with no role check. Any authenticated user can read.
- **Migration `CREATE EXTENSION IF NOT EXISTS` is idempotent.** Safe to re-run; `IF NOT EXISTS` guards on indexes too. `down` drops the indexes but intentionally leaves `pg_trgm` installed because other tables/queries may rely on it (consistent with `20260101000000_enable_uuid_extension.js`).

### FN-1111 (subtask)

- **Subscribe to `valueChanges` once in `ngOnInit`, not on `openForm`.** Subscribing on form-open leads to easy lifecycle bugs (forgetting to unsubscribe on close, double-subscribing on second open). The pipeline gates *whether to fire the API* on `editingPartId === null` and `!duplicateWarningDismissed`, so it's safe and cheap to keep subscribed for the component lifetime.
- **`duplicateCheck` method went on `ApiService`, not a new `parts.service.ts`.** The Jira description hints at extending `parts.service.ts`, but no such file exists in this codebase — every `/parts/*` HTTP call already lives in `api.service.ts` next to `getParts`/`createPart`. Adding a parallel service for one extra method would have been an unnecessary abstraction.
- **Collision check uses the in-memory parts list, no extra HTTP roundtrip.** The Add Part modal opens after `loadParts()` populates `this.parts`, so the SKU generator can consult that Set in O(n). For a single tenant's catalog this is well within range, and it avoids a synchronous-but-async UX where the user clicks "Generate SKU" and waits on a network call. The BE will still reject a duplicate SKU on save (existing 409 path), which is the actual correctness backstop.
- **`<MFG>-<CAT>-<NNNN>` abbreviation is alphanumeric-strip + uppercase + first-3-chars.** "Fleetguard" → "FLE", "Oil Filter" → "OIL", "AC" → "AC". Falls back to "GEN" if all chars are stripped (purely numeric input). Picked over a fancy slug-with-dictionary scheme because the Jira AC says simply "3-letter abbreviations" — anything more invents requirements.
- **Random retry first, then deterministic scan.** 10 random attempts almost always succeed on a sparse keyspace; the deterministic 0000–9999 fallback only matters at the asymptote where almost all slots are taken. The fallback also means the function always returns *some* SKU when a free slot exists, which is the correctness contract.
- **`limit=5` is hard-coded in the FE call.** The BE caps at 25 and defaults to 5; the AC says "up to 5 candidates", so we send the explicit value rather than relying on the BE default.
- **Errors swallowed in the duplicate-check pipeline.** A 500 from the BE would otherwise unsubscribe the entire `valueChanges` stream and silently disable duplicate detection for the rest of the session. `catchError → of([])` keeps the user typing and surfaces no candidates; the parent will simply not show the warning until a successful call returns a populated list.
