# FN-524: Show success message then auto-navigate to consent section after employment application submit

## Requirement
After a driver successfully submits the employment application, display a success message for 2-3 seconds, then automatically navigate to the consent forms section of the onboarding packet — so the driver can continue without manual intervention.

Previously the component set `this.submittedEmployment = true` but no navigation occurred.

## Acceptance Criteria
- [x] Success message is displayed immediately after successful submit ("Application submitted successfully! Taking you to consent forms…")
- [x] After 2.5 seconds, the page automatically navigates to the consent forms section
- [x] Navigation uses the onboarding packet's existing `setStep()` mechanism (not a hard page reload)
- [x] If the component is destroyed before the timer fires (e.g. user navigates away), the timer is safely cancelled via `ngOnDestroy`

## Agent
frontend

## Implementation Summary

### `onboarding-packet.component.ts`
- Added `OnDestroy` interface to the component class declaration
- Added `private autoNavTimer: ReturnType<typeof setTimeout> | null = null` property to track the pending timer
- Updated `submitEmploymentApplication()` success handler:
  - Changed success message to: `'Application submitted successfully! Taking you to consent forms…'`
  - Added `setTimeout(() => this.setStep('consent_forms'), 2500)` — navigates to the `consent_forms` step after 2.5 s using the existing step mechanism
  - Timer reference stored in `autoNavTimer` for cleanup
- Added `ngOnDestroy()` — clears the timer if the component unmounts before the 2.5 s fires (prevents memory leaks / stale callbacks)

### Key Decisions
- 2.5 s delay: within the 2-3 s range specified, long enough to read the message
- Used `setStep('consent_forms')` (existing method) rather than a hard route navigation — preserves packet state (packetId, token, sections) and matches the existing UX pattern
- `OnDestroy` cleanup: defensive measure for edge cases where the user navigates away or the packet reloads before the timer fires
