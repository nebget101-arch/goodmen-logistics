# FN-385: Driver Position shows wrong previous delivery — not using correct load ordering

## Requirement
Fix the previous delivery query in `computeTripMetrics()` so it correctly finds the chronologically previous load's delivery for a driver, not just the most recent delivery stop across all loads.

## Acceptance Criteria
- [x] Driver Position shows the delivery location from the chronologically previous load for that driver
- [x] For sequential loads, each load's Driver Position reflects the prior load's delivery
- [x] Query correctly orders by temporal sequence, handling NULL dates gracefully
- [x] Works correctly when loads have same-day dates

## Agent
backend

## Implementation Summary
Fixed the previous delivery query in `computeTripMetrics()` to:
1. Compute the current load's effective date from its first pickup stop date (or fallbacks)
2. Add a `<=` date constraint so only deliveries chronologically before or on the current load's date are considered
3. Add secondary sort by `l.created_at DESC` to break ties on same-day loads

## Files Changed
- `backend/packages/goodmen-shared/routes/loads.js` — fixed previous delivery query in `computeTripMetrics()`

## Key Decisions
- Used `COALESCE($3::date, l.created_at)` for the date constraint upper bound — if the current load has no pickup date, fall back to its created_at so the constraint is always meaningful
- Added `l.created_at DESC` as tiebreaker for same-day loads
- The `$3` parameter passes the current load's effective date (first pickup stop_date > loadRow.pickup_date > loadRow.created_at)
