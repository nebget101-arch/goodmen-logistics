# FN-501: Idle Truck Notification System

## Requirement
Detect trucks with active drivers that have had no recent loads but still accrue recurring deductions. Create alerts and send email + in-app notifications at two escalation levels.

## Acceptance Criteria
- [x] Identify trucks idle 7+ days with always-on recurring deductions
- [x] Create `week_1_idle` alert and notify dispatch, accounting, admin, manager
- [x] Identify trucks idle 14+ days with no response to week_1 alert
- [x] Create `week_2_no_response` alert with admin escalation
- [x] Avoid duplicate open alerts (idempotent per vehicle per alert_type)
- [x] `POST /api/idle-truck-monitor/run` ‚Äî trigger the check
- [x] `GET /api/idle-truck-monitor/alerts` ‚Äî list alerts with filters
- [x] `PATCH /api/idle-truck-monitor/alerts/:id/respond` ‚Äî acknowledge/resolve/escalate
- [x] In-app notification bell records per user
- [x] Angular alert history panel at `/idle-truck-alerts`

## Implementation Summary

### `idle-truck-monitor-service.js` (NEW)
- `runIdleTruckCheck(knex, userId)` ‚Äî main daily job
  - Fetches all active drivers with a truck assigned (`drivers.truck_id NOT NULL, driver_status = 'active'`)
  - For each: gets `lastLoadDate` (latest of pickup/delivery date from `loads`)
  - Skips if `idleDays < 7` or no always-on recurring deductions
  - **Week 1 (7‚Äì13 days idle)**: creates `week_1_idle` alert if no open alert exists; notifies dispatch/accounting/admin/manager via email + in-app
  - **Week 2 (14+ days idle)**: creates `week_2_no_response` alert if unresponded week_1 alert is >7 days old; notifies admin only with deactivation recommendations; back-fills week_1 if missing
- `getAlwaysOnDeductionTotal(knex, driverId, tenantId)` ‚Äî sums `recurring_deduction_rules` where `applies_when = 'always' OR NULL`
- `createAlert()` ‚Äî guarded by `hasTable('idle_truck_alerts')` for safe pre-migration
- `sendAlertNotifications()` ‚Äî fires email + in-app bell for each target user

### `idle-truck-monitor.js` (NEW route)
- `POST /run` ‚Äî admin/manager only; calls `runIdleTruckCheck`; returns `{ stats: { checked, week1Created, week2Created, errors } }`
- `GET /alerts` ‚Äî tenant-scoped; filters: `alert_type`, `response_status`, `vehicle_id`, `driver_id`, `limit`, `offset`; joins vehicles + drivers for display names
- `PATCH /alerts/:id/respond` ‚Äî validates `response_status` in `[acknowledged, resolved, escalated]`; records `responded_by`, `responded_at`

### `notifications.js` (NEW route ‚Äî FN-507)
- `GET /api/notifications` ‚Äî list notifications for current user
- `GET /api/notifications/unread-count` ‚Äî count for badge
- `PATCH /api/notifications/:id/read` ‚Äî mark one read
- `PATCH /api/notifications/read-all` ‚Äî mark all read

### `logistics-service/server.js` (MODIFIED)
- Mounted `idleTruckMonitorRouter` at `/api/idle-truck-monitor`
- Mounted `notificationsRouter` at `/api/notifications`

### Angular `IdleTruckAlertsComponent` (NEW ‚Äî FN-508)

- Filters: alert_type and response_status dropdowns (auto-apply on change)

- Unresolved badge count in page header (red pill)
- "Fleet Management" sidebar section added to `nav.config.ts`
