# FN-501: Idle Truck Notification System

## Requirement
As a fleet manager, I need to be notified when trucks are idle but still accruing recurring deductions, so I can take action to reduce unnecessary costs.

## Acceptance Criteria
- [ ] Daily scheduled job checks all active vehicles with assigned drivers
- [ ] 7+ days idle with always-on deductions → week_1_idle alert with accrued deduction total
- [ ] Notification to: dispatch manager, accounting team, Equipment Owner
- [ ] 14 days idle + no response on week_1 → week_2_no_response alert to admin
- [ ] Week 2 alert suggests removing truck from insurance and marking inactive
- [ ] Notification channels: in-app notification bell + email (if configured)
- [ ] Alert history visible in admin panel

## Implementation Summary (FN-506)
- Created idle truck monitoring service that scans all active vehicles with assigned drivers per tenant
- Detects vehicles idle 7+ days (no DELIVERED loads since completed_date) with always-on recurring deductions
- Calculates accrued deductions by summing enabled recurring_deduction_rules (applies_when = 'always') daily rates over the idle period
- Creates week_1_idle alerts at 7+ days, week_2_no_response alerts at 14+ days (only if week_1 alert is unresponded)
- Skips vehicles already alerted at the same tier (no duplicate alerts)
- Created REST API for listing, viewing, responding to, and manually triggering idle truck alerts
- All endpoints are tenant-scoped

## Key Decisions
- Mounted in vehicles-maintenance-service (port 5005) since alerts are vehicle-centric
- Loads table uses truck_id (not vehicle_id) for the vehicle FK -- join uses loads.truck_id = vehicles.id
- recurring_deduction_rules lacks tenant_id; rules are matched via driver_id / equipment_id
- Weekly deduction amounts are converted to daily rates for accrual calculation
- Vehicles with no delivered loads at all are skipped (no anchor date to measure idle from)
