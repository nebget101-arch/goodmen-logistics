## Root cause

`backend/packages/goodmen-shared/storage/r2-storage.js` requires `@aws-sdk/client-s3` and `@aws-sdk/s3-request-presigner` at module load (lines 2 and 8). Both are declared as deps of `@goodmen/shared`, but because `@goodmen/shared` is consumed via `file:../../packages/goodmen-shared`, npm does NOT reliably hoist transitive deps from `file:` deps to the consuming microservice. Every microservice that imports `r2-storage` must list these two packages in its OWN `package.json`. Three peers already do (`integrations-service`, `vehicles-maintenance-service`, `drivers-compliance-service`); `ai-service` did not.

`parts-invoice-intake-handler.js` and `parts-photo-intake-handler.js` (added by FN-1101 and FN-1096 respectively) lazy-require `r2-storage` *inside the request body* — so the service boots cleanly, and empty-body POSTs return a clean 400 from multer before the require fires. Only a real multipart upload (or valid JSON `imageBase64` payload) triggers the require, throws `MODULE_NOT_FOUND`, and crashes the Node process. Render auto-restarts the instance (~30s); during the window every in-flight request sees 502 — including a flood of cron-batch `/score-alert` POSTs at 00:09:19 and 00:09:34 UTC on 2026-05-06, which is what the user's screenshots captured.

Full Render runtime evidence (4 separate uncaught crashes between 00:08-00:16 UTC) is captured in the FN-1349 RCA comment on FN-1348.
