# FN-1598 — [P1] [Loads Import] Stage & Commit response field-name drift

**Type:** Bug · **Priority:** Highest (P1) · **Epic:** FN-1583 (Loads Import) · **Agent:** frontend
**Branch:** `frontend/FN-1598/loads-import-stage-commit-field-drift`

## Symptom

Step 4 (Validate) "Continue to Commit" stays disabled even when 272/272 rows are
OK. The wizard never reaches `/commit/:batchId`. The same FE/BE field-name drift
extends to the commit response, so the result step would render `0` for every
count even after the validate-step fix.

## Root cause

Wizard FE was authored against an earlier wire shape; FN-1584/FN-1585 landed the
BE with a different field naming. Two interfaces drifted:

- `StageResponse` — FE expected `okCount` / `needsReviewCount` / `errorCount`;
  BE returns `ok` / `needsReview` / `errors`. `undefined + undefined === NaN`,
  so `canProceed` stayed `false`.
- `CommitResponse` / `CommitDuplicate` — FE expected flat counts and rich
  duplicate metadata; BE returns `{ created: { auto, needsReview }, duplicates:
  [{ rowIndex, loadNumber, existingLoadId, existingLoadKey }], errors: [...] }`.

Folded both fixes into one branch — same RCA pattern.

## Implementation Summary

### Stage response (validate-step unblock)

- `StageResponse` interface in `loads-import.model.ts`:
  - `okCount` → `ok`
  - `needsReviewCount` → `needsReview`
  - `errorCount` → `errors`
  - `rows: StageRow[]` → `rows?: StageRow[]` (BE doesn't return per-row data
    today; validate-step gracefully degrades with a hint message)
- `validate-step.component.ts:canProceed` reads `result.ok + result.needsReview`.
- `validate-step.component.html` chips read the new field names; added a hint
  paragraph when `rows` is empty but `totalRows > 0`.
- `commit-step.component.ts:totalCommittable` and `commit-step.component.html`
  summary rows read the new field names.

### Commit response (option (b) — adapter at service)

- `CommitDuplicate` aligned to BE shape:
  - `rowNumber` → `rowIndex`
  - `attemptedLoadNumber` → `loadNumber`
  - Dropped unused `rate?` / `brokerName?` / `pickupCity?` / `deliveryCity?`
  - Added `existingLoadKey?: string | null` for the deep-link label.
- `CommitResponse` keeps its FE-flat shape. `loads-import.service.ts` adds a
  private `CommitWireResponse` type and an `adaptCommitResponse` mapper invoked
  from `commit()`. Components see the same FE shape they always did.
- `CommitResponse.errors` aligned to `{ rowIndex, message }` (matches BE; no
  components currently render `error.rowNumber`, so this is type-clean).
- `duplicate-review-modal.component.ts` CSV export now emits
  `row_index, load_number, existing_load_id, existing_load_key`.
- `duplicate-review-modal.component.html` simplified to three columns
  (Row / Attempted Load # / Existing Load) — broker/rate/pickup/delivery columns
  were always `'—'` because the BE never returned that data; surfacing them
  belongs on a follow-up if the modal genuinely needs the context. Existing-load
  link uses `existingLoadKey` as its visible label and `existingLoadId` as the
  query param.

### Result step

No changes — `result-step.component.ts` getters (`duplicatesCount`,
`errorsCount`) already fall back to `.length`, and the adapter populates
`autoCreatedCount` / `needsReviewCount` / `duplicatesSkippedCount` /
`errorCount` directly.

## Key Decisions

- **Picked option (b) — adapter at service boundary** for `CommitResponse`.
  The result-step + duplicate-review-modal already had partial fallbacks
  (`?? .length`, etc.); adapting once at the HTTP boundary keeps every
  downstream consumer untouched. Less code than scattering getters across
  components, and the `CommitWireResponse` type stays private to the service so
  components never see the BE shape.
- **Aligned `CommitDuplicate` to BE shape** rather than adapting it. The
  Jira AC explicitly asked for `rowIndex` / `loadNumber` and adding
  `existingLoadKey`. Adapting these would have been more code without payoff —
  the duplicate-review-modal is the only consumer.
- **Made `StageResponse.rows` optional** rather than removing `StageRow`. The
  type stays exported so a follow-up endpoint can populate it without another
  schema migration on the FE side.
