# FN-1077: Auto-Create-from-PDF flow drops PDFs and bypasses the 4-step wizard

## Requirement
Two related defects in the loads dashboard's AI "Auto-Create from PDF" modal (`showAutoModal`), reported 2026-05-03:

1. "Add More PDFs" appears to add a PDF but overwrites the previously selected PDF, so only the latest file is extracted and a single load is created.
2. After AI extraction (regardless of confidence), the flow opens the legacy `showManualModal` inline form instead of the new 4-step Load Wizard.

Both behaviors live in the same component and the same call path, so they should be fixed together.

## Acceptance Criteria
- [ ] Selecting a 2nd PDF in the Auto-Create modal does NOT silently replace the first; either the user is routed to the bulk extraction grid (preferred — `openBulkExtractionGrid([oldFile, newFile])` already exists) OR is shown an explicit "Replace selected PDF?" confirmation.
- [ ] Successful AI extraction (any confidence) closes `showAutoModal` and opens the 4-step wizard (`showLoadWizard`) with `wizardBasics` / `wizardStops` / `wizardDriverId` prefilled from the extraction payload, the source PDF queued in `wizardAiExtractedPdf`, and `wizardAiPrefilledFields` populated so per-field confidence badges (FN-818) render correctly.
- [ ] Low-confidence (e.g. `overall_confidence < 0.6`), vision-only/scanned PDFs (`provider === 'none'` with warning), and "no data" responses still route through the new wizard — the legacy `showManualModal` path is removed from `runAutoExtraction()`. Low-confidence fields show the existing red/yellow/green per-field highlighting; user can still edit and submit.
- [ ] Network/extraction errors keep the user in the Auto-Create modal with the existing error message AND offer a "Continue manually" button that opens the new wizard with just the PDF queued (no prefill).
- [ ] No regression to the bulk-upload modal flow (`bulkPdfFiles` / `runBulkUpload` / `BulkExtractionGridComponent`) — it already accumulates correctly.
- [ ] Cypress regression: upload one PDF → wizard opens prefilled. Upload PDF, then drop a second on the dropzone → bulk grid opens with both files (or replace-confirm prompt fires, depending on chosen UX). Force a low-confidence extraction (mock) → wizard opens, low-conf fields highlighted, submit succeeds.

### FN-1078 (subtask)

**Branch:** `frontend/FN-1078/auto-create-wizard-routing` (off `integration/FN-1077`)

- **`runAutoExtraction()`** — replaced all four `this.showManualModal = true` branches. Every successful response now flows through the new helper `_continueAutoToWizard(pdfFile, extraction)`, which closes `showAutoModal` and calls `openLoadWizardWithExtraction({ extraction, pdfFile })`.
  - Normal extraction → wizard with prefill + AI sparkle markers (FN-818).
  - `provider === 'none'` (scanned/vision-only) + warning → wizard with PDF queued only; warning surfaces on the dashboard via `errorMessage` (auto-clears after 6s).
  - Empty `data` → same wizard-with-PDF-only path; surface a transient "Extraction returned no data — continuing with manual entry" banner.
  - HTTP error → keep user in `showAutoModal`. New `autoError` copy directs them to the **Continue manually** footer button (`continueAutoManually()`), which routes to the wizard with PDF queued and no prefill.
- **`openLoadWizardWithExtraction()`** — mirrors `openLoadWizard()` (reset → smart defaults) and then layers extraction overrides on top. Smart-default pickup/delivery dates remain when extraction omits them.
- **`applyExtractionToWizard()`** — patches `wizardBasics` (brokerName, poNumber/loadId/orderId/proNumber fallback chain, rate, notes), replaces `wizardStops` when the extraction has structured stops (otherwise folds `pickup`/`delivery` onto the smart-default stops), and builds `wizardAiPrefilledFields` so step-basics renders the AI sparkle on AI-derived fields (`brokerName`, `poNumber`, `rate`, `notes`).
- **`_handleAutoModalFiles()`** — when a PDF is already queued and the user selects more (single or multi), merge `[queued, ...new]` (deduped, capped at 10) and route to `openBulkExtractionGrid`. No silent replacements.
- **HTML** — added a `[Continue manually]` footer button next to `Run Extraction`, visible only when `autoError && autoPdfFile && !autoExtracting`. Existing `clearAutoPdf()` button preserved.
- **Spec** — new `loads-dashboard.component.spec.ts` covers normal/no-data/vision-only/error extraction routing plus the re-select → bulk-extraction-grid path. Construction is direct (`new LoadsDashboardComponent(...)`) with stubbed services since the constructor itself is side-effect-free.

## Key Decisions
- **Re-select → grid, not confirm dialog**: AC offered grid-or-confirm; chose grid. It matches the existing `openHeroBulkPdfs`/`openBulkExtractionGrid` UX, has zero new components, and keeps both PDFs (vs. a confirm dialog that would force one to lose).
- **Vision-only routes to wizard, not modal**: AC explicitly removed the legacy `showManualModal` branch. Surface the warning via the dashboard's `errorMessage` (auto-clears after 6s) so the wizard isn't cluttered with a duplicate banner.
- **Smart defaults preserved**: `openLoadWizardWithExtraction` calls `applySmartDefaults()` *before* layering extraction. When extraction omits dates (common for OCR-fragile PDFs), the wizard still gets today/tomorrow defaults instead of empty fields.
- **`wizardAiPrefilledFields` built locally, not from `confidence_tiers`**: The Jira description referenced `data.confidence_tiers`, but the actual `LoadAiEndpointExtraction` model uses `confidence` (per-section) and `fieldConfidences` (per-field). FN-818 step-basics already infers tier color from `aiMetadata.fields` once the load is persisted; for the pre-save wizard, populating `wizardAiPrefilledFields` from the presence of extracted values is sufficient to drive the sparkle marker.
- **No karma `ng test` target**: angular.json has only `build` + `serve`. The new spec is type-checked by `tsc -p tsconfig.app.json` (clean) and the production build (clean). Functional regression is owned by the Cypress subtask FN-1079.
