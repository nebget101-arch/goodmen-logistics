# FN-771 — "New Load" button missing on Loads Dashboard

**Type:** Bug (High) · **Agent:** frontend · **Regression from:** FN-770
**Branch:** `frontend/FN-771/shared-permission-directives`

## Implementation Summary

FN-770 moved `LoadsDashboardComponent` into a new lazy-loaded `LoadsDashboardModule`.
The template uses `*appHasPermission="'loads.create'"` to gate the "New Load" dropdown,
but `HasPermissionDirective` + `HasAnyPermissionDirective` were declared **only** in
`AppModule`. Angular lazy-loaded modules have their own template compilation scope
and do **not** inherit `AppModule` declarations — so the structural directive was a
silent no-op in the lazy template, the wrapped `<div>` was never instantiated, and
the "New Load" button disappeared from the DOM.

**Fix:** move both directives into `SharedModule` (declare + export). Every FN-770
lazy module (including `LoadsDashboardModule`) already imports `SharedModule`, so
they automatically pick up the directives. Removed the now-duplicate declarations
from `AppModule`.

## Files Changed

1. `frontend/src/app/shared/shared.module.ts`
   - Import `HasPermissionDirective`, `HasAnyPermissionDirective` from `../directives/has-permission.directive`
   - Added both to `declarations` and `exports`

2. `frontend/src/app/app.module.ts`
   - Removed the two directives from the `declarations` array
   - Removed the import statement for them (no other references in the file)
   - Added a short comment pointing to `SharedModule` for discoverability
   - `SharedModule` is already in `AppModule.imports`, so the directives remain
     available in non-lazy template scopes used by `AppComponent` et al.

No changes to `has-permission.directive.ts` itself.

## Lazy-Module Audit (AC: "audit other FN-770 lazy modules")

Searched for `*appHasPermission` / `*appHasAnyPermission` structural-directive usage
in the templates of the four FN-770 lazy modules:

| Lazy Module | Template `*appHasPermission` usage? |
| --- | --- |
| `LoadsDashboardModule` | **Yes** — `loads-dashboard.component.html:13` (the bug) |
| `MaintenanceModule` (`components/maintenance/**`) | None |
| `WorkOrderModule` (`components/work-order/**`, incl. `tabs/*`) | None |
| `BarcodeManagementModule` (`components/barcode-management/**`) | None |

`.ts` files in those modules reference `access.hasPermission(...)` as method calls
against `AccessControlService`, which is NOT affected by module scope — only
structural-directive usage is. No other templates are silently hidden by this
regression. Fix is exactly one template's worth of impact, but centralizing the
directives in `SharedModule` also future-proofs any further lazy-module extractions.

## Key Decisions

- **Moved to `SharedModule`, not a new `DirectivesModule`** — `SharedModule` already
  exists, is already imported by every lazy module FN-770 touched, and is the
  conventional Angular location for shared structural directives. Adding a separate
  `DirectivesModule` would have been over-engineered for two directives.
- **Kept `AppModule` importing `SharedModule`** — it already did, so `AppComponent`'s
  own template scope continues to have access to the directives.
- **Did not touch `has-permission.directive.ts`** — directive logic is correct; this
  is purely a module-scope wiring bug.
