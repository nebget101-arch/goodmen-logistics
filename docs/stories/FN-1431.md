# FN-1431: D1 — AI Load-to-Driver Assignment

## Requirement

When a dispatcher opens an unassigned load, give them an AI-ranked list of candidate drivers with rationale (HOS, location, equipment match, prior history with the customer). Dispatcher picks one, assignment is recorded with the AI rationale.

## Acceptance Criteria

- [ ] AI service: `POST /api/ai/loads/recommend-driver` per the contract in the Jira story.
- [ ] Logistics route: `POST /api/loads/:id/recommend-driver` aggregates load + candidate-driver state and proxies to AI.
- [ ] Top 5 candidates; only positive HOS eligible.
- [ ] "Suggest driver" button on load drawer → modal with rationale → assign records the AI rationale.
- [ ] AI failure → "No suggestions available" with link to manual assign.
- [ ] Cypress E2E + p95 < 4s on dev seed.
- [ ] AI handler uses prompt caching on the static system prompt.

## Implementation Summary

### FN-1437 (subtask)

Built the AI handler that ranks candidate drivers for an unassigned load.

- New endpoint `POST /api/ai/loads/recommend-driver` registered in `ai-router.js` with full OpenAPI spec.
- Input validation: `loadId`, `load { originLat, originLng, equipmentClass, ... }`, non-empty `candidateDrivers`, optional `topN` (1–10, default 5). Bad input → `400 AI_BAD_REQUEST`.
- Pre-LLM filter: drivers with `hosRemainingHours <= 0` (or missing) are dropped before the model call. Equipment-mismatch drivers are still sent to the LLM with `equipmentMatch=false` so the model can mark them score 0 with a clear rationale.
- Distance computed server-side via haversine and passed to the model in the user message; the LLM is not asked to compute distance.
- Anthropic SDK call uses `system: [{ type: 'text', text: SYSTEM_PROMPT, cache_control: { type: 'ephemeral' } }]` to enable prompt caching on the static system prompt (FN-1431 acceptance criterion).
- Response merge: per-driver structured fields (`hosRemaining`, `distanceMiles`, `equipmentMatch`, `lastLoadWithCustomer`) are server-authored from pre-filter state — the LLM only contributes `score` and `rationale`. Hallucinated `driverId`s are dropped; eligible drivers missing from the AI response are backfilled with `score: 0` so the consumer always sees the full ranked field.
- Short-circuit: if zero drivers are eligible after pre-filter, the handler returns `{ candidates: [], reasoning: 'No drivers had sufficient HOS...' }` without calling the LLM.
- Error paths: parse failure → `502 AI_PARSE_ERROR`; SDK throw → `502 AI_UNAVAILABLE`. All paths emit `logAiInteraction` events.
- Unit spec covers: input validation matrix, zero/negative/missing HOS dropped, haversine sanity, LLM hallucination protection, eligible-driver backfill, `topN` truncation, schema-conforming happy path with prompt-caching assertion, zero-eligible short-circuit (no LLM call), 400 on bad input, 502 paths for parse and upstream errors.

## Key Decisions

### FN-1437 (subtask)
- **Server computes distance, not the LLM.** Haversine in the handler keeps `distanceMiles` deterministic and prevents the LLM from hallucinating geography. The LLM only synthesizes the score and natural-language rationale.
- **Equipment mismatch is not a pre-filter, HOS is.** A mismatched driver still goes to the LLM (with `equipmentMatch=false` in context) so the rationale explicitly explains why they were disqualified. HOS=0 drivers are pure noise — dropped pre-LLM to save tokens.
- **Structured fields are server-authored, not echoed from the LLM.** `mergeAndRank` ignores any `distanceMiles` / `equipmentMatch` / `hosRemaining` / `lastLoadWithCustomer` the model returns and reattaches the values from the pre-filter step. This is a hallucination-resistance measure proven by tests.
- **Anthropic prompt caching via system block array.** Used the `system: [{ type: 'text', text, cache_control: { type: 'ephemeral' } }]` form (SDK ≥ 0.27, current is 0.80). The system prompt is fully static — no per-request interpolation — so it caches cleanly.
- **`deps.anthropic` injection for testability** (matches `loads-nlq-handler` pattern). Mock client is exercised in unit tests; production path falls back to a lazy-initialized real client.
