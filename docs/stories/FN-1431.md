# FN-1431: D1 — AI Load-to-Driver Assignment

## Requirement

When a dispatcher opens an unassigned load, give them an AI-ranked list of candidate drivers with rationale (HOS, location, equipment match, prior history with the customer). Dispatcher picks one, assignment is recorded with the AI rationale.

## Acceptance Criteria

- [ ] AI service: `POST /api/ai/loads/recommend-driver` per the contract in the Jira story.
- [ ] Logistics route: `POST /api/loads/:id/recommend-driver` aggregates load + candidate-driver state and proxies to AI.
- [ ] Top 5 candidates; only positive HOS eligible.
- [ ] "Suggest driver" button on load drawer → modal with rationale → assign records the AI rationale.
- [ ] AI failure → "No suggestions available" with link to manual assign.
- [ ] Cypress E2E + p95 < 4s on dev seed.
- [ ] AI handler uses prompt caching on the static system prompt.

## API Contract

**AI service** — `POST /api/ai/loads/recommend-driver`

Request:
```json
{
  "loadId": "uuid",
  "load": { "originLat": 32.7, "originLng": -96.8, "pickupAt": "2026-05-07T08:00Z", "equipmentClass": "53FT_DRY", "customerId": "uuid" },
  "candidateDrivers": [
    { "driverId": "uuid", "name": "...", "lat": 32.5, "lng": -96.9, "hosRemainingHours": 6.5, "equipmentClass": "53FT_DRY", "lastLoadWithCustomer": "2026-04-12" }
  ]
}
```

Response:
```json
{
  "candidates": [
    { "driverId": "uuid", "score": 0.92, "rationale": "85mi away, 6.5h HOS, perfect equipment match, hauled for ACME 4 times in last 90d", "hosRemaining": 6.5, "distanceMiles": 85, "equipmentMatch": true, "lastLoadWithCustomer": "2026-04-12" }
  ],
  "reasoning": "Selected drivers within 200mi with sufficient HOS..."
}
```

**Logistics-service** — `POST /api/loads/:id/recommend-driver` — same response shape.

## Implementation Summary

### FN-1437 (subtask)

Built the AI handler that ranks candidate drivers for an unassigned load.

- New endpoint `POST /api/ai/loads/recommend-driver` registered in `ai-router.js` with full OpenAPI spec.
- Input validation: `loadId`, `load { originLat, originLng, equipmentClass, ... }`, non-empty `candidateDrivers`, optional `topN` (1–10, default 5). Bad input → `400 AI_BAD_REQUEST`.
- Pre-LLM filter: drivers with `hosRemainingHours <= 0` (or missing) are dropped before the model call. Equipment-mismatch drivers are still sent to the LLM with `equipmentMatch=false` so the model can mark them score 0 with a clear rationale.
- Distance computed server-side via haversine and passed to the model in the user message; the LLM is not asked to compute distance.
- Anthropic SDK call uses `system: [{ type: 'text', text: SYSTEM_PROMPT, cache_control: { type: 'ephemeral' } }]` to enable prompt caching on the static system prompt (FN-1431 acceptance criterion).
- Response merge: per-driver structured fields (`hosRemaining`, `distanceMiles`, `equipmentMatch`, `lastLoadWithCustomer`) are server-authored from pre-filter state — the LLM only contributes `score` and `rationale`. Hallucinated `driverId`s are dropped; eligible drivers missing from the AI response are backfilled with `score: 0` so the consumer always sees the full ranked field.
- Short-circuit: if zero drivers are eligible after pre-filter, the handler returns `{ candidates: [], reasoning: 'No drivers had sufficient HOS...' }` without calling the LLM.
- Error paths: parse failure → `502 AI_PARSE_ERROR`; SDK throw → `502 AI_UNAVAILABLE`. All paths emit `logAiInteraction` events.
- Unit spec covers: input validation matrix, zero/negative/missing HOS dropped, haversine sanity, LLM hallucination protection, eligible-driver backfill, `topN` truncation, schema-conforming happy path with prompt-caching assertion, zero-eligible short-circuit (no LLM call), 400 on bad input, 502 paths for parse and upstream errors.

### FN-1438 (subtask)

Built the logistics-service route that feeds the AI handler from FN-1437.

- New endpoint `POST /api/loads/:id/recommend-driver` in `backend/packages/goodmen-shared/routes/loads.js`. Role-gated to `admin`/`dispatch`; tenant- and operating-entity-scoped via `req.context`.
- Load lookup uses a single CTE-flavoured query that joins `load_stops` (first PICKUP) and `zip_codes` to resolve the origin lat/lng. Tenant + OE bound positionally — out-of-tenant ids return 404.
- `equipmentClass` is read from `loads.ai_metadata->>'equipmentClass'` (the existing JSONB column added in FN-816). `customerId` is the load's `broker_id` (no separate customer entity in this schema).
- Candidate aggregation runs a second query: tenant + OE drivers with `status = 'active'`, joined to (a) the latest `hos_records` row, (b) the driver's last DELIVERY stop (zip → `zip_codes` for current location), (c) the driver's truck (`vehicles.vehicle_type` as equipment class), and (d) the most recent prior load with the same broker (treated as "lastLoadWithCustomer"). Drivers with no HOS record at all are kept (forwarded with `hosRemainingHours: null`) so the AI can decide; drivers with a known but spent HOS (driving ≥ 11h or duty ≥ 14h) are dropped pre-AI.
- App-side haversine sorts the eligible pool ascending by distance and caps it at 25 before posting to the AI service. Drivers with no resolved location sort last.
- AI call uses `process.env.AI_SERVICE_URL` (defaults to `http://localhost:4100`), `POST /api/ai/loads/recommend-driver`, with a 15s `AbortSignal.timeout` so a slow upstream cannot pin a request thread past the p95 budget. Per-request timing is logged via `dtLogger.trackRequest`.
- AI failure paths (network throw, non-2xx) collapse to `200 { success: true, candidates: [], reasoning: 'AI service unavailable' }` so the dispatcher UI (FN-1439) shows the manual-assign fallback rather than a hard error.
- AI rationale persistence is **not** in this subtask — the route is read-only. The dispatcher's accept action (FN-1439) writes the rationale into the existing `loads.ai_metadata` JSONB column (FN-816). No new migration was needed; the description's "OR write to an existing JSONB metadata column" branch was taken.
- Unit spec covers: 400 on non-uuid load id, tenant/OE binding on the load lookup (params positional), HOS pre-filter (spent driver dropped before AI call), AI throw → graceful 200 fallback, AI non-2xx → graceful 200 fallback, 25-driver cap honoured by ascending distance, role gate (drivers forbidden).

## Key Decisions

### FN-1437 (subtask)
- **Server computes distance, not the LLM.** Haversine in the handler keeps `distanceMiles` deterministic and prevents the LLM from hallucinating geography. The LLM only synthesizes the score and natural-language rationale.
- **Equipment mismatch is not a pre-filter, HOS is.** A mismatched driver still goes to the LLM (with `equipmentMatch=false` in context) so the rationale explicitly explains why they were disqualified. HOS=0 drivers are pure noise — dropped pre-LLM to save tokens.
- **Structured fields are server-authored, not echoed from the LLM.** `mergeAndRank` ignores any `distanceMiles` / `equipmentMatch` / `hosRemaining` / `lastLoadWithCustomer` the model returns and reattaches the values from the pre-filter step. This is a hallucination-resistance measure proven by tests.
- **Anthropic prompt caching via system block array.** Used the `system: [{ type: 'text', text, cache_control: { type: 'ephemeral' } }]` form (SDK ≥ 0.27, current is 0.80). The system prompt is fully static — no per-request interpolation — so it caches cleanly.
- **`deps.anthropic` injection for testability** (matches `loads-nlq-handler` pattern). Mock client is exercised in unit tests; production path falls back to a lazy-initialized real client.

### FN-1438 (subtask)
- **No new migration.** `loads.ai_metadata` JSONB exists from FN-816, and `equipmentClass` lives there already. The dispatcher's accept flow (FN-1439) will write the rationale into the same column — keeping the storage decision aligned with how AI metadata is already persisted on loads.
- **Pool cap on the route side, not on the LLM side.** FN-1437's handler can take any number of drivers and trims to top-N, but sending hundreds of candidates per request bloats latency and tokens. Capping at 25 closest by haversine on the route side stays well under the 200-mile spec radius for a typical fleet without needing an explicit radius parameter.
- **Drivers with unknown HOS are forwarded, not dropped.** The route only drops a driver when we have a positive HOS record but it's spent. A driver with no `hos_records` row at all is still included (with `hosRemainingHours: null`) and the AI handler decides — partial dev seed data should not silently empty the candidate list.
- **AI failure → success response, never a 5xx.** The acceptance criteria are explicit: dispatchers should never be blocked. So fetch throws and AI non-2xx both collapse to `200 { candidates: [], reasoning: 'AI service unavailable' }`. The frontend reads `candidates.length === 0` and shows the manual-assign fallback.
- **Equipment class derived from `vehicles.vehicle_type`.** The drivers schema has no explicit equipment field; the truck assigned to the driver carries it. Forwarded as-is (string or null) so the AI handler's equipment-match logic can compare against the load's `equipmentClass` literal.
- **Broker as customer.** No separate `customers` table exists; loads are keyed by `broker_id`. Treating the broker as the customer for the "lastLoadWithCustomer" lookup matches how the rest of the loads UI groups history (e.g., the broker pages in the dispatch board).

### FN-1439

- **Reused `PUT /api/loads/:id` for the assignment** instead of inventing a separate `assign-driver` endpoint. Per the parent story AC, only the *trigger* and *rationale capture* are new — the field set on the assignment endpoint is unchanged except for three opt-in metadata fields (`assignmentSource`, `assignmentRationale`, `assignmentScore`).
- **CTA gated to `!driverId` only**, matching the AC "AI failure → 'No suggestions available' with link to manual assign." Re-suggesting after an existing assignment was deemed out of scope; manual driver dropdown remains the source of truth for re-assigns.
- **Avoided getter-based `[options]` bindings** for the candidate list — used a plain `*ngFor` over an array property, per the FN-317 RCA project memory (infinite change-detection loops).
- **AI failure surfaces as the empty state** instead of a hard error banner, per the AC. The error banner inside the modal is reserved for assignment failures (i.e., when the chosen candidate can't be saved).
- **Modal lives in `loads-dashboard/`** (not a separate `dispatch-board/` directory) because the load drawer is owned by `LoadsDashboardModule`. The Jira description's "dispatch board" wording refers to the dispatcher-facing view, not a separate module.
