# FN-1632: DQF — AI CDL upload accepts multiple files (batch extract + sequential review)

## Requirement

The DQF "Upload CDL" CTA shipped in FN-1625 only supports one CDL at a time. Onboarding teams routinely receive a stack of CDLs and have to repeat the upload + modal-review cycle once per driver. Let the same flow accept multiple CDLs in one selection, extract each independently, and surface them in a small review queue so the user can step through Add-New-Driver pre-fill modals one after another (current single-driver save semantics preserved).

## Acceptance Criteria

- [ ] CTA file input on `/drivers/dqf` accepts multiple files; button label switches to "Upload CDLs". Single-file selection still works (regression-safe).
- [ ] N files (1 ≤ N ≤ 10) → ONE multipart POST to `/api/dqf/cdl-extract` with all files. Per-file MIME + size pre-validation client-side; rejected files listed back inline; only accepted files in the call.
- [ ] Backend accepts `upload.array('files', 10)` AND remains backward-compatible with `upload.single('file')`. Per-file mime/size enforcement at route level mirrors current single-file rules.
- [ ] Backend extracts each file independently via the existing `extractCdl()` service. Returns `{ results: CdlExtractionResponse[] }` in upload order. **Decision (to confirm at impl):** single-file POSTs return either the legacy single shape OR the new array shape — picked-one shape documented here once FE+BE agree.
- [ ] FE shows a compact CDL extraction queue under the header (filename + status pill: Ready / Couldn't read / Done). Review opens existing Add-New-Driver modal pre-filled. Save returns to queue with row marked Done.
- [ ] Single failure in batch isolates: failed row keeps the existing fallback toast text; other rows still reviewable.
- [ ] File cap enforced both client- and server-side at 10. Server returns `413 { message: "Too many files (max 10)" }`.
- [ ] No PII in logs (preserves FN-1625 invariant). Per-file metadata only — `dtLogger` records counts, total bytes, processingMs.
- [ ] All three subtasks (FE, BE, QA) merge to `integration/FN-1632`; final PR `integration/FN-1632 → dev`.

## Architecture note

The AI handler (`backend/microservices/ai-service/src/handlers/cdl-vision-handler.js`) and the per-file extraction service (`backend/packages/goodmen-shared/services/cdl-extraction-service.js`) stay byte-identical — this is purely a thin "iterate over N inputs" change in the route + a queue UI in the FE. Confidence floor, server-side re-validation, and the camelCase wire contract per file are unchanged.

## Out of scope

- Bulk-save (auto-creating N drivers without per-driver review). Per-driver modal review is preserved.
- Parallelism tuning beyond a simple `Promise.all` over the batch on the backend.
- Re-ordering / drag of the queue.
- Persisting uploaded CDL bytes (still intentionally not stored, per FN-1625).

## Implementation Summary

### FN-1634 (subtask — backend)

`POST /api/dqf/cdl-extract` now accepts up to 10 CDL files in one multipart upload. The per-file `extractCdl()` service is unchanged — this is purely a thin route-layer change.

- Multer config switched from `upload.single('file')` to `upload.fields([{ name: 'files', maxCount: 10 }, { name: 'file', maxCount: 1 }])`. Both shapes coexist.
- Handler normalizes `req.files.files` (new) or `req.files.file[0]` (legacy) into a `files` array, validates each entry against `CDL_ALLOWED_MIMETYPES` (whole batch rejected on first invalid mime → 400), then `Promise.all`s `extractCdl({ fileBuffer, mimeType })`.
- 413 mappings: `LIMIT_FILE_SIZE` → `"File exceeds 10 MB limit"` (unchanged); new `LIMIT_FILE_COUNT` / `LIMIT_UNEXPECTED_FILE` → `"Too many files (max 10)"`.
- New batch-complete log line: `dtLogger.info('cdl_extract_batch_complete', { fileCount, successCount, failureCount, totalBytes, processingMs })`. No filenames, no field values.
- `extractCdl()`'s existing per-file failure mode (`{ success:false, reason }`, never throws) means each batch entry is always populated even when the AI rejects an individual file.

### FN-1633 (subtask — frontend)

The DQF "Upload CDL" CTA on `/drivers/dqf` now accepts up to 10 CDLs in one
selection. CTA copy switched to "Upload CDLs" / "Reading CDLs…". A compact queue
card renders under the page header (only when `cdlBatchQueue.length > 0`) with
one row per upload — filename, status pill (Ready / Couldn't read / Done),
optional reason text for failures, and a Review button on Ready rows.

Selection flow:

1. `onCdlFileSelected` splits the picked files into `accepted` (mime + size +
   max-count pre-validated) and `rejected` rows. Rejected rows go straight into
   the queue marked `failed` with their reason. If `> 10` files are selected,
   the queue stays empty and an inline error toast is shown — no POST is made.
2. `extractCdls(accepted)` issues ONE multipart POST to `/api/dqf/cdl-extract`
   with each file appended under the `files` key (matches the FN-1634
   `upload.fields([{ name: 'files', maxCount: 10 }, …])` server config).
3. `populateQueueFromResponse` appends one queue row per server result in upload
   order: `success && extracted` → `ready`, otherwise `failed` with the existing
   "Couldn't read CDL — please enter manually." copy.
4. Clicking Review on a Ready row stashes the row as `activeQueueRow` and calls
   the existing `applyCdlExtraction(row.response)` — which opens the Add New
   Driver modal pre-filled exactly as the single-file flow did.
5. When the modal closes (save via `addDriver` or cancel via `toggleAddForm`),
   `completeActiveQueueRow()` flips the active row's status to `done` and
   returns the user to the queue. The user steps through the remaining Ready
   rows the same way until the queue is empty or they Clear it.

Concurrency: the existing `extractingCdl` flag now also gates the multi-file
POST and disables the Review button while a batch is in flight. The single-file
`extractCdl(file)` API method on `ApiService` is intentionally retained for
backward-compat per the ticket constraint — the drivers component no longer
calls it; only `extractCdls` is used.

Manual-only fields (`phone`, `hireDate`, `medicalCertExpiry`) remain protected
by the existing `cdlManualOnlyFields` defensive filter on every applied row.

## Key Decisions

### FN-1634: response-shape branching (legacy vs. batch)

**Chosen:** dual-shape — branch on the field name used in the upload.

- `files` field present (1..10 entries) → `{ results: CdlExtractionResponse[] }` envelope. Always an array, even when only one file is uploaded under the new field name. **FE-1633 should always use `files`** so it can rely on a single response shape.
- `file` field present (1 entry) and no `files` field → returns the single `CdlExtractionResponse` object unchanged from FN-1627. Preserved purely for backward-compat with any out-of-band caller still on the pre-batch contract.

Rationale for dual-shape (vs. always returning `{results:[]}`): keeps the FN-1627 wire contract intact byte-for-byte for the legacy `file` field, so anything currently calling that field — internal scripts, manual curl, the original FN-1625 FE if it were ever rolled back — continues to work without code change. New FE work targets `files` and never sees the legacy shape.

### FN-1633: FE wire contract aligned with FN-1634 dual-shape

- **FE always posts under the `files` field** (even for a single selection) and consumes the `{ results: CdlExtractionResponse[] }` envelope. The drivers component never sees the legacy single-object response.
- **`extractCdl(file)` retained on `ApiService`** (posts under the legacy `file` field) per the ticket constraint, even though no FE caller invokes it now. Kept until a follow-up confirms there are no out-of-band callers (curl scripts, future external integrations) that rely on the method's legacy behavior. Safe to delete in a later cleanup pass.
- **Queue persistence:** the queue is plain template state on the drivers component (no global service, no router state). Picking another batch from the CTA replaces the queue. The "Clear" button explicitly drops it.
- **Done vs. Skipped:** the ticket allows distinct internal states; we intentionally collapse both modal-close paths (save and cancel) into a single `done` status because the pill column is `Ready / Couldn't read / Done` and the user only needs to know the row no longer needs action.
- **Single-file UX preserved:** selecting one file via the multi picker still surfaces a one-row queue card → Review → modal pre-fill → save. No regression to the FN-1625 / FN-1628 happy path.
