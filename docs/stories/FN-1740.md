# FN-1740: Render shop logo on shop/parts invoices

## Requirement
Render the shop's logo on generated shop/parts invoices. Replace the existing `INVOICE_LOGO_PATH` environment-variable lookup in `invoice-pdf.js` with a DB-driven resolution: `invoice.location_id` → location logo → tenant fallback → no-logo layout. Pull the buffer from R2.

- [x] `INVOICE_LOGO_PATH` env lookup removed (or demoted to dev-only fallback — document the choice).

## Implementation Summary
The invoice PDF logo is now DB-driven. When `POST /api/invoices/:id/pdf` runs, the route resolves a branding logo via `resolveInvoiceLogoBuffer(invoice, location)`:
1. `locations.logo_storage_key` (the invoice's service location), else
2. `tenants.logo_storage_key` (tenant-level fallback keyed on `invoice.tenant_id`).

The resolved R2 object is downloaded via the existing `r2-storage.downloadBuffer(key)` and passed as a `logoBuffer` into `buildInvoicePdf`. The PDF embeds it at the existing ~80px header placement (`doc.image(logoBuffer, startX, y, { width: 80 })`). Resolution and embedding are both best-effort: a missing key, an R2 fetch failure, or a corrupt/unsupported image all fall back to the existing no-logo layout and the invoice still generates.

`INVOICE_LOGO_PATH` is demoted to a **dev-only fallback**: it is only consulted when no DB logo buffer was provided, and only for local rendering. The previous hardcoded `../assets/logo.png` default was removed.

## Key Decisions
- **Logo resolution lives in the route, not the PDF util.** `buildInvoicePdf` stays a pure synchronous PDF builder that takes a ready `logoBuffer`; the route owns DB + R2 I/O (matching the existing pattern where the route already does `uploadBuffer`/`getSignedDownloadUrl`). This keeps the util easily unit-testable without mocking DB/R2.
- **Did not reuse FN-1746's `logo-embed.helper.js`** — it hadn't landed on `dev` (targets `integration/FN-1739`). Resolving inline avoids a cross-integration-branch dependency. If/when that helper lands, `resolveInvoiceLogoBuffer` can delegate to it.
- **`INVOICE_LOGO_PATH` demoted, not removed**, to preserve a zero-config local rendering path for devs without R2 credentials. It is never used when a DB logo is present.
- **Non-fatal everywhere**: resolution returns `null` on any failure; embedding is wrapped in try/catch. An invoice never fails to generate because of a logo.
