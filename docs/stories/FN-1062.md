## Implementation Summary

### FN-1063 (subtask)

Single frontend pass that delivers all five items.

**1. Page-size selector** — `pageSize` default flipped from 25 to 100; new `pageSizeOptions` array (25 / 50 / 100 / 200) drives a `<select>` next to the pagination buttons. `onPageSizeChange()` resets `page = 1`, re-fetches loads, and persists via `userPreferences.patchLoadsDashboard({ pageSize })`. `loadUserPreferences()` hydrates `prefs.pageSize` and re-fetches once if it differs from the in-memory default. Backend already supports 1–200 (FN-769 / `MAX_PAGE_SIZE`), so no API change needed.

**2. Action-row visibility** — `.row-toolbar` now starts at `opacity: 0.55` with `pointer-events: auto` and brightens to `1` on row hover / focus / drawer-active state. Removed the previous `transform: translateY(1px)` shimmy since buttons are no longer "appearing." Buttons remain hidden by `*ngIf="canDeleteLoad(load)"` / `*appHasPermission="'loads.create'"` for users without permission — that gating is correct and out of scope here.

**3. View / Edit / load-number → drawer** — load-number link, View button, and Edit button now invoke `openLoadDrawer()` instead of the legacy `openEdit()` / `openDetails()` modals. `openLoadDrawer(loadId, focusLowConfidence, mode)` gained a third `mode: 'view' | 'edit'` argument that flows into the drawer via a new `[initialMode]` input. `openDetails()` and `openEdit()` are intentionally retained for `openEditAndShowMap()` and the `loadId` query-param entry path (line ~1000 in ngOnInit).

**4. Clone Existing Load picker** — `openCloneLoad()` was a stub that flashed "Clone load is coming soon"; replaced with a real picker. It opens an inline modal (`showCloneLoadPickerModal`) with a debounced search input that hits `loadsService.listLoads({ q, page: 1, pageSize: 20 })` and renders the top 20 matches in a clickable list. Selection calls `loadsService.cloneLoad(id)` → the backend returns an in-memory draft (`buildDraftFromLoad`, no DB write) → frontend persists it via `createLoad(draft)` → `loadLoads()` refreshes the list → `openLoadDrawer(newId, false, 'edit')`. The stale `// TODO: ... (FN-724 sibling subtask)` comment is gone.

**5. Row Clone button** — `cloneLoad(load)` now persists the returned draft via `createLoad(draft)` instead of seeding the legacy manual-entry form (`applyDraftToManualForm`). On success it refreshes the list and opens the new row in the drawer in edit mode. `applyDraftToManualForm` and `openManualEntry` remain in place for `createReturnLoad`, which still uses the modal flow (out of scope for this story).

**Drawer initialMode input** — `LoadDetailDrawerComponent` gained `@Input() initialMode: 'view' | 'edit' = 'view'`. The drawer's footer now switches between an "Edit" affordance (in view mode) and the existing Save button (in edit mode); the Cancel button label flips to "Close" in view mode. `enterEditMode()` flips the local `mode` flag to `'edit'`. `ngOnChanges` resets `mode` to `initialMode` whenever `loadId` changes so prev/next navigation respects the entry mode.

## Key Decisions

1. **Edit-mode entry point** — went with the recommended `@Input() initialMode: 'view' | 'edit'`. The drawer keeps a public `mode` flag (mirrors the input on init / loadId change) and renders either an "Edit" button (view mode) or the existing "Save" button (edit mode). Form fields stay visually editable in both modes — view mode just gates *persistence* by hiding Save. This avoids touching every nested step component (`step-basics`, `step-stops`, `wizard-step-driver`, `step-attachments`) to plumb a `readonly` prop, which would have ballooned scope. The edit affordance gives users a clear way to opt into editing without an extra round-trip click.

2. **Clone picker** — went with an **inline modal** in the dashboard component rather than a separate `load-clone-picker/` component. The picker is ~50 lines of HTML and zero non-trivial logic (debounced search → `listLoads` → click → `cloneLoad` → `createLoad` → `openLoadDrawer`). Splitting it into its own component would have added a new file with one consumer and forced a parent → child event API for what is fundamentally three callbacks. We can extract later if a second consumer appears.

3. **Action-row visibility** — confirmed the root cause was `opacity: 0` + `pointer-events: none` until row hover. The toolbar is now always visible at `opacity: 0.55` and brightens to full on row hover / focus / `.row-drawer-active`. This makes actions discoverable for keyboard users and first-time users without the "wall of icons" feel of fully-saturated chrome on every row. Permission-gated buttons (Clone, Delete) remain hidden when the user lacks the role — that's correct behavior, separate from visibility.

4. **Persisting the clone instead of seeding a modal** — the spec said "open the resulting draft in the side drawer", but the backend `POST /loads/:id/clone` returns an in-memory payload with no DB id (`buildDraftFromLoad` does not insert). To open the drawer (which needs an id to fetch detail), we call `createLoad(draft)` immediately after the clone, then open the drawer on the new id. Trade-off: clicking Clone now creates a real DRAFT row in the list (consistent with how AI-extracted DRAFT loads work). Users can delete the row from the toolbar if they cancel out — same as today's AI-extraction flow.
