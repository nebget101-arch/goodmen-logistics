# FN-1499: [Bug] Repair history widget shows "Failed to fetch repair history summary" for vehicles with no history

## Requirement
Vehicles with no prior repair history must NOT trigger a red error in the WO repair-history widget. They should render the same neutral "Not enough history" pill that the AI handler already produces when `insufficientHistory: true`.

## Reported error string
> Failed to fetch repair history summary

This text comes from the 500 catch in `backend/packages/goodmen-shared/routes/vehicles.js:1651`.

## Root Cause (proposed)
Two layers contribute:

1. **Backend** (`services/vehicle-repair-history.service.js`):
   - `fetchVehicleWorkOrderHistory` resolves only against `vehicles`/`all_vehicles`, not `customer_vehicles`. WO form's `vehicleId` comes from `customer_vehicles` → null → 404.
   - `requestRepairHistorySummary` axios call lacks try/catch around the network call itself (validateStatus only suppresses HTTP-status errors, not DNS / ECONNRESET / timeouts) → bubbles to route catch → 500 instead of 502.
   - No structured short-circuit when vehicle exists but `history.length === 0` — sends an empty history slice to AI which returns `insufficientHistory: true`, but if AI is down or slow, this becomes a 500/502 instead of a free 200.

2. **Frontend** (`repair-history-widget.component.ts:96-101`):
   - Any non-2xx → `state = 'error'` + the response body's message string. 404 ("vehicle not found") and 500 ("Failed to fetch") both render as red errors. Should render as neutral empty state for 404, retry chip for 502, error UI only for 500.

## Acceptance Criteria
- [ ] Vehicle with no history → 200 `{ insufficientHistory: true, count: 0, ... }` from backend
- [ ] vehicleId resolved against both `vehicles` AND `customer_vehicles`
- [ ] AI down → 502 reliably (axios throws translated)
- [ ] Frontend: 404 → neutral pill; 502 → retry chip; 500 → error UI
- [ ] Component specs cover all four paths
- [ ] No "Failed to fetch repair history summary" string visible to user in normal "no history" path

## Implementation Summary

### FN-1500 (subtask)
Backend service `vehicle-repair-history.service.js` and the `/repair-history-summary` route catch reshaped so that "no history" is a 200, network failures map cleanly to `ai_unavailable`, and unhandled 500s carry actionable context to SigNoz.

- `fetchVehicleWorkOrderHistory` now tries `vehicles`/`all_vehicles` first and falls back to `customer_vehicles WHERE vehicle_uuid = $1 AND tenant_id = $2` before returning null. Both lookups stay tenant-scoped — a foreign tenant's vehicle still 404s.
- `getRepairHistorySummary` short-circuits when `fetched.history.length === 0` and returns `{ insufficientHistory: true, count: 0, vin, windowDays }` without round-tripping to the AI service. The empty body is cached under the same 5-minute TTL so re-renders within the window skip both DB and AI.
- `requestRepairHistorySummary` wraps the axios call in try/catch. Network/DNS/timeout/ECONNRESET errors no longer bubble — they translate to `{ status: 0, data: null }` so the existing non-200 branch in the orchestrator returns `{ ok: false, reason: 'ai_unavailable' }` and the route maps that to 502.
- Route 500 catch replaces `console.error` with `dtLogger.error('vehicle_repair_history_summary_failed_unhandled', { error, stack, vehicleId, tenantId, windowDays })` so unhandled failures are debuggable in SigNoz with full request context. The redundant first `dtLogger.error` was folded into the same call.

## Key Decisions

### FN-1500 (subtask)
- **Short-circuit lives in the orchestrator, not the AI handler.** The spec gave a choice between letting the AI handler keep deciding "insufficient history" or short-circuiting client-side. We picked the orchestrator because (1) it eliminates a guaranteed-pointless network round-trip per render, (2) the response is now resilient when the AI service is down (a no-history vehicle still gets a 200 instead of falling through to 502), and (3) the body shape `{ insufficientHistory, count, vin, windowDays }` is what the widget needs anyway.
- **Empty body cached under same TTL.** A page revisit within 5 minutes shouldn't re-hit Postgres for a known empty result.
- **Folded the duplicate dtLogger.error into the new richer call.** The original ticket said "replace console.error with dtLogger.error". Leaving two dtLogger.error calls in one catch would produce duplicate SigNoz events with overlapping fields. Consolidated into one call carrying error.message, stack, vehicleId, tenantId, windowDays.
- **Did not change the contract for vehicles that exist but have a null VIN.** Still returns `{ vin: null, history: [] }` from the fetcher → orchestrator now short-circuits that as `insufficientHistory: true` (previously it would have called AI with `vin: null` which is wrong).

### FN-1501 (subtask)
- **Frontend-side 404 fold-in is independent of the backend fix (FN-1500).** Even though FN-1500 now returns 200 for no-history vehicles, this guard keeps the widget non-noisy if any other 404 path slips through (vehicle deleted mid-session, customer/tenant mismatch, etc.). Defence-in-depth, not redundant.
- **Reused existing tokens.** AC required "no new tokens." `__badge--insufficient` already produces a neutral grey pill; `__badge--unavailable` reuses the warning palette already used by `risk-medium`. No new CSS variables added.
- **Retry button stops propagation.** The widget's outer row is `role="button"` for expand-toggle. Without `event.stopPropagation()`, the Retry click would also bubble to that handler (`canExpand` is false here so it'd no-op, but stopping propagation is more deliberate and survives future template refactors).
