### FN-1765 (subtask) ‚Äî Admin-gated entry point + route to branding/logo settings

**Implementation summary**
- Added a **Branding & Logos** admin menu link to `app.component.html`, pointing at the existing `/admin/branding` route. Added to both admin menu blocks (desktop dropdown + mobile dropdown), placed directly after the *Inbound Email* link.
- Each link is gated with `*ngIf="canSeeAdminMenuLink('/admin/branding', adminMenuPermissions)"` ‚Äî identical pattern to the sibling admin links (`/admin/multi-mc`, `/admin/locations`, `/admin/inbound-email`). `adminMenuPermissions = [PERMISSIONS.ROLES_MANAGE, PERMISSIONS.ACCESS_ADMIN, PERMISSIONS.USERS_EDIT]`, so the link renders for admins of any tenant and is hidden for non-admins.
- **Route was already registered and gated** by FN-1744 in `app-routing.module.ts`:
  ```
  { path: 'admin/branding', component: BrandingSettingsComponent,
    canActivate: [AuthGuard, PermissionGuard],
    data: { anyPermission: [PERMISSIONS.ACCESS_ADMIN, PERMISSIONS.ROLES_MANAGE, PERMISSIONS.LOCATIONS_MANAGE] } }
  ```
  No route change needed ‚Äî the "register/confirm route" criterion is satisfied (confirmed). The remaining gap closed by this subtask was the missing **menu entry point**.
- No tenant allowlist / feature flag added ‚Äî link shows for admins of every tenant.

**Files Changed**
- `frontend/src/app/app.component.html` ‚Äî added the Branding & Logos admin link in both admin menu blocks (desktop + mobile).

**Key decisions**
- Reused the existing `canSeeAdminMenuLink(path, adminMenuPermissions)` helper rather than introducing a new gate, keeping the link consistent with every other admin entry point. The route's own `PermissionGuard` provides defense-in-depth for direct navigation.
- Icon: `branding_watermark` (Material Symbols), label "Branding & Logos".
- Route path is `/admin/branding` (as registered by FN-1744), not `/settings/branding` as the ticket's older note suggested ‚Äî wired the link to the route that actually exists on `dev`.
