# FN-412: Update exception handling and import history for split rows, skipped rows, and new exception types

## Requirement
Fleet operators need import history to show raw/split/skipped row counts, and the exception queue must handle new exception types (missing_card_number, money_code_skip, low_confidence_mapping).

## Acceptance Criteria
- [x] Import history shows raw_rows, split_rows, skipped_rows counts
- [x] Exception queue handles missing_card_number type
- [x] Exception queue handles money_code_skip type
- [x] Batch detail view can filter by split/skipped rows
- [x] Audit trail preserved: batch_row raw_payload for all row types

## Implementation Summary

### Migration
- Added `raw_rows` (nullable integer) and `split_rows` (default 0) to `fuel_import_batches`

### fuel-service.js
- `stageBatch()` now tracks `raw_rows` (original file count) and computes `split_rows` (expanded - raw count)
- Return value includes `rawRows` and `splitRows` in addition to existing counts
- `buildExceptionMessage()` handles `money_code_skip` and `low_confidence_mapping` types

### fuel.js routes
- Batch detail endpoint (`GET /import/batches/:id`) accepts `?resolution_status=` query param to filter rows by status (skipped, valid, failed, etc.)
- Exceptions endpoint (`GET /exceptions`) accepts `?exception_type=` query param to filter by exception type (missing_card_number, unmatched_truck, etc.)

## Files Changed

## Key Decisions
- `raw_rows` is nullable (for backcompat with existing batches that don't have this tracked)
- `split_rows` computed as `expandedRowNum - rawRows.length` (difference between post-split and pre-split counts)
- Exception type filtering done server-side via query param rather than client-side filtering
