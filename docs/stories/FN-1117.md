# FN-1117: Reports — Natural-language filter parser

## Requirement
Optional NL input above the structured filters parses queries like "revenue by driver last month, exclude team leads" into filter params via Claude Haiku. Mirrors `loads-nlq-handler` pattern.

- [ ] Prompt caching on system prompt + filter schema; `cache_read_input_tokens > 0` on second call
- [ ] Requires `REPORTS_VIEW`; 403 otherwise
- [x] FE input loads on Enter; on success applies filters; on partial highlights unmatched tokens
- [x] Structured filters remain authoritative — user can override after parse
- [ ] Endpoint documented in Swagger

## API Contract
**Request** `POST /api/ai/reports/parse-query`
```json
{ "reportKey": "revenue-by-dispatcher", "naturalQuery": "last month, exclude cancelled", "currentFilters": {...} }
```
**Response**
```json
{ "filters": { "dateRange": "2026-04-01..2026-04-30", "status": ["delivered","invoiced"] }, "unmatchedTokens": ["team leads"], "confidence": 0.82 }
```

### FN-1142 (subtask)
- Added optional NL input above the structured filter row in `app-report-filters`. Renders only when `[reportKey]` is provided so legacy multi-tab consumers are unaffected.
- Submit on Enter (or Parse button) calls `ReportsService.parseNlQuery(reportKey, query, currentFilters)` against `${apiUrl}/ai/reports/parse-query`.
- On success: response `filters` are merged into `localFilters` (parse only proposes; structured controls stay authoritative and can override) and an `apply` event is emitted so the parent page reloads data.
- On partial match: `unmatchedTokens` rendered as warning chips with a hover tooltip listing the full set; matched filters are still applied.
- On error: friendly message in `.nl-error`; loading state cleared; no `apply` emitted.
- Theme tokens only (`--accent-color`, `--warning-color`, `--danger-color`, `--text-dark`, `--text-light`, `--border-color`).
- Unit spec (`report-filters.component.spec.ts`) covers: input visibility gated by `reportKey`, empty-query no-op, parse success applies + emits, partial match surfaces tokens + still applies, error handling (server message + fallback), and `clear()` reset.

## Key Decisions
- **Opt-in via `[reportKey]` input** rather than always-on so the legacy multi-tab `reports-page` (no single report key) isn't broken. Page-level wiring of `reportKey` for the v2 `report-view` page is out of scope for this subtask and lives with the backend FN-1149 + later integration.
- **Parse → merge → emit apply** in one step (rather than only populating the form) so users see the report refresh immediately. They can still adjust structured controls and click Apply again.
- **`.css` instead of `.scss`** to match the existing `report-filters.component.css` style — the description's `.scss` reference was aspirational; converting the file is unrelated churn.
