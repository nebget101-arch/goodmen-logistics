# FN-1117: Reports — Natural-language filter parser

## Requirement
Optional NL input above the structured filters parses queries like "revenue by driver last month, exclude team leads" into filter params via Claude Haiku. Mirrors `loads-nlq-handler` pattern.

## API Contract
**Request** `POST /api/ai/reports/parse-query`
```json
{ "reportKey": "revenue-by-dispatcher", "naturalQuery": "last month, exclude cancelled", "currentFilters": {...} }
```
**Response**
```json
{
  "success": true,
  "fallback": false,
  "filters": { "date_from": "2026-04-01", "date_to": "2026-04-30", "status": ["delivered","invoiced"] },
  "unmatchedTokens": ["team leads"],
  "confidence": 0.82,
  "meta": { "reportKey": "revenue-by-dispatcher", "model": "claude-haiku-4-5-20251001", "cacheReadTokens": 1500, "cacheWriteTokens": 0, "processingTimeMs": 312 }
}
```
On any failure (unconfigured API key, upstream error, unparseable model output) the response shape stays `{ success: true, fallback: true, filters: {}, unmatchedTokens: [], confidence: 0, meta: { reason: ... } }` — never a 500 — so the FE keeps the structured filters authoritative.

## Implementation Summary

### FN-1142 (subtask)
- Added optional NL input above the structured filter row in `app-report-filters`. Renders only when `[reportKey]` is provided so legacy multi-tab consumers are unaffected.
- Submit on Enter (or Parse button) calls `ReportsService.parseNlQuery(reportKey, query, currentFilters)` against `${apiUrl}/ai/reports/parse-query`.
- On success: response `filters` are merged into `localFilters` (parse only proposes; structured controls stay authoritative and can override) and an `apply` event is emitted so the parent page reloads data.
- On partial match: `unmatchedTokens` rendered as warning chips with a hover tooltip listing the full set; matched filters are still applied.
- On error: friendly message in `.nl-error`; loading state cleared; no `apply` emitted.
- Theme tokens only (`--accent-color`, `--warning-color`, `--danger-color`, `--text-dark`, `--text-light`, `--border-color`).
- Unit spec (`report-filters.component.spec.ts`) covers: input visibility gated by `reportKey`, empty-query no-op, parse success applies + emits, partial match surfaces tokens + still applies, error handling (server message + fallback), and `clear()` reset.

### FN-1149 (subtask)
Added `POST /api/ai/reports/parse-query` to ai-service following the FN-1134 anomaly-handler pattern (`loadAuthContext` + `hasReportsView`).

- **Per-report filter schemas** — `REPORT_FILTER_SCHEMAS` map covers the 10 v2 reports listed in `goodmen-shared/routes/reports.js` (`total-revenue`, `rate-per-mile`, `revenue-by-dispatcher`, `revenue-by-driver`, `payment-summary`, `gross-profit`, `gross-profit-per-load`, `profit-loss`, `direct-load-profit`, `fully-loaded-profit`). Each schema lists allowed filter keys with type/constraint metadata. Unknown reportKeys return 400 (`AI_UNKNOWN_REPORT`) before any model call.
- **Schema enforcement** — `validateFilters()` drops any key not in the report's schema, plus any value that fails its type validator (ISO date, date range, positive number, string, string array, bool). Source tokens for dropped filters bubble up via the model's `tokenMap` envelope into `unmatchedTokens`. The model also has a `_unmatched` sentinel for tokens it could not bind to any filter.
- **Prompt caching** — System uses two cached blocks (`cache_control: { type: 'ephemeral' }`): the static system prompt loaded from `src/prompts/reports-parse-query.md`, then the per-report schema. The same `reportKey` warms its own cache slot, so the second call reports `meta.cacheReadTokens > 0` (verified in unit test).
- **RBAC** — Requires `reports.view`; super_admin bypasses. Caller without permission gets 403 with `{ code: 'AI_FORBIDDEN', required: 'reports.view' }` and Claude is never invoked.
- **Model** — `claude-haiku-4-5-20251001` (overridable via `ANTHROPIC_REPORTS_NLQ_MODEL`). Temperature 0 for deterministic parses.
- **Fallbacks** — Unparseable model output, schema-mismatched envelope, anthropic upstream errors, and missing `ANTHROPIC_API_KEY` all degrade to `{ success: true, fallback: true, filters: {} }`. The FE keeps structured filters authoritative.
- **Tolerant envelope** — Model may return either `{ filters, tokenMap, confidence }` or a bare filter object; handler accepts both.
- **Tests** — 30 standalone Node tests via `node:assert`; suite wired into `npm test` in `ai-service/package.json`.

## Key Decisions

### FN-1142
- **Opt-in via `[reportKey]` input** rather than always-on so the legacy multi-tab `reports-page` (no single report key) isn't broken. Page-level wiring of `reportKey` for the v2 `report-view` page is out of scope for this subtask and lives with the backend FN-1149 + later integration.
- **Parse → merge → emit apply** in one step (rather than only populating the form) so users see the report refresh immediately. They can still adjust structured controls and click Apply again.
- **`.css` instead of `.scss`** to match the existing `report-filters.component.css` style — the description's `.scss` reference was aspirational; converting the file is unrelated churn.

### FN-1149 (subtask)
- **Permission code**: used `reports.view` (matches existing `reports-anomalies-handler` and `auth-context.js` `ROLE_PERMISSIONS` map) instead of the AC's literal `REPORTS_VIEW`. Net effect identical — caller needs the same role membership — and using the project's actual code lets `loadAuthContext` middleware enforce it without changes.
- **Schema-locked allowlist over dynamic introspection**: the per-report filter schema is a hand-curated map in the handler, not auto-derived from `v2Builders`. Tradeoff: schema drift requires manual update, but the upside is that ad-hoc internal filter params can't accidentally become AI-exposed surface area, and unknown filters are rejected with a clear 400 (`AI_UNKNOWN_REPORT`) before any model spend.
- **`tokenMap` envelope**: instead of asking the model to inline `unmatchedTokens` (which ties unmatched tracking to whatever the model decides to emit), the handler computes `unmatchedTokens` server-side from `tokenMap[droppedKey]`. This makes unmatched-token reporting deterministic relative to the schema enforcement step.
- **Bare-object fallback**: handler accepts both `{ filters, tokenMap, confidence }` and a bare `{ key: value }` shape so a smaller/older Haiku rev that doesn't include the envelope still produces useful filters.
- **No 5xx on AI failures**: every degraded path returns `{ success: true, fallback: true, filters: {} }`. The FE (FN-1142) is expected to leave the structured filters untouched on `fallback: true`, matching the AC bullet "structured filters remain authoritative".

## Deployment Handoff

### FN-1149 (ai-service)
- **Service**: `ai-service` redeploy. Restart picks up the new route, the prompt file, and the test wiring.
- **Migrations**: none.
- **Env vars**:
  - `ANTHROPIC_API_KEY` (already set on ai-service in Render — no change). When missing, the handler degrades to `fallback: true` so a missing key won't 500.
  - `ANTHROPIC_REPORTS_NLQ_MODEL` (optional override, defaults to `claude-haiku-4-5-20251001`).
- **Cache**: none at the app level. Anthropic prompt cache is ephemeral (5-min TTL), warms automatically on second call per `reportKey`.
- **Feature flags**: none.
- **Verification**:
  1. `curl -X POST $GATEWAY/api/ai/reports/parse-query -H "Authorization: Bearer <token>" -H "Content-Type: application/json" -d '{"reportKey":"revenue-by-driver","naturalQuery":"last month over $1000"}'` → expect `success:true`, populated `filters.date_from`/`date_to`/`min_revenue`.
  2. Repeat the same call within 5 minutes → `meta.cacheReadTokens` should be > 0 (prompt cache hit).
  3. Call without Authorization or with a `driver`-role token → expect 403 `AI_FORBIDDEN`.
  4. Call with `reportKey: "no-such-report"` → expect 400 `AI_UNKNOWN_REPORT`.
  5. Swagger UI at `$AI_SERVICE/api-docs` lists `/api/ai/reports/parse-query` under the AI tag.

### Known limitations / follow-ups
- `REPORT_FILTER_SCHEMAS` is hand-curated and must stay aligned with new v2 reports added to `goodmen-shared/routes/reports.js`. A future ticket could derive the schema from a shared declaration to eliminate drift.
- Per-tenant filter customisation (e.g. custom dispatcher exclude lists) is not supported; if a tenant wants to whitelist additional keys, that's a separate enhancement.
- Page-level wiring (passing `[reportKey]` to `app-report-filters` from a v2 report page) is out of scope for both subtasks — it's done when the v2 report-view adopts `app-report-filters`.
