# FN-1472: [Quick Add Part] AI invoice + photo upload skips auto-barcode generation and never extracts category

## Requirement

The AI-driven Quick Add Part flows (Scan Invoice + Photo Identify) create parts that **(a) do not get an auto-generated barcode** and **(b) have no category populated**.

- `FN-1400` was scoped to the **single-create path** (`createPart()` in `parts.service.js`). The AI invoice flow uses bulk-create (FN-1103) and the photo flow uses a separate write path — both bypass `createPart()`, so the `FN-XXXXXXXX` generator never runs.
- The AI prompts in `parts-invoice-vision-handler.js` (FN-1102) and the photo identify handler do not request a `category` field. After FN-1363 relaxed the NOT-NULL constraint, the FE just leaves the column empty — the AI was never wired to actually classify the part.

## Acceptance Criteria

- [ ] AI invoice review modal: every parsed line has a `category` field prefilled by the AI; user can edit before save.
- [ ] AI photo prefill modal: the `category` field is prefilled by the AI; user can edit before save.
- [ ] After save, all AI-created parts have a non-null `barcode` matching `^FN-[A-HJ-NP-Z2-9]{8}$` and a non-null `category`.
- [ ] Manual create flow (FN-1398/FN-1400) is unaffected.
- [ ] QA evidence (screenshots + DB row dumps) committed to `docs/stories/evidence/FN-1472/`.

## Implementation Summary

### FN-1473 (subtask)

Extended both Claude vision flows used by Quick Add Part to extract a `category` per item with a `confidence.category` score.

- **Invoice flow** (`parts-invoice-vision-handler.js`): prompt schema now includes `category` and `confidence.category` per `lineItems[i]`. Added a `## Category classification` section documenting the fixed vocabulary `Brakes | Filters | Electrical | Fluids | Engine | Tires | Body | Other` (or `null` when too ambiguous). `normalizeLineItem` propagates `category` (string|null, trimmed; empty/whitespace → null) and clamps `confidence.category` to `[0, 1]`. Top-level response shape unchanged — only the per-line shape grew, so the existing FN-1103 bulk-create wiring keeps working.
- **Photo identify flow** (`parts-vision-handler.js`, hit by both `/parts/identify-vision` and `/parts/identify-from-photo` via `parts-photo-intake-handler`): prompt vocabulary realigned to the same fixed list (was previously `Filtration / Hydraulics / Suspension / Lighting`). Schema/validators already had `category` + `confidence.category` from FN-1097 — no validator changes needed, only the prompt + the few-shot example.

The actual photo identify handler in this codebase is `parts-vision-handler.js`, not the `parts-photo-identify-handler.js` filename suggested in the ticket — `parts-photo-intake-handler.js` is the multipart-upload wrapper that delegates to it.

### FN-1474 (subtask)

`bulkCreateParts()` in `parts.service.js` was minting parts without ever populating `barcode` — the field was simply missing from the insert payload, so the column landed in the DB as NULL for every AI-invoice row. `createPart()` (FN-1400) already used the shared `resolveBarcodeForCreate()` helper from `services/barcode-generator.js`; the fix re-uses that helper inside the bulk path's pre-transaction prep loop.

Because `isBarcodeTaken()` only sees committed rows, a single batch could in theory mint the same value twice in a row (extreme tail risk against a 32^8 ≈ 1.1e12 keyspace, but free to defend). A `mintedInBatch` Set is wrapped around `isBarcodeTaken` so duplicates within a single bulk request are caught before insert.

`category` already round-trips correctly via `normalizeCategory(c.category)` (FN-1364), so no new code was required for the category half of the AC; FN-1473 (AI prompt) + FN-1475 (FE form) close the loop. The photo prefill flow goes through `POST /api/parts` → `createPart()`, which has been barcode-aware since FN-1400, so no route change was needed there.

## Key Decisions

### FN-1473 (subtask)

- **Aligned the photo handler's category vocabulary with the documented invoice vocabulary** rather than keeping its FN-1097 list (`Filtration / Hydraulics / Suspension / Lighting`). The downstream FE (FN-1475) needs a single canonical set of labels to render a dropdown; two separate vocabularies would have forced a translation map and a UI fork between flows.
- **Did not enforce the vocabulary in the validator** — `normalizeCategory` accepts any string and trims it; if Claude returns `Filters` we keep `Filters`. The prompt steers the model toward the fixed list and FE can either show the value verbatim or coerce unknown values to `Other` at render time. Validating server-side would silently lose perfectly reasonable AI guesses.
- **Empty / whitespace category coerces to `null`, not `""`.** This matches how `parts-vision-handler.js` already treats string fields (`null` is the universal "no value" signal across both flows) and avoids the FE having to handle two empty-state representations.

### FN-1474 (subtask)

- **Reused existing helper instead of re-extracting.** `resolveBarcodeForCreate` already lives in `services/barcode-generator.js` and is already injected with a `checkExists` callback for testability — exactly what the bulk path needed. Extracting it again into `parts.service.js` (as the original ticket suggested) would have duplicated working code.
- **Intra-batch dedup via in-memory Set.** Resolving barcodes outside the transaction matches the existing FK-resolution pattern, but means `isBarcodeTaken` can't see uncommitted batch inserts. Wrapping it with a Set is cheaper than moving generation inside the trx and tying up a connection per row.
- **Did not touch `POST /api/parts` (single create).** AC #4 ("manual create flow unaffected") — the photo flow already goes through this same endpoint, so it's already barcode-aware.
