## Implementation Summary
Implemented all four upstream endpoints in their existing shared route files, mirroring the FN-1303 pattern (Daily AI Briefing upstreams). Each endpoint follows the same skeleton: parse `?limit` (default 20, clamp to [1, 100], 400 on non-numeric), bind tenant scope, query the relevant table(s), and return a JSON array consumed directly by the aggregator's `normalize*` functions. All four degrade to `[]` (200) when an underlying table is missing on a partial dev schema, so the aggregator's `upstreamErrors` stays clean instead of paging on every dashboard load.

### FN-1309 (subtask)
- `GET /api/hos/violations/imminent` (hos.js): finds drivers whose latest `hos_records` row is within 60 minutes of the 11-hour drive cap or 14-hour duty cap. `windowEndsAt` is approximated as `now + minutesRemaining` (the schema has daily totals, not running window starts).
- `GET /api/drivers/fatigue/top` (drivers.js): inline heuristic ā€” `LEAST(100, (on_duty_hours/14)*70 + (driving_hours/11)*30)` per driver, ranked DESC. 70/30 weighting biases toward duty hours since cumulative cognitive load drives fatigue more strongly than wheel time alone.
- `GET /api/vehicles/inspections/overdue` (vehicles.js): vehicles where `inspection_expiry < CURRENT_DATE`, ordered by oldest expiry first. Schema only tracks one inspection date today, so all rows are labeled `inspectionType: 'annual'`.
- `GET /api/loads/late-risk` (loads.js): registered BEFORE `/:id` to avoid the same UUID-cast crash that FN-1303 fixed for `/throughput`. Loads with last delivery stop in the past and not in `EXCLUDED_FROM_OVERDUE`; `etaDelta` = positive minutes between now and the delivery appointment.

Co-located regression test file `routes/smart-alerts-upstreams.test.js` (17 tests) covers the response shape, default limit, clamp-to-100, 400 on non-numeric, and graceful-degrade-on-missing-table for all four endpoints. Plus a route-ordering regression for `/late-risk` to mirror the FN-1303 throughput regression.

## Key Decisions
- **Fatigue heuristic over materialized view**: synchronous, cheap, and operates only on `hos_records` which all tenants already populate. A materialized view would add a cron + a refresh-staleness window for a panel that loads every 30s. The 70/30 duty-vs-drive weighting is documented inline at `drivers.js` route comment.
- **Approximated `windowEndsAt` for HOS imminent**: schema stores daily on/off duty totals, not window-start timestamps. Aggregator only uses this field for display ā€” exact precision isn't required for "violation in 30 min" alerts.
- **Single `inspectionType: 'annual'`**: schema has one `inspection_expiry` column. Splitting annual/DOT/FMCSA would require a new table and is out of scope for this bug fix.
- **`/late-risk` registered before `/:id`**: same route-ordering trap that bit FN-1303. Failing to register early would route `/late-risk` into the UUID `/:id` handler and crash on the cast.
- **Graceful 200+`[]` on missing table**: aggregator's failure mode is HTTP error ā†’ `upstreamErrors`. We treat schema-evolution gaps as "no signals" rather than errors so the panel never regresses on an empty-row tenant.
