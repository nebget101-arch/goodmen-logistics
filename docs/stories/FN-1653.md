## Implementation Summary

### FN-1661 (backend subtask)
Implemented the provider-agnostic telematics ingestion layer:

- **`TelematicsAdapter` interface** (`goodmen-shared/services/telematics/telematics-adapter.js`) — base class with `verifyWebhookSignature`, `parseEvent`, `normalizePing`, `fetchLatestPosition`, plus shared timing-safe HMAC + coercion helpers (`safeEqual`, `hmacSha256Hex`, `toNumber`, `toDate`).
- **`SamsaraAdapter`** — Stripe-style signature `X-Samsara-Signature: v1=<hmac>` over `v1:{timestamp}:{rawBody}` with a 5-minute replay guard; REST polling via `GET /fleet/vehicles/stats?types=gps`.
- **`MotiveAdapter`** — hex HMAC-SHA256 of the raw body in `X-Motive-Signature` (legacy `X-Keeptruckin-Signature` accepted); REST polling via `GET /v1/vehicle_locations`.
- **Registry** (`telematics/index.js`) — `getAdapter(code)` + `SUPPORTED_PROVIDERS`.
- **Webhook ingress** (`integrations-service/routes/telematics-webhook.js`) — `POST /api/webhooks/telematics/:provider?secret=<...>`, two auth layers (shared secret + provider HMAC), parse → normalize → persist, always-200 to avoid retry storms. Mirrors the SendGrid inbound-email webhook.
- **Ingest persistence** (`integrations-service/services/telematics-ingest-service.js`) — resolves `external_device_id` → device → vehicle, dedups on `(vehicle_id, source_event_id)`, inserts into `vehicle_position_pings`, updates `last_seen_at`. Schema-defensive (no-ops pre-migration).
- **Polling fallback** (`integrations-service/services/telematics-polling-service.js`) — finds devices whose `last_seen_at` is stale (`TELEMATICS_POLL_STALE_MINUTES`, default 10), pulls last position per provider, persists. In-process interval scheduler (`TELEMATICS_POLL_INTERVAL_MINUTES`) **or** external Render cron via `POST /api/telematics/poll`.
- **Raw body capture** — `server.js` adds a `bodyParser.json({ verify })` hook so adapters can HMAC the exact received bytes.
- **Tests** — `telematics/telematics-adapter.test.js` (20 cases: registry, helpers, signature valid/tamper/replay/missing/dev-bypass, parse, normalize for both providers). `node --test`, all green.

### FN-1660 (subtask) — Telematics schema + retention (database)

Created three migrations and one seed in `backend/packages/goodmen-database`:

- `20260603100000_create_telematics_providers.js` — `telematics_providers` lookup
  (`id`, `code` UNIQUE, `name`, `is_active`, timestamps).
- `20260603100100_create_telematics_devices.js` — `telematics_devices`
  (`vehicle_id` FK→`vehicles` CASCADE, `provider_id` FK→`telematics_providers` RESTRICT,
  `external_device_id`, `serial`, `paired_at`, `last_seen_at`, timestamps).
  UNIQUE `(provider_id, external_device_id)`; indexes on `vehicle_id` and `last_seen_at`
  (the latter feeds the FN-1661 polling-fallback "stale device" scan).
- `20260603100200_create_vehicle_position_pings.js` — `vehicle_position_pings`,
  **RANGE-partitioned by `ts` into daily partitions** (`id`, `vehicle_id`, `ts`, `lat`,
  `lng`, `speed_mph`, `heading_deg`, `source_event_id`, `payload jsonb`, `created_at`).
  PK `(id, ts)`; index `(vehicle_id, ts DESC)` for latest-position lookups; UNIQUE
  `(vehicle_id, source_event_id, ts)` as a race-safe dedup net matching FN-1661's
  `(vehicle_id, source_event_id)` app-level dedup key (the partition key `ts` is appended
  because a unique index on a partitioned table must include the partition column);
  CHECK constraints on lat/lng/speed/heading.
  Ships a `manage_vehicle_position_pings_partitions(forward_days, backfill_days, retention_days)`
  SQL function and pre-creates a 9-day partition window so ingestion works on day one.
- `seeds/08_telematics_providers_seed.js` — idempotent upsert of `samsara` + `motive`.

**Verification:** ran all three migrations up + down against a throwaway local PG 18,
plus the seed. Confirmed: 9 initial partitions created; device + ping insert; duplicate
`(vehicle_id, source_event_id, ts)` rejected (23505); multiple NULL `source_event_id` rows
allowed (polling fallback); `heading_deg=400` rejected (23514); `manage_…()` drops a
40-day-old partition (retention horizon 30d) and is idempotent; `down()` removes all three
tables + the function cleanly.

## Key Decisions

- Providers = Samsara + Motive (intake). Retention = 30 days raw.
- **New `telematics-ingest-service` vs extending `integrations-service` → EXTEND `integrations-service` (FN-1661 decision).**
  Rationale: the SendGrid inbound-email webhook + shared-secret verification, the knex bootstrap, the Bull/cron scheduler, and the gateway path conventions already live here. A webhook + a polling cron are exactly the workloads this service already runs, so a separate service would duplicate boot/DB/secret plumbing for no isolation benefit at current volume. Revisit (split out `telematics-ingest-service`) if ping ingest volume threatens the FMCSA/email workloads' latency (Story G load test, FN-1659, will inform this).
- **Retention strategy = native PostgreSQL declarative range partitioning (daily), NOT TimescaleDB (FN-1660 decision).**
  Prod-class Render Postgres (dev instance `dpg-…oregon-postgres`, **PG 18.3**) was checked:
  - `timescaledb` 2.23.0 is **available and installable** (`CREATE EXTENSION` succeeded in a
    rolled-back tx) — but it is **not installed** and used nowhere else in the schema.
  - `pg_cron` is **not available**; `shared_preload_libraries` is not readable by our role, so
    TimescaleDB's background-job scheduler (and thus its automatic retention policies) cannot be
    relied upon.
  - **Rationale:** a TimescaleDB hypertable would make this core table un-migratable on local-dev /
    CI (plain `postgresql` per `knexfile.js`, no extension). Native daily range partitioning is
    portable across all environments, gives precise 30-day retention via partition `DROP` (no
    per-row delete bloat), and needs no extra extension. Partition lifecycle (pre-create future
    days + drop days older than 30) is driven by an **app cron** calling
    `manage_vehicle_position_pings_partitions()` daily — required regardless, since `pg_cron` is
    unavailable on Render.
- **Schema contract — FN-1660 ↔ FN-1661 reconciled (verified against `telematics-ingest-service.js`):**
  - `telematics_providers (id, code, name, is_active)` — `code` ∈ {`samsara`, `motive`} (seeded). Resolved case-insensitively by `code`. ✅
  - `telematics_devices (id, vehicle_id, provider_id, external_device_id, serial, paired_at, last_seen_at)` — device lookup by `(provider_id, external_device_id)`; `last_seen_at` touched on every persisted ping. ✅
  - `vehicle_position_pings (id, vehicle_id, ts, lat, lng, speed_mph, heading_deg, source_event_id, payload jsonb, created_at)`. ✅
    - **Dedup:** FN-1661 dedups in app code via `SELECT … WHERE vehicle_id AND source_event_id` then a plain `INSERT` (no `ON CONFLICT`). FN-1660 adds a UNIQUE `(vehicle_id, source_event_id, ts)` net so a concurrent double-webhook race is rejected on the same key the app dedups on; the rejected insert is caught/logged by the ingest's try/catch (no 500). NULL `source_event_id` (polling) is exempt.
    - **`provider_id` / `tenant_id` on pings are intentionally NOT added** (not in the AC; this repo scopes tenancy via `vehicles`/`operating_entities`, not a ping-level `tenant_id`). FN-1661 only writes these "when the column exists", so their absence degrades gracefully — no code change needed.
- `vehicle_position_pings.vehicle_id` has **no hard FK** (write-optimized ingest path); referential integrity is upheld upstream by `telematics_devices.vehicle_id → vehicles.id`.
- Geometry/PostGIS decisions belong to Story B/C — pings store plain `lat`/`lng` doubles here.
- **Signature schemes** are implemented against documented Samsara (Stripe-style `v1=` with timestamp) and Motive (hex HMAC of raw body) conventions. QA (FN-1663) validates against recorded provider fixtures; tolerances live in env (`TELEMATICS_SIGNATURE_MAX_AGE_S`).

## Deployment Handoff

**Deploy order:** FN-1660 (migrations + seed) → FN-1662 (env vars + gateway) → redeploy `integrations-service` (FN-1661 already merged into integration) → schedule the partition-maintenance cron.

**FN-1660 (database):**
- **Migrations to run** (in order, via `fleetneuron-db-migrations`):
  1. `20260603100000_create_telematics_providers.js`
  2. `20260603100100_create_telematics_devices.js`
  3. `20260603100200_create_vehicle_position_pings.js`
- **Seed to run:** `seeds/08_telematics_providers_seed.js` (idempotent; seeds `samsara` + `motive`).
- **Scheduled job required (FN-1662 to wire; consumed by FN-1661 ingest):** a **daily cron** must
  call `SELECT manage_vehicle_position_pings_partitions();` to (a) pre-create upcoming daily
  partitions and (b) drop partitions older than 30 days. `pg_cron` is **not** available on Render —
  this must be an application-side scheduled task (a natural companion to the polling cron). **If
  this cron does not run for ≳7 days, inserts for new days will fail** (no partition); the function
  pre-creates a 7-day forward buffer. Override window/retention via args, e.g.
  `manage_vehicle_position_pings_partitions(14, 2, 30)`.
- No env vars, no service restart, no cache invalidation for the DB subtask itself.

**FN-1661 (backend) — service to redeploy:** `integrations-service` (Render: see `.agent/docs/render_services.md`). No new Render service created — ingestion was folded into `integrations-service`. Until FN-1660 migrations run, the webhook accepts requests but persists nothing (schema-defensive no-op).

**Environment / config (provisioned by FN-1662 on `integrations-service`):**
- `TELEMATICS_WEBHOOK_SECRET` — shared-secret gate for `?secret=` (recommended; if unset, the shared-secret layer is skipped in dev).
- `TELEMATICS_WEBHOOK_SECRET_SAMSARA`, `TELEMATICS_WEBHOOK_SECRET_MOTIVE` — provider HMAC signing secrets (if unset, provider signature check is skipped in dev).
- `TELEMATICS_SAMSARA_API_TOKEN`, `TELEMATICS_MOTIVE_API_TOKEN` — REST tokens for the polling fallback.
- Optional tuning: `TELEMATICS_POLL_INTERVAL_MINUTES` (enables in-process scheduler; default off — use Render cron instead), `TELEMATICS_POLL_STALE_MINUTES` (default 10), `TELEMATICS_POLL_MAX_DEVICES` (default 500), `TELEMATICS_SIGNATURE_MAX_AGE_S` (default 300), `TELEMATICS_POLL_TIMEOUT_MS` (default 10000), `TELEMATICS_SAMSARA_API_BASE` / `TELEMATICS_MOTIVE_API_BASE` (override for sandbox).

**Gateway (FN-1662):** register `/api/webhooks/telematics/:provider` (public — auth via secret + signature) and `/api/telematics/poll` (cron-triggered) to the `integrations-service` upstream.

**Polling cron (FN-1662):** either set `TELEMATICS_POLL_INTERVAL_MINUTES` on the service, or add a Render cron job hitting `POST /api/telematics/poll` every N minutes. The route + in-process scheduler are mutually optional — pick one to avoid double-polling.

**Post-deploy verification:**
1. `POST /api/webhooks/telematics/samsara?secret=...` with a valid-signature fixture → 200 `{ inserted: 1 }` and a new `vehicle_position_pings` row.
2. Bad signature/secret → 401.
3. Stale a device's `last_seen_at`, run `POST /api/telematics/poll` → device polled, ping inserted (requires live/sandbox provider token).
4. Confirm `SELECT manage_vehicle_position_pings_partitions();` runs and a 31-day-old partition is dropped.
