# FN-1039: Load detail drawer (edit-load side panel) does not match the AI dark theme

## Requirement
Re-tokenize `frontend/src/app/components/loads-dashboard/load-detail-drawer/load-detail-drawer.component.scss` so the right-side load detail drawer adopts the AI dark theme defined in `frontend/src/styles.css` (FN-149/150/160/180). The drawer was introduced in FN-808 with hard-coded light tokens; the rest of the app is dark-only, so the drawer looks visually inconsistent.

## Acceptance Criteria
- [ ] Drawer panel, header, tabs row, banner, footer, and unsaved-changes dialog use the AI dark surfaces (matching `.card` / `.topbar` / app-shell).
- [ ] Drawer borders use `var(--border-color)` (or equivalent dark token), not light gray.
- [ ] Active tab indicator and hover accents use the cyan/sky accent palette (`#67e8f9` / `rgba(56,189,248,*)`), not legacy `#14b8a6`.
- [ ] Body text and load number use `#e5e7eb` / `#e5f9ff`; muted text uses `#94a3b8`.
- [ ] Form fields rendered inside the drawer pick up the global `.form-input` AI styling (dark gradient bg, cyan focus ring, 10px radius).
- [ ] Resize handle hover, nav buttons, and icon buttons render legibly on the dark surface (no light-on-light).
- [ ] Visual parity check vs. loads table and onboarding modal — surfaces, shadows, and accents are consistent.
- [ ] No regression in light-content components rendered inside the drawer (status pill, confidence badge, AI sparkle).

## Out of Scope
- Adding a true light-mode toggle. The app is dark-only today.

## Dependencies
- None (FN-808 is already Done)

### FN-1040 (subtask)
- Replaced the six light SCSS variables with AI dark equivalents and added two helpers (`$drawer-text-strong`, `$accent-soft`/`$accent-border`) so emphasized text and hover/active states pull from a consistent palette.
- Surfaces: drawer body uses `rgba(15,23,42,0.96)` (matches `.card`); header/footer/banner use `rgba(2,6,23,0.92)`; tabs row now also uses the panel surface (was hard-coded `#ffffff`).
- Borders use `var(--border-color)`. Active tab + nav/icon button hover use the cyan/sky accent (`#67e8f9`, `rgba(56,189,248,*)`) replacing the legacy `#14b8a6` teal.
- Drawer shadow promoted from a light `-8px 0 24px rgba(15,23,42,0.08)` to the deeper `0 18px / 24px / 44px rgba(2,6,23,0.85) + 1px hairline` pattern used by `.topbar` / `.card`.
- Unsaved-changes overlay darkened (`rgba(2,6,23,0.7)`) and dialog re-themed onto the dark surface with a hairline border and stronger shadow.
- Error banner palette switched from `#fef2f2` / `#b91c1c` to `rgba(127,29,29,0.35)` / `#fca5a5` so the contrast holds on the dark surface.
- Drawer HTML untouched — child step components (`app-step-basics`, `app-step-stops`, `app-wizard-step-driver`, `app-step-attachments`) own their own dark-themed form-field styling from prior FN-149/150/160/180 work, so no class hooks were needed to opt the drawer body into global `.form-input`.
- Build verification: `sass` compiles the file cleanly. `ng build` was not run end-to-end on this branch because the working tree carries pre-existing untracked Finder-duplicate files (`load-wizard.component 2.ts`, `empty-state.component 2.ts`, etc.) that break Angular template type-checking; those duplicates exist on `dev` already and are unrelated to this subtask.
