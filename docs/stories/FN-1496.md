# FN-1496: [Bug] POST /api/work-orders fails with work_orders_status_check violation on dev

## Requirement
Restore WO creation on dev. Repro: any `POST /api/work-orders` payload (no status field) → 500 with `new row for relation "work_orders" violates check constraint "work_orders_status_check"`.

## Repro request body
```json
{"vehicleId":"1b83e006-7f3f-4ce3-ba03-bfbb452dac18","customerId":"e2497242-fe3a-4ac1-b0c5-d31ef57bab2d","locationId":"2cacd454-300a-4b5d-b20f-b2bb1bc359ff","type":"REPAIR","priority":"NORMAL","description":"Goodmen Logistics 1098","odometerMiles":765489,"assignedMechanicUserId":null,"requestedBy":"nebyou.solomon","labor":[],"fees":[],"taxRatePercent":8.5}
```

## Root Cause (proposed)
Dev DB's `work_orders_status_check` no longer allows the legacy lowercase values (`open`, `in_progress`, `completed`, `closed`) that `services/work-orders.service.js:361` writes via `normalizeStatus(payload.status, 'open')`. The constraint drifted from the original migration — likely a manual DBA ALTER on dev to align with the upper-case workflow values from `STATUS_TRANSITIONS` (`work-orders.service.js:26-33`).

- [ ] All status-transition flows still work end-to-end (DRAFT → IN_PROGRESS → COMPLETED → CLOSED)

## Key Decisions
- **Widen rather than canonicalize.** The service writes legacy lowercase today (`normalizeStatus` defaults to `'open'`). Forcing canonical-only would require a synchronized backend refactor and a data backfill that is out of scope for a 500-on-dev hotfix. Widening unblocks both vocabularies; the canonicalization decision is captured under "Open Items".
- **Defensive ENUM/column coercion even though dev doesn't need it.** The original `add_locations_and_work_orders` migration uses `table.enu(...)` which Knex implements as CHECK + text on Postgres, but `update_work_orders_hub` contains an `ALTER TYPE` ENUM block that runs only when a native ENUM exists. Keeping the cleanup logic in this migration means the fix is safe to run on any environment whose schema drifted differently than dev's.
- **Backfill guard with explicit abort.** Rather than letting Postgres throw an opaque `check constraint violation` mid-migration, we count + summarize orphan rows and `throw new Error(...)` with the per-value breakdown so the operator knows exactly which rows to fix before re-running.
- **Down keeps the orphan guard.** A naive down would re-add a stricter constraint and crash on legacy rows already inserted post-up. The guard surfaces that condition before Postgres does.

## Deployment Handoff
- **Service to redeploy:** none. This is a schema-only change. After the migration runs, `work-orders-service` (and any service that hits `POST /api/work-orders`) will succeed on the next request — no restart required because the constraint is enforced inside Postgres.
- **Migration to run on dev:** `20260508120000_fix_work_orders_status_check.js` is already applied to `goodmen_logistics_dev` as part of this work (validated up → down → up). The Render-managed migration job will replay it cleanly on next deploy because the migration is idempotent (drops with `IF EXISTS`, adds with `NOT EXISTS` guard, no-ops if the constraint already matches).
- **Migration to run on staging/prod:** apply via `cd backend/packages/goodmen-database && knex migrate:latest --env staging` (and `--env production`). Pre-flight on staging/prod first by running the same three queries from the Pre-flight section — if the constraint already matches the new union, the migration is a no-op; if there are out-of-set rows, the migration aborts with a row-count summary so you can clean them up before retrying.
- **Env / config / cache / feature flags:** none.
- **Post-deploy verification:** `psql ... -c "SELECT pg_get_constraintdef(oid) FROM pg_constraint WHERE conname='work_orders_status_check'"` — should list all 10 union values. Then re-run the FN-1496 repro request body — expect 201.
