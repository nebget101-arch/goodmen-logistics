## Implementation Summary

### FN-1482 (subtask) â€” Backend DRAFT-resume + today summary
- Added `GET /api/receiving/draft?locationId=` â€” returns the current authenticated user's most recent open DRAFT receiving ticket at the given location, with line items pre-joined to `parts`. Responds 204 No Content when no DRAFT exists for that user/location, so the frontend can decide whether to resume or create a new ticket.
- Added `GET /api/receiving/summary/today?locationId=` â€” returns `{ totalParts, totalLines, totalTickets }` aggregated from `receiving_tickets` where `status = 'POSTED'` and `posted_at >= today 00:00` (server local time) at the given location. Lines are joined to tickets and aggregated via `count(l.id)` and `coalesce(sum(l.qty_received), 0)`.
- Both endpoints registered before the `GET /:id` matcher so the literal segments are not captured as a UUID.
- Role gates on the existing write endpoints were left unchanged. The Jira ticket asked us to also accept `'parts_receiver'` only if that role exists. It does not â€” `backend/packages/goodmen-shared/utils/rbac-compat.js` defines `parts_manager` and `parts_clerk` but not `parts_receiver`. The production gap is flagged in the Deployment Handoff.
- Added `backend/packages/goodmen-shared/routes/receiving.test.js` (Node 18+ `node:test`) with 8 cases covering missing-locationId, no-DRAFT, most-recent-DRAFT, foreign-user-DRAFT, foreign-location-DRAFT, missing-locationId summary, zero-summary, and aggregated counts. All 8 pass under `node --test`.

### FN-1483 (subtask) â€” Frontend ticket-lifecycle rewire + layout redesign
Rewrote `WarehouseReceivingComponent` to drive the receiving-tickets lifecycle end-to-end and rebuilt the page layout as a 3-zone responsive grid with AI dark theme tokens.

- **Lifecycle**: location change resolves to a DRAFT (resume via `GET /api/receiving/draft?locationId=`, otherwise `POST /api/receiving`). Scans add lines through `POST /api/receiving/:id/lines`; remove uses `DELETE /api/receiving/:ticketId/lines/:lineId`; "Post Receiving" calls `POST /api/receiving/:id/post`. After a successful post, the success banner names the posted ticket number, the today-summary refreshes, and a fresh DRAFT is opened automatically.
- **Layout** (3 zones): header (location + ticket# + vendor + reference + today metrics), center (scan input + lines table + post action), right rail (two empty `<ng-content>` slots: `[recv-rail-quick-add]` for FN-1479, `[recv-rail-invoice-upload]` for FN-1480). Hidden via `:empty` until consumers project content.
- **Tokens**: matched the load-detail-drawer (FN-1040) palette â€” `var(--border-color)`, slate `rgba(15,23,42,0.96)` cards, cyan accent `#67e8f9` for ticket# emphasis. Encoded as locally-namespaced CSS custom properties on `.recv-page` so the visual language is self-contained and easy to evolve.
- **Preserved**: hardware scanner focus management, phone QR bridge (SSE + QRCode encode), image decode upload, qty multiplier.
- **Removed**: legacy `api.receiveInventory` client-side fan-out â€” the backend now owns the inventory transactions.
- **Contract alignment with FN-1482**: `getReceivingTodaySummary` reads `totalParts` / `totalTickets` (with snake_case fallbacks for forward-compat). The DRAFT call tolerates `null` / 204 by falling through to `createReceivingTicket`.

## Key Decisions

### Backend (FN-1482)
- **Test runner**: chose Node's built-in `node:test` to match the existing `routes/user-preferences.test.js` and `routes/loads-ai-insights.test.js` pattern. The Jira ticket said "Jest" but the package's `npm test` is `node --test`, so using `node:test` keeps the file in the same lane CI already runs.
- **Auth in tests**: mocked `middleware/auth-middleware.js` via `require.cache` so the test owns `req.user` deterministically, instead of minting a JWT.
- **`/draft` precedence**: registered before `GET /:id` so the literal `draft` segment is not matched as a UUID path param.
- **Today window**: server-local midnight (`new Date(); setHours(0,0,0,0)`). Matches existing aggregation conventions in `routes/inventory.js` and avoids surfacing a TZ negotiation in the page-header strip. If a per-location TZ is needed later, swap `new Date()` for a location-tz lookup in one place.
- **Role gate relaxation**: NOT applied â€” `parts_receiver` does not exist in `utils/rbac-compat.js`. See Deployment Handoff.

### Frontend (FN-1483)
- **Endpoint contract for FN-1482 codified by the frontend caller.** `getReceivingDraft` expects `{ success, data }` for 200 and tolerates 204/null bodies (falls through to create). `getReceivingTodaySummary` reads `totalParts` / `totalTickets` from the documented payload, with `partsReceived` / `parts_received` and `ticketsPosted` / `tickets_posted` accepted as fallbacks.
- **Vendor / reference are model-bound on the DRAFT view but not currently round-tripped to the server.** The backend doesn't expose a PATCH for the receiving ticket (today). They survive in-memory while the user works the same DRAFT, and on resume the backend-stored values are the source of truth. A follow-up to add `PATCH /api/receiving/:id` is recommended if vendor edits need to outlive a session.
- **Right rail uses content-projection slots, not nested components.** This keeps FN-1479 and FN-1480 free to ship as standalone components without a forced API on the receiving component.
- **Component remains module-declared, not standalone.** Aligns with the surrounding components in `app.module.ts`; the spec uses `declarations` + `FormsModule` + `CUSTOM_ELEMENTS_SCHEMA` rather than the standalone-imports pattern.
