### FN-1482 (subtask) — Backend DRAFT-resume + today summary
- Added `GET /api/receiving/draft?locationId=` — returns the current authenticated user's most recent open DRAFT receiving ticket at the given location, with line items pre-joined to `parts`. Responds 204 No Content when no DRAFT exists for that user/location, so the frontend can decide whether to resume or create a new ticket.
- Added `GET /api/receiving/summary/today?locationId=` — returns `{ totalParts, totalLines, totalTickets }` aggregated from `receiving_tickets` where `status = 'POSTED'` and `posted_at >= today 00:00` (server local time) at the given location. Lines are joined to tickets and aggregated via `count(l.id)` and `coalesce(sum(l.qty_received), 0)`.
- Both endpoints registered before the `GET /:id` matcher so the literal segments are not captured as a UUID.
- Role gates on the existing write endpoints were left unchanged. The Jira ticket asked us to also accept `'parts_receiver'` only if that role exists. It does not — `backend/packages/goodmen-shared/utils/rbac-compat.js` defines `parts_manager` and `parts_clerk` but not `parts_receiver`. The production gap is flagged in the Deployment Handoff.
- Added `backend/packages/goodmen-shared/routes/receiving.test.js` (Node 18+ `node:test`) with 8 cases covering missing-locationId, no-DRAFT, most-recent-DRAFT, foreign-user-DRAFT, foreign-location-DRAFT, missing-locationId summary, zero-summary, and aggregated counts. All 8 pass under `node --test`.

## Key Decisions
- **Test runner**: chose Node's built-in `node:test` to match the existing `routes/user-preferences.test.js` and `routes/loads-ai-insights.test.js` pattern. The Jira ticket said "Jest" but the package's `npm test` is `node --test`, so using `node:test` keeps the file in the same lane CI already runs.
- **Auth in tests**: mocked `middleware/auth-middleware.js` via `require.cache` so the test owns `req.user` deterministically, instead of minting a JWT.
- **`/draft` precedence**: registered before `GET /:id` so the literal `draft` segment is not matched as a UUID path param.
- **Today window**: server-local midnight (`new Date(); setHours(0,0,0,0)`). Matches existing aggregation conventions in `routes/inventory.js` and avoids surfacing a TZ negotiation in the page-header strip. If a per-location TZ is needed later, swap `new Date()` for a location-tz lookup in one place.
- **Role gate relaxation**: NOT applied — `parts_receiver` does not exist in `utils/rbac-compat.js`. See Deployment Handoff.

## Deployment Handoff
- **Service to redeploy**: `goodmen-logistics-service` (the receiving routes are mounted in `backend/microservices/logistics-service` via `@goodmen/shared`). Frontend build is unaffected by this subtask.
- **No DB migrations**: both endpoints query existing `receiving_tickets` and `receiving_ticket_lines` tables. No schema changes.
- **No env / config changes**.
- **No feature flags**.
- **Cache invalidation**: none.
- **Deploy order**: backend (this subtask) before FN-1483 frontend rewire — the frontend will call `/draft` and `/summary/today`, so the backend must ship first or the FE will see 404s.
- **Post-deploy verification**:
  1. `curl -H "Authorization: Bearer <token>" "$LOGISTICS_URL/api/receiving/draft?locationId=<uuid>"` → 200 with most-recent DRAFT for that user/location, or 204 if none.
  2. `curl ".../api/receiving/summary/today?locationId=<uuid>"` → 200 `{ totalParts, totalLines, totalTickets }`.
  3. Regression: `GET /api/receiving/:id` still returns ticket-by-id (verifies `/draft` does not shadow `/:id`).
- **Production role gap (carried forward to parent story)**: `requireRole(['admin', 'parts_manager'])` on `POST /api/receiving`, `POST /:id/lines`, `DELETE /:ticketId/lines/:lineId`, and `POST /:id/post` is unchanged by this subtask. If production parts staff are provisioned as `parts_clerk` rather than `parts_manager`, those write routes will 403. The parent story's existing Open Item already tracks this — flagging again so it's resolved before launch.
