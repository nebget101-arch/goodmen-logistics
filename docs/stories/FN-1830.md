# FN-1830: Company-profile header on DQF consent documents

## Requirement
All DQF consent PDFs render `[COMPANY NAME]` / `[ADDRESS, PHONE, EMAIL]` placeholders instead of the selected company profile. Populate the header from the operating-entity profile (the company shown on the company admin page). Consent-document half of Epic FN-1829 issue 1.

`operating_entities` already stores `name`, `address_line1/2`, `city`, `state`, `zip_code`, `phone`, `email` — no schema change. Consent PDFs are built by `driver-onboarding-pdf.js` (`generateConsentPdf`) + `consent-service.js` (`{{companyName}}`/`{{companyAddress}}`).

## Acceptance Criteria
- [x] Every consent PDF shows the selected company's name + address (+ phone/email when set), matching the operating-entity profile.
- [x] No placeholder text when a profile exists; missing optional fields omitted cleanly.
- [x] Multi-MC: driver's operating entity drives the header, not the tenant default.
- [x] No regression to consent body, signature, or MC-logo branding (FN-1739).

## Implementation Summary
Both consent-signing flows now resolve the operating-entity profile and render it in the consent-PDF header instead of the placeholder:

- **Root cause of the empty header:** `consent-service.js` queried a non-existent `operating_entities.address` column, so the lookup threw, was swallowed, and `company` stayed `null` — the header always fell back to `'Company Name'` with an empty address. Fixed by selecting the discrete address columns (`address_line1/2`, `city`, `state`, `zip_code`) plus `phone`/`email` and composing the address.
- **Shared composition helper** (`consent-company-profile.js`, new) — pure, dependency-free functions `composeEntityAddress()` and `buildConsentCompany()` mirror the lookup/compose logic in `employment-application.service.js`. Address format: `address_line1[, address_line2], city, state zip` (state + zip space-joined; missing pieces dropped with no stray separators). `buildConsentCompany()` returns `null` when no entity exists (header falls back), maps `name || legal_name`, and **preserves `logo_storage_key`/`logo_mime_type` for MC-logo branding (FN-1739)**.
- **`consent-service.js`** (authenticated signing) — rewritten lookup uses the helper.
- **`public-consents.js`** (public driver-link signing) — three duplicated inline address blocks (body-placeholder substitution, preview JSON, PDF generation) collapsed onto the shared helpers; the PDF flow now also passes `phone`/`email`. This standardized the address format to `city, state zip` (previously `city, state, zip` with an extra comma).
- **`driver-onboarding-pdf.js`** (`generateConsentPdf`) — header now renders a contact line = `address [ | phone] [ | email]`, filtering out absent pieces so a partial profile shows no empty separators and a real profile never emits the `[ADDRESS, PHONE, EMAIL]` placeholder. Body `{{companyAddress}}` substitution unchanged (postal address only).

All consent templates (clearinghouse, PSP, FCRA disclosure/auth, release of information, drug/alcohol release, MVR disclosure/auth/release-of-liability) share this generator, so the fix applies to every template.

Unit tests: `consent-company-profile.test.js` (8 cases, all passing under Node 20 `node --test`) cover full/partial/empty address composition and the company-object mapping incl. logo preservation and `legal_name` fallback.

## Key Decisions
- **Extracted a shared helper** rather than duplicating the compose logic a 4th time. It also made the logic unit-testable without installing the package's express/db deps (requiring the router module fails with `Cannot find module 'express'`).
- **Phone/email live in the header contact line, not in `{{companyAddress}}`** — consent templates only define `{{companyName}}`/`{{companyAddress}}`, so the body "Address:" line stays clean (postal address only) while the header carries the full `[ADDRESS, PHONE, EMAIL]` equivalent.
- **Re-classified the story to Shape B at runtime.** Intake had recorded an integration branch, but the story has only one non-QA subtask. Per the worktree CLAUDE.md (count subtasks at runtime), the single subtask branch off `dev` is the PR head; the prematurely-created `integration/FN-1830` (identical to `dev`, no dependents) was deleted.
