## Implementation Summary
FN-881 adds `<app-load-wizard-attachments>` under `frontend/src/app/components/load-wizard/steps/attachments/`. It wraps the drag-drop zone, existing-attachments list, pending-upload queue, per-file progress bar, retry button, and delete confirm in a single standalone component driven by the parent wizard's `attachments.queued` FormArray.

Mode branching:
- `create` — files are queued in memory; the parent wizard's existing batch-upload path (post-`createLoad`) picks them up unchanged.
- `edit` / `ai-extract` with a known `loadId` — each dropped file kicks off `LoadsService.uploadAttachmentWithProgress(...)` immediately. On success, the entry moves out of the queued FormArray and into the "existing" list above; on failure, the entry stays in the queue with an inline error + Retry button, and the error field is non-validating so it never blocks `canProceed` / Submit.
- `view` — read-only list of existing attachments (with download links), no dropzone, no controls.

The parent `LoadWizardComponent` gains a new `existingAttachments: LoadAttachment[]` Input (defaults to `[]`) and two callback wiring points (`onAttachmentUploaded`, `onExistingDeleted`) so FN-867 / FN-868 can flow edit/view prefill through without further churn to this file. `buildAttachment` on the parent grew sibling `progress` + `uid` controls so progress updates are reactive and `trackBy` is stable across re-renders.

A new `LoadsService.uploadAttachmentWithProgress(...)` wraps the existing POST endpoint with `HttpRequest({ reportProgress: true })` and maps events to `{ progress, result }` tuples so the template can render a live percent bar without bypassing the service layer.

File filter: `.pdf / .png / .jpg / .jpeg` (by MIME or by extension fallback). Non-matching files are silently skipped.

## Key Decisions
- **Sub-component owns its FormGroup builder.** The child re-implements `buildAttachmentGroup(file, type)` instead of calling back into the parent — avoids tight coupling to a private method and keeps the child unit-testable in isolation. Both producers must keep the shape in sync; a comment in the child flags this.
- **Progress via a new service method** (`uploadAttachmentWithProgress`) rather than mutating the existing `uploadAttachment` callers. Keeps the 4 other call sites (AI extract, legacy wizard, import flow, etc.) on the simpler `{ success, data }` contract.
- **Upload errors aren't form-valid errors.** Per-file `error` lives in a non-validating control, so `attachmentsGroup.valid` stays true even with failed uploads in the queue. This matches AC: "single failure does not block wizard Submit".
- **Inline delete confirm**, not a separate modal — the row flips to a yes/no action strip. Avoids stacking dialogs inside the wizard modal.
