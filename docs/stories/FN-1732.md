# FN-1732: Billing blocked for unapproved trial tenants — "Stripe customer is not initialized" + no trial-status visibility

## Requirement
A trial user reported three symptoms: (1) no trial end date visible anywhere, (2) cannot upgrade plan, (3) adding a payment method fails with "Stripe customer is not initiated for this tenant".

**Root cause (one gap, three symptoms):** trial self-signup (`completeSignupFromToken()` in `backend/packages/goodmen-shared/services/trial-request-service.js`) creates the tenant but does NOT activate the trial. `trialService.activateTrial()` (`backend/packages/goodmen-shared/services/trialService.js:18`) only runs on admin approval of the trial request — and is the only code path that creates a Stripe customer. Pre-approval, `trial_start`/`trial_end`/`trial_status`/`stripe_customer_id` are all NULL:

- `GET /api/billing/trial-status` returns nulls → trial banner + `/billing` render no trial info
- `POST /api/billing/setup-intent` (billing.js:392), `payment-method/confirm` (:457), `portal-session` (:719) hard-fail with the initialization error — which also blocks the upgrade flow (needs customer + card)

**Product decision (2026-06-12):** keep admin approval as the trial-activation gate. Fix around it: lazy Stripe customer creation in billing endpoints + explicit "pending activation" UI state.

**Already shipped — do not rebuild:** FN-1688 (change-plan/cancel/portal/invoices API, PR #785) and FN-1689 (billing page redesign with upgrade/cancel UI, PR #786) merged to `dev` 2026-06-04.

## Acceptance Criteria

## Implementation Summary

### FN-1734 (subtask) — frontend pending-activation state
Branch: `frontend/FN-1734/trial-pending-state` (off `integration/FN-1732`)

Added an explicit "pending activation" state so a self-signup trial tenant
whose trial has not been admin-approved (`trial-status` returns 200 with
`trial_status: null` and no subscription) no longer sees a blank banner or empty
trial fields.

- **`trial-state.service.ts`** — `TrialState` gains a `pending` flag, derived in
  the success path as `!trial_status && !subscription.status`. The discriminator
  is robust: a genuine non-trial user's request errors (403/404) and yields a
  `null` state (banner stays hidden), so `pending` only ever fires for a
  loaded-but-unapproved tenant.
- **`trial-banner.component.ts/html/scss`** — new `'pending'` variant in
  `bannerVariant` (ordered after the past-due override, before the trial-status
  branches): "Your trial is pending activation — you'll see your trial end date
  here once it's approved." Styled with `.banner-pending` using AI dark-theme
  tokens only (`--card-background`, `--ai-accent-info`, `--ai-text-body`) — no
  new hex.
- **`billing.component.ts/html`** — `isPending` getter; status banner shows a
  "Trial pending activation" message instead of the misleading "Subscription
  active"; Status KPI reads "Pending"; `statusSeverity` returns `info` for
  pending.
- **`payment-method-form.component.ts`** — billing-setup (`setup-intent`)
  failures now show a friendly fallback with a support hint
  (`SETUP_FALLBACK_ERROR`) instead of surfacing the raw API error (e.g. "Stripe
  customer is not initialized for this tenant"). After FN-1733 this path
  shouldn't trigger, but the guard remains for any residual 4xx/5xx.
- **Specs** — new `trial-banner.component.spec.ts` (variant decision table incl.
  pending + past-due override), new `trial-state.service.spec.ts` (pending
  derivation incl. error→null case), extended `billing.component.spec.ts`
  (pending severity; active subscriber not pending).

FN-317 compliance: no getter-based `[options]` bindings introduced. Build
verified under nvm Node 20 (`ng build` — billing/trial files compile clean).
