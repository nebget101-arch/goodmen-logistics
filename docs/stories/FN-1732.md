# FN-1732: Billing blocked for unapproved trial tenants ‚Äî "Stripe customer is not initialized" + no trial-status visibility

## Requirement
A trial user reported three symptoms: (1) no trial end date visible anywhere, (2) cannot upgrade plan, (3) adding a payment method fails with "Stripe customer is not initiated for this tenant".

**Root cause (one gap, three symptoms):** trial self-signup (`completeSignupFromToken()` in `backend/packages/goodmen-shared/services/trial-request-service.js`) creates the tenant but does NOT activate the trial. `trialService.activateTrial()` (`backend/packages/goodmen-shared/services/trialService.js:18`) only runs on admin approval of the trial request ‚Äî and is the only code path that creates a Stripe customer. Pre-approval, `trial_start`/`trial_end`/`trial_status`/`stripe_customer_id` are all NULL:

- `GET /api/billing/trial-status` returns nulls ‚Üí trial banner + `/billing` render no trial info
- `POST /api/billing/setup-intent` (billing.js:392), `payment-method/confirm` (:457), `portal-session` (:719) hard-fail with the initialization error ‚Äî which also blocks the upgrade flow (needs customer + card)

**Product decision (2026-06-12):** keep admin approval as the trial-activation gate. Fix around it: lazy Stripe customer creation in billing endpoints + explicit "pending activation" UI state.

**Already shipped ‚Äî do not rebuild:** FN-1688 (change-plan/cancel/portal/invoices API, PR #785) and FN-1689 (billing page redesign with upgrade/cancel UI, PR #786) merged to `dev` 2026-06-04.

## Implementation Summary

### FN-1733 (subtask) ‚Äî lazy Stripe customer creation
Added `trialService.ensureStripeCustomer(tenantId)`: returns the tenant's existing `stripe_customer_id`, or creates the Stripe customer (reusing `stripeService.createCustomer` with the same email/legal_name identity as `activateTrial`), persists it to `tenants.stripe_customer_id`, and returns it.

- **Idempotent / race-safe:** persist is a conditional `UPDATE ‚Ä¶ WHERE id = ? AND stripe_customer_id IS NULL`. If the update claims 0 rows (a concurrent caller won), it re-reads and returns the persisted id instead of the just-created orphan. Backed by the existing partial unique index `uq_tenants_stripe_customer_id_not_null`.
- **Stripe-disabled:** added `stripeService.isStripeConfigured()`; `ensureStripeCustomer` throws a clear `503 / STRIPE_NOT_CONFIGURED` ("Billing is not configured‚Ä¶") instead of the misleading "not initialized" 400.
- **Refactor:** `activateTrial()` now delegates its inline customer-creation block to `ensureStripeCustomer` (single source of truth; also keeps the tenant-existence check).
- **Wiring:** `POST /api/billing/setup-intent`, `POST /api/billing/payment-method/confirm`, and `POST /api/billing/portal-session` call `trialService.ensureStripeCustomer(tenant.id)` and use the returned id, replacing the three hard `400 "Stripe customer is not initialized"` branches. Read-only endpoints (`GET /invoices`, `/trial-status`, `/subscription`, `/payment-method`) are unchanged ‚Äî no customer creation on reads.
- **Tests:** `trialService.test.js` (node:test, in-memory knex stub) ‚Äî existing-id no-op, create+persist, email‚Üílegal_name‚Üíname fallback, idempotency, race-safe winner-wins, disabled-stripe 503, 404 missing tenant, 400 missing tenantId. 8/8 pass under Node 20.

### FN-1734 (subtask) ‚Äî frontend pending-activation state
Branch: `frontend/FN-1734/trial-pending-state` (off `integration/FN-1732`)

Added an explicit "pending activation" state so a self-signup trial tenant whose trial has not been admin-approved (`trial-status` returns 200 with `trial_status: null` and no subscription) no longer sees a blank banner or empty trial fields.

- **`trial-state.service.ts`** ‚Äî `TrialState` gains a `pending` flag, derived in the success path as `!trial_status && !subscription.status`. The discriminator is robust: a genuine non-trial user's request errors (403/404) and yields a `null` state (banner stays hidden), so `pending` only ever fires for a loaded-but-unapproved tenant.
- **`trial-banner.component.ts/html/scss`** ‚Äî new `'pending'` variant in `bannerVariant` (ordered after the past-due override, before the trial-status branches): "Your trial is pending activation ‚Äî you'll see your trial end date here once it's approved." Styled with `.banner-pending` using AI dark-theme tokens only (`--card-background`, `--ai-accent-info`, `--ai-text-body`) ‚Äî no new hex.
- **`billing.component.ts/html`** ‚Äî `isPending` getter; status banner shows a "Trial pending activation" message instead of the misleading "Subscription active"; Status KPI reads "Pending"; `statusSeverity` returns `info` for pending.
- **`payment-method-form.component.ts`** ‚Äî billing-setup (`setup-intent`) failures now show a friendly fallback with a support hint (`SETUP_FALLBACK_ERROR`) instead of surfacing the raw API error (e.g. "Stripe customer is not initialized for this tenant"). After FN-1733 this path shouldn't trigger, but the guard remains for any residual 4xx/5xx.
- **Specs** ‚Äî new `trial-banner.component.spec.ts` (variant decision table incl. pending + past-due override), new `trial-state.service.spec.ts` (pending derivation incl. error‚Üínull case), extended `billing.component.spec.ts` (pending severity; active subscriber not pending).

FN-317 compliance: no getter-based `[options]` bindings introduced. Build verified under nvm Node 20 (`ng build` ‚Äî billing/trial files compile clean).
