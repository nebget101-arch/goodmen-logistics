# FN-1726: Manual load create — Submit on final wizard step silently no-ops (create-load API never called)

## Requirement
When a user creates a load manually through the 4-step V2 Load Wizard (`mode="create"`) and clicks **Submit** on the final (Attachments) step, the load can fail to be created with no feedback — `POST /api/loads` is never called and no error is shown. The wizard must never silently swallow a Submit click.

## Root Cause
`frontend/src/app/components/load-wizard/load-wizard.component.ts`:
- **`canProceed` (~L265):** on the `attachments` step returns only `attachmentsGroup.valid && !submitting` — so the shell's Submit button (`wizard-shell.component.ts onSubmit`, gated on `canProceed`) is enabled based on the attachments group alone.
- **`onSubmit()` (~L571):** guards on the **whole** `this.form.valid`. When an earlier, non-visible step has an invalid required control, it calls `markAllAsTouched()` and `return`s silently — no message, no navigation, no API call.
- Mismatch → Submit button is enabled but the click does nothing. Backend (`POST /api/loads`, logistics-service) is healthy and never reached.

Secondary: `uploadQueuedAttachments` (~L653) swallows attachment-upload errors via `catchError(() => of(null))` and the dashboard success toast masks the transient failure message — a separate, lower-severity silent path.

## Acceptance Criteria
- [x] Valid form → `POST /api/loads` fires and the load is created (happy path preserved).
- [x] Invalid form → visible, actionable error + navigation to the first invalid step (field touched/highlighted); no silent no-op.
- [x] Submit button enabled/disabled state is consistent with whether submission proceeds.
- [x] Unit specs cover both the valid (API called) and invalid (API not called, error shown, step jump) paths.

## Implementation Summary
FN-1727 makes the load wizard's Submit click impossible to silently swallow:

1. **Single source of truth for step validity.** Extracted the per-step validity rules (previously inlined in `canProceed`) into a private `isStepValid(stepId)` helper and a `stepOrder` array. `canProceed` now delegates to `isStepValid(currentStepId)` (still `&& !submitting` on the final step), and a new `firstInvalidStepId()` walks `stepOrder` to find the first invalid group — so forward navigation and submit share one definition of "which step is invalid."
2. **`onSubmit()` jump-to-invalid.** Instead of `if (!this.form.valid) { markAllAsTouched(); return; }`, `onSubmit()` computes `firstInvalidStepId()`; when non-null it sets `currentStepId` to that step, calls `markAllAsTouched()` on *that step's* group, sets `errorMessage = "Please complete the highlighted fields before creating the load."`, and `markForCheck()`s. Only when every step is valid does it fall through to `submitCreate()` / `submitUpdate()` (happy path unchanged).
3. **Button stays honest (option b).** The final-step Submit button remains enabled whenever the attachments group is valid (no "enabled button that does nothing"), because the click now always produces feedback — either a created load or a visible error + jump. No `canProceed` change was needed beyond the `isStepValid` refactor.
4. **Always-visible error surface.** Moved the `submit-error` banner out of the attachments step body and into a shared banner rendered at the top of the wizard-shell content (above the `ngSwitch`), so the message is visible on whichever step the user is jumped to. Added a `.submit-error--global` SCSS modifier (top→bottom margin) for the banner placement. The stale banner is cleared on `onNext`/`onBack`/`onStepChange` once the user moves on.

The secondary `uploadQueuedAttachments` toast-masking issue was **not** addressed — it requires touching the parent dashboard (`loads-dashboard.component.ts`) toast logic, which is outside this focused, low-risk pass; `uploadQueuedAttachments` already sets `errorMessage` on partial failure.

## Files Changed
- `frontend/src/app/components/load-wizard/load-wizard.component.ts` — `isStepValid`/`firstInvalidStepId`/`groupForStep` helpers + `stepOrder`; `canProceed` refactor; `onSubmit` jump-to-invalid; clear `errorMessage` on navigation.
- `frontend/src/app/components/load-wizard/load-wizard.component.html` — global `submit-error` banner above the step bodies; removed the attachments-only error block (kept the `submitting` hint).
- `frontend/src/app/components/load-wizard/load-wizard.component.scss` — `.submit-error--global` modifier.
- `frontend/src/app/components/load-wizard/load-wizard.component.spec.ts` — new `FN-1727 submit validation guard` describe: valid→API called; invalid earlier-step→no API + error + jump; stops-invalid jump; button stays enabled; DOM banner renders; banner clears on navigation.

`wizard-shell.component.ts` was reviewed but needed no change — gating it on `canProceed` (kept enabled) is sufficient.

## Key Decisions
- **Chose option (b)** from the ticket (keep button enabled + jump-to-invalid in `onSubmit`) over option (a) (disable button via full-form validity). Rationale: option (a) reproduces a different "dead button" confusion (button greyed with no explanation of which far-away step is wrong), whereas (b) guarantees every click yields actionable feedback and points the user at the exact step. `isStepValid` still gives a single source of truth.
- **Global banner instead of duplicating per-step** to avoid multiple elements sharing `data-testid="submit-error"` and to keep one render path.
