# FN-1841: Agreements nav link never renders — tab gated on un-seeded permission code `agreements.view`

## Requirement
The `/agreements` page loads when typed directly, but the **"Agreements"** sidebar link never appears for any user. The nav tab `agreements` is gated on permission code `agreements.view`, which no backend migration ever grants — the seed uses `agreement.templates.view` instead. Align the gate so authorized users see the link.

## Type
Bug (under epic FN-1786, lane `epic:agreements-esign`)

## Root Cause (verified on `origin/dev`)
- Frontend gate: `frontend/src/app/models/access-control.model.ts:253` (`AGREEMENTS_VIEW: 'agreements.view'`) + `:295` (`agreements: [PERMISSIONS.AGREEMENTS_VIEW]`); `frontend/src/app/app.component.ts:224` hides the link when `!access.canSee(link.tab)`.
- Backend seed: `backend/packages/goodmen-database/migrations/20260613120100_add_agreement_permissions.js` grants `agreement.templates.view` (+ create/edit/delete) but never `agreements.view`.
- Route `/agreements` is only `AuthGuard`-protected (no tab guard), so direct URL works while the nav link stays hidden.

## Recommended Fix
Frontend-only: point `AGREEMENTS_VIEW` (or a new `AGREEMENT_TEMPLATES_VIEW`) at `agreement.templates.view`, the code actually seeded. Semantically correct — `/agreements` is the templates surface (FN-1837). No migration or backfill. Drop the stale FN-1792 comment.

**Alternative (heavier):** seed `agreements.view` in the migration + assign to roles + backfill existing tenants. Only if product wants the codes to diverge.

## Acceptance Criteria
- [ ] Users with agreement-templates view permission (admin, accounting, carrier_accountant, etc.) see **Agreements** under **Compliance** in the sidebar.
- [ ] Users without that permission do not see the link.
- [ ] No new un-granted permission code introduced (frontend fix) OR new code seeded + assigned + backfilled (backend fix).
- [ ] `/agreements` still loads by direct URL for authorized users.

## Implementation Summary
Applied the recommended **frontend-only** fix. The `agreements` nav tab was gated on
`PERMISSIONS.AGREEMENTS_VIEW`, whose value was `agreements.view` — a code no backend
migration ever seeds or assigns to a role. Repointed `AGREEMENTS_VIEW` at
`agreement.templates.view`, the code that migration
`20260613120100_add_agreement_permissions.js` actually seeds and grants
(super_admin, admin, company_admin, finance_manager, accounting, carrier_accountant).

`TAB_PERMISSIONS.agreements` already references `PERMISSIONS.AGREEMENTS_VIEW`, so the
single constant change propagates to the only gate. No other code references the old
`agreements.view` literal. The `/agreements` route remains `AuthGuard`-only (no tab
guard), so direct-URL access is unchanged.

Added two regression specs in `access-control.service.spec.ts` asserting the agreements
tab is visible to a user granted `agreement.templates.view` and hidden from one without
it, plus an assertion pinning the constant's value — guarding against the gate drifting
back to an un-seeded code.

Note: `ng test` cannot run in this environment (no local karma; build needs Node 20).
Changes are a string-literal value change plus additive specs with all consumers verified
by grep.

## Key Decisions
- **Frontend fix over backend fix.** Chose the recommended minimal path: no new permission code, no migration, no per-tenant backfill. `/agreements` is the agreement-templates surface (FN-1837), so gating it on `agreement.templates.view` is semantically correct, not a workaround.
- **Reused the existing `AGREEMENTS_VIEW` constant name** rather than introducing `AGREEMENT_TEMPLATES_VIEW`, to keep the change to a single line and avoid touching the `TAB_PERMISSIONS` mapping. The comment documents the intent.
