### FN-1668 (subtask) — geofence_events table
Migration `20260603100300_create_geofence_events.js` creates the `geofence_events`
table that the FN-1669 Bull worker writes to (one row per detected crossing) and
that load-status automation reads from.

Schema:
- `id` UUID PK
- `geofence_id` UUID NOT NULL → FK `geofences.id` **ON DELETE CASCADE**
- `vehicle_id` UUID NOT NULL — **no hard FK** (copied from the triggering ping; mirrors `vehicle_position_pings`)
- `event_kind` TEXT NOT NULL CHECK `('enter','exit','dwell')`
- `ts` TIMESTAMPTZ NOT NULL — crossing time (= triggering ping's `ts`)
- `ping_id` UUID NOT NULL — **soft reference** to `vehicle_position_pings.id`, no hard FK
- `load_id` UUID NULL → FK `loads.id` **ON DELETE SET NULL** (FK wired only when `loads` exists)
- `created_at` TIMESTAMPTZ DEFAULT now()

Constraints / indexes:
- `UNIQUE (ping_id, geofence_id, event_kind)` — idempotency net so the Bull worker can safely retry/reprocess a ping (one enter/exit/dwell per ping+geofence)
- `INDEX (vehicle_id, ts DESC)` — worker lookup: latest events per vehicle
- `INDEX (geofence_id, ts DESC)` — worker lookup: events per geofence
- `INDEX (load_id) WHERE load_id IS NOT NULL` — load-status automation reads events by load

Key decisions:
- **`ping_id` has no hard FK by design.** `vehicle_position_pings` (FN-1660) is RANGE-partitioned by `ts` with a composite PK `(id, ts)`. PostgreSQL forbids a FK that references only part of a partitioned table's key, and adding `ts` to the FK would couple this table to the partition key. `ping_id` is stored as a plain UUID soft reference, consistent with how the pings table itself treats `vehicle_id` (no FK on the write-optimized ingest path).
- **`vehicle_id` has no hard FK**, matching the pings ingest pattern; the value is copied straight off the triggering ping.
- **`geofence_id` uses CASCADE**: an event log row is meaningless once its geofence is gone. Geofences are soft-disabled via `is_active` (FN-1664), so hard deletes are rare.
- **`load_id` uses SET NULL**: events are a historical record and should survive a load deletion.

Verified: ran the full migration chain (211 migrations) on a scratch Postgres — table, FKs (`geofences` CASCADE, `loads` SET NULL), check constraint, unique constraint, and all four indexes created as specified. Rollback (`down` = `dropTableIfExists`) confirmed.

### FN-1669 (subtask) — geofence-event worker + load-status automation
Bull-backed worker that consumes new `vehicle_position_pings`, computes geofence
crossings app-side (no PostGIS — reuses `geofence-service.js`'s
`geofenceContainsPoint`), writes one `geofence_events` row per crossing, and
drives load-status automation off those events.

**Modules**
- `services/geofence-event-worker.js`
  - `createGeofenceEventWorker({ redisUrl, knex, dwellMinutes, deliveryDwellMinutes })` — Bull queue `geofence-events` (mirrors the FN-1424 fmcsa-import-queue shape: lazy ioredis client, TLS for `rediss://`, ready/error wiring, 3 attempts with exponential backoff — safe because writes are idempotent). Exposes `enqueuePing(ping)`, `enqueuePings(pings)` (the "batched per minute" path — one job, many pings), `processPing`, `isReady`, `shutdown`.
  - `processPing(ping)`: resolves the vehicle's tenant (`vehicles.tenant_id` — pings carry no tenant), loads active tenant geofences, edge-detects enter/exit/dwell per geofence against the **event log** (latest event per geofence), persists new events idempotently, then runs automation per new event.
  - Pure helpers (unit-tested): `pingPoint`, `decideEventKind`.
- `services/load-status-automation.js`
  - `nextLoadStatus({ currentStatus, eventKind, insideMinutes, deliveryDwellMinutes })` — **pure** state machine (see Key Decisions for the confirmed table).
  - `findActiveLoadForVehicle` (loads on `truck_id`, status ∈ in-flight set, most-recently-updated wins), `minutesInsideSinceEnter` (delivery-dwell from the matching enter event), `applyForEvent` (updates `loads.status` with a from-status guard + stamps `geofence_events.load_id`).
- `migrations/20260603140000_add_arrived_statuses_to_loads.js` — widens `loads_status_check` to add `ARRIVED_AT_PICKUP` / `ARRIVED_AT_DELIVERY` (additive; existing values intact). Required because the PM-confirmed machine writes those values and the constraint is strict.

**Idempotency:** crossings are de-duped at two layers — the DB `UNIQUE (ping_id, geofence_id, event_kind)` (insert via `onConflict(...).ignore()`), and the from-status–guarded `loads.status` UPDATE. Reprocessing a ping is a no-op.

**Tests:** `services/geofence-event-worker.test.js` — 27 tests (Node test runner + in-memory knex stub, no Postgres/Redis). Covers the pure state machine (all transitions incl. the >5-min delivery rule and terminal-status inertness), the pure crossing classifier (enter/exit/dwell/no-op incl. one-dwell-per-stay), and end-to-end `processPing` (each transition, idempotent reprocess, dwell event, no-tenant / no-fix / not-in-flight short-circuits). All green under Node 20.

## Key Decisions
- **Load-status state machine (confirmed with PM):**
  | Current status | Crossing | → Next status |
  |----------------|----------|---------------|
  | DISPATCHED / EN_ROUTE | enter | ARRIVED_AT_PICKUP |
  | ARRIVED_AT_PICKUP | exit | IN_TRANSIT |
  | IN_TRANSIT | enter | ARRIVED_AT_DELIVERY |
  | ARRIVED_AT_DELIVERY | exit, inside > 5 min | DELIVERED |
  | ARRIVED_AT_DELIVERY | exit, inside ≤ 5 min | (no change — drive-through) |
  | any | dwell | (no change — event only, Phase 1) |
- **New statuses `ARRIVED_AT_PICKUP` / `ARRIVED_AT_DELIVERY`** added to `loads_status_check` (PM chose to capture "arrived" precisely rather than collapse onto existing values). Frontend status pills/filters must be extended to render them — see Open Items.
- **Pickup vs. delivery is inferred from the load's current status**, not a geofence↔stop link (none exists in the schema). Per PM, **any** active tenant geofence the vehicle crosses can drive status — the worker does **not** gate on `geofence_triggers`.
- **Dwell** is recorded as an event only (no detention billing in Phase 1). Generic dwell threshold is `GEOFENCE_DWELL_MINUTES` (default 30); the delivery-confirmation dwell is `GEOFENCE_DELIVERY_DWELL_MINUTES` (default 5).
- `geofence_events.ping_id` is a soft reference (no FK) because the pings table is partitioned with a composite PK — see FN-1668 notes.
