# FN-466: Implement Toll Transponder -> Vehicle -> Driver Resolution Chain

## Requirement
Build the core matching logic that resolves toll transactions to vehicles and drivers. This includes device-to-vehicle assignment management endpoints and a `resolveTollTransaction()` service that walks the transponder -> vehicle -> driver chain with date-range awareness and driver-override support.

## Acceptance Criteria
- [ ] Device assignment endpoints (list, assign-vehicle, remove-vehicle, assign-driver)
- [ ] `resolveTollTransaction()` matching service with multi-step resolution
- [ ] Driver override on device skips vehicle->driver chain
- [ ] Assignment history used with date-range matching
- [ ] Loads table takes priority over drivers.truck_id for driver resolution
- [ ] Exception types: unmatched_device, unmatched_driver (with truck info)
- [ ] All endpoints tenant-scoped and operating-entity filtered

### FN-468 (subtask)
**Backend changes:**

1. **New file: `backend/packages/goodmen-shared/services/toll-matching-service.js`**
   - `resolveTollTransaction(knex, tenantId, transaction)` — multi-step resolution:
     - Step 1: Find device by device_number_masked or plate_number
     - Step 2: Resolve vehicle via assignment history (date-range), then toll_devices.truck_id fallback, then plate match
     - Step 3: Resolve driver via loads table (priority) then drivers table
     - Step 4: Return { truck_id, driver_id, toll_device_id, matched_status, exceptions[] }
   - Driver override (device.driver_id set) skips Steps 2-3

2. **Updated file: `backend/packages/goodmen-shared/routes/tolls.js`** — 4 new endpoints:
   - `GET /api/tolls/devices/:deviceId/assignments` — list assignment history
   - `POST /api/tolls/devices/:deviceId/assign-vehicle` — assign with auto-remove of previous
   - `POST /api/tolls/devices/:deviceId/remove-vehicle` — clear assignment
   - `POST /api/tolls/devices/:deviceId/assign-driver` — direct driver override

**Render service:** logistics-service (tolls routes are mounted in logistics)
