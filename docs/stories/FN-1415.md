# FN-1415: Manual FMCSA import trigger (FleetNeuron-internal only) + biweekly cron + admin UI

## Requirement

Wire the importers from FN-1413 and FN-1414 into a runtime control plane:

1. **Biweekly cron** (every 14 days, Sunday 4 AM UTC) — auto-runs all five importers
2. **Manual trigger API** — `POST /api/fmcsa/imports/run` with `{ files: [...] }`. Restricted to FleetNeuron internal tenant via `tenants.is_internal = true`. 403 for everyone else.
3. **Import history API** — `GET /api/fmcsa/imports` returns recent runs from the ledger
4. **Admin UI** — minimal Angular page for FleetNeuron internal users only: trigger panel + history table with live status polling

## Acceptance Criteria

- [x] Cron registered for biweekly cadence; first scheduled run logs `triggered_by = 'cron'`
- [x] `POST /api/fmcsa/imports/run` enqueues selected importers, returns 202 + run IDs
- [x] `dryRun: true` mode parses without persisting
- [x] `GET /api/fmcsa/imports` returns ledger ordered by `started_at DESC`
- [x] Both endpoints return 403 for non-FleetNeuron tenants and for callers without `fmcsa.imports.manage`
- [x] Cron disabled in `NODE_ENV=development` by default
- [x] Admin UI page: trigger panel with checkboxes per file, dry-run toggle, history table auto-refreshing every 5s while runs are in progress
- [x] Non-internal users redirected away from the UI route
- [x] Cron expression configurable via `FMCSA_IMPORT_CRON` env var

## Implementation Summary
_To be filled by implementing agent_

### FN-1424 (subtask)

Stood up the FMCSA import control plane on top of the FN-1412 schema (`fmcsa.import_runs`, `tenants.is_internal`, `getFmcsaKnex()`):

- **`requireInternalTenant` middleware** — reads `req.context.tenantId` (populated by the existing `tenantContextMiddleware`), looks up `tenants.is_internal`, and returns 403 for `false` / `NULL` / unresolved tenants. Six unit tests cover the true / false / null / missing-tenant / missing-context / DB-error branches.
- **`fmcsa.imports.manage` permission** — added via migration `20260506110300_add_fmcsa_imports_permission.js`, assigned to `super_admin` and `platform_admin` roles only (deliberately narrow — only FleetNeuron-internal staff trigger imports).
- **`fmcsa-import-queue.js`** — new Bull queue with: an importer registry (FN-1413 / FN-1414 plug in here at startup), a single `run-import` processor that updates `fmcsa.import_runs` from `queued → running → success/error`, and a `cron-trigger` processor that fans out to all five files. Cron defaults to a true 14-day cadence (`every: 14*24*60*60*1000`); set `FMCSA_IMPORT_CRON` to override with a cron expression. Cron is gated off in `NODE_ENV=development` unless `FMCSA_IMPORT_CRON_ENABLED=true`.
- **`routes/fmcsa-imports.js`** — `POST /api/fmcsa/imports/run` validates the `files` array against `SUPPORTED_FILES`, enqueues one ledger row + Bull job per file, and returns `202 { success: true, data: { runIds } }`. `GET /api/fmcsa/imports?limit=N` returns `{ success: true, data: [...] }` ordered by `started_at DESC` (limit clamped to 1..200, default 50).
- **Server wiring (`integrations-service/server.js`)** — mounts the import routes behind the full chain `authMiddleware → tenantContextMiddleware → requireInternalTenant → loadUserRbac → requirePermission('fmcsa.imports.manage')`. Returns 503 when Redis is unavailable. The queue's `initScheduler()` runs after `app.listen()` like the existing scrape queue.

### FN-1425 (subtask)

Built the Angular admin page that consumes the FN-1424 control-plane endpoints. Highlights:

- New `FmcsaImportsAdminComponent` under `components/admin/fmcsa-imports/` (matching existing `components/admin/inbound-email-settings` layout — the codebase has no `admin/` feature module, so the route is registered inline in `app-routing.module.ts` rather than introducing a new module just for one page).
- New `FmcsaImportsService` (HTTP wrapper for `GET /api/fmcsa/imports` and `POST /api/fmcsa/imports/run`).
- New `InternalTenantGuard` redirects non-internal tenants to `/dashboard` (not 403) so the admin route is silent for everyone outside FleetNeuron.
- Threaded an `isInternal` flag through `UserAccess` + `AccessControlService.isInternalTenant()` so the guard prefers the backend-supplied flag once FN-1424 ships and falls back to the legacy `tenantName === 'fleetneuron default tenant'` check until then.
- Added a "FMCSA Imports" link to both the desktop and mobile user-menu nav, gated by `canSeeFmcsaImportsAdmin()`.
- Live polling: the history table re-fetches every 5s while any run is `queued` / `running` and stops once they all reach `success` / `error`. Confirmation modal blocks blind clicks on the Run button. Dry-run rows render with a `DRY RUN` tag and a tinted background so they cannot be confused with live runs.
- Unit specs cover service HTTP shape and the component's selection / submit / polling / formatting helpers.
- Styling follows the existing dark `ai-admin-shell` / `ai-panel` tokens used by the inbound-email admin page.

## Key Decisions
_To be filled by implementing agent_

### FN-1424 (subtask)
- **Response shape matches the frontend, not the original ticket description.** The FN-1424 description proposed `{ runs: [...] }`; the FN-1425 service merged into `integration/FN-1415` already calls `{ success, data: { runIds } }` / `{ success, data: [...] }` — the standard envelope used elsewhere in the codebase. I kept the standard envelope rather than diverging both subtasks.
- **The queue is a new file, not a "modify" as the ticket said.** The ticket assumed `services/fmcsa-import-queue.js` already existed; on this integration branch it does not (FN-1413 / FN-1414 — the importer stories — have not landed yet). I created the queue with an importer registry so FN-1413 / FN-1414 can call `registerImporter('census', fn)` etc. at startup without further plumbing.
- **No-importer behavior:** before FN-1413 / FN-1414 land, `dryRun: true` runs succeed with `rows_inserted = 0` (the AC explicitly allows this); real runs fail loudly with `status='error'` and `error_message="No importer registered for file '<file>'"` so the operator sees the gap.
- **Cron cadence uses `repeat: { every: ms }` for true biweekly.** Cron expressions can't cleanly express "every 14 days" — `every: 14*24*60*60*1000` actually delivers the cadence. The `FMCSA_IMPORT_CRON` env var override lets ops switch to a cron expression if needed (e.g. weekly on Sunday).
- **Permission seeded via migration, not `config/permissions.js`.** The repo has no `config/permissions.js`; permissions live in the `permissions` table and are seeded by migrations (see the FN-1417 `add_fmcsa_safety_permissions` migration for the pattern this one follows).
- **Lazy router binding in `server.js`.** The queue is constructed inside `app.listen()` (after `initImportQueue()`), so the route is mounted with a closure that lazily creates the `createImportsRouter({ importQueue })` once the queue exists. Returns 503 when the queue is null (Redis unavailable).

### FN-1425 (subtask)
- **No new admin module.** The original ticket suggested an `admin.module.ts` / `admin-routing.module.ts` pair, but the existing codebase already registers admin routes inline in `app-routing.module.ts` and houses admin components under `components/`. Adding a fresh module just for one page would diverge from the established pattern with no upside, so the FMCSA admin page follows the existing convention.
- **Two-source `isInternal` resolution.** The backend (FN-1424) is expected to add `isInternal` to `/api/auth/me`. The frontend reads that field if present and otherwise falls back to the legacy tenant-name check (`'fleetneuron default tenant'`) so the UI works regardless of subtask landing order on `integration/FN-1415`.
- **`5s` polling implemented as a self-rescheduling `setTimeout`.** Cleaner than `interval()` because the cancellation logic is purely status-driven — when no run is `queued` / `running`, no timer is scheduled.

## Operator notes — FMCSA imports 503 messages (FN-1453)

`POST /api/fmcsa/imports/run` and `GET /api/fmcsa/imports` return one of three distinct 503 bodies when the queue is unavailable. Each maps to a different operator action:

| 503 `error` value | Cause | Fix |
|---|---|---|
| `FMCSA import queue not initialized: REDIS_URL is not configured on integrations-service` | `REDIS_URL` env var is unset on `fleetneuron-integrations-service`. Boot logged `[integrations] REDIS_URL not set — FMCSA import queue disabled.` | Set `REDIS_URL` in Render env vars and redeploy. |
| `FMCSA import queue failed to initialize: <message>` | `createImportQueue(...)` threw at boot — typically a malformed `REDIS_URL` (bad scheme, invalid host) or a missing `fmcsa.import_runs` schema. The captured error message is appended verbatim (≤500 chars, no stack). | Read the appended message, fix the underlying config (correct `REDIS_URL`, run pending migrations, etc.), redeploy. |
| `FMCSA import queue not ready: Redis connection lost` | Init succeeded earlier but Redis is currently disconnected (transient network blip, Redis restart, etc.). `isReady()` returns false because `redisConnected` flipped on `end`/`close`. | Usually self-heals once the ioredis client reconnects (retryStrategy keeps trying for up to 10 attempts). If persistent, check Redis service health on Render. |

The existing `503 FMCSA import queue unavailable (Redis not connected)` string from before FN-1453 should no longer appear in production responses. If you see it in logs, you are looking at an old build.
