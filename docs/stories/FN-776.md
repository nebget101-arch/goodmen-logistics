# FN-776: Load wizard steps not rendered on dev — placeholders remain; FN-733 never merged

## Requirement

Restore the FN-725 "New Load" wizard on `dev` to render its real step components instead of the placeholder stubs currently showing in the UI.

## Root Cause

Two independent problems merged into dev as part of the FN-725 story:

1. The FN-732 wizard shell seeded `<div class="wizard-step-placeholder">` stubs inside `<app-load-wizard>` in `loads-dashboard.component.html:1799-1810`. No subsequent merge replaced them with the real step components.
2. FN-733 (Step 1 Basics) was never merged to dev. Commit `e9110a2` exists only on `origin/frontend/FN-718/smart-dropdowns` and `step-basics/` is absent from dev.

- [ ] Wizard steps 1–4 render their real components on dev.
- [ ] End-to-end load creation works through the wizard on dev.
- [ ] All subtask branches merged into a single Bug merge PR.

## Implementation Summary

### FN-777 (subtask)

- Cherry-picked `e9110a2` from `origin/frontend/FN-718/smart-dropdowns` (the only place the FN-733 commit lived).
- Kept only the three step-basics files; discarded cherry-pick edits to `app.module.ts` and `docs/stories/FN-725.md` — those conflicted with the FN-770 module restructure and were out of scope for this subtask.
- Added `StepBasicsComponent` import + declaration to `LoadsDashboardModule` (not `AppModule`, per FN-770).
- No template wiring performed — placeholders in `loads-dashboard.component.html` are intentionally left for FN-778.

### FN-778 (subtask)

- Replaced the four `<div class="wizard-step-placeholder">` stubs in `loads-dashboard.component.html` with the real step components: `<app-step-basics>`, `<app-step-stops>`, `<app-wizard-step-driver>`, `<app-step-attachments>`. Each selector verified against the component's `@Component({ selector: ... })` decorator — note the driver step uses `app-wizard-step-driver`, not `app-step-driver`.
- Added wizard form state fields on `LoadsDashboardComponent`: `wizardBasics` (`StepBasicsData`), `wizardStops` (`LoadStop[]`), `wizardDriverId`/`wizardTruckId`/`wizardTrailerId`, `wizardAttachments` (`WizardAttachment[]`), `wizardAiExtractedPdf`, `wizardAiPrefilledFields`.
- Added per-step handlers that funnel child outputs into wizard state:
  - `onWizardBasicsChange/Valid`, `onWizardStopsChange/Valid`, `onWizardDriver/Truck/TrailerChange`, `onWizardDriverValid`, `onWizardAttachmentsChange`.
  - All value-change handlers flip `wizardDirty = true` so the shell's unsaved-changes guard fires on close.
  - Validity handlers use an immutable `updateStepValid(index, valid)` helper that returns a new array so the `[stepValid]` binding on the shell sees a reference change.
- Added `resetWizardFormState()` called from `openLoadWizard()` and `onWizardSaveAndNew()` to blank the step data; `closeWizard()` already resets navigation state and edit-mode flags.
- Removed `.wizard-step-placeholder` and `.wiz-placeholder-text` SCSS rules from `loads-dashboard.component.scss`.
- **Intentionally out of scope** (not a regression — mirrors pre-existing behavior): the actual POST/PATCH on wizard save, AI-pre-fill wiring for create-from-AI, and hydrating `wizardBasics`/`wizardStops`/driver/etc. from `wizardPrefilledData` in edit mode. `onWizardSave()` still just closes the wizard, same as before FN-778. The Save button inside `<app-step-attachments>` and the shell footer will both emit `save` when pressed — de-duping the buttons is a UX follow-up, not a wiring concern.
- Build verified: `ngc --noEmit -p tsconfig.app.json` produces 0 errors; pre-existing NG8107/NG8102 warnings in unrelated components (drivers, billing) are untouched.

## Key Decisions

### FN-777
