## Implementation Summary
Authored a research-only deliverable at `docs/research/FN-1620-contract-win-analysis.md` covering all six required sections plus a short up-front recommendation and an effort-estimate appendix. The headline finding: a **Phase 2a** build on free, public-domain sources (SAM.gov + SEC EDGAR, plus optional BLS / state DOTs) is feasible in 4–6 weeks at ~$100/day Claude infra cost, and is recommended. **Phase 2b** (paid load boards, shipper portals like Amazon Relay / Walmart Connect, GovWin IQ, BidNet) is recommended for **deferral** until customer pull and partnership economics justify it — those sources have ToS or per-tenant credential constraints that don't fit a platform-wide ingest model.

The doc surveys 8 sources (target was ≥6), maps every carrier-fit feature back to a concrete source — FN-1411's `fmcsa.*` schema and FleetNeuron's `loads`, `vehicles`, `drivers` tables — and specifies a two-call AI architecture: a Haiku 4.5 win-probability scorer (cheap, batched, deterministic, JSON-mode) and a Sonnet 4.6 strategy generator (called only on user click, higher temperature, structured output for talking points and bid recommendation). System prompts on both calls are designed for prompt caching so the per-call cost is dominated by the small per-tenant input. Both handlers would live in `backend/microservices/ai-service/src/handlers/`.

The doc also sketches the ingest pipeline (new `opportunities.*` schema, a new `fleetneuron-opportunities-worker` Render service, daily SAM.gov ingest, weekly EDGAR enrichment, daily nightly scoring), and includes a per-source legal/ToS table flagging which sources are redistributable across tenants and which are BYO-credentials only.

Eight open questions are listed; the most material is the calibration-data question — win-probability without a feedback loop is decorative, so the build plan needs design-partner carriers to label historical wins/losses before the model output is trustworthy.

## Key Decisions
1. **Tier-only-1 build set.** Phase 2a builds only on freely redistributable, public-domain sources. Everything else is documented but explicitly out of scope for the proposed implementation epic. Avoids the licensing/scraping minefield that kills similar projects in year two.
2. **Two AI calls, not one.** Splitting scoring (Haiku, batched, deterministic) from strategy (Sonnet, on-click, creative) costs ~10x less than a single Sonnet call and gives an auditable numeric `winProbability` per (tenant x RFP) pair that we can calibrate over time.
3. **New Render service** for the sourcing worker rather than co-locating in `integrations-service` or `ai-service`. The job is fundamentally a cron-driven ingest — different scaling profile and different failure modes — and mirrors the existing `fleetneuron-db-migrations` "single-purpose service" pattern.
4. **Hard-filter in SQL, soft-rank in AI.** Authority age, equipment type, and insurance minimums become SQL filters at ingest/scoring time; the AI ranks the survivors. Cheap, deterministic, and produces a clean UX.
5. **Calibration is the gate, not infra.** The doc explicitly flags that win-probability is meaningless without ~50–150 labeled historical bids from design-partner carriers. Building the pipeline before this exists ships a number with no meaning. Suggest gating Phase 2a kickoff on calibration commitments.
6. **`loads.is_dedicated` flag is a prerequisite small story** — surfaced in Open Questions so it doesn't get lost when planning the implementation epic.
