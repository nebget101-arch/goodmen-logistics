### FN-1102 (subtask)

**Implementation:** Built `parts-invoice-vision-handler.js` following the toll-invoice handler pattern. Handler accepts either `imageBase64` (jpeg/png/webp/gif) or `pdfBase64`; PDFs use Anthropic's `document` content block while images use `image`. Output schema matches the AC exactly: `{vendor, invoiceNumber, confidence:{vendor,invoiceNumber}, lineItems[{sku,description,qty,unitCost,manufacturer,confidence:{...}}], warnings[]}`.

**Key decisions:**
- **Dependency injection for the Anthropic client** (`deps.anthropic`) so unit tests do not require a real API key. Falls back to lazy-init `getAnthropicClient()` in production. The toll handler used pure lazy-init and was hard to test — this matches the newer `loads-nlq-handler` pattern instead.
- **Per-line confidence is enforced in `validateExtractionResult`** with `clampConfidence` so any out-of-range numbers from the model (or missing fields) get coerced to `[0,1]`. `qty`/`unitCost` are coerced to numbers via `toNumber` (strips `$` and `,`). This protects FN-1103's downstream bulk-create from bad types.
- **422 `AI_INVOICE_UNREADABLE`** when vendor is empty AND `lineItems` is empty — gives the FE a distinct status to show a "couldn't read this invoice" retry prompt instead of a generic 5xx. AC explicitly required structured-error behavior.
- **Model**: `process.env.ANTHROPIC_VISION_MODEL || 'claude-sonnet-4-20250514'` — same env var and default as the toll handler so they roll forward together.

**Files:**
- `backend/microservices/ai-service/src/handlers/parts-invoice-vision-handler.js` (new, 230 lines)
- `backend/microservices/ai-service/src/handlers/__tests__/parts-invoice-vision-handler.spec.js` (new, 250 lines, 10 cases)
- `backend/microservices/ai-service/package.json` (added new spec to `test` script)

**Not in scope (handed off to FN-1103):**
- `POST /api/ai/parts/extract-from-invoice` route registration in `ai-router.js`
- R2 upload of the original invoice image
- Wiring extracted data → existing parts bulk-create endpoint

**Tests (all 10 pass):** validators (clamp/coerce/defaults, markdown-fence parsing); 400 missing-input; 400 unsupported mediaType; happy image multi-line (incl. partial-confidence line); happy PDF (verifies `document` block); 502 parse error on prose; 422 unreadable invoice; 502 SDK throw.

### FN-1103 (subtask)

**Implementation:** Wired the FN-1102 vision handler behind a real route, persisted invoice files to R2, and added a transactional bulk-create endpoint for the FE confirm step.

**Key decisions:**
- **No new `routes/ai-parts.js` file.** FN-1098's `/parts/identify-from-photo` registers inline in `ai-router.js` via `buildAiRouter(deps)`; the AC's "match its pattern exactly" wins over its "lives in `routes/ai-parts.js`" — the file doesn't exist in this codebase. Added the new route alongside `/parts/identify-from-photo`.
- **Don't modify FN-1102.** FN-1102's `handlePartsInvoiceVision(req, res)` writes the AI envelope directly to `res`. Instead of pulling out a "pure" extractor (out of scope), the new intake handler builds a synthetic `req` and a tiny `{status, json}` `res` shim, captures the envelope, then re-emits it on the real `res` with `r2Key` appended.
- **R2 upload comes BEFORE the vision call.** If R2 fails, we 502 with `R2_UPLOAD_FAILED` and never spend Anthropic tokens. If R2 succeeds but vision fails, the file is still safely persisted for retry/audit.
- **New endpoint: `POST /api/parts/bulk`** (did not exist). The closest pre-existing one — `POST /api/parts/bulk-upload` — is Excel-multipart with row-by-row inserts (no transaction); leaving it untouched. The new JSON endpoint:
  - Dedupes by SKU within the request (`duplicate_in_request`).
  - One `WHERE UPPER(sku) IN ($1)` lookup against existing rows (`sku_exists`) — no N+1.
  - Reuses the FN-1091/1093 `resolveManufacturerVendor()` helper for FK auto-create.
  - All inserts wrapped in `db.transaction` — full rollback on any failure.
- **RBAC**: `admin` + `parts_manager`, mirroring single-create `POST /api/parts`.
- **JSON body field name**: AC specifies `{base64, mimeType}`; defensively also accepts FN-1102's native `imageBase64`/`pdfBase64` if a caller passes them straight through.

**Files:**
- `backend/microservices/ai-service/src/handlers/parts-invoice-intake-handler.js` (new)
- `backend/microservices/ai-service/src/handlers/__tests__/parts-invoice-intake-handler.spec.js` (new, 13 cases)
- `backend/microservices/ai-service/src/ai-router.js` (modified — registers `POST /parts/extract-from-invoice` with `multer.single('image')`)
- `backend/microservices/ai-service/package.json` (modified — appended new spec to `test`)
- `backend/packages/goodmen-shared/routes/parts.js` (modified — added `POST /api/parts/bulk`)
- `backend/packages/goodmen-shared/services/parts.service.js` (modified — added `bulkCreateParts()`)

**Tests (all pass):** ai-service `npm test` (9 spec files, including 13 new intake-handler cases); shared `parts.service.bulk.test.js` (4 cases — empty input, AC mixed-payload happy path, validation, single-roundtrip-query verification); full sweep of shared parts tests = 27/27.
