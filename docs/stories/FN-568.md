# FN-568: Expense responsibility data not persisting — lost on driver page reload

## Requirement
After saving expense responsibility values on the driver edit page, navigating away and returning shows blank/zeroed fields. The data appears to save (no error shown) but is not returned on reload.

## Root Cause Analysis

Three compounding bugs were identified:

### Bug 1 — POST always INSERTs (no close of old record)
`POST /drivers/:driverId/expense-responsibility` in `settlements.js` performs a plain INSERT every time the driver is saved. It never closes/end-dates the previous record. This causes multiple `expense_responsibility_profiles` rows to accumulate for the same driver — all with the same `effective_start_date` (today's date).

### Bug 2 — GET ORDER BY is non-deterministic on duplicate dates
Both GET queries (`settlements.js` ~line 722 and `drivers.js` ~line 354) use `ORDER BY effective_start_date DESC LIMIT 1`. When multiple rows share the same `effective_start_date` (caused by Bug 1), PostgreSQL returns an arbitrary row — which may be an older record, making it appear as if the latest save was lost.

### Bug 3 — saveExpenseResponsibility only fires inside savePayeeAssignment success
In `dispatch-drivers.component.ts`, `saveExpenseResponsibility(driverId)` is called only from within the `savePayeeAssignment` success callback (~line 1811). If the driver has no payee assignment change, or if the payee assignment save is skipped/fails, expense responsibility is never sent to the backend — silently discarding the user's input.

## Acceptance Criteria
- [ ] POST expense-responsibility closes/end-dates the previous active record before inserting new one
- [ ] GET expense-responsibility returns the most-recently-created record (add `created_at DESC` as tiebreaker)
- [ ] `saveExpenseResponsibility` fires directly after driver update/create success, independent of payee assignment
- [ ] Expense responsibility persists after save + page reload
- [ ] Expense responsibility saves even when no payee assignment change is made
- [ ] DB: only one un-end-dated expense responsibility row per driver after multiple saves

## Implementation Summary

### FN-569 (backend subtask)
**Branch**: `backend/FN-569/expense-responsibility-persist-fix`

Three changes across two files:

**`settlements.js` POST `/drivers/:driverId/expense-responsibility`**
- Wrapped the handler in `knex.transaction()` for atomicity
- Before INSERT: `UPDATE expense_responsibility_profiles SET effective_end_date = newStart, updated_at = now() WHERE driver_id = ? AND effective_end_date IS NULL`
- This ensures only one open (un-end-dated) row exists per driver at any time
- Mirrors the pattern in `POST /drivers/:driverId/payee-assignment/resolve`

**`settlements.js` GET `/drivers/:driverId/expense-responsibility`**
- Added `created_at DESC` as secondary sort: `.orderBy([{column: 'effective_start_date', order: 'desc'}, {column: 'created_at', order: 'desc'}])`
- Ensures the most-recently created row wins when multiple rows share the same `effective_start_date` (defense-in-depth for existing duplicate data in the DB)

**`drivers.js` GET `/:id`** (embedded expense query)
- Added `, created_at DESC` to the raw SQL ORDER BY clause
- Same tiebreaker logic as the settlements.js GET fix

## Key Decisions
- Mirror the close-old-record pattern already used in `POST /drivers/:driverId/payee-assignment/resolve` (settlements.js ~line 629): UPDATE old record `effective_end_date = CURRENT_DATE - 1` before INSERT.
- `expense_responsibility_profiles` has `created_at` column (added by `timestamps(true, true)` in migration) — safe to use as secondary sort.
