# FN-1071: [Bug] Load Detail Drawer — Attachments tab is empty

## Requirement
The Phase-2 Load Detail Side Drawer's Attachments tab renders nothing for loads that have saved attachments, even though the API returns them. Backend + APIs already work — this is a pure frontend gap. The wizard's `step-attachments` component only models pending uploads (`WizardAttachment` with `file: File`) and has no UI for saved server-side `LoadAttachment` rows. Additionally, the drawer's `populateFromDetail()` discards `d.attachments` from the API response.

## Acceptance Criteria
- [ ] Existing attachments render in the Attachments tab with: filename, type label, uploaded date, download link (opens signed URL in new tab), delete button.
- [ ] Delete calls `DELETE /api/loads/:id/attachments/:attachmentId` (existing endpoint at `backend/packages/goodmen-shared/routes/loads.js:3200`) and removes the row on success.
- [ ] Pending-upload UX (drag/drop + per-file metadata) continues to work and is visually separated from the saved list.
- [ ] Saving the load uploads new pending files (existing behavior preserved).
- [ ] Works in both the side drawer and the legacy full-page edit modal (`loads-dashboard.component.html:2519` also uses `<app-step-attachments>`).

## Dependencies

## Implementation Summary

### FN-1072 (subtask)
Branch: `frontend/FN-1072/drawer-saved-attachments`

**What changed**
- `LoadDetailDrawerComponent.populateFromDetail()` no longer drops `d.attachments`. The drawer now keeps two parallel lists:
  - `attachments: WizardAttachment[]` — pending (drag/drop) uploads, unchanged.
  - `savedAttachments: LoadAttachment[]` — server-side persisted attachments, populated from `d.attachments`.
- Drawer template binds `[saved]="savedAttachments"` and `(deleteSaved)="onDeleteSavedAttachment($event)"` on `<app-step-attachments>`.
- `onDeleteSavedAttachment(id)` does an optimistic remove from both `savedAttachments` and `loadDetail.attachments`, calls `LoadsService.deleteAttachment(loadId, id)`, and rolls back + sets `errorMessage` on failure. No-op when `loadId` is null.
- `StepAttachmentsComponent` gained:
  - `@Input() saved: LoadAttachment[] = []`
  - `@Output() deleteSaved = new EventEmitter<string>()`
  - Helpers `trackBySavedId`, `getSavedFileIcon` (mime-driven), `formatSavedDate`, `onDeleteSavedClick`.
- Template adds a "Saved attachments" section above the dropzone (visually distinct card list with type label, formatted date, file_url link in new tab, delete icon-button). Empty state now also checks `saved.length`.
- SCSS adds `.sa-saved-section` and friends, reusing the existing AI-dark-theme palette (no new colors).

**What did not need to change**
- `LoadsService.deleteAttachment(loadId, attachmentId)` already existed (`frontend/src/app/services/loads.service.ts:207`).
- The "legacy edit modal" referenced in the ticket description does **not** use `<app-step-attachments>` for saved rows — `loads-dashboard.component.html:1679–1720` already renders saved attachments via its own `documents-table` with edit + delete. No change needed there.
- The only other `<app-step-attachments>` call site (`loads-dashboard.component.html:2538`) is the create-wizard (no saved attachments yet); the new `[saved]` input defaults to `[]`, so the call site continues to work unchanged.

## Key Decisions
- **Two parallel arrays, not one.** `savedAttachments` (LoadAttachment) and `attachments` (WizardAttachment) have different shapes and lifecycles — the description explicitly called out not co-mingling them. Saved is read-only-from-server + server-mutating-via-DELETE; pending is local-state-only until `Save`.
- **Parent owns the DELETE call.** Child stays loadId-agnostic — emits the id, parent calls the service. Same shape as the description.
- **Optimistic delete with rollback.** Removes immediately for snappier UX, restores the full previous list on failure (atomic — no half states). Failure surfaces via the existing `errorMessage` banner.
- **Both `savedAttachments` and `loadDetail.attachments` updated.** Keeping `loadDetail` in sync means a re-`populateFromDetail()` call (e.g. after save) won't resurrect a just-deleted row.
- **Touched only the drawer, not the legacy modal.** The legacy edit modal already has its own attachment table — wiring it through `<app-step-attachments>` would duplicate UI, not consolidate it.
