# FN-1398: Barcode generator + printable label on part create

## Requirement
When a user adds a part to the parts catalog, the backend auto-generates a unique scannable barcode value and the frontend offers an immediate "Print label" action that produces a sheet-ready label containing the part's name, SKU, a Code-128 1D barcode, and a QR code — both encoding the same generated barcode value.

Closes a gap in the FN-1090 quick-add flow: today `parts.barcode` is a manual free-text field, so warehouse staff have no way to print a scannable label at part-creation time and fall back to typing SKUs by hand at receiving/picking.

## Acceptance Criteria
- [ ] On `POST /api/parts`, if `barcode` is empty, backend generates `FN-XXXXXXXX` (8 alphanumeric chars from `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` — no `O/0/I/1`) and persists it in `parts.barcode`.
- [ ] If the client supplies a `barcode` value, it is preserved verbatim (no regeneration).
- [ ] Generated value is unique across `parts.barcode`; on collision, retry up to 5 times before failing the request.
- [ ] After successful part create, parts-catalog UI shows a "Print label" action in the success modal.
- [ ] Print template renders: part name (large), SKU, barcode value as text, Code-128 SVG, QR SVG — both encoding the same value.
- [ ] `window.print()` produces a clean label with no app chrome (sidebar/header/buttons hidden via `@media print`).
- [ ] A real 1D laser scanner reads the Code-128; a phone camera reads the QR — both decode to the stored value.
- [ ] No DB migration needed (uses existing `parts.barcode` column).

## Implementation Summary

### FN-1400 (subtask) — Backend barcode auto-generation

Added `barcode-generator.js` helper exposing `resolveBarcodeForCreate(supplied, checkExists)`:

- If `supplied` is a non-empty string, it is preserved verbatim (no regeneration, no validation — matches existing behavior for clients that bring their own barcode).
- Otherwise generates `FN-XXXXXXXX` from the safe alphabet `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (no `O/0/I/1`) and retries up to 5 times if `checkExists` reports a collision. After 5 collisions it throws an `Error` with `statusCode = 500`.

Wired into `partsService.createPart` via `isBarcodeTaken(value)`, which checks **both** `parts.barcode` and `part_barcodes.barcode_value` so the generated value cannot collide with anything a scanner could match. The resulting barcode is included in the `parts` insert; `POST /api/parts` already returns the row via `.returning('*')` so the frontend receives the value without a re-fetch.

Unit tests in `barcode-generator.test.js` cover: candidate format match, single-shot success, retry path with simulated collisions, exhaustion (throws after 5 attempts), `supplied → preserved`, and `empty/whitespace/null/undefined → generated`. 11/11 pass under `node --test`.

## Key Decisions
- **Option B chosen for value strategy**: backend generates `FN-XXXXXXXX` rather than reusing SKU. Rationale: decouples printed labels from SKU edits; avoids exposing user-typed SKUs in scanned form.
- **Both Code-128 and QR on the same label**: Code-128 for cheap warehouse 1D scanners, QR as fallback for phone cameras. Same value encoded in both.
- **Browser print, not Zebra/Dymo**: scoped to `window.print()` + `@media print`. Printer-specific formats deferred.
