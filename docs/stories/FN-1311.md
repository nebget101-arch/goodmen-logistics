# FN-1311: Smart Alert "Reassign load" quick action opens legacy load drawer/modal instead of V2 wizard

## Type
Bug

## Requirement
Clicking the **Reassign load** quick-action button on a Smart Alert card (introduced in FN-1129) navigates to `/loads?action=reassign&loadId=<X>` and opens the legacy load detail drawer / `<app-load-wizard>` (FN-732) — not the V2 wizard (FN-862 `<app-load-wizard-v2>`). User expectation: load-context quick-action deep-links should land in the V2 wizard in `mode=edit` with `loadId` prefilled.

## Root cause
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts:941-953` — `ngOnInit` queryParams subscription reads only `loadId` and triggers the legacy detail drawer (`showDetailsModal = true`); `action` is ignored.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.html:2549-2554` — V2 wizard binding is hard-coded to `mode="create"`; `mode="edit"` + `[loadId]` are never bound.

## Acceptance Criteria
- [ ] Clicking **Reassign load** on a Smart Alert opens `<app-load-wizard-v2>` with `mode="edit"` and `[loadId]` set to the alert's load id.
- [ ] Legacy detail drawer (`showDetailsModal`) does NOT open when `action=reassign` is present.
- [ ] Legacy `<app-load-wizard>` does NOT open for `action=reassign`.
- [ ] Closing/saving V2 wizard clears `action` and `loadId` query params from the URL.
- [ ] `?loadId=X` without `action` continues to open the detail drawer (no regression on unrelated deep-links).
- [ ] Users without load-edit permission cannot reach edit mode.

## Dependencies
None — frontend-only fix.

## Caused by
FN-1129 (S5: Quick Actions) — shipped the buttons, but the loads-dashboard handler was never extended to interpret `action=reassign`.

## Implementation Summary

### FN-1312 (subtask)
Extended the `loads-dashboard` `queryParams` subscription so the `?action=reassign&loadId=X` deep-link (emitted by Smart Alert / Predictive Insight quick-action buttons from FN-1129) opens `<app-load-wizard-v2>` in `mode='edit'` with `[loadId]` prefilled — instead of the legacy `<app-load-detail-drawer>` / `<app-load-wizard>`.

Changes:
- Pulled the `route.queryParams` body out of `ngOnInit` into a new public method `applyRouteQueryParams(params)` so unit tests can drive the routing branches without spinning up the full component lifecycle.
- Added `loadWizardV2Mode` and `loadWizardV2LoadId` state, wired into the V2 wizard binding in the template (replaces hard-coded `mode="create"`).
- Added `(updated)` handler `onLoadWizardV2Updated` for the edit-mode save path (success toast + grid reload + clear deep-link params).
- `closeLoadWizardV2` and `onLoadWizardV2Updated` now strip `action` + `loadId` from the URL via `router.navigate([], { queryParams: { action: null, loadId: null }, queryParamsHandling: 'merge', replaceUrl: true })`. Create-mode close (e.g. "+ New Load" cancel) does NOT navigate, so unrelated query state stays put.
- Permission gate: when `accessControl.hasPermission(PERMISSIONS.LOADS_EDIT)` is false, the `?action=reassign&loadId=X` deep-link falls through to the read-only detail drawer — matches the V2 wizard's own `canEdit` gate.
- Bare `?loadId=X` (no action) still opens the detail drawer (regression check covered by spec).
- Spec: 8 new cases under `LoadsDashboardComponent.applyRouteQueryParams (FN-1312)` covering edit-mode routing, permission fallback, drawer regression, query-param cleanup, and the create-mode no-regression path.

## Files Changed
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` — new state, `AccessControlService` injection, `applyRouteQueryParams` method, `clearLoadActionQueryParams` helper, `onLoadWizardV2Updated` handler, updated `closeLoadWizardV2` / `onLoadWizardV2Created` / `openLoadWizardV2` to manage the new mode state.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.html` — V2 wizard binding now `[mode]="loadWizardV2Mode"` + `[loadId]="loadWizardV2LoadId"` and listens to `(updated)`.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.spec.ts` — `makeComponent` factory takes optional `getLoad`, `hasPermission`, `routerNavigate` overrides; constructs the new `AccessControlService` 12th arg; new `FN-1312` describe block with 8 specs.

## Key Decisions
- **Extracted `applyRouteQueryParams` instead of inlining the new logic in the subscription callback.** The existing spec file deliberately skips `ngOnInit` (FN-1078 comment in the spec explains why — it pulls in 11+ services and is too brittle to drive end-to-end). A pure method is the cheapest way to lock in the routing branches without TestBed.
- **Permission gate falls through to the detail drawer rather than showing a toast.** Read-only users still get a useful UX (they can see the load) instead of being blocked outright. This matches the V2 wizard's own `canEdit` philosophy (header Edit button is gated, but view mode is allowed).
- **Only edit-mode close clears the URL params.** The "+ New Load" entry point opens the wizard with no query params, so calling `router.navigate` on close would be a no-op at best and could clobber unrelated params (e.g. `status=NEW`) at worst. We track the mode at close time and only navigate when transitioning out of edit.
- **Used `queryParamsHandling: 'merge'` with explicit `null` values.** This is the Angular idiom for removing specific keys from the URL while preserving others. `replaceUrl: true` avoids polluting the back-stack with the cleanup nav.
- **Bound the wizard's `(updated)` output.** Previously only `(created)` was bound (wizard was create-only at the dashboard entry point); edit-mode save would have left the wizard open. The new handler closes the wizard, refreshes the grid, shows a success toast, and clears the deep-link params.
