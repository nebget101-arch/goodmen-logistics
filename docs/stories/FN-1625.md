# FN-1625: DQF — Upload CDL, AI-extract identity + license fields, prefill Add New Driver modal

## Requirement
Let DQF users upload a CDL (image or PDF), have Claude vision OCR the document, then open the existing Add New Driver modal with the extractable fields pre-populated and clearly marked as AI-extracted. Phone, hire date, and medical cert expiry are explicitly excluded from extraction (they don't appear on a CDL) and stay blank with manual-entry hints.

## Acceptance Criteria
- [ ] DQF page surfaces "Upload CDL to add driver" CTA with image/PDF picker (max 10 MB).
- [ ] On extraction success, Add New Driver modal opens pre-filled with: firstName, lastName, dateOfBirth, streetAddress, city, state, zipCode, cdlNumber, cdlState, cdlClass, cdlExpiry (and middleName if present).
- [ ] Each pre-filled field shows an "AI" pill / accent indicator + a banner reads "AI extracted these fields — please review."
- [ ] phone, hireDate, medicalCertExpiry are NOT touched and show "Manual entry required" hint.
- [ ] Confidence floor (default 0.6) drops low-confidence values to null server-side; FE shows them empty.
- [ ] Failure paths (non-CDL upload, AI 5xx, all fields below floor) → modal opens empty with toast "Couldn't read CDL — please enter manually." Existing manual flow unchanged.
- [ ] No PII logged. CDL bytes are never persisted (no R2, no disk, no DB) — they live only in the request lifecycle.
- [ ] Existing Add New Driver flow continues to work without an upload.
- [ ] All four subtasks merge to `integration/FN-1625`; final PR `integration/FN-1625 → dev`.

## Wire contract (camelCase, used by both FN-1627 and FN-1628)
```json
{
  "success": true,
  "extracted": {
    "firstName": "John", "middleName": null, "lastName": "Doe",
    "dateOfBirth": "1985-04-12",
    "streetAddress": "123 Main St", "city": "Dallas", "state": "TX", "zipCode": "75201",
    "cdlNumber": "12345678", "cdlState": "TX", "cdlClass": "A", "cdlExpiry": "2028-04-12"
  },
  "extractedFields": ["firstName","lastName","dateOfBirth","streetAddress","city","state","zipCode","cdlNumber","cdlState","cdlClass","cdlExpiry"],
  "meta": { "lowConfidenceFields": [], "processingMs": 0 }
}
```
Failure shape: `200 { success: false, extracted: null, reason: 'low_confidence' | 'ai_unavailable' }`. **Do NOT collapse to a generic "AI service unavailable" string** (lesson from FN-1605).

## Out of scope
- CDL back-side OCR (endorsements/restrictions detail)
- Storing the CDL image as a driver document
- Auto-creating the driver without user review (user always clicks Save)
- Medical card / med cert extraction (separate document, separate story)

## Implementation Summary

### FN-1626 (subtask, ai)
Added `POST /api/ai/drivers/cdl-vision` to the `ai-service` microservice. Pipeline:

1. **Pre-Anthropic validation** — `imageBase64` required + non-empty, `mimeType` in the three-element allowlist (`image/jpeg`, `image/png`, `application/pdf`), decoded payload size ≤ 10 MB. All three reject with `{ success: false, error, code: 'AI_BAD_REQUEST' }` (HTTP 400) before any LLM call.
2. **Claude vision call** — uses the existing shared model constant (`process.env.ANTHROPIC_VISION_MODEL || 'claude-sonnet-4-20250514'`, identical to `psp-report-vision-handler.js` and `toll-invoice-vision-handler.js`). PDFs go in via the `document` content block, JPEG/PNG via `image`. The system prompt is sent as a `text` block with `cache_control: { type: 'ephemeral' }` — the prompt is static, so every request after the 5-minute warmup hits the cache.
3. **Hallucination protection** — every field returned by the model is re-validated server-side before responding. Anything failing its validator is coerced to `{ value: null, confidence: 0 }`:
   - `zipCode` matches `^\d{5}(-\d{4})?$`
   - `state` / `cdlState` are members of the 50-state + DC + 5-territory `Set`
   - `cdlClass` is exactly `'A'`, `'B'`, or `'C'`
   - `dateOfBirth` parses via `Date.parse` AND year strictly between 1900 and the current year (no future DOB)
   - `cdlExpiry` parses via `Date.parse` AND year ≤ currentYear + 20 (past expiry dates allowed — represents an expired CDL the user is updating)
4. **Output guarantee** — handler always returns all 12 keys, every key has shape `{ value: string|null, confidence: 0..1 }`. Missing keys, non-finite confidence, and out-of-range confidence are normalized.
5. **Anthropic failure handling** — any throw from the SDK call returns 502 `{ success: false, error, code: 'AI_UPSTREAM_ERROR' }`. Both happy-path and failure-path emit a `logAiInteraction` row.
6. **PII-free logging** — `logAiInteraction` records route (`/drivers/cdl-vision`), success flag, processing time, model, and an aggregate count of fields at confidence ≥ 0.6. **No field values** ever reach the log buffer. Test suite asserts this by stringifying every `logAiInteraction` argument and confirming the mocked first name, last name, and CDL number never appear.

### FN-1627 (subtask, backend)
Added `POST /api/dqf/cdl-extract` to the DQF router. Pipeline:

1. `multer.single('file')` (memory storage, 10 MB cap) — wrapped in a small middleware that translates `LIMIT_FILE_SIZE` to a clean `413 { message: 'File exceeds 10 MB limit' }` instead of leaking a 500.
2. Strict mime allowlist for this route only: `image/jpeg`, `image/png`, `application/pdf`. The shared `upload` filter at the top of `dqf.js` permits TIFF for other routes; the CDL route narrows that with an inline check after multer.
3. Hands `req.file.buffer` to a new helper module `cdl-extraction-service.js` which base64-encodes, POSTs to `${AI_SERVICE_URL}/api/ai/drivers/cdl-vision` (FN-1626), and applies a per-field confidence floor (default `0.6`, override via `CDL_EXTRACT_CONFIDENCE_FLOOR`).
4. Confidence-floor logic accepts either `{ fields: { firstName: { value, confidence } } }` or a flat `{ firstName: { value, confidence } }` from the AI service. Values without an attached confidence are treated as 1.0 (passthrough).
5. Returns the camelCase wire-contract payload exactly as documented at the top of this story:
   - `200 { success: true, extracted, extractedFields, meta: { lowConfidenceFields, processingMs } }` on extraction
   - `200 { success: false, extracted: null, reason: 'ai_unavailable' | 'low_confidence' }` on graceful failure (AI 5xx, network error, all fields below floor)

The buffer is never written anywhere — no R2, no disk, no DB. Logging is metadata-only via the existing `dtLogger`: `extractedFieldCount`, `lowConfidenceFieldCount`, `processingMs`, `mimeType`, `fileSizeBytes`. Field values never reach the log buffer (covered by an explicit "no PII in logs" assertion in the test suite).

Auth is the router-level `auth(['admin', 'safety'])` gate already declared at the top of `dqf.js` — no extra RBAC check is needed on this route.

### FN-1628 (subtask, frontend)
Added the `Upload CDL` CTA + extraction-driven pre-fill flow on `/drivers/dqf`. Click → hidden `<input type="file">` → `apiService.extractCdl(file)` (multipart POST `/api/dqf/cdl-extract`) → on success the existing Add New Driver modal opens with the extracted camelCase fields applied and an `AI` pill on each pre-filled label. The user can review, edit, and save via the unchanged `addDriver()` path; or hit `Clear AI fields` in the modal banner to reset just the 11 CDL-extractable fields and re-enter manually. On any failure (success:false, network/API error, or all fields below floor) the modal still opens but empty, with a banner reading "Couldn't read CDL — please enter manually." `phone`, `hireDate`, `medicalCertExpiry` are never touched by the AI path and each shows a "Manual entry required" hint.

The existing manual `Add Driver` button + flow are untouched. Cancelling the modal also clears any AI state so the next manual open is clean.

## Key Decisions

### FN-1626 (subtask, ai)
- **Server-side re-validation, not "trust the model"**: the prompt instructs Claude to drop confidence to ≤0.4 on uncertainty, but a hallucinated high-confidence value (e.g., `zipCode: 'ABCDE'` at 0.9) would still poison the downstream prefill. The handler's post-validation pass coerces any field whose value fails its shape rule to `{ value: null, confidence: 0 }` — the FN-1627 confidence floor then drops it to null on the wire. Two layers, defense in depth.
- **Cached system prompt only, not the image**: image bytes are unique per request and have no cache value. The system prompt (~1 KB describing the document, schema, and rules) is identical every call and ideal for ephemeral caching. The 5-minute TTL is a non-issue at expected DQF traffic.
- **PDFs via the `document` block, not pre-rendered to image**: Claude ingests PDFs natively. Pre-rendering would require a PDF→image dependency in the ai-service container and adds a failure mode for nothing. The mvr-vision handler already uses the document path successfully.
- **`deps.anthropic` injection for tests but lazy module-level client in prod**: the route mount in `ai-router.js` does not pre-build an Anthropic client (no other handler reads `deps.anthropic`). Following the `load-driver-match-handler.js` pattern, the handler accepts `deps.anthropic` as an override (used by every test case to stub `messages.create`) and falls back to a single lazy-init module-level client in production. Avoids per-request client churn while keeping the unit tests fully hermetic.
- **Tests under plain `node`, not jest**: ai-service has no jest dep and no jest binary; `npm test` runs every test file with `node`. Mirrored `loads-nlq-handler.test.js` (uses `node:assert/strict` and a hand-rolled `makeRes()` mock). Verification command: `node src/handlers/__tests__/cdl-vision-handler.test.js`.

### FN-1627 (subtask, backend)
- **Strict mime allowlist on this route only, not the file-level multer**: the `upload` instance at the top of `dqf.js` is shared with the MVR upload route and permits TIFF. The CDL route narrows mime types via an inline post-multer check rather than introducing a second multer instance — keeps file-upload behavior centralized while still enforcing the spec's strict allowlist.
- **413 for oversize, 400 for mime rejection**: wrapped `upload.single('file')` in a small middleware that maps `LIMIT_FILE_SIZE` to 413 and any other multer error to 400. This avoids the default 500-on-multer-error that the MVR upload route inherits.
- **Confidence floor lives in the service, not the route**: the route is a thin proxy. The service owns the AI call, the floor application, and the structured failure shape. This keeps the route file (already 1.9k lines) from growing more business logic, and lets the test suite exercise floor logic in isolation.
- **Floor applies to the field, not the AI's "confident enough overall" signal**: each field is judged independently. If 8 of 11 fields meet the floor, those 8 land in `extractedFields` and the other 3 land in `meta.lowConfidenceFields` so the FE could surface them later if we wanted.
- **`success: false reason: 'ai_unavailable'` is a 200, not a 502**: explicit lesson from FN-1605. The FE needs a structured signal to swap into the empty-modal-with-toast fallback; a non-200 would push the FE into a generic error toast that masks the actionable "Couldn't read CDL" copy.
- **Logger spy assertion enforces PII discipline**: the test serializes every `dtLogger` call's args and asserts `'John'`, `'Doe'`, `'12345678'` never appear. Future contributors who add a `dtLogger.info('cdl_done', { firstName: extracted.firstName })` will fail the suite immediately.

### FN-1628 (subtask, frontend)
- **Concurrency: ignore re-clicks while an extraction is in flight.** Both `openCdlPicker()` and `onCdlFileSelected()` early-return when `extractingCdl` is true; the CTA is also `[disabled]`. The cancel-and-replace alternative would have required tracking an inner subscription and isn't justified by user value.
- **One-way contract guard on the FE.** Even though the backend should never return `phone` / `hireDate` / `medicalCertExpiry`, `applyCdlExtraction` filters those keys out as a defensive measure. Cheap insurance against a future backend regression slipping these through.
- **Spec scope.** The existing DriversComponent has no Karma spec; standing up a full TestBed harness for an unrelated component would be a large yak shave. The new spec instantiates the component manually with stub services and exercises the CDL methods directly. This covers the AC-listed cases (extraction success populates `newDriver`, excluded fields stay empty, failure path opens the empty modal with the toast text, concurrent click ignored, AI-pill / banner state booleans). Visual rendering is verified by the FN-1629 manual QA pass.
- **Distinct visual identity for the CTA.** `btn-cdl-upload` uses a purple/indigo gradient instead of reusing the teal `btn-primary` of the manual `Add Driver` button. The two paths are now visually distinguishable in the header, so a user choosing between "I have a CDL image" and "I'll type it in" can tell them apart at a glance.
