# FN-1625: DQF — Upload CDL, AI-extract identity + license fields, prefill Add New Driver modal

## Requirement
Let DQF users upload a CDL (image or PDF), have Claude vision OCR the document, then open the existing Add New Driver modal with the extractable fields pre-populated and clearly marked as AI-extracted. Phone, hire date, and medical cert expiry are explicitly excluded from extraction (they don't appear on a CDL) and stay blank with manual-entry hints.

## Acceptance Criteria
- [ ] DQF page surfaces "Upload CDL to add driver" CTA with image/PDF picker (max 10 MB).
- [ ] On extraction success, Add New Driver modal opens pre-filled with: firstName, lastName, dateOfBirth, streetAddress, city, state, zipCode, cdlNumber, cdlState, cdlClass, cdlExpiry (and middleName if present).
- [ ] Each pre-filled field shows an "AI" pill / accent indicator + a banner reads "AI extracted these fields — please review."
- [ ] phone, hireDate, medicalCertExpiry are NOT touched and show "Manual entry required" hint.
- [ ] Confidence floor (default 0.6) drops low-confidence values to null server-side; FE shows them empty.
- [ ] Failure paths (non-CDL upload, AI 5xx, all fields below floor) → modal opens empty with toast "Couldn't read CDL — please enter manually." Existing manual flow unchanged.
- [ ] No PII logged. CDL bytes are never persisted (no R2, no disk, no DB) — they live only in the request lifecycle.
- [ ] Existing Add New Driver flow continues to work without an upload.
- [ ] All four subtasks merge to `integration/FN-1625`; final PR `integration/FN-1625 → dev`.

## Wire contract (camelCase, used by both FN-1627 and FN-1628)
```json
{
  "success": true,
  "extracted": {
    "firstName": "John", "middleName": null, "lastName": "Doe",
    "dateOfBirth": "1985-04-12",
    "streetAddress": "123 Main St", "city": "Dallas", "state": "TX", "zipCode": "75201",
    "cdlNumber": "12345678", "cdlState": "TX", "cdlClass": "A", "cdlExpiry": "2028-04-12"
  },
  "extractedFields": ["firstName","lastName","dateOfBirth","streetAddress","city","state","zipCode","cdlNumber","cdlState","cdlClass","cdlExpiry"],
  "meta": { "lowConfidenceFields": [], "processingMs": 0 }
}
```
Failure shape: `200 { success: false, extracted: null, reason: 'low_confidence' | 'ai_unavailable' }`. **Do NOT collapse to a generic "AI service unavailable" string** (lesson from FN-1605).

## Implementation Summary

### FN-1627 (subtask, backend)
Added `POST /api/dqf/cdl-extract` to the DQF router. Pipeline:

1. `multer.single('file')` (memory storage, 10 MB cap) — wrapped in a small middleware that translates `LIMIT_FILE_SIZE` to a clean `413 { message: 'File exceeds 10 MB limit' }` instead of leaking a 500.
2. Strict mime allowlist for this route only: `image/jpeg`, `image/png`, `application/pdf`. The shared `upload` filter at the top of `dqf.js` permits TIFF for other routes; the CDL route narrows that with an inline check after multer.
3. Hands `req.file.buffer` to a new helper module `cdl-extraction-service.js` which base64-encodes, POSTs to `${AI_SERVICE_URL}/api/ai/drivers/cdl-vision` (FN-1626), and applies a per-field confidence floor (default `0.6`, override via `CDL_EXTRACT_CONFIDENCE_FLOOR`).
4. Confidence-floor logic accepts either `{ fields: { firstName: { value, confidence } } }` or a flat `{ firstName: { value, confidence } }` from the AI service. Values without an attached confidence are treated as 1.0 (passthrough).
5. Returns the camelCase wire-contract payload exactly as documented at the top of this story:
   - `200 { success: true, extracted, extractedFields, meta: { lowConfidenceFields, processingMs } }` on extraction
   - `200 { success: false, extracted: null, reason: 'ai_unavailable' | 'low_confidence' }` on graceful failure (AI 5xx, network error, all fields below floor)

The buffer is never written anywhere — no R2, no disk, no DB. Logging is metadata-only via the existing `dtLogger`: `extractedFieldCount`, `lowConfidenceFieldCount`, `processingMs`, `mimeType`, `fileSizeBytes`. Field values never reach the log buffer (covered by an explicit "no PII in logs" assertion in the test suite).

Auth is the router-level `auth(['admin', 'safety'])` gate already declared at the top of `dqf.js` — no extra RBAC check is needed on this route.

## Key Decisions

### FN-1627 (subtask, backend)
- **Strict mime allowlist on this route only, not the file-level multer**: the `upload` instance at the top of `dqf.js` is shared with the MVR upload route and permits TIFF. The CDL route narrows mime types via an inline post-multer check rather than introducing a second multer instance — keeps file-upload behavior centralized while still enforcing the spec's strict allowlist.
- **413 for oversize, 400 for mime rejection**: wrapped `upload.single('file')` in a small middleware that maps `LIMIT_FILE_SIZE` to 413 and any other multer error to 400. This avoids the default 500-on-multer-error that the MVR upload route inherits.
- **Confidence floor lives in the service, not the route**: the route is a thin proxy. The service owns the AI call, the floor application, and the structured failure shape. This keeps the route file (already 1.9k lines) from growing more business logic, and lets the test suite exercise floor logic in isolation.
- **Floor applies to the field, not the AI's "confident enough overall" signal**: each field is judged independently. If 8 of 11 fields meet the floor, those 8 land in `extractedFields` and the other 3 land in `meta.lowConfidenceFields` so the FE could surface them later if we wanted.
- **`success: false reason: 'ai_unavailable'` is a 200, not a 502**: explicit lesson from FN-1605. The FE needs a structured signal to swap into the empty-modal-with-toast fallback; a non-200 would push the FE into a generic error toast that masks the actionable "Couldn't read CDL" copy.
- **Logger spy assertion enforces PII discipline**: the test serializes every `dtLogger` call's args and asserts `'John'`, `'Doe'`, `'12345678'` never appear. Future contributors who add a `dtLogger.info('cdl_done', { firstName: extracted.firstName })` will fail the suite immediately.

  - `curl -X POST -H "Authorization: Bearer <admin-jwt>" -F "file=@cdl.jpg" "https://<service>/api/dqf/cdl-extract"` → `200 { success: true, extracted: { firstName, ... } }`.
