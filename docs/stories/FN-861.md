## Implementation Summary

### FN-871 (subtask)
Implemented `WizardShellComponent` at `frontend/src/app/components/shared/wizard/`. Standalone Angular component that renders the fuel-import-wizard progress rail (active / done / upcoming states) and a footer that adapts to `mode` and `canProceed`. Purely presentational ŌĆö the parent owns `currentStepId`, step gating, and form state; the shell only projects the step body via `<ng-content>` and emits navigation intent.

API as shipped:
- **Inputs**: `steps: WizardStepDef[]`, `currentStepId: string | null`, `canProceed: boolean`, `mode: 'create' | 'edit' | 'view'`
- **Outputs**: `stepChange(id)`, `back`, `next`, `submit`, `close`
- **Types**: `WizardStepDef { id: string; label: string; icon?: string }`, `WizardMode`

Footer behavior:
- `create` / `edit`: `[Back] [Continue]` ŌĆö Continue disabled when `canProceed === false`. On the final step, Continue becomes `Submit` (emits `submit`).
- `view`: only `[Close]` ŌĆö submit is hidden on every step including the final; progress rail remains fully clickable so reviewers can inspect any step.

Rail click gating:
- In `create` / `edit`: only previously-completed steps are clickable (matches fuel-import-wizard).
- In `view`: any non-current step is clickable.

Unit spec covers rail rendering, active/done class application, `canProceed` gating of Continue, final-step Continue ŌåÆ Submit swap, view-mode footer (Close only, Submit hidden), close emission, and step-change gating per mode.

## Key Decisions

- **Standalone component** over NgModule declaration. Lets S2's LoadWizardModule import `WizardShellComponent` directly without plumbing a new shared module. Matches the spec's "tree-shakeable / standalone or module-exported cleanly" option and keeps the shell self-contained.
- **OnPush change detection** to avoid pulling parent CD cycles into the rail render path ŌĆö state is fully input-driven.
- **Added `close` output** to FN-861's proposed API so `view` mode can signal its Close button without overloading `back` or `submit`. This is an additive change; `create`/`edit` modes are unchanged.
- **Ported fuel-import-wizard SCSS verbatim** for the rail, panel, and button visuals so `stepState` (active/done/upcoming) matches pixel-for-pixel. Did not extract into a shared SCSS partial ŌĆö the styles live inside the component to avoid coupling new consumers to global selectors.
- **Rail cursor gating**: `.rail-step.clickable` only when navigation is permitted (done step in edit, or any non-current step in view). Matches the existing UX of "click to revisit" without suggesting forward-skip.
- **Step icon support**: `WizardStepDef.icon` is rendered via `material-symbols-outlined` for the active/upcoming bubble; done state always shows a check to match fuel-import-wizard. No icon fallback falls back to the step number.
