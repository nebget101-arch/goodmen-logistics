# FN-1554: [Work Order] Customer still not auto-selected on WO open — payload reads `customer_id` but backend returns `shop_client_id` (regression after FN-1518)

## Requirement
After FN-1518 was merged (commit `4c9845cb`), the customer name input on the Basics tab still does not auto-populate when opening an existing Work Order. The FN-1518 fix to `populateCustomerDisplay()` is correct but never executes because the upstream payload mapping in `work-order.component.ts:152` reads `wo.customer_id || wo.customerId` — but the backend returns the field as `shop_client_id`. Result: `this.workOrder.customerId` is null, `populateCustomerDisplay()` early-returns.

One-line fix: change line 152 to `customerId: wo.shop_client_id || wo.shopClientId || wo.customer_id || wo.customerId`.

## Implementation Summary
Single-line fix in the WO load payload mapping. `loadWorkOrder()` was mapping `customerId` from `wo.customer_id || wo.customerId`, but the backend `getWorkOrderById()` returns the customer reference as `shop_client_id` (no `customer_id` column exists on `work_orders`). With `this.workOrder.customerId = null`, the `setTimeout` callback then called `populateCustomerDisplay()` and `checkCustomerCredit()` — both early-return on a null id, so the customer name input stayed blank, "Selected: …" line never rendered, and credit pre-check never fired.

The fix prepends `wo.shop_client_id || wo.shopClientId` to the `||` chain. The legacy `customer_id` / `customerId` fallbacks are kept in case any caller still emits them (defensive only — no current backend path does).

Note: line numbers in the original ticket (`work-order.component.ts:152`) have shifted to **161** on dev due to FN-1521 (tax-rule auto-apply) extending the `Object.assign` block. The fixed line is the same expression the ticket flagged.

Verified no other stale read sites:
- `submitWorkOrder()` (line 250) reads `this.workOrder.customerId` — fed by the now-correct load mapping, so save round-trip works without further changes.
- `basics-tab.component.ts` reads only `this.workOrder.customerId` (the in-memory normalized field) and already handles vehicle-side `shop_client_id ?? customer_id` correctly.
- `service-details-tab.component.ts:61` reads `this.workOrder.customerId` — same in-memory field.

## Files Changed
- `frontend/src/app/components/work-order/work-order.component.ts` (line 161 — extend `||` chain to read `shop_client_id` first)

## Key Decisions
- **Kept the `customer_id` / `customerId` fallbacks** rather than replacing them with only `shop_client_id`. They're harmless (no current backend code path emits them) and removing them would be a behavior change beyond the bug's scope.
- **Did not touch the save path.** The save payload at line 250 reads the in-memory `this.workOrder.customerId`, which becomes correct as soon as the load mapping is fixed. Backend `payload.customerId → shop_client_id` mapping in `work-orders.service.js` (lines 357, 434) is unaffected.
- **Did not change line numbers in the ticket description.** They referenced pre-FN-1521 dev. The expression matched verbatim.
