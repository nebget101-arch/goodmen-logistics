# FN-1739: Render MC logo on dispatch documents (settlements + driver docs)

## Requirement
Embed the operating-entity (MC) logo on generated dispatch documents: driver settlements, the employment application, and consent forms. The document's `operating_entity_id` resolves which logo to load from R2. Gracefully fall back to the existing text header when no logo is set.

## Acceptance Criteria
- [x] Settlement PDF (`settlement-pdf.service.js`) renders the operating entity's logo in the header (replacing the hardcoded "FleetNeuron" wordmark).
- [x] Employment application PDF (`pdf.service.js`) renders the MC logo in its header.
- [x] Consent PDF (`driver-onboarding-pdf.js`) renders the MC logo in its header.
- [x] Logo buffer loaded from R2 via `operating_entities.logo_storage_key` (resolved through the document's `operating_entity_id`).
- [x] No logo set → existing text header renders unchanged; logo fetch failure is non-fatal.

## Implementation Summary
A new shared helper `logo-embed.helper.js` fetches the operating-entity logo from R2 (`r2-storage.js` → `downloadBuffer`) and embeds it into a pdf-lib document, sniffing PNG/JPEG by magic bytes (falling back to the stored mime type). Every path is graceful: a missing `logo_storage_key`, an R2 download failure, or an unsupported image type all resolve to `null`, and each PDF builder falls back to its existing text header so the document still generates.

The three PDF builders embed the logo once per document and draw it in the header:
- **Settlement** (`settlement-pdf.service.js`): the logo replaces the "FleetNeuron" / "AI FLEET INTELLIGENCE" wordmark block in the per-page header band; the operating-entity company name/address on the right is unchanged. Embedded image is carried on the document state so every page header reuses it.
- **Employment application** (`pdf.service.js`): the logo is drawn at the left of the page header and the company name/address shift right; threaded through `drawPageHeader`/`newPageWithHeader`/`checkPage` so continuation pages also show it.
- **Consent** (`driver-onboarding-pdf.js`): the logo is drawn at the left of the consent header (closure-scoped, so all pages reuse it); company text shifts right.

The logo is resolved via the document's `operating_entity_id`. The calling routes that build the operating-entity / company object were extended to select `logo_storage_key` + `logo_mime_type` and pass them through (settlement already selected `*`, so no change there).

## Key Decisions
- **Graceful-by-default helper**: all failure modes (no key, R2 error, bad bytes, unsupported type) return `null` rather than throwing — the AC requires logo fetch failure to be non-fatal. Verified by smoke-testing both the no-logo and logo-present paths for all three builders.
- **Type sniffing over trusting mime**: PNG (`89 50 4E 47`) / JPEG (`FF D8 FF`) detected by magic bytes first, mime hint second. pdf-lib only supports PNG/JPEG (no WebP embed), so WebP logos fall back to the text header even though FN-1737 accepts WebP on upload.
- **Embed once, draw per page**: the embedded image is created a single time per document and reused across page headers (carried on document state / closure) to avoid re-embedding the same bytes on every page.
- **Namespace import of `r2-storage`** in the helper (not destructured) so `downloadBuffer` is resolvable/mockable at call time in unit tests.
- **No route signature changes**: the logo columns ride along on the existing operating-entity/company objects already passed to the builders, so no new parameters were threaded through route handlers.
