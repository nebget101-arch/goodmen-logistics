# FN-1359: Control Center KPI strip returns zeros — `/api/dashboard/stats?window=*` filters state-of-fleet KPIs by created_at

## Symptom
Control Center KPI strip ("Your Control Center") shows all zeros on dev despite real data in the same tenant:
- ACTIVE DRIVERS `0/0` with `−5` delta (smoking gun: previous window had 5 → drivers exist, current window excludes them)
- DQF COMPLIANCE `0%`, MEDICAL CERTS `0`, MAINTENANCE DUE `0`, VEHICLES OOS `0`, HOS VIOLATIONS `0`
- DQF page shows 6 drivers (5 active + 1 inactive), Rishawn Williams med cert expired 2026-04-29, dqf_completeness values present
- Action Queue (different endpoint, same page) correctly shows "5 vehicles overdue", "3 due soon", "Medical certificate expired", "19 loads late delivery risk"

## Root Cause
`backend/packages/goodmen-shared/routes/dashboard.js` → `buildStatGroups`:

1. `drivers` group uses `driversWindow = AND created_at >= $3 AND created_at < $4` → counts drivers *created* in window, not *existing*. Active/total drivers are point-in-time state.
2. `compliance` group uses `driversAsOf = created_at < $4`, which evaluates NULL for rows with NULL `created_at` and silently excludes them.
3. `vehicles` group has no SQL windowing yet still returns zero — needs RCA (status enum mismatch vs alerts path? silent `degradedGroups` failure?).

### FN-1360 (subtask) — implementation notes

**Branch:** `backend/FN-1360/dashboard-stats-windowing`

**What changed in `backend/packages/goodmen-shared/routes/dashboard.js`:**

1. `drivers` group now filters by `AND ${driversAsOf}` (NULL-safe `(created_at IS NULL OR created_at < $4)` when windowed; `TRUE` when unwindowed) instead of `AND created_at >= $3 AND created_at < $4`. The `driversWindow` fragment was removed entirely. Drivers are STATE, not activity — they must reflect "who exists at windowEnd," not "who was created in this window."
2. `compliance` group's `driversAsOf` predicate is now NULL-safe so legacy/seed rows with `created_at IS NULL` are no longer silently excluded.
3. `vehicles` group `activeVehicles` and `oosVehicles` filters now normalize the status enum: `UPPER(REPLACE(COALESCE(status::text, ''), '-', '_'))` and accept `'IN_SERVICE'` for active and `'OUT_OF_SERVICE' / 'OOS'` for OOS. This matches the tolerant check the `/api/dashboard/alerts` route already uses (`vehicle.status === 'out-of-service' || vehicle.status === 'OOS'`). `vehiclesNeedingMaintenance` filter additionally guards `next_pm_due IS NOT NULL`.
4. Windowing-contract comment added at the top of `buildStatGroups` listing every state KPI vs activity KPI so the next contributor doesn't reintroduce the same regression.
5. Telemetry: `dtLogger.sendMetric('custom.dashboard.stats.degraded', degradedGroups.length)` is now emitted on both the legacy and windowed paths.

**Vehicles-zero RCA conclusion:** the silent failure was a status-enum mismatch, not a `degradedGroups` failure. The `vehicles` group succeeded on dev but returned `0` for both `activeVehicles` and `oosVehicles` because the stored status values didn't match the kebab-case literal in the FILTER clause. The new tolerant predicate matches whatever encoding the data uses. Telemetry on degraded-count is added defensively in case future schema drift produces actual group failures.

**What did NOT change:**
- `services/window-stats.js` — no helper extraction was needed; all fixes live in the route.
- `loads`, `billing`, `hos` groups — these are activity KPIs and are correctly windowed by `created_at` / `record_date`.
- The legacy no-window response shape — backwards compat preserved.

**Tests added (`routes/dashboard.test.js`, FN-1360 describe block):**
- Drivers SQL has no `created_at >= $3` and uses the NULL-safe predicate.
- Compliance SQL is NULL-safe.
- Vehicles SQL uses `UPPER(REPLACE(...))` and matches `'IN_SERVICE'` / `'OUT_OF_SERVICE'` / `'OOS'`.
- `?window=7d` and `?window=30d` return identical `current.activeDrivers` when the underlying fleet is stable across both windows (state stability).
- Legacy no-window drivers query has no `$3`/`$4` references.

All 29 tests pass under `node:test` (24 existing + 5 new).
