# FN-719: Story 6 — Auto-Generate Invoice on Completion

## Requirement
When a work order transitions to COMPLETED status, automatically create a DRAFT invoice with all line items (labor, parts, fees). Show a toast with the invoice number. The invoice should appear inline in the Financials tab so the user can review before sending. Non-billable (Internal) work orders should skip invoice generation and show a badge instead.

## Acceptance Criteria
- [x] Transitioning WO to COMPLETED auto-creates DRAFT invoice server-side
- [x] Toast notification confirms creation with invoice number
- [x] Financials tab shows invoice inline with all line items
- [x] "Send Invoice" button works (DRAFT → SENT via existing invoice status endpoint)
- [x] Customer Credit toggle applies credit to balance with correct math (existing `generateInvoiceForWorkOrder` handles this)
- [x] Non-billable work orders do NOT generate invoices ("Internal Work Order — No Invoice" badge shown)
- [x] Backend `work-orders-hub.js` `PUT /:id` and `PATCH /:id/status` hooks trigger invoice creation
- [x] Invoice service `generateInvoiceForWorkOrder()` used (already handles idempotency — returns existing invoice if one exists)
- [x] Idempotent: calling twice returns existing invoice, no duplicate created

## Implementation Summary

### Backend — `work-orders-hub.js`

Two route handlers modified — both now call `generateInvoiceForWorkOrder()` (non-blocking) when the work order reaches COMPLETED status:

1. **`PUT /:id`** — full update route: checks `workOrder.status === 'completed'` after save
2. **`PATCH /:id/status`** — status-only route: checks `nextStatus === 'COMPLETED'`

Both routes:
- Skip invoice generation if `cost_type === 'internal'` (non-billable)
- Use `try/catch` so invoice failure never blocks the work order update
- Add `invoice: autoInvoice` to the response payload (null if skipped)
- Use the existing `workOrdersService.generateInvoiceForWorkOrder()` which is already idempotent

## Key Decisions

1. **Non-blocking hook**: Auto-invoice failures (e.g., customer not set) are caught and logged with `dtLogger.warn` — the WO status update always succeeds regardless.
2. **Idempotency via existing service**: `generateInvoiceForWorkOrder()` already returns the existing invoice if one exists (no `regenerate` flag), so re-completing doesn't duplicate.
3. **Send = SENT status**: The existing `setInvoiceStatus()` backend function handles `SENT` status with validation (requires `issued_date` and `due_date`). If those aren't set, the Send will fail with a user-visible error.
4. **Removed ApiService from financials-tab**: The component was previously injecting ApiService but never calling it directly. All invoice calls now flow through the parent component.
