# FN-1755: Story A — Geofence alert recipients + email/in-app dispatch + broker updates

## Requirement
Make geofences actually alert people. Phase 1 (Epic FN-1652) computes enter/exit/dwell `geofence_events` but `geofence-event-worker.js` **does not gate crossings on `geofence_triggers`** — only load-status automation runs. The `notify` action is dead and there is no recipient model. This story adds a recipient model + dispatch engine: when a unit crosses a geofence with a `notify` trigger, the configured recipients (internal users, external emails, brokers — inside and outside the org) get an **email (SendGrid) + in-app** notification. Broker emails carry load context.

Covers three of the user's asks: email when a unit is outside a fence (exit event), broker email updates, and a clean way to notify specific people in/out of the org.

## Acceptance Criteria
- [x] `geofence_trigger_recipients` table (trigger_id, recipient_type user|email|broker, user_id/email/broker_id, channel email|in_app|both) — FN-1757
- [ ] Worker consumes `geofence_triggers` on each event; `action='notify'` + matching `event_kind` (+ per-vehicle scoping) → resolve recipients → dispatch. Load-status automation NOT regressed.
- [ ] Dispatch via `notification-service.js`: SendGrid email + in-app `user_notifications` (`type='geofence_alert'`). No SMS.
- [ ] Broker recipients get a load-context email (load #, stop, status) keyed off the event's `load_id`.
- [ ] Idempotent: at most one dispatch per recipient per `geofence_event`.
- [ ] Recipient CRUD under `/api/geofences` trigger endpoints; payload shape documented below.

## Implementation Summary

### FN-1757 (subtask) — database — Done
Added the recipient model migration `20260612130000_create_geofence_trigger_recipients.js`, following the Phase 1 geofence migration conventions (`20260603100100_create_geofence_triggers.js`):

- **Table `geofence_trigger_recipients`**: `id` UUID PK (`uuid_generate_v4()`), `trigger_id` UUID NOT NULL FK→`geofence_triggers.id` ON DELETE CASCADE, `recipient_type` TEXT, `user_id` UUID NULL FK→`users.id` (CASCADE), `email` TEXT NULL, `broker_id` UUID NULL FK→`brokers.id` (CASCADE), `channel` TEXT NOT NULL DEFAULT `'both'`, `created_at` TIMESTAMPTZ DEFAULT now().
- **Conditional FK wiring**: `user_id`/`broker_id` FKs are wired only when the referenced table is present (`hasTable` guard), mirroring the `address_id`/`locations` pattern in `20260603100000_create_geofences.js`. Idempotent `hasTable('geofence_trigger_recipients')` guard wraps the whole `up`.
- **Named CHECK constraints**:
  - `chk_geofence_trigger_recipients_type` — `recipient_type IN ('user','email','broker')`
  - `chk_geofence_trigger_recipients_channel` — `channel IN ('email','in_app','both')`
  - `chk_geofence_trigger_recipients_target` — exactly one of (`user_id`,`email`,`broker_id`) is set **and** it is consistent with `recipient_type` (a `user` row carries only `user_id`, etc.). This is the integrity net the FN-1758 dispatch resolver relies on.
- **Index** `idx_geofence_trigger_recipients_trigger_id` on `(trigger_id)` (`CREATE INDEX IF NOT EXISTS`) — recipients are always loaded per-trigger.

**Verification** (scratch Postgres DB, prerequisite tables stubbed): `up` DDL applies clean; valid `user`/`email`/`broker` rows accepted; rows with a missing/mismatched/duplicate target rejected by `chk_..._target`; bad `recipient_type` and bad `channel` rejected; deleting a `geofence_trigger` CASCADE-deletes its recipients; `down` (`dropTableIfExists`) drops clean. `node --check` passes.

**Contract note for FN-1758/FN-1759**: the `recipients[]` payload in the API Contract above maps 1:1 to this table. The DB enforces the one-target-per-`recipient_type` rule, so the backend may rely on it but should still validate at the route layer for friendly 400s.
