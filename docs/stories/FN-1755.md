# FN-1755: Story A — Geofence alert recipients + email/in-app dispatch + broker updates

## Requirement
Make geofences actually alert people. Phase 1 (Epic FN-1652) computes enter/exit/dwell `geofence_events` but `geofence-event-worker.js` **does not gate crossings on `geofence_triggers`** — only load-status automation runs. The `notify` action is dead and there is no recipient model. This story adds a recipient model + dispatch engine: when a unit crosses a geofence with a `notify` trigger, the configured recipients (internal users, external emails, brokers — inside and outside the org) get an **email (SendGrid) + in-app** notification. Broker emails carry load context.

Covers three of the user's asks: email when a unit is outside a fence (exit event), broker email updates, and a clean way to notify specific people in/out of the org.

## API Contract (FINALIZED by FN-1758 — frontend builds against this)

Recipients are an embedded `recipients[]` array on a trigger (no separate sub-resource endpoint — the embedded array is the cleaner shape and covers create/replace in one round-trip). The shape is **camelCase**, consistent with the rest of the geofence wire API (`eventKind`, `vehicleId`, `dwellMinutes`). This supersedes the snake_case draft from intake.

**Recipient object (wire shape):**
```jsonc
{
  "recipientType": "user" | "email" | "broker",   // required
  "userId":  "<uuid>",      // required when recipientType = 'user'
  "email":   "ops@acme.com",// required when recipientType = 'email' (must be a valid email)
  "brokerId":"<uuid>",      // required when recipientType = 'broker'
  "channel": "email" | "in_app" | "both"          // optional, default 'both'
}
```
Exactly the identity field matching `recipientType` is honored; the others are stored null. GET responses include the recipient's `id`.

**Where it's accepted (all existing trigger endpoints — no new routes):**
- `POST /api/geofences` — `triggers[].recipients[]` on create.
- `PUT  /api/geofences/:id` — `triggers[].recipients[]`; when `triggers` is present it fully replaces the trigger set (and their recipients).
- `POST /api/geofences/:id/triggers` — `recipients[]` on the new trigger.
- `PUT  /api/geofences/:id/triggers/:triggerId` — `recipients[]` fully replaces that trigger's recipient set.

**Where it's returned:** every geofence/trigger read (`GET /api/geofences`, `GET /api/geofences/:id`, and the create/update responses) carries `triggers[].recipients[]`.

**Channel semantics:** `in_app` is only meaningful for `user` recipients (they have an in-app bell). `email` and `broker` recipients are email-only regardless of `channel`.

**Example — trigger create/update body:**
```jsonc
{
  "eventKind": "exit",
  "action": "notify",
  "vehicleId": null,                 // null = all units; else scope to one vehicle
  "recipients": [
    { "recipientType": "user",   "userId": "<uuid>",     "channel": "both"  },
    { "recipientType": "email",  "email": "ops@acme.com", "channel": "email" },
    { "recipientType": "broker", "brokerId": "<uuid>",    "channel": "email" }
  ]
}
```

## Implementation Summary

### FN-1757 (subtask) — database — Done
Added the recipient model migration `20260612130000_create_geofence_trigger_recipients.js`, following the Phase 1 geofence migration conventions (`20260603100100_create_geofence_triggers.js`):

- **Table `geofence_trigger_recipients`**: `id` UUID PK (`uuid_generate_v4()`), `trigger_id` UUID NOT NULL FK→`geofence_triggers.id` ON DELETE CASCADE, `recipient_type` TEXT, `user_id` UUID NULL FK→`users.id` (CASCADE), `email` TEXT NULL, `broker_id` UUID NULL FK→`brokers.id` (CASCADE), `channel` TEXT NOT NULL DEFAULT `'both'`, `created_at` TIMESTAMPTZ DEFAULT now().
- **Named CHECK constraints**: `chk_..._type` (`recipient_type IN ('user','email','broker')`), `chk_..._channel` (`channel IN ('email','in_app','both')`), and `chk_..._target` — exactly one of (`user_id`,`email`,`broker_id`) set **and** consistent with `recipient_type`. This is the integrity net the FN-1758 dispatch resolver relies on.
- **Index** `idx_geofence_trigger_recipients_trigger_id` on `(trigger_id)` — recipients are always loaded per-trigger.

### FN-1758 (subtask) — worker dispatch + recipient CRUD + templates
When the geofence-event worker writes a **new** `geofence_events` row, after load-status automation runs it now hands the event to a new **`geofence-notify-dispatcher.js`**:
1. Loads the geofence's `geofence_triggers` where `action='notify'` and `event_kind` matches the crossing, respecting per-vehicle scoping (`vehicle_id` NULL = all units, else must equal the event's vehicle).
2. Resolves each matching trigger's `geofence_trigger_recipients` (de-duplicating a recipient configured on multiple triggers).
3. Dispatches per recipient via the existing `notification-service.js`:
   - `user` → in-app `user_notifications` (`type='geofence_alert'`) + email to the user's account email, honoring `channel`.
   - `email` → SendGrid email only.
   - `broker` → SendGrid email only, using a **load-context template** (load #, status, and the relevant stop — delivery on exit, pickup otherwise) keyed off the event's `load_id` (stamped by load-status automation, which runs first).

Recipient CRUD was added to `geofence-service.js` (validation + persistence + wire mapping); it flows through the existing `/api/geofences` trigger endpoints unchanged, so `routes/geofences.js` needed no edits.

**Idempotency:** the worker only dispatches for a newly-inserted event — `insertEvent` returns null on the `(ping_id, geofence_id, event_kind)` unique conflict, so a reprocessed ping never re-dispatches. That event-level uniqueness is the at-most-once guard; no separate dispatch ledger was added.

**No regression:** dispatch runs after `loadStatusAutomation.applyForEvent` and is wrapped in try/catch in the worker, so any notification failure is logged and never blocks event writing or load-status transitions. A regression test asserts a load still transitions when a notify trigger is present.

### FN-1759 (subtask) — frontend recipient panel — Done
Added a per-trigger **Notify recipients** panel to the geofence editor (`geofences.component`), rendered only when `action === 'notify'`:
- **Add recipients** via a type selector (Internal user / External email / Broker): users and brokers from `app-ai-select` pickers (sourced from `/api/users` and `/api/brokers`), external emails via a validated text input (regex + dedupe).
- Recipients render **grouped "Inside the org" (users) vs "Outside the org" (emails + brokers)** — the "cleaner way to notify in/out of the org" ask.
- Per-recipient **channel** selector (`Email + in-app` / `Email only` / `In-app only`) for users; external email/broker rows show a static "Email" pill since they are email-only (matches the backend channel semantics above).
- Recipients persist nested in the existing trigger payload (`triggers[].recipients`) — consistent with the FINALIZED contract; the editor repopulates from `trigger.recipients` on read/edit. No new endpoint on the frontend side.
- `geofence.model.ts` gains `GeofenceRecipientType`, `GeofenceRecipientChannel`, `GeofenceTriggerRecipient`, and `recipients?` on `GeofenceTrigger`; `geofence.service.ts` gains `listUsers()` / `listBrokers()` projections for the pickers.
- **FN-317 compliance:** all `[options]` arrays are class fields (static `readonly` for type/channel; once-assigned fields for the async user/broker lists) — no getter-backed option arrays. Verified by spec.
- **AI dark-theme tokens only** — reused the existing component palette, no new hex.
- Verified with a Node 20 `ng build` (geofences lazy chunk compiles clean) and unit specs for add/validate/dedupe/group/serialize + option mapping.

## Key Decisions
- **Channels: email + in-app only (no SMS)** — intake 2026-06-12.
- **Reuse `notification-service.js`; do not revive dead `backend/emails/`.** No extension was needed — the dispatcher owns the templates and passes the `geofence_alert` type to the existing generic helpers.
- **Webhook trigger action remains out of scope.**
- **Dispatch lives in a dedicated `geofence-notify-dispatcher.js`**, not inline in the worker — keeps the worker focused and the dispatch logic unit-testable with an injected notifier.
- **Recipient wire shape is camelCase** (`recipientType`/`userId`/`brokerId`), overriding intake's snake_case draft, for consistency with the existing geofence wire contract.
- **Embedded `recipients[]` array, no separate sub-resource route** — one round-trip create/replace; simpler for the frontend.
- **Idempotency via the existing event-uniqueness net**, not a new dispatch-ledger table (would be out of this subtask's scope and unnecessary).
- **Reads degrade gracefully** if `geofence_trigger_recipients` is absent (pre-FN-1757), so the backend subtask is safe to land before the migration on the integration branch.
- **Frontend forces `channel: 'email'` for external email/broker recipients** (no in-app account) and defaults `user` recipients to `both` — aligned with the backend channel semantics.

The **frontend** (FN-1759) ships with the normal Angular build/deploy — geofences is a lazy-loaded module, no separate service.
- **Jobs/workers affected:** the `geofence-events` Bull worker (`createGeofenceEventWorker`) now dispatches notifications per new crossing. No new queue.
- **DB migrations:** **FN-1757**'s `20260612130000_create_geofence_trigger_recipients.js` (`knex migrate:latest`) — additive new table, already merged onto `integration/FN-1755`. Apply it before the FN-1758 code runs; until applied, recipient reads return empty and no alerts are sent (graceful). Rollback via `knex migrate:rollback` (`dropTableIfExists`) is safe once the FN-1758 service is rolled back first.
- **Env/config:** no new env vars. Email requires `SENDGRID_API_KEY` (+ `ONBOARDING_FROM_EMAIL` for the From address) — already used by notification-service. With no key, email sends no-op (sent:false) and in-app still works. No Twilio (no SMS).
- **Feature flags:** none.
- **Deploy order:** apply FN-1757 migration → deploy logistics-service (API + worker) → deploy frontend (FN-1759). Order between migration and backend is not strict (reads degrade gracefully), but alerts only flow once the migration is applied, the backend is live, and recipients are configured via the UI.
- **Post-deploy verification:** configure a geofence `notify` trigger with a user + email + broker recipient via the UI; drive a unit across the fence (or replay a ping); confirm an in-app bell + emails are produced and the broker email carries load context. Confirm load-status automation still transitions normally.
