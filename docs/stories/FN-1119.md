## Implementation Summary

### FN-1183 (subtask)
Added a pure `DrilldownService` that constructs nav targets (commands +
queryParams) from a report row or KPI card, an `@Injectable` shared by the
report-view (centered reports center) and the shared `app-report-table` /
`app-kpi-cards` components. Drill-downs resolve in priority order: load_id →
customer/broker → driver → revenue-by-dispatcher's dispatcher_id. When no
useful destination exists, the service returns null and components render
plain table rows / cards (no hover affordance, no anchor).

URL contracts are documented in `docs/reports/drilldown-contracts.md`,
including which params the receiving screens (loads-dashboard, dispatch-
drivers, customer-management) currently honor and which are documented for
future receiver-side enhancement.

Date range from the source report (`startDate`/`endDate`, plus legacy
`date_from`/`date_to`) carries into the navigation as `from`/`to`. The
`reports.models.ts` exports new types (`DrilldownTarget`,
`DrilldownDestination`, `RowDrilldownFn`, `CardDrilldownFn`).

To avoid change-detection thrash from getter-based bindings (FN-317 RCA),
`report-view` pre-resolves targets into `displayCards` / `displayRows`
arrays once per fetch — bindings see stable references for the lifetime of
the rendered data.

The shared `app-report-table` and `app-kpi-cards` accept new optional
`@Input()` callbacks (`rowDrilldown`, `cardDrilldown`) so any caller — not
only report-view — can opt into drill-down without code duplication.
Existing callers (legacy `reports-page`) leave them unset and continue to
render plain.

`RouterModule` was added to `ReportsModule` imports so the new `routerLink`
bindings resolve. `DrilldownService` is `providedIn: 'root'`.

## Key Decisions

1. **DrilldownService is pure & central**: a single source of truth that
   inspects rows/cards and produces navigation targets. Components render
   anchors when a target is non-null and plain elements otherwise. Adding a
   new report → just extend the service (no template churn).

2. **Pre-resolve targets per fetch (`displayCards`, `displayRows`)**: getter-
   based bindings (`*ngIf="cardDrilldown(card) as target"`) work but
   re-evaluate per CD cycle, returning new array/object references and
   forcing routerLink to recompute. Pre-resolving once per fetch keeps
   bindings stable. Lesson from FN-317.

3. **Receiver-side gaps documented, not blocked**: loads-dashboard currently
   only reads `status`/`billingStatus`/`loadId`. The contract defines
   `from`/`to`/`driverId`/`dispatcherId`/`customerId` so future loads-
   dashboard work has a target to aim at. Drill-downs still navigate today;
   filter pre-population for the listed params is a follow-up.

4. **Shared component opt-in**: `rowDrilldown` / `cardDrilldown` inputs
   default to null. Legacy `reports-page` (which uses `app-report-table` +
   `app-kpi-cards`) is unchanged — rows/cards render plain.
