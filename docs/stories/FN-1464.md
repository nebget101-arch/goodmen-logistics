# FN-1464: POST /api/vehicles/customer fails — `column "customer_id" of relation "customer_vehicles" does not exist`

## Requirement
Fix the production regression where adding a vehicle for a customer on the Work Orders page returns HTTP 500. Root cause: migration `20260317093000_rename_customers_to_shop_clients.js` renamed `customer_vehicles.customer_id` → `shop_client_id` but `backend/packages/goodmen-shared/routes/vehicles.js` and `frontend/.../basics-tab.component.ts` were not updated.

## Acceptance Criteria
- [ ] POST `/api/vehicles/customer` with either `customer_id` or `shop_client_id` in the body succeeds and persists the vehicle linked to the specified shop client
- [ ] PUT `/api/vehicles/:id` updates of `shop_client_id` on `customer_vehicles` rows succeed
- [ ] Work Orders Basics tab — adding a vehicle for a selected customer:
  - New vehicle appears in the customer-scoped vehicle list immediately
  - Re-loading the work order shows the vehicle linked to the customer
- [ ] No console errors or 500s in the e2e flow
- [ ] Existing customer-vehicle linkage continues to render correctly (no regression on stored rows)

## API Contract (canonical, to allow parallel work)
- **Field name on the wire**: `shop_client_id` (canonical). Backend MUST also accept `customer_id` as an alias on incoming JSON during the transition.
- **POST `/api/vehicles/customer`** request body includes `shop_client_id` (or alias `customer_id`); response row exposes `shop_client_id`.
- **GET / readback** (`SELECT * FROM customer_vehicles WHERE vehicle_uuid = $1` and the `all_vehicles` view) — already returns `shop_client_id` post-rename. No backend change needed for reads.
- Frontend reads SHOULD prefer `vehicle.shop_client_id ?? vehicle.customer_id` to tolerate any cached/stale responses.

## Implementation Summary

### FN-1465 (subtask)
Migrated the Work Orders Basics tab vehicle owner field from `customer_id` to `shop_client_id`. All reads use `vehicle.shop_client_id ?? vehicle.customer_id` to tolerate cached/old responses; the create-vehicle write now sends `payload.shop_client_id` (backend accepts both names during the transition per the API contract).

Changes in `frontend/src/app/components/work-order/tabs/basics-tab/basics-tab.component.ts`:
- `onVehicleSelect()` (~line 320): customer-link branch resolves owner via `shop_client_id ?? customer_id` and uses the resolved id for both `workOrder.customerId` assignment and customer lookup.
- `createVehicleFromWorkOrder()` (~line 468): outgoing payload now sets `shop_client_id` instead of `customer_id`.
- `getVehicleSearchPool()` (~line 513): customer-scoped vehicle filter prefers `v.shop_client_id`, falls back to `v.customer_id`/`v.customerId`.
- `applyCustomerVehicleFilter()` (~line 537): mismatch comparison reads `selectedVehicle.shop_client_id ?? selectedVehicle.customer_id`.

`frontend/src/app/services/api.service.ts` was not touched — vehicle responses are typed as `any` and have no `customer_id` field declaration to update.
