### FN-800 (subtask)
Added Claude-powered NLQ handler in `ai-service` that converts free-text queries (e.g. "Smith's pending loads over $1000") into a whitelisted filter object the loads list can consume.

- Route: `POST /api/ai/loads/nlq` — handled by `handleLoadsNlq`
- Model: `process.env.ANTHROPIC_NLQ_MODEL || 'claude-haiku-4-5-20251001'` (temperature 0, max_tokens 512)
- Validation drops unknown/invalid fields silently; if the result is empty or the model output is unparseable, returns `{ success: true, fallback: true }` so the backend can fall back to keyword search
- Upstream Claude errors (timeouts, 5xx, rate limits) also return `{ fallback: true }` with HTTP 200 and `meta.reason: 'ai_upstream_error'` — preserves graceful degradation
- Input validation: `query` must be a non-empty string → else HTTP 400 `AI_BAD_REQUEST`
- Query is trimmed and truncated to 500 chars before being sent to the model

**ALLOWED_FILTERS** (exported for reuse by FN-801): `driver_name`, `broker_name`, `load_number`, `billing_status` (enum), `delivery_status` (enum), `pickup_state` / `delivery_state` (2-letter codes), `pickup_city`, `delivery_city`, `rate_min`, `rate_max`, `date_from`, `date_to` (strict ISO YYYY-MM-DD).

Test file `src/handlers/__tests__/loads-nlq-handler.test.js` uses a mocked Anthropic client and exercises 10 assertions: 5 sample queries from the AC, unknown-fields filtered, unparseable-output fallback, missing/non-string `query` 400s, and bad-value normalisation. Runs via `npm test` — no external deps, uses `node:assert` only.

## Key Decisions
### FN-800
- **Haiku 4.5 over Sonnet**: filter extraction is a constrained structured-output task; Haiku 4.5 is ~5× cheaper and fast enough. Overridable via `ANTHROPIC_NLQ_MODEL`.
- **Fallback on upstream error returns 200, not 502**: the whole point of the fallback signal is graceful degradation; returning 5xx would force the caller to build separate retry logic for the common case (rate limits, transient 5xx).
- **Empty filters → fallback**: a parseable-but-empty `{}` model response counts as "no filters extracted" and falls through to keyword search. `meta.reason` distinguishes `no_filters_extracted` vs `unparseable_model_output` vs `ai_upstream_error` for observability.
- **`deps.anthropic` injection supported**: handler honours `deps.anthropic` when present so unit tests can mock without monkey-patching module state. Production paths unchanged — lazy-init from env still wins when `deps.anthropic` is absent.
- **Strict ISO-date validation**: regex + `new Date().toISOString()` round-trip rejects impossible dates like `2025-13-40`.
