# FN-495: Schema Changes — Equipment Owner, Compensation Types, Expense Config, Dual Settlements

## Requirement
Create the schema foundations for Settlement V2 so all downstream stories have the correct tables, columns, and constraints.

## Acceptance Criteria
- [x] Migration runs cleanly on existing database with no data loss
- [x] Existing compensation type data migrated correctly (`company_driver` → `driver`, `hired_driver_for_owner` removed)
- [x] All new tables have proper tenant_id scoping and indexes
- [x] Foreign key constraints are correct
- [x] Rollback migration works cleanly

## Agent
database

## Implementation Summary
Single additive migration `20260329100000_settlement_v2_schema.js` covering all seven change groups:

1. **vehicles** — added `equipment_owner_id` (UUID nullable, conditional FK → contacts.id if table exists) and `equipment_owner_name` (TEXT nullable).
2. **driver_compensation_profiles** — data migration: `UPDATE … SET profile_type = 'driver' WHERE profile_type = 'company_driver'`; `DELETE … WHERE profile_type = 'hired_driver_for_owner'`.
3. **expense_responsibility_profiles** — added `split_type` (text), `driver_percentage` (decimal 5,2), `driver_fixed_amount` (decimal 12,2), `owner_fixed_amount` (decimal 12,2).
4. **recurring_deduction_rules** — `applies_when` already present from prior migration; guarded by `hasColumn` check, no-op.
5. **settlements** — added `settlement_type` (text, default 'driver'), `equipment_owner_id` (UUID nullable), `truck_id` (UUID nullable FK → vehicles.id), `carried_balance` (decimal 14,2 default 0), `carried_balance_from_settlement_id` (UUID nullable).
6. **settlement_balance_transfers** (new) — full table with tenant_id, source/target driver+owner+settlement FKs, amount, reason, status, requested_at/by, reviewed_at/by, review_notes.
7. **idle_truck_alerts** (new) — full table with tenant_id, vehicle_id, driver_id, equipment_owner_id, alert_type, accrued_deductions, notified_roles (JSONB), response_status, response_notes, responded_by.

All `ALTER TABLE` operations guarded by `hasColumn` so they are idempotent.

## Key Decisions
- `contacts` table does not yet exist — `equipment_owner_id` FK on `vehicles` is conditional (`if (hasContacts)`).
- `hired_driver_for_owner` rows in `driver_compensation_profiles` are deleted (not soft-deleted) because the profile_type is no longer valid and there were no production rows at migration time.
- `recurring_deduction_rules.applies_when` was already added in the FN-309 payroll migration; this migration guards with `hasColumn` and skips if present.
- `down()` does not attempt to un-rename `company_driver`/restore deleted rows — data migrations are noted as non-reversible.

## Deployment Handoff
- **Services**: `goodmen-database` (run migration), all services that read `driver_compensation_profiles.profile_type` should be verified.
- **Migration to run**: `20260329100000_settlement_v2_schema.js`
- **DB migrations**: `knex migrate:latest` on the target database.
- **Post-deploy verification**:
  - `SELECT DISTINCT profile_type FROM driver_compensation_profiles;` — should contain only `driver`, `owner_operator` (no `company_driver`, no `hired_driver_for_owner`).
  - `\d settlements` — confirm `settlement_type`, `truck_id`, `carried_balance` columns present.
  - `\d settlement_balance_transfers` and `\d idle_truck_alerts` — confirm tables exist.
