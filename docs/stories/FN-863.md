### FN-875 (subtask)
- New standalone component `<app-load-wizard-basics>` renders all 8 fields against the parent wizard's `basics` FormGroup.
- Status dropdown limited to `DRAFT` / `NEW` in create mode (per AC); full status list in edit / view / ai-extract.
- Billing-status dropdown mirrors the legacy manual form (`PENDING`, `BOL_RECEIVED`, `INVOICED`, `SENT_TO_FACTORING`, `FUNDED`, `PAID`, `CANCELLED`).
- Broker combo backed by `LoadsService.getBrokers()`; client-side filter; selecting writes `brokerId` on the FormGroup. The `+ Create new broker` link opens an inline sub-modal stacked **above** the wizard (`z-index: 1300` over the wizard's `1100`); on POST success the new broker is auto-selected and the modal closes. Wizard form state is preserved (the modal lives inside the same component tree).
- Rate field gets `Validators.required` and a custom non-negative numeric validator added on init; `formatRate()` rounds to 2 decimals on blur.
- Load number gets `Validators.required` added on init and is auto-prefilled (`LD-YYMMDD-####`) on create when blank, mirroring legacy UX so the form is valid out-of-the-box and `canProceed` flips immediately.
- Dispatcher input is read-only and prefilled from `localStorage.fn_user`; `dispatcherId` on the FormGroup is set to the current user's id (matches the legacy "current user is dispatcher" pattern — the only existing dispatcher source).
- `mode='view'` → `basics.disable()` so every control is read-only.
- `<app-load-wizard-v2>`'s basics step now renders the new sub-component; the parent's `canProceed` already gates on `basics.valid` so no parent change was needed beyond the slot replacement.

## Implementation Summary
FN-875 delivers the full Step 1 body for the new load wizard. The sub-component owns no form state — it renders controls against the parent `basics` FormGroup exposed by `LoadWizardComponent` — and layers on extra Validators (rate ≥ 0, load number required) so `canProceed` in the shell footer reflects real basics rules. The inline broker-create sub-modal uses the existing `POST /api/brokers` endpoint; on success the newly-created broker is prepended to the local combo list and auto-selected, with the wizard's form values left untouched.

## Key Decisions
- **Dispatcher source**: AC asks for the "existing dispatcher list source," but the legacy form treats Dispatcher as the current logged-in user only — there is no list endpoint or service. I kept that pattern and rendered Dispatcher as a read-only display tied to the current user (id stored on `dispatcherId`). If the team later builds a real dispatcher list, swap this for a combo without touching the wizard parent.
- **Load number prefill**: the AC table doesn't require it, but adding `Validators.required` on `loadNumber` would otherwise leave the create-mode form invalid out-of-the-box. Auto-generating an `LD-YYMMDD-####` placeholder (legacy behavior) keeps `canProceed` flipping to true as soon as the user lands on Step 1 — the user is free to edit it.
- **Sub-modal stacking**: the broker create modal is rendered inside the basics component (not portalled) because the wizard form lives there; this preserves wizard state without ngTemplate gymnastics. The modal's `z-index: 1300` puts it above the wizard's `1100`.
- **No getter `[options]` bindings**: per the FN-317 lesson, all option lists (`statusOptions`, `billingStatusOptions`, `filteredBrokers`) are plain arrays mutated in component methods, not getters.
