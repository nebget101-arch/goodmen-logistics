# FN-1703: Story B — Session-aware auth + login conflict/takeover + logout

## Requirement
Make auth session-aware and add the conflict/takeover/logout flow. Core of the Epic and **cross-cutting** — the auth-middleware change affects every service. API shapes are defined below so Story D (FN-1705) builds in parallel.

## Acceptance Criteria
- [ ] Login records a `user_sessions` row (captures User-Agent + IP, derives `device_label`); JWT carries a `sid` claim.
- [ ] Conflict detection: existing non-revoked, non-expired session → `409` conflict payload (no token), high-level details only.
- [ ] Takeover revokes the existing session (`revoked_reason='takeover'`) and issues the new token; exactly one active session remains.
- [ ] Session-aware `auth-middleware.js`: validate `sid` present + `revoked_at IS NULL` + not expired, else `401 { reason: 'session_revoked' }`; short-TTL read-through cache (mirror `plan-access-middleware`); throttled `last_seen_at`.
- [ ] `POST /api/auth/logout` revokes the current session (`revoked_reason='logout'`).
- [ ] Works across gateway + microservices; no auth bypass; no info leak.

## API contract (define up front — frontend builds against this)
> Implementing backend agent: finalize and keep authoritative.
- `POST /api/auth/login` `{ username, password }` → success `{ token, role, username, ... }` OR conflict `409 { conflict: true, existingSession: { deviceLabel, browser, os, ipMasked, lastSeenAt } }`
- Takeover: `POST /api/auth/login` `{ username, password, takeover: true }` (or `POST /api/auth/sessions/takeover`) → `{ token, ... }`
- `POST /api/auth/logout` → `{ ok: true }`
- Revoked-session API responses → `401 { reason: 'session_revoked' }`
