# FN-1368: Parts catalog: expand row shows "locationId query parameter is required"

## Requirement

On the Parts Catalog page, expanding a part row (chevron ▶) is supposed to show that part's stock breakdown across every location ("SHOP-A (A-3): 5 units, WAREHOUSE-1 (W-27): 20 units"). Today it instead renders the inline error `locationId query parameter is required`.

The frontend service `getInventoryByPart()` (added in FN-708) calls `GET /api/inventory?partId=X` without `locationId`, but that endpoint requires `locationId` and was never intended to support cross-location queries. Add a new backend route `GET /api/inventory/by-part/:partId` that returns rows for one part across all locations, and point the frontend at it.

## Acceptance Criteria

- [ ] New route `GET /api/inventory/by-part/:partId` returns `{ success: true, data: [...] }` with one row per (location, bin) carrying the part. Empty array (200) when none.
- [ ] Each row exposes `location_id`, `location_name`, `on_hand_qty`, `reserved_qty`, `available_qty`, `bin_location` (and bin id/code/name if schema supports), matching what `formatStockLine()` consumes.
- [ ] OpenAPI doc block added for the new route.
- [ ] Existing `GET /api/inventory?locationId=X` route is unchanged (still used by inventory list view).
- [ ] Frontend `getInventoryByPart()` calls the new route.
- [ ] Expanding any part row in `/parts` shows the per-location breakdown with no error.
- [ ] Empty-stock part shows a friendly empty state, not an error.

## Implementation Summary

### FN-1371 (subtask)

Added `GET /api/inventory/by-part/:partId` to `backend/packages/goodmen-shared/routes/inventory.js`. The handler joins `inventory` with `locations` (for `location_name`) and left-joins `location_bins` on `inventory.bin_id` (for `bin_id`/`bin_code`/`bin_name`), filters by `inventory.part_id = :partId`, and returns one row per inventory record across all locations. Available qty is computed as `on_hand_qty - reserved_qty`. Empty result returns `{ success: true, data: [] }` (200, not 404). The route is mounted before `router.put('/:id', ...)` so future `GET /:id` additions won't shadow it. OpenAPI JSDoc block added.

Tenant filtering matches the existing `GET /` route (no explicit filter — both rely on parent-resource access control via `authMiddleware` + service-level checks). The existing `GET /api/inventory?locationId=X` is untouched.

### FN-1373 (subtask)

Single-line change in `frontend/src/app/services/api.service.ts` at `getInventoryByPart()`: swap the URL from query-param form `${baseUrl}/inventory?partId=${id}` to path-param form `${baseUrl}/inventory/by-part/${id}` (matching the route added in FN-1371). Kept `encodeURIComponent(partId)` so any special chars in the id are preserved through the path segment. No other call sites of `getInventoryByPart()` exist in the repo, and no template/component changes were needed — `parts-catalog.component.ts` already consumes the response via `formatStockLine()`.

## Files Changed

## Key Decisions

- **Left-join `location_bins`** instead of inner-join because `inventory.bin_id` is nullable (legacy rows may only have free-text `bin_location`). The response carries both the FK-based fields and the legacy `bin_location` text so the frontend can render whichever is populated.
- **No service extraction** — the query is a single `select` and only used by this route; inlining keeps it consistent with `GET /` style.
- **No `parts` join** — caller already has the part. Joining would just add columns the consumer (`formatStockLine`) doesn't use.
- **Order by `locations.name`** to give the breakdown a stable, predictable display order.
