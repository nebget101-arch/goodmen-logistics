# FN-1305: Gateway crash loop — `/api/insights/trends` pulls knex into gateway

## Defect
After PR #651 (FN-1126 / FN-1152) merged to `dev` at 2026-05-04T16:37:47Z, `fleetneuron-logistics-gateway` enters a crash loop on Render boot:

```
Error: Cannot find module 'knex'
  at requireFromRoot (backend/packages/goodmen-shared/internal/require-from-root.js:8:12)
  at Object.<anonymous> (backend/packages/goodmen-shared/config/knex.js:2:14)
  at Object.<anonymous> (backend/gateway/index.js)
```

## Root Cause
FN-1152 (commit `660ca5c`) added `const trendKnex = require('../packages/goodmen-shared/config/knex')` to `backend/gateway/index.js:541`. The gateway has never declared `knex`/`pg` — it's a thin Express proxy. `requireFromRoot('knex')` finds nothing in the gateway's `node_modules` and the boot fails. Sibling Control Center features (`briefing-aggregator.js`, `ask-forwarder.js`) do this correctly: they take a `fetcher` + microservice URLs.

## Key Decisions

- **Cache lives in reporting-service, not the gateway.** The bug originated from putting DB-touching code in the gateway. Keeping the cache co-located with the aggregator (which is the only thing that fills it) avoids a second cache layer at the proxy and keeps the gateway concerns purely transport. A proxy cache would require duplicate invalidation paths (`?refresh=true` would have to invalidate two caches) and risks serving stale data when only one layer is busted. The downside — every gateway request crosses the network to reporting-service — is fine: reporting-service is the same Render network, latency is single-digit ms, and the 10-min TTL absorbs traffic spikes per tenant.
- **No JWT-decoding change in the route.** Kept `buildInsightsRouter` verbatim (own JWT verify) instead of switching to the shared `authMiddleware` + `tenantContextMiddleware` chain that other reporting-service routes use. Reasons: (1) FN-1305 is a hot-fix bug, scope creep is risky; (2) the existing test suite verifies the JWT decode path exactly; switching middleware would force test rewrites without any wire-shape benefit; (3) shared `tenantContextMiddleware` resolves tenant via DB lookups, which would be a behavioral change vs. the FN-1152 contract that trusts the JWT `tenant_id` claim. A future cleanup can unify after this lands.
- **Did not add a gateway boot smoke test in code.** A pre-deploy boot check is best owned by CI / Render's own deploy probe rather than a unit test (which can't reproduce a require-tree resolution failure inside `node_modules`-isolated unit harnesses without a special install). Documented the local boot smoke as a runbook step instead — see Deployment Handoff. FN-1307 will run the same probe via Render-deployed gateway.

**Env vars**: `REPORTING_SERVICE_URL` must be set on the gateway. It's almost certainly already configured (the gateway has been calling `/api/dashboard`, `/api/reports`, `/api/audit` against it since the microservice split). **No cache to invalidate.** The trend cache is in-memory per-process; restarts implicitly clear it. Both services restart on this deploy, so all caches reset.

**No feature flags.** `/api/insights/trends` was added in FN-1152 (already on `dev`); this just relocates the implementation.
