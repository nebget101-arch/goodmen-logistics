# FN-1573: [Locations Admin] Users tab shows empty state — frontend reads `res.data` but API returns a bare array (also wrong field shape)

## Requirement
The Edit Location modal's **Users** tab renders the empty state on dev even when the API returns rows. Two distinct frontend↔backend contract mismatches:

1. **Envelope** — backend `user-locations.js:80` returns `res.json(result.rows)` (a bare array); frontend `api.service.ts` typed `getLocationUsers()` as `Observable<{ data: LocationUserRecord[] }>` and the component reads `res?.data ?? []` → always falls back to `[]`.
2. **Field shape** — the SELECT in `user-locations.js:71` projects `{id, first_name, last_name, email, role, assigned_at}`; the frontend `LocationUserRecord` interface declares `user_id` and `username`, and the template + trackBy + remove handler reference fields that don't exist on the response.

Backend is the source of truth (matches sibling endpoints like bins). Fix is frontend-only.

## Implementation Summary
Aligned the `LocationUserRecord` TypeScript interface and its consumers with the actual `GET /api/locations/:id/users` payload (a bare array of `{id, first_name, last_name, email, role, assigned_at}`).

The frontend was previously expecting an envelope (`{ data: [...] }`) and additional fields (`user_id`, `username`) that the backend never returns. Result: the Users tab always rendered the empty state and any row-level action (`trackBy`, Remove confirm) referenced `undefined`.

### FN-1574 (subtask)
**Service** (`api.service.ts`)
- Removed `user_id` and `username` from `LocationUserRecord`.
- `getLocationUsers()` now returns `Observable<LocationUserRecord[]>` (bare array).
- `assignLocationUsers()` typed as `Observable<unknown>` — the consumer doesn't read the body, and the actual backend response (`{assigned, already_assigned, invalid_user_ids, total_requested}`) is irrelevant to the caller.

**Component** (`users-tab.component.ts`)
- `loadAssignedUsers().next` assigns `res ?? []` (no more `.data` peeling).
- `formatName()` falls back to `email` then `'—'` (the `username` fallback no longer exists in the model).
- `trackByUserId()` returns `user.id`.
- `filterSuggestions()` builds the assigned-id set from `u.id`.

**Template** (`users-tab.component.html`)
- Replaced every `user.user_id` reference with `user.id` (3 occurrences: confirm-state guard + Remove + executeRemove).
- Removed `<span class="username-sub">@{{ user.username }}</span>` row — the API doesn't return `username`.

**Styles** (`users-tab.component.css`)
- Removed the orphaned `.username-sub` rule.

## Key Decisions
- **Typed `assignLocationUsers()` as `Observable<unknown>`** rather than the actual `{assigned, already_assigned, invalid_user_ids, total_requested}` shape. The component already ignores the body and just calls `loadAssignedUsers()` on success; no need to bake a body shape into the public service signature.
- **`formatName()` fallback chain `full → email → '—'`** instead of just `email`. Guards against the (very unlikely) case where the backend ever returns a row with no name and no email.
- **Removed orphaned `.username-sub` CSS** since the markup that referenced it is gone — keeps the stylesheet honest.
