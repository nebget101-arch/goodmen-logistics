# FN-1567: Print Label PDF for new part includes full UI screenshot (chrome not hidden)

## Requirement
After adding a part in the Parts Catalog and clicking **Print label**, the generated PDF (Chrome → Save as PDF from `window.print()`) currently contains both the intended 4×2" barcode label and a full screenshot of the parts catalog UI (sidebar, filters, parts table, footer). The label sits in the upper-left while the rest of the page bleeds through behind it.

This is a regression of FN-1398 acceptance criterion: "`window.print()` produces a clean label with no app chrome (sidebar/header/buttons hidden via `@media print`)."

## Acceptance Criteria
- [ ] Saving the Print-label dialog as PDF on dev shows ONLY the 4×2" label — no sidebar, no parts table, no filters, no footer.
- [ ] Both entry points work: (a) **Print label** in success modal after creating a new part; (b) **Print label** row action on existing parts.
- [ ] No regression in normal screen rendering — the parts catalog UI is unchanged for non-print viewing.
- [ ] Code-128 and QR SVGs render in the printed PDF (regression check on FN-1399).

## Root Cause
`frontend/src/app/components/parts-catalog/parts-catalog.component.css:1120-1145` declares:

```css
@media print {
  body * { visibility: hidden; }
  .label-print-area, .label-print-area * { visibility: visible; }
  .label-print-area { position: absolute; left: 0; top: 0; ... }
}
```

This pattern works only with global CSS. The parts-catalog component uses Angular's default `ViewEncapsulation.Emulated`, which rewrites `body *` to a scoped attribute selector (e.g. `body[_ngcontent-xxx] *`). That selector matches nothing at runtime, so the `visibility: hidden` rule is silently dropped — every element on the page remains visible during print.

## Suggested Fix Direction
Option 1 (recommended, smallest blast radius): move the `@media print` block from the component CSS to a global stylesheet so `body *` is unscoped at runtime. Verify with Chrome DevTools (Rendering → Emulate CSS media type: print) and by saving the print dialog as PDF.

Alternates considered:
- Set `ViewEncapsulation.None` on the parts-catalog component — exposes all of the component's other selectors globally, bigger blast radius.
- Render the label into a hidden iframe and call `iframe.contentWindow.print()` — most robust but a larger refactor.

## Implementation Summary

### FN-1568 (subtask) — frontend
Moved the `@media print` block out of the component CSS (where Angular's `ViewEncapsulation.Emulated` was rewriting `body *` to a scoped attribute selector that matched nothing) into the global stylesheet at `frontend/src/styles.css`. The screen-mode off-screen positioning rule (`.label-print-area { position: fixed; left: -10000px; top: -10000px; }`) had to move with it, otherwise the component-scoped (0,2,0) screen rule would override the now-global (0,1,0) print rule and the label would stay off-screen during print. The component CSS file now carries only a comment pointing readers to the global location and explaining why.

Both entry points — the **Print label** button in the post-create success modal and the **Print label** row action on existing parts — funnel through the same `printLabelForPart` flow that mounts the `.label-print-area` div, so the single CSS move covers both paths.

The optional spec smoke test was skipped: parsing the built CSS bundle from `parts-catalog.component.spec.ts` would not catch the actual specificity-driven rule collision, which only surfaces in a real DOM under print emulation. QA (FN-1569) will provide the higher-signal verification via Chrome "Save as PDF" evidence.

## Files Changed
- `frontend/src/app/components/parts-catalog/parts-catalog.component.css` — removed the `.label-print-area` off-screen rule and the `@media print { body * { visibility: hidden; } ... }` block (lines ~1110–1145); replaced with a comment pointing to the global stylesheet and the FN-1568 / FN-1567 context.
- `frontend/src/styles.css` — appended the off-screen `.label-print-area` rule and the `@media print` block (with the `body *` visibility flip, the `.label-print-area` on-screen reposition, and `@page { margin: 0.25in; }`).

## Key Decisions
- **Option 1 (global CSS) over Options 2 & 3.** Set `ViewEncapsulation.None` on the parts-catalog component would have made every other selector in the file leak globally — too broad. A hidden-iframe re-implementation would have been a larger refactor with no incremental benefit for the immediate regression. Moving just the print rules + the matched off-screen rule is the smallest blast-radius fix.
- **Co-locate the off-screen `.label-print-area` rule with the print rule globally.** Splitting them between component scope (screen) and global scope (print) inverts their specificity and silently breaks the off-screen → on-screen flip during print. The original code worked because both rules sat in the same stylesheet at equal specificity and source order resolved the conflict; preserving that same-specificity / source-order pairing is what makes the fix actually take effect.
- **No change to `partForLabel` / `printLabelForPart` TypeScript.** Per the ticket's out-of-scope guidance — the timing and lifecycle work; only CSS scope was wrong.
- **`drivers.component.css` print rules left untouched.** Per the ticket's out-of-scope guidance — separate concern, file a separate bug after QA verifies if the same defect exists there.
