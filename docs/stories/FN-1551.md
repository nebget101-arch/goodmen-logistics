# FN-1551: [Work Order] Requested By / Assigned To show username instead of first + last name

## Requirement
On the Work Order detail page → Basics tab → Work Order Info, the **Requested By** and **Assigned To** read-only inputs render the user's `username` (`admin`, `biz.work`). They should render `first_name + ' ' + last_name`, with fallback to `username` if either name is missing.

## Acceptance Criteria
- [x] Open a WO whose `requested_by_user_id` resolves to a user with first/last name → field shows "First Last"
- [x] Open a WO whose `assigned_mechanic_user_id` resolves similarly → field shows "First Last"
- [x] Users with no first/last on file fall back to `username` (no blank field)
- [x] Saving the WO does NOT round-trip the display name back into `requested_by_user_id` (resolve the ID separately)
- [x] No regression in WO list view if it surfaces these columns

## Implementation Summary
Backend-only change in `getWorkOrderById()`. The frontend's `[(ngModel)]="workOrder.requestedBy"` and `[(ngModel)]="workOrder.assignedTo"` (`frontend/src/app/components/work-order/tabs/basics-tab/basics-tab.component.html:151,155`) bind to camelCase keys that the service was not setting — the existing snake_case `requested_by_username` / `assigned_to` keys held the username, and the frontend's fallback logic in `work-order.component.ts` was therefore picking up the username.

The fix introduces a small pure helper `formatUserDisplayName(user)` that returns `first_name + ' ' + last_name` (trimmed) and falls back to `username`, then assigns its output to `workOrder.requestedBy` and `workOrder.assignedTo`. Existing `requested_by_username` / `assigned_mechanic_username` / `assigned_to` keys are preserved unchanged so any other consumers (frontend fallback, controllers) keep working.

`listWorkOrders()` does not select user columns at all, so the WO list view surfaces no user display data and is not affected (acceptance criterion 5 satisfied by inspection).

For round-trip safety on save (acceptance criterion 4): `updateWorkOrder()` already preserved the existing `requested_by_user_id` via `workOrder.requested_by_user_id || resolvedUserId`, and live controllers always pass `req.user?.id` (a UUID) which short-circuits the username fallback. As a defensive hardening, the legacy username-resolution fallback in both `createWorkOrder` and `updateWorkOrder` was reordered from `payload.requestedBy || payload.requestedByUsername` to `payload.requestedByUsername || payload.requestedBy`, so any caller that explicitly passes a username (instead of just a display string) wins.

## Files Changed
- `backend/packages/goodmen-shared/services/work-orders.service.js` — added `formatUserDisplayName` helper; set `workOrder.requestedBy` / `workOrder.assignedTo` in `getWorkOrderById`; reordered username-fallback resolution in `createWorkOrder` + `updateWorkOrder` to prefer explicit `requestedByUsername`; exported helper for tests.
- `backend/packages/goodmen-shared/routes/__tests__/work-orders-display-name.test.js` — new file. 6 unit tests covering both names, missing names (fall back to username), blank/whitespace-only names, single-name-present, and null/undefined input.

## Key Decisions
- **Kept `assigned_to` (username) alongside new `assignedTo` (display name).** Two coexisting keys differ only by case but JS treats them as distinct properties. Removing the old key risks breaking any unknown consumer; the cost of keeping it is one extra payload field.
- **Helper exported for test, not extracted to a util module.** It is only used inside this service; lifting it would be premature.
- **Defensive reorder of `payload.requestedByUsername || payload.requestedBy`.** Doesn't change current behavior (no caller sends both; live callers don't reach this branch) but makes the AC structurally airtight against future callers that might mistakenly send the display name.
- **No frontend changes needed.** The `[(ngModel)]` already binds to `workOrder.requestedBy` / `workOrder.assignedTo`; once the backend populates them, the UI consumes them automatically. The existing fallback chain in `work-order.component.ts` (`requested_by_username` → `requestedBy`) keeps legacy behavior for any WO where the new field is missing.
