# FN-408: Implement skip/exception rules for Money Code rows and card-less transactions

## Requirement
Fleet operators need non-fuel Money Code rows and card-less transactions to be automatically skipped or routed to exceptions during fuel import, keeping fuel data clean.

## Acceptance Criteria
- [x] Money Code rows are detected and skipped (not imported as fuel_transactions)
- [x] Skipped rows tracked in batch results with reason
- [x] Card-less rows create exceptions with type 'missing_card_number'
- [x] Skip count visible in import results summary
- [x] Existing rows with valid data are unaffected
- [x] Detection works across EFS, Comdata, WEX, RTS formats

## Implementation Summary

### Rule 1: Money Code / Non-Fuel Skip
- Added `isMoneyCode(row)` function in `fuel-parser.js` that checks `product_type` and `vendor_name` fields against a regex pattern list (MONEY CODE, CASH ADVANCE, ATM, MONEY ORDER, CASH, FEE, SERVICE CHARGE, REBATE, REFUND)
- `validateRow()` now returns `{ errors, warnings, skipReason }` — if `skipReason` is non-null, the row is a non-fuel skip
- `stageBatch()` marks these rows with `resolution_status: 'skipped'` and increments `skippedCount`
- Skipped rows are NOT processed by `commitBatch()` (since they're not in 'valid'/'warning' status)

### Rule 2: Card-Less Row Handling
- In `commitBatch()`, after creating the fuel_transaction, if `card_number_masked` is empty, creates an exception with `exception_type: 'missing_card_number'`
- The transaction is still created (with `matched_status` based on other entity matches) but flagged for review

### Schema
- Added `skipped_rows` integer column to `fuel_import_batches` via migration

## Files Changed
- `backend/packages/goodmen-shared/services/fuel-parser.js` — `isMoneyCode()`, updated `validateRow()` return type
- `backend/packages/goodmen-shared/services/fuel-service.js` — skip logic in `stageBatch()`, missing_card exception in `commitBatch()`, updated `buildExceptionMessage()`
- `backend/packages/goodmen-database/migrations/20260328120000_add_skipped_rows_to_fuel_batches.js` — new migration

## Key Decisions
- Patterns check both `product_type` and `vendor_name` fields (some providers put non-fuel indicators in vendor name instead of product type)
- Skip detection runs before other validation to avoid false errors on rows that will be skipped anyway
- `missing_card_number` exception is separate from `unmatched_card` — the latter means we tried to match but failed, the former means there was nothing to match against
