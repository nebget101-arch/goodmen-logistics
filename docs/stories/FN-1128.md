# FN-1128: S4 — Smart Alerts panel on Control Center

## Requirement
A Smart Alerts panel surfaces severity-ranked alerts derived from cross-service signals: HOS violations imminent, vehicle inspection overdue, load late risk, driver fatigue. Severity computed by AI service using rules + scoring; alerts dismissible per user.

## Acceptance Criteria
- [ ] `GET /api/alerts/smart` returns alerts ranked by severity
- [ ] AI service `/score-alert` endpoint scores severity 0-100
- [ ] Frontend renders top 5 alerts with severity badge, action link, dismiss button
- [ ] Dismissed alerts persist in `user_alert_dismissals` table; do not re-surface for 24h
- [ ] WebSocket push updates the panel in real time

### FN-1159 (subtask) — Severity scoring `/score-alert` endpoint
**Branch:** `ai/FN-1159/score-alert`

ai-service gains `POST /api/ai/score-alert` consumed by the gateway aggregator from FN-1161. The handler combines a deterministic rule-based baseline (from `services/severity-scorer.js`) with a Claude-derived contextual boost so severity stays predictable when the model is unavailable. The contract matches what FN-1161 already calls: request `{ tenantId, alert: { id, type, subjectId, subjectKind, facts } }` → response `{ severity: 0-100, reasoning, action, meta: { baseScore, boost, scoredBy, model?, processingTimeMs } }`.

**Rule-based baselines** per alert type:
- `hos_imminent` — bucketed by `facts.minutesRemaining` (≤15→95, ≤30→90, ≤60→85, ≤120→80, ≤240→75, else 70)
- `fatigue` — uses `facts.fatigueScore` directly (clamped 50-95) when present, otherwise bucketed by `facts.consecutiveDutyHours`
- `inspection_overdue` — bucketed by `facts.daysOverdue` (≥30→90, ≥14→80, ≥7→70, ≥1→65, else 60)
- `late_load_risk` — bucketed by `facts.etaDelta` minutes late (≥240→80, ≥120→70, ≥60→60, ≥30→55, else 50)

The Claude call (Haiku by default, override with `ANTHROPIC_SEVERITY_MODEL`) returns `{ boost: -10..+20, reasoning, action }`. Final severity = `clamp(baseScore + clampBoost(boost), 0, 100)`. The system prompt (`src/prompts/severity.md`) constrains the model to JSON only, max-160-char reasoning, max-80-char imperative action, and forbids inventing facts beyond the alert payload.

**Failure modes** (handler always returns 200 with a finite severity):
- `ANTHROPIC_API_KEY` unset → `meta.scoredBy='rules:no-anthropic'`, baseline + canned reasoning/action
- Claude returns non-JSON → `meta.scoredBy='rules:unparseable-ai-response'`
- Claude throws / times out → `meta.scoredBy='rules:ai-error'`
- Boost outside [-10, +20] is clamped; reasoning/action over the length cap is ellipsis-truncated.

**Tests** (`node src/handlers/__tests__/score-alert-handler.test.js`, 17 cases, runs via `npm test`):
- Pure scorer: clamp, boost clamp, combine, all 4 alert-type baselines, validateAlert reject paths
- Handler: happy path (base + boost merge), boost over/under-clamp, unparseable Claude output fallback, upstream-error fallback, missing-alert 400, unsupported-type 400, length truncation, inspection_overdue end-to-end

### FN-1161 (subtask) — Smart Alerts aggregator + WebSocket push
**Branch:** `backend/FN-1161/smart-alerts`

The gateway gains a local `/api/alerts` router (mounted before the catch-all proxy) that fans out to drivers-compliance, vehicles-maintenance, and logistics for raw signals (HOS imminent, fatigue, inspection overdue, late-load risk), normalizes each into `{ id, type, subjectId, facts }`, then asks ai-service `/api/ai/score-alert` to assign a 0-100 severity per alert. The route filters out dismissals belonging to the requesting user, sorts by severity desc, and returns the ranked top N. A `POST /api/alerts/smart/:id/dismiss` endpoint records a dismissal and broadcasts an `alerts.smart.dismissed` Socket.IO event to the tenant room; `GET /api/alerts/smart` also broadcasts an `alerts.smart.update` event so other connected clients in the same tenant stay in sync without polling.

Failure modes are intentionally lenient:
- A failing upstream signal source produces an entry in `upstreamErrors` and the response still returns 200 with whatever did load (mirrors the FN-1141 briefing pattern).
- A failing AI score-alert call falls back to a per-type baseline severity (`hos_imminent=75`, `inspection_overdue=65`, `fatigue=60`, `late_load_risk=55`) so the panel still shows alerts even if ai-service is degraded.
- WS broadcast failures never fail the HTTP request — they are warn-logged.

**Dismissal persistence (24h TTL):** ships with an in-memory `MemoryDismissalsStore` whose `{ isDismissed, dismiss }` interface matches the future Postgres-backed store. Once FN-1165 lands the `user_alert_dismissals` migration, swapping in a `PgDismissalsStore` against that table is a one-line change in `backend/gateway/index.js`. The 24h non-resurface contract is honored today on a single instance.

**Tests** (`node backend/gateway/__tests__/alerts.test.js`):
- Unauthorized requests (no token / invalid / missing tenant / missing sub) — 401, no fan-out
- Happy path — all 4 sources contribute, ranked by severity desc, broadcast emitted
- Partial upstream failure — degraded sources tracked in `upstreamErrors`, surviving alerts still ranked
- AI scoring fallback — when `/score-alert` errors, alerts get baseline severity and `scoredBy='fallback:ai-error'`
- Dismissal flow + tenant isolation — dismissed alert disappears for the dismissing user; other tenants are unaffected
- Dismissal TTL expiry — entry expires after 24h (driven by injected clock)
- Dismissal alert id required — whitespace id rejected with 400

### FN-1163 (subtask) — Smart Alerts panel component
**Branch:** `frontend/FN-1163/smart-alerts-panel`

Standalone `<app-smart-alerts>` panel mounted on the Control Center (`/dashboard`) after `<app-daily-briefing>`. Renders the top 5 alerts ranked by severity with a categorical badge + 0-100 score, an action link, and a dismiss button. `SmartAlertsService` fetches `GET /api/alerts/smart`, exposes a `BehaviorSubject` of the full ranked list, subscribes to Socket.IO events `alerts.smart.update | alerts.smart.dismissed` (the contract emitted by FN-1161's `backend/gateway/services/alerts-ws.js`) for live updates, and POSTs dismissals to `/api/alerts/smart/:id/dismiss`. The component is OnPush with a loading skeleton (`prefers-reduced-motion` fallback), error fallback (`role="alert"` + retry), empty state, and a11y labels (region, aria-busy, aria-labelledby, score in badge aria-label, per-row dismiss aria-label).

The frontend matches the wire shape FN-1161 returns (no separate categorical severity, no `detail` field — bucket and detail are derived client-side):

```ts
interface SmartAlert {
  id: string;
  type: 'hos_imminent' | 'fatigue' | 'inspection_overdue' | 'late_load_risk';
  subjectId: string;
  subjectKind: 'driver' | 'vehicle' | 'load';
  title: string;
  facts: Record<string, unknown>;        // type-specific normalized facts
  severity: number;                       // 0-100 from /score-alert
  action?: { label: string; routerLink?: (string|number)[]; queryParams?: ...; href?: string };
  reasoning?: string;                     // optional, from AI scorer
  scoredBy?: string;                      // optional provenance
}
interface SmartAlertsResponse {
  tenantId: string;
  alerts: SmartAlert[];
  upstreamErrors?: { source: string; error: string }[];
  generatedAt: string;
}
```

The service exports three pure helpers used by the component:
- `severityBucket(score)` → `'critical' | 'high' | 'medium' | 'low'` (≥80 / ≥60 / ≥40 / else).
- `detailFor(alert)` renders the `facts` blob into a single human-readable line per `type` (handles both camelCase and snake_case keys).
- `defaultActionFor(alert)` synthesizes a router link from `subjectKind`+`subjectId` when the backend doesn't attach one.

WS event names (matching FN-1161 emitter):
- `alerts.smart.update` — full snapshot `{ tenantId, alerts, generatedAt }` → service replaces the local list.
- `alerts.smart.dismissed` — `{ tenantId, userId, alertId, dismissedAt }` → service removes by id.

## Key Decisions
- **Reuse Socket.IO infrastructure rather than a new `/ws/control-center/alerts` path.** The gateway already has FN-811's tenant-scoped Socket.IO server with JWT handshake auth and `tenant:<id>` rooms. Adding a parallel raw WS server would duplicate auth and CORS surfaces. Frontend listens for `alerts.smart.update` / `alerts.smart.dismissed` on the existing socket connection.
- **In-memory dismissals as a placeholder, with a stable interface.** The Postgres migration is owned by FN-1165 and not yet integrated. The store interface (`isDismissed`, `dismiss`) is the only surface the route depends on, so the swap is mechanical once `user_alert_dismissals` exists.
- **Lenient failure semantics**, mirroring FN-1141's briefing aggregator. A flaky downstream service degrades the panel rather than blanking it.
- **AI score-alert call is per-alert and parallel.** `Promise.all` over rawAlerts; in practice each upstream returns ≤20 items so the worst case is ~80 concurrent AI calls. If this proves too aggressive in load testing (FN-1166), a chunked or batched score endpoint is the natural follow-up.

### FN-1163 (subtask)
- **Snapshot model on the WS, not deltas.** FN-1161 emits a full ranked snapshot on every gateway-side change. The service mirrors that — on each `alerts.smart.update`, the local list is replaced (rather than merged), so a client that briefly disconnects can't fall out of sync after reconnect.
- **Severity bucket derived client-side.** The wire shape carries one numeric `severity` (0-100). The component derives the categorical bucket (`critical/high/medium/low`) at render time so the backend can tune scoring thresholds without a frontend change. Bucket boundaries (≥80, ≥60, ≥40) are intentionally co-located with the badge styling and are easy to retune.
- **`facts` rendered, not piped.** Each alert type carries a different `facts` shape (HOS imminent has `minutesRemaining` + `windowType`, fatigue has `fatigueScore`, etc.). The `detailFor(alert)` helper handles both camelCase and snake_case keys (the aggregator preserves either depending on upstream service), so the panel shows a useful line even if upstream changes its casing.
- **Top-5 in component, full list in service.** `SmartAlertsService.alerts$` exposes the entire ranked feed; the component slices the first 5. This keeps "top 5" as a presentation rule (a future notifications drawer can subscribe without re-querying) and lets the local cap (50) stay independent of the panel's display cap.
- **Optimistic dismiss with WS reconciliation.** `dismiss(id)` removes the alert locally before the POST resolves so the UI feels instant; the gateway then broadcasts `alerts.smart.dismissed`, which is consumed by the same client (idempotent — already removed) and by other tabs in the tenant room. On POST failure, the next snapshot reconciles state and the component surfaces a soft error message.
- **No getter-based bindings.** Following the FN-317 RCA, the template references plain class fields (`topAlerts`, `dismissing`) populated in subscribers — no getters that allocate per change-detection tick.
