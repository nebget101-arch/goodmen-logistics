# FN-1790: Adapter — Lease-to-Own driver agreement signing

## Requirement
Wire the engine (FN-1787 + FN-1788) into the existing **Lease-to-Own** module so a lease-to-own agreement can be **sent to the driver to e-sign**. Reuse the existing `lease_agreements` table fields (`status`, `sent_for_signature_at`, `signed_at`, `document_storage_key`). From the lease-financing dashboard/detail, the internal user uploads/prepares the agreement and sends it to the driver; the driver signs via the public web interface (or the existing `driver-lease-view`); on signature the agreement flips `pending_signature → active`, with `signed_at` and the signed-PDF `document_storage_key` persisted.

## Acceptance Criteria
- [ ] "Send for signature" action from lease-financing (dashboard/detail) creates a signature request for the agreement's driver via the engine.
- [ ] Driver signs through the public e-sign interface; existing `driver-lease-view` surfaces/links the signing.
- [ ] On signature: `lease_agreements.status` → `active`, `signed_at` set, signed PDF stored in `document_storage_key`; `sent_for_signature_at` set on send.
- [ ] Reuses the generic engine endpoints — no duplicate signing logic; existing lease status lifecycle respected.

## Frontend ↔ Backend contract (defined by FN-1804, to be implemented by FN-1803)
The frontend "Send for signature" action calls a **lease-financing** endpoint (not the generic agreements endpoint directly) so the request↔lease linkage and status lifecycle stay server-side:

- `POST /api/lease-agreements/:id/send-for-signature` → creates a signature request via the generic engine (`signature-service`) for the agreement's **driver**, sets `lease_agreements.sent_for_signature_at` + `status='pending_signature'`. **Returns** the updated agreement plus the engine's public signer link:
  ```json
  { "agreement": { ...LeaseAgreement, "status": "pending_signature", "sent_for_signature_at": "..." },
    "signature_request_id": "<uuid>", "signer_link": "<PUBLIC_APP_URL>/agreements/sign/<token>" }
  ```
- On signature completion (FN-1803 hook in `signature-service`): set `lease_agreements.signed_at`, persist signed-PDF key into `document_storage_key`, transition `status` `pending_signature → active`.
- `GET /api/lease-agreements/:id` (existing) should additionally surface `sent_for_signature_at`, `signed_at`, `document_storage_key`/`document_download_url`, and optionally `signer_link` so the detail/driver views reflect signing state on reload.

The frontend reads these fields off `LeaseAgreement` and degrades gracefully if any are absent (the UI just won't render that row/banner).

## Implementation Summary
### FN-1803 (subtask) — backend
Wired the lease-to-own module into the generic e-sign engine (FN-1787/FN-1788) on the **backend**, reusing `signature-service` end-to-end — no duplicate signing logic. Implements the contract above exactly.

- **Migration `20260614120000_add_signature_request_to_lease_agreements.js`**: adds nullable `lease_agreements.signature_request_id` (FK → `signature_requests`, `ON DELETE SET NULL`, indexed). This is the single link between a lease and its engine request — the completion hook resolves the lease back from it, so the generic `signature_requests` table stays adapter-agnostic. Additive/guarded (hasColumn + conditional FK).
- **`signature-service.js` (engine, completion hook)**: added a generic `onSigned(handler)` registry + `fireSignedHooks(ctx)`, invoked at the end of `submitSignature` after the signed PDF is generated. The engine never imports any adapter — adapters subscribe. Hooks are best-effort (a throwing handler is logged, never fails the signature). Exported `onSigned`.
- **`lease-financing-service.js`**:
  - `chooseSignatureTemplate(templates, templateId?)` — pure: resolve a finalized (`status='ready'`) `lease_agreement` template (explicit id, else newest).
  - `planSignatureCompletion(agreement, {signedPdfStorageKey, hasTruckConflict})` — pure lifecycle decision: only auto-activate out of `pending_signature`, and only when the truck has no competing active/overdue lease; never regress `active`/other statuses.
  - `applySignatureCompletion({requestId, documentType, signedPdfStorageKey, db})` — the registered hook handler. No-ops for non-`lease_agreement` documents / unlinked requests. In one transaction: sets `signed_at`, persists the signed-PDF key into `document_storage_key`, and (when eligible) flips `pending_signature → active` with the same `vehicles` ownership update + risk snapshot + audit log the manual `/activate` route performs. Registered on the engine at module load via `require('./signature-service').onSigned(...)`.
- **`lease-financing.js` (route)**: added `POST /lease-agreements/:id/send-for-signature` (`lease.financing.edit`). Guards state (`draft`/`pending_signature` only), loads the driver as signer (requires email or phone), resolves a finalized lease_agreement template, calls `signature-service.createSignatureRequest` (which sends the secure link via email/SMS), links the request to the lease, sets `sent_for_signature_at` + `status='pending_signature'`, and returns `{ agreement, signature_request_id, signer_link }`. `GET /lease-agreements/:id` already spreads all `lease_agreements` columns, so `signature_request_id` / `sent_for_signature_at` / `signed_at` / `document_download_url` surface automatically (the post-creation `signer_link` is intentionally not reconstructable — the token is hashed — matching the FE's graceful degradation).
- **Tests** (`lease-financing-service.test.js`, `node --test`): 14 DB-free unit tests for `chooseSignatureTemplate` + `planSignatureCompletion` (lifecycle guard matrix) + engine-hook wiring smoke. Existing `signature-service` (25) and `agreement-service` (33) suites stay green.

**Key decision — one link column, generic engine hook.** Rather than add adapter-specific columns (`linked_entity_type/id`) to the shared `signature_requests` table or import the lease module into the engine, the lease owns the forward link (`signature_request_id`) and the engine exposes a generic `onSigned` registry. All three routers (`lease-financing`, `agreements`, `public-agreements`) run in the same `logistics-service` process, so the in-process hook fires reliably when the driver signs.

## Key Decisions
- **"Send for signature" calls a lease-financing endpoint, not the generic `/agreements` flow directly.** Rationale: FN-1803's own scope says the backend "creates a signature request via signature-service for the agreement's driver" — the request↔`lease_agreement` join and the `pending_signature → active` transition must happen server-side. This keeps FN-1804 within its listed files, honors "no new signing UI," and gives FN-1803 a single clean endpoint to implement. "Routes into the prepare/send flow" is satisfied by surfacing the engine's `signer_link` (returned by the endpoint) on the lease detail.
- **No edits to the generic agreements components** (FN-1787/FN-1788) — reuse only via the backend engine + the public signer page already at `/agreements/sign/:token`.
- **Graceful degradation**: all new UI is guarded on the presence of the new fields, so the frontend can merge and render correctly before FN-1803 lands.

## Deployment Handoff (FN-1803 backend)
- **Service to redeploy**: `fleetneuron-logistics-service` (mounts `lease-financing`, `agreements`, and `public-agreements` routers). No gateway change.
- **DB migration (run before/with deploy)**: `20260614120000_add_signature_request_to_lease_agreements.js` — adds `lease_agreements.signature_request_id` + index. Additive, backward-compatible, reversible (`down` drops the column/index). Requires `signature_requests` (FN-1788) to already exist for the FK — it does on `dev`.
- **Env/config**: none new. The engine's existing optional vars still apply — `PUBLIC_APP_URL` (base of the signer link), `AGREEMENT_SIGN_EXPIRY_DAYS`, and Twilio/SendGrid creds for link delivery (best-effort; absence degrades to no-send but still sets `pending_signature` and returns the link to copy).
- **Prerequisite data**: a finalized (`status='ready'`) `lease_agreement` agreement template must exist for the tenant (uploaded/finalized via the Agreements module). Without one, `send-for-signature` returns `400` with a clear message.
- **Deploy order**: backend (FN-1803) before or with frontend (FN-1804). Both ship together in the `integration/FN-1790 → dev` PR.
- **Post-deploy verification**: run the migration; from a lease detail with a `draft` agreement (driver has email/phone + a ready lease template exists), `POST /lease-agreements/:id/send-for-signature` → `200`, status `pending_signature`, `signer_link` returned; open the link, sign on `/agreements/sign/:token`; the lease flips to `active` with `signed_at` + the signed PDF in `document_storage_key` (downloadable via `document_download_url`).
