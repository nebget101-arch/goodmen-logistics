# FN-772: Loads dashboard actions (3-dot) dropdown hidden behind next table row

## Requirement
On the Loads Dashboard (dashboard updates landed with FN-725), clicking the 3-dot **Actions** button on a load row opens the dropdown menu, but the menu renders behind the next `<tr>`. Items like *Edit Load*, *Copy Load*, *Show on Map*, *Delete*, *Approve* are partially/fully hidden and unclickable on any row that is not the last row. Fix the stacking/overflow so the menu is always fully visible and interactive.

## Acceptance Criteria
- [ ] Clicking 3-dot on any row shows the full dropdown menu above following rows
- [ ] All menu items are clickable (Edit Load, Copy Load, Show on Map, Delete, Approve)
- [ ] Works on the last row (no viewport clipping; flip-up acceptable, not required)
- [ ] No regression to other dropdowns on the dashboard (broker/driver/truck/trailer inline dropdowns, new-load menu)
- [ ] Verified in light and dark themes

### FN-773 (subtask)
**Root cause:** `td { overflow: hidden }` at `frontend/src/app/components/loads-dashboard/loads-dashboard.component.scss:440` clips the absolutely-positioned `.dropdown-menu`. The menu's containing block is `.dropdown { position: relative }` which lives inside the `<td>`, so the td's overflow-hidden clips the menu at the row's bottom edge. The original "hidden under next row" symptom was clipping, not a stacking-context issue.

**Fix (minimal, scoped):**
- HTML: added `class="actions-cell"` to the actions `<td>` only.
- SCSS: new rule `td.actions-cell { overflow: visible; }` — other cells keep ellipsis behavior. Also raised `.actions-dropdown .dropdown-menu { z-index: 30 }` defensively.

**Regression surface:** `.actions-dropdown` is only referenced in `loads-dashboard.component.html`/`.scss` — no other callers.

## Implementation Summary
Single-file scope: `loads-dashboard` component template + styles. Two-line HTML change (class on td), two-line SCSS change (new rule block + z-index).

## Files Changed

- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.scss`

## Key Decisions
- **Scoped override, not global**: kept `td { overflow: hidden }` for text-ellipsis in data cells; only the actions cell escapes. Avoids touching existing ellipsis behavior.
- **No DOM restructuring / CDK Overlay**: the simpler CSS-only fix resolves the observed symptom. Escalating to `@angular/cdk/overlay` would be a heavier refactor with no additional user-visible benefit here.
- **Defensive z-index** (20 → 30): belt-and-suspenders in case a future sibling introduces a stacking context.
