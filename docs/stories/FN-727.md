### FN-755 (subtask) â€” Frontend â€” Load Templates admin page

**Branch**: `frontend/FN-755/load-templates-admin`

**Scope delivered**
- New route `/loads/templates` rendering an admin list of templates with columns **Name Â· Description Â· Broker Â· Route Â· Last Used Â· Actions**
- `Templates` button in the loads-dashboard header navigates to the list
- **Save as Template** button on an open saved load (edit modal footer) opens a prompt for name + description and calls `POST /api/load-templates`
- **Use Template** button on the list navigates back to `/loads` via router state; loads-dashboard pre-fills the inline new-load form with the template's broker, stops, rate, driver/truck/trailer, and notes (dates and PO are intentionally cleared) and status is set to `DRAFT`
- `markUsed(templateId)` (`POST /:id/use`) is called on Use to bump `last_used_at`
- **Edit** opens a modal to update name/description only (template_data is immutable from the UI)
- **Delete** confirms then removes the template

**Files changed**
- `frontend/src/app/services/load-templates.service.ts` (new) â€” typed HTTP client for the template API
- `frontend/src/app/components/loads-dashboard/load-templates/load-templates.component.ts|html|scss` (new) â€” admin list page at `/loads/templates`
- `frontend/src/app/components/loads-dashboard/loads-dashboard.module.ts` â€” declares the new component and adds the `templates` child route
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` â€” adds `LoadTemplatesService`/`Router` injection, `navigateToTemplates()`, Save-As-Template modal state + submit, `applyTemplateToManualForm()`, and router-state detection in `ngOnInit` (`applyUseTemplateFromRouterState()`)
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.html` â€” Templates button in header, Save-as-Template button in edit modal footer, Save-As-Template prompt modal
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.scss` â€” styles for Templates button and Save-As-Template modal

**Key decisions**
- Separate route (`/loads/templates`) rather than a tab, because the existing loads-dashboard component is already ~2800 lines; a child route keeps concerns separate and is cheaper to maintain.
- Use Template delivers via router state + inline new-load form, not the not-yet-implemented 4-step wizard (FN-732 placeholder). When the wizard is wired up in FN-733 through FN-736, `applyTemplateToManualForm()` can be retargeted to seed wizard state instead, without changing the URL-level contract.
- Dates and PO are not templated â€” seeding a date would almost always be wrong.
- Edit is name/description only. `template_data` edits are out of scope â€” users re-snapshot by re-saving a load.

**API contract consumed (owned by FN-753)**
- `GET /api/load-templates` â†’ `{ success, data: LoadTemplateListItem[] }`
- `GET /api/load-templates/:id` â†’ `{ success, data: LoadTemplateDetail }` (includes `template_data`)
- `POST /api/load-templates` body `{ load_id, name, description? }`
- `PATCH /api/load-templates/:id` body `{ name?, description? }`
- `DELETE /api/load-templates/:id`
- `POST /api/load-templates/:id/use` â€” bumps `last_used_at`

The list response is expected to include flattened display columns: `broker_name`, `first_pickup_city/state`, `last_delivery_city/state`, `last_used_at`.

### FN-756 (subtask) â€” Frontend â€” Clone & Return Load actions

**Branch**: `frontend/FN-756/load-clone-return-actions`

**Scope delivered**
- **Row action menu** on the loads list gains two new items:
  - `Clone Load` (icon `file_copy`) â€” calls `POST /api/loads/:id/clone` and opens the inline new-load form pre-filled from the returned draft payload
  - `Create Return Load` (icon `swap_horiz`) â€” calls `POST /api/loads/:id/return-load` and opens the inline new-load form pre-filled
- **Load detail / edit modal** trip-actions row gains matching `Clone` and `Create Return Load` buttons, operating on `editingLoadDetail`.
- Both actions are gated by `*appHasPermission="'loads.create'"` â€” the same RBAC used by the New Load flows â€” so read-only viewers don't see them.
- Nothing is persisted server-side until the user saves from the pre-filled form (per FN-754 contract: endpoints return a draft payload, they do not insert a row).

**Files changed**
- `frontend/src/app/services/loads.service.ts` â€” adds `cloneLoad(id)` and `returnLoad(id)` against `POST /api/loads/:id/clone` and `POST /api/loads/:id/return-load`
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` â€” adds `cloneLoad(load)`, `createReturnLoad(load)`, and the shared private `applyDraftToManualForm(detail)` helper that seeds `manualLoadForm` + `sortedStops` from the returned draft
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.html` â€” Clone Load / Create Return Load items in the row actions menu; matching `Clone` / `Create Return Load` buttons in the edit-modal trip-actions row

**Key decisions**
- Clone and Return Load are **server-driven**: the backend (FN-754) is the single source of truth for what carries over, what gets cleared, and what the new load number looks like. The frontend just patches the form from whatever the server returns. That way any later tweak to the copy/flip rules happens in one place (loads.js), not both ends.
- We reuse the existing **inline new-load form** to show the pre-filled draft, consistent with FN-755's Use-Template flow. Once the 4-step wizard (FN-732â†’FN-736) goes live, `applyDraftToManualForm()` can be retargeted without changing the public action surface.
- Kept the existing `copyLoad()` action untouched â€” it's a different UX (immediately persists a copy and opens it for edit). Clone never persists until the user saves, so it's listed separately in the menu.
- The hero-CTA `openCloneLoad()` stub ("Clone Existing Load" message) is unchanged in this subtask â€” that flow needs a load-picker which is out of FN-756's scope.

**API contract consumed (owned by FN-754)**
- `POST /api/loads/:id/clone` â†’ `{ success, data: LoadDetail }` where data is a draft (new `load_number`, dates cleared, `status=DRAFT`, PO cleared; stops, rate, broker, driver, truck, trailer copied)
- `POST /api/loads/:id/return-load` â†’ `{ success, data: LoadDetail }` where data has stops reversed, rate cleared, dates cleared, same broker (editable), driver/truck/trailer kept, `status=DRAFT`
