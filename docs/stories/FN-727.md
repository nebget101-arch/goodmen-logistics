## Implementation Notes

### FN-752 — Migration: load_templates

**Branch**: `database/FN-752/load-templates-table`

**Migration file**: `backend/packages/goodmen-database/migrations/20260418150000_create_load_templates.js`

**Schema:**
- `id UUID PK` (default `uuid_generate_v4()`)
- `tenant_id UUID NOT NULL`
- `name TEXT NOT NULL`
- `description TEXT NULL`
- `template_data JSONB NOT NULL` — full snapshot consumed by the load wizard
- `created_by UUID NOT NULL` → FK `users(id)` `ON DELETE RESTRICT`
- `last_used_at TIMESTAMPTZ NULL`
- `created_at TIMESTAMPTZ DEFAULT now()`
- `updated_at TIMESTAMPTZ DEFAULT now()`

**Constraints / indexes:**
- `UNIQUE (tenant_id, name)` → constraint `uq_load_templates_tenant_name` — prevents duplicate template names within a tenant (AC).
- `INDEX (tenant_id)` → `idx_load_templates_tenant_id` — list/scope queries are always tenant-scoped.

**Key decisions:**
1. **`onDelete('RESTRICT')` on `created_by`** — `created_by` is `NOT NULL` per AC, so a `SET NULL` cascade would violate the constraint if a user row were ever deleted. Most user removals in this codebase are soft (deactivation), so `RESTRICT` is the safe default.
2. **`uuid_generate_v4()` default** — consistent with the recent `load_ai_extractions` migration (FN-741) and other uuid-extension tables in this package.
3. **No `updated_at` trigger** — keeping parity with sibling migrations; the application/route layer is responsible for bumping `updated_at` on PATCH (consistent with FN-753 endpoints).
4. **`hasTable` guard on up** — idempotent up matches the FN-741 pattern, in case a partial deploy has already created the table.

### FN-753 — Backend load templates CRUD endpoints

**Branch**: `backend/FN-753/load-templates-route`

Added `backend/packages/goodmen-shared/routes/load-templates.js` with six handlers, all gated by `auth(['admin', 'dispatch'])` and scoped to `tenant_id` from `req.context` (via `tenantContextMiddleware`). Route mounted on the logistics service and proxied via the gateway at `/api/load-templates`.

**`POST /api/load-templates`**
- Body: `{ name, description?, load_id? | template_data? }`
- If `load_id` is supplied, fetches the load + stops (scoped by `tenant_id` and, when present, `operating_entity_id`) and snapshots into `template_data`:
  - `load`: `status='DRAFT'`, `billing_status`, `broker_id`, `broker_name`, `driver_id`, `truck_id`, `trailer_id`, `rate`, `notes`
  - `stops[]`: `stop_type, stop_date, city, state, zip, address1, address2, sequence`
  - Also records `source_load_id` for traceability.
- If the caller supplies `template_data` directly (e.g. wizard not yet saved), it is stored as-is.
- Sets `created_by = req.user.id`, returns `201`.
- Unique-constraint conflict (`23505`) on `(tenant_id, name)` returns `409`.

**`GET /api/load-templates`** — list scoped to `tenant_id`. Optional `?q=` ILIKEs `name` and `description`. Paginated via `?page`/`?pageSize` (default 50, max 100). Ordered by `COALESCE(last_used_at, updated_at, created_at) DESC, name ASC`.

**`GET /api/load-templates/:id`** — single template (404 if not owned by tenant).

**`PATCH /api/load-templates/:id`** — partial update of `name`, `description`, or `template_data`. Uses `hasOwnProperty` so callers can explicitly null a field. Duplicate-name conflict returns `409`.

**`DELETE /api/load-templates/:id`** — hard delete within tenant scope. Returns `{ id }` or `404`.

**`POST /api/load-templates/:id/use`** — stamps `last_used_at = NOW()` and returns the full template row so the wizard can pre-fill.

### Key Decisions (FN-753)
1. **Tenant + operating-entity scoping on the source load** — the `POST` snapshot honors `req.context.operatingEntityId` when present so users in a multi-entity tenant can only template loads they can see. List/read/update/delete intentionally scope by `tenant_id` only (templates are tenant-wide, matching the FN-752 schema that has no `operating_entity_id` column).
2. **Two creation modes** — `load_id` for "Save as Template" from an existing load; `template_data` for a power-user path (wizard draft, API client). Both paths write the same JSONB shape.
3. **`POST /:id/use` vs side-effect on load creation** — chose an explicit endpoint because FN-754's `/clone` and `/return-load` return a payload without persisting, so the frontend controls when a template is "used".
4. **`COALESCE` sort on list** — templates with `last_used_at` bubble to the top; never-used templates fall back to `updated_at`, then `created_at`.
5. **Role gating `admin + dispatch`** — loads allow drivers, but templates are a dispatch/admin administrative surface.
6. **JSONB string-serialize** — `JSON.stringify(...)::jsonb` on insert/update to avoid pg driver quirks.
