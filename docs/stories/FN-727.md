# FN-727: Load Templates, Cloning & Return Load

## Requirement
Support three load-reuse patterns: **Templates** (named, reusable snapshots of a load), **Clone** (duplicate with new dates/number), **Return Load** (flip origin/destination).

## Implementation Summary

### FN-753 — Backend load templates CRUD endpoints (branch: `backend/FN-753/load-templates-route`)

Added `backend/packages/goodmen-shared/routes/load-templates.js` with six handlers, all gated by `auth(['admin', 'dispatch'])` and scoped to `tenant_id` from `req.context` (via `tenantContextMiddleware`). Route mounted on the logistics service and proxied via the gateway at `/api/load-templates`.

**`POST /api/load-templates`**
- Body: `{ name, description?, load_id? | template_data? }`
- If `load_id` is supplied, fetches the load + stops (scoped by `tenant_id` and, when present, `operating_entity_id`) and snapshots into `template_data`:
  - `load`: `status='DRAFT'`, `billing_status`, `broker_id`, `broker_name`, `driver_id`, `truck_id`, `trailer_id`, `rate`, `notes`
  - `stops[]`: `stop_type, stop_date, city, state, zip, address1, address2, sequence`
  - Also records `source_load_id` for traceability.
- If the caller supplies `template_data` directly (e.g. wizard not yet saved), it is stored as-is.
- Sets `created_by = req.user.id`, returns `201`.
- Unique-constraint conflict (`23505`) on `(tenant_id, name)` returns `409`.

**`GET /api/load-templates`** — list scoped to `tenant_id`. Optional `?q=` ILIKEs `name` and `description`. Paginated via `?page`/`?pageSize` (default 50, max 100). Ordered by `COALESCE(last_used_at, updated_at, created_at) DESC, name ASC`.

**`GET /api/load-templates/:id`** — single template (404 if not owned by tenant).

**`PATCH /api/load-templates/:id`** — partial update of `name`, `description`, or `template_data`. Uses `hasOwnProperty` so callers can explicitly null a field. Duplicate-name conflict returns `409`.

**`DELETE /api/load-templates/:id`** — hard delete within tenant scope. Returns `{ id }` or `404`.

**`POST /api/load-templates/:id/use`** — stamps `last_used_at = NOW()` and returns the full template row so the wizard can pre-fill.

### FN-754 — Backend `/clone` and `/return-load` endpoints (branch: `backend/FN-754/load-clone-return-endpoints`)

Added two POST handlers to `backend/packages/goodmen-shared/routes/loads.js`. Both are gated by `requireRole(['admin', 'dispatch'])`, use the existing `getLoadDetail(client, loadId, req.context)` helper (so they inherit tenant + operating-entity scoping and stop ordering), and **never persist** — each returns a wizard-ready draft payload that the frontend opens in the create-load wizard for the user to confirm and save.

A new helper, `buildDraftFromLoad(sourceLoad, sourceStops, options)`, centralizes draft construction. Options: `clearBroker`, `clearPo`, `clearRate`, `newLoadNumber`, `flipStops`. The draft uses the same camelCase shape `POST /api/loads` already accepts (`loadNumber`, `status`, `billingStatus`, `brokerId`, `brokerName`, `driverId`, `truckId`, `trailerId`, `rate`, `poNumber`, `notes`, `stops[]`), so the wizard can round-trip the draft back to the existing create endpoint without mapping.

**`POST /api/loads/:id/clone`**
- Returns a draft with:
  - `loadNumber`: freshly-generated via `generateLoadNumber(client)` (SELECT-only; no insert)
  - `status`: `DRAFT`
  - `billingStatus`: `PENDING`
  - `poNumber`: `null` (cleared per spec)
  - `rate`, `brokerId`, `brokerName`, `driverId`, `truckId`, `trailerId`, `notes`: copied
  - `stops[]`: copied 1:1 with `sequence` preserved; `date`, `stopDate`, `stop_date` all cleared (wizard reads any of these)
  - `source_load_id`: original load id
- 404 when the source load is missing or out of tenant/operating-entity scope.

**`POST /api/loads/:id/return-load`**
- Same draft shape as clone, with two differences:
  - `rate`: cleared (spec: "rate cleared")
  - `stops[]`: reversed, and each stop's `stopType` is flipped (`PICKUP` ↔ `DELIVERY`). The last delivery of the source load becomes the first pickup of the return load; intermediate stops invert in-place.
- Broker/driver/truck/trailer are kept (spec: "same broker (editable client-side)", "driver/truck/trailer kept if currently assigned"). Frontend can null them if the user reassigns.

## Key Decisions

### FN-753
1. **Tenant + operating-entity scoping on the source load** — the `POST` snapshot honors `req.context.operatingEntityId` when present so users in a multi-entity tenant can only template loads they can see. List/read/update/delete intentionally scope by `tenant_id` only (templates are tenant-wide, matching the FN-752 schema that has no `operating_entity_id` column).
2. **Two creation modes** — `load_id` for "Save as Template" from an existing load; `template_data` for a power-user path (wizard draft, API client). Both paths write the same JSONB shape.
3. **`POST /:id/use` vs side-effect on load creation** — chose an explicit endpoint because FN-754's `/clone` and `/return-load` return a payload without persisting, so the frontend controls when a template is "used" (i.e., when the wizard actually opens). Clean separation; no hidden coupling to load POST.
4. **`COALESCE` sort on list** — templates with `last_used_at` bubble to the top; never-used templates fall back to `updated_at`, then `created_at`.
5. **Role gating `admin + dispatch`** — loads allow drivers, but templates are a dispatch/admin administrative surface.
6. **JSONB string-serialize** — `JSON.stringify(...)::jsonb` on insert/update to avoid pg driver quirks with nested objects.

### FN-754
1. **Return the same camelCase shape that `POST /api/loads` already accepts** — so the frontend wizard can take the draft, let the user edit, and POST it back to the existing create endpoint without a separate mapping layer. The spec explicitly says "frontend shows in wizard and user saves."
2. **Reuse `getLoadDetail` for reads** — inherits tenant + operating-entity scoping, stop ordering, and the status/billing normalization the rest of the file relies on, so clone/return-load can't leak cross-tenant data.
3. **Generate a new `loadNumber` server-side** — `generateLoadNumber` already handles collision retries against the `loads` table. Doing this on the server means the wizard opens with a real, unique number shown to the user; the user can still override at save time.
4. **Stop-flip semantics for return-load** — reverse the array AND toggle `stopType` on each stop. Example: original `[P1, P2, D1, D2]` → return `[P-from-D2, P-from-D1, D-from-P2, D-from-P1]`. This matches the spec "last delivery → first pickup; intermediate stops reversed" and produces a valid wizard payload where PICKUP stops precede DELIVERY stops in sequence.
5. **Clear all three date aliases on each stop** — the wizard/create endpoint accepts `date | stopDate | stop_date`; setting all three to `null` prevents stale data from leaking through whichever path the wizard uses to read.
6. **`source_load_id` in response** — non-normative field useful for analytics and for the frontend to show "Cloned from #XXXX" in the wizard header. Does not survive into the eventual created load row unless the frontend chooses to persist it.

## Deployment Handoff
- **Services to restart**: `fleetneuron-gateway` (FN-753 only) and `fleetneuron-logistics` (FN-753 + FN-754). Reference `.agent/docs/render_services.md`.
- **Jobs/workers affected**: None.
- **DB migrations to run**: `FN-752` migration creating `load_templates` — **must be deployed before FN-753's route, or the route will 500 on every call**. The logistics service auto-applies migrations on boot (`knex.migrate.latest()` in `server.js`). FN-754 needs no migrations.
- **Env/config changes**: None.
- **Cache invalidation**: None.
- **Feature flags**: None.
- **Deploy order**:
  1. FN-752 (migration) first — merge and deploy logistics so the new table exists.
  2. FN-753 + FN-754 backend — both logistics + gateway restart (FN-754 is independent of the migration but ships with the story).
  3. FN-755 + FN-756 frontend (no service restart required beyond the UI bundle).
- **Post-deploy verification**:
  - FN-753: `POST /api/load-templates {load_id, name}` → 201; `GET ?q=` filters; `POST /:id/use` updates `last_used_at`; duplicate name → 409; cross-tenant → 404.
  - FN-754: `POST /api/loads/:id/clone` → 200 with `status=DRAFT`, new `loadNumber`, stops copied, `poNumber=null`; verify the loads table row count is unchanged. `POST /api/loads/:id/return-load` → 200 with stops reversed and types flipped, `rate=null`; verify loads row count unchanged. Cross-tenant load id → 404.
