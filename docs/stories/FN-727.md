## Implementation Notes

### FN-752 (subtask) — Migration: load_templates

**Branch**: `database/FN-752/load-templates-table`

**Migration file**: `backend/packages/goodmen-database/migrations/20260418150000_create_load_templates.js`

**Schema:**
- `id UUID PK` (default `uuid_generate_v4()`)
- `tenant_id UUID NOT NULL`
- `name TEXT NOT NULL`
- `description TEXT NULL`
- `template_data JSONB NOT NULL` — full snapshot consumed by the load wizard
- `created_by UUID NOT NULL` → FK `users(id)` `ON DELETE RESTRICT`
- `last_used_at TIMESTAMPTZ NULL`
- `created_at TIMESTAMPTZ DEFAULT now()`
- `updated_at TIMESTAMPTZ DEFAULT now()`

**Constraints / indexes:**
- `UNIQUE (tenant_id, name)` → constraint `uq_load_templates_tenant_name` — prevents duplicate template names within a tenant (AC).
- `INDEX (tenant_id)` → `idx_load_templates_tenant_id` — list/scope queries are always tenant-scoped.

**Key decisions:**
1. **`onDelete('RESTRICT')` on `created_by`** — `created_by` is `NOT NULL` per AC, so a `SET NULL` cascade would violate the constraint if a user row were ever deleted. Most user removals in this codebase are soft (deactivation), so `RESTRICT` is the safe default.
2. **`uuid_generate_v4()` default** — consistent with the recent `load_ai_extractions` migration (FN-741) and other uuid-extension tables in this package.
3. **No `updated_at` trigger** — keeping parity with sibling migrations; the application/route layer is responsible for bumping `updated_at` on PATCH (consistent with FN-753 endpoints).
4. **`hasTable` guard on up** — idempotent up matches the FN-741 pattern, in case a partial deploy has already created the table.

**Live up/down verification:** not run locally — no Postgres available in this worktree (Docker daemon down). Syntax-checked with `node -c`. Up/down must be exercised in staging during deploy (see Deployment Handoff).

## Deployment Handoff
- **Migrations to run**: `20260418150000_create_load_templates.js`
- **Services to restart**: any service that opens new pooled DB connections after the schema change — primarily the service hosting the templates routes from FN-753 (`load-templates.js`). Coordinate with the FN-727 merge PR; on its own this subtask only adds a table.
- **Env/config changes**: none.
- **Cache invalidation**: none.
- **Feature flags**: none.
- **Deploy order**: 1) run migration, 2) deploy backend with FN-753 routes (already merged), 3) frontend can light up once FN-755 ships.
- **Post-deploy verification**:
  1. `\d load_templates` shows the columns/types above.
  2. `\d+ load_templates` shows `uq_load_templates_tenant_name` and `idx_load_templates_tenant_id`.
  3. Insert two rows with same `(tenant_id, name)` → second insert fails with `uq_load_templates_tenant_name` violation.
  4. Insert with a non-existent `created_by` → fails with FK violation.
  5. `knex migrate:rollback` cleanly drops the table; `knex migrate:latest` re-creates it.
