# FN-1806 — Visual bbox field-placement editor (drag fields onto document preview)

**Type:** Story · **Parent epic:** FN-1786 (Agreements / e-sign engine) · **Status:** In Progress
**Integration branch:** `integration/FN-1806` (Shape C — multi-agent)

## Requirement

Uploaded agreements are scanned images; signed-PDF generation overlays
values/signatures at AI-detected `bbox` coordinates, so placement is only as
good as detection. Give internal users a visual editor to correct field
placement: render each page of the source doc as a preview and let the user
drag, resize, add, and delete field boxes directly on the page, persisting the
corrected `page` + `bbox` back onto the template's field map. This raises overlay
accuracy and complements the confidence-flag review in FN-1794 (which only flags
— this lets the user fix geometry).

## Acceptance Criteria

- [x] Document pages render as previews (reuse the source-doc signed URL from FN-1793).
- [x] Each detected field shows as a draggable/resizable box positioned by its `bbox`; low-confidence boxes visually distinguished.
- [x] User can move/resize a box, add a new field box, or delete one; role/label still editable (reuse FN-1794 controls); type editable on new boxes.
- [x] Save persists `page` + `bbox` (and adds/deletes) via the field-map endpoint; a re-opened template shows the corrected geometry. *(frontend FN-1807 + backend FN-1808)*
- [x] Coordinate space is consistent with the signed-PDF overlay (FN-1797) — canonical convention in `docs/design/agreements-bbox-coordinates.md`.

## Canonical coordinate convention

`bbox = [x, y, width, height]` in **PDF points, top-left origin**, per page.
Shared by the AI detector, the FN-1807 editor, the FN-1808 persistence layer,
and the FN-1797 signed-PDF overlay (the only place the y-axis flips, to pdf-lib's
bottom-left origin). Authoritative spec: **`docs/design/agreements-bbox-coordinates.md`**.

## Implementation Notes

### FN-1808 (subtask) — Backend: field-map endpoint geometry edits/adds/deletes

Extends `PATCH /api/agreements/templates/:id/fields` (from FN-1793) so the visual
editor can persist geometry corrections.

**What changed**

- **PATCH contract extended** — body is now `{ fields, adds, deletes, finalize }`:
  - `fields[]`: `{ id, role?, label?, page?, bbox? }` — existing edits plus the
    new geometry columns `page` + `bbox`. `sort_order` is preserved (geometry
    edits never reorder).
  - `adds[]`: `{ fieldType, page, bbox, label?, role?, fieldKey? }` — user-drawn
    boxes, persisted with `confidence: null` (manual, no AI), appended after the
    current max `sort_order`.
  - `deletes[]`: `[fieldId]` or `[{ id }]` — fields the user erased.
- **Validation** — `validatePage(page, pageCount)` enforces `1 ≤ page ≤
  page_count`; `validateBbox(bbox)` enforces a 4-number non-negative `[x,y,w,h]`.
  Invalid page/bbox is dropped from edits and rejects an add — never persisted.
- **Tenant scoping** — the template is verified against the caller's tenant up
  front; every field write is additionally constrained by `template_id`, so a
  tenant can only touch its own template's fields. All three mutation sets run
  in one transaction (delete → edit → add → finalize).
- **Canonical coordinate convention documented** —
  `docs/design/agreements-bbox-coordinates.md`.

**Files changed**

- `backend/packages/goodmen-shared/services/agreement-service.js` — `validateBbox`,
  `validatePage`, `sanitizeNewField`, `normalizeDeleteIds`, `fieldUpdateToRow`;
  `sanitizeFieldUpdate` now accepts `page`/`bbox`; `updateTemplateFields` handles
  `updates`/`adds`/`deletes`.
- `backend/packages/goodmen-shared/routes/agreements.js` — PATCH handler reads
  `adds`/`deletes`; updated OpenAPI doc.
- `backend/packages/goodmen-shared/services/agreement-service.test.js` — 26 new
  unit tests. 59/59 pass under Node 20.

**Key decisions**

- bbox stored as a top-left PDF-points `[x,y,w,h]` JSONB array (matches what the
  detector already emits). The y-axis flip to PDF bottom-left space is isolated
  to the FN-1797 overlay.
- Manual adds carry `confidence: null` so the FE's low-confidence flagging
  (FN-1794) treats them as user-authored.
- Geometry edits preserve `sort_order`; adds append at the end.

### FN-1807 (subtask) — frontend: visual bbox editor

New `agreement-placement` feature under `frontend/src/app/agreements/`, reached
from the FN-1794 review screen ("Edit field placement") and routed at
`/agreements/:id/placement`.

**Page rendering — `pdfjs-dist`.** The frontend had no PDF-rendering library;
faithfully previewing a scanned PDF page to map `bbox ↔ pixels` requires
rasterization. Added `pdfjs-dist@^3.11.174` (Angular-17 / TS-5.2 compatible) and
render each page to a `<canvas>` at a known scale, overlaying an absolutely
positioned box layer. The worker is shipped as a static asset
(`assets/pdfjs/pdf.worker.min.js`, copied via `angular.json`). pdf.js is loaded
**lazily** (`await import('pdfjs-dist')`) so it doesn't bloat the initial bundle.
If the PDF can't be rasterized (e.g. signed-URL/CORS failure) the editor degrades
gracefully: boxes remain editable over a blank, correctly-proportioned page.

**Coordinate math + reducers are pure + unit-tested.** All geometry
(points↔pixels, clamp, move, resize) and the edit reducers (add / delete /
update-meta / serialize-to-payload) live in `bbox-editor.logic.ts` with no
Angular/DOM/pdf.js dependency, fully covered by `bbox-editor.logic.spec.ts`. The
component is the thin pdf.js + pointer-event shell. Convention: `bbox = [x,y,w,h]`
in PDF points, top-left origin — a pure scale, no axis flip (see the design doc).

**Interactions.** Pointer-event driven, deltas applied incrementally through the
reducers: drag a box (move), drag any of 8 handles (resize, min 14 pt, clamped to
the page), toggle "Add field" then click-drag to draw a new box, delete via the
inspector or the field-list row. The inspector edits label / type / role; the
field list lets you jump to a field on any page. Low-confidence boxes are
amber-flagged (carried over from FN-1794); user-added boxes are green/dashed.

**Save.** `AgreementService.savePlacement()` PATCHes the FN-1808 field-map body
`{ fields, adds, deletes, finalize }`. Only changed fields are sent (dirty
existing → `fields`, new → `adds`, tombstoned existing → `deletes`); the response
re-seeds the editor so server-assigned ids for new boxes are picked up.

**Known limitation.** The FN-1808 `fields[]` update honors `role`/`label`/`page`/
`bbox` but not `fieldType`, so changing the *type* of an **existing** field is not
persisted (type is persisted for **new** boxes via `adds[]`). Detection rarely
mis-types; a future backend follow-up can add `fieldType` to the update sanitizer.

**Files**
- `frontend/src/app/agreements/agreement-placement/bbox-editor.logic.ts` (new) — geometry + reducers
- `frontend/src/app/agreements/agreement-placement/bbox-editor.logic.spec.ts` (new) — specs
- `frontend/src/app/agreements/agreement-placement/agreement-placement.component.{ts,html,css,spec.ts}` (new) — editor
- `frontend/src/app/agreements/agreement.service.ts` — `savePlacement()`
- `frontend/src/app/agreements/agreements.module.ts`, `agreements-routing.module.ts` — declare + route
- `frontend/src/app/agreements/agreement-review/agreement-review.component.{html,css}` — "Edit field placement" link
- `frontend/angular.json` — pdf worker asset glob
- `frontend/package.json` / `package-lock.json` — `pdfjs-dist@^3.11.174`

## Deployment Handoff

### FN-1808 (backend)
- **Service to redeploy:** `fleetneuron-logistics-service` (mounts the agreements
  router and bundles `goodmen-shared`). No gateway change.
- **DB migrations:** none — `agreement_template_fields.page` / `.bbox` columns
  already exist (FN-1792).
- **Env / config / flags:** none.
- **Post-deploy verification:** `PATCH /api/agreements/templates/:id/fields` with
  a `bbox`/`page` edit, an `adds[]` box, and a `deletes[]` id; re-`GET` and confirm
  geometry persisted, the new field has `confidence: null`, out-of-bounds
  `page`/malformed `bbox` is rejected, cross-tenant id returns 404.

### FN-1807 (frontend)
- **Service to redeploy:** the Angular frontend (Render static site serving
  `dist/goodmen-logistics`). A fresh `npm ci && ng build` is required — **a new
  runtime dependency (`pdfjs-dist`) was added**, so the build must install it and
  copy `pdf.worker.min.js` into `assets/pdfjs/` (already wired in `angular.json`).
  Build under **Node 20** (repo standard).
- **DB migrations / env / config:** none (frontend-only subtask).
- **New asset served:** `assets/pdfjs/pdf.worker.min.js` (~1 MB) — ensure the
  static host serves it; pdf.js fetches it at runtime via `GlobalWorkerOptions.workerSrc`.
- **Deploy order:** backend (FN-1808) first so geometry saves are honored, then
  the frontend. Both are already merged on `integration/FN-1806`.
- **Post-deploy check:** open a finalized/draft template → "Edit field placement"
  → page renders, boxes overlay at their `bbox`, drag/resize/add/delete work,
  Save returns 200 and persists on reload.

## Out of scope
- Re-running AI detection from the editor (user can re-upload).
- Round-trip validation against the signed PDF — that is **FN-1809 (QA)**.
