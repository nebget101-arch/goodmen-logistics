# FN-1837: Agreement templates list UI — browse & manage uploaded templates

## Requirement
The agreements feature (FN-1787) lets a user upload a PDF and review/place fields, but there is **no list/index view of templates already added**. Today `/agreements` routes straight into the upload screen, so a user cannot see what templates exist, their status, or re-open one. Backend `GET /api/agreements/templates` already returns the data (`backend/packages/goodmen-shared/routes/agreements.js:185`); this story adds the missing frontend surface.

## Acceptance Criteria
- [x] `/agreements` shows a list of the tenant's agreement templates (name, type, pages, status, created)
- [x] "Upload new agreement" button routes into the existing upload component; after upload the user lands on review as today
- [x] `agreement.service.ts` has a `listTemplates()` method hitting `GET /api/agreements/templates`
- [x] Rows are clickable and route to review (draft) / placement (ready) without breaking existing deep links
- [x] Empty state renders when there are no templates; loading + error states handled
- [x] Existing `/agreements/:id/review`, `/agreements/:id/placement`, and `/agreements/sign/:token` routes unaffected

## Implementation Summary
Added an `AgreementListComponent` as the new `/agreements` landing view. It loads the tenant's templates via the (already-present) `AgreementService.listTemplates()` → `GET /api/agreements/templates`, and renders a table of name / type / pages / status / created with loading, error (retry), and empty states.

Routing was rewired so the list owns `''` and the upload screen moves to `/agreements/new`:
- `''` → `AgreementListComponent`
- `new` → `AgreementUploadComponent` (unchanged; still navigates to `…/review` after upload)
- `:id/review`, `:id/send`, `:id/placement` → unchanged deep links

Rows route by lifecycle status: `draft` → `…/review`, `ready` → `…/placement`. The "Upload new agreement" header/empty-state buttons route to `/agreements/new`.

Because the upload screen moved off `''`, the two equipment-lease entry points that previously navigated to `/agreements` (carrying subject context query params, FN-1801) were updated to navigate to `/agreements/new` so that flow still lands on the upload screen. The review/placement "cancel/finalize → `/agreements`" navigations now correctly land on the new list (an improvement, no change needed).

Styling uses the existing AI dark-theme tokens lifted from the upload screen for visual consistency.

## Key Decisions
- **Ready → placement** (not send): followed the AC literally (`route to … placement (ready)`). Placement is the visual field editor; from there the user can proceed to send.
- **Upload moved to `/agreements/new`** rather than a separate sibling route, keeping the whole flow under the lazy `agreements` module. Updated the two FN-1801 subject-context entry points to match so equipment-lease initiation still opens the upload screen.
- Table layout with a mobile breakpoint (hides the Type column under 640px) rather than cards, matching the data-dense list convention elsewhere in the app.
