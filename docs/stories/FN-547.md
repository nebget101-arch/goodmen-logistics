# FN-547: Settlements endpoint returns HTTP Status 0 / Unknown Error — connection dropped before response

## Problem
`GET /api/settlements` and `GET /api/settlements/settlements/:id` returned **Status 0** (no response headers) in the browser. The root cause was two related issues in `backend/gateway/index.js`:

1. **No `proxyTimeout`** — `buildProxy` had no timeout configured. When `logistics-service` cold-starts on Render free tier, the proxy connection hangs indefinitely. The browser eventually drops the TCP connection, producing Status 0. The `onError` handler never fires because the error occurs at the socket level before `http-proxy-middleware` can intercept it.

2. **No `headersSent` guard in `onError`** — If the proxy times out after partial headers are written, calling `res.status(502).json(...)` on an already-sent response throws an uncaught exception.

## Path doubling — NOT a bug
The frontend `api.service.ts` calls `/settlements/settlements/:id` (doubled path). This is intentional:
- Gateway mounts `buildProxy(LOGISTICS_URL, 'logistics')` at `/api/settlements`
- `pathRewrite` reconstructs the full path: `/api/settlements/settlements/:id`
- `logistics-service` strips `/api/settlements` via its route prefix
- Remaining path `/settlements/:id` matches `router.get('/settlements/:id')`

Removing the double path would break routing. It was not changed.

## Fix

### `backend/gateway/index.js`
```js
// Added to buildProxy createProxyMiddleware options:
proxyTimeout: 55000,

// Added headersSent guard to onError:
onError: (err, req, res) => {
  console.error('[gateway] proxy error', err.message);
  if (res.headersSent) return;
  res.status(502).json({ error: 'Bad Gateway', message: 'Unable to reach the API. Please try again later.' });
}
```

`proxyTimeout: 55000` (55 seconds) gives Render free tier services up to 55 s to respond before the proxy closes the socket and fires `onError` — returning a clean 502 to the browser instead of Status 0.
