# FN-1548: Render production build fails — initial bundle exceeded 3 MB budget

## Requirement
Render's Angular production build is failing because the initial bundle (3.07 MB) exceeds the configured 3.00 MB error budget by 66.77 kB. The same build also emits ~13 TS warnings of the form `*.spec.ts is part of the TypeScript compilation but it's unused`, indicating `tsconfig.app.json` is pulling spec files into the app compilation. We need to (a) clean up the tsconfig so specs are globally excluded, and (b) trim the initial bundle below budget — primarily by lazy-loading components currently declared eagerly in `app.module.ts`.

## Render log excerpt
```
Warning: bundle initial exceeded maximum budget. Budget 1.00 MB was not met by 2.07 MB with a total of 3.07 MB.
Error: bundle initial exceeded maximum budget. Budget 3.00 MB was not met by 66.77 kB with a total of 3.07 MB.
==> Build failed
```

## Implementation Summary

### FN-1549 (subtask)

**tsconfig fix.** Replaced the per-file spec exclude list in `frontend/tsconfig.app.json` with a single `src/**/*.spec.ts` glob. Also added `src/environments/environment.dev.ts` to the exclude list (file-replacement target, parallel to the existing `environment.prod.ts` entry). All thirteen `*.spec.ts is part of the TypeScript compilation but it's unused` warnings are gone after this single change.

**Unused-file cleanup.** The first build after the tsconfig fix surfaced six remaining "unused file" warnings on barrel re-exports and one stray service. Verified each had zero consumers (`grep -r` for both the file path and the exported symbol) and removed:
- `src/app/services/entity-cache.service.ts`
- `src/app/shared/components/ai-sparkle/index.ts`
- `src/app/shared/components/confidence-badge/index.ts`
- `src/app/shared/components/grouped-alert-row/index.ts`
- `src/app/shared/components/severity-badge/index.ts`

Each component is still imported directly from its `.component.ts` everywhere, so the barrel exports were dead.

**Bundle trim — five new lazy feature modules.** Identified the heaviest eagerly-imported routes by inspecting source-folder sizes in `app.module.ts` declarations (stats.json from `--stats-json` did not include module-level data on Angular 17, so source size was the proxy). Created small `<feature>.module.ts` files following the existing `MaintenanceModule` / `WorkOrderModule` / `LoadsDashboardModule` pattern and converted these five routes to `loadChildren`:

| Route | Module | Source size moved | Notes |
|-------|--------|-------------------|-------|
| `/receiving` | `WarehouseReceivingModule` | ~210 KB | Heaviest single route — parent + 5 child components (invoice upload card, invoice review modal, quick-add panel, receiving activity, receiving activity drawer). |
| `/onboard/:packetId` | `OnboardingPacketModule` | ~200 KB | Public driver onboarding packet — parent + 3 tiered children (consent form, employer history tiered, disqualification history). Never visited from authenticated flows. |
| `/employment-application` | `EmploymentApplicationModule` | ~92 KB | Public employment application form. Standalone, large reactive form. |
| `/admin/fmcsa-imports` | `FmcsaImportsAdminModule` | ~52 KB | Internal-only FMCSA bulk import tooling. Guard `InternalTenantGuard` moved into the lazy module. |
| `/admin/trial-requests` | `TrialRequestsAdminModule` | ~52 KB | Internal-only trial-request triage. Guard `InternalTrialAdminGuard` moved into the lazy module. |

**Result.** Production build (`npm --prefix frontend run build -- --configuration production`) now exits 0 with:
- Initial bundle: **2.66 MB** (down from 3.07 MB — saved ~410 KB; **~340 KB headroom** under the 3 MB error budget, well past the 170 KB headroom target).
- Zero spec warnings.
- Zero "unused file" warnings.
- The remaining warning (`bundle initial exceeded maximum budget. Budget 1.00 MB was not met by 1.66 MB`) is the long-standing >1 MB *warning* threshold; it was a warning before this change too. Only the >3 MB *error* gate matters for Render deploy.

## Key Decisions

- **Did not lazy-load `/billing`.** `PaymentMethodFormComponent` is opened as a `MatDialog` from the eagerly-loaded `TrialBannerComponent`, so moving it into a lazy module would force the chunk back into the eager graph (or require restructuring the dialog wiring). Leaving billing eager kept the change scoped — and the five other lazy modules already cleared the budget by ~340 KB headroom, so billing was not needed.
- **Did not touch `frontend/angular.json` budgets.** Per the AC, the 3 MB error budget is a regression guardrail; bundle came under 2.9 MB on lazy-loading alone, so no budget tweak was needed.
- **Lazy-loaded admin guards inside the lazy module.** `InternalTrialAdminGuard` and `InternalTenantGuard` were only used by the two admin routes converted to `loadChildren`. Moved the guard imports into each lazy module so they're not pulled into the eager bundle just to declare a route.
- **Used the existing module-per-route pattern** (`MaintenanceModule`, `WorkOrderModule`, `LoadsDashboardModule`) rather than introducing standalone-component lazy loading, to minimize churn and stay consistent with the rest of the codebase.
