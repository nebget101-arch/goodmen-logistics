# FN-1302: S1.bug — Daily AI Briefing aggregator calls 4 nonexistent upstream endpoints

## Requirement
The Daily AI Briefing aggregator (FN-1141, shipped under FN-1124) fans out to four upstream endpoints that were never implemented. The briefing card on `/dashboard` always renders "No data" for every section, and the logistics-service emits 500 errors on every dashboard load (because `/api/loads/throughput` matches `router.get('/:id')` and PG rejects "throughput" as a UUID).

Implement the four missing endpoints and harden the `/:id` handler against non-UUID ids so the failure mode is 404, not 500.

## Acceptance Criteria
- [ ] `GET /api/loads/throughput?date=YYYY-MM-DD` returns 200 with `{ success, data: { date, loadCount, deliveredCount, exceptionCount, totalRevenue } }`, tenant-scoped
- [ ] `GET /api/loads/exceptions/count?date=YYYY-MM-DD` returns 200 with `{ success, data: { date, count, breakdown } }`
- [ ] `GET /api/drivers/risk/top?limit=N` returns 200 with `{ success, data: [{ driverId, name, riskScore, topFactor }] }`
- [ ] `GET /api/vehicles/risk/top?limit=N` returns 200 with `{ success, data: [{ vehicleId, unitNumber, riskScore, topFactor }] }`
- [ ] Both new logistics routes registered before `router.get('/:id')` in `loads.js`
- [ ] `GET /api/loads/<non-uuid>` returns 404 (not 500) and emits no `loads_get_failed` log
- [ ] `GET /api/ai/briefing` on dev returns `upstreamErrors: []`
- [ ] Briefing card renders real numeric metrics on `/dashboard`

**Route-ordering fix:** `loads.js` `router.get('/:id')` now shape-checks the id against a UUID regex BEFORE hitting Postgres, so a missing static route surfaces as `404 Load not found` instead of `500 + loads_get_failed + invalid input syntax for type uuid`. This belt-and-suspenders the new `/throughput` and `/exceptions/count` static routes (which are also registered before `/:id`).

## Key Decisions
- **`exception_count` derivation.** No `load_exceptions` table exists, and `loads` has no first-class exception flags. The closest "open exceptions for this day" signal already in the codebase is the `needs_attention` composite used by `/api/loads/ai-insights` (drafts_ready + overdue + missing_docs). The new endpoints reuse the same definition so the briefing card and the intelligence panel stay consistent.
- **Driver `topFactor`.** Pulled from the highest value in `driver_risk_scores.category_scores` (JSONB). When the JSONB blob is absent or all-zero, `topFactor` is `null` per the story description.
- **Vehicle composite.** No "open DOT inspection violations per internal vehicle" linkage exists in the schema; the composite is bounded to data we already collect. Pending maintenance, overdue service intervals, and recent high-priority maintenance act as the breakdown proxy. The vehicle source table is detected at runtime via `to_regclass`, and tenant scoping is only applied when the column actually exists, matching the existing graceful-degrade pattern in `routes/vehicles.js`.
- **Test convention.** Co-located `*.test.js` next to the route file (matching `routes/loads-ai-insights.test.js`) instead of a separate `__tests__/` directory, since that's the pattern actually present on `dev`.
