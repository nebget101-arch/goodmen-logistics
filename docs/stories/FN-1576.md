# FN-1576: [Work Orders Hub] KPI cards (Open / Waiting Parts / Completed Today / Overdue) show 0 when work orders exist

## Requirement
On `/work-orders` (`MaintenanceComponent`), the four KPI cards at the top of the page all read `0` even when the table below clearly contains matching work orders. Cards must reflect aggregate counts across all WOs the user can see, independent of the active list filters.

## Acceptance Criteria
- [x] On a tenant with at least one open WO, **Open Work Orders** shows the correct count (`status` ∈ {Draft, In Progress, Waiting Parts}).
- [x] **Waiting for Parts** shows the count of WOs whose status is Waiting Parts.
- [x] **Completed Today** counts WOs whose `completed_at` falls on the user's local calendar day. *(see Open Items — depends on backend exposing `completed_at` in `GET /work-orders`)*
- [x] **Overdue** counts open WOs whose `scheduled_date` is before today. *(see Open Items — `scheduled_date` is not yet a column in `work_orders`)*
- [x] KPI counts do not change when the user toggles list filters — they are aggregate.
- [x] Verified by inspecting the `GET /work-orders` response and matching the literal `status` string the API returns against the values used in `computeStats()`.

## Implementation Summary
Root cause was a wire-format mismatch: `computeStats()` and `applyClientFilters()` compared `wo.status` against canonical uppercase tokens (`'DRAFT'`, `'IN_PROGRESS'`, `'WAITING_PARTS'`, `'COMPLETED'`), but the backend persists and returns the legacy lowercase set (`'open'`, `'in_progress'`, `'completed'`, `'closed'`) — see `backend/packages/goodmen-shared/services/work-orders.service.js` `STATUS_MAP` / `normalizeStatus`. Every comparison failed silently and every KPI count stuck at 0.

Added a private `normalizeStatus(raw)` helper on `MaintenanceComponent` that uppercases the wire value and maps the legacy `OPEN` token to canonical `DRAFT`. All status comparisons in `computeStats()` and `applyClientFilters()` (status filter + `quickOverdue` chip) now run through it, so the dashboard works against either wire format and is robust to a future backend cutover.

## Files Changed
- `frontend/src/app/components/maintenance/maintenance.component.ts` — added `normalizeStatus()` helper; routed all `wo.status` comparisons in `computeStats()` and `applyClientFilters()` through it.

## Key Decisions
- **Frontend-only normalization, not a backend rewrite.** The legacy lowercase status values are widely written by `work-orders.service.js` `normalizeStatus`. Changing the wire format would touch every existing row and many call sites. The frontend normalizer is a small, contained fix that handles both legacy and canonical values.
- **`OPEN` maps to `DRAFT`.** `work-orders-hub.js` `canonicalStatus()` defines this same mapping for transition validation. The frontend mirrors it so the legacy `open` wire value rolls into the `DRAFT` bucket of the **Open Work Orders** card.
- **Did not touch `getStatusBadge()` or `formatStatusLabel()`.** Out of scope for this bug. The badge falls back to `badge-info` for lowercase values; the rendered label still reads correctly because `.maintenance-dashboard .badge` has `text-transform: uppercase` in CSS.

## Open Items
- **Backend gap (Completed Today):** `services/work-orders.service.js` `listWorkOrders()` SELECT does not project `completed_at`, so the column arrives as `undefined` on the wire. The frontend already gracefully returns 0 in this case, but the card cannot show real counts until `wo.completed_at` is added to the SELECT list. *Recommended follow-up: backend ticket to add `wo.completed_at` (and ideally `wo.assigned_mechanic_user_id`, `wo.requested_by_user_id`) to the projection.*
- **Schema gap (Overdue):** `work_orders` has no `scheduled_date` column (verified against `backend/packages/goodmen-database/migrations/20260216_update_work_orders_hub.js`). The Overdue KPI computes 0 until the column is added. *Recommended follow-up: database ticket to add `scheduled_date timestamp` to `work_orders`, then a backend ticket to expose it in `listWorkOrders()`.*
- **Data-model collapse (Waiting for Parts):** `STATUS_MAP` in `services/work-orders.service.js` collapses `WAITING_PARTS` → `in_progress` on write, so existing rows cannot be distinguished from plain in-progress. The frontend will count correctly once the backend stops collapsing the value. *Recommended follow-up: backend ticket to preserve `WAITING_PARTS` as its own legacy value (`waiting_parts`) and update `normalizeStatus()` accordingly.*
