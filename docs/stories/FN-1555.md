## Implementation Summary

Added `deactivatePart(id)` to `parts.service.js` as a soft-delete that sets `status='INACTIVE'` when the parts table has the `status` column, falling back to `is_active=false` on the legacy boolean schema (the same dual-schema pattern already used by `getActivePartsCondition`, `getCategories`, and `getManufacturers` in this service). The new function looks up the row first and throws `Error('Part <id> not found')` when missing, mirroring `getPartById` and `deletePart`. Bumped the route's `catch` to return **404** for "not found" errors and **400** otherwise.

## Key Decisions

- **Soft-delete semantics, not hard-delete**: The endpoint name and product intent ("Deactivate") imply the row stays in the database for audit/inventory history. `deletePart` (hard `del()`) is a separate, intentional path; we did not re-route `/deactivate` to it.
- **Schema-aware via `columnInfo()`**: Mirrors what `getActivePartsCondition` already does. Some environments shipped with `status TEXT` (ACTIVE/INACTIVE) and some with `is_active BOOLEAN`. The function handles both rather than assuming one shape and breaking the other.
- **404 via error-message regex in the route**: Cheapest fix that matches the existing pattern in `GET /:id` (which already does `res.status(404)` for any thrown error from `getPartById`). A typed error class would be cleaner but is out of scope for a one-line bug fix.
- **`updated_at` set via `db.fn.now()` only when the column exists** â€” avoids breaking inserts on environments where the column wasn't migrated.
