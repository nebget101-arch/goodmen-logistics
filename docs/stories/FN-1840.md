# FN-1840: Branding — auto-downscale oversized logos on upload

## Requirement
Uploading a normal high-res logo on the branding page fails with `{ "message": "Logo dimensions must be 1024x1024 or smaller" }` (reproduced with a 3500×3500 PNG). The backend cap `MAX_LOGO_DIMENSION = 1024` in `backend/packages/goodmen-shared/routes/branding.js:15` (rejection at L229-231) is intentional — logos are embedded small in PDFs/headers. The gap is UX: a valid source logo is bounced instead of fitted. Fix: downscale client-side before upload, keeping the backend cap as a safety net.

## Implementation Summary
Frontend-only change. Oversized logos are now downscaled in the browser to fit within
1024×1024 (aspect ratio preserved, original MIME type kept so PNG transparency survives)
*before* the existing multipart upload. Images already ≤1024px upload byte-identical — no
re-encode. The backend 1024px cap in `branding.js` is untouched and remains the safety net.

Flow in `BrandingSettingsComponent.uploadFile`:
1. Validate MIME type up front (PNG/JPEG/WebP).
2. `prepareLogoForUpload(file)` reads dimensions; if either axis > 1024 it draws the image to
   an offscreen canvas scaled to fit and exports a Blob in the same format; otherwise it
   returns the original file unchanged with `resized: false`.
3. Enforce the 2MB cap on the *final* blob (so a >2MB-after-resize file still surfaces an
   error; a fine-sized 4000px source that happens to be small passes).
4. Upload the (possibly resized) blob. On success, if it was resized, show a non-blocking
   notice: "Logo was resized to fit 1024×1024." (distinct from the error row, `role="status"`).

The pure fit math is factored into an exported `fitWithin()` so it's unit-testable without the
DOM; the canvas/Image/URL plumbing stays in `prepareLogoForUpload`.

## Key Decisions
- **Client-side canvas downscale, no backend change** — per the ticket decision; avoids adding a
  server-side image lib (`sharp`) the backend doesn't have, and keeps the 1024 cap as a safety net.
- **Original bytes preserved when within bounds** — `prepareLogoForUpload` returns the untouched
  `File` (no canvas round-trip) so already-fine logos aren't needlessly re-encoded (AC #3).
- **Size cap applied after resize** — a large-pixel source is resized first, then size-checked, so
  the ">2MB after resize" AC is met without bouncing valid high-res sources early.
- **MIME type preserved on export** — `canvas.toBlob(cb, file.type)`; PNG keeps transparency,
  JPEG/WebP stay in format (AC #1, #6).
- **`fitWithin` extracted for testability** — `ng test` can't run locally (no karma); the DOM-free
  math is covered by a spec that CI/karma can run.

## Deployment Handoff
- **Frontend-only.** No backend, DB, or env changes. Backend `branding.js` cap unchanged.
- **Service to redeploy:** the Angular frontend (static build). No microservice restarts, no
  migrations, no env vars, no feature flags, no cache invalidation.
- **Post-deploy verification:** on the Branding settings page, upload a >1024px logo (e.g. the

  intact; upload a ≤1024px logo → uploads unchanged, no notice; upload a non-image / corrupt file →
  clear error.
