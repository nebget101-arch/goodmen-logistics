# FN-1729 (Bug): Password reset email link points to localhost

## Symptom
Clicking "Reset Password" in the password-reset email did not land the user on the
reset page — the emailed link was `http://localhost:4200/reset-password?token=...`.

## Root cause
`backend/packages/goodmen-shared/routes/auth.js` builds the reset link from
`PASSWORD_RESET_URL_BASE || FRONTEND_URL || 'http://localhost:4200/reset-password'`.
Neither env var was declared for `fleetneuron-auth-users-service` in `render.yaml`
/ `render-dev.yaml`, so the link silently fell back to localhost. The frontend
route (`/reset-password`) and token param (`?token=`) were already correct, so the
flow works once the base URL is right.

## Fix (Shape A — standalone bug)

1. **render.yaml / render-dev.yaml** — declare `PASSWORD_RESET_URL_BASE` on
   `fleetneuron-auth-users-service` with the deployed frontend host:
   - prod: `https://fleetneuron.ai/reset-password`
   - dev: `https://dev.fleetneuron.ai/reset-password`
   Hosts taken from the existing `CORS_ORIGIN` allowlist. Set as an explicit
   `value:` (not `sync: false`) — see Key Decisions.
2. **backend/packages/goodmen-shared/routes/auth.js** — hardening:
   - Added a derived fallback `${PUBLIC_APP_BASE_URL || APP_BASE_URL}/reset-password`
     (trailing slash stripped) between the explicit vars and the localhost default,
     mirroring `roadside.service.js` / the other email-link builders. The link is
     now correct even if `PASSWORD_RESET_URL_BASE` is never set, as long as the
     service has a frontend base configured.
   - Added a loud `console.warn` when the resolved base contains `localhost` while
     `NODE_ENV=production`, so a misconfigured deploy fails loudly instead of
     silently emailing dead links.
3. **.env.example** — documented `PASSWORD_RESET_URL_BASE`.

## Files Changed
- `render.yaml` — `PASSWORD_RESET_URL_BASE=https://fleetneuron.ai/reset-password` on `fleetneuron-auth-users-service`.
- `render-dev.yaml` — `PASSWORD_RESET_URL_BASE=https://dev.fleetneuron.ai/reset-password` on `fleetneuron-auth-users-service-dev`.
- `backend/packages/goodmen-shared/routes/auth.js` — derived fallback + production localhost warning.
- `.env.example` — documented the var.

## Key Decisions
- **Explicit `value:` instead of `sync: false`.** The ticket suggested `sync: false`,
  but the whole bug is "nobody set the var." The reset URL is public (it's emailed to
  users), and `CORS_ORIGIN` already hardcodes these exact hosts as `value:` in
  `render.yaml`. Hardcoding the public URL makes the deploy self-correcting — no
  manual dashboard step, no recurrence. (Can still be overridden in the dashboard if
  ever needed.)
- **Derived fallback uses `PUBLIC_APP_BASE_URL`/`APP_BASE_URL`** — already configured
  on the service and used by every other email-link builder, so the reset flow is
  robust to the explicit var being unset. `APP_BASE_URL` alone is the frontend origin
  (no path), so it's composed with `/reset-password` rather than used raw.

## Deployment Handoff
- **Services to redeploy:** `fleetneuron-auth-users-service` (prod) and
  `fleetneuron-auth-users-service-dev` (dev) — they mount `goodmen-shared/routes/auth.js`
  and own the new env var. Redeploy picks up both the blueprint env change and the
  code change.
- **Env/config changes:** new `PASSWORD_RESET_URL_BASE` declared in `render.yaml`
  (prod) and `render-dev.yaml` (dev) with explicit public values — no dashboard step
  required. If overriding per environment, set it in the Render dashboard.
- **DB migrations:** none.
- **Feature flags / cache:** none.
- **Deploy order:** single service; no ordering constraints.
- **Post-deploy verification:**
  1. Trigger a password reset on dev; confirm the email link host is
     `https://dev.fleetneuron.ai/reset-password?token=...` (not localhost).
  2. Click it → lands on the reset form with the token populated.
  3. Set a new password and log in with it.
  4. Check the auth-users-service logs show no `[auth] PASSWORD RESET misconfig`
     warning (it would fire if the base still resolved to localhost in production).
