## Implementation Summary

### FN-1527 (subtask)

Hardened the orchestrator → ai-service path so a single transient blip stops surfacing as a user-facing `502 "AI summary service unavailable"` on the Work Order page.

**Root-cause hypothesis:** the orchestrator's axios timeout (`AI_REPAIR_HISTORY_TIMEOUT_MS`) defaulted to 12s, which is shorter than `fleetneuron-ai-service` cold-start latency on Render's free tier when the service has been idle. A first request after sleep would either time out (`ECONNABORTED`/`ETIMEDOUT`) or get an `ECONNRESET` mid-flight, the orchestrator's `try/catch` translated that to `{status: 0, data: null}`, the route returned a flat `502 {message: "AI summary service unavailable"}` with no `Retry-After`, and the widget's retry button hit the still-cold service before warm-up finished.

**Fix:**

1. **Orchestrator (`vehicle-repair-history.service.js`)**
   - Default `AI_REPAIR_HISTORY_TIMEOUT_MS` raised from `12000` → `30000` ms to cover Render cold-start.
   - Added one retry on transient failures: HTTP `502 / 503 / 504` and network errors `ECONNRESET / ETIMEDOUT / ECONNABORTED`. Backoff defaults to 2s, configurable via `AI_REPAIR_HISTORY_RETRY_BACKOFF_MS`.
   - `ECONNREFUSED` is intentionally **not** retried — it usually signals a non-transient outage (bad deploy / mis-set `AI_SERVICE_URL`) and retrying just delays the user-visible error.
   - Structured logs via the shared `utils/logger`: each attempt logs `vin`, attempt number, downstream status / error code, duration, and `historyLen`. Final fail logs at `error` level so it lands in Render error logs.
2. **Route (`routes/vehicles.js`)**
   - The 502 response on the `/api/vehicles/:id/repair-history-summary` path now sets `Retry-After: 5`. The frontend retry button can read this and stop the spinner timer at a sensible cap.
3. **AI handler (`vehicle-repair-history-handler.js`)**
   - Defense-in-depth: catch block now logs `err.status`, Anthropic `err.error.type` (e.g. `overloaded_error`, `rate_limit_error`), `err.code`, `processingTimeMs`, and `model` — so when the orchestrator retries and Render shows two consecutive log lines, on-call can immediately tell whether Anthropic is overloaded vs the handler itself blew up.
   - Handler's own 502 response also sets `Retry-After: 5` so the orchestrator (and any direct caller) gets a consistent retry hint regardless of which leg failed.

**Tests added** (orchestrator):
- `retries once on transient 503 then surfaces 200`
- `retries once on ECONNRESET then surfaces 200`
- `ECONNREFUSED is NOT retried` — explicit negative test guarding the conservative classification.

**Tests added** (handler):
- `Anthropic upstream failure → 502 + Retry-After header` — exercises the previously-uncovered catch-block path.

## Key Decisions

1. **Default timeout 30s** — Anthropic Haiku latency is well under 5s warm; 30s is generous enough to absorb a Render cold-start (~10-20s) plus one Anthropic call. The env var override remains so production can tune.
2. **One retry only, 2s backoff** — Per the ticket. Two retries at 2s each plus 30s timeouts could pile up to ~64s before the user sees an error, which is worse UX than a clean 502 with `Retry-After`. One retry covers the common case (cold start, single 503) without piling.
3. **`ECONNREFUSED` not in transient set** — Distinguishes "service warming up" (retry helps) from "service down" (retry just wastes seconds). Explicit test guards this.
4. **Logging via shared `utils/logger`, not `console`** — Keeps the orchestrator log line consistent with the rest of `goodmen-shared` (timestamp, level, app field) so Render's structured-log dashboards pick it up uniformly.
