# FN-1126: S3 — Predictive Insights & Trends widget

## Requirement
A widget on Control Center surfaces 7-day predictive trends: load volume forecast, predicted maintenance windows, projected on-time %, fuel cost trend. Backed by a new aggregation endpoint that derives trends from existing tables.

## Acceptance Criteria
- [ ] `GET /api/insights/trends?range=7d` returns the 4 trend series
- [ ] Each series includes `predicted` and `actual` arrays for charting
- [ ] Frontend renders 4 sparklines with current value + delta vs. last week
- [ ] Caches at gateway for 10 minutes per tenant
- [ ] Works for tenants with sparse data (<7 days) — returns nulls, not errors

## Implementation Summary

### FN-1152 (subtask) — backend trends aggregator
- New gateway-local route `GET /api/insights/trends?range=7d` that JWT-authenticates the caller, derives `tenant_id` from the token, and returns four series (`loadVolume`, `maintenance`, `onTimePct`, `fuelCost`).
- Each series is `{ actual: [{ date, value }], predicted: [{ date, value }] }`. `actual` is the past 7 UTC days ending today (oldest → newest); `predicted` is the next 7 days, computed via simple linear regression over the actual values.
- Sparse-data graceful nulls: missing day → `value: null`. Insufficient history (<2 actual points) → predicted is all-null. Per-series query failure is isolated; the failed series gets all-null arrays and is reported in `upstreamErrors[]` while the other three series still serve.
- Cache: in-memory map keyed by `tenantId:range`, 10-minute TTL. `?refresh=true` bypasses and refills the cache.
- Aggregator and cache are constructor-injected (`buildTrendAggregator({ knex, cache, now })`, `buildTrendCache({ ttlMs, now })`) so tests can inject a fake knex and a deterministic clock.

### FN-1154 (subtask) — Predictive Insights widget
- Added `<app-predictive-insights>` standalone Angular component rendering 4 sparkline cards (load volume, maintenance, on-time %, fuel cost) with current value, delta vs. start of the 7-day window, and an inline SVG sparkline overlaying actual + predicted.
- Added `InsightsService` consuming `GET /api/insights/trends?range=7d` with types matching the FN-1152 contract exactly: each series is `{ actual: TrendPoint[], predicted: TrendPoint[] }` where `TrendPoint = { date: string; value: number | null }`. Top-level response includes `window`, `upstreamErrors`, and `cached`.
- Current/previous/delta/direction are derived **client-side** from `actual` (latest non-null vs. earliest non-null) — keeping the backend contract narrow and avoiding UI-domain knowledge in the aggregator.
- Wired the widget into the Control Center by importing `PredictiveInsightsComponent` in `app.module.ts` and embedding `<app-predictive-insights>` between the Daily Briefing and the dashboard grid in `dashboard.component.html`.
- Loading skeleton, error fallback with retry, and per-cell empty state ("Not enough data yet — collecting baseline.") satisfy the sparse-data acceptance criterion. Per-series `upstreamErrors` are surfaced as a "stale source" badge on the affected card.
- Component spec covers loading, success render of 4 cards, value formatting per unit, sparkline path generation, direction classification semantics, empty state, upstream-error badge, error fallback, refresh re-entrancy, and a11y labels.

## Key Decisions

### FN-1152 (subtask)
- **Direct DB access at the gateway** rather than HTTP fan-out to downstream services. Reasons: (a) the AC says "derives trends from existing tables", (b) there is no existing 7-day time-series endpoint on logistics/vehicles/integrations services, and (c) the briefing/ask predecessors already established a `goodmen-shared/config/knex` import path from the gateway via `routes/stripe.js`. The aggregator is constructor-injected so it remains test-friendly.
- **Linear regression for the `predicted` array** rather than a heavier ML approach. Suitable for a 7-day sparkline preview, predictable, and degrades cleanly to nulls under sparse data. Easy to swap behind the same interface later (e.g., when FN-1132 adds maintenance forecasting).
- **`upstreamErrors[]`** mirrors the briefing-aggregator contract so the frontend (FN-1154) can render a "stale source" badge per series without coupling to specific failure modes.
- **`?refresh=true`** override on the route follows the existing `/api/ai/briefing?refresh=true` pattern so the frontend has a uniform cache-bust knob across Control Center widgets.

### FN-1154 (subtask)
- **Inline SVG sparklines over chart.js**: 4 small inline `<svg>` paths are cheaper than mounting 4 chart.js canvases and avoid the lazy-load round-trip per card. Predicted segment is rendered as a dashed path so the visual difference between actual and forecast is unambiguous.
- **Standalone component, OnPush change detection**: matches the `DailyBriefingComponent` pattern (FN-1124). Imported into `AppModule.imports[]` like its sibling.
- **Current/previous/delta computed client-side from `actual[]`**: the backend contract carries time-series only — no precomputed scalars. Computing in the view keeps the API narrow and keeps "good vs. bad direction" (UI semantics) out of the aggregator.
- **Direction semantics encoded in the component**: "good direction" is data-domain-specific — `fuelCost` and `maintenance` going up is bad, `loadVolume` and `onTimePct` going up is good. This is a UI concern, not a server concern.
- **Per-series `upstreamErrors` badge**: when the backend reports a partial failure for one series, the affected card renders a "stale source" badge instead of failing the whole widget.

## Deployment Handoff

### FN-1152 (subtask)
- **Service to redeploy**: `fleetneuron-logistics-gateway` (only — gateway-local route, no microservice change).
- **Migrations**: none. All four queries read from existing tables (`loads`, `load_stops`, `work_orders`, `fuel_transactions`).
- **Env vars**: none new. Reuses existing `JWT_SECRET` and the gateway's existing access to `@goodmen/shared/config/knex` (which uses `goodmen-database/knexfile`).
- **Cache invalidation**: in-process map; entries expire after 10 minutes. No Redis involvement. Fresh deploy → empty cache (acceptable; first hit per tenant repopulates in <1s on typical data volumes).
- **Feature flags**: none.
- **Deploy order**: gateway only.
- **Post-deploy verification**:
  1. Hit `https://fleetneuron-logistics-gateway-dev.onrender.com/api/insights/trends?range=7d` with a valid bearer token; expect 200 with `series.{loadVolume,maintenance,onTimePct,fuelCost}` each having `actual` and `predicted` arrays of length 7.
  2. Repeat the call within 10 minutes; expect `cached: true`.
  3. Confirm a tenant with no data returns 200 with all-null series and `upstreamErrors: []` (no 5xx).

### FN-1154 (subtask)
- **Service to redeploy**: `fleetneuron-logistics-ui` — frontend bundle changed (new component + service + dashboard wiring + app module).
- **Migrations**: none.
- **Env vars**: none. `environment.apiUrl` already points at the gateway.
- **Cache invalidation**: none for the frontend. Gateway cache (10-minute TTL per tenant) is owned by FN-1152.
- **Feature flags**: none.
- **Deploy order (for the eventual FN-1126 PR, which bundles FN-1152 + FN-1154)**:
  1. Deploy `fleetneuron-logistics-gateway` (FN-1152 endpoint) **first** so the frontend has a backing endpoint.
  2. Deploy `fleetneuron-logistics-ui` (FN-1154 widget) **second**.
  If deployed in reverse order the widget renders its error fallback ("Trends unavailable right now…") with a retry button until the gateway is up — non-fatal, just visually noisy.
- **Post-deploy verification**:
  - Open Control Center as an authenticated user; widget appears between Daily Briefing and the metrics grid.
  - 4 cards render with values, deltas, and sparklines.
  - Refresh button works and shows the spinning state during the request.
  - For a tenant with no historical data, each card shows "Not enough data yet — collecting baseline." instead of erroring.
