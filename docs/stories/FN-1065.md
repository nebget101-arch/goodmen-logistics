# FN-1065: Auto-Create modal "has-file" state lacks visible affordances to replace file or add more PDFs

## Requirement
Follow-up bug to FN-1056 / FN-1057 (merged to dev as commit `806e074`). After clicking **Upload Rate Confirmation** on the Loads page and selecting one PDF, the Auto-Create modal renders only the filename, a "Ready for AI extraction" hint, and **Run Extraction** / **Close** buttons.

The hidden file `<input>` overlays the dropzone and the `(drop)` handler is wired, so multi-file selection logic from FN-1057 actually works — but no visible UI tells the user that. Users have no way to discover that they can replace the selected PDF or add more PDFs (which would route 2–10 selections to the bulk extraction flow).

This is a pure UI affordance bug. No backend / AI / DB / devops work.

## Acceptance Criteria
- [ ] In the "has-file" state, a visible **"Replace file"** affordance is present and reopens the file picker when clicked.
- [ ] In the "has-file" state, a visible **"Add more PDFs"** affordance OR an inline hint is present, making it discoverable that the dropzone still accepts more files.
- [ ] Adding a 2nd–10th PDF (via the new affordance OR via drop) closes the Auto-Create modal and opens the bulk flow with all selected PDFs preserved (existing FN-1057 logic — verify no regression).
- [ ] Optional **✕** button (if added) clears `autoPdfFile` and returns the dropzone to the empty state without closing the modal.
- [ ] The 1-PDF Auto-Create flow still works (no regression).
- [ ] No console errors. Existing unit specs pass.

## Out of Scope

- Backend / AI service / DB / devops.

## Root Cause Reference
File: `frontend/src/app/components/loads-dashboard/loads-dashboard.component.html`
Lines: ~2411–2416 on `origin/dev`

```html
<div class="ai-dropzone-content" *ngIf="autoPdfFile && !autoExtracting">
  <span class="material-symbols-outlined ai-file-icon">description</span>
  <p class="ai-dropzone-filename">{{ autoPdfFile.name }}</p>
  <p class="ai-dropzone-hint">Ready for AI extraction</p>
  <input type="file" accept="application/pdf" multiple
         (change)="onAutoFileSelected($any($event.target).files); $any($event.target).value = ''"
         class="ai-dropzone-input">
</div>
```

The hidden `<input>` overlays the dropzone via CSS, so clicking the file display *does* reopen the picker. The dropzone wrapper (lines 2398–2404) binds `(drop)`/`(dragover)`/`(dragleave)` and routes through `_handleAutoModalFiles`. All logic exists; only visible affordances are missing.

## Implementation Summary

### FN-1066 (subtask)
Replaced the implicit click-anywhere-to-replace behavior of the Auto-Create modal "has-file" state with three explicit, visible affordances:

1. **Replace file** — a `<label for="autoReplaceInput">` styled as a small AI-themed pill button. Triggers a single-file picker. Reuses `onAutoFileSelected(...)` and the existing `(change)` reset pattern (`$any($event.target).value = ''`) so the same file can be re-selected.
2. **Add more PDFs** — a second `<label for="autoAddMoreInput">` styled identically, opens a multi-file picker. Hits the same `onAutoFileSelected` handler, which already routes 2+ PDFs through `_handleAutoModalFiles` to the bulk extraction flow (FN-1057).
3. **✕ clear button** — a small absolute-positioned circular icon button in the top-right corner of the dropzone content. Calls a new `clearAutoPdf()` method that resets `autoPdfFile`, `autoError`, and `autoExtraction` to null without closing the modal.
4. **Inline bulk hint** — italic line below the actions: "Adding more PDFs (up to 10) switches to bulk extraction" — makes the bulk routing discoverable.

## Key Decisions

- **Dropped the `<input class="ai-dropzone-input">` overlay in the has-file state** rather than restyling it. The overlay's purpose was to make the entire dropzone reopen the picker on click — but that's the affordance bug we're fixing. Explicit labeled buttons are clearer and let us provide separate single-vs-multi pickers (`Replace` is single-file, `Add more` is multi-select). The dropzone's drag-and-drop handlers stay on the wrapper `<div>` and continue to work in this state.
- **Two separate hidden inputs** (`autoReplaceInput` and `autoAddMoreInput`) instead of one. This lets the buttons present different intents to the OS file picker (single vs. multi-select) and keeps the existing `_handleAutoModalFiles` logic untouched — both inputs ultimately funnel through `onAutoFileSelected`.
- **Used `display: none` on hidden inputs** rather than the visually-hidden absolute-positioning trick. Labels with `for=` still trigger the picker, and `display: none` avoids any layout interference inside the dropzone.
- **Did not modify** `_handleAutoModalFiles`, `onAutoFileDropped`, or any routing logic, per the out-of-scope guidance. The bulk-flow routing still happens via the existing length-check in `_handleAutoModalFiles`.
- **`clearAutoPdf()` does not call `closeAutoModal()`** — it only nulls the file/error/extraction state, returning the dropzone to its empty state, as the AC specifies.
