# FN-551: Equipment Owner & Effective Dates not persisted on Pay Routing save

## Requirement
Bug fix: payee assignment effective dates and equipment owner data are not persisted correctly when saving from the driver edit page.

## Acceptance Criteria
- [ ] POST payee-assignment/resolve stores effective_start_date and effective_end_date from request body
- [ ] GET payee-assignment returns effective dates in the response
- [ ] Previous open-ended assignments are end-dated when a new assignment is created (no overlapping open assignments)

### FN-559 (subtask)
**Audit findings:**
1. `driver_payee_assignments` table schema: Has `effective_start_date` (NOT NULL) and `effective_end_date` (nullable) — correct
2. POST `/resolve` (line 622): Was writing effective dates correctly, but was not end-dating previous open assignments — **fixed**
3. POST `/payee-assignments` (line 585): Same issue — **fixed**
4. GET `/payee-assignment` (line 558): Returns full row from `getActivePayeeAssignment` which includes all columns — **correct, no change needed**

**Fix applied:** Both POST endpoints now end-date any existing open assignment (`effective_end_date IS NULL`) for the driver before inserting the new one. This prevents overlapping assignments and ensures `getActivePayeeAssignment` always returns the correct latest assignment.

## Files Changed
- `backend/packages/goodmen-shared/routes/settlements.js` (POST payee-assignments + POST resolve)

## Key Decisions
- End-dating uses the new assignment's start date as the old assignment's end date (clean handoff)
- Only open-ended assignments (NULL end date) are end-dated — pre-existing bounded assignments are left unchanged
