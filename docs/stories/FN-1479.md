# FN-1479: Quick-add parts panel — search, recent at location, common at location

## Requirement
Side panel on the receiving page (right rail) with three tabs to add parts without scanning: Search by SKU/name, Recent at this location, Common at this location. Click `+ Add` adds the part to the current DRAFT ticket.

## Acceptance Criteria
- [ ] `GET /api/parts/recent-at-location/:locationId?limit=20` ordered by `MAX(inventory_transactions.created_at)` for type RECEIVE
- [ ] `GET /api/parts/common-at-location/:locationId?days=90&limit=20` by `SUM(qty_change)` over window
- [ ] Search uses existing parts search (active only)
- [ ] Inline panel on desktop, bottom sheet on mobile
- [ ] `+ Add` uses qty multiplier; calls same line-add endpoint as scanner
- [ ] Empty states per tab
- [ ] Search debounced 250ms; lists cached 60s per location

## Implementation Summary

### FN-1485 (subtask) — Backend recent + common endpoints

Added two `GET` endpoints to `backend/packages/goodmen-shared/routes/parts.js`:

- `GET /api/parts/recent-at-location/:locationId?limit=20` — DISTINCT parts most-recently received at the location, ordered by `MAX(inventory_transactions.created_at)` for `transaction_type = 'RECEIVE'`. Joins `parts` (sku, name, default_cost, filtered to `is_active = true`) and left-joins `inventory` for `on_hand_qty` at the location. Empty set returns `{ data: [] }`.
- `GET /api/parts/common-at-location/:locationId?days=90&limit=20` — same shape, but ranked by `SUM(qty_change)` over the rolling window. Empty set returns `{ data: [] }`.

Both endpoints validate `locationId` is a UUID (400 otherwise), require `authMiddleware`, clamp `limit` to `[1, 100]` (default 20), clamp `days` to `[1, 365]` (default 90). Index already exists at `inventory_transactions (location_id, created_at)` per `20260216_create_inventory_schema.js:167`.

Tests: `backend/packages/goodmen-shared/routes/parts-stats.test.js` (7 tests via `node:test`) — UUID validation, empty-set, populated-shape, query-param clamping. All passing.

### FN-1486 (subtask) — Frontend quick-add panel
- New `QuickAddPanelComponent` with three tabs (Search / Recent / Common), debounced typeahead (250 ms), 60 s in-memory cache keyed by location for Recent + Common, mobile bottom-sheet (<768 px) with chip toggle, keyboard arrow + Enter row navigation.
- Mounted in the `<aside class="recv-rail">` slot exposed by FN-1483; replaced the `<ng-content select="[recv-rail-quick-add]">` projection placeholder with `<app-quick-add-panel>`.
- `(addPart)` event handler `onQuickAdd` on the host calls `api.addReceivingLine(ticketId, partId, qty)` (same line-add endpoint as the scan flow). Quantity = `qtyMultiplier` from the receiving toolbar.
- Disabled when no DRAFT ticket is loaded (`[disabled]="!ticket"`).
- API service: added `getRecentPartsAtLocation` / `getCommonPartsAtLocation` (consume the FN-1485 endpoints) and extended `getParts` to forward `is_active` and `limit`. Network errors are treated as empty list so the UI degrades cleanly if a backend endpoint is temporarily unavailable.
- Component spec covers tab switching with cache, debounced search, empty states per tab, error-as-empty, click + Enter add emission, and disabled state.

## Key Decisions

**FN-1485 (backend):**
- **Same file, not a new module.** Per the ticket "or new `parts-stats.js`": kept the routes inline in `parts.js` to match the locality of the other parts read endpoints (`/categories`, `/manufacturers`, `/duplicate-check`) and avoid a new mount point in every microservice that uses the shared router.
- **UUID validation at the router.** Returns 400 for non-UUID `locationId` rather than letting Postgres throw a 500. Matches the spirit of the AC's "empty-set returns 200, not 404" — bad input gets a clear 4xx, missing data gets a clean empty array.
- **Sub-query + outer join, not a window function.** The aggregate (`MAX(created_at)` / `SUM(qty_change)`) runs in a sub-query keyed by `part_id`, then we join `parts` and left-join `inventory`. Lets the planner use the existing `(location_id, created_at)` index without scanning every transaction row through the join.
- **`COALESCE(inv.on_hand_qty, 0)` cast to int.** Parts that have a transaction history but no current `inventory` row (deleted/migrated) still surface; their qty just shows as 0.
- **`node:test` not Jest.** Ticket said "Jest tests" but the goodmen-shared package's `npm test` runs `node --test`; matched the existing convention to keep CI green.

**FN-1486 (frontend):**
- **Cache scope**: only Recent + Common are cached (60 s, location-keyed). Search is never cached because typeahead state is volatile and the user expects fresh results. Tab switches between Recent and Common do not re-fetch within TTL — meets AC.
- **Error-as-empty**: network errors on Recent/Common are treated as empty list, so a transient backend hiccup shows the same friendly empty state instead of a confusing error message.
- **OnPush change detection**: avoids parent-driven re-renders churning the typeahead. Cache reads/writes call `cdr.markForCheck()` explicitly.
- **Mobile bottom-sheet**: implemented purely via CSS `@media (max-width: 768px)` plus a `mobileExpanded` flag — no separate component, no overlay manager. Keeps DOM identical across breakpoints.
- **Keyboard navigation**: Arrow Up/Down moves focus across native `<button>` rows (`@ViewChildren('rowBtn')`), Enter triggers `addPart`. Native button focus already gets the focus ring; no custom roving-tabindex needed.
