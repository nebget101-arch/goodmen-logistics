# FN-1080: Bulk extraction grid auto-runs on open with no review/confirm step

## Requirement
The `BulkExtractionGridComponent` calls `startProcessing()` from `ngOnInit`, so dropping 2+ PDFs anywhere on the dashboard fires AI extraction immediately with no chance to review, remove, or replace files. Combined with `onBulkFilesSelected` silently appending instead of replacing, users believe their second drop replaced the first but actually triggered extraction of both.

Add an explicit "Review Files" pre-flight panel before extraction starts, and clarify the replace-vs-add affordance in the bulk-upload modal.

## Acceptance Criteria
- [x] After dropping 2+ PDFs anywhere on the dashboard, the user sees the file list with edit/remove controls and an explicit "Start extraction" button before any AI call fires.
- [x] In the bulk-upload modal, dropping a second time after files are queued does not silently append — the user clearly sees they're adding (or chooses replace).
- [x] No regression in the single-PDF auto-create flow (`onHeroSinglePdf` → `showAutoModal`).
- [x] No regression in `BulkExtractionGridComponent` happy path: clicking "Start extraction" still extracts and creates loads for all queued PDFs.

## Implementation Summary

### FN-1083 (subtask)

**Review gate in `BulkExtractionGridComponent`:**
- `ngOnInit` no longer calls `startProcessing()`. It only builds the `rows` array from the input files. The grid opens in a new "review" phase.
- Added a derived getter `inReview` (`!processing && !completed`) so existing template guards continue to work without introducing a separate `phase` enum.
- New review-panel template renders each queued file with a per-row remove button, an inline drag-drop dropzone for adding more PDFs (capped at 10), and a `Start extraction` primary action in the footer. Cancel in the footer remains and closes the modal without firing any AI call.
- Removing the last queued file emits `close` to dismiss the empty modal.
- Added `BulkExtractionGridComponent.MAX_FILES = 10` constant; `addReviewFiles` enforces the cap and surfaces an inline `reviewNotice`.

**Replace-vs-add in the bulk-upload modal (chose option (b) — inline confirm):**
- New `bulkPendingAddFiles: File[]` state holds the second selection until the user picks a path.
- `onBulkFilesSelected` no longer silently appends. If `bulkPdfFiles.length > 0`, the new selection is parked in `bulkPendingAddFiles` and the dropzone is replaced by an inline prompt offering **Replace queue / Add to queue / Cancel**. First-time selections still go straight into the queue (no prompt).
- Refactored the queue-append logic into a private `appendBulkFiles(pdfs)` helper shared by the initial path and the new `confirmBulkAdd()` action. `confirmBulkReplace()` clears `bulkPdfFiles` first, then appends. `cancelBulkPending()` discards the pending selection.
- Dropzone copy updated from "Add more PDFs" → "Add more PDFs — you'll be asked whether to replace or add".

**`onPageDrop` (step 3, optional):** intentionally NOT modified. The existing path opens `BulkExtractionGridComponent`, which now starts in review phase, so the user already gets the explicit confirmation step. Restructuring the page-drop handler to detour through the bulk-upload modal would be invasive and add no behavior beyond what the new review gate already provides.

## Key Decisions

- **Replace-vs-add: chose option (b) inline confirm prompt over (a) split dropzone.** Splitting the dropzone would require restructuring the existing `.bulk-ai-dropzone` SCSS (single rounded zone with conditional copy/icon) into two flex children, retuning the empty-state, and threading a new `bulkAppendMode` boolean through every drop/file-input handler. The inline prompt is additive: a new conditional block above the dropzone, three new component methods, no changes to the existing dropzone CSS or to the empty-state UX. The first-selection path is unchanged (no prompt), so the modal still feels frictionless until ambiguity actually arises.
- **`onPageDrop` left alone (step 3 was optional).** The bulk-extraction grid now opens in review phase, so the page-drop path already lands the user on an explicit confirm step. Routing the page drop through the bulk-upload modal instead would be a larger UX change without additional safety benefit.
- **Reused `processing`/`completed` rather than introducing a `phase` enum.** Added a derived `inReview` getter. This keeps the existing `processing`/`completed` template guards untouched (lower regression risk) while still giving the template a clear flag for review-only blocks.
- **Added `BulkExtractionGridComponent.MAX_FILES = 10` static constant** — single source of truth shared by the in-component cap logic and the templated max-reached notice. Matches the limit already enforced in `loads-dashboard.component.ts:openBulkExtractionGrid` and `onBulkFilesSelected`.
- **Spec file added but cannot be executed locally.** This Angular workspace's `angular.json` has only `build` / `serve` targets — no `test` target is configured (matches every other adjacent `.spec.ts` in the repo, all of which are unwired). The spec was added per the ticket's "Spec coverage" requirement and follows the structure/imports of the existing `load-detail-drawer.component.spec.ts`. `npx ng build --configuration=development` succeeds with no errors in the changed files.

### Manual Verification (run on dev after deploy)
1. Open the loads dashboard. Drag-drop 2 PDFs anywhere on the page → bulk extraction grid opens in **Review Files** state. Confirm the file list shows both PDFs with remove buttons and that **no AI/network call fires** (check Network tab — no `/api/loads/ai-extract` request).
2. In the review panel, drag a 3rd PDF onto the "Drag & drop more PDFs here" zone → it appears in the list with a "Added 1 file(s)." notice. Click the X next to one file → it's removed.
3. Click **Start extraction** → grid switches to processing view; `/api/loads/ai-extract` requests fire sequentially. After completion, the summary screen renders as before.
4. Open the **Bulk Upload Rate Confirmations** modal (hero CTA → Bulk). Drop 1 PDF; the file list shows 1 queued. Drop another PDF or click the dropzone and pick more → instead of silent append, the **Replace queue / Add to queue / Cancel** prompt appears. Pick **Replace queue** → the queue swaps to the new selection. Re-trigger and pick **Add to queue** → both selections are queued.
5. Single-PDF auto-create flow: drop **one** PDF anywhere on the dashboard. The auto-create modal (`showAutoModal`) opens unchanged — the bulk grid does NOT open (existing `onPageDrop` requires `>= 2`).
6. Cancel during review (Cancel button or backdrop) → modal closes, no extraction fires.
