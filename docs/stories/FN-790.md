## Implementation Notes

### FN-811 (subtask) — Backend WS infra + load CRUD emits

Branch: `backend/FN-811/websocket-infra`

**Architecture**

- **Gateway owns the Socket.IO server.** `backend/gateway/index.js` now wraps the
  Express app in an `http.Server`, attaches Socket.IO on path `/socket.io`, and
  authenticates each handshake with the existing JWT secret. On connect the
  socket joins `tenant:<tenantId>`.
- **JWT now carries `tenant_id`.** `routes/auth.js` login resolves the default
  tenant for the user (via `tenantContextService.getDefaultContextForUser`) and
  includes it in the signed JWT. This removes the need for the gateway to hit
  the database during the WS handshake.
- **Shared emit helper.** `backend/packages/goodmen-shared/services/websocket.service.js`
  exposes `emitToTenant({ tenantId, event, payload })`. It fire-and-forgets a
  POST to the gateway's `/internal/ws/emit` endpoint (authed by
  `INTERNAL_WS_SECRET`). The helper never throws, never blocks the caller.
- **Internal emit endpoint on gateway.** `POST /internal/ws/emit` validates the
  `X-Internal-Token` header and broadcasts the body to `tenant:<tenantId>`.
  Rejects with 401 on bad token, 503 when WS is not initialized.

**Load CRUD events (rolled up from FN-812)**

`routes/loads.js` now emits after successful commits:

| Endpoint | Event | Payload |
|---|---|---|
| `POST /api/loads` | `load:created` | full load detail |
| `PUT /api/loads/:id` | `load:updated` | full load detail |
| `PUT /api/loads/:id` (status changed) | `load:status_changed` | `{ id, previousStatus, status, load }` |
| `DELETE /api/loads/:id` | `load:deleted` | `{ id }` |
| `PATCH /api/loads/:id/approve-draft` | `load:status_changed` + `load:updated` | `{ id, previousStatus: DRAFT, status: DISPATCHED, load }` |
| `POST /api/loads/bulk-update` | `load:updated` (per id) + `load:status_changed` (per id when status changed) | `{ id, changes }` / `{ id, status }` |
| `POST /api/loads/bulk-delete-drafts` | `load:deleted` (per id) | `{ id }` |

All emits happen **after** the commit path; a rolled-back transaction emits nothing.

**Envelope shape** (all events):

```json
{
  "tenantId": "<uuid>",
  "event": "load:created",
  "payload": { /* load or { id } */ },
  "emittedAt": "2026-04-23T00:00:00.000Z"
}
```

**Tests**

`backend/packages/goodmen-shared/services/websocket.service.test.js` — 4 tests
covering missing args, missing bridge env, envelope shape, and null-payload
normalization. All shared service tests pass (69/69).

### FN-813 (subtask) — Frontend WS client + loads-dashboard integration

Branch: `frontend/FN-813/ws-client-service`

**Client architecture**

- **Service**: `frontend/src/app/services/websocket.service.ts` — a singleton
  `WebsocketService` that wraps `socket.io-client`. It pulls the JWT from
  `localStorage.getItem('token')` and connects to the gateway origin derived
  from `environment.apiUrl` on path `/socket.io`. The handshake passes the
  token via `auth.token`, matching `backend/gateway/index.js` (FN-811).
- **Connect trigger**: `AppComponent.ngOnInit` calls `websocket.connect()`
  when `isLoggedIn()` is true. `logout()` calls `websocket.disconnect()` so
  the session doesn't leak across users on a shared device.
- **Reconnect**: delegated to socket.io-client's built-in reconnect manager
  (`reconnectionDelay: 1000`, `reconnectionDelayMax: 30000`, capped at
  `MAX_RECONNECT_ATTEMPTS = 5`). After the cap is hit the service emits
  `status = 'disconnected'` and starts a 30s polling loop via
  `pollTick$`. Subscribers re-run their own list fetch on each tick.
- **Event API**: `on<T>('load:created')` returns a shared RxJS Subject per
  event name. Socket listeners are attached lazily the first time a component
  subscribes, then re-bound to the new socket on reconnect so listeners
  survive a transport upgrade.
- **NgZone**: socket callbacks run outside Angular via `runOutsideAngular` to
  avoid a CD pass on every heartbeat; `zone.run` is used when pushing into
  `status$` / event subjects so templates re-render.

**Loads dashboard integration**

- **Subscriptions** wired in `ngOnInit` (and torn down via `takeUntil(destroy$)`):
  - `load:created` → prepends to `loads[]`, total++, row gets
    `.row-just-created` for 1.5s (fade-in highlight).
  - `load:updated` and `load:status_changed` → shallow-merges into the
    existing row, marks `.row-just-updated` for 1.2s (subtle pulse).
  - `load:deleted` → marks `.row-just-deleted` (fade-out for 400ms), then
    splices out of `loads[]` and decrements `total`.
  - `presence:join` / `presence:leave` → maintains `presenceUsers[]`. Only
    joins with `page === 'loads'` are accepted to keep the stack focused.
  - `websocket.pollTick$` → triggers `loadLoads()` so the list still
    refreshes when we fall back to polling.
- **Animations**: CSS-only, defined in the component SCSS. Keyframes live at
  the top level so the animations don't re-trigger on CD passes. Row
  animations target `<td>` (not `<tr>`) so the virtual-scroll viewport
  doesn't bounce on height changes.
- **In-transit indicator**: `.in-transit-dot` pulses green, and an
  `.in-transit-eta` pill shows a short countdown (`delivery_date - now`)
  whose label is refreshed every 60s by an `etaTick` counter incremented
  from a single `setInterval`.
- **Presence**: stacked circular avatars (`.realtime-presence__avatar`) in
  the page header with a "+N" more chip after 4. Initials fall back when no
  `avatarUrl` is provided.
- **Connection indicator**: a 10px dot (`.realtime-status--connected`,
  `--reconnecting`, `--disconnected`) + a "Reconnecting to live updates…"
  banner during backoff.
- **Presence announcement**: the dashboard `emit('presence:join', { page: 'loads' })`
  on mount and `presence:leave` on destroy. The backend does **not** yet
  fan-out presence (see below), so the UI silently tolerates the absence.

**Acceptance criteria status**

- [x] Connection + reconnect works — socket.io's reconnect manager handles
      exponential backoff (1s → 2s → 4s → 8s → 16s, jitter ±25%, capped 30s).
- [x] Polling fallback engages after N failed reconnects
      (`MAX_RECONNECT_ATTEMPTS = 5`, then 30s `pollTick$` refresh).
- [x] Connection status observable emits correct values
      (`connected` / `reconnecting` / `disconnected`).
- [x] Events reflect on the list without manual refresh
      (load:created / updated / status_changed / deleted).
- [x] Animations are CSS-only and bounded so they stay smooth in DevTools.
- [x] Connection indicator always accurate (wired to `status$`).
- [~] Presence avatars update within 2s of join/leave — **client-side wired;
      server-side presence broadcast is not in FN-811 scope yet. UI is ready
      for the backend emission; no change needed on the client when it lands.**
