# FN-1405: Wire Bins/Users/Supply-Rules tabs into Edit Location modal (replace placeholders)

## Requirement
Replace the three placeholder blocks in `location-edit-dialog.component.html` with the existing tab components so users can actually manage bins, user assignments, and supply rules from the Edit Location modal.

## Background — Jira/code drift
FN-682 (parent Story for the locations admin), FN-699 (Bins tab), FN-700 (Users tab), FN-701 (Supply Rules tab) are all marked **Done** in Jira, and the standalone tab components do exist in the repo:
- `frontend/src/app/components/locations-admin/bins-tab/bins-tab.component.{ts,html,scss}` — full CRUD + bulk-create dialog
- `frontend/src/app/components/locations-admin/location-edit-dialog/users-tab/users-tab.component.{ts,html,css}` — assign/remove with autocomplete
- `frontend/src/app/components/locations-admin/location-edit-dialog/supply-rules-tab/supply-rules-tab.component.{ts,html,scss}` — full CRUD with primary/auto-replenish

But `location-edit-dialog.component.html` on `origin/dev` still hard-codes "Coming in a future update" placeholders for all three tabs (lines 254–279). The wire-up step was missed when those tickets shipped. Backend APIs (`location-bins.js`, `user-locations.js`, `warehouse-supply-rules.js`) and DB tables (`location_bins`, `user_locations`, `warehouse_shop_supply_rules`) are all in place — no backend or DB work needed.

## Implementation Summary
The three placeholder blocks in `location-edit-dialog.component.html` (lines 254–279) were replaced with the existing tab component selectors:

- `<app-bins-tab>` bound to `[locationId]="location?.id || ''"` and `[locationType]="locationType"`.
- `<app-users-tab>` bound to `[locationId]="location?.id || ''"`.
- `<app-supply-rules-tab>` bound to `[locationId]="location?.id || null"` and `[locationType]="locationType"`.

`locations-admin.module.ts` already declares all three components, so no module changes were required. The dialog's existing `showBinsTab` / `showSupplyRulesTab` getters continue to gate tab visibility for non-WAREHOUSE/SHOP types (those tab buttons remain `[disabled]` in the tab bar).

For the Add-mode case (no `location.id`), the children receive an empty string / null for `locationId`. Each tab component already short-circuits its data load when `locationId` is falsy, so the panes render an empty state until the location is saved.

### FN-1407 (subtask)

- Verified production build compiles cleanly for the changed components (pre-existing baseline errors in unrelated components — Billing, Drivers, Vehicles, Reports, InvestigationPanel — are not introduced by this change).

## Key Decisions
- **Treating prior FN-682/699/700/701 as Done and not reopening them.** Those tickets shipped the components but skipped the wire-up. Reopening four closed tickets would be noisy; instead this Story explicitly carries the wire-up as the remaining work.
- **No backend/DB work.** APIs and schema for bins, user-locations, supply-rules are all live on `dev`.
- **Bind nulls/empties for Add-mode `locationId`**, relying on each child's existing `if (!locationId) return;` guards rather than wrapping the tab bodies in `*ngIf="isEditMode"`. Keeps the tab bar consistent with current behavior; users see empty data and are expected to save first.
- **No `.ts` or module changes needed.** Selectors + inputs match what the ticket described, and `locations-admin.module.ts` already declares all three tabs — the wire-up was a single template change.
