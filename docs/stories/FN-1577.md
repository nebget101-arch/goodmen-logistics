# FN-1577: [Work Orders Hub] "Assigned To" mechanics filter dropdown shows raw UUIDs instead of First + Last name

## Requirement
On `/work-orders`, the **Assigned To** filter dropdown shows technician UUIDs (e.g. `f7781c4b-7d37-4370-9282-676c564d5017`) instead of "First Last". Each option must show the user's First + Last name, with `username` as fallback when names are missing. UUIDs should never be the visible label.

## Acceptance Criteria
- [x] Open the Work Orders Hub → click "Assigned To" → every option shows "First Last" (or `username` fallback). No UUIDs.
- [x] Selecting a mechanic still correctly filters the WO list (the option's underlying `value` remains the mechanic's `id`).
- [x] No regression on other pages that consume `getDrivers()` (Loads, Driver Hub, etc.).

## Implementation Summary
Frontend-only fix. The `GET /api/drivers` response shape is camelCase (the route runs every row through `transformRows()` in `backend/packages/goodmen-shared/utils/case-converter.js`, which maps snake_case columns → camelCase keys). The maintenance component's `loadDriversAsMechanics()` mapper was reading `d.first_name` / `d.last_name` (snake_case), which resolve to `undefined` against the camelCase payload, so the join produced an empty string and the `|| d.id` fallback rendered the raw UUID.

Updated the mapper to read `d.firstName` / `d.lastName` and added `d.email` as an intermediate fallback before `d.id`. Tightened the inline TS types into a single `MechanicRow` alias to remove duplication.

## Files Changed
- `frontend/src/app/components/maintenance/maintenance.component.ts` — fixed `loadDriversAsMechanics()` to read camelCase keys (`firstName`/`lastName`); added `email` fallback before `id`; consolidated inline types into a `MechanicRow` alias.

## Key Decisions
- **Fix path: frontend, not backend.** The bug ticket listed two possible paths and asked the implementing agent to confirm which applies. Inspection of `backend/packages/goodmen-shared/routes/drivers.js` confirmed the legacy/default `GET /api/drivers` query already selects `d.*` from the `drivers` table (which has NOT NULL `first_name` and `last_name`) and pipes the result through `transformRows()`. So the BE response already contains `firstName`/`lastName` — the bug is the FE reading the wrong key shape. Fixing the BE was unnecessary and would have introduced an inconsistent dual-shape contract for other consumers that already use camelCase.
- **Agent reassignment: `agent:backend` → frontend.** The Jira label `agent:backend` was set on the assumption the BE was missing columns. Since the actual change lives entirely in `frontend/src/...`, the branch uses the `agent/frontend/...` prefix to accurately reflect the diff. Story doc updated; Jira label left as-is for historical context.
- **Username fallback → email fallback.** The `drivers` table schema has no `username` column (it has `first_name NOT NULL`, `last_name NOT NULL`, `email UNIQUE NOT NULL`). The bug's "username" mention was a generic placeholder. In the unlikely event both names are blank on a driver row, the dropdown now falls back to `email`, which is always present and human-readable. The final `d.id` fallback is preserved as a defensive last resort.
- **No changes to other `getDrivers()` consumers.** Audited the eight other call sites (Loads, Driver Hub, Tolls, Settlements, Fuel Cards, FMCSA Inspections); none were touched. The fix is local to the maintenance component, so regression risk on other pages is zero.
