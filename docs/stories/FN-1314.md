# FN-1314: Reports narrative API returns 403 AI_FORBIDDEN ("Cannot verify bearer token") for all users

## Symptom

Admin (and any other allowed role) calling `POST /api/ai/reports/:reportKey/narrative` receives:

```json
{ "success": false, "error": "Cannot verify bearer token", "code": "AI_FORBIDDEN" }
```

The narrative panel on report-view never renders â€” it stays in the silent error-hide state from FN-1121.

## Root Cause

`backend/microservices/ai-service/src/handlers/reports-narrative-handler.js#resolveUser` (lines 104-129) tries:

1. Use `req.user` if the gateway attached it.
2. Otherwise, locally `jwt.verify` via `jsonwebtoken`.

Both paths fail:

1. The gateway's `buildProxy` (`backend/gateway/index.js:300`) is plain `http-proxy-middleware` â€” it forwards headers but never decodes JWT or sets `req.user` for `/api/ai/*`. Other services (e.g. `reporting-service/middleware/auth-middleware.js`) attach `req.user` themselves.
2. `jsonwebtoken` is **not** in `backend/microservices/ai-service/package.json` â€” `tryRequireJwt()` returns `null`, and the handler falls through to `403 AI_FORBIDDEN: "Cannot verify bearer token"` (line 118).

## Implementation Summary

FN-1315 added `jsonwebtoken: ^9.0.3` to `ai-service` so the narrative handler's local-verify fallback (`tryRequireJwt()`) succeeds. The gateway proxies `/api/ai/*` without decoding the JWT, so handlers must verify it themselves; previously the lazy `require('jsonwebtoken')` returned `null` and every request fell through to `403 AI_FORBIDDEN: "Cannot verify bearer token"`.

The handler logic itself was already correct â€” it reads `payload.role` (matching `reporting-service/middleware/auth-middleware.js`'s `decoded.role`) and routes through `hasAllowedRole`. The fix is purely the missing dependency. New unit tests exercise the local-verify path end-to-end (positive admin JWT, all four allowed roles, plus negative cases for disallowed role, expired token, bad signature, JWT without role claim).

## Key Decisions

- **Pinned `^9.0.3`**: matches `reporting-service/package.json` exactly. Keeps the JWT verification surface consistent across services.
- **Did not touch the handler code**: the existing `resolveUser` path (line 104-129) already handles `decoded.role` correctly via `auth.user.role`. Adding the dep alone was sufficient.
- **Did not move JWT decoding to the gateway**: kept scope tight per subtask description. Future story can centralize this.
- **Test secret matches handler fallback (`dev_secret`)**: the handler uses `process.env.JWT_SECRET || 'dev_secret'`, and tests mirror that fallback so they remain hermetic without env setup.
- **Added negative tests beyond the strict ask**: the AC explicitly lists "missing/expired/invalid token, disallowed role"; only "missing" and "invalid bearer" were already covered. Added expired-token, bad-signature, and missing-role-claim cases to lock down the contract.
