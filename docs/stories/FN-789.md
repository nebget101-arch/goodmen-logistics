## Implementation Summary

### FN-816 (subtask)

**Migration** — `backend/packages/goodmen-database/migrations/20260423043941_add_ai_metadata_to_loads.js` adds a nullable `jsonb` column `ai_metadata` to the `loads` table plus a `GIN` index on that column.

**Guarded + idempotent** — Matches the style of the sibling `20260401100000_add_settlement_ai_insights.js` and `20260418140000_add_loads_needs_review.js` migrations: `hasTable('loads')` and `hasColumn('loads', 'ai_metadata')` guards let the migration re-run cleanly on environments that were manually fixed. The index is created via `CREATE INDEX IF NOT EXISTS` so a rerun after a partial failure doesn't crash.

**GIN index for field-name lookups** — Stored payload has a nested `fields` object keyed by load field name. A GIN index on the whole `ai_metadata` column supports containment and key-exists operators (`?`, `?|`, `@>`) so queries like `WHERE ai_metadata->'fields' ? 'broker_name'` or `WHERE ai_metadata @> '{"fields":{"broker_name":0.98}}'` are indexable without per-subfield expression indexes.

**Down migration** — Drops the index first (`DROP INDEX IF EXISTS idx_loads_ai_metadata`), then the column. Safe to run on environments that never had the column.

**Existing rows unaffected** — Column is nullable with no default; existing loads (including AI-created loads that pre-date this migration) remain NULL and the backend (FN-817) can treat NULL as "no confidence available" without a backfill.

### FN-817 (subtask)

**`buildAiMetadata(data, sourceDocument)` helper** — Added to `backend/packages/goodmen-shared/services/load-ai-extractor.js`. Transforms the existing extractor payload (`{ confidence: { brokerName, poNumber, rate, pickup, delivery }, overall_confidence, overall_confidence_tier }`) into the JSONB shape documented in the FN-816 migration:

```json
{
  "overall_confidence": 0.83,
  "overall_confidence_tier": "yellow",
  "extracted_at": "2026-04-23T…Z",
  "source_document": "ratecon.pdf",
  "fields": { "brokerName": 0.97, "rate": 0.83, "pickup": 0.9, "delivery": 0.91 }
}
```

Returns `null` when the payload carries no confidence signal so callers skip the column write and leave `ai_metadata` NULL (preserves the "never AI-extracted" signal for pre-existing rows). Non-finite values (NaN/Infinity) are dropped so corrupted model output cannot poison the JSONB column. Field keys mirror the extractor's `confidence` object names (camelCase) — this is the contract the PUT handler and frontend (FN-818) read from.

**Persist on AI-created loads (two write paths)**
- `backend/packages/goodmen-shared/routes/loads.js :: processSingleRateConfirmation` — after the load-insert transaction commits, a separate `UPDATE loads SET ai_metadata = $1::jsonb WHERE id = $2` writes the payload. Wrapped in try/catch so a missing column (migration lag) logs `loads_ai_metadata_write_failed` instead of failing the load creation. This covers both the single-upload `/api/loads/ai-extract` → create flow and the bulk `/api/loads/bulk-rate-confirmations` flow (the bulk handler uses the same helper for each file).
- `backend/microservices/integrations-service/services/inbound-email-service.js :: createDraftLoadFromExtraction` — adds `ai_metadata` to the knex insert payload guarded by `hasColumn('loads', 'ai_metadata')` (matches the existing `hasColumn('loads', 'source')` pattern). `JSON.stringify`'d so knex sends a JSONB literal rather than a Postgres row/array.

**API responses include `ai_metadata`**
- `GET /api/loads` (list) — added `l.ai_metadata` to the SELECT column list (`listLoads` / `dataSql` in `loads.js`). Null for legacy rows; populated for AI-created rows once this change deploys.
- `GET /api/loads/:id` (detail) — `getLoadDetail` already selects `l.*`, so `ai_metadata` flows through automatically once the column exists. No code change needed.

**Clear per-field confidence on user edit (FN-817 AC#3)** — In `PUT /api/loads/:id`, after the transaction commits, the handler walks the request body and for each edited field that has a confidence entry it strips the corresponding key from `ai_metadata.fields` via `ai_metadata = ai_metadata #- $1::text[]` with the JS parameter `['fields', key]`. Mappings:

| Body key edited | Field cleared from `ai_metadata.fields` |
|-----------------|-----------------------------------------|
| `brokerName` | `brokerName` |
| `brokerId` (broker swap is an explicit broker confirmation) | `brokerName` |
| `poNumber` | `poNumber` |
| `rate` | `rate` |
| `stops` / `pickup` / `delivery` (any stops edit) | `pickup` and `delivery` |

The per-key clears run against the live load (`AND ai_metadata IS NOT NULL`) so legacy rows stay untouched, and the whole block is wrapped in a try/catch that logs `loads_ai_metadata_clear_failed` — a clear-path failure never breaks the primary load update.

**Why post-commit rather than in-transaction** — If the `ai_metadata` column does not yet exist in the target environment, a failing UPDATE inside the transaction would roll back the load create/update. Running the metadata statement *after* the commit means the core mutation is durable even when the migration lags the code deploy (the `loads_ai_metadata_write_failed` warning surfaces the discrepancy in ops logs).

**Tests** — `backend/packages/goodmen-shared/services/load-ai-extractor.test.js` (new, 9 cases) covers:
- `confidenceTier` banding (green/yellow/red).
- `computeOverallConfidence` (weakest-field semantics).
- `buildAiMetadata`: null/empty input handling, canonical field-key filtering, NaN/Infinity rejection, explicit-tier precedence, and `source_document` fallback.

The test stubs `axios` and `pdf-parse` via `Module._resolveFilename` so it runs standalone without needing the microservice-host `node_modules` (goodmen-shared does not install those deps directly).

## Key Decisions

### FN-816
- **Single whole-column GIN index instead of expression indexes per subfield.** The payload's `fields` shape evolves as new AI-extractable fields are added; a whole-column GIN index absorbs future keys without a schema change. Cost: slightly larger index than, say, `GIN ((ai_metadata->'fields'))`, but negligible at expected row counts.
- **Nullable with no default.** A JSONB default (`{}` or `null`) would either (a) mask the "never AI-extracted" signal the backend/frontend need, or (b) force a table-wide rewrite on PostgreSQL versions that materialize the default. Nullable is the cheapest, clearest choice.
- **Guards (`hasColumn`/`hasTable`) match sibling migrations.** Matches `settlement_ai_insights` and `loads_needs_review` patterns so the migration is safe to re-run after partial failures and on non-prod environments with manually-applied schema changes.

### FN-817
- **camelCase field keys (`brokerName`, `poNumber`, `rate`, `pickup`, `delivery`) in `ai_metadata.fields`.** These match the extractor's existing `confidence` object and the PUT endpoint's `fieldMap` body keys. Using snake_case (the FN-816 docstring example used `broker_name`) would require a translation layer at every read/write site; the migration doesn't enforce key names, so we standardize on the shape the code already produces. Exported as `AI_METADATA_FIELD_KEYS` so any future field additions happen in one place.
- **Post-commit metadata writes, not in-transaction.** The column is guaranteed to exist in steady state, but during a rolling deploy the backend change can land a few seconds before the migration finishes. Running the metadata UPDATE after COMMIT keeps the primary load insert durable and turns schema drift into a warn-level log instead of a 500 for end users.
- **`brokerId` edit also clears `brokerName` confidence.** Swapping the resolved broker is an explicit manual confirmation of the broker field by the dispatcher, so the sparkle should come down even though the extracted `brokerName` string didn't change. Explicit entry in the mapping, not an accident.
- **Any stops edit clears both `pickup` and `delivery`.** The two confidence values are derived from the first-pickup / last-delivery stops respectively. Partial stop edits (adding a mid-route stop, editing a ZIP) can change which stop is "first pickup" or "last delivery", so we pessimistically clear both rather than trying to diff. Cheap and obviously correct.
- **GET /api/loads list adds `l.ai_metadata`, not a derived subset.** The frontend confidence-badge component (FN-818) needs the per-field object to render drawer tooltips, so projecting only `overall_confidence` would force a second detail fetch for every drawer open. Full JSONB on the list row is cheap (bounded to a dozen numeric fields) and keeps the drawer snappy.
