## Implementation Summary

### FN-816 (subtask)

**Migration** — `backend/packages/goodmen-database/migrations/20260423043941_add_ai_metadata_to_loads.js` adds a nullable `jsonb` column `ai_metadata` to the `loads` table plus a `GIN` index on that column.

**Guarded + idempotent** — Matches the style of the sibling `20260401100000_add_settlement_ai_insights.js` and `20260418140000_add_loads_needs_review.js` migrations: `hasTable('loads')` and `hasColumn('loads', 'ai_metadata')` guards let the migration re-run cleanly on environments that were manually fixed. The index is created via `CREATE INDEX IF NOT EXISTS` so a rerun after a partial failure doesn't crash.

**GIN index for field-name lookups** — Stored payload has a nested `fields` object keyed by load field name. A GIN index on the whole `ai_metadata` column supports containment and key-exists operators (`?`, `?|`, `@>`) so queries like `WHERE ai_metadata->'fields' ? 'broker_name'` or `WHERE ai_metadata @> '{"fields":{"broker_name":0.98}}'` are indexable without per-subfield expression indexes.

**Down migration** — Drops the index first (`DROP INDEX IF EXISTS idx_loads_ai_metadata`), then the column. Safe to run on environments that never had the column.

**Existing rows unaffected** — Column is nullable with no default; existing loads (including AI-created loads that pre-date this migration) remain NULL and the backend (FN-817) can treat NULL as "no confidence available" without a backfill.

## Key Decisions

### FN-816
- **Single whole-column GIN index instead of expression indexes per subfield.** The payload's `fields` shape evolves as new AI-extractable fields are added; a whole-column GIN index absorbs future keys without a schema change. Cost: slightly larger index than, say, `GIN ((ai_metadata->'fields'))`, but negligible at expected row counts.
- **Nullable with no default.** A JSONB default (`{}` or `null`) would either (a) mask the "never AI-extracted" signal the backend/frontend need, or (b) force a table-wide rewrite on PostgreSQL versions that materialize the default. Nullable is the cheapest, clearest choice.
- **Guards (`hasColumn`/`hasTable`) match sibling migrations.** Matches `settlement_ai_insights` and `loads_needs_review` patterns so the migration is safe to re-run after partial failures and on non-prod environments with manually-applied schema changes.
