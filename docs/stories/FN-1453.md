# FN-1453: FMCSA imports 503 reports "Redis not connected" when REDIS_URL env var is missing — misleading + silent boot failure

## Requirement

Replace the single opaque `503 FMCSA import queue unavailable (Redis not connected)` body returned by `/api/fmcsa/imports/*` with three distinct messages so operators can tell:

1. `REDIS_URL` is unset (config bug on integrations-service).
2. The queue threw at boot (malformed URL, missing schema, etc. — surface the captured error).
3. The queue is up but Redis is currently disconnected (transient outage).

## Acceptance Criteria

- [x] When `REDIS_URL` is missing, the 503 explicitly says the env var is unset (`"FMCSA import queue not initialized: REDIS_URL is not configured on integrations-service"`).
- [x] When `createImportQueue(...)` throws at boot, capture the error message on `app.locals._fmcsaImportInitError` (truncated ~500 chars, no stack) and include it in the 503 body.
- [x] When `importQueueInstance` exists but Redis is currently disconnected, surface a different message (`"FMCSA import queue not ready: Redis connection lost"`). Use the existing `redisConnected` flag tracked in `fmcsa-import-queue.js` event listeners; expose via a small `isReady()` getter.
- [x] Update `docs/stories/FN-1415.md` with a short "Operator notes" section listing the three 503 messages and what each one means.
- [x] No change to the 202 success path.

## Agent

backend

## Files Changed

- `backend/microservices/integrations-service/server.js` — split the route gate into three branches (REDIS_URL missing / init threw / Redis disconnected); persist init failure on `app.locals._fmcsaImportInitReason` + `app.locals._fmcsaImportInitError` (≤500 chars, no stack).
- `backend/packages/goodmen-shared/services/fmcsa-import-queue.js` — added `'end'` and `'close'` listeners on `queue.client` so `redisConnected` flips back to false on disconnect; exposed `isReady()` on the returned API.
- `docs/stories/FN-1415.md` — added "Operator notes — FMCSA imports 503 messages (FN-1453)" section documenting the three 503 bodies and the operator action for each.

## Key Decisions

- **`isReady()` reflects current state, not just initial connection.** The pre-existing `redisConnected` flag flipped to `true` on `'ready'` but never back to `false`. Without that, an `isReady()` getter using only the existing flag would return `true` after Redis went down — and the new "Redis connection lost" branch would never fire. Added `'end'` + `'close'` listeners so the flag accurately reflects whether the ioredis client is currently connected. ioredis emits `'end'` after retries are exhausted and `'close'` per socket tear-down; together they cover transient and exhausted-retry cases.
- **Three branches, not two.** A defensive fallback "FMCSA import queue not initialized" exists for the rare case a request lands before `initImportQueue()` has finished (the queue init runs after `app.listen()`). It's narrow but real on cold start.
- **Init error message capped at 500 chars, stack stripped.** The AC said ~500 chars; using `String(err.message).slice(0, 500)` and never serializing the stack keeps the public payload bounded and avoids leaking stack frames through a 503 body.
- **No new test files.** The fix is a small, route-gate-level change in a service whose existing test surface only covers the queue's `_isCronEnabled` helper. Writing a request-level test would require a meaningful Express test harness that doesn't exist in this service today — out of scope per the AC.

## Deployment Handoff

- **Service to redeploy:** `fleetneuron-integrations-service` only. The routing/queue changes live entirely inside this service.
- **Migrations:** none.
- **Env vars:** none added or changed. `REDIS_URL` continues to be required for the queue to initialize at all.
- **Cache invalidation:** none.
- **Feature flags:** none.
- **Deploy order:** standalone — no cross-service coordination needed. The only public contract change is the wording of the 503 body, and no current frontend caller depends on the exact string.
- **Post-deploy verification (manual, on a non-prod environment):**
  1. **REDIS_URL missing branch.** Temporarily unset `REDIS_URL` and redeploy the service. Hit `POST /api/fmcsa/imports/run` (as an internal-tenant `super_admin`) → expect 503 `"FMCSA import queue not initialized: REDIS_URL is not configured on integrations-service"`.
  2. **Init-error branch.** Set `REDIS_URL=redis://invalid-host-that-does-not-resolve:6379` (or any value that makes `createImportQueue` throw — e.g. an unreachable hostname combined with a process-startup error). Redeploy. Same call → expect 503 `"FMCSA import queue failed to initialize: <message>"` with the captured error.
  3. **Redis-disconnected branch.** Run with a valid `REDIS_URL`, then stop Redis. After the existing retry strategy gives up (or after `'end'`/`'close'` fires), the same call should return 503 `"FMCSA import queue not ready: Redis connection lost"`.
  4. **Success path.** Restore `REDIS_URL` and Redis, redeploy, confirm `POST /api/fmcsa/imports/run` still returns 202 with `{success:true, data:{runIds:[...]}}`.
