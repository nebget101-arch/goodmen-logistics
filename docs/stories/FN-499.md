## Implementation Summary

### New Service: `settlement-engine-v2.js`

`generateDualSettlements(payrollPeriodId, driverId, dateBasis, userId, knex, context)`:

1. **Loads period, driver, truck, Equipment Owner info** from existing tables
2. **Resolves expense profile split ratios** (`driver_percentage` / `owner_percentage`) from `expense_responsibility_profiles`
3. **Gets eligible loads** via existing `getEligibleLoads()` from `settlement-service.js`
4. **Queries fuel_transactions + toll_transactions** by `truck_id` in period
5. **Checks prior period's settlement** for negative balance rollover via `carried_balance`
6. **Creates DRIVER settlement** (`settlement_type = 'driver'`):
   - Revenue = gross Ă— driver split %
   - Deductions: fuel (driver share), tolls (driver share), recurring rules (excluding `equipment_owner_only`), carried balance
7. **Creates EO settlement** (`settlement_type = 'equipment_owner'`):
   - Revenue = gross Ă— owner split %
   - Deductions: fuel (owner share), tolls (owner share), recurring rules (excluding `driver_only`), carried balance, approved balance transfers
   - Marks applied balance transfers as `status = 'applied'`
8. **OO special case**: Single settlement, 100% expenses, no EO settlement generated

Balance transfer CRUD: `createBalanceTransfer`, `approveBalanceTransfer`, `rejectBalanceTransfer`, `listBalanceTransfers`.

### Key Logic Decisions

**Fuel/toll split**: EO gets `amount Ă— ownerPct`, driver gets `amount Ă— driverPct`. Both adjustments are inserted as separate `settlement_adjustment_items` â€” one per settlement. Transactions are linked (`settlement_link_status = 'linked'`) on the driver settlement (first to write), and a parallel link on the EO settlement re-uses the same `source_reference_id`.

**Recurring deduction routing**: `applies_when = 'always'` â†’ both settlements (split by ratio). `applies_when = 'driver_only'` â†’ driver settlement only. `applies_when = 'equipment_owner_only'` â†’ EO settlement only, full amount.

**Carried balance**: Tracked via `settlements.carried_balance` and `carried_balance_from_settlement_id`. The carried amount is also inserted as a `settlement_adjustment_items` deduction of type `carried_balance` for audit visibility.

**Balance transfers**: Created with `status = 'pending_approval'`. After accounting approves (`status = 'approved'`), the next `generate-dual` run for the EO picks them up and marks them `applied`.
