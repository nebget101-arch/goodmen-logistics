### FN-1493 (subtask) â€” Backend activity endpoints

**Branch:** `backend/FN-1493/activity-endpoints` (off `integration/FN-1481`)

**Endpoints added in `routes/receiving.js`:**
- `GET /api/receiving/activity` â€” paginated POSTED tickets (default 25/page, max 200) with `posted_by_name` + `location_name` resolved on each ticket and lines expanded for the current page only. Top-level aggregations (`totalParts`, `totalLines`, `totalCost`, `byUser[]`, `byVendor[]`) are computed across the **full filtered set**, not just the page, so the summary tiles stay stable across pagination. `X-Total-Count` header carries the unpaged total.
- `GET /api/receiving/activity.csv` â€” streams CSV via knex `.stream()`, one row per line item; columns `ticket_number, posted_at, location_name, vendor_name, reference_number, sku, part_name, qty_received, unit_cost, posted_by_name`. CSV escaping wraps any cell containing `,`, `"`, `\r`, or `\n` and doubles embedded quotes.

**Filter set** (shared by both endpoints, all optional, AND-composed via the `applyActivityFilters` helper):
- `locationId` â†’ `receiving_tickets.location_id =`
- `from` / `to` â†’ inclusive `posted_at` window (parsed to `Date`; invalid values ignored)
- `userId` â†’ `receiving_tickets.posted_by =`
- `vendor` â†’ exact `vendor_name =` (frontend dropdown drives this)

`receiving_tickets.status = 'POSTED'` is always applied. Auth required; no role gate beyond auth (read-only audit data is broadly visible).

**Tests** â€” `routes/receiving-activity.test.js` (15 cases): filter composition, date-range, DRAFT/other-location exclusion, pagination + `X-Total-Count`, aggregation correctness, aggregations stable across pagination, CSV streaming end-to-end (1500+ rows), CSV escaping for commas and embedded quotes, empty-result shape.

## Key Decisions
- **Aggregate in JS over the filtered ticket set, not via SQL `GROUP BY`.** Avoids a dialect-specific aggregation query and keeps the test stub simple. Trade-off: pulls all filtered ticket rows into memory once. For audit reports scoped to a location + date range this is bounded by ticket count (not lines Ă— tickets) and fine.
- **Aggregations cover the full filtered set, not the page.** Otherwise the summary tiles would shift every time the user paged, which is useless. The page is only the `data` array; `totalParts`/`totalLines`/`totalCost`/`byUser`/`byVendor` and `X-Total-Count` all describe the unpaged result.
- **CSV streams via knex `.stream()`.** Required for the >10k-row criterion; `res.write()` per row keeps server memory bounded. The header row is written before the stream attaches so partial-failure responses still have something useful.
- **`from`/`to` are normalized to `Date` objects in the route** (not just passed through). Production-equivalent behavior across DB and tests; invalid values are silently dropped rather than returning 400 (matches the rest of the receiving routes' tolerance for missing optional filters).
- **Vendor filter is exact-match.** Frontend will populate from a vendor dropdown (the same vendor strings already on tickets); substring matching can be added later if a free-text input replaces the dropdown.
- **Aliased the CSV `parts.name` column to `part_name`** to disambiguate from `posted_by_name` and `location_name` in the row object.
