# FN-1542: [Warehouse Receiving] Quick-Add qty input + Receive-lines column alignment + Header card sizing

## Requirement
Three UI defects on `/warehouse-receiving`:

1. **Quick Add panel has no qty input** — clicking **+ Add** always inserts a Receive line with `qty = 1`. Need a per-row qty input (default 1) consistent with the QTY MULTIPLIER used by the scan flow.
2. **Receive lines column misalignment** — `QTY` and `UNIT COST` values render visually offset from their headers. Headers and values must align.
3. **Top header card field sizing** — Ticket # is huge and wraps to two lines (`RCV-E8B7-1778201604345-6960` overflows); Location / Vendor / Reference look cramped. Fields need proportional sizing; Ticket # must never wrap.

Bundled into one Bug + one frontend subtask because all three changes touch the same component files (.html / .scss / .ts) — splitting would force serial execution from overlapping files.

## Implementation Summary

### FN-1546 (subtask) — frontend

All three UI fixes landed in a single subtask because they share component files. Branch: `agent/frontend/FN-1546/receiving-ui-fixes` (off `origin/integration/FN-1542`).

**2A — Quick Add per-row qty input.** Refactored each row in `quick-add-panel.component.html` from a single `<button>` (which clicked anywhere on the row to add `qty=qtyMultiplier`) into a 3-zone grid: part info / qty `<input type=number>` / `+ Add` button. The qty input is per-part, defaults to 1, and is sanitized to integer ≥ 1 in `setRowQty()` (non-numeric / negative / fractional values fall back to 1). On click, the panel emits `qty = userQty × qtyMultiplier`, preserving the scan flow's QTY MULTIPLIER contract — when `userQty=1` (default), behavior is identical to the previous emit. Keyboard navigation (`ArrowUp`/`ArrowDown`/`Enter`) on the `+ Add` button is preserved; pressing Enter inside the qty input also adds. Pressing tab moves: row1-qty → row1-add → row2-qty → row2-add.

**2B — Receive lines column alignment.** Two changes in `warehouse-receiving.component.{html,css}`:
1. Added `<colgroup>` with explicit `width` (in `ch`) for SKU/Name/Qty/Unit Cost/Action and set `table-layout: fixed` so column widths are stable as rows are added.
2. Added a more-specific selector `.recv-table thead th.recv-table__num { text-align: right }` — without it, the existing `.recv-table thead th { text-align: left }` rule outranked `.recv-table__num` and forced numeric headers to the left while cells stayed right-aligned (root cause of the misalignment).

**2C — Header card field sizing.** In `warehouse-receiving.component.css`:
- Changed grid template on `.recv-ticket-card__row` from `minmax(180px, 220px) minmax(200px, 280px) ...` to proportional `1fr / 1.6fr / 1.2fr / 1.2fr` (with sensible min-widths). Ticket # column gets the most flex space because the generated value is the longest token.
- `.recv-ticket-number__value`: dropped font from 18px → 13px (monospace), added `white-space: nowrap`, `overflow: hidden`, `text-overflow: ellipsis`, `min-height: 40px`. Truncated values surface the full text via the `title` attribute on the parent div.
- Added `min-height: 40px` to `.recv-control` so all controls in the card are tap-friendly on mobile.

## Key Decisions

- **Row is no longer a single click target.** The previous "click anywhere on the row to add" behavior was incompatible with placing an editable input inside the same area (a `<button>` cannot legally contain `<input>`). The ticket explicitly asks for a qty input next to `+ Add`, so the row is now `[info] [qty] [+ Add]` and the button is the sole add trigger. Keyboard nav still works through the `+ Add` button.
- **`table-layout: fixed` over auto.** The existing misalignment was partly caused by auto-layout shifting column widths as content grew. Fixed layout with explicit `<colgroup>` widths makes the headers and cells lock to the same grid lines.
- **Ticket # font size: 13px not 14px.** 14px monospace would still wrap `RCV-E8B7-1778201604345-6960` at the 1024px breakpoint where the column drops below ~280px. 13px gives ~30 chars of headroom in the smallest card layout.
- **No client-side dedup change.** The ticket says "follow whatever existing dedup/append behavior the scan flow uses." The scan flow simply prepends a new line per add (no client dedup); Quick Add already mirrored that. No change.
