# FN-1456: FMCSA imports — manual file upload (replaces FMCSA_*_URL env-var path)

## Requirement

Add a manual bulk-file upload path to the FMCSA imports admin so operators can run imports without depending on FMCSA's gated `ai.fmcsa.dot.gov/SMS/Tools/Downloads.aspx` endpoints (Basic-auth + open captcha problem from FN-1422) **and** without depending on `data.transportation.gov/api/views/<slug>/rows.csv?accessType=DOWNLOAD` for the census/authority slugs that turned out to be `assetType:"href"` / `"file"` (FN-1455).

Operator flow: download the bulk CSV (or `.csv.gz`) from FMCSA on their workstation → click an "Upload bulk file" affordance in the FMCSA Imports admin UI → pick the file type (`census` | `authority` | `inspections` | `crashes` | `sms`) and optional `dryRun` → backend streams the upload to a tmp path → enqueues a `run-import` job whose source descriptor is `{ type: 'path', value: '/tmp/...' }` → existing parsers do the rest → tmp file deleted after the job completes.

This decouples FleetNeuron from FMCSA's auth/captcha mechanics entirely and reuses 100% of the parsers shipped in FN-1413 / FN-1414 / FN-1420 / FN-1422.

## Acceptance Criteria

- [ ] `POST /api/fmcsa/imports/run-upload` accepts a multipart body and returns 202 with `{ runId, file, uploadedSizeBytes }`. Gated by the same chain as `POST /run`.
- [ ] Multer saves uploads to a tmp path (`FMCSA_UPLOAD_DIR` or default `/tmp/fmcsa-uploads`), per-file size limit 1 GB, 413 on overflow.
- [ ] Queue's `enqueueImportRun(...)` accepts an optional `source: { type, value, auth? }` and forwards it on `job.data`; the `run-import` processor passes `{ dryRun, source }` to the registered importer adapter.
- [ ] Each adapter prefers an explicit `source` over its `FMCSA_*_URL` env-var fallback. census/authority pass `source` to the runner (translating `path → filePath`, `url → url`); inspections/crashes/sms open a stream from `source.value`.
- [ ] Tmp file is deleted in the processor's `finally` (best-effort; no error if cleanup fails).
- [ ] Admin UI ("FMCSA Imports") shows an "Upload bulk file" affordance: file picker + file-type select + dryRun toggle → multipart POST → optimistic refresh of the ledger; progress bar via `HttpClient.reportProgress`.
- [ ] Existing `POST /run` route keeps working; this is additive.
- [ ] dryRun short-circuit (FN-1452) still applies even when `source` is provided.

## API Contract (BE + FE share this)

```http
POST /api/fmcsa/imports/run-upload
Authorization: Bearer <internal-tenant-token>
Content-Type: multipart/form-data

  fields:
    file       (binary, the CSV or CSV.GZ)
    fileType   (string; one of census|authority|inspections|crashes|sms)
    dryRun     (string "true"|"false"; optional, defaults to false)

→ 202 { success: true, data: { runId, file, uploadedSizeBytes } }
→ 400  validation
→ 403  permission denied
→ 413  > 1 GB
→ 503  queue unavailable
```

`GET /api/fmcsa/imports` ledger response is unchanged — uploaded runs appear with `triggered_by='manual'`.

## Implementation Summary

### FN-1457 (subtask)

Backend wiring for the manual file-upload path:

- **`POST /api/fmcsa/imports/run-upload`** — new multipart route on the existing `/api/fmcsa/imports` mount, gated by the same chain (`auth → tenant → requireInternalTenant → loadUserRbac → requirePermission('fmcsa.imports.manage')`). Single-file multer storage with `limits.fileSize = 1 GB`, destination `process.env.FMCSA_UPLOAD_DIR || '/tmp/fmcsa-uploads'` (mkdir-recursive on first use), MIME filter for `text/csv` / `application/gzip` / `application/x-gzip` / `application/octet-stream`, deterministic-but-unique filenames (`fmcsa-<timestampBase36>-<6byteHex><ext>`). Multer error → `LIMIT_FILE_SIZE` returns 413, `UNSUPPORTED_MIME` returns 400. On any pre-enqueue failure (validation / RBAC / queue) the saved tmp file is unlinked before the response is returned.
- **`enqueueImportRun({ ..., source })`** — accepts an optional `source: { type: 'path' | 'url', value, auth? }` and forwards it on `job.data` only when set (so the existing cron path's job data is byte-identical to before).
- **`queue.process('run-import', ...)`** — passes `{ dryRun, source }` to the registered importer. The full body is wrapped in `try { … } finally { fs.promises.unlink(source.value).catch(() => {}) }` when `source.type === 'path'`, so failed runs don't leak tmp files on the worker dyno.
- **Adapters** in `register-importers.js` now prefer an explicit `source` over the env-var fallback. Census/authority translate `{ type: 'path', value }` to the runner's `{ filePath: value }` and `{ type: 'url', value }` to `{ url: value }` (the existing `runner.openSource` already accepts both). Inspections/crashes/sms open a Readable from the descriptor via the existing `openSourceStream` helper and pass it as `source: stream` to their runner. The "throw if env var unset" path remains as the safety net for cron triggers.
- **dryRun short-circuit (FN-1452)** still applies BEFORE the source path: a dryRun adapter call returns `{ rowsInserted: 0, rowsUpdated: 0, rowsSkipped: 0 }` without touching network/DB even when a source is provided.
- **Tests** — added 17 new node:test cases (`sourceToRunnerInput()` translation, census/authority path-source forwarding, snapshot-adapter path-source forwarding with garbage env vars set to prove the env-var path is bypassed, and dryRun-with-source for all five files). All 26 cases in `register-importers.test.js` pass.
- **multer dependency** — already present in both `backend/microservices/integrations-service/package.json` and `backend/packages/goodmen-shared/package.json`; no `package.json` change needed despite the ticket's expected-files list.

### FN-1458 (subtask) — Frontend upload modal

Adds the operator-facing "Upload bulk file" path on top of the existing FMCSA Imports admin page (FN-1425):

- **Service** `frontend/src/app/services/fmcsa-imports.service.ts`: new `runUpload(file, fileType, dryRun)` that builds a `FormData`, POSTs to `${apiUrl}/fmcsa/imports/run-upload` with `reportProgress: true` + `observe: 'events'`, and resolves with the API's `{ runId, file, uploadedSizeBytes }`. Upload-progress percentage is exposed on a separate `uploadProgress$` stream so the modal can render a progress bar without each subscriber re-issuing the HTTP request.
- **Component** `fmcsa-imports.component.ts/html/css`: header gains an "Upload bulk file" button; each per-row file card gains a small "Upload" button that opens the same modal with that file type pre-selected. Modal contains a `<input type="file" accept=".csv,.csv.gz,.gz,...">`, a file-type `<select>`, a dry-run checkbox, an `aria-live` progress bar, and inline error display.
- **A11y**: dialog has `role="dialog"`, `aria-modal="true"`, `aria-labelledby` on its title; the file input is auto-focused on open; `Esc` closes the modal when no upload is in flight (and is a no-op while uploading — chosen over a "Cancel upload?" prompt because aborting an `XMLHttpRequest` mid-stream is brittle and the operation is short).
- **Specs**: `fmcsa-imports.component.spec.ts` covers open with/without prefill, submit-disabled gating, success → ledger refresh, 413 error path, progress relay, and Esc behavior. `fmcsa-imports.service.spec.ts` covers FormData shape, response value, progress events, and 413 error propagation.

## Key Decisions

### FN-1457 (subtask)

- **Filenames are tmp-only and not persisted on `fmcsa.import_runs`** — the ticket calls this out as a possible follow-up; we ship without it. The existing ledger row is sufficient for v1; if audit demands the original filename, add it as a separate migration.
- **Multer middleware is constructed once per router instance, not per request** — `buildUploadMiddleware()` is called inside `createImportsRouter` so each integrations-service boot gets one instance. The diskStorage closure mkdir's lazily on first request so a cold deploy with a non-existent `FMCSA_UPLOAD_DIR` still succeeds.
- **`source` is omitted from `job.data` when not provided** — keeps cron-trigger jobs byte-identical to the pre-FN-1457 contract so any inspection-tool diffing the ledger doesn't see noise.
- **Cleanup is in the processor's `finally`, not the route** — the route can't `unlink` after handing off because the worker may be on a different dyno. The processor owns the file's lifetime end-to-end once it accepts the job.
- **Snapshot adapters' "throw if env var unset" message now also points operators at the upload route** — so the failure surface guides them to the new path instead of just the env-var fix.

### FN-1458 (subtask)

- **Single observable returns the final result; progress is on a separate stream.** The Jira spec asked for `runUpload(...): Observable<{ runId, file, uploadedSizeBytes }>` — i.e. the resolved value is the result, not the union of progress + result. Implementing that cleanly means filtering the `HttpEvent` stream down to the `Response` event in the service, and tee'ing `UploadProgress` events into a singleton `BehaviorSubject<number>` exposed as `uploadProgress$`. The component subscribes to both: `runUpload(...)` for the result, `uploadProgress$` for the bar. This matches Angular's idiomatic split, avoids forcing every consumer to type-narrow on `HttpEventType`, and avoids two HTTP requests that would happen if we naively `share()`d two consumers off the events stream.
- **Esc during upload is a no-op (documented in `closeUpload`).** Aborting a multipart `XMLHttpRequest` mid-stream is brittle (the server may already have buffered the file, the queue may already have enqueued the job), and the upload is single-shot up to 1 GB so the wait is bounded. A "Cancel upload?" prompt would add UX without correctness — kept simple per the ticket's "pick whichever is simpler" guidance.
- **Per-row Upload button + global Upload button.** The Jira description offered "next to (or replacing) the existing per-row trigger" — kept the existing per-row file checkboxes (used for URL-based runs) and added a small per-row "Upload" button **plus** a header-level "Upload bulk file" button, so operators can either upload a specific file from its row (pre-fills the file type) or open a blank modal and pick the type themselves.
- **Progress bar fed via `BehaviorSubject` rather than `Subject`.** A new subscriber reads the most recent percentage immediately, so the modal renders a non-zero bar even if it subscribes mid-upload. `finalize` resets to `0` so the next upload starts clean.
