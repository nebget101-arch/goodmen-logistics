# FN-1506: [Work Orders] Save/Get fail with column "mechanic_user_id" does not exist on work_order_labor_items

## Requirement
Add the missing `mechanic_user_id` column on `work_order_labor_items` so the existing backend code (which already references the column in SELECT/JOIN, INSERT, and UPDATE statements) starts working. This unblocks all WO save and detail-load operations on dev.

## Reported Symptoms (all share one root cause)
1. **GET 500** — `GET /api/work-orders/:id` fails: `column woli.mechanic_user_id does not exist`. (`services/work-orders.service.js:302-306`)
2. **PUT 400** — `PUT /api/work-orders/:id` with a `labor` array fails: `column "mechanic_user_id" of relation "work_order_labor_items" does not exist`. (`services/work-orders.service.js:457`, also `:380, :860, :887`)
3. **Parts list stale** — `POST /api/work-orders/:id/parts` returns 201, but the new part doesn't appear without manual refresh. Transitive cause: the post-add reload calls GET → 500 (Symptom 1), so the UI keeps the stale list.

`down` reverses all three (drop index, drop FK, drop column) using `IF EXISTS` / `hasColumn` so the down → up cycle is also a clean no-op.

## Key Decisions

- **Timestamp `20260509100000`**: latest existing migration on dev is `20260508120000_fix_work_orders_status_check.js`; the ticket's suggested timestamp range was outdated, so picked the next ordered slot.
- **`hasColumn` guard on `up` in addition to FK/index `IF NOT EXISTS`**: the spec only guarded the FK and index, but `alterTable.uuid('mechanic_user_id')` would throw on a second `up`. Wrapping it with `hasColumn` makes the whole migration truly idempotent (AC: "re-running up after up is a no-op").
- **`ON DELETE SET NULL`**: matches the spec — deleting a user shouldn't delete their labor history; the labor row stays but loses the mechanic reference.
- **No backfill of existing rows**: the column is nullable; existing labor items remain `NULL` (no historical mechanic data to recover).

## Deployment Handoff

- **DB migration to run on dev:** `20260509100000_add_mechanic_user_id_to_work_order_labor_items.js`
  - Run with `cd backend/packages/goodmen-database && npm run migrate:dev` (or via the standard dev migration job).
  - Locking: `ALTER TABLE ... ADD COLUMN` of a nullable column is metadata-only in Postgres (no table rewrite). Adding the FK takes an `ACCESS EXCLUSIVE` lock briefly to validate references, but the table is empty of `mechanic_user_id` values so validation is instant. `CREATE INDEX` (non-`CONCURRENTLY`) takes a `SHARE` lock; on dev the table is small so this is acceptable. **For prod**, consider running with `CONCURRENTLY` if/when promoted (separate hotfix consideration).
- **Services to restart:** none required — the backend code already references `mechanic_user_id`. As soon as the migration runs, the existing services will start succeeding. No deploy of `backend/web-server` or `backend/work-orders-service` needed.
