# FN-1836: Navigation entry point for generic Agreements (e-sign) template surface

## Requirement
FN-1787 built the standalone agreement-template surface — `/agreements` (upload) and
`/agreements/:id/review` (AI field-mapping + role review) — as a lazy-loaded Angular module
(`frontend/src/app/agreements/**`, registered in `app-routing.module.ts:188` under FN-1794,
`AuthGuard`-gated). However, **nothing in the app navigation links to it**:
`frontend/src/app/config/nav.config.ts` has no `/agreements` entry, and no sidebar item, menu,
button, or `routerLink` anywhere points to the route. The feature is reachable only by manually
typing the URL.

The two adapter stories (FN-1789 Equipment lease, FN-1790 Lease-to-Own) add **contextual** entry
points for `lease_agreement` documents launched from specific records — they do **not** provide a
top-level home for the **generic** agreement-template surface (the detect-fields contract supports
`documentType: "generic"`, not only leases). This story closes that gap.

## Implementation Summary
Added an **"Agreements"** child link (`/agreements`) under the existing **Compliance** nav section
(`gavel` icon), alongside "IFTA Quarterly". The link routes to the agreement upload screen; the
`/agreements/:id/review` deep-link is untouched and continues to work.

Nav links require a permission `tab` key to render (`AccessControlService.canSee(tab)` returns
`false` for any tab not present in `TAB_PERMISSIONS`). The `/agreements` route itself is only
`AuthGuard`-gated and there was no agreements permission in the frontend model yet (the
`agreements-routing.module.ts` comment notes backend grants land in FN-1792). Rather than reuse an
unrelated tab (which would conflate concerns) or invent an always-true tab (which would expose
agreement authoring to every authenticated role), I introduced a dedicated, forward-compatible tab:

- New permission constant `PERMISSIONS.AGREEMENTS_VIEW = 'agreements.view'`.
- New tab mapping `agreements: [PERMISSIONS.AGREEMENTS_VIEW]` in `TAB_PERMISSIONS`.
- The Compliance section's `tabs` array extended to `['compliance', 'agreements']` for consistency
  (section/command-palette visibility helpers read this).

Because `AccessControlService.hasPermission()` grants **any** permission code to
`super_admin` / `admin` / `company_admin` (existing wildcard), the entry is immediately visible to
those administrative roles — the roles that author agreement templates — while every other role
stays hidden until backend RBAC (FN-1792) explicitly grants `agreements.view`. This satisfies
"renders in the sidebar" today without "silently exposing it to roles that shouldn't author
agreements."

## Key Decisions
- **Placement under Compliance (not a new section):** the e-sign agreement surface is a
  compliance/documents concern and the existing `gavel` Compliance section is the natural home;
  matches the AC's suggested grouping and avoids a one-item section.
- **Dedicated `agreements.view` permission/tab instead of plain AuthGuard or a borrowed tab:**
  nav rendering is permission-tab-driven, so an AuthGuard-only entry isn't expressible without a
  tab. A dedicated tab keeps visibility scoped to agreement authors and is forward-compatible with
  FN-1792's backend grants — no rework needed when those land. Admin roles see it now via the
  existing role wildcard in `hasPermission()`.
- **Route guard unchanged:** out of scope to modify the agreement components/route; the route stays
  `AuthGuard`-gated. The nav tab is an additional, stricter visibility filter, not a replacement.

## Follow-ups
