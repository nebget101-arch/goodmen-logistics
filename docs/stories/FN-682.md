### FN-702 (subtask)
**Delete flow — dependency detection dialog (soft vs hard delete)**

**Branch:** `frontend/FN-702/locations-delete-flow`

1. **`frontend/src/app/models/location.model.ts`** — Added `LocationDependencies` interface: `{ work_orders, inventory_items, users, vehicles }`

2. **`frontend/src/app/components/locations-admin/delete-location-dialog/delete-location-dialog.component.ts`** (new):
   - `@Input() location: LocationListItem | null`, `@Input() isOpen`
   - `@Output() deleted` (after hard delete), `@Output() markedInactive` (after soft delete), `@Output() close`
   - **Three dialog phases**: `confirm` → `has_deps` → `done`
   - `onConfirmDelete()` — attempts DELETE; on **200** emits `deleted`, moves to `done`; on **409** extracts `err.error.dependencies` and pivots to `has_deps`; on **404** treats as already-deleted success
   - `onMarkInactive()` — calls PATCH `{ active: false }` on the location; on success emits `markedInactive`, moves to `done`

3. **`frontend/src/app/components/locations-admin/delete-location-dialog/delete-location-dialog.component.html`** (new):
   - **Phase 1 (confirm)**: location name, type/city pill, "Delete" (red) button → triggers delete attempt
   - **Phase 2 (has_deps)**: 2×2 dependency grid showing counts; "Mark Inactive" (amber) button + Cancel
   - **Phase 3 (done)**: success state with different copy for hard vs soft delete

4. **`frontend/src/app/components/locations-admin/delete-location-dialog/delete-location-dialog.component.scss`** (new):
   - `.dld-*` namespace; colored left-border on header per phase
   - Dependency grid cards with amber highlight when count > 0

5. **`frontend/src/app/services/api.service.ts`** (modified):
   - JSDoc on `deleteLocation` documents the 409 response shape

6. **`frontend/src/app/app.module.ts`** (modified):
   - Declared `DeleteLocationDialogComponent`

**Key decisions:**
- The "Delete" button in phase 1 triggers the actual DELETE request — 409 carries `{ dependencies: { work_orders, inventory_items, users, vehicles } }`
- Soft-delete is a PATCH with `{ active: false }`; 404 on delete is treated as success
