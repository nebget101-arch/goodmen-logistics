### FN-701 (subtask)
**Supply Rules tab (WAREHOUSE/SHOP only) with primary + auto-replenish**

**Branch:** `frontend/FN-701/locations-supply-rules-tab`

1. **`frontend/src/app/models/location.model.ts`** (new on this branch):
   - Added `SupplyRuleRow` interface: `id`, `warehouse_location_id`, `shop_location_id`, `warehouse_name`, `shop_name`, `is_primary_supplier`, `auto_replenish`, `delivery_days`, `notes`, `active`, `created_at`
   - All other types same as FN-698

2. **`frontend/src/app/components/locations-admin/location-edit-dialog/supply-rules-tab/supply-rules-tab.component.ts`** (new):
   - `@Input() locationId: string | null` ā€” null shows "save first" empty state
   - `@Input() locationType: string` ā€” determines direction (WAREHOUSE vs SHOP)
   - `isWarehouse` / `isShop` getters; component hidden for other types
   - `loadRules()` ā€” GET /api/locations/:id/supply-rules; response joined with warehouse_name/shop_name
   - `loadCounterparts()` ā€” listLocations({ type: 'SHOP' or 'WAREHOUSE', active: 'true' })
   - `availableCounterparts` getter ā€” filters already-linked locations from Add dropdown
   - Add/Edit inline dialog: `selectedCounterpartId`, `isPrimary`, `autoReplenish`, `deliveryDays`, `notes`
   - `saveRule()` ā€” POST with `warehouse_location_id`/`shop_location_id` swap based on locationType; PATCH for edit
   - Inline delete with two-step confirmation (`confirmDelete` ā†’ `deleteRule`)
   - 409 duplicate error mapped to field-level error on counterpart select
   - OnPush with explicit `cdr.markForCheck()` after all async ops

3. **`frontend/src/app/components/locations-admin/location-edit-dialog/supply-rules-tab/supply-rules-tab.component.html`** (new):
   - Unsaved-location guard; wrong-type guard
   - Header with "+ Add Supply Rule" button
   - Rules table: counterpart name, Primary badge (star), Auto-Replenish badge, Lead Days, Edit/Delete actions
   - Inline delete confirmation (no modal)
   - Add/Edit dialog with: counterpart dropdown (filtered, disabled in edit), Primary toggle + hint, Auto-Replenish toggle, Lead Days number input, Notes textarea

4. **`frontend/src/app/components/locations-admin/location-edit-dialog/supply-rules-tab/supply-rules-tab.component.scss`** (new):
   - `.srt-*` class namespace
   - Dark fleet theme matching other tabs
   - Inline dialog with backdrop blur

5. **`frontend/src/app/services/api.service.ts`** (modified):
   - Added `listLocations()` with type/active/search/page/pageSize filters
   - Added `getLocationById()`, `createLocation()`, `updateLocation()`, `deleteLocation()`
   - Added `getLocationSupplyRules()`, `createLocationSupplyRule()`, `updateLocationSupplyRule()`, `deleteLocationSupplyRule()`

6. **`frontend/src/app/app.module.ts`** (modified):
   - Declared `SupplyRulesTabComponent`

**Key decisions:**
- Backend (FN-693) already stores `is_primary_supplier`, `auto_replenish`, `delivery_days` in DB ā€” no migration needed
- Primary toggle enforced at the API level (backend unsets other primaries on PATCH/POST)
- The component guards against `locationType !== 'WAREHOUSE' && locationType !== 'SHOP'` so it never renders for YARD/OFFICE/etc.
- `availableCounterparts` excludes already-linked locations from the Add dropdown (still shown in Edit mode)
- Supply rule pair is immutable after creation (warehouse + shop IDs cannot change) ā€” counterpart select is disabled in edit mode
