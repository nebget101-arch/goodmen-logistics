# FN-1299: Single-PDF auto-create — route through V2 wizard (ai-extract mode)

## Requirement
Migrate the single-PDF auto-create flow to use `<app-load-wizard-v2>` in `mode='ai-extract'` with the dropped PDF pre-loaded. This (a) gives single-PDF the bulk-style animated step progress UI, (b) eliminates the broken legacy save path tracked by FN-1298 by reusing the V2 wizard's working `submitCreate`, and (c) consolidates two of the three load-creation entry points onto the same component.

## Bulk vs Single — Decision
| Aspect | Single (current legacy) | Single (target — this story) | Bulk (unchanged) |
|---|---|---|---|
| Component | `showAutoModal` → `showManualModal` | `<app-load-wizard-v2>` (`mode='ai-extract'`, `[initialPdfFile]`) | `<app-bulk-extraction-grid>` |
| Progress UI | spinner + indeterminate bar | animated 4-step list (built in) | per-file row with status pill + % |
| Save model | review then save (currently broken) | review then save (V2 `submitCreate`) | auto-approve high-confidence; DRAFT low-confidence |
| Use case | "I have one rate-con; let me check before saving" | same | "stack of rate-cons; auto-process and surface review queue" |

**Keep bulk separate.** Bulk's auto-approve + DRAFT-queue model is a different product than single's review-before-save. Folding bulk into V2 would require adding a "save as DRAFT" mode + batch summary view to V2 — significant scope; defer.

## Acceptance Criteria
- [ ] Single PDF dropped or selected in the hero CTA opens `<app-load-wizard-v2>` directly with `mode='ai-extract'` and the file pre-loaded; extraction begins immediately.
- [ ] The wizard's existing animated progress (Extracting → Identifying → Parsing → Calculating) plays during extraction.
- [ ] After success, Steps 1-2 are prefilled; field-confidence badges render where confidence < threshold.
- [ ] Submit on the last step issues `POST /api/loads`, attaches the source PDF as `RATE_CONFIRMATION`, closes the wizard, refreshes the grid, and shows a success toast.
- [ ] Extraction failure shows the existing inline error + retry; no fallback to the legacy manual modal.
- [ ] Legacy `showAutoModal` and the `runAutoExtraction()` → `showManualModal` wiring are removed for the single-PDF path. Manual-create-from-empty (no PDF) still works using the legacy manual modal until separately migrated.
- [ ] Bulk flow (2-10 PDFs) is unchanged.
- [ ] FN-1298 is closed as superseded once this story merges.
- [ ] QA evidence captured in `docs/stories/evidence/FN-1299/`.

## Implementation Summary
Single-PDF hero drops now route directly into `<app-load-wizard-v2>` running in `ai-extract` mode with the source PDF pre-loaded via the new `[initialPdfFile]` input. The V2 wizard auto-fires `runExtraction()` on init, plays its built-in animated 4-step progress UI, prefills Steps 1–2, and on submit issues `POST /api/loads` with the PDF auto-attached as `RATE_CONFIRMATION` (existing wiring at `load-wizard.component.ts:592-600`). The legacy `showAutoModal` → `showManualModal` pipeline that fed the broken save path tracked by FN-1298 is gone.

## Key Decisions
- **Pre-load via input, not method call.** The PDF is passed as `[initialPdfFile]` and the wizard auto-feeds it inside `ngOnInit`, so the dashboard does not need a `@ViewChild` reference and timing works regardless of `*ngIf` mount order.
- **`wizardMode` field on the dashboard, not a derived getter.** Drives the V2 wizard `[mode]` binding directly from a plain field, avoiding the per-CD-cycle re-evaluation pattern that has caused infinite-loop regressions on `app-ai-select` (per `feedback_no_getter_options_bindings.md`).
- **Reset ai-extract state on every V2 close path.** `closeLoadWizardV2`, `onLoadWizardV2Created`, and `openLoadWizardV2` all clear `singlePdfForWizard` and force `wizardMode = 'create'` so the next New-Load menu entry never re-uses a stale hero PDF.
- **Left V1 wizard bridge helpers in place.** `openLoadWizardWithExtraction` and `applyExtractionToWizard` had only the now-removed `_continueAutoToWizard` as a caller, but the description explicitly carves the legacy `<app-load-wizard>` and `showLoadWizard` paths out of this scope. They become dead private methods (no warning, since TS doesn't flag unused private members) and will be removed when the V1 wizard is retired in FN-869.
