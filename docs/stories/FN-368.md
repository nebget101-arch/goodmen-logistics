# FN-368: Approve button should save all form changes before transitioning draft to DISPATCHED

## Summary
When reviewing a draft load, clicking Approve should save all pending form changes (rate, stops, driver, broker, notes, etc.) before transitioning status to DISPATCHED, in a single atomic transaction.

## Subtasks

| Key | Summary | Agent | Status | Branch |
|-----|---------|-------|--------|--------|
| FN-372 | Update approve-draft endpoint to accept and persist form changes | backend | Done | `backend/FN-372/approve-saves-changes` |
| FN-373 | Send current form state when calling approve on draft load | frontend | Done | `frontend/FN-373/approve-sends-form-data` |
| FN-374 | QA: Validate approve button saves form changes before transitioning | qa | Backlog | — |

## Acceptance Criteria
- [ ] Approve button saves all pending form changes (rate, stops, driver, broker, notes, etc.) before transitioning status
- [ ] Save + transition happens in a single database transaction
- [ ] If save fails, status does not change and user sees an error
- [ ] Existing behavior preserved when no changes were made (just approve)
- [ ] Works for both single-stop and multi-stop loads

### FN-372 (subtask) — Backend: Update approve-draft endpoint

**Implementation Summary:**
Modified `PATCH /api/loads/:id/approve-draft` to accept an optional request body with load fields and persist them in a single DB transaction before transitioning status to DISPATCHED.

**Files Changed:**
- `backend/packages/goodmen-shared/routes/loads.js` — approve-draft endpoint (~line 831)

---

### FN-373 (subtask) — Frontend: Send current form state when calling approve

**Implementation Summary:**
Updated `approveDraftLoad()` in loads-dashboard.component.ts and `approveDraft()` in loads.service.ts to collect all current form field values (rate, stops, driver, broker, notes, etc.) and send them as the request body to the approve-draft endpoint.

**Changes:**
- `loads.service.ts`: `approveDraft()` now accepts an optional `body` parameter (defaults to `{}`) and sends it as the PATCH request body
- `loads-dashboard.component.ts`: `approveDraftLoad()` now builds a payload from `manualLoadForm` values and `sortedStops` when called from the detail modal (where `editingLoadId` exists). When called from the table row action menu (no modal context), sends empty body preserving existing behavior.

**Files Changed:**
- `frontend/src/app/services/loads.service.ts` — approveDraft method
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` — approveDraftLoad method

**Key Decisions:**
- Payload is built from `manualLoadForm.getRawValue()` (the reactive form) rather than `editingLoadDetail` to ensure all pending form edits are captured, even if not yet synced to the detail object
- Reuses same field shape as the `createLoad()` edit payload for consistency with the backend `PUT /:id` fieldMap
