## Implementation Summary

### FN-748 — Backend granular stop endpoints (branch: `backend/FN-748/load-granular-stop-endpoints`)

Added 4 new route handlers to `backend/packages/goodmen-shared/routes/loads.js`:

**`POST /api/loads/:id/stops`**
- Requires admin or dispatch role
- Validates `stop_type` against `STOP_TYPES` enum (PICKUP/DELIVERY)
- Auto-assigns `sequence = MAX(existing) + 1` via `COALESCE(MAX(sequence), 0)`
- Returns full `getLoadDetail` response with recalculated trip metrics

**`PATCH /api/loads/:id/stops/reorder`** *(registered before `:stopId` to avoid route collision)*
- Body: `[{stopId, newSequence}, …]`
- Iterates items and updates each stop's `sequence` field
- Returns full `getLoadDetail` response

**`PATCH /api/loads/:id/stops/:stopId`**
- Partial update — only fields present in body are written
- Validates `stop_type` if provided
- Dynamic `SET col = $N` builder; returns 400 if no updatable fields
- Returns full `getLoadDetail` response

**`DELETE /api/loads/:id/stops/:stopId`**
- Hard deletes the stop row
- Fetches remaining stops `ORDER BY sequence, created_at` and renumbers them 1-based
- Returns full `getLoadDetail` response

All 4 handlers use `getClient()` transactions (BEGIN/COMMIT/ROLLBACK) and return the complete load detail including recalculated `empty_miles`, `loaded_miles`, `total_miles`, `rate_per_mile`.

## Key Decisions
1. **Route registration order** — `PATCH /:id/stops/reorder` is registered before `PATCH /:id/stops/:stopId` so the literal string "reorder" is not captured as a `:stopId` parameter by Express.
2. **`COALESCE(MAX(sequence), 0) + 1`** — safe for loads with zero existing stops; sequence starts at 1.
3. **Renumber after delete** — after removing a stop, all remaining stops are renumbered 1-based in `sequence, created_at` order to keep sequences contiguous.
4. **Partial update pattern** — `addField()` helper builds `SET col = $N` dynamically; returns 400 early if no recognized field is present, avoiding a no-op UPDATE.
5. **Transaction wraps INSERT/UPDATE/DELETE only** — `getLoadDetail` (which runs `computeTripMetrics`) is called after `COMMIT` so it reads committed data including the latest stop changes.

## Deployment Handoff
- **Services to restart**: `logistics-service` (loads routes live here)

- **Env/config changes**: None
- **Deploy order**: Restart `logistics-service` after deploy

  4. `DELETE /api/loads/:id/stops/:stopId` — remove a stop; confirm remaining stops renumbered from 1
