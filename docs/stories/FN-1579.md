# FN-1579: Work Order Basics tab → vehicle search not scoping to selected customer (regression after shop_client_id rename)

## Requirement
On the Work Order page → Basics tab → Vehicle Info section, the **Search Vehicle (Unit, VIN, Make, Model)** input must surface the operator's vehicles when the operator's own customer entry is selected (e.g. selecting "GOODMEN LOGISTICS LLC" should show GOODMEN's fleet trucks, including unit 1098). It must continue to scope to a third-party customer's `customer_vehicles` rows when one of them is selected.

## Real Root Cause (revised after FN-1580 investigation)
The originally-filed cause (`customer_vehicles.customer_id → shop_client_id` rename leaving the filter stale) was already addressed in **FN-1465** (commit `bcb845d0`, PR #713, merged 2026-05-06). That fix restored the `customer_vehicles` path. The user's reported failure on dev — typing "1098" under GOODMEN returns nothing — is a *different*, structural issue:

- Unit 1098 is a fleet truck (`vehicles` table, `vehicle_source='fleet'`, `operating_entity_id=7e5a7f03-…`, `operating_entity_name='Goodmen Logistics'`).
- Per the `all_vehicles` view (migration `20260506045211_update_all_vehicles_view_ownership_type.js`), the fleet branch sets `shop_client_id = NULL` for **every** fleet row. By design.
- GOODMEN-as-customer is a `shop_clients` row (id `e2497242-…`, dot_number `3761784`), separate from the `operating_entities` row representing GOODMEN-as-carrier (id `7e5a7f03-…`, also dot_number `3761784`). No FK links them.
- The current filter (`v.shop_client_id ?? v.customer_id` === selected `customerId`) cannot match a fleet truck under any shop_client. This was true before the rename too — fleet trucks have never been visible under a shop_client filter.

DOT number is the natural anchor between the two tables: `operating_entities.dot_number` has a unique index (`uq_operating_entities_dot_not_null`), and `shop_clients` carries `dot_number` from FMCSA lookup.

## Acceptance Criteria
- [ ] With customer "GOODMEN LOGISTICS LLC" (DOT 3761784) selected and "1098" typed into Search Vehicle, unit 1098 appears in the dropdown.
- [ ] With "SHEBA LOGISTICS LLC" (DOT 2461599) selected, GOODMEN's fleet vehicles do **not** appear (no DOT cross-leak).
- [ ] With no customer selected, the dropdown falls back to the full vehicle pool.
- [ ] When a customer with no DOT is selected, only that shop_client's `customer_vehicles` appear (no DOT-based extension fires).
- [ ] Selecting a customer-owned vehicle (with no customer pre-selected) auto-populates the customer field (existing behavior, unchanged).
- [ ] Switching customer clears any previously selected vehicle that doesn't belong to the new customer (existing behavior, applied DOT-aware).

## Implementation Summary

### FN-1582 (backend subtask)
Added `oe.dot_number AS operating_entity_dot_number` to both SELECT branches of `GET /api/vehicles` (the LEFT-JOIN-on-`operating_entities` branch and the no-table fallback). The join itself already existed for `oe.name`; this is purely an additional column. No other endpoints touched. `customer_vehicles`-sourced rows naturally surface `null` for the field because they have `operating_entity_id = NULL` in the `all_vehicles` view.

### FN-1580 (frontend subtask)
Refactored the vehicle-search predicate in `basics-tab.component.ts` into a single `vehicleBelongsToCustomer(vehicle, customerId)` helper, used by both `getVehicleSearchPool()` and `applyCustomerVehicleFilter()`. The helper accepts a vehicle as belonging to the customer if **either** the vehicle's `shop_client_id` (or legacy `customer_id`/`customerId`) equals the selected customerId, **or** the vehicle's `operating_entity_dot_number` (from FN-1582) equals the selected customer's `dot_number`. The DOT branch only fires when the customer has a non-null DOT, so third-party customers without FMCSA data still get strict shop_client filtering.

## Key Decisions
- **DOT match over name match**: chose `operating_entity_dot_number === customer.dot_number` rather than name matching ("GOODMEN LOGISTICS LLC" ≈ "Goodmen Logistics"). DOT is unique-indexed and immutable; names drift.
- **No schema FK between `shop_clients` and `operating_entities`**: deferred. Adding a `shop_clients.operating_entity_id` would be cleaner long-term but expands scope to migration + backfill + cross-feature audit. Filed as a follow-up consideration.
- **Backend extends payload, frontend filters**: kept the existing client-side filter design rather than introducing `?customer_id=` server filter. Vehicle counts are small enough that client-side scales for now.
