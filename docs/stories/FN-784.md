## Implementation Summary

### FN-794 (subtask)
Intelligence Panel shell ships with four metric cards and a pill period selector above the loads list.

- **New `IntelligencePanelComponent`** (`frontend/src/app/components/loads-dashboard/intelligence-panel/`)
  - Presentational / dumb. Accepts `[period]`, `[metrics]`, `[loading]`, `[needsAttentionActive]`; emits `(periodChange)` + `(needsAttentionClick)`.
  - Pill selector: Today / Week / Month / All — one active pill at a time.
  - Four cards: **Gross** (sum of rates, $-formatted), **Delivered** (count of DELIVERED/COMPLETED), **In Transit** (count of IN_TRANSIT / EN_ROUTE / PICKED_UP / DISPATCHED, with a pulsing green dot in the header), **Needs Attention** (clickable `<button>`, active highlight when the filter is on).
  - Trend arrows render only when `previous` is supplied on the metric — otherwise the card hides the trend row so we never show a misleading 0%. Direction inferred from a ±0.5% dead-band. Colors: green up / red down / slate flat. For the Needs Attention card the semantics invert (more attention = worse), so the CSS flips the red/green assignment via `data-direction`.
- **Wiring in `LoadsDashboardComponent`**
  - New state: `intelligencePeriod`, `intelligenceMetrics`, `needsAttentionActive`, plus a private `prevIntelligenceAggregate` cache.
  - `onIntelligencePeriodChange(period)` updates the pill, maps to the existing `grossPeriod` values (today, this_week, this_month, all), reloads the current page, and fires a background fetch for the previous equivalent period used to compute trends.
  - `recomputeIntelligenceMetrics()` assembles the four cards from the currently loaded rows and the cached previous-period aggregate. Runs after every `loadLoads()` success.
  - `isNeedsAttention(load)` — composite predicate: DRAFT, `needs_review === true`, or a delivery date in the past with a non-terminal status. Powers both the card count and the client-side filter.
  - Client-side `filteredLoads` gains a `needsAttentionActive` guard so toggling the card instantly narrows the table; no extra server request.
  - Panel embedded in the template just below the alerts, above the existing `.status-summary-bar`, and hidden for driver-role views.
- **Module**: `IntelligencePanelComponent` declared in `LoadsDashboardModule` (it's only used here, so it stays out of `SharedModule`).

### Files changed (FN-794)
- `frontend/src/app/components/loads-dashboard/intelligence-panel/intelligence-panel.component.{ts,html,scss}` — **new**.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.module.ts` — declare the new component.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` — intelligence state + handlers + metric computation + previous-period fetch + needs-attention predicate + `filteredLoads` guard.

### Key Decisions (FN-794)
- **Dumb component, smart parent.** The panel receives metrics — it does not fetch them. When FN-793 ships `/api/loads/ai-insights`, the parent can swap its aggregate source without touching the panel.
- **Trend hidden until previous is known.** Rendering a "0.0% vs prev." on first paint would suggest stable performance even when we simply haven't fetched prior-period data yet. `null previous → no trend row` keeps the UI honest.
- **Client-side Needs Attention filter.** The server has no composite "drafts + overdue + needs_review" filter, and the three components are cheap to check on the currently loaded page. This mirrors how the existing `needs_review` chip narrows results client-side.
- **Previous-period fetch caps pageSize at 500.** The existing list endpoint doesn't expose period aggregates, so we do a second listLoads call bounded to 500 rows for trend. That's enough for realistic trucking fleets and bounded enough to avoid accidentally paging the entire dataset. When FN-793 lands, the fetch will shrink to a single `/ai-insights` call.
- **Existing `grossPeriod` retained.** Rather than splitting the period state, the Intelligence pills drive `grossPeriod` via a small mapping so the existing status summary bar + list filters keep working while we ship the new panel. FN-795 removes / collapses the old bar as part of its scope.

### FN-795 (subtask)
Ships the AI insights list (with 60-second polling) and collapses the Status / Billing chip rows into a default-hidden breakdown.

- **New `IntelligenceInsightsComponent`** (`frontend/src/app/components/loads-dashboard/intelligence-panel/intelligence-insights.component.{ts,html,scss}`)
  - Polls `LoadsService.getAiInsights(period)` on mount, on every `[period]` change, and every 60 seconds (`timer(0, refreshMs)`). The Rx stream is cancelled on destroy.
  - Renders each insight as a card with an icon (type-derived or service-supplied), title, optional subtitle, and an optional right-aligned action button. Severity-based colored left border (blue / amber / red).
  - Supports an optional `href` for navigation via `routerLink` and an `(action)` output for app-level handlers (e.g. "open wizard for this draft").
  - Empty state: after the first fetch completes with `[]`, the panel shows a check_circle + "everything looks healthy" message — not a spinner or error. Before the first fetch it says "Loading insights…".
  - Shows a small inline spinner while a refresh is in flight.
- **Shared insight contract** (`loads.service.ts`)
  - New types `AiInsight`, `AiInsightType`, `AiInsightSeverity` locked to what the panel consumes. FN-793's backend endpoint will fill this shape.
  - `getAiInsights(period)` returns the typed payload and **degrades gracefully** when the endpoint is missing (catches 404/error → emits `{ success: false, data: [] }`). This is the contract that lets FN-795 ship independently of FN-793 — the empty state renders instead of a broken UI.
- **Collapsible Status/Billing chip rows**
  - `status-summary-bar` gets a new "Show/Hide breakdown" toggle in its header (chevron icon + label, `aria-expanded`).
  - `chipRowsOpen` defaults to `false`, so on first render only the period pills + TOTAL remain visible.
  - When expanded, the section now splits into **two** sub-sections: Status breakdown (the existing track + legend) and a new **Billing breakdown** with its own color palette (`billingColorMap`) and `billingBarSegments` getter.
  - Clicking a billing legend item toggles `filters.billingStatus` via the existing `setBillingFilter()` helper.
- **Module registration**: `IntelligenceInsightsComponent` declared in `LoadsDashboardModule` next to the panel shell (both live in `intelligence-panel/`).

### Files changed (FN-795)
- `frontend/src/app/components/loads-dashboard/intelligence-panel/intelligence-insights.component.{ts,html,scss}` — **new**.
- `frontend/src/app/services/loads.service.ts` — `getAiInsights(period)`, plus `AiInsight*` shared types.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.module.ts` — register the new insights component.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` — `chipRowsOpen` state, `toggleChipRows()`, `billingColorMap`, `billingBarSegments` getter.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.html` — embed insights next to the panel, wrap Status/Billing chip rows in the collapsible with the new toggle.
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.scss` — toggle button, collapsible container, section labels, reveal animation.

### Key Decisions (FN-795)
- **Polling via `timer(0, 60_000)`.** Fires an immediate fetch *and* every 60s thereafter. Period changes call `_restartPolling()` so the user doesn't wait up to 60s for fresh data.
- **Graceful 404 degradation.** The service's `catchError(() => of({ ... empty }))` means the insights panel doesn't flash an error banner while FN-793 is still in flight across environments — the empty-state message serves both "no insights to show" and "backend not ready yet", which is a reasonable product behavior.
- **Component stays out of SharedModule.** Intelligence panel pieces are only used on `/loads`, so they live in `LoadsDashboardModule` next to the shell. SharedModule continues to carry only multi-feature widgets.
- **`<details>` vs manual toggle.** Used a manual `*ngIf` toggle so the toggle button's appearance, reveal animation, and `aria-expanded` are styled consistently with the dark theme. `<details>` was tempting for the accessibility baseline but the summary styling is a ways off the rest of the panel.
- **Billing breakdown added alongside Status.** The original `.status-summary-bar` showed only Status. The AC calls for "Status/Billing chip rows" in the collapsible, so the billing breakdown was added as a sibling section — using existing `summaryTotals.byBilling` data that was already populated but never rendered.

## Deployment Handoff

### FN-794
- **Services to redeploy**: `fleetneuron-frontend` (Angular bundle only).
- **Database migrations**: none.
- **Environment / config changes**: none.
- **Cache invalidation**: none.
- **Feature flags**: none.
- **Deploy order**: standalone frontend deploy — no backend dependency yet (FN-793 will add the backend aggregate endpoint; the panel tolerates its absence).
- **Post-deploy verification**:
  1. Open Loads → the new Intelligence Panel renders above the summary bar with Gross, Delivered, In Transit (with pulsing green dot), and Needs Attention cards.
  2. Click **Week** → panel pills switch, list reloads for the current week, Needs Attention count updates. Trend arrows appear on Gross / Delivered / In Transit / Needs Attention when the previous-week fetch returns (may briefly be absent during the network round-trip — expected).
  3. Click **Needs Attention** card → the card gets an orange active ring and the table narrows to drafts + overdue + needs-review rows. Click again → clears.
  4. Click **All** → period shows "All", trend arrows disappear (no previous-period comparison available for "all").
  5. Driver role login → Intelligence Panel must not render (it's hidden alongside the other dispatcher-only controls).

### FN-795
- **Services to redeploy**: `fleetneuron-frontend` (Angular bundle only). No backend dependency — `GET /api/loads/ai-insights` returns 404 until FN-793 ships, and the insights panel shows an empty state in that case.
- **Database migrations**: none.
- **Environment / config changes**: none.
- **Cache invalidation**: none.
- **Feature flags**: none.
- **Deploy order**: standalone frontend deploy. When FN-793 merges and the backend endpoint is live, no frontend change is needed — the insight cards simply start appearing on the next 60-second tick.
- **Post-deploy verification**:
  1. Open Loads → below the 4 metric cards a "AI insights" panel is visible. Before FN-793 ships: reads "everything looks healthy" after the first fetch. After FN-793 ships: shows 0–N insight cards.
  2. Open DevTools Network tab → confirm `GET /api/loads/ai-insights?period=all` fires on load and again ~60 seconds later (with the current period in the query string).
  3. Click a period pill → `/api/loads/ai-insights?period=week` (or today/month/all) fires immediately, and the 60s timer restarts.
  4. Status summary bar — "Show breakdown" button renders on the right side of the header. Click → Status + Billing sections reveal (each with its own track + legend chips). Click a billing chip → the list filters by that billing status. Click "Hide breakdown" → both sections collapse.
  5. First-time user: the breakdown is collapsed by default (no status/billing chips visible until they expand it).
