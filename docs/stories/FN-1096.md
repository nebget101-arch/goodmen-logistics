# FN-1096: AI Photo-of-Part intake

## Requirement
Let users snap or upload a photo of a part and auto-fill the Add Part form with confidence-scored suggestions, modeled on the existing Claude Sonnet 4 vision pattern.

## Acceptance Criteria
- [ ] AI handler returns structured fields + per-field confidence (0–1).
- [ ] `POST /api/ai/parts/identify-from-photo` wraps handler, uploads image to R2.
- [ ] On Save, image URL persists to `parts.image_url`.
- [ ] "Quick Add → Snap Photo" entry point on parts catalog.
- [ ] Modal pre-fills with high/medium/low confidence badges per field.
- [ ] Manufacturer suggestion binds to FN-1091 typeahead.
- [ ] QA: happy path, low-confidence path, R2 upload verified.

## Implementation Summary
_To be filled by implementing agent_

### FN-1097 (subtask)
Implemented the `parts-vision-handler` AI module and wired it to a public route at `POST /api/ai/parts/identify-vision`. Used the Anthropic SDK with the same Claude vision model as the toll handler (`claude-sonnet-4-20250514` via `ANTHROPIC_VISION_MODEL` env var). The handler accepts `{imageBase64, mimeType}` and returns the spec'd output shape:

```
{
  manufacturer, partNumber, category,
  descriptionGuess, dimensionsGuess,
  confidence: { manufacturer, partNumber, category, description, dimensions },
  isUnreadable, warnings
}
```

Adopted the `loads-nlq-handler` deps-injection pattern (`deps.anthropic`) so unit tests can mock the model without real API calls. The system prompt includes three few-shot examples covering sharp/blurry/unreadable cases and explicit confidence-scoring rules to keep the model honest.

When the AI flags `isUnreadable: true`, the handler returns HTTP 422 with code `AI_IMAGE_UNREADABLE` (structured error, not a throw — per AC). Malformed model output returns 502 `AI_PARSE_ERROR`. Upstream Anthropic errors return 502 `AI_VISION_ERROR`. All confidence values are clamped to `[0, 1]`.

11 unit tests cover: confidence clamping, validation defaults, happy path, low-confidence response, malformed response, markdown-fenced JSON, unreadable image, missing input, unsupported media type, upstream error. Wired into `package.json` test script alongside the existing handler tests.

FN-1098 can now `require('./handlers/parts-vision-handler').handlePartsVision` and call it from its own R2-aware route.

### FN-1098 (subtask)
Added `POST /api/ai/parts/identify-from-photo` — a multipart-aware wrapper around the FN-1097 vision handler that uploads the photo to R2 first and returns both the AI extraction and the storage key. The endpoint accepts either `multipart/form-data` (field `image`) or JSON `{imageBase64, mimeType}`, both capped at 10MB.

Refactored `parts-vision-handler.js` to expose a pure `extractPartFromImage({imageBase64, mediaType, deps, route})` helper that returns a tagged result (`{kind, status, body, processingTimeMs, model}`). `handlePartsVision` (the existing `/identify-vision` route handler) now thin-wraps it, and the new photo-intake handler reuses it after R2 upload — no HTTP fan-out, single Anthropic call per request.

`r2Key` is built as `parts/photos/<uuid>.<ext>` using `crypto.randomUUID()` for unguessable, collision-free keys. The key is returned even when the AI flags the image unreadable (HTTP 422), so the FE (FN-1099) can decide whether to retry or discard. R2 upload failure short-circuits to 502 `R2_UPLOAD_FAILED` without spending an Anthropic call.

Extended `parts.service.js` with `resolveImageR2Patch()` and wired it into `createPart` / `updatePart` so the FE can save the prefilled form by sending `image_r2_key`. The patch is purely additive: when `image_r2_key` is supplied, the service writes its value to `parts.image_url`; when it's absent, partial-update semantics are preserved (no clobber). `null` or empty string clears the image. Stored value is the R2 object key (not a pre-signed URL), since pre-signed URLs expire and must be generated on read.

13 unit tests cover the photo-intake handler (multipart + JSON happy paths, unreadable image with r2Key, R2 failure, missing image, oversized image, unsupported mime, R2 key shape). 8 unit tests cover the parts.service patch helper. All pass; full ai-service `npm test` is green.

### FN-1099 (subtask)
Wired the parts-catalog page to the FN-1098 endpoint:

- **Quick Add split-button** in the page header opens a 3-item dropdown. Only "Snap Photo" is enabled — "Paste SKUs" and "Clone Existing" are present but disabled-with-tooltip per the AC ("others come from later stories — leave their menu items disabled-with-tooltip"). The button is disabled while an AI call is in flight to avoid double-fires.
- **Snap Photo trigger** uses a hidden `<input type="file" capture="environment" accept="image/jpeg,png,webp,gif">`. On mobile, browsers honor `capture=environment` and open the rear camera; desktop browsers fall back to the standard file picker. Resetting `value` before `.click()` lets the user re-pick the same file back-to-back.
- **`AiPartsService` (new, `services/ai-parts.service.ts`)** wraps `POST /api/ai/parts/identify-from-photo`. Sends multipart (HttpClient sets the boundary; we never set `Content-Type` manually). Normalizes the response (defensively defaults missing nested fields). Maps known BE error codes (`AI_IMAGE_UNREADABLE`, `AI_IMAGE_TOO_LARGE`, `AI_BAD_REQUEST`, `R2_UPLOAD_FAILED`) to friendly messages and a 0-status network error.
- **Loading state** is a centered overlay with a spinner + Cancel button. Cancel emits via a per-call `Subject` piped through `takeUntil`, which terminates both the FE observable and the underlying XHR (HttpClient unsubscribes the request).
- **Prefill flow** opens the existing Add Part modal and patches `sku` (← AI `partNumber`), `category`, and `description`. Manufacturer is special: per the AC, the suggestion is shown in the FN-1091 typeahead with `id: null` so the user must intentionally pick an existing master record OR explicitly create a new one — the prefill never silently creates a `manufacturers` row.
- **Confidence badges** are a new shared component (`components/shared/confidence-badge/`). Buckets follow the AC scale: high (≥0.85, green), medium (0.6–0.84, yellow), low (<0.6, red). Badges render inline with each field label (SKU, Category, Manufacturer, Description) when an AI confidence value is present.
- **Override clears the badge.** A `valueChanges` listener wired in `ngOnInit` watches `sku`, `category`, and `description`; the first edit clears the matching `aiConfidence[*]` slot. Manufacturer clears in `onManufacturerPick` since the typeahead doesn't fire `valueChanges` directly. The `destroy$` Subject tears these down on component destroy.
- **Save includes `image_r2_key`.** When `aiR2Key` is set, `savePart` adds `image_r2_key` to the createPart/updatePart payload — the FN-1098 BE persists it to `parts.image_url`.
- **Failure path**: any non-2xx response or AI `isUnreadable: true` opens the form **empty** + sets `errorMessage` so the user can fill manually (per AC). The `errorMessage` slot was already present on the page; no new toast widget needed.
- **AI warnings panel.** When `aiResult.warnings[]` is populated (model surfaces partial reads / glare / low-light), they render in a small warning block at the bottom of the form so the user knows which fields to scrutinize.
- **Tests.** Three new spec files compile under strict TS:
  - `confidence-badge.component.spec.ts` (5 tests): bucket boundaries, null/undefined/NaN, percentage formatting.
  - `ai-parts.service.spec.ts` (5 tests): multipart POST shape, sparse-response normalization, error code → friendly message mapping for unreadable / oversize / unknown.
  - `parts-catalog.component.spec.ts` (10 tests, FN-1099-scoped): toggle dropdown; happy path (AI call → modal open → prefill → manufacturer with id=null); `image_r2_key` in createPart payload; badge clears on each field override (sku/category/description/manufacturer); error path (open empty + errorMessage); `isUnreadable=true` soft-failure; cancel-during-loading interrupts the in-flight call; closeForm clears AI state.

`ng build --configuration development` is green on the worktree (Node 20). The repo's `ng test` runner is not wired (no karma/jasmine devDeps in `frontend/package.json`); all new specs typecheck under strict-mode and follow the existing FN-1094 spec patterns so they'll run as soon as the runner is provisioned.

## Key Decisions
_To be filled by implementing agent_

### FN-1097 (subtask)
- **Reused `ANTHROPIC_VISION_MODEL` env var** (default `claude-sonnet-4-20250514`) — matches toll-invoice-vision-handler exactly per AC's "reuse exactly" wording.
- **Deps injection over module-level lazy client** — toll handler uses `getAnthropicClient()` directly; we kept that as a fallback but accept `deps.anthropic` first (loads-nlq-handler pattern). This makes the mocked unit tests possible without monkey-patching globals.
- **Added `isUnreadable` flag to the AI output schema** — the spec required "structured error (not throw) when image is unreadable." Letting the model self-flag is cleaner than threshold-on-confidence (which would conflate "blurry but identifiable" with "unusable").
- **Public route `/api/ai/parts/identify-vision`** — followed the toll-invoice-vision naming convention. FN-1098 will add a separate `/identify-from-photo` route that wraps multipart upload + R2 storage and calls the vision handler internally.

### FN-1098 (subtask)
- **Reused FN-1097 handler via in-process function call, not HTTP** — extracted `extractPartFromImage` as a pure helper and import it from the photo-intake handler. Avoids an internal HTTP hop, double-logging, and double-counting in analytics. Closes the "Open Items" entry from FN-1097.
- **Two input formats — multipart `image` AND JSON `{imageBase64, mimeType}`** — the FN-1099 modal will likely send multipart from a `<input type=file>`, but mobile/in-process callers may already have base64. Supporting both means no second route. Multer is wired only for the new endpoint and uses `memoryStorage` (10MB limit) — the buffer goes straight to R2 without touching disk.
- **`r2Key` returned even on 422 unreadable** — the FE can offer "this image isn't readable, retry?" with the option to delete the upload, instead of orphaning it in R2. The cleanup of orphaned objects is out of scope for this story (a future lifecycle policy or sweeper job).
- **Stored value in `parts.image_url` is the R2 key, not a pre-signed URL** — pre-signed URLs expire (default 15 min); persisting them creates broken links. Reads should call `getSignedDownloadUrl(key)` on demand. This is consistent with how the rest of the app uses `r2-storage.js` (the helper returns `{key}` from `uploadBuffer`).
- **`crypto.randomUUID()` for the key, not `Date.now()+random`** — unguessable, collision-free, and matches the URL-friendly base62-ish convention. Files are namespaced under `parts/photos/` so a future R2 lifecycle rule can target them without affecting other prefixes.
- **No service-side rate limiting** — consistent with every other `/api/ai/*` route. The gateway (`fleetneuron-api-gateway`) is the chokepoint and already enforces JWT verification + global rate limits before requests reach this service.
- **Defensive base64 size check** — multer enforces 10MB on multipart, but the JSON path uses `bodyParser.json({limit: '20mb'})` from server.js, so we re-validate the decoded base64 byte count and return 413 `AI_IMAGE_TOO_LARGE` to keep the contract consistent across both input formats.

### FN-1099 (subtask)
- **AI partNumber → SKU control.** The form has no separate `partNumber` input; `sku` is the primary identifier and was the obvious target for the AI partNumber suggestion. The user can override (e.g., to add a fleet prefix) and the badge clears on first edit.
- **Manufacturer prefill uses `id: null`** even when the AI confidently identifies "Bosch". This forces the user to *intentionally* pick an existing master record from the FN-1091 typeahead OR explicitly use the typeahead's "Create new" affordance. Without this, the prefill would silently create master rows on every Save and pollute the manufacturers table — the AC's "creating a new master record is intentional" requirement.
- **Treat `isUnreadable: true` as a soft-failure.** It comes back HTTP 200 from the BE (it's a successful classification — the model just couldn't read it), but for the user the experience should be "AI couldn't help — fill it in manually." We open an empty form + show the model's first warning. The R2 image is discarded for now (we don't keep `aiR2Key` on isUnreadable paths) — orphan-cleanup is out of scope per FN-1098.
- **Loading overlay over Cancel inside the modal.** The catch is timing: the modal isn't open yet when the AI call fires, so a per-modal cancel button doesn't exist. Instead we use a fixed-position overlay that covers the whole page and offers a single Cancel button — visually consistent with how progress is shown on other AI flows in this app (`AiExtractionFlowComponent`).
- **No new toast widget.** The page already has an `errorMessage` slot at the top. Reusing it keeps the visual design consistent and avoids introducing a third notification surface.
- **No getter-bound `[options]`** anywhere — the Quick Add menu items are static literals, the typeahead uses arrow-class-property `searchFn`/`createFn` per the FN-1094 / FN-317 RCA convention.
- **`?? null` coercion in template badge bindings.** Angular's strict template checker treats `@Input x: number | null | undefined = ...` as `number | null` once a default is provided. Rather than weaken the input type with `!:` or use `transform`, we coerce undefined→null at the binding site (`aiConfidence.category ?? null`). Keeps both the template and the component cleanly typed.
