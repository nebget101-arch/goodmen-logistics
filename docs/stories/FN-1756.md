### FN-1761 (subtask) — backend geocode proxy + vehicle_id filter

**New `services/geocode-service.js`** — server-side Nominatim/OSM forward-geocode proxy:
- Descriptive `User-Agent` (`FleetNeuron/1.0 (ops@fleetneuron.ai)`) per the Nominatim usage policy.
- In-process **short-TTL cache** (60 s) keyed by the normalized query (lowercased, whitespace-collapsed); LRU eviction at 500 entries. The cache is the primary mechanism for respecting the ~1 req/s policy — repeated keystrokes / re-searches don't re-hit upstream. `meta.cached` flags a cache hit.
- Base URL from `GEOCODER_BASE_URL` (defaults to `https://nominatim.openstreetmap.org`).
- `axios` is **lazy-required** inside the HTTP call so the module loads without the dependency present (keeps unit tests hermetic; tests inject `opts.search`).
- **`address_id` enrichment** (best-effort): for each result, looks up the tenant's saved `locations` (raw `query` bridge) and stamps `address_id` when a location's normalized `address` is contained in the result label (min length 5, conservative to avoid false links). `locations` has no lat/lng columns, so the match is text-based. A miss simply leaves `address_id` unset; absent `locations`/`query` degrades to "no match".

**`routes/geofences.js`**:
- `GET /api/geofences/geocode?q=<address>` — registered **before** `/:id` so "geocode" isn't captured as an id. Returns `{ data: [{ label, lat, lng, type, address_id? }], meta: { total, cached } }`. `400` blank `q`, `403` no tenant, `502` upstream error.
- `GET /api/geofences?vehicle_id=<id>` — new list filter (accepts `vehicle_id` or `vehicleId`).

**`services/geofence-service.js`** — `listGeofences` `vehicleId` filter: keep geofences with a trigger scoped to that vehicle **or** a tenant-wide (`vehicle_id` NULL) trigger that also applies to it; geofences with no applicable trigger are dropped.

**Tests** (`node --test`, Node 20): `geocode-service.test.js` (row→wire mapping, invalid-coord drop, cache-hit path with mocked Nominatim, blank-query short-circuit, `address_id` matched/unmatched) + a `vehicle_id` filter case added to `geofences.test.js`. 27 + 6 green; geofence-event-worker suite still green (no regression).

> **API contract for FN-1762 (frontend):**
> - `GET /api/geofences/geocode?q=` → `{ data: [{ label, lat, lng, type, address_id? }], meta: { total, cached } }`. Debounce the address box; when a chosen result carries `address_id`, set it on the created geofence.
> - `GET /api/geofences?vehicle_id=<vehicleId>` → existing `{ data, meta }` list envelope, narrowed to fences relevant to that unit. Use it for the per-unit view.
> - Per-unit scoping persists via `geofence_triggers.vehicle_id` on trigger create/update (already supported by the trigger endpoints; wire payload key is `vehicleId`).

### FN-1762 (subtask) — frontend address search + per-unit scoping + nav entry

**Address search** (`geofences.component`):
- New search box at the top of the editor form. Input feeds a `Subject` debounced 350 ms + `distinctUntilChanged` → `GeofenceService.geocode(q)` (queries ≥ 3 chars; shorter input clears results). A spinner shows while in flight; upstream errors surface a non-blocking banner.
- Results render as a dropdown; selecting one **flies the map** (`setView` zoom 14), **drops an editable Leaflet circle** at the point (default radius **200 m**, captured as the geometry), pre-fills the name (when blank) and stamps `addressId` on the payload when the result carries one.
- `GeofenceService.geocode()` maps the wire `address_id` → `addressId`; consumes the `{ data, meta }` envelope.

**Per-unit scoping** (`geofences.component`):
- New **"Applies to"** segmented control (`app-ai-segmented-control`): *All units* (default) vs *Specific unit*. Choosing *Specific unit* lazily loads the vehicle list (`ApiService.getVehicles`) into an `app-ai-select`. `vehicleOptions` is a plain mutable field assigned once after load — **not** a getter (FN-317).
- On save, `toPayload` stamps the chosen vehicle id onto **every** trigger's `vehicleId` (or clears it for *All units*). The per-trigger free-text "Vehicle ID" input was removed in favor of this control. `edit()` derives the scope back from the triggers (single shared vehicle id ⇒ unit-scoped).
- **Deep link from Equipment**: `vehicles.component` "Create geofence for this unit" button (vehicle detail modal) → `router.navigate(['/geofences'], { queryParams: { vehicle_id, unit } })`. The page reads the query params, filters the saved list by `vehicle_id`, shows a scope banner with a "Show all" reset, and opens the editor pre-scoped (applies-to = unit, vehicle preselected, a default trigger seeded so the scope persists).

**Nav entry** (`nav.config.ts`): added `{ path: '/geofences', label: 'Geofences', tab: 'vehicles' }` to the **Equipment** section children. (Optional `Live Map`/`/tracking` link intentionally left to FN-1751, which owns the tracking page, to avoid a dead link / duplicate.)

**Theme/tests**: AI dark-theme tokens only, reusing the existing geofence palette (no new hex). Extended `geofences.component.spec.ts` (geocode selection drops a circle + pre-fills; per-unit stamps/clears `vehicleId`; `addressId` in payload) and `geofence.service.spec.ts` (`geocode` mapping; `vehicle_id` filter). Verified via `ng build` (Node 20) — clean compile (AOT), bundle emitted; karma not runnable in this environment.

## Key Decisions
- Geocoder: Nominatim/OSM, backend proxy (free, no key) — intake 2026-06-12.
- Per-unit scoping reuses existing `geofence_triggers.vehicle_id` — no new schema.
