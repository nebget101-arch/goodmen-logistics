# FN-1452: FMCSA import queue never calls registerImporter — manual triggers fail with "No importer registered"

## Symptom

`POST /api/fmcsa/imports/run` with `{"files":["census","authority"]}` against `dev` returned 202 with `runIds`, but every ledger row finished with:

```json
{ "status": "error", "rows_inserted": 0, "rows_updated": 0, "rows_skipped": 0,
  "error_message": "No importer registered for file 'authority'" }
```

Same for `census` / `inspections` / `crashes` / `sms`. Reproduced 2026-05-06 against `dev`.

## Root cause

The control plane (FN-1415 / FN-1424) merged `services/fmcsa-import-queue.js` to `dev` with a `registerImporter(file, fn)` registry that was **never invoked anywhere** (`grep -rn "registerImporter\b" backend` returns 0 callers outside the queue itself). The five importer modules (FN-1413/FN-1420 census + authority; FN-1414/FN-1422 inspections + crashes + sms) shipped separately and self-register nothing. Every `run-import` job hit the no-importer branch at `fmcsa-import-queue.js:128` and recorded `status='error'`.

## Implementation Summary

### Adapter shapes

The queue calls each registered importer as `importer(knex, { dryRun })` and expects `{ rowsInserted, rowsUpdated, rowsSkipped }`. The two existing driver families have different shapes, so each adapter normalizes:

- **census / authority** — call `runCensusImport({ knex, source: { url: DEFAULT_CENSUS_URL }, triggeredBy: 'manual' })` (same for authority with `DEFAULT_AUTHORITY_URL`). The runner returns `{ runId, counts: { inserted, updated, skipped } }`; the adapter maps `counts.* → rows*`.
- **inspections / crashes / sms** — open a Readable stream from `process.env.FMCSA_INSPECTION_URL` / `FMCSA_CRASH_URL` / `FMCSA_SMS_URL` (with optional `FMCSA_DOWNLOAD_USER` / `FMCSA_DOWNLOAD_PASS` for Basic auth) and call `runInspectionImport({ source: stream, triggeredBy: 'manual' })`. These drivers already return the `rows*` shape verbatim.

### dryRun behaviour

Every adapter short-circuits to `{ rowsInserted: 0, rowsUpdated: 0, rowsSkipped: 0 }` before doing any work. Two reasons:

1. **AC compliance** — the queue's existing `dryRun` short-circuit only fires when *no* importer is registered. Once we register importers, `dryRun` would otherwise fall through to the importer and do real work; the AC requires it to record zero rows.
2. **Avoids duplicate `import_runs` rows** — the underlying `runner.js` (census/authority) and `inspections.js`/`crashes.js`/`sms.js` each insert their *own* `import_runs` row. Short-circuiting on dry runs keeps the dev-environment dry-run truly side-effect-free.

There's a residual quirk on real (non-dry) runs: census/authority/inspections/crashes/sms drivers each create a second `import_runs` row internally in addition to the one the queue creates. This pre-existing duplication is **out of scope per the ticket** ("Changes to the importer modules themselves — they work; only the wiring is missing"). A follow-up should refactor the drivers to accept an externally-managed `runId` so the queue's row is the single source of truth.

### Source URL convention for snapshot files

Inspections / crashes / sms expect a Readable stream — there is no public Socrata URL like census/authority have. The adapters read the source URL from env vars, mirroring the existing `FMCSA_DOWNLOAD_USER`/`FMCSA_DOWNLOAD_PASS` convention from `fmcsa-importer/index.js`:

| File | Env var |
|------|---------|
| inspections | `FMCSA_INSPECTION_URL` |
| crashes | `FMCSA_CRASH_URL` |
| sms | `FMCSA_SMS_URL` |
| (all three, optional Basic auth) | `FMCSA_DOWNLOAD_USER` / `FMCSA_DOWNLOAD_PASS` |

When an env var is unset, the adapter throws `${ENV_VAR} is not set — cannot run '<file>' import. Set ${ENV_VAR} (and FMCSA_DOWNLOAD_USER/FMCSA_DOWNLOAD_PASS if the snapshot endpoint requires Basic auth), or trigger this file with dryRun=true.` The queue's `process('run-import')` catches it and writes the message into `import_runs.error_message`. So in dev (no snapshot URLs configured), inspections/crashes/sms `dryRun: true` succeeds with zero rows; a real trigger errors loudly with the remediation hint instead of silently no-op'ing.

### Captcha / token-broker for snapshot endpoints

Out of scope per the ticket — still an open item from FN-1422. Until snapshot URLs are configured, snapshot-file imports work via `dryRun: true` only.
