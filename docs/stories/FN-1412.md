# FN-1412: FMCSA schema + getFmcsaKnex() accessor + tenants.is_internal flag

## Requirement

Foundational data layer for the FMCSA reference dataset. Three pieces:
1. `fmcsa.*` schema with skeleton tables (carriers, authorities, inspections, violations, crashes, basic_scores, import_runs)
2. Shared accessor `getFmcsaKnex()` that all FMCSA reads/writes go through — Phase 1 returns the same Knex as the main DB; Phase 2 (future) can swap to a separate connection without consumer changes
3. `tenants.is_internal BOOLEAN DEFAULT false` column to mark the FleetNeuron internal tenant — used by FN-1415 to gate the manual import trigger

### FN-1417 (subtask) — fmcsa.* schema + tenants.is_internal

Three new migrations under `backend/packages/goodmen-database/migrations/`:

- `20260506110000_create_fmcsa_schema.js` — creates the `fmcsa` schema and 7 tables (carriers, authorities, inspections, violations, crashes, basic_scores, import_runs). Adds GIN trigram indexes on `carriers.legal_name`, `carriers.dba_name`; b-tree indexes on `carriers.mc_number`, `inspections.dot`, `inspections.inspection_date`, `crashes.dot`. CHECK constraints on `import_runs.file` (`census|authority|inspections|crashes|sms`) and `import_runs.triggered_by` (`manual|cron`). Down: `DROP SCHEMA fmcsa CASCADE`.
- `20260506110100_add_is_internal_to_tenants.js` — adds `tenants.is_internal BOOLEAN NOT NULL DEFAULT false`. Idempotent.
- `20260506110200_seed_fleetneuron_internal_tenant.js` — flips `is_internal=true` on the FleetNeuron Default Tenant row (lookup by exact name first, then ILIKE `'FleetNeuron%'` fallback). Clears any other rows to enforce a single internal tenant. Logs a warning if no FleetNeuron tenant is found instead of failing the migration (so first-boot environments stay green).

Verified locally:
- Fresh DB: `migrate:latest` runs all 189 migrations cleanly. After completion: 7 fmcsa tables, 13 indexes (PKs + 6 acceptance-criteria indexes), exactly 1 tenant with `is_internal=true`.
- Dev DB copy: `migrate:latest` applied cleanly. With 2 pre-existing tenants (FleetNeuron + Totot Logistics), only FleetNeuron is flagged internal.
- Per-migration rollback (`migrate:down`) for all 3 migrations cleanly removes the schema, the column, and resets the flag. Re-applying after rollback is idempotent.

## Key Decisions

### FN-1417
- **Schema-qualified tables via `knex.schema.withSchema('fmcsa')`** — keeps the FMCSA reference data in its own namespace, easy to drop wholesale (`DROP SCHEMA fmcsa CASCADE`) and easy to point a future Phase 2 separate-connection Knex at.
- **`bigint` for DOT** — DOT numbers are 7 digits today but the FMCSA registry has been allocating into the 4M+ range; `bigint` future-proofs without much storage cost.
- **CHECK constraints, not native PG ENUMs**, on `import_runs.file` and `triggered_by` — easier to extend later (just `ALTER TABLE … DROP CONSTRAINT … ADD CONSTRAINT …`) than dropping/re-creating an enum type.
- **Down migration uses `DROP SCHEMA fmcsa CASCADE`** instead of dropping each table — cheaper, complete, and avoids ordering bugs around the violations→inspections FK.
- **Extensions left in place on rollback** — `pg_trgm` and `uuid-ossp` are used by other features; dropping them on FMCSA rollback would break unrelated code.
- **Seed migration is forgiving** — falls back from exact tenant name → `ILIKE 'FleetNeuron%'` → warning + skip. This keeps fresh-environment migrations green even if a future seed renames the default tenant.

### FN-1418
- **Returns the shared main-DB Knex, not a wrapped instance.** Wrapping would force consumers to learn a new API and complicate Phase-2 swap. The accessor is the chokepoint *file*, not a chokepoint object — same Knex API, single import path.
- **Test calls `mainKnex.destroy()` in `after()`.** Without it, Knex's connection pool keeps the event loop alive and `node --test` hangs/cancels with a timeout. `node --test` runs each file in its own process, so destroying the shared pool here does not affect other test files.
- **SELECT 1 includes a graceful skip path** for `ECONNREFUSED` / `ENOTFOUND` / auth / missing-DB errors so unrelated dev environments do not break the suite. Live-DB smoke at the FMCSA-schema level is FN-1419's responsibility.

**Post-deploy verification (run on prod DB):**
```sql
SELECT count(*) FROM information_schema.tables WHERE table_schema = 'fmcsa';
-- expect: 7

SELECT count(*) FROM tenants WHERE is_internal = true;
-- expect: 1 (the FleetNeuron tenant)

SELECT name FROM tenants WHERE is_internal = true;
-- expect: 'FleetNeuron Default Tenant' (or whatever the prod row is named)
```

If `is_internal` count is `0` on prod (no `FleetNeuron%` named tenant), the seed migration logs a warning and exits — set the flag manually:
```sql
UPDATE tenants SET is_internal = true WHERE id = '<prod-fleetneuron-uuid>';
```

**No env vars, feature flags, or cache invalidation needed for FN-1417.**
