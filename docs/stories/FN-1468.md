# FN-1468: POST /api/work-orders fails — work_orders + invoices schema drift

## Requirement
The user reported a 500 from `POST /api/work-orders`:

```
column "requested_by_user_id" of relation "work_orders" does not exist
```

Root-cause investigation showed `work-orders.service.js` references **multiple** columns that no migration ever added. PostgreSQL only reports the first missing column, so fixing one surfaces the next. The same schema drift affects `invoices` (the WO "details" path), so similar errors will fire on labor/parts/charges/status/generate-invoice once create stops failing first.

- [ ] Migration adds to `invoices`: `tenant_id`, `operating_entity_id`

## Implementation Summary

### FN-1469 (subtask) — database
Added an idempotent migration that supplies the columns `work-orders.service.js` and `work-orders-hub.js` already read/write but the schema did not have. PostgreSQL only reports the first missing column per failed statement, which is why `requested_by_user_id` masked the rest. The migration uses `hasTable` + `hasColumn` guards so it is safe to apply on any environment, including ones where parts of the schema were already added by `20260310101000_add_multi_mc_scope_columns.js` (which independently added `tenant_id` and `operating_entity_id`).

Verified against the dev DB: `up → up → down → down → up` was fully idempotent. `down()` intentionally drops only the columns this migration owns (`cost_type`, `requested_by_user_id`) and the three indexes it creates — `tenant_id` / `operating_entity_id` belong to the prior migration and are preserved on rollback so we don't over-revert.

### FN-1470 (subtask) — backend
Fixed the `ReferenceError: context is not defined` time bomb in `updateWorkOrderStatus` and plumbed `cost_type` end-to-end so the schema added in FN-1469 is actually populated.

- `updateWorkOrderStatus` signature now `(workOrderId, nextStatus, userRole, userId, context = null)` — `context` was previously a free identifier that would throw the moment a tenant filter actually fired. Route at `work-orders-hub.js:1091` now passes `req.user?.id, req.context || null`, matching the convention sibling routes in this file already use.
- `createWorkOrder` writes `cost_type` (defaulting to `'BILLABLE'`); `updateWorkOrder` only writes it when the caller supplies it — never forces a default on edit.
- Both `cost_type` and `costType` casings are accepted from request bodies to match the field-marshalling style used elsewhere in this hub.

A schema-vs-code audit was run across every WO service method. PASS for the entire failing payload from FN-1468 (`POST /api/work-orders` with `vehicleId`, `customerId`, `locationId`, `type`, `priority`, `description`). Two unrelated, pre-existing column drifts were spotted that do **not** block FN-1468's reproduction — see *Open Items*.

## Key Decisions
- **`cost_type` default `'BILLABLE'`** — matches the ticket and the consumer in `work-orders-hub.js` lowercases before comparing to `'internal'`, so any non-internal value (including `BILLABLE`) is treated as billable. No retroactive backfill needed.
- **FK semantics** —
  - `tenant_id → tenants(id) ON DELETE RESTRICT` (matches the `20260310101000` pattern)
  - `operating_entity_id → operating_entities(id) ON DELETE RESTRICT` (same pattern)
  - `requested_by_user_id → users(id) ON DELETE SET NULL` (preserve WO history if a requesting user is deleted)
- **Asymmetric down()** — `up()` is hasColumn-guarded so it only adds what's missing; `down()` mirrors that by only dropping columns this migration introduced. Rolling back never touches columns owned by `20260310101000`.
- **Indexes** — `idx_work_orders_tenant`, `idx_invoices_tenant`, `idx_invoices_op_entity` created via `CREATE INDEX IF NOT EXISTS` (kept distinct from the `*_id`-suffixed indexes the `20260310101000` migration creates, so neither migration's `down()` clobbers the other's indexes).
- **`updateWorkOrderStatus` `userId` parameter accepted but unused** — there is no audit column on `work_orders` for "status changed by". The parameter is plumbed in for forward compatibility (and to mirror sibling service methods that take `userId`), but no synthetic audit behavior was invented. Adding such a column is a separate ticket.
- **`cost_type` only written on update when supplied** — `updateWorkOrder` uses spread-conditional inclusion (`...(costTypePatch !== undefined ? { cost_type } : {})`) so a PATCH that omits the field doesn't reset it to the default. Mirrors how other "preserve existing on omit" fields are handled in the same function.
