## Key Decisions
1. **Use sibling DB tables, not extend FN-1124's `daily_briefings`.** Snapshot is global (per-day, shared across tenants); per-tenant briefing is per-tenant-per-local-date. Forcing both into one table breaks either normalization or the prompt-cache strategy.
2. **Snapshot canonical timezone = US Eastern (single snapshot/day).** Avoids snapshot fan-out across tenant timezones; AI cache stays effective. HI/AK tenants see "yesterday's snapshot" until ~04:00 local — accepted tradeoff (revisit if HI/AK tenants exist and complain).
3. **Linear weighted-sum scoring, not ML.** Auditability + cold-start + tunable per tenant. Revisit ML in Phase-3 once 6+ months of `score → tenant-converted-load` data exists.
4. **Sonnet 4.6 for synthesis, Haiku 4.5 for GDELT relevance filtering.** Haiku is the right cost/quality fit for high-volume, low-stakes article classification; Sonnet is needed for the dispatcher-facing narrative.
5. **GDELT 2.0 over NewsAPI for production.** NewsAPI free tier is dev-only; GDELT is unlimited and designed for redistribution.
6. **Honor FN-1610 `localDate` contract on the new endpoint.** `tenant_market_briefings` is keyed on `(tenant_id, local_date)`; `?refresh=true` invalidates the per-tenant row only (not the snapshot).
7. **Don't replace FN-1124 — sibling card.** Today's-Operations + Market-&-Positioning answer different questions; both belong on Control Center.
