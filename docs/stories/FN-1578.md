# FN-1578: [Work Orders Hub] Date From/To filter excludes work orders created on the To date (timezone bug)

## Requirement
On `/work-orders`, the Date From / Date To range filter is exclusive of the user's selected end date in non-UTC timezones. A WO created on 2026-05-08 does not appear when Date To = `05/08/2026`, but does when Date To = `05/09/2026`. The filter must be inclusive of the To date in the user's local calendar — picking 05/08 must include all WOs created on 05/08 (any time).

## Acceptance Criteria
- [ ] User in any timezone picks Date To = today → a WO created today (any time) is included.
- [ ] User picks Date From = Date To = today → list contains exactly today's WOs (no boundary slop).
- [ ] User picks Date From = 7 days ago, Date To = today → all WOs in that 8-day local window appear, none missing on the boundaries.
- [ ] No regression in the unfiltered view.
- [ ] Test pinning `process.env.TZ` to a non-UTC zone (e.g. `America/Chicago`) confirms the boundary is correct.

## Implementation Summary
Replaced the buggy `new Date('YYYY-MM-DD')` + `setHours(...)` pattern in `applyClientFilters()` with two pure helpers — `parseLocalDateStart` (00:00:00.000 local) and `parseLocalDateEnd` (23:59:59.999 local) — that build boundaries directly from the year/month/day fields using the local-component `Date` constructor.

Root cause: `new Date('2026-05-08')` follows the date-only ISO production and is parsed as UTC midnight. The follow-up `setHours(23,59,59,999)` then writes hour fields in the runtime's local time zone, which leaves the underlying instant on the wrong calendar day in any non-UTC zone (silently shifting the boundary by the local UTC offset).

Fix: `new Date(y, m-1, d, ...)` is a local-time constructor — the resulting `Date` represents the picked calendar day in the user's TZ regardless of where the runtime sits. Both `from` and `to` are now constructed the same way, eliminating the asymmetry that made only `to` exclude same-day WOs.

The helpers are exported so they can be unit-tested without instantiating the whole component.

## Key Decisions
- **Strict YYYY-MM-DD regex** rather than free-form parsing. The two callers (the `<input type="date">` two-way binding and the `quickLast7Days()` shortcut at lines 489–491) both produce that format; rejecting anything else avoids silently re-introducing UTC-instant parsing if a future caller passes an ISO string.
- **Helpers are module-scoped exports**, not component methods. Pure functions make the test surface trivial and keep the regression evidence localized to one spec file.
- **No TZ pinning in the unit tests.** `process.env.TZ` only affects Node, and Karma/Angular component specs run in a real browser. The test runner config in `frontend/angular.json` has no `test` target, so the existing `.spec.ts` files in this repo are not executed by CI today; the new spec follows the same convention. The assertions verify the function contract directly (output components, not absolute UTC instants), which is TZ-agnostic — the same assertions hold in any zone the runner happens to be in.
