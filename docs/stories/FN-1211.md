# FN-1211 — Roadside v2 Story 9.1: nightly rollup tables + cron

## Implementation

### FN-1279 (subtask — backend)

**Branch**: `backend/FN-1279/nightly-rollup`

**Files changed** (4):
- `backend/microservices/reporting-service/services/rollup.service.js` (new)
- `backend/microservices/reporting-service/cron/rollup.cron.js` (new)
- `backend/microservices/reporting-service/telemetry/rollup.telemetry.js` (new)
- `backend/microservices/reporting-service/__tests__/rollup.service.spec.js` (new)

**Design**:
- `buildRollupService({ knex })` factory — `rollupTenant(tenantId, day)` + `runForDay(day)`.
- `runForDay` fetches all non-expired tenants and runs per-tenant rollup sequentially to keep DB pressure bounded.
- Per-table errors isolated: one failing upsert does not abort the other two tables for the same tenant.
- `rollup.cron.js` is a standalone script (not embedded in the HTTP server); Render Cron Job calls `node cron/rollup.cron.js`. Override day via `ROLLUP_DATE=YYYY-MM-DD` env var for backfill.
- Non-zero exit only when ALL tenants fail (vs. partial failures which are logged but non-fatal).
- Source tables: `roadside_calls` (incident metrics), `roadside_dispatch_assignments` (vendor SLA), `roadside_payments` (payment metrics).
- All source queries bind `tenant_id` first, `day` second — consistent with project conventions.

**Tests**: 9 cases covering happy path, tenant binding, ON CONFLICT clause, null aggregates, per-table error isolation, empty source, multi-tenant iteration, no-tenant case, missing-knex guard.

### FN-1280 (subtask — database)

**Branch**: `database/FN-1280/rollup-tables`

**Files changed** (1):
- `backend/packages/goodmen-database/migrations/20260610000000_create_rollup_tables.js` (new)

**Design**:
- Three tables created idempotently (`hasTable` guard + `IF NOT EXISTS` indexes).
- Each table keyed by `UNIQUE (tenant_id, day)` — matches the `ON CONFLICT (tenant_id, day) DO UPDATE` clause in `rollup.service.js`.
- Columns derived directly from upsert calls in `rollup.service.js` to guarantee schema/code alignment.
- `float` used for nullable aggregates (`avg_*`) and `total_amount`; `integer` for counts; `timestamptz computed_at` for freshness tracking.
- `(tenant_id, day DESC)` index on each table for efficient per-tenant time-series dashboard queries.
- `down` migration drops all three tables in reverse dependency order.
