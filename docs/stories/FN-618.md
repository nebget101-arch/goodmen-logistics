## Implementation Summary

### FN-627 (Gateway trace-context propagation + docker-compose OTel env vars)
**Gateway trace propagation:**
- Updated `buildProxy()` in `backend/gateway/index.js` to explicitly forward `traceparent` and `tracestate` W3C headers on every proxied request (both prod and dev)
- The `onProxyReq` callback now always runs (previously was `undefined` in prod) — reads trace context from incoming request and sets it on the outgoing proxy request
- This ensures distributed traces link: browser → gateway → downstream service → DB

**Docker-compose OTel env vars:**
- Added `OTEL_EXPORTER_OTLP_ENDPOINT=http://signoz-otel-collector:4318` and `OTEL_SERVICE_NAME=fleetneuron-<service>` to all 9 services
- Uses Docker service name `signoz-otel-collector` for internal networking (will resolve when SigNoz services from FN-648 are added to docker-compose)

## Files Changed
- `backend/gateway/index.js` (modified — trace context propagation in buildProxy)
- `docker-compose.yml` (modified — added 18 env vars across 9 services)

## Key Decisions
- **Always forward trace headers** — moved `onProxyReq` from dev-only to always-on; dev logging is still conditional via `if (!isProd)`
- **Docker internal URL** — uses `http://signoz-otel-collector:4318` which will resolve within docker-compose network once FN-648 SigNoz services are merged
- **No SIGNOZ_ACCESS_TOKEN** — self-hosted SigNoz doesn't need it

## Deployment Handoff
**Gateway service needs redeployment** for trace propagation change.
- `fleetneuron-logistics-gateway` — redeploy for `onProxyReq` trace header forwarding
- Docker-compose changes only affect local dev — no Render impact from env var additions in docker-compose.yml
- Render env vars (`OTEL_EXPORTER_OTLP_ENDPOINT`) are set separately in the Render dashboard per FN-636

**Post-deploy:** Verify gateway logs show `[tracing] OpenTelemetry initialized for fleetneuron-gateway` and that traces from gateway requests appear in SigNoz linked to downstream service traces.
