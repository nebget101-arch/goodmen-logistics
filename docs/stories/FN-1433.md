# FN-1433: T2 — VIN Repair History Lookup

## Requirement

When a tech opens a work order, surface a 1–2 line AI summary of the vehicle's repair history with comeback-risk grading (low/medium/high) and recurring-issue patterns linked to the underlying WOs.

## Acceptance Criteria

- [ ] AI service: `POST /api/ai/vehicles/repair-history-summary` with prompt-cached system prompt; short-circuit for <2 prior WOs.
- [ ] Backend route: `GET /api/vehicles/:id/repair-history-summary?windowDays=365` — tenant-scoped, default window 365d, capped at 50 WO rows sent to AI.
- [ ] Frontend: WO header widget with comeback-risk badge + expandable panel listing recurring patterns and linked WOs.
- [ ] In-memory cache (1h TTL) keyed by VIN.
- [ ] Cypress E2E: high-comeback case + insufficient-history case.

## Implementation Summary

### FN-1445 (subtask)

New `POST /api/ai/vehicles/repair-history-summary` endpoint in `fleetneuron-ai-service`. Given a VIN and a list of recent work-order rows, the handler returns:

- a 1–2 sentence plain-English summary,
- an array of recurring complaint patterns (each linked back to the underlying WO IDs from the input),
- a comeback-risk grade in `{low, medium, high}`.

Implementation highlights:

- **Prompt caching**: the static system prompt is sent as a `system: [{ type:'text', text, cache_control:{ type:'ephemeral' } }]` block so subsequent calls within the cache window pay only for the dynamic user message. Tokens are surfaced in `meta.cacheCreationInputTokens` and `meta.cacheReadInputTokens`.
- **Short-circuit**: when fewer than 2 prior WOs are supplied, the handler returns `{ summary: 'Not enough history', recurringIssues: [], comebackRisk: 'low' }` without making an LLM call.
- **In-memory cache**: results are cached per `(VIN, sorted set of WO IDs, count)` for ~1h with a 500-entry FIFO cap, so repeat opens of the same WO header do not re-spend tokens. Backend (FN-1446) can flush by changing the input set.
- **Defensive validation**: `validateResult` drops any `workOrderIds` not in the input list (model cannot hallucinate IDs into the response), enforces `comebackRisk ∈ RISK_LEVELS`, clamps the summary to 280 chars, and requires `occurrences >= 2`.
- **Rows cap**: `history` is truncated to 50 rows (`MAX_WO_ROWS`) to keep the user-message payload small.
- **Model**: `process.env.ANTHROPIC_REPAIR_HISTORY_MODEL` with default `claude-haiku-4-5-20251001` (fast and cheap; the static system prompt does most of the heavy lifting).
- **Logging**: every call goes through `logAiInteraction` with route `/vehicles/repair-history-summary`, success/error code, and `processingTimeMs`.

### FN-1447 (subtask) — Frontend repair-history widget

Adds a compact repair-history strip directly under the WO status bar plus an expandable patterns panel. The strip shows prior-WO count, last visit date, and a comeback-risk badge (low / medium / high). When `insufficientHistory: true` (fewer than 2 prior WOs), the risk badge is suppressed and a neutral "Not enough history" pill is rendered while the count remains visible. Expanding the panel reveals AI-detected recurring patterns with occurrence counts and a list of router-linked WO references that deep-link into `/work-order/:id`.

The widget is owned by `WorkOrderModule` and is fed `[vehicleId]="workOrder?.vehicleId"`. Loading is driven by `ngOnChanges` so the widget refetches when the user picks a different vehicle on the Basics tab. Per project memory (FN-317 RCA), no getter-based `[options]` bindings were used; bindings are plain field references.

A new `VehicleService` (`providedIn: 'root'`) hosts the typed `getRepairHistorySummary(vehicleId, windowDays)` call against the FN-1446 endpoint contract.

## Key Decisions

### FN-1445 (subtask)
- **Default to Haiku 4.5**, not Sonnet — the schema is small and the recurring-pattern detection is light reasoning. Operators can override via `ANTHROPIC_REPAIR_HISTORY_MODEL` if quality regresses.
- **Cache key is the set of WO IDs, not just the VIN** — adding new WOs invalidates automatically, while reopening the same WO list hits the cache.
- **No backend-call shape contract added here** — that is owned by FN-1446 (vehicles route). This subtask ships only the AI primitive; FN-1446 will define how the backend caps WOs to 50 and forwards `windowDays`.

### FN-1447 (subtask)
- **Single component, not two.** The compact strip and expandable panel live in one component to keep the `vehicleId → fetch → render` lifecycle in one place; the panel is just a conditional `*ngIf="expanded && canExpand"` block.
- **Service decoupled from AI shape.** `RepairHistorySummary` mirrors the contract FN-1446 will return after calling AI handler FN-1445. We treat `comebackRisk` as nullable so the widget gracefully degrades when the backend short-circuits the AI call (e.g. <2 WOs).
- **Expansion gated on patterns.** `canExpand` requires `patterns.length > 0`, so the WO header doesn't show a useless caret in the very-first-visit case.
- **No getter-bound `[options]`.** Followed FN-317 RCA — bindings are plain inputs / fields only; no app-ai-select used.

A second call with `history.length >= 2` and the prompt cache warm should report `meta.cacheReadInputTokens > 0`.

- **Feature flags:** None.

- **Post-deploy verification:** Open `/work-order/:id` for a vehicle with ≥3 WOs, confirm strip renders count + last-visit date + risk badge, confirm caret expands and pattern → WO links navigate.
