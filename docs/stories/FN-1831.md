## Key root causes (from analysis)
- Previous-street blank: `employment-application.service.js:76`/`:223` map `street: r.street || null` only (employers at L16 accept both casings).
- Work-auth color inverted: `pdf.service.js:204` colors `answer === 'YES'` red unconditionally â€” needs per-question adverse-answer.
- Signature audit: render path exists (`pdf.service.js` ~L684-714) but `context.auditTrail` is never populated; capture IP/UA server-side at submit into `applicant_snapshot.auditTrail`.

## Implementation Summary
All seven fixes implemented in the backend file set (FN-1834):

1. **Header company profile** â€” `submitApplication` OE resolution is now reliable: it uses a caller-supplied `context.operatingEntity` only when it actually has a `name`, otherwise resolves through the id chain **application â†’ request context (`context.operatingEntityId`) â†’ driver**. Name falls back `name || dba_name || legal_name`; address composes `address_line1, address_line2, city, state, zip` (already present from the FN-1787 MC-logo work, which is preserved). Missing fields are omitted gracefully. The placeholder text only appears when no profile exists at all.
2. **"PLEASE READ COMPLETELY" box** â€” taller (72â†’96pt) with explicit inner-padding (`noticePadX = 12`) and roomier line spacing (9â†’9.5pt); managed via a local `ny` cursor so content sits comfortably inside the box.
3. **Address History previous-street** â€” residency normalization in both `createDraft` and `updateDraft` now accepts `r.street || r.streetAddress || r.street_address`, so a form payload using camelCase `streetAddress` (the previous-address rows) no longer persists NULL.
4. **Employer Name width** â€” `drawFieldPair` gained an `opts.ratio1` so EMPLOYER NAME claims 64% of the row, and a new `fitText()` helper ellipsizes any value that would overflow its underline (applied across `drawFieldPair`/`drawSingleField` and the header text).
5. **Work Authorization color** â€” `drawQuestionRow` gained an `adverseAnswer` param; red now marks only the adverse selection: **NO** for "legally authorized to work", **YES** for felony/ADA and all drug/alcohol questions. A YES-authorized answer renders in the normal primary color.
6. **Signature date + audit trail** â€” the submit route captures `x-forwarded-for`/`req.ip`, `user-agent`, and a submit timestamp; `submitApplication` persists them into `applicant_snapshot.auditTrail` (JSONB, no migration) and sets `signed_certification_at` (if not already set). The existing PDF audit-trail block + signature-date render now populate.
7. **Alignment/layout polish** â€” notice-box spacing, header text fitting, and value truncation tidy the overall layout; no changes to the MC-logo branding path.

## Key Decisions
- **No migration**: audit trail lives in the existing `applicant_snapshot` JSONB; `signed_certification_at` column already exists (FN-215).
- **IP/UA captured server-side** in the route (client self-report is untrustworthy); `x-forwarded-for` first element preferred (Render runs behind a proxy), then `req.ip`.
- **Truncation over wrapping** for employer/header values â€” single-line ellipsis keeps the fixed-height row layout intact and avoids cascading y-shift regressions; the widened EMPLOYER NAME field handles the common case without truncation.
- **Preserved the FN-1787 MC-logo header path** â€” the OE object still carries `logo_storage_key`/`logo_mime_type`; only the resolution reliability and name/address composition changed.
