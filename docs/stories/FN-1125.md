# FN-1125: S2 — Ask FleetNeuron natural-language bar on Control Center

## Requirement
A persistent natural-language input bar at the top of `/dashboard` lets the user ask any fleet question. Answers route through the briefing context and a query-classification AI endpoint.

## Acceptance Criteria
- [ ] Input bar visible on Control Center, focusable via `/` keyboard shortcut
- [ ] Submit calls `POST /api/ai/ask` with prompt + briefing context
- [ ] Response renders inline below the bar with structured cards (table, chart, or text)
- [ ] Empty/error states with retry
- [ ] a11y: aria-live region announces results, ESC closes results

## Implementation Summary

### FN-1146 (subtask)
Built `<app-ask-bar>` standalone Angular component, wired into the existing `DashboardComponent` (the rendered `/dashboard` Control Center) above the daily briefing card:

- **Persistent input** with autofocus on view init; the global `/` keyboard shortcut focuses the input but only when the user isn't already typing in another `<input>`/`<textarea>`/`<select>`/contenteditable.
- **Submit** posts `{ prompt, briefingContext }` to `POST /api/ai/ask` via `AskService`. The TypeScript types match FN-1148's contract one-to-one (`success: true`, `intent`, `answer: { kind, headline, detail }`, `classification`, `meta`). The discriminated union `AskAnswer = AskAnswerText | AskAnswerTable | AskAnswerChart | AskAnswerMetric` is forward-compatible: today the backend only emits `text`, but adding `table`/`chart`/`metric` later requires no frontend re-architecture — the component already switches on `kind`.
- **Result rendering**: a single answer card with the headline as `<h3>`, plus kind-specific body — paragraph for `text`, big value + sub for `metric`, scrollable `<table>` for `table`, CSS-only horizontal-bar list for `chart` (no charting dependency added in v1).
- **Intent badge** (`Loads` / `Drivers` / `Vehicles` / `General`) renders next to the answer so dispatchers see how the classifier interpreted the prompt.
- **Error handling**: surfaces the gateway's structured error message (e.g. `prompt is required` from a 400) when the response body has an `error` string; falls back to friendly copy on transport failures (status 0) and 5xx. The `Try again` button re-submits the last prompt.
- **a11y**: `role="region"` with `aria-labelledby`, `aria-busy` toggling on load, `aria-live="polite"` status announcer reads the answer headline (or error). ESC closes the result pane and returns focus to the input. `prefers-reduced-motion` honored on shimmer + spinner. Input is capped at 1000 chars to mirror the gateway validation.
- Component is registered in `AppModule` imports (matches the FN-1124 pattern for `DailyBriefingComponent`).

**Note on file paths**: the ticket listed `frontend/src/app/components/control-center/control-center.component.html`, but the Control Center is rendered by the existing `DashboardComponent` at `/dashboard`; there is no separate control-center page component. The ask-bar was wired into `dashboard.component.html` to match the actual page composition (same place FN-1124 wired the daily briefing).

### FN-1148 (subtask)
Wired the gateway-local `POST /api/ai/ask` route plus the ai-service classifier and answer pipeline. The flow:

1. **Gateway** verifies the JWT, validates the `{ prompt, briefingContext }` envelope, and forwards `{ tenantId, prompt, briefingContext }` to ai-service `POST /api/ai/ask`.
2. **AI service classifier** (`src/services/query-classifier.js`) uses Claude Haiku to bucket the prompt into `loads | drivers | vehicles | generic`. A keyword heuristic acts as both a deterministic fallback (when Claude is unreachable / returns unparseable JSON) and the source for empty prompts.
3. **AI service answerer** (`src/handlers/ask-handler.js`) composes a domain-specific system prompt (one per intent), feeds the briefing context as JSON, and asks Claude Sonnet for a `{ kind:"text", headline, detail }` envelope. Validation enforces the shape and trims overlong fields.
4. The response carries the classified intent, the answer envelope, classification metadata (confidence, source), and timing meta — enough for FN-1146 to render structured cards.

Mocked-Anthropic test suites cover both layers:
- `backend/microservices/ai-service/src/handlers/__tests__/ask-handler.test.js` — 26 cases (keyword heuristic, classifier fallback paths, validation, happy path, parse/schema/upstream errors).
- `backend/gateway/__tests__/ask.test.js` — 7 cases (JWT enforcement, validation, happy-path forwarding, upstream error pass-through, transport failures, dep guards).

The mandatory `askForwarder` dep on `buildAiRouter` mirrors the existing `aggregator` dep (FN-1141) — briefing tests were updated to pass a no-op stub.

## Key Decisions

### FN-1146
- **Host page.** Wired into the existing `DashboardComponent` (Control Center) — there is no separate `control-center.component.*`, so the ticket's listed path was adjusted to the real host. Same pattern FN-1124 used for the briefing card.
- **Discriminated-union answer type.** The backend currently emits only `kind: 'text'`, but the frontend types and template already cover `text | table | chart | metric` so the next backend iteration can extend `ANSWER_KINDS` without a frontend rewrite. Each kind has its own `<ng-container *ngIf="asXxxAnswer(...)">` branch with type-narrowed access.
- **Chart fallback.** v1 renders charts as a CSS-only horizontal-bar list (label + proportional bar + value). Avoids pulling chart.js into the bundle for a feature that today returns only text answers.
- **Error message passthrough.** Render the gateway's `error` field verbatim when present (so `prompt is required` reaches the user). Only fall back to generic copy for network failures or unstructured 5xx.
- **OnPush + standalone.** Matches `DailyBriefingComponent` for consistency.
- **Briefing context wiring deferred.** The component sends `briefingContext: null` for now. Pulling the latest briefing payload from `BriefingService` and stuffing it into context is captured under Open Items — the backend already accepts an opaque object so this can land as a small follow-up.

### FN-1148
- **Classifier model split.** Used Claude Haiku for classification (low cost, low latency, JSON-only output) and Claude Sonnet for the answer (higher quality, can use briefing context). Both are environment-overridable via `ANTHROPIC_CLASSIFIER_MODEL` and `ANTHROPIC_ASK_MODEL`.
- **Keyword fallback.** If the classifier fails for any reason (parse / upstream / unknown intent), fall back to a deterministic keyword heuristic. The handler always produces an answer — no 5xx for classification failures alone.
- **Single answer kind for now.** The envelope is `{ kind, headline, detail }` and the only valid kind is `text`. Adding `table` / `chart` later means widening `ANSWER_KINDS` and the `validateAnswer` schema — frontend (FN-1146) will switch on `kind`.
- **Gateway-local route, not pure proxy.** Mirrors the FN-1141 briefing pattern: JWT verification + body validation happen at the gateway, and `tenantId` is sourced from the JWT (not the body). The Authorization header is forwarded to ai-service for downstream auth/audit.
- **Briefing context is opaque.** The handler accepts any object shape so the route doesn't break when FN-1124's briefing schema evolves. The classifier never sees the context; only the per-intent answer prompt does.

- `curl -X POST .../api/ai/ask` (no Authorization) → 401.
- `curl -X POST .../api/ai/ask -H 'Authorization: Bearer <token>' -d '{}'` → 400 with `prompt is required`.
