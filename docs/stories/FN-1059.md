# FN-1059: Loads Dashboard — user-resizable column widths with per-user persistence

## Type
Story (under Epic FN-723: Loads Page Redesign)

## Requirement
Let users manually drag the column dividers on the Loads Dashboard table to resize columns, and persist the chosen widths per user across sessions — same per-user prefs surface as visibility (FN-767), density (FN-821), and the drawer width pattern (FN-808).

## Background
- **Component**: `frontend/src/app/components/loads-dashboard/loads-dashboard.component.{html,ts,scss}`
- Two synchronized `<table>` elements that must stay aligned:
  - `table--header` sticky header — `loads-dashboard.component.html:725` (colgroup 726–741)
  - `table--body` cdk-virtual-scroll body — `loads-dashboard.component.html:868` (colgroup 869–884)
- `table-layout: fixed` already applied at `loads-dashboard.component.scss:486`.
- Column registry: `columnDefs` at `loads-dashboard.component.ts:415` — 14 columns; `load_number` and `actions` are `alwaysVisible` and not resizable.
- Existing skeleton-only `columnWidthMap` at `loads-dashboard.component.ts:92` supplies defaults.
- Persistence: extend `LoadsDashboardPrefs` at `frontend/src/app/services/user-preferences.service.ts:24` with optional `columnWidths?: Record<string, number>`; persist via `patchLoadsDashboard(...)`. Backend `backend/packages/goodmen-shared/routes/user-preferences.js` round-trips JSONB and does NOT need changes.

## Acceptance Criteria
- [ ] Each non-locked column header exposes a draggable resize handle on its right edge (`cursor: col-resize` on hover)
- [ ] Dragging updates width live in BOTH the sticky header table and the virtualized body table without visual desync
- [ ] Width clamped between 60px (min) and 600px (max)
- [ ] On `mouseup`/`touchend`, width is persisted via `patchLoadsDashboard({ columnWidths })` with ~400ms debounce; optimistic local update
- [ ] On reload, persisted widths are restored (single repaint on prefs load OK — match FN-767)
- [ ] "Reset column widths" action in the column-picker dropdown clears `columnWidths`
- [ ] Keyboard a11y: focused handle responds to `ArrowLeft`/`ArrowRight` (±8px), `Shift+Arrow` (±32px) with same clamp
- [ ] Hidden columns retain their stored width for when re-shown
- [ ] Works with all density modes (compact / comfortable / spacious) and with virtual scroll (FN-766)
- [ ] No regression in column sorting — handle `mousedown`/`click` must `stopPropagation` so `<th>` sort does not fire

## Out of Scope
- Drag-to-reorder columns
- Column pinning/freezing
- Per-saved-view widths (widths are global per user)
- Backend schema changes
- Mobile-specific gestures beyond basic touch drag

## Implementation Summary

**FN-1060 (frontend slice)** — landed the user-resizable column behavior plus per-user persistence on the Loads Dashboard. A new attribute directive `appResizableColumn` injects a draggable + keyboard-accessible resize handle into each non-pinned `<th>`; both the sticky header `<colgroup>` and the cdk-virtual-scroll body `<colgroup>` now read px widths from a single component-owned `columnWidths` map, so they stay aligned during drag and on prefs restore. Resize commits are debounced 400 ms before being patched into `loadsDashboard.columnWidths` via `UserPreferencesService`, mirroring the FN-808 / FN-821 prefs pattern.

### FN-1060 (subtask)

- New directive: `resizable-column.directive.ts` — Renderer2 + DOM `mousedown` / `mousemove` / `mouseup` (and `touchstart` / `touchmove` / `touchend`) listeners, plus `keydown` for `ArrowLeft` / `ArrowRight` (±8 px) and `Shift+Arrow` (±32 px). Width is clamped to [60, 600] px on every emit. The handle is `tabindex=0`, `role="separator"`, `aria-orientation="vertical"`, with `aria-valuemin` / `aria-valuemax` / `aria-valuenow` kept in sync. `mousedown` and `click` on the handle `stopPropagation` so the host `<th>` sort handler does not fire (AC10).
- Wireup: header + body `<colgroup>` switched from `style="width: X%"` to `[style.width.px]="getColWidth(...)"`. The `<select>` column is pinned to `36px` (still not in `columnDefs`). `load_number` and `actions` are `alwaysVisible` per `columnDefs`, so they intentionally do **not** carry the resize handle.
- Persistence: `LoadsDashboardPrefs.columnWidths?: Record<string, number>` added to `user-preferences.service.ts`; `loadUserPreferences()` restores into the px defaults map; a `Subject<void>` wired to `debounceTime(400)` in `ngOnInit` flushes `patchLoadsDashboard({ columnWidths })` once the user stops dragging.
- Reset: a "Reset column widths" action below the existing column-visibility checklist in the Columns dropdown wipes both the in-memory map (back to `columnWidthDefaultsPx`) and the persisted prefs (writes `{ columnWidths: {} }`).
- Hidden columns retain their stored width — the visibility checklist mutates `visibleColumns`, never `columnWidths`, so re-showing a hidden column restores the user's last drag width (AC8).
- Tests: `resizable-column.directive.spec.ts` covers handle injection + ARIA, drag emits, min/max clamp on drag, `stopPropagation` on mousedown + click (AC10), keyboard ±8 / ±32 stepping, keyboard clamp, aria-valuenow refresh on input change, live emission cadence during drag, and listener teardown on destroy.
- AOT: `resizable-column.directive.spec.ts` is excluded from `tsconfig.app.json` (matches the existing project pattern for spec fixtures with template HostComponents — see `hover-popover.component.spec.ts`).

## Key Decisions

- **Px, not %, in the live colgroups**: the existing `<col style="width: 6%">` setup couldn't represent a user override in any unit other than %. Switching the live tables to `[style.width.px]` keeps `table-layout: fixed` happy and lets the directive emit a single canonical unit. The skeleton's `columnWidthMap` (still %) is unchanged — it only paints during the first list fetch and doesn't need to mirror user widths.
- **Single px-defaults map (`columnWidthDefaultsPx`) lives next to the legacy `columnWidthMap`**: I kept both rather than refactor every consumer of the % map (skeleton component) for a behavior change that is already shipping.
- **Debounce on the component, not the directive**: the directive emits live (every move) so both colgroups can repaint synchronously without desync. The 400 ms `Subject` in the component coalesces those into one `patchLoadsDashboard` PUT, matching the AC4 wording.
- **No third-party drag lib**: `@angular/cdk/drag-drop` is overkill for a 1-axis resize and would require a wrapping `cdkDrag` host. Renderer2 + DOM events keeps the directive standalone and easy to unit-test.
- **Spec excluded from AOT**: the existing project pattern (see `hover-popover.component.spec.ts`) is to exclude spec files that contain template HostComponents from `tsconfig.app.json` because the workspace has no Karma/Jasmine runner configured — spec files document intent and remain compilable under a future `tsconfig.spec.json`.

## Deployment Handoff

- **Services to redeploy**: `goodmen-frontend` only. (Backend `user-preferences` route stores `loadsDashboard` as a JSONB blob and round-trips arbitrary keys — no backend change required for the new `columnWidths` field.)
- **Jobs/workers**: none.
- **DB migrations**: none. New widths persist inside `users.preferences` JSONB under `loadsDashboard.columnWidths`.
- **Env / config changes**: none.
- **Cache invalidation**: none. The first user load after deploy re-reads prefs and merges the new `columnWidths` field if present.
- **Feature flags**: none.
- **Deploy order**: single-service — `goodmen-frontend`.
- **Post-deploy verification**:
  1. Open `/loads`, drag the right edge of a non-pinned column header (e.g., "Broker") → header + virtualized body shift in lockstep without desync.
  2. Refresh the page → the new width is restored before first paint.
  3. Open Columns dropdown → "Reset column widths" reverts every column to its default px and the next reload returns to defaults.
  4. Tab onto a resize handle → `ArrowLeft` / `ArrowRight` step the column ±8 px; `Shift+Arrow` ±32 px; widths clamp at 60 px / 600 px.
  5. Click a sortable header (e.g., "Rate") → sort still toggles. Click the resize handle → sort does **not** toggle (AC10).
