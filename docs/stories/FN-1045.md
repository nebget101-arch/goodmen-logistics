# FN-1045: Trip Metrics on Stops tab show "--" / are not calculating in the drawer

## Type
Bug — Severity: Medium

## Symptom
Trip Metrics panel on the Stops tab shows "--" for Total Miles, Empty Miles, Loaded Miles, and Rate / Mile, even on loads where backend has computed values.

## Confirmed Root Cause (pure frontend wiring)
- Backend `getLoadDetail()` (`backend/packages/goodmen-shared/routes/loads.js:683`) DOES call `computeTripMetrics()` and returns `{ empty_miles, loaded_miles, total_miles, rate_per_mile }`.
- `load-detail-drawer.component.html:127–132` rendered `<app-step-stops>` without passing `[totalMiles]`, `[emptyMiles]`, `[loadedMiles]`.
- `populateFromDetail()` (drawer.ts:151–180) never read the three metrics off LoadDetail.
- `step-stops.recalcMetrics()` (line 190) saw null inputs → rendered "--" everywhere.

- [x] Editing Rate on Basics and switching back to Stops updates Rate / Mile

## Implementation Summary
The backend `getLoadDetail()` already returns `total_miles`, `empty_miles`,
`loaded_miles`, and `rate_per_mile`, and `<app-step-stops>` already declares
matching `@Input()`s — but the drawer was never reading them off `LoadDetail`
or forwarding them to the step component. As a result the Trip Metrics panel
ran on null inputs and showed "--" for every value.

`LoadDetail` already exposes typed optional fields for the four metrics, so no
model change was needed (FN-1054's "out of scope" note about model edits was
already satisfied).

### FN-1054 (subtask)
- Added `totalMiles`, `emptyMiles`, `loadedMiles` fields on
  `LoadDetailDrawerComponent` (default `null`).
- `populateFromDetail()` now reads `d.total_miles / d.empty_miles /
  d.loaded_miles` through a new `toFiniteNumber()` helper that coerces
  Postgres NUMERIC strings to numbers and returns `null` for missing /
  invalid / NaN values. The empty-detail branch (`!d`) also resets all three
  back to `null`.
- The drawer template now passes `[totalMiles]`, `[emptyMiles]`,
  `[loadedMiles]`, and `[rate]` to `<app-step-stops>` **before** `[stops]`.
  This ordering is load-bearing: `StepStopsComponent`'s `stops` setter calls
  `recalcMetrics()`, which reads the metric inputs — Angular applies input
  bindings in template attribute order, so the metric inputs must be set
  first.
- New unit spec covers numeric pass-through, string-numeric coercion,
  null/undefined/NaN safety, empty-string handling, and the null-detail reset.

## Key Decisions
- **Server values are the source of truth.** The frontend never recomputes
  metrics — `computeTripMetrics()` runs on the backend with full ZIP/coord
  data and the drawer just forwards what the API returns.
- **No `LoadDetail` model edits needed.** The interface in
  `frontend/src/app/models/load-dashboard.model.ts` already declares
  `total_miles`, `empty_miles`, `loaded_miles`, and `rate_per_mile` as
  optional `number | null` fields (lines 129–133). Reading them is type-safe
  without an `as any` cast.
- **Order template inputs deliberately.** The metric inputs are listed before
  `[stops]` because `StepStopsComponent.stops`'s setter calls
  `recalcMetrics()` which reads `this.totalMiles / emptyMiles / loadedMiles`.
  Angular applies @Input bindings in template attribute order, so binding
  metrics first guarantees the setter sees real values on the very first
  change-detection pass.
- **`toFiniteNumber()` over inline `Number(x)`.** PG NUMERIC columns
  deserialize as strings, and a load with no metrics sends nulls. Centralizing
  the coercion (with `Number.isFinite` + empty-string guard) keeps `NaN` out
  of the panel and makes the behavior easy to unit-test.
- **No change to `step-stops.recalcMetrics()`.** The existing logic already
  derives `loadedMiles` from `total - empty` if the server sends only those
  two, and computes `ratePerMile` from rate + total. Leaving it intact keeps
  the wizard (which uses the same component) unchanged.
