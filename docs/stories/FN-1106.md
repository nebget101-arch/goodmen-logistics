# FN-1106: Barcode/QR scanner integration into Add Part

## Requirement
Wire the existing scanner from `/barcode-management` into the parts catalog as a "Quick Add ‚Üí Scan Barcode" option. Match ‚Üí Edit modal; no match ‚Üí Add modal with barcode prefilled.

- [ ] Lookup uses existing `barcodes.js` route (confirm shape during impl).
- [ ] Matched ‚Üí Edit Part modal.
- [ ] Unmatched ‚Üí Add Part modal with barcode prefilled (read-only).
- [ ] Malformed ‚Üí in-scanner toast; scanner stays open.
- [ ] QA: matched, unmatched, malformed.

## Implementation Summary

### FN-1107 (subtask)
Added a third "Scan Barcode" entry to the parts-catalog Quick Add menu and wired it through a new shared `BarcodeScannerDialogComponent`. The dialog reuses the existing decode/bridge backend services (`/api/barcodes/decode-image`, `/api/scan-bridge/session`) and emits a captured barcode value to its host. The host (parts-catalog) calls `lookupBarcode(code)` and routes:

- **Matched (200 with `data.part.id`)** ‚Äî close dialog, resolve the full part record from the local `this.parts` cache and open the existing Edit Part modal pre-populated.
- **Unmatched (404 or 200 with no part)** ‚Äî close dialog, open the Add Part modal with the barcode field prefilled and locked (`barcodePrefilled = true` ‚Üí `[readOnly]`).
- **Malformed/server error** ‚Äî keep the dialog open and surface the error inline (`scannerError`) so the user can retry. The dialog itself handles "no code detected in image" with a local toast (no `scanned` event fires).

The lookup endpoint returns only `{id, sku, name, category, unit_price, unit_cost, default_retail_price, default_cost, taxable}` ‚Äî not enough for the Edit form (no manufacturer/vendor/uom/qty). Rather than introducing a new endpoint (per scope guard in the subtask Notes), we resolve the full record from the local catalog cache that `loadParts()` already populates on init. If the part is missing from the cache we fall back to the lookup partial; the user can fill the rest and Save.

The existing `/barcode-management` page is unchanged ‚Äî we built a fresh dialog rather than refactor that page's intermixed chrome, but the dialog uses the same `decodeBarcodeFromImage` and `createScanBridgeSession` API methods (no second barcode library).

## Key Decisions

### FN-1107 (subtask)
- **No backend change.** The barcode lookup endpoint is partial; the parts-catalog already loads the full catalog up front, so we resolve the full record by id from `this.parts`. Avoids the scope-creep "do not silently introduce a new lookup endpoint" guard.
- **Did not refactor `/barcode-management`.** Its scanner UI is intermixed with page-specific chrome (part selection + assign-barcode form). Building a separate dialog that reuses the same backend services is lower-risk and the AC explicitly allows this ("refactor to extract a reusable child component **if** the existing component embeds page-specific chrome").
- **Dialog never self-closes.** It only closes when the host clears `scannerOpen` (or the user cancels) ‚Äî keeps the malformed/lookup-error path simple (just re-render with `externalError` set).
- **Manual-entry kept** in the dialog as a third path so a desktop user without a camera can still test the flow.
