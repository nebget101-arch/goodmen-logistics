## Implementation Summary

### FN-808 (subtask)

Implemented the Phase 2 Load Detail Side Drawer with full row-click wiring
and prev/next navigation. Consolidated the originally-separate FN-809
(row wiring) work into this subtask per the "one subtask per agent per
story" rule.

- New `LoadDetailDrawerComponent` — slides in from the right with a
  `cubic-bezier(0.4, 0, 0.2, 1)` animation, 720 px default width,
  drag-resizable between 600–800 px via a left-edge drag handle.
- Header: prev/next nav arrows, load number, status pill, expand-to-modal,
  close button. Footer: Cancel / Save (primary).
- Tabbed body (Basics / Stops / Driver / Attachments) reuses the existing
  Load Wizard step components (`<app-step-basics>`, `<app-step-stops>`,
  `<app-wizard-step-driver>`, `<app-step-attachments>`) so field-level
  behavior stays consistent with the FN-725 wizard.
- Save persists via `LoadsService.updateLoad` (PATCH). On success the
  drawer refreshes its local copy and emits `(savedLoad)` so the parent
  can refresh the list row.
- Unsaved-changes guard: prompts the user on Esc, close click, or
  prev/next navigation when the drawer is dirty.
- Drawer width is persisted per-user via `UserPreferencesService`
  (`loadsDashboard.drawerWidth`).

Row-level wiring in `LoadsDashboardComponent`:

- Table `<tr>` now has a `(click)` handler (`onRowClick`) that opens the
  drawer for the clicked load. Interactive child cells
  (checkbox/actions/status dropdown) already stop propagation, and the
  handler additionally bails out if the click target is an `<a>`,
  `<button>`, `<input>`, `<select>`, or `<textarea>` — so inline
  controls (status pill, bulk-select, actions menu, etc.) retain their
  existing behavior.
- Active row gets a teal left-border (inset box-shadow) highlight.
- When the drawer is open, `.loads-dashboard` receives inline
  `padding-right` equal to the drawer width, animated with the same
  cubic-bezier curve; the table reflows into the remaining viewport
  without overlap or layout jump.
- Prev/Next cycle through `filteredLoads` with wraparound, so the user
  can step through the current filter/sort result set without
  closing/reopening.
- Clicking a different row while the drawer is open updates the
  drawer content in-place (no close/reopen) via
  `LoadDetailDrawerComponent.ngOnChanges`.

## Key Decisions

- **Drawer positioning** — fixed to the viewport's right edge (not a grid
  cell) so it stays in place while the user scrolls the loads list. The
  dashboard container takes on right padding equal to the drawer width,
  which produces the "table reflows, no overlap" behavior the AC asks
  for while keeping the existing dashboard layout intact.
- **Step component reuse** — rather than duplicate field logic, the
  drawer tabs host the existing wizard step components directly. This
  keeps Basics/Stops/Driver/Attachments behavior identical between the
  wizard and the drawer, at the cost of slightly looser typing for
  shared wizard types.
- **Old modal kept as "Expand to full"** — clicking the load-number
  link still opens the full-screen wizard, and the drawer header has an
  "Expand to modal" icon that hands off cleanly. This gives power
  users the larger editing surface without blocking the new fast path.
- **Width persistence via user prefs, not localStorage** — reuses the
  existing `users.preferences` JSONB surface so the value syncs across
  devices for the same user.

## Deployment Handoff

- **Services to redeploy**: `goodmen-frontend` (Render) — frontend-only
  change to the Loads page.
- **Backend changes**: none. The drawer uses the existing
  `GET /api/loads/:id` and `PATCH /api/loads/:id` endpoints — no new
  routes, no new fields.
- **DB migrations**: none. `drawerWidth` is stored inside the existing
  `users.preferences` JSONB column and fits the existing
  `UserPreferencesService.patchLoadsDashboard` flow.
- **Environment/config**: none.
- **Cache invalidation**: none (Angular build bust handles component
  updates).
- **Feature flags**: none. The drawer is the default edit path for row
  clicks; existing load-number-link and actions-menu edit paths are
  untouched, so there is no staged-rollout switch.
- **Deploy order**: frontend-only — deploy in a single pass.
- **Post-deploy verification**:
  1. Open `/loads`, click any row — the drawer should slide in from the
     right and populate with the load's details within ~300 ms.
  2. Drag the left edge of the drawer left/right — width clamps to
     600–800 px and persists across reload for the same user.
  3. Click a different row — drawer content swaps in place; active-row
     teal left-border moves to the new row.
  4. Click prev/next in the drawer header — drawer advances through
     `filteredLoads` with wraparound.
  5. Edit a field, press Esc — unsaved-changes dialog appears; "Keep
     editing" dismisses; "Discard changes" closes the drawer.
  6. Edit a field, click Save — list refreshes and success toast
     appears.
