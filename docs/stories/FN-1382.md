# FN-1382: Vehicle Maintenance History tab — work orders + invoices on vehicle detail

## Requirement
Surface every shop work order and its generated invoice for a given vehicle directly on the vehicle detail drawer, joined via VIN through the `customer_vehicles` mirror table, so dispatch and accounting can see lifetime maintenance spend without leaving the Trucks/Trailers page.

## Acceptance Criteria
- [x] `GET /api/vehicles/:id/maintenance-history` returns paginated WO + invoice rows for fleet AND customer-owned vehicles, joined by VIN.
- [x] Vehicle detail drawer shows a Maintenance History tab next to Overview.
- [x] Tab lists WOs with totals and clickable invoice download links.
- [x] Header strip shows lifetime spend, last-WO date, WO count.
- [x] Invoice column hidden for users without `invoices.view`. _(backend strips the per-row `invoice` field; FE additionally hides the column based on the same permission)_
- [x] Empty state shown for vehicles with zero WOs.
- [x] No regressions in existing Overview content. _(existing details-grid wrapped in `*ngIf="activeDetailTab === 'overview'"`; no fields touched)_

## Implementation Summary
_To be filled by implementing agent_

### FN-1389 (subtask)
- New endpoint `GET /api/vehicles/:id/maintenance-history` mounted on the vehicles router.
- Resolution flow: tenant-scoped vehicle lookup against `all_vehicles` (or `vehicles`) → VIN → all `customer_vehicles.vehicle_uuid` rows for the same tenant → `work_orders` LEFT JOIN LATERAL latest `invoices` row per WO + `locations` for shop name.
- Response matches the shape documented on FN-1382 (top-level `data[]` + `meta { page, pageSize, total, lifetime_spend }`).
- `lifetime_spend` is `SUM(work_orders.total_amount)` excluding `CANCELED` WOs (canceled spend is not real cost).
- `pageSize` clamped at 100 (default 25); `page` defaults to 1.
- Permissions: `loadUserRbac` + `requirePermission('work_orders.view')` gate access. The per-row `invoice` field is omitted when the caller lacks `invoices.view` (super_admin bypasses both checks).
- Tenant scoping enforced in every step: vehicle lookup, VIN→vehicle_uuid join, WO aggregate, and WO rows query — including the LATERAL invoice subquery. Cross-tenant `:id` returns 404.

### FN-1390 (subtask)
- Vehicle detail drawer (`vehicles-details-modal`) now exposes two tabs: **Overview** (existing content, unchanged) and **Maintenance History**.
- Maintenance History tab is **lazy-loaded**: the API call only fires the first time the tab is opened for a given vehicle. Reopening the same drawer for a different vehicle resets the cached state.
- **Header strip** shows lifetime spend (currency, no decimals), last WO date (most recent `completion_date` or `request_date` from page 1), and WO count.
- **Table columns**: WO #, Date, Type, Title (truncated, ellipsis), Shop Location, Total (right-aligned currency), Invoice (download pill, conditional on `invoices.view`), Status badge.
- **Row interaction**: click or Enter/Space → opens `/work-order/:id` in a new tab via `window.open(..., '_blank', 'noopener')`.
- **Invoice pill**: opens `${environment.apiUrl}/invoices/:id/pdf` in a new tab. The pill swallows row-click via `event.stopPropagation()` so a click on the pill never navigates to the WO route.
- **States**: skeleton-style loading message, error state with Retry, and "No maintenance history yet." empty state.
- **Pagination**: defaults to `pageSize=25`; Prev/Next controls appear only when `total > pageSize`. Each page change triggers a fresh fetch.
- **RBAC**: invoice column header + cells are hidden client-side via `canViewInvoices()` (`PERMISSIONS.INVOICES_VIEW`). Backend additionally strips the `invoice` field when the caller lacks the permission, so the FE check is defense-in-depth, not the primary gate.
- New API service method `getVehicleMaintenanceHistory(id, page=1, pageSize=25)` added next to existing `getVehicle(id)`.

## Key Decisions
_To be filled by implementing agent_

### FN-1389 (subtask)
- **Inline RBAC, not router-level.** The vehicles router already runs a role-based `vehiclesRoleGate`; switching the whole router to `loadUserRbac` would change the access surface for every existing endpoint. The maintenance-history route mounts `loadUserRbac` + `requirePermission` as per-route middleware to keep the change scoped.
- **`includeInvoices` is a service-level flag, not a SQL gate.** The query always returns the joined invoice row (cheap LATERAL with index on `invoices.work_order_id`) and the service drops the field when the caller lacks `invoices.view`. This keeps a single SQL path and lets the test stub one shape.
- **`lifetime_spend` excludes CANCELED.** Including canceled WOs would inflate the aggregate; the meta value is intended as actual maintenance cost for accounting.
- **VIN is the join key, not `vehicle_id`.** Matches the schema established by `20260228182000_update_work_orders_vehicle_fk_to_customer_vehicles` — `work_orders.vehicle_id` references `customer_vehicles.vehicle_uuid`, and a fleet `vehicles.id` won't match directly. Resolving via VIN handles both fleet and customer-owned vehicles transparently.

### FN-1390 (subtask)
- **Lazy fetch, not eager.** Maintenance history is fetched the first time the tab is opened, not on drawer open. Most users open the drawer to read Overview / safety summary; an eager fetch would issue a redundant query for every drawer open. `maintenanceHistoryLoaded` gates the call.
- **Existing details-grid wrapped, not duplicated.** The Overview tab simply guards the existing `details-grid` with `*ngIf="activeDetailTab === 'overview'"`. No fields were renamed, removed, or reorganized — preserves the "no regressions" acceptance criterion without churn.
- **Invoice pill bypasses row-click.** The pill calls `event.stopPropagation()` before opening the PDF so users can download an invoice without simultaneously opening the WO route in another tab.
- **Status badges reuse existing `.badge` classes** with two new modifiers (`badge-info`, `badge-warning`, `badge-muted`). `badge-success` and `badge-danger` already existed. Statuses map: COMPLETED/CLOSED/INVOICED → success, CANCELED → muted, IN_PROGRESS/OPEN → info, everything else (PENDING/etc.) → warning.
- **Last-WO date is page-1-only.** The header strip's "Last WO" is derived from row 0 of page 1, since the backend already orders by `created_at DESC`. Showing it from later pages would be misleading. On page 2+, it reads `—`.
- **Use `window.open` for invoice download instead of `InvoiceService.getPdf`.** The existing `InvoiceService.getPdf(id)` returns an `Observable<any>` from `http.get`, which doesn't trigger a browser download. Per scope, opening the gateway URL in a new tab is the documented fallback ("else `/api/invoices/:id/pdf` in new tab").
- **No spec file added.** No existing `vehicles.component.spec.ts` exists in the repo; per the parent story, FN-1391 covers manual QA validation. Adding scaffolding only for this subtask would expand scope.
