# FN-1737: Logo storage model + upload/serve API (foundation)

## Requirement
Foundation for customer logo branding under epic FN-1736. Add the storage model (DB columns) and a backend API to upload, replace, delete, and fetch logos for an operating entity (MC) and for a shop location. This story unblocks the frontend UI (FN-1738) and both document-rendering stories (FN-1739, FN-1740). The API contract is defined here up front so the frontend can build in parallel.

## API Contract (canonical — frontend builds against this)
| Method | Path | Body | Response |
|--------|------|------|----------|
| POST | `/api/branding/operating-entities/:id/logo` | multipart `file` | `{ logoUrl, mimeType, uploadedAt }` (signed URL) |
| GET | `/api/branding/operating-entities/:id/logo` | — | `{ logoUrl, mimeType, uploadedAt }` or `{ logoUrl: null }` |
| DELETE | `/api/branding/operating-entities/:id/logo` | — | `{ ok: true }` |
| POST | `/api/branding/locations/:id/logo` | multipart `file` | `{ logoUrl, mimeType, uploadedAt }` |
| GET | `/api/branding/locations/:id/logo` | — | `{ logoUrl, mimeType, uploadedAt }` or `{ logoUrl: null }` |
| DELETE | `/api/branding/locations/:id/logo` | — | `{ ok: true }` |

- **Validation:** PNG/JPEG/WebP only, max 2MB, max dimensions ~1024×1024 → else 400 with message.
- **Tenant scoping:** all endpoints scoped to caller `tenant_id`; cross-tenant id → 403/404.
- **Signed URLs:** 15-min TTL, same mechanism as DQF docs.

## Acceptance Criteria

- [ ] GET endpoints return a signed R2 URL (15-min TTL) or null.

- [ ] All endpoints tenant-scoped; cross-tenant access denied.

## Implementation Summary

### FN-1741 (subtask) — DB logo columns
Knex migration adding the branding-logo storage columns to the three branding-bearing
tables. No data backfill; `down()` drops the columns. Each table receives:

- `logo_storage_key` — `text`, nullable, no default (R2 object key)
- `logo_mime_type` — `text`, nullable, no default (e.g. `image/png`)
- `logo_uploaded_at` — `timestamptz` (`{ useTz: true }`), nullable, no default

The migration iterates the three tables with `hasTable`/`hasColumn` guards, so both
`up()` and `down()` are idempotent and safe to re-run.

**Verification:** ran `up()`/`down()` against a throwaway local Postgres DB seeded
with stub `operating_entities`, `locations`, `tenants` tables. Confirmed: 3 columns
added per table, all `is_nullable=YES` with `column_default=null`; `up()` idempotent;
`down()` drops all columns and is idempotent. All checks passed.

### FN-1742 (subtask) — Logo upload/serve/delete API
New shared router `routes/branding.js` implementing all six endpoints behind a single
`/:resource/:id/logo` handler set, where `:resource` is `operating-entities` or
`locations`. Mirrors the multer `memoryStorage` + R2 pattern from `dqf-documents.js`
and reuses the existing `uploadBuffer`/`getSignedDownloadUrl`/`deleteObject` helpers in
`storage/r2-storage.js` (no new R2 helper needed — `deleteObject` already existed).

- **POST** validates type (PNG/JPEG/WebP via mime allow-list), size (≤2MB, multer
  `LIMIT_FILE_SIZE` → 400), and dimensions (≤1024×1024). Stores to R2 under a stable
  key `branding/<resource>/<id>/logo.<ext>`, deletes the prior object if the extension
  changed, then persists `logo_storage_key`/`logo_mime_type`/`logo_uploaded_at`. Returns
  `{ logoUrl (15-min signed), mimeType, uploadedAt }`.
- **GET** returns a 15-min signed URL + metadata, or `{ logoUrl: null }` when unset.
- **DELETE** removes the R2 object and clears the three columns (idempotent).
- **Tenant scoping:** every request loads the target row and rejects missing/cross-tenant
  ids with 404 (404 over 403 so existence isn't leaked); missing tenant context → 403.
- **Dimension validation** uses a bespoke, dependency-free header parser
  (`readImageDimensions`) covering PNG (IHDR), JPEG (SOF markers), and WebP
  (VP8/VP8L/VP8X) — `jimp` (the only image dep present) can't decode WebP.

Mounted in **logistics-service** (`/api/branding`, after `authMiddleware` +
`tenantContextMiddleware` + `requireActiveSubscription`, mirroring `/api/locations`)
and proxied through the gateway (`/api/branding` → `LOGISTICS_SERVICE_URL`). Chose
logistics-service because it already owns `/api/locations` and the tenant-context
middleware that supplies `req.context.tenantId`; `operating_entities` is queryable from
the shared DB regardless of owning service.

**Verification:** `node --test routes/branding.test.js` — 8/8 pass. Covers the header
parser (PNG/JPEG/WebP + corrupt→null) and the no-R2 route paths (unknown resource 404,
no-logo GET → `{ logoUrl: null }`, cross-tenant 404, idempotent DELETE). `node --check`
clean on `branding.js`, `logistics-service/server.js`, and `gateway/index.js`. Upload
success/R2 paths are validated by QA (FN-1743) against the deployed service.

## Key Decisions

- **`text` over `varchar`** for `logo_storage_key`/`logo_mime_type` — matches existing
  convention in `goodmen-database` migrations (e.g. DQF tables) and avoids arbitrary
  length caps on storage keys.
- **`timestamptz`** (`{ useTz: true }`) for `logo_uploaded_at`, consistent with all
  other timestamp columns in the package.
- **Single migration for all three tables** — the columns are identical and introduced
  for one feature; one reversible migration keeps the change atomic.
- **`tenants` included** as the tenant-level fallback logo for shop invoices, per scope.
