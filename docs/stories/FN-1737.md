# FN-1737: Logo storage model + upload/serve API (foundation)

## Requirement
Foundation for customer logo branding under epic FN-1736. Add the storage model (DB columns) and a backend API to upload, replace, delete, and fetch logos for an operating entity (MC) and for a shop location. This story unblocks the frontend UI (FN-1738) and both document-rendering stories (FN-1739, FN-1740). The API contract is defined here up front so the frontend can build in parallel.

## API Contract (canonical ÔÇö frontend builds against this)
| Method | Path | Body | Response |
|--------|------|------|----------|
| POST | `/api/branding/operating-entities/:id/logo` | multipart `file` | `{ logoUrl, mimeType, uploadedAt }` (signed URL) |
| GET | `/api/branding/operating-entities/:id/logo` | ÔÇö | `{ logoUrl, mimeType, uploadedAt }` or `{ logoUrl: null }` |
| DELETE | `/api/branding/operating-entities/:id/logo` | ÔÇö | `{ ok: true }` |
| POST | `/api/branding/locations/:id/logo` | multipart `file` | `{ logoUrl, mimeType, uploadedAt }` |
| GET | `/api/branding/locations/:id/logo` | ÔÇö | `{ logoUrl, mimeType, uploadedAt }` or `{ logoUrl: null }` |
| DELETE | `/api/branding/locations/:id/logo` | ÔÇö | `{ ok: true }` |

- **Validation:** PNG/JPEG/WebP only, max 2MB, max dimensions ~1024├ù1024 ÔåÆ else 400 with message.
- **Tenant scoping:** all endpoints scoped to caller `tenant_id`; cross-tenant id ÔåÆ 403/404.
- **Signed URLs:** 15-min TTL, same mechanism as DQF docs.

## Acceptance Criteria
- [ ] Migration adds `logo_storage_key`, `logo_mime_type`, `logo_uploaded_at` to `operating_entities`, `locations`, and `tenants`.
- [ ] Upload endpoints validate type/size/dimensions, store to R2 via `r2-storage.js`, persist key + mime + timestamp.
- [ ] GET endpoints return a signed R2 URL (15-min TTL) or null.
- [ ] DELETE endpoints remove the R2 object and clear the columns.
- [ ] All endpoints tenant-scoped; cross-tenant access denied.

## Implementation Summary

### FN-1741 (subtask) ÔÇö DB logo columns
Knex migration adding the branding-logo storage columns to the three branding-bearing
tables. No data backfill; `down()` drops the columns. Each table receives:

- `logo_storage_key` ÔÇö `text`, nullable, no default (R2 object key)
- `logo_mime_type` ÔÇö `text`, nullable, no default (e.g. `image/png`)
- `logo_uploaded_at` ÔÇö `timestamptz` (`{ useTz: true }`), nullable, no default

The migration iterates the three tables with `hasTable`/`hasColumn` guards, so both
`up()` and `down()` are idempotent and safe to re-run.

**Verification:** ran `up()`/`down()` against a throwaway local Postgres DB seeded
with stub `operating_entities`, `locations`, `tenants` tables. Confirmed: 3 columns
added per table, all `is_nullable=YES` with `column_default=null`; `up()` idempotent;
`down()` drops all columns and is idempotent. All checks passed.

## Key Decisions

- **`text` over `varchar`** for `logo_storage_key`/`logo_mime_type` ÔÇö matches existing
  convention in `goodmen-database` migrations (e.g. DQF tables) and avoids arbitrary
  length caps on storage keys.
- **`timestamptz`** (`{ useTz: true }`) for `logo_uploaded_at`, consistent with all
  other timestamp columns in the package.
- **Single migration for all three tables** ÔÇö the columns are identical and introduced
  for one feature; one reversible migration keeps the change atomic.
- **`tenants` included** as the tenant-level fallback logo for shop invoices, per scope.

## Deployment Handoff

- **DB migration to run:** `20260612120000_add_logo_columns_to_branding_tables.js`
  ÔÇö run `knex migrate:latest` (or the platform's migrate step) on the target DB.
  Pure additive DDL on `operating_entities`, `locations`, `tenants`; nullable columns,
  no backfill, no lock-heavy rewrite. Reversible via `migrate:rollback`.
- **Services to redeploy:** none required for the migration itself. The backend
  consuming these columns ships with FN-1742.
- **Env/config changes:** none.
- **Deploy order:** migration must land before FN-1742 backend code reads/writes the
  columns.
- **Post-deploy verification:** confirm the 3 columns exist on each of the three tables
  (`\d operating_entities` etc.) and are nullable.
