# FN-1521: [Work Order] Financials — auto-apply state-based tax rules from selected location

## Requirement

Replace the hardcoded 8.5% on parts with a state-aware tax engine. Per state DoR rules: tax rate, plus per-component flags (labor / parts / fees taxable). Example: Texas — parts taxable @ 6.25%, labor not. Use selected Shop Location's state to pick the rule. Manual override remains available via `tax_rate_percent`.

## Acceptance Criteria

- [ ] `state_tax_rules` table seeded from a researched 50-state CSV with citations
- [ ] `recomputeWorkOrderTotals()` consumes location → state → rules; per-component flags honored
- [ ] `tax_breakdown` JSONB column persists the breakdown; returned in GET response
- [ ] Frontend Financials tab shows computed tax + tooltip + override toggle
- [ ] Test cases: TX (parts only), CA (both), MT (none), FL (parts only) all match expected calculations
- [ ] Existing WOs recompute on next save (no auto-backfill in scope)

## Dependencies

- Subtask chain: ai (research CSV) → database (table + seed) → backend (engine) → frontend (UI) → qa
- Recommended sequencing with FN-1520 (parts selling price): FN-1520 ships first so the parts subtotal is correct before tax computes against it. Not a hard block, but a coordination note.

## Implementation Summary

### FN-1536 (subtask) — AI

Researched and produced the 50-state + DC sales-tax rules reference dataset at `docs/reference/state-sales-tax-rules-2026.csv`. The CSV is the seed source for the `state_tax_rules` table built by FN-1537 and consumed by the tax engine in FN-1538.

**Approach.** State-by-state across DoR publications, statutes, and regulations. Aggregators (Avalara, Tax Foundation) used only as triage; primary state sources resolved every classification. Five spot-check states verified hand-to-DoR (TX, CA, FL, NY, PA).

**Key finding.** The labor-taxability axis is the dominant variable. Jurisdictions cluster:
- **No-tax (5)**: AK, DE, MT, NH, OR — all flags `false`, rate `0`
- **Labor exempt when separately stated (~26)**: AL, AZ, CA, CO, GA, ID, IL, IN, MA, MD, ME, MI, MN, MO, NE, NV, ND, OH, OK, RI, SC, TX, VA, VT
- **Labor fully taxable (~16 + DC)**: AR, CT, DC, FL, HI, IA, KS, KY, LA, MS, NC, NJ, NM, NY, PA, SD, TN, WA, WI, WV, WY
- **Edge cases**: MA inconsequential rule, VA diagnostic-labor exemption (2023), KY 2018 statute change, FL/NJ "labor taxable when parts furnished"

**Sanity-check confirmation** (FN-1536 acceptance criteria):
- TX → labor `false`, parts `true`, rate `0.0625` (Tex. Tax Code §151.0048; Comptroller Pub. 94-113)
- AK / DE / MT / NH / OR → all flags `false`, rate `0`

**Methodology + caveats**: see `docs/reference/state-sales-tax-rules-2026.md` for full methodology, the `fees_taxable` discussion, spot-check evidence table, and limitations (local rates, sector exemptions, warranty work).

**CSV consumer contract** (for FN-1537 seed script):
- 51 rows + header. UTF-8. RFC 4180 quoting (notes contain commas → quoted).
- Booleans lowercase (`true`/`false`) — Postgres-friendly without coercion.
- `default_sales_tax_rate` is decimal (`0.0625`, not `6.25`).
- `effective_from` = `2026-01-01` for all rows.
- Recommended Postgres column types listed in the markdown.

### FN-1537 (subtask) — Database

Created the `state_tax_rules` reference table per the spec in FN-1537, plus an idempotent seed that loads it from `docs/reference/state-sales-tax-rules-2026.csv` (FN-1536 output).

**Migration** (`20260507230000_create_state_tax_rules.js`)
- Schema matches the ticket exactly: `state_code VARCHAR(2) PK`, `state_name VARCHAR(64)`, `default_sales_tax_rate NUMERIC(6,4)`, three boolean flags (labor / parts / fees taxable), `notes`, `source_url`, `effective_from / effective_to`, timezone-aware `created_at / updated_at`.
- Two CHECK constraints — `default_sales_tax_rate >= 0`, and effective range well-formedness (`effective_to IS NULL OR effective_to >= effective_from`) — to keep the table self-validating against bad future seed updates.
- Down migration drops the table cleanly.

**Seed** (`07_state_tax_rules_seed.js`)
- Reads the CSV directly so the in-repo human source of truth and the DB never drift. RFC 4180 inline parser handles the quoted `notes` fields (which contain commas inside citations).
- Header validation: errors loudly if the CSV column order changes — preferable to silent data corruption.
- Single-statement upsert: `INSERT ... ON CONFLICT (state_code) DO UPDATE` over 9 columns + `updated_at`. Re-running the seed is idempotent; row count stays at 51, only `updated_at` advances.
- Final row-count assertion (must be ≥ 51) catches a half-written CSV.

**Verification (against fresh Postgres DB `fn_1537_test`):**
- `migrate:up` → table created with the documented schema (verified via `\d state_tax_rules`); both CHECK constraints present.
- `seed:run` → 51 rows. Spot-checks against the methodology doc:
  - TX `0.0625` / labor=f / parts=t / fees=t ✓
  - CA `0.0725` / labor=f / parts=t / fees=f ✓
  - FL / NY / PA `0.06–0.07` / all three flags `t` ✓
  - AK / DE / MT / NH / OR `0` / all flags `f` ✓
  - HI `0.04` / all flags `t` (GET treated as sales tax) ✓
- Re-running `seed:run` → row count remains 51; `MAX(updated_at)` advances; no duplicates.
- `migrate:down` → table dropped cleanly. Re-running `migrate:up` succeeds.

### FN-1538 (subtask) — Backend

State-aware tax engine in `recomputeWorkOrderTotals`. Replaces the per-line
`taxable` flag (kept as legacy fallback) with `state_tax_rules` per-component
flags driven off `locations.state`.

**Rate selection priority** (matches the FN-1538 algorithm spec):
1. `tax_rate_override === true` → user-supplied `tax_rate_percent / 100`.
2. Active rule found in `state_tax_rules` → `rule.default_sales_tax_rate`.
3. Location with state but no rule → legacy `0.085` + warn log.
4. Location with no state → legacy `0.085` + warn log.
5. No `location_id` → rate `0` (skip tax) + warn log.

**Per-component taxability**: rule flags win. Without a rule, the legacy
per-line `taxable` flag is used so historical numbers don't shift silently
for tenants that never had state rules. Manual rate override still respects
the rule's per-component flags (a TX override of 10% taxes only parts at
10%, not labor or fees).

**Discount apportionment**: discounts reduce the taxable subtotal in
proportion to its share of the overall subtotal. A 10% off coupon reduces
the taxable base by 10%, not the full discount amount.

**Schema additions** (`20260510120000_add_workorder_tax_breakdown.js`):
- `tax_breakdown` JSONB nullable — `{ rule_state, rate, override, fallback_reason, labor_taxable, parts_taxable, fees_taxable, labor_subtotal, parts_subtotal, fees_subtotal, taxable_subtotal, taxable_after_discount, discount_amount, tax_amount }`.
- `tax_rate_override` BOOLEAN default false — disambiguates "user explicitly set 0%" from "use the rule" when `tax_rate_percent = 0`.

**API surface**:
- `getWorkOrderById` already returns the full `work_orders` row, so `tax_breakdown` is exposed automatically (Knex/pg returns JSONB as a parsed object).
- `createWorkOrder` / `updateWorkOrder` / `updateCharges` accept `taxRateOverride: boolean` from the payload.
- `computeWorkOrderTotals` and `loadStateTaxRule` are exported for unit testing.

**Tests** (`backend/packages/goodmen-shared/test/work-orders-tax.test.js`,
10 cases, all passing under `node:test`):
- TX (parts only @ 6.25%), CA (all @ 7.25%), MT (none), FL (parts @ 6%).
- Manual override with rule flags applied.
- No-rule legacy 8.5% fallback against per-line flags.
- No-location skip (rate 0).
- Location with no state → legacy 8.5%.
- Discount apportioned proportionally.
- Explicit 0% override on CA (zero tax even though rule rate is 7.25%).

### FN-1539 (subtask) — Frontend

Wired the Financials tab to the FN-1538 tax engine. The Tax field is now a
read-only display fed by `workOrder.tax_amount`, with an info-button tooltip
that surfaces the breakdown audit (state code + name, effective rate, per-
component flags, taxable subtotal, discount adjustment, fallback reason if
any). Total Cost is also driven from the server's `total_amount` instead of
the legacy 8.5%-on-parts client estimate.

**Override flow.** The Tax field exposes an "Override" button that flips
`workOrder.taxRateOverride = true` and reveals an inline `Tax Rate %` input
seeded with the current effective rate. While override is on, the save
payload sends `taxRateOverride: true` so the backend honors the rate (per-
component flags from the rule still apply per the FN-1538 algorithm). A
"Revert" button clears the override (`taxRateOverride = false`,
`taxRatePercent = null`) so the next save returns to the state default.

**Loader changes** (`work-order.component.ts`):
- Reads `tax_amount`, `total_amount`, `labor_subtotal`, `parts_subtotal`,
  `fees_subtotal`, `tax_rate_percent`, `tax_rate_override`, `tax_breakdown`
  from the GET response and maps them onto the model. JSONB is parsed
  defensively (object or string).
- `computeFinancials()` is replaced by `recomputeActualCost()`, which only
  computes `actualCost = labor + parts + fees` from line items as a pre-save
  sanity figure. **No more client-side 8.5% tax calc** — that was overwriting
  the server-computed values on reload. Fees are now included in
  `actualCost` (closes the parent-story note that it previously didn't).
- `submitWorkOrder` adds `taxRateOverride` to the payload so the backend can
  distinguish "use the rule" from "user picked 0%".

**Tooltip rendering** (`financials-tab.component.ts`):
- `taxTooltipTitle` — e.g. `Texas (TX) — 6.25%` or `Manual override: 8.25%`.
- `taxTooltipDetail` — concise sentence(s) covering: override note, fallback
  reason ("no location selected — tax skipped" / "no state — using legacy
  8.5%" / "no rule for state — using legacy 8.5%"), per-component flags
  ("labor not taxable; parts taxable; fees taxable"), taxable subtotal, and
  the after-discount taxable when a discount apportions.
- US state code → name map for the 50 states + DC so the tooltip reads as a
  human-friendly summary instead of a two-letter abbreviation.

**Estimated Cost** stays a manual user-entered field (no auto-fill) per the
acceptance criteria — confirmed by leaving its `[(ngModel)]` binding alone.

### FN-1538 (this subtask) — Backend
- **Migration to run**: `20260510120000_add_workorder_tax_breakdown.js` adds two columns to `work_orders` (`tax_breakdown` JSONB, `tax_rate_override` BOOL default false). Idempotent — skips columns that already exist. Run on `fleetneuron-db-migrations` before promoting the service.
- **Depends on**: FN-1537's `20260507230000_create_state_tax_rules.js` migration + `07_state_tax_rules_seed.js` seed must be applied first (already on the integration branch). Confirm `SELECT COUNT(*) FROM state_tax_rules` returns 51 post-deploy.

- **Recompute on next save**: existing WOs are NOT auto-backfilled. Tax recomputes whenever a WO is saved (create/update/charges/labor/parts/fees). WOs in states with non-default rules (e.g., TX where labor was previously taxed at 8.5%) will see a one-time tax delta on next save — flag in release notes.
- **Env / cache / flags**: none.
