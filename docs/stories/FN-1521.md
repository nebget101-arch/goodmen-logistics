# FN-1521: [Work Order] Financials — auto-apply state-based tax rules from selected location

## Requirement

Replace the hardcoded 8.5% on parts with a state-aware tax engine. Per state DoR rules: tax rate, plus per-component flags (labor / parts / fees taxable). Example: Texas — parts taxable @ 6.25%, labor not. Use selected Shop Location's state to pick the rule. Manual override remains available via `tax_rate_percent`.

## Acceptance Criteria

- [ ] `state_tax_rules` table seeded from a researched 50-state CSV with citations
- [ ] `recomputeWorkOrderTotals()` consumes location → state → rules; per-component flags honored
- [ ] `tax_breakdown` JSONB column persists the breakdown; returned in GET response
- [ ] Frontend Financials tab shows computed tax + tooltip + override toggle
- [ ] Test cases: TX (parts only), CA (both), MT (none), FL (parts only) all match expected calculations
- [ ] Existing WOs recompute on next save (no auto-backfill in scope)

## Dependencies

- Subtask chain: ai (research CSV) → database (table + seed) → backend (engine) → frontend (UI) → qa
- Recommended sequencing with FN-1520 (parts selling price): FN-1520 ships first so the parts subtotal is correct before tax computes against it. Not a hard block, but a coordination note.

## Implementation Summary

### FN-1536 (subtask) — AI

Researched and produced the 50-state + DC sales-tax rules reference dataset at `docs/reference/state-sales-tax-rules-2026.csv`. The CSV is the seed source for the `state_tax_rules` table built by FN-1537 and consumed by the tax engine in FN-1538.

**Approach.** State-by-state across DoR publications, statutes, and regulations. Aggregators (Avalara, Tax Foundation) used only as triage; primary state sources resolved every classification. Five spot-check states verified hand-to-DoR (TX, CA, FL, NY, PA).

**Key finding.** The labor-taxability axis is the dominant variable. Jurisdictions cluster:
- **No-tax (5)**: AK, DE, MT, NH, OR — all flags `false`, rate `0`
- **Labor exempt when separately stated (~26)**: AL, AZ, CA, CO, GA, ID, IL, IN, MA, MD, ME, MI, MN, MO, NE, NV, ND, OH, OK, RI, SC, TX, VA, VT
- **Labor fully taxable (~16 + DC)**: AR, CT, DC, FL, HI, IA, KS, KY, LA, MS, NC, NJ, NM, NY, PA, SD, TN, WA, WI, WV, WY
- **Edge cases**: MA inconsequential rule, VA diagnostic-labor exemption (2023), KY 2018 statute change, FL/NJ "labor taxable when parts furnished"

**Sanity-check confirmation** (FN-1536 acceptance criteria):
- TX → labor `false`, parts `true`, rate `0.0625` (Tex. Tax Code §151.0048; Comptroller Pub. 94-113)
- AK / DE / MT / NH / OR → all flags `false`, rate `0`

**Methodology + caveats**: see `docs/reference/state-sales-tax-rules-2026.md` for full methodology, the `fees_taxable` discussion, spot-check evidence table, and limitations (local rates, sector exemptions, warranty work).

**CSV consumer contract** (for FN-1537 seed script):
- 51 rows + header. UTF-8. RFC 4180 quoting (notes contain commas → quoted).
- Booleans lowercase (`true`/`false`) — Postgres-friendly without coercion.
- `default_sales_tax_rate` is decimal (`0.0625`, not `6.25`).
- `effective_from` = `2026-01-01` for all rows.
- Recommended Postgres column types listed in the markdown.

### FN-1537 (subtask) — Database

Created the `state_tax_rules` reference table per the spec in FN-1537, plus an idempotent seed that loads it from `docs/reference/state-sales-tax-rules-2026.csv` (FN-1536 output).

**Migration** (`20260507230000_create_state_tax_rules.js`)
- Schema matches the ticket exactly: `state_code VARCHAR(2) PK`, `state_name VARCHAR(64)`, `default_sales_tax_rate NUMERIC(6,4)`, three boolean flags (labor / parts / fees taxable), `notes`, `source_url`, `effective_from / effective_to`, timezone-aware `created_at / updated_at`.
- Two CHECK constraints — `default_sales_tax_rate >= 0`, and effective range well-formedness (`effective_to IS NULL OR effective_to >= effective_from`) — to keep the table self-validating against bad future seed updates.
- Down migration drops the table cleanly.

**Seed** (`07_state_tax_rules_seed.js`)
- Reads the CSV directly so the in-repo human source of truth and the DB never drift. RFC 4180 inline parser handles the quoted `notes` fields (which contain commas inside citations).
- Header validation: errors loudly if the CSV column order changes — preferable to silent data corruption.
- Single-statement upsert: `INSERT ... ON CONFLICT (state_code) DO UPDATE` over 9 columns + `updated_at`. Re-running the seed is idempotent; row count stays at 51, only `updated_at` advances.
- Final row-count assertion (must be ≥ 51) catches a half-written CSV.

**Verification (against fresh Postgres DB `fn_1537_test`):**
- `migrate:up` → table created with the documented schema (verified via `\d state_tax_rules`); both CHECK constraints present.
- `seed:run` → 51 rows. Spot-checks against the methodology doc:
  - TX `0.0625` / labor=f / parts=t / fees=t ✓
  - CA `0.0725` / labor=f / parts=t / fees=f ✓
  - FL / NY / PA `0.06–0.07` / all three flags `t` ✓
  - AK / DE / MT / NH / OR `0` / all flags `f` ✓
  - HI `0.04` / all flags `t` (GET treated as sales tax) ✓
- Re-running `seed:run` → row count remains 51; `MAX(updated_at)` advances; no duplicates.
- `migrate:down` → table dropped cleanly. Re-running `migrate:up` succeeds.
