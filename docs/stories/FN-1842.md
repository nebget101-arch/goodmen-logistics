# FN-1842: Lease-to-own create fails with DB type error — Driver/Truck ID free-text inputs, no UUID validation

## Requirement
Creating a lease-to-own agreement fails when a non-UUID value reaches the driver/vehicle lookup. Observed in production (logistics-service):

```
lease_agreement_create_failed:
select * from "drivers" where "id" = $1 and "tenant_id" = $2 and "operating_entity_id" = $3 limit $4
- invalid input syntax for type uuid: "SN-3918"
```

`SN-3918` is a vehicle unit number; it was entered where a driver/truck **UUID** is required. Two compounding defects: (1) the form lets users hand-type IDs, and (2) the backend doesn't validate UUID format, so Postgres throws an uncaught type-cast error instead of a clean 400.

## Type
Bug (no epic — core lease-to-own financing; no lease-financing epic exists)

## Root Cause (verified on `origin/dev`)
- **Frontend (primary):** free-text inputs in `frontend/src/app/lease-financing/pages/lease-agreement-form/lease-agreement-form.component.ts:22` (`driver_id`) and `:26` (`truck_id`). No picker constrains values to real entity UUIDs.
- **Backend (defensive gap):** `backend/packages/goodmen-shared/routes/lease-financing.js` `POST /lease-agreements` reads `driver_id`/`truck_id` (lines ~618-619) and queries `drivers`/`vehicles` by `id` (~631-640) with no UUID check → pg `invalid input syntax for type uuid` escapes uncaught.

## Implementation Summary

### FN-1844 (subtask) — Backend: UUID validation + clean 400

Hardened `routes/lease-financing.js` so non-UUID identifiers are rejected with a
`400` before they reach any Postgres `uuid` column (which throws an uncaught
`invalid input syntax for type uuid` → 500). Three guard points:

1. **New shared helper** `utils/uuid.js` → `isUuid(value)` (strict RFC-4122 v1–5
   regex, string-only). Centralizes the pattern previously inlined in
   `routes/work-orders.js`. Unit-tested in `utils/uuid.test.js` (4 cases / 16
   assertions: valid v1/v3/v4, the `SN-3918` bug input, malformed shapes, and
   non-string input).
2. **`router.param('id', …)`** — one guard covers every `:id` lifecycle endpoint
   (`GET/PUT /lease-agreements/:id`, `:id/activate`, `:id/terminate`,
   `:id/upload-contract`, `:id/payment-schedule`, `:id/manual-payment`,
   `:id/sign`). Registered after the RBAC middleware so auth still runs first; a
   non-UUID `:id` → `400 { error: 'Invalid lease agreement id' }`.
3. **Body ids** — `POST /lease-agreements` validates `driver_id` + `truck_id`
   (→ `400 { error: 'driver_id and truck_id must be valid identifiers' }`,
   the FN-1842 root-cause path); `:id/manual-payment` validates the optional
   `payment_schedule_id` (queried by uuid) and `settlement_id` (inserted into a
   uuid column via `applyPayment`).

This is the backend hardening half of FN-1842; the FE picker (FN-1843) is the
primary fix that stops users entering free-text ids in the first place.

## Key Decisions

- **Self-contained regex helper, not the `uuid` package's `validate`.** Zero
  runtime dep, trivially unit-testable without `node_modules`, and consistent
  with the existing inline check in `work-orders.js`. All entity ids are
  `uuid.v4()`-generated, so the strict v1–5 regex matches every real id.
- **`router.param` over per-handler guards** for `:id` — every `:id` in this
  router is a lease-agreement id, so one param hook is DRY and impossible to
  forget on a future endpoint. Returns `400` (malformed input), distinct from
  the `404` returned when a well-formed id simply isn't found.
- **Scope:** path/body ids per the ticket. List-endpoint query-string filters
  (`?driver_id=`/`?truck_id=`/`?mc_id=` in `mapAgreementFilters`) are a related
  but separate 500 vector — see Open Items; not changed here to avoid altering
  list-filter behavior outside this subtask's AC.
