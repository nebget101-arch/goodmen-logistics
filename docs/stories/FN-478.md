# FN-478: AI-Powered PSP Report Analysis at Onboarding

## Requirement
After a PSP (Pre-employment Screening Program) report is uploaded to the DQF, auto-trigger Claude Vision to extract inspection records and crash data, import them into `fmcsa_inspection_history` with `source='psp_upload'`, create driver risk events, and generate a pre-hire assessment event when both MVR and PSP are present.

## Acceptance Criteria
- [x] `psp_report` document type supported in DQF upload
- [x] After upload, auto-trigger Claude Vision extraction (background, non-blocking)
- [x] Extracted inspections imported into `fmcsa_inspection_history` (deduped by report_number)
- [x] Each inspection matched to driver via `matched_driver_id` (psp_upload method)
- [x] `driver_risk_events` created for each imported inspection
- [x] Pre-hire assessment risk event created when both MVR and PSP documents are present
- [x] DQF requirement `psp_report_document` marked complete after processing
- [x] PSP vision HTTP endpoint in ai-service (`POST /drivers/psp-vision`)

## Implementation Summary

### PSP Extraction Service (`psp-extraction-service.js`)
- Calls Anthropic Vision API via axios (same pattern as `mvr-extraction-service.js`)
- Supports both image/* (JPEG, PNG, WebP) and application/pdf inputs
- Extracts: driver name/CDL, all inspection records with BASIC violation categories, crash records
- Returns normalized structure with confidence score and warnings

### AI Service Handler (`psp-report-vision-handler.js`)
- HTTP endpoint using Anthropic SDK (same pattern as `toll-invoice-vision-handler.js`)
- Reuses the prompt and normalizer from `psp-extraction-service.js`
- Route: `POST /drivers/psp-vision`

### DQF Upload Integration (`dqf-documents.js`)
- After upload of `documentType === 'psp_report'`, fires `processPspReport()` as a background async (non-blocking, errors caught and logged)
- `processPspReport`:
  1. Calls `extractPspData()` → Claude Vision
  2. Inserts each inspection into `fmcsa_inspection_history` with `match_status='manual'`, `match_method='psp_upload'`, `matched_driver_id=driverId`
  3. Creates `driver_risk_events` for each inspection via `createRiskEvent()`
  4. Calls `createPreHireAssessmentIfReady()` — checks for existing MVR doc, creates combined `pre_hire_assessment` event if both present
  5. Calls `upsertRequirementStatus(driverId, 'psp_report_document', 'complete', docId)` to mark DQF requirement done

## Key Decisions
- PSP processing is fire-and-forget (background) — upload response is not delayed by AI extraction
- Inspections from PSP are matched with `match_status='manual'` and `match_method='psp_upload'` — confidence reflects AI extraction confidence
- Pre-hire assessment is deduplicated: only one per driver per source
- `psp_report_document` DQF requirement upsert is non-fatal (key may not exist in all tenant configurations)

## Deployment Handoff

### Services to redeploy
- `goodmen-shared` (backend) — dqf-documents route + psp-extraction-service
- `ai-service` — new PSP vision handler + route

### DB migrations
- None (uses existing `fmcsa_inspection_history`, `driver_risk_events` tables from FN-474)
- To enable DQF requirement tracking: run `INSERT INTO dqf_requirements (key, label, category) VALUES ('psp_report_document', 'PSP Report', 'pre_hire') ON CONFLICT DO NOTHING;`

### Environment/config changes
- `ANTHROPIC_API_KEY` — required for PSP extraction (same key used by MVR and toll vision)
- `ANTHROPIC_VISION_MODEL` — optional, defaults to `claude-sonnet-4-20250514`

### Post-deploy verification
1. Upload a PSP report document (PDF or image) via DQF upload with `documentType=psp_report`
2. Check server logs for `psp_processing_complete` event
3. Query `fmcsa_inspection_history` — verify records appear with `match_method='psp_upload'`
4. Query `driver_risk_events` — verify events created for the driver
5. Upload an MVR report for the same driver → verify `pre_hire_assessment` risk event created
6. Verify `psp_report_document` requirement status is `complete` in `dqf_driver_status`
