# FN-1552: [Work Order] Financials tab — Tax field shows wrong amount; calculation does not match parts + labor

## Requirement
The Tax field on the Financials tab does not reflect a correct calculation against the parts + labor on the Work tab. Root causes traced in Jira description: (1) hard-coded 8.5% rate at `work-order.component.ts:16`, (2) `computeFinancials()` overwrites manual edits even when "Override" is engaged, (3) reactivity from Work-tab line-item edits is unclear.

## Acceptance Criteria
- [ ] Tax recomputes immediately when a Work-tab part qty / unit price / new labor line changes (not only on save/reload)
- [ ] Manual edits to the Tax field are preserved when "Override" is engaged and NOT overwritten by `computeFinancials()`
- [ ] Total Cost = Actual Cost + Tax, displayed consistently in the Financials section AND in the auto-generated Invoice card
- [ ] On save, the persisted `work_orders.tax` matches what the user sees on screen
- [ ] No regression in invoice line items / invoice totals when tax is non-zero

## Implementation Summary
FN-1521 had already shipped state-based tax rules and the FN-1538 server tax engine, so the rate-source half of the bug was already solved. This change closes the remaining two AC: (1) the Financials tab now stays in sync with unsaved Work-tab edits via a client-side preview, and (2) PATCH responses now mirror the server-computed totals onto the camelCase fields the Financials tab actually binds to (previously only snake_case `tax_amount` / `total_amount` were updated, leaving the on-screen Tax / Total Cost stale until a full reload).

A new `recomputeFinancialsPreview()` on `WorkOrderComponent` computes Actual Cost (labor + parts + fees), a Tax preview that mirrors the server's logic — taxability flags from `tax_breakdown` win, override rate replaces only the rate, discount is apportioned proportionally — and Total Cost = subtotal − discount + tax. The preview runs whenever `WoWorkTabComponent` (labor add / remove / hours-or-rate edit, plus PATCH-response sync) or `WoFinancialsTabComponent` (Override engaged / reverted / rate changed) emits the new `financialsChanged` event. Server values still win on save: `loadWorkOrder()` was switched to `recomputeActualCost()` (Actual Cost only) so authoritative `tax` / `totalCost` from the server are not overwritten.

## Files Changed
- `frontend/src/app/components/work-order/work-order.component.ts` — added `recomputeFinancialsPreview()` and helpers (`sumLaborCost`, `sumPartsCost`, `sumFeesAmount`, `resolveEffectiveRatePercent`, `computeTaxableSubtotal`, `computeDiscountAmount`); narrowed `recomputeActualCost()` to Actual Cost only.
- `frontend/src/app/components/work-order/work-order.component.html` — wired `(financialsChanged)="recomputeFinancialsPreview()"` on both `<app-wo-work-tab>` and `<app-wo-financials-tab>`.
- `frontend/src/app/components/work-order/tabs/work-tab/work-tab.component.ts` — added `financialsChanged` output; emits on `addLabor` / `removeLabor` / `updateLaborTotals`; mirrors `tax_amount` → `workOrder.tax`, `total_amount` → `workOrder.totalCost`, and recomputes `actualCost` from server subtotals after each PATCH so the camelCase bindings update without a reload round-trip.
- `frontend/src/app/components/work-order/tabs/financials-tab/financials-tab.component.ts` — added `financialsChanged` output; emits on `enableTaxOverride` / `revertTaxOverride` / `onTaxRatePercentChange`.

## Key Decisions
- **Server stays source of truth.** Preview is only for the editing window. After save, `loadWorkOrder()` rewrites `tax` / `totalCost` from the server response, and the (narrowed) `recomputeActualCost()` no longer touches them.
- **Preview mirrors server taxability semantics.** Override changes the rate but does not flip per-component taxable flags (matches `computeWorkOrderTotals` in `backend/packages/goodmen-shared/services/work-orders.service.js:229-268`). Without a `tax_breakdown` (new WO never saved) the preview falls back to parts-only at 8.5%, which is the same legacy fallback the server uses.
- **Override field stays the rate, not the tax amount.** The Tax input remains read-only. The bug description allowed either "preserve a typed tax amount" or "preserve a typed rate"; the rate-based override that landed with FN-1521 is the one that round-trips cleanly through the FN-1538 engine, so we kept it and made sure the displayed Tax tracks the override rate live.
- **PATCH-response camelCase mirror is the smaller fix.** The work-tab already syncs snake_case totals from the server PATCH; adding the camelCase mirror was the minimal change to make Financials reactive to part-qty edits without changing the API contract.
- **No new spec file.** No work-order specs exist today and standing one up was out of scope for a P2 reactivity bug. Verified via `ng build --configuration=development` (work-order module compiles clean) and manual reasoning against server logic.
