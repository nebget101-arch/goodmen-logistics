# FN-1789: Adapter — Equipment / Motor-Carrier Lease Agreement signing

## Requirement
Wire the upload→scan→fill→sign engine (FN-1787 + FN-1788) into the **Equipment / Motor-Carrier Lease Agreement** workflow. From the Equipment context (a leased unit / equipment owner — `vehicles.ownership_type='leased'`, `equipment-owners` settlement area), an internal user uploads the lease PDF, prepares it via the engine, and sends it to the **equipment owner / lessor** to e-sign. The signed agreement is linked back to the vehicle/equipment-owner record.

## Acceptance Criteria
- [x] Entry point to start an "Equipment Lease Agreement" signing from the Equipment/Vehicles or Equipment-Owners area (document_type = `lease_agreement`). _(backend: `POST /api/agreements/equipment-lease/requests`; frontend wiring is FN-1801)_
- [x] Backend links a completed signature request to the vehicle/equipment-owner (store request id + signed PDF key on the linkage). _(`equipment_lease_signings` join; signed-PDF key backfilled on read)_
- [x] Internal user can see lease signing status (sent / viewed / signed) and download the signed PDF from the equipment record. _(backend: `GET /api/agreements/equipment-lease/requests?subjectType=&subjectId=`; UI is FN-1801)_
- [x] Reuses the generic engine endpoints — no duplicate signing logic. _(adapter delegates wholly to `signature-service`)_

### FN-1800 (subtask) — backend linkage
A **thin adapter** that ties the generic e-signature engine to an equipment subject without duplicating any signing logic:

- **Schema** — new `equipment_lease_signings` join table: `(id, tenant_id, operating_entity_id, subject_type ['vehicle'|'equipment_owner'], subject_id, signature_request_id → signature_requests, document_type default 'lease_agreement', signed_pdf_storage_key, created_by, timestamps)`. Unique on `signature_request_id` (one subject per request); index on `(tenant_id, subject_type, subject_id)`. Additive, `hasTable`-guarded, FK attached only when `signature_requests` exists.
- **Service** — `equipment-lease-signing-service.js`:
  - `createEquipmentLeaseSigning` validates the subject + signer, **delegates the full fill→send→sign lifecycle to `signature-service.createSignatureRequest`**, then inserts the linkage row pointing at the engine-created request.
  - `listEquipmentLeaseSignings` lists a subject's signings (newest first), enriches each with live status + signed-PDF URL from `signature-service.getRequestById`, and **lazily backfills** the signed-PDF R2 key onto the linkage once the request is signed.
- **Routes** — `equipment-lease-signings.js`, mounted at `/api/agreements/equipment-lease` **ahead of** the generic `/api/agreements` router (the generic `POST /:templateId/requests` would otherwise capture `…/equipment-lease/requests`):
  - `POST /api/agreements/equipment-lease/requests` → `{ subjectType, subjectId, templateId, fieldValues?, signer:{name,email,phone,role}, expiresInDays? }` → `201 { requestId, signerLink, status, link }`.
  - `GET /api/agreements/equipment-lease/requests?subjectType=&subjectId=` → `200 { data: [{ id, subjectType, subjectId, signatureRequestId, documentType, createdAt, request:{ status, signerName, sentAt, viewedAt, signedAt, signedPdfUrl, … } }] }`.
- **No gateway change** — `/api/agreements/*` already proxies to logistics-service.
- **Tests** — `test/equipment-lease-signing-service.test.js` (Node `node --test`): 18 assertions over the pure helpers + create/list orchestration with an injected fake knex + fake signature-service (confirms delegation, linkage persistence, and signed-key backfill). All green.

## Key Decisions
- **Join table over `subject_type`/`subject_id` columns on `signature_requests`.** Keeps the engine schema document-type-agnostic; the equipment-domain linkage lives in the adapter's own table. Story 4 (lease-to-own) can add its own linkage the same way.
- **Mount under `/api/agreements/equipment-lease`, not a new top-level prefix.** Reuses the existing gateway proxy + auth/tenant/subscription guards — zero gateway change. Registered before the generic router so the specific path wins.
- **Signed-PDF key stored on the linkage but backfilled lazily on read** (not via a webhook/callback from the engine), so the adapter has no coupling into the engine's signing transaction. The request remains the source of truth; the linkage is a denormalized cache for the equipment record.
- **Delegation, not duplication.** The adapter never touches tokens, PDFs, or status transitions — it calls `signature-service` for everything and only owns the subject↔request mapping.
