# FN-1789: Adapter ‚Äî Equipment / Motor-Carrier Lease Agreement signing

## Requirement
Wire the upload‚Üíscan‚Üífill‚Üísign engine (FN-1787 + FN-1788) into the **Equipment / Motor-Carrier Lease Agreement** workflow. From the Equipment context (a leased unit / equipment owner ‚Äî `vehicles.ownership_type='leased'`, `equipment-owners` settlement area), an internal user uploads the lease PDF, prepares it via the engine, and sends it to the **equipment owner / lessor** to e-sign. The signed agreement is linked back to the vehicle/equipment-owner record.

## Acceptance Criteria
- [x] Entry point to start an "Equipment Lease Agreement" signing from the Equipment/Vehicles or Equipment-Owners area (document_type = `lease_agreement`). _(backend: `POST /api/agreements/equipment-lease/requests`; frontend wiring FN-1801 ‚Äî vehicle detail "Lease Agreement" tab + equipment-owner row action)_
- [x] Backend links a completed signature request to the vehicle/equipment-owner (store request id + signed PDF key on the linkage). _(`equipment_lease_signings` join; signed-PDF key backfilled on read)_
- [x] Internal user can see lease signing status (sent / viewed / signed) and download the signed PDF from the equipment record. _(backend: `GET /api/agreements/equipment-lease/requests?subjectType=&subjectId=`; UI FN-1801 ‚Äî status table + download link on the vehicle's Lease Agreement tab)_
- [x] Reuses the generic engine endpoints ‚Äî no duplicate signing logic. _(adapter delegates wholly to `signature-service`; frontend reuses the FN-1787/FN-1788 upload‚Üíreview‚Üísend components with no new signing UI)_

### FN-1800 (subtask) ‚Äî backend linkage
A **thin adapter** that ties the generic e-signature engine to an equipment subject without duplicating any signing logic:

- **Schema** ‚Äî new `equipment_lease_signings` join table: `(id, tenant_id, operating_entity_id, subject_type ['vehicle'|'equipment_owner'], subject_id, signature_request_id ‚Üí signature_requests, document_type default 'lease_agreement', signed_pdf_storage_key, created_by, timestamps)`. Unique on `signature_request_id` (one subject per request); index on `(tenant_id, subject_type, subject_id)`. Additive, `hasTable`-guarded, FK attached only when `signature_requests` exists.
- **Service** ‚Äî `equipment-lease-signing-service.js`:
  - `createEquipmentLeaseSigning` validates the subject + signer, **delegates the full fill‚Üísend‚Üísign lifecycle to `signature-service.createSignatureRequest`**, then inserts the linkage row pointing at the engine-created request.
  - `listEquipmentLeaseSignings` lists a subject's signings (newest first), enriches each with live status + signed-PDF URL from `signature-service.getRequestById`, and **lazily backfills** the signed-PDF R2 key onto the linkage once the request is signed.
- **Routes** ‚Äî `equipment-lease-signings.js`, mounted at `/api/agreements/equipment-lease` **ahead of** the generic `/api/agreements` router (the generic `POST /:templateId/requests` would otherwise capture `‚Ä¶/equipment-lease/requests`):
  - `POST /api/agreements/equipment-lease/requests` ‚Üí `{ subjectType, subjectId, templateId, fieldValues?, signer:{name,email,phone,role}, expiresInDays? }` ‚Üí `201 { requestId, signerLink, status, link }`.
  - `GET /api/agreements/equipment-lease/requests?subjectType=&subjectId=` ‚Üí `200 { data: [{ id, subjectType, subjectId, signatureRequestId, documentType, createdAt, request:{ status, signerName, sentAt, viewedAt, signedAt, signedPdfUrl, ‚Ä¶ } }] }`.
- **No gateway change** ‚Äî `/api/agreements/*` already proxies to logistics-service.
- **Tests** ‚Äî `test/equipment-lease-signing-service.test.js` (Node `node --test`): 18 assertions over the pure helpers + create/list orchestration with an injected fake knex + fake signature-service (confirms delegation, linkage persistence, and signed-key backfill). All green.

### FN-1801 (subtask) ‚Äî frontend entry point + status display
Reuses the FN-1787/FN-1788 agreements components end to end ‚Äî **no new signing UI**. The only behavioural change to the generic engine is which endpoint the final send call hits when the flow is scoped to an equipment subject.

- **Service client** ‚Äî added `startEquipmentLeaseSigning()` (POST adapter) and `listEquipmentLeaseSignings(subjectType, subjectId)` (GET adapter; unwraps the `{ data }` envelope) to `AgreementService`. New `equipment-lease-signing.model.ts` mirrors the FN-1800 contract DTOs.
- **Subject-context threading** ‚Äî the entry points navigate into the existing `/agreements` upload flow with `subjectType` / `subjectId` / `subjectLabel` / `documentType=lease_agreement` query params. Upload ‚Üí review preserve the params (`queryParamsHandling: 'preserve'`); on finalize, the review screen continues into the **existing** `:id/send` screen (instead of the template list) when a subject is present. The send screen, when scoped to a subject, routes through `startEquipmentLeaseSigning` so the resulting request is linked back to the subject; otherwise it uses the generic `createRequest` exactly as before. A small "Lease agreement for ‚Ä¶" banner is shown on upload/send for context.
- **Vehicle detail status display** ‚Äî a new **"Lease Agreement" tab** on the Vehicle Details modal, shown only for **leased** units (`getOwnership(v) === 'leased'`). It carries the "Upload & send lease agreement" entry button plus a table of the unit's signings with status (sent/viewed/signed ‚Äî reusing the engine's `statusLabel`), sent/signed dates, and a **Download** link to the signed PDF (`request.signedPdfUrl`). Signings load lazily on first tab open and reset on modal open/close.
- **Equipment-owners entry point (optional)** ‚Äî a per-row **"Lease agreement"** action in the Equipment Owners table that starts the same flow with `subjectType=equipment_owner`.
- **Tests** ‚Äî `agreement.service.spec.ts` (new): POST payload + GET query-param/`data`-unwrap for the adapter methods. `agreement-send.component.spec.ts` (new): asserts the send branches to the adapter when a subject is present and to the generic engine otherwise. `agreement-review.component.spec.ts` (updated): query-param stub + a test that finalize continues into `:id/send` (preserving params) when scoped to a subject. _(Karma can't run in this environment ‚Äî `ng build` validates compilation/templates; specs are authored to the existing Jasmine patterns.)_

## Key Decisions
- **Join table over `subject_type`/`subject_id` columns on `signature_requests`.** Keeps the engine schema document-type-agnostic; the equipment-domain linkage lives in the adapter's own table. Story 4 (lease-to-own) can add its own linkage the same way.
- **Mount under `/api/agreements/equipment-lease`, not a new top-level prefix.** Reuses the existing gateway proxy + auth/tenant/subscription guards ‚Äî zero gateway change. Registered before the generic router so the specific path wins.
- **Signed-PDF key stored on the linkage but backfilled lazily on read** (not via a webhook/callback from the engine), so the adapter has no coupling into the engine's signing transaction. The request remains the source of truth; the linkage is a denormalized cache for the equipment record.
- **Delegation, not duplication.** The adapter never touches tokens, PDFs, or status transitions ‚Äî it calls `signature-service` for everything and only owns the subject‚Üîrequest mapping.
