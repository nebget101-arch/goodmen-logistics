# FN-1081: Bulk-extracted load missing source PDF as RATE_CONFIRMATION attachment

## Requirement
`BulkExtractionGridComponent.processFile` calls `aiExtractFromPdf` + `createLoad` but never uploads the source PDF as an attachment, so loads created via the bulk grid have zero attachments. The single-PDF wizard path (`LoadWizardComponent.submitCreate`) does this correctly. Add the missing `uploadAttachment` call after `createLoad` succeeds, with proper partial-failure handling.

The longer-term refactor to consolidate on the existing `POST /api/loads/bulk-rate-confirmations` endpoint (which extracts AND attaches in one transaction) is deferred to a follow-up story — for this bug, just add the missing attachment step in the current client-side flow.

## Acceptance Criteria
- [ ] After bulk-extraction completes for a PDF, opening the resulting load in the wizard or detail drawer shows the source PDF in the Attachments section as type `RATE_CONFIRMATION`.
- [ ] If load creation succeeds but attachment upload fails, the grid row shows a clear partial-success state (load created link + warning that PDF didn't attach), not a green "SUCCESS" tick.
- [ ] No regression in the existing "load created" link / row navigation behavior on full success.
- [ ] Unit spec coverage for both happy path and partial-success.

## Implementation Summary
`BulkExtractionGridComponent.processFile` now chains `uploadAttachment(load.id, row.file, 'RATE_CONFIRMATION')` after a successful `createLoad`. The row resolves to `SUCCESS` only when both calls succeed. If the upload fails, the row enters a new `PARTIAL_SUCCESS` state — the load link is still visible (so the user can open the load and attach the PDF manually) but the badge is amber with a warning icon, not a green tick. The summary screen surfaces a fourth stat (`PDF not attached`) when any row is in this state. Retry is intentionally restricted to `FAILED` rows because retrying a `PARTIAL_SUCCESS` row would create a duplicate load.

A new sub-status `ATTACHING` is shown briefly (with a spinning paperclip icon) between `CREATING` and the terminal state, mirroring the existing `EXTRACTING`/`CREATING` progress badges. The summary table's "retrying" badge now also covers `ATTACHING`.

A new spec file (`bulk-extraction-grid.component.spec.ts`) covers the three terminal paths: happy (all three calls succeed), partial (createLoad ok, uploadAttachment fails), and full failure (createLoad fails, no upload call made).

## Key Decisions
- **PARTIAL_SUCCESS as a first-class status** rather than reusing `SUCCESS` + a flag, so the row class, badge, and summary-stat all branch cleanly on `row.status` alone. Prevents accidental "green tick + attached" rendering anywhere downstream consumers may key off `successCount`.
- **`autoApprovedCount`/`needsReviewCount` continue to filter `SUCCESS` only** — partial-success loads are reported in their own tile. This keeps the existing "auto-approved" semantics ("create load + attach RATE_CON in one go succeeded") instead of muddling them.
- **No retry for PARTIAL_SUCCESS** — the load is already created in the DB, so a retry would create a duplicate. The user is directed (via the warning + tooltip with the upload error) to open the load and attach the PDF manually from the detail drawer.
- **No second pass for the bigger refactor** — keeps this PR scoped to the bug. The follow-up story tracked in Open Items will swap the per-file `aiExtract`+`createLoad`+`uploadAttachment` chain for a single `bulkUploadRateConfirmations` call (the existing backend endpoint that does both atomically).
