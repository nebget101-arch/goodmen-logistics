# FN-1381: Add/Edit Vehicle form redesign + explicit OO vs Company-owned classification

## Requirement
Restructure the Add/Edit Vehicle modal into a 5-section layout shared by trucks and trailers, and introduce a first-class **Ownership Type** segmented control (Company | OO | Leased) with conditional fields. Persist as a structured `vehicles.ownership_type` enum so settlements, work-order routing, and maintenance billing can rely on it.

## Acceptance Criteria
- [ ] Add Truck and Add Trailer modals share the same 5-section layout (Basic Info, Ownership Type, Compliance & Maintenance, Status, Documents) plus a collapsible Trailer Details accordion.
- [ ] Segmented Ownership control writes `ownership_type` and shows/hides conditional fields.
- [ ] OO save persists `equipment_owner_name`, `operating_entity_id`, `equipment_owner_percentage`.
- [ ] Leased save persists `lessor_name`, `lease_date`, `lease_payment_amount`.
- [ ] Edit modal pre-populates segmented control from `ownership_type`.
- [ ] Validation: OO requires Equipment Owner; Leased requires Lessor.
- [ ] No regressions in VIN decode, document upload, trailer-type chip-search.
- [ ] `company_owned` boolean kept for settlements backward compatibility.

## Implementation Summary
All three implementing subtasks (DB, BE, FE) are complete and integrated on `integration/FN-1381`. Manual QA (FN-1388) is the remaining gate before the story PR opens.

### FN-1387 (subtask)
- **Component restructured into a single shared 5-section template.** Deleted the old branching `*ngIf="isTrailerMode; else legacyVehicleForm"` split. One template now covers trucks and trailers; a sixth section — a collapsible **Trailer Details** accordion — appears only when `isTrailerMode`. Sections in order: Basic Information, Ownership Type, Compliance & Maintenance, Status, (Trailer Details), Documents.
- **Ownership Type segmented control.** Three-button radio-group (Company | OO | Leased) bound to `formData.ownership_type`. `setOwnershipType()` clears stale fields from the now-hidden branch so the save payload never leaks across modes (e.g., switching OO → Company wipes equipment_owner_name/id/percentage/operating_entity_id; switching to Company wipes lessor_name/lease_date/lease_payment_amount). Conditional fields render below the segmented buttons inside an `.ownership-fields` panel.
- **OO branch — equipment-owner typeahead.** Wired to existing `ApiService.searchPayees(term, 'all', 20)` with a 250ms debounce; selecting a result captures `equipment_owner_id` AND mirrors the name into `equipment_owner_name`. Free-text typing also persists as `equipment_owner_name` (backend FN-1386 only requires the name string), so users can add a brand-new owner without leaving the modal. **Resolves the FN-1381 Open Item** asking to confirm the endpoint — there is no `/api/equipment-owners` endpoint today, but `searchPayees` already returns equipment-owner payees and matches the backend's `equipment_owner_name` shape.
- **OO branch — operating entity dropdown** uses existing `ApiService.listOperatingEntities()` and the `app-ai-select` component with a cached options array (per the `app-ai-select` infinite-CD-loop guidance: never bind a getter that returns a new array on every CD pass).
- **Leased branch — trucks** get top-level `lessor_name` / `lease_date` / `lease_payment_amount` fields. Trailer leases also write the same shared fields up top, and `onSubmit()` mirrors them into `trailer_details.lessor_name` / `lease_date` / `ownership='leased'` so legacy trailer-card readers and the backend's trailer-nested validator path both keep working.
- **Validation parity with backend FN-1386.** `validateForm()` enforces: OO requires `equipment_owner_name`; Leased requires `lessor_name` (truck path) — same rules backend rejects with 400, but caught client-side so users see the error inline before the network round-trip. Errors clear immediately when the user fixes the field or switches ownership type.
- **Edit mode pre-population.** `loadFormData()` reads `vehicle.ownership_type` from the API; falls back to `trailer_details.ownership === 'leased'` (legacy trailer path) and then to `company_owned === false` (legacy OO path). For trailers stored leased pre-FN-1387, the mirrored top-level lessor fields are seeded from `trailer_details` so the redesigned panel shows the existing data on first open.
- **Trailer-type chip-search preserved** — moved from the trailer-only `<ng-template>` block into the Trailer Details accordion. Chip filtering / cached `_filteredTrailerTypeOptionsCache` / dropdown z-index all unchanged. Accordion auto-expands on Edit when the trailer has any meaningful detail (`trailer_type_code`, `assigned_driver_id`, `fid_number`, `address`, `city`, `notes`, `history`).
- **VIN decode + document upload untouched.** `onVinChange()` / `decodeVin()` / `onFileSelect()` all still work the same way; the VIN input + decode message live in Section 1 of the shared template and document upload is Section 5, so no behavior change.
- **Cached options pattern preserved.** Added `operatingEntitySelectOptionsCached` and `equipmentOwnerResults` arrays — both populated only inside subscribe callbacks, never via getters that allocate per access. Equipment-owner typeahead debounces input at 250ms via a single tracked timer that's cleared in `ngOnDestroy()`.
- **Build verification.** `ng build --configuration development` runs clean (EXIT=0, ~90s, only pre-existing unused-spec warnings; no template/TS errors introduced).

### FN-1386 (subtask)
- Added `validateOwnership(body)` helper to `backend/packages/goodmen-shared/routes/vehicles.js` and exported it (with `OWNERSHIP_TYPES`) on the router for unit tests. Validation only fires when the caller supplies `ownership_type`, so partial PUTs that don't touch ownership stay backwards-compatible.
- `POST /api/vehicles`: validates ownership_type → 400 on bad payload before any DB write. After the existing INSERT (and the existing `trailer_details` follow-up UPDATE), runs a single dynamic UPDATE that writes `ownership_type` and derives `company_owned` from it (`true` only when `'company'`). Conditional fields (`equipment_owner_name`, `equipment_owner_id`, `lessor_name`, `lease_date`, `lease_payment_amount`) are persisted only when the column exists on `vehicles` — keeps the route resilient to schema variation across environments. Replaces the prior unconditional `company_owned = true` write that was forcing every newly created vehicle to look company-owned to settlements. Response is re-read from `vehicles` so the caller sees the persisted ownership shape.
- `PUT /api/vehicles/:id`: same validation gate. When `ownership_type` is in the body, derives `company_owned` from it (overriding any inconsistent `company_owned` the caller sent) before the existing dynamic-column update path runs, so settlements never sees `ownership_type='oo'` paired with `company_owned=true`.
- `GET /api/vehicles`: added `?ownership_type=` query filter (rejects values outside `OWNERSHIP_TYPES` with 400) for the FN-1383 fleet-list filter chips. Filter is only applied if the underlying view/table exposes the column — defensive against environments where FN-1385 hasn't run yet.
- OpenAPI annotations updated on the list/post block to advertise the new query filter, the ownership_type field, and the conditional fields.
- New test file `backend/packages/goodmen-shared/test/vehicles-ownership-validation.test.js` exercises the validator across all four shapes (absent, company, oo, leased — including the truck top-level `lessor_name` path and the trailer-nested `trailer_details.lessor_name` path) plus invalid inputs. 13/13 pure-function cases pass under a local Node 20 sanity harness.
- Resolves the FN-1381 Open Item about top-level lessor columns: `lessor_name`/`lease_date`/`lease_payment_amount` are not present on `vehicles` today, but FN-1386 accepts and forwards them — they'll simply be ignored at the SQL layer until FN-1385 (or a follow-up DB ticket) adds the columns. Trailer leases continue to round-trip via `trailer_details.lessor_name`. No-op for FN-1387's UI binding.

### FN-1385 (subtask)
- Added `vehicles.ownership_type` as a nullable-free `text` column (NOT NULL, DEFAULT `'company'`) backed by a `CHECK (ownership_type IN ('company','oo','leased'))` constraint. Chose CHECK + text over a Postgres native `CREATE TYPE … AS ENUM` because the rest of the codebase uses CHECK-style "enums" (e.g., `customer_billing`, `compensation_profiles`) — values can be added later with `ALTER TABLE … DROP CONSTRAINT … ADD CONSTRAINT …`, no `ALTER TYPE` ceremony.
- Backfill runs in two `UPDATE`s in priority order:
  1. `'oo'` for `company_owned = false`
  2. `'leased'` for `trailer_details->>'ownership' = 'leased'` (overrides `'oo'` when both apply, since a leased trailer driven by an OO is still a leased asset for billing).
- Composite index `(tenant_id, ownership_type)` added for the tenant-scoped ownership filter introduced in FN-1383.
- `all_vehicles` view recreated to expose `ownership_type` on the fleet branch; the shop-client (`customer_vehicles`) branch returns `NULL::text` because customer-owned vehicles aren't fleet-classified. View shape otherwise matches the post-FN-1317 baseline.
- Verified with synthetic rows on local Postgres: 4-bucket backfill (company / oo / leased-trailer / leased-OO) all classify correctly and `down()` cleanly drops index + constraint + column and restores the prior view.

## Key Decisions
- **CHECK constraint over native PG ENUM** — matches existing convention, avoids `ALTER TYPE … ADD VALUE` migration friction if a 4th ownership category is added later.
- **`'leased'` wins over `'oo'`** in backfill — chosen because leased equipment carries a lessor-payment obligation regardless of who drives it, and downstream settlements/maintenance code keys on lease vs. OO billing in that priority order.
- **Shop-client vehicles get `NULL` ownership_type** in the view — the column models fleet-side ownership classification only; customer vehicles aren't fleet equipment.
- **`company_owned` preserved** — explicit AC requirement; FN-1386 will write both columns until settlements migrates off `company_owned`.
- **(FN-1387) Equipment-owner typeahead reuses `searchPayees`, not a new endpoint.** The Open Item asked to confirm `/api/equipment-owners`. That endpoint does not exist; building one would expand scope across two services. `searchPayees` already returns equipment-owner payees with the right `{id, name}` shape and is used by other surfaces (driver assignment, settlements). Choosing reuse keeps FN-1387 strictly UI-scoped and avoids a backend round-trip that nobody else needs.
- **(FN-1387) Free-text owner names allowed alongside payee selection.** Backend FN-1386 only requires `equipment_owner_name` (string) for `'oo'` saves; `equipment_owner_id` is optional. Forcing typeahead-only would block users adding a brand-new owner inline and contradict the relaxed validator. The typeahead surfaces existing payees but the input itself is the source of truth for the persisted name.
- **(FN-1387) Single shared template, not two.** The legacy `<ng-template #legacyVehicleForm>` branch is gone. Truck and trailer modes diverge through `*ngIf` on individual fields and the accordion, not by maintaining two parallel forms. This is the explicit AC ("no second modal, just conditional sections via `isTrailerMode`") and removes the long-running drift risk between the two paths.
- **(FN-1387) Ownership Type writes mirror into `trailer_details` for trailers.** Existing readers (trailer-card list, integrations) still inspect `trailer_details->>'ownership'` and `lessor_name`. Writing both the new top-level enum AND the legacy nested fields means we don't have to touch every reader in this story to avoid regressions.
