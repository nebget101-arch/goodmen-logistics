# FN-542: 7 of 8 microservices missing knex.migrate.latest() on startup — DB schema never applied

## Problem
Only `logistics-service` called `knex.migrate.latest()` on startup. The remaining microservices never ran DB migrations, meaning views, tables, and columns added by recent migrations were absent from their databases — silently causing endpoints to return empty data or errors.

Root cause confirmed in FN-541/FN-543: `GET /api/equipment` returned `[]` because the `all_vehicles` view was missing from `vehicles-maintenance-service`'s DB (already fixed in FN-543).

## Fix
Added `knex.migrate.latest()` to the `app.listen()` startup callback of 5 remaining services — matching the `logistics-service` pattern exactly.

`ai-service` was excluded: it has no DB connection (OpenAI proxy only — no knex import).
`vehicles-maintenance-service` was excluded: already fixed in FN-543.

### Pattern applied to each service:
```js
app.listen(PORT, async () => {
  try {
    await knex.migrate.latest();
    console.log('✅ Database migrations applied');
  } catch (err) {
    console.error('⚠️  Migration error (non-fatal):', err.message);
  }
  console.log(`... service running on http://localhost:${PORT}`);
});
```

For `integrations-service` (already async), migration was prepended before the existing `await initScrapeQueue()` call.
For `auth-users-service`, migration was prepended before the existing `startTrialConversionJob()` call.
