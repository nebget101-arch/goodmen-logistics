# FN-1480: Invoice upload + AI line extraction — drop file, auto-fill receiving lines

## Requirement
Upload a vendor invoice (jpg/png/heic/pdf) on the receiving page → AI service extracts vendor, reference, and line items (sku/qty/unitCost) using Claude Vision → user reviews extracted lines in a modal → confirmed lines POST into the current DRAFT ticket. File URL is persisted on the ticket and surfaced in the activity report (FN-1481).

## Acceptance Criteria
- [ ] DB migration adds `invoice_file_url`, `invoice_extracted_at`, `invoice_extracted_data jsonb` to `receiving_tickets` (idempotent)
- [ ] Backend `POST /api/receiving/:id/invoice` (multipart) uploads, attaches, calls AI, returns extracted lines
- [ ] AI `POST /ai/invoice/extract` returns `{ vendor, reference, invoiceDate, lines: [{ sku?, description, qty, unitCost, match }] }` using Claude Vision with prompt caching
- [ ] Frontend: drop-zone card + review modal with editable rows; unmatched SKUs route to Quick Add Part (FN-1090) and back
- [ ] Vendor + reference auto-fill on the ticket header (don't overwrite user values)
- [ ] Re-upload replaces previous file/extraction
- [ ] Logs: no PII / no invoice content
- [ ] Tests: backend (mocked AI), AI service (mocked Anthropic), frontend (component specs)

## Implementation Summary

### FN-1488 (subtask)
Added the database surface area for invoice attachment on receiving tickets. New Knex migration adds three nullable columns to `receiving_tickets`:
- `invoice_file_url` (varchar) — uploaded file URL
- `invoice_extracted_at` (timestamptz) — AI extraction timestamp
- `invoice_extracted_data` (jsonb) — Claude Vision payload (vendor, reference, invoiceDate, lines[])

Migration is idempotent via `hasColumn` guards on both up and down. Verified locally with a forward → reverse → forward → reverse cycle against `goodmen_logistics`: types correct, all nullable, no data loss, idempotent on repeat runs.

`backend/packages/goodmen-shared/services/inventory.service.js` does not reference `receiving_tickets`, so no downstream column-info refresh is required.

### FN-1489 (subtask)
New AI service endpoint `POST /api/ai/invoice/extract` that takes a vendor invoice (image or PDF) and returns structured `{ vendor, reference, invoiceDate, lines: [{ sku, description, qty, unitCost, match }] }`.

- Accepts `{ fileUrl }` OR `{ base64, contentType }`. Supports `image/jpeg|png|webp|gif|heic|heif` and `application/pdf` (sent as a Claude `document` block — no pre-rendering).
- Default model `claude-haiku-4-5-20251001`, override via `AI_INVOICE_MODEL`.
- **Prompt caching**: persona + extraction schema split into two `text` blocks with `cache_control: { type: 'ephemeral' }` so back-to-back uploads reuse the cached prefix. Usage stats (`cacheReadTokens`, `cacheCreationTokens`) bubble up in the response `meta`.
- **Resilience**: one parse-failure retry with stricter "JSON only" reinforcement; 422 on second failure. 502 on Anthropic upstream errors. DB query failures degrade gracefully — lines come back with `match: null`.
- **Parts matching**: SKUs from extracted lines are looked up against `parts` table case-insensitively in batches of 200. Matched lines get `match: { partId, sku, name }`.
- **Logging**: only file size + content-type are logged; never invoice bytes, vendor names, or line items.
- **Tests**: 19 cases in `src/handlers/__tests__/invoice-extractor-handler.test.js`. Anthropic SDK and pg pool injected via `deps` for full isolation. Runs as part of `npm test` in the ai-service package.

### FN-1490 (subtask)
New backend endpoint `POST /api/receiving/:id/invoice` (multipart) that uploads a vendor invoice, persists the storage reference on the receiving ticket, and forwards a short-lived signed URL to the AI service for line extraction.

- **Multipart**: 15 MB cap, restricted to `image/jpeg | image/png | image/heic | image/heif | application/pdf` (mimetype + extension both checked). Multer error → 400 with a clear message.
- **Validation**: 404 on missing ticket, 400 on non-DRAFT, 400 on missing file, 403 on caller without `admin` or `parts_manager` (via existing `requireRole`).
- **Crash-safety on AI failure**: file is uploaded to R2 and `invoice_file_url` is stamped *before* the AI call. If extraction fails, the endpoint still returns 200 with `extraction: null` and `aiError: <code>` so the frontend can decide whether to retry.
- **Auto-fill rule**: vendor and reference are written from the extraction only when the ticket's value is currently null — never overwrites a user-set value.
- **Re-upload**: replaces previous `invoice_file_url`, clears `invoice_extracted_data`/`invoice_extracted_at` before the new AI call, then re-stamps on success. Orphaned R2 objects are reaped by a separate sweeper (out of scope, see FN-1480 open items).
- **Logging**: only ticket id, line count, and an "auto-filled" boolean are logged — never invoice bytes, vendor names, or line item details (matches the AI service's logging policy from FN-1489).
- **AI client**: new `services/receiving-invoice.service.js` posts to `${AI_SERVICE_URL}/api/ai/invoice/extract`, forwards the caller's bearer token, 45s timeout (long enough for Claude Vision on a multi-page PDF), `validateStatus: () => true` so non-200s degrade gracefully.
- **Tests**: 8 cases in `test/receiving-invoice.test.js` using `node:test` with the route's `processInvoiceUpload` core extracted for dependency injection — covers all guards, happy-path persistence ordering, AI-failure resilience, and the don't-overwrite-user-values rule. Run: `cd backend/packages/goodmen-shared && node --test test/receiving-invoice.test.js`.

## Key Decisions

### FN-1488 (subtask)
- **Filename prefix**: ticket spec proposed `20260508_…` but a sibling migration already exists at `20260508120000_…`. Used full timestamp `20260508130000_…` to keep lexical ordering consistent with the rest of the directory and ensure this migration runs after the existing same-day migration.
- **`timestamp` column type**: Knex's `table.timestamp()` produces `timestamp with time zone` in PostgreSQL — same convention as the existing `posted_at`/`created_at`/`updated_at` columns on this table.
- **Down migration drops in reverse-add order** for predictability, with `hasColumn` guards so partial state from a failed up doesn't break a re-run.

### FN-1489 (subtask)
- **File layout**: ticket sketched `routes/`/`services/` directories, but the ai-service convention is a flat `handlers/` folder. Followed the existing pattern (handler + test under `handlers/__tests__/`) so future maintainers don't have to context-switch between two layouts.
- **PDF strategy**: Claude 4.x supports inline PDFs as a `document` block, so we pass the file straight through instead of pre-rendering pages. Falls back to the same image-block path only when the caller flags it as image content.
- **Parts matching is best-effort, not authoritative**: case-insensitive `UPPER(sku) IN (…)` keyed by the *input* SKU so the frontend can correlate matches back to the original extracted line, even if the user later overrides. DB failure does not fail the request — lines just come back with `match: null`.
- **Test runner**: matched the existing `node:assert/strict` + plain-`node` pattern from `loads-nlq-handler.test.js` instead of pulling in jest, keeping the test surface dependency-free.
- **Dep injection vs lazy singleton**: handler accepts `{ anthropic, pool }` via deps for tests. Production calls fall through to lazy SDK + lazy `@goodmen/shared/config/database` pool. Used `'pool' in deps` (not `||`) so tests can pass `pool: null` to skip DB without inadvertently triggering the real connection.

### FN-1490 (subtask)
- **Storage column holds the R2 key, not a URL**: the FN-1488 migration named the column `invoice_file_url` but R2 signed URLs expire in 15 minutes, so persisting a URL would be useless. Stored value is the R2 storage key; signed URLs are minted on-demand for the AI call (and will be for the future activity-report download in FN-1481). Documented inline in the route handler. The column name is a known wart — renaming requires another migration and isn't worth blocking this story.
- **File persisted before AI call**: AI failure does not lose the upload. The endpoint always responds 200 once the file is saved; an extraction failure surfaces as `{ extraction: null, aiError: <code> }` in the response so the frontend can offer a "retry extraction" UX without re-uploading.
- **Auto-fill is one-shot, never overwrite**: vendor and reference fields are only stamped when currently null. Tested explicitly — re-uploading after the user manually corrected vendor will NOT clobber the user's value. Acceptance criterion called this out specifically.
- **Bearer token forwarding**: AI client follows the existing `services/ai-narrative-client.js` pattern — pulls `Authorization: Bearer …` off `req.headers` and re-attaches. Lets the AI service's local JWT verifier identify the calling user when the gateway hasn't already attached `req.user`.
- **Test runner choice**: 46 of 51 existing test files in this package use `node:test` (the `npm test` runner). Five use jest globals but `jest` is not in `package.json` dependencies, so those don't actually run in CI. Wrote this test in `node:test` so it runs alongside the rest. The route handler was refactored to expose a pure `processInvoiceUpload(input, deps)` so we can test all branches without standing up Express, multer, R2, or a live DB.
- **No new env vars by default**: reused `AI_SERVICE_URL` (already set in production for `ai-narrative-client`). New optional `AI_INVOICE_EXTRACT_TIMEOUT_MS` defaults to 45000 — long enough for Claude Vision on a multi-page PDF. Only set in Render if observed timeouts on real invoices warrant tuning.
