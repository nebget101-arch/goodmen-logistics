# FN-1327: Role-based default layouts (Owner / Dispatcher / Compliance presets)

## Requirement
Ship role-based default layouts as presets layered on top of the existing per-user drag-to-reorder system (FN-1130). New users (or "Reset to default" users) see a layout tuned to their role rather than the single owner default.

## Acceptance Criteria
- [ ] New table `dashboard_layout_presets` stores admin-editable preset configurations.
- [ ] `GET /api/users/me/dashboard-layout` returns the role preset when no per-user override exists, sourced from `dashboard_layout_presets`.
- [ ] Three new shipped presets seeded at migration time: Owner, Dispatcher, Compliance.
- [ ] Existing four role keys (dispatcher / safety / maintenance / owner) continue to work; new presets supplement, not replace.
- [ ] Reset-to-default returns the role-correct preset.
- [ ] Settings UI lets users preview and switch between presets before drag-reordering.

## Dependencies
- Built on top of FN-1130 / FN-1171 / FN-1172 / FN-1174 (per-user layouts).

## Schema Contract
```
dashboard_layout_presets
  preset_key          varchar PK
  role_key            varchar (indexed)
  display_name        varchar
  layout_json         jsonb default '{}'
  is_default_for_role boolean
  created_at          timestamptz default now()
  updated_at          timestamptz default now()
```
Seeded rows: one per role marked `is_default_for_role = true`. Layouts mirror current hard-coded values in `frontend/src/app/components/control-center/role-layouts.ts`.

## API Contract
```
GET /api/users/me/dashboard-layout/presets
→ 200 [{ preset_key, role_key, display_name, layout, is_default_for_role }]

PUT /api/users/me/dashboard-layout/preset
body { preset_key }
→ 200 { layout }
```

## Key Decisions
_To be filled by implementing agent_

### FN-1341 (subtask)
- **Migration location**: ticket suggested `backend/microservices/auth-users-service/migrations/`, but the actual codebase keeps all shared-DB migrations under `backend/packages/goodmen-database/migrations/` (sibling of `knexfile.js`). `auth-users-service` has no migrations directory. Used the real location.
- **`compliance` → role_key `safety`**: parent story notes "the existing four role keys (dispatcher / safety / maintenance / owner) continue to work; new presets supplement, not replace." `safety` is the established RBAC role key for compliance/HOS work, so the Compliance preset attaches to `role_key='safety'`. UI surfaces it as "Compliance" via `display_name`. No new role key is introduced.
- **Three presets, three default rows**: AC requires "one row per role marked `is_default_for_role = true`" — interpreted as one default per role *that has a preset*, i.e. the three seeded rows are each the default for their role. `maintenance` has no preset yet (out of scope for this subtask); FN-1342 keeps falling back to layout-store's hard-coded `maintenance` default for now.
- **Unique partial index on `(role_key) WHERE is_default_for_role`**: prevents two rows from both being marked default for the same role. Belt-and-suspenders for FN-1342's read path.
- **Seed both at migration time and via a separate seed file**: AC says both. Migration uses `ON CONFLICT (preset_key) DO NOTHING` so it never overwrites edited rows; the separate seed uses `ON CONFLICT … MERGE` to actively reset the canonical three back to defaults when explicitly run.

### FN-1342 (subtask)
- **Preset store as a separate module, not a method on layout-store**: keeps the read-path concerns split. `preset-store.js` owns the table contract (column list, `is_default_for_role = true` filter, jsonb parsing); `layout-store.js` owns the fallback-and-deep-copy policy. FN-1343's switcher will likely add `listAll()` / `getByPresetKey()` to preset-store without touching layout-store.
- **Defense-in-depth fallback on `try/catch`, not just on null**: the AC says "if `dashboard_layout_presets` is empty, behavior matches current hard-coded fallback" — but in practice the table can also be *unreachable* (DB outage, pre-migration deploy, role missing from PG search_path). `resolveRoleDefaultLayout` swallows the throw, logs once, and falls through to `ROLE_DEFAULTS`. This makes the FN-1342 deploy strictly safer than FN-1172: there is no scenario where the user gets a 500 they did not get before.
- **Deep-copy the preset layout on every read**: the preset-store's parsed JSON object is reused if knex caches the row; without a deep copy, a caller mutating `result.layout.cards` could corrupt the cached object. Added `JSON.parse(JSON.stringify(layout))` in `resolveRoleDefaultLayout` and a regression test for it.
- **`maintenance` role intentionally has no preset row**: FN-1341 only seeded three (owner / dispatcher / compliance). For `maintenance`, the preset-store returns null and layout-store falls back to the hard-coded entry. Kept this as the default rather than adding a fourth preset row in this subtask, because (a) the parent story's "Three new shipped presets" AC is explicit about the count, and (b) the hard-coded `maintenance` default is identical to dispatcher's, so adding a row would be cosmetic. Tracked as future work.
- **DELETE path unchanged at the store level**: the route already does `deleteLayout` → `getLayout`, so DELETE automatically picks up the new preset path through `getLayout`. No new `deleteLayout` signature, and no new "reset" entry point. Kept the surface area minimal.
- **No new public endpoints**: the parent story lists `GET /dashboard-layout/presets` and `PUT /dashboard-layout/preset` as future API contracts; those are out of scope for this subtask (FN-1342 explicitly only updates the existing endpoints) and will be added by FN-1343 / a follow-up if needed.
- **Tests use a knex builder mock, not a real Postgres**: matches the FN-1172 pattern in `dashboard-layout.test.js`. The full Postgres round-trip is the QA subtask's job (FN-1344 — Karate against a seeded dev DB).
- **Test runner fixture**: confirmed `npm test` (`node --test routes/ services/`) picks up the new `services/*.test.js` files automatically — no `package.json` change needed.
