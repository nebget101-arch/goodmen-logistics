### FN-1177 (subtask)

**What changed:**
- Added `backend/gateway/services/explain-forwarder.js` ‚Äî small forwarder that GETs `/api/ai/explain/:token?tenantId=‚Ä¶` from ai-service. Mirrors `ask-forwarder` shape (injected `fetcher` is the test seam, AbortController-based timeout).
- Extended `backend/gateway/routes/ai.js` with `GET /explain/:token`. The route reuses the existing `authenticate(req, jwtSecret)` helper to decode the JWT and pin the upstream call to the JWT's `tenant_id`. Tokens are pre-validated against `^[A-Za-z0-9_-]{16,128}$` (base64url-ish), and malformed tokens short-circuit to a generic 404 without calling ai-service. Wrong-tenant and expired-token 404s from ai-service are passed through verbatim so the two cases are indistinguishable to the caller.
- Wired `explainForwarder` as a required dep in `buildAiRouter` and instantiated it in `backend/gateway/index.js` next to the existing `briefingAggregator` / `askForwarder`.
- Updated existing `ask.test.js` and `briefing.test.js` to inject an `EXPLAIN_FORWARDER_STUB` since `explainForwarder` is now mandatory on `buildAiRouter`.

## Key Decisions
- **Tenant pinning at the gateway, not just downstream.** The gateway extracts `tenant_id` from the JWT and forwards it as a query parameter to ai-service so the upstream check has an authoritative value (not just whatever the caller put in the URL). Defense-in-depth ‚Äî ai-service still owns the actual store lookup and 404-on-mismatch.
- **Generic 404 on malformed tokens, not 400.** Returning 404 for both "token doesn't exist" and "token shape is wrong" prevents probing the token format via status codes.
- **Token regex `^[A-Za-z0-9_-]{16,128}$`.** Matches base64url (the natural shape for opaque tokens minted by ai-service in FN-1176). 16-char floor blocks trivial brute-force; 128-char ceiling blocks accidental URL bombs.
