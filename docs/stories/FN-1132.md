# FN-1132: S7 — Drill-down explanation panel for AI outputs

## Requirement
Any AI-derived value (briefing claim, predictive trend, alert severity) is clickable; clicking opens a side panel that explains *why* — showing source signals, rule traces, and source rows. Improves trust and debuggability.

## Acceptance Criteria
- [ ] AI service returns `explainabilityToken` with each AI output
- [ ] `GET /api/ai/explain/:token` returns rationale (sources, rules, scores)
- [ ] Frontend side panel renders rationale with collapsible sections
- [ ] "View source row" links jump to underlying record
- [ ] Tokens expire after 30 days

### FN-1176 (subtask)

**What changed:**
- Added `backend/microservices/ai-service/src/services/explainability-store.js` — in-memory Map-backed store keyed by `expl_<32hex>` tokens (16 random bytes from `crypto.randomBytes`). Default TTL is 30 days (matches the AC); entries are evicted lazily on `get()` and eagerly via `purgeExpired()`. Mirrors the pattern in `cache/briefing-cache.js`.
- Added `backend/microservices/ai-service/src/handlers/explain-handler.js` — `GET /api/ai/explain/:token`. Returns `400 AI_BAD_TOKEN` for malformed tokens, `404 AI_TOKEN_NOT_FOUND` for unknown/expired tokens, `200` with `{ data, meta: { token, createdAt, expiresAt, processingTimeMs } }` on hit. Logs every lookup via the existing `ai_interaction` analytics logger.
- Added tenant scoping (defense-in-depth for FN-1177 gateway): when `?tenantId=…` is passed and the rationale was minted with a different `tenantId`, returns 404 (indistinguishable from "not found") so token shape can't leak across tenants. The query parameter is optional so direct ai-service callers still work in single-tenant local dev.
- Wired `GET /explain/:token` in `backend/microservices/ai-service/src/ai-router.js` with full OpenAPI/Swagger annotations (visible at `/api-docs`).
- `briefing-handler.js`: after `validateBriefing` succeeds, mint one token per section (`throughput`, `exceptions`, `driverRisk`, `vehicleRisk`, `recommendedAction`) and attach as `explainabilityToken` on each section. Mutation happens *before* the briefing is cached so cached responses keep the same tokens until the briefing TTL expires.
- `score-alert-handler.js`: extracted a `mintSeverityRationale()` helper and attached `meta.explainabilityToken` on every return path (no-AI fallback, unparseable AI, AI success, upstream error). Rationale captures `baseScore` / `boost` / `finalSeverity`, the alert facts snapshot, the `scoredBy` mode, and the model id.

**Tests:** All run with plain `node` (matches existing convention; no jest):

- `backend/microservices/ai-service/src/handlers/__tests__/explain-handler.test.js` — 12 cases including end-to-end mint-via-`briefing-handler` (cached + uncached) and mint-via-`score-alert-handler` (AI path + no-AI fallback) → resolve via `explain-handler`. **Token format:** `expl_<32 hex chars>` (37 chars total). Compatible with the FN-1177 gateway's pre-validation regex `^[A-Za-z0-9_-]{16,128}$`.

## Key Decisions

### Backend (FN-1176 / FN-1177)
- **Tenant pinning at the gateway, not just downstream.** The gateway extracts `tenant_id` from the JWT and forwards it as a query parameter to ai-service so the upstream check has an authoritative value (not just whatever the caller put in the URL). Defense-in-depth — ai-service still owns the actual store lookup and 404-on-mismatch.
- **Generic 404 on malformed tokens, not 400.** Returning 404 for both "token doesn't exist" and "token shape is wrong" prevents probing the token format via status codes.
- **Token regex `^[A-Za-z0-9_-]{16,128}$`.** Matches base64url (the natural shape for opaque tokens minted by ai-service in FN-1176). 16-char floor blocks trivial brute-force; 128-char ceiling blocks accidental URL bombs.
- **In-memory store, not Redis/Postgres.** The ai-service already uses Map-backed in-memory caching (`cache/briefing-cache.js`). Tokens are non-load-bearing — if the process restarts, the explanation panel can show "rationale unavailable" and the user can request a fresh AI output. Adding a persistence layer for explainability would dwarf the implementation cost without proportional benefit. Revisit if multiple ai-service replicas are introduced.
- **Mint tokens after validation, before cache write.** Briefing tokens are mutated onto the validated section objects *before* `briefingCache.set`, so cached briefing reads return identical tokens for the lifetime of the briefing cache entry. Token store TTL (30d) outlives the briefing cache TTL (24h), so the cache eviction path is the binding constraint.
- **Per-section briefing tokens, single severity token.** Each briefing claim (5 sections) is independently explainable, so each gets its own token. Severity is one decision per alert call, so one token per response. Frontend can attach the same `[appAiExplainable]` directive to either with no special-casing.
- **`mintSeverityRationale()` helper to avoid drift.** All four return paths in `score-alert-handler` (no-AI fallback, parse fallback, AI success, upstream error) use the same helper so the rationale shape is identical regardless of how severity was derived. The `scoredBy` field tells the frontend which path produced the score.

### Frontend (FN-1179)
- **Singleton at app root.** Panel is mounted once in `app.component.html` (same pattern as `<app-keyboard-shortcuts-help>` / `<app-command-palette>`). Triggers anywhere in the app open the same panel via the `ExplainService` BehaviorSubject — no per-feature wiring required.
- **Per-token cache in the service** (in-memory, page-session lifetime). Repeated opens of the same explanation are instant. The token itself has a 30-day TTL on the backend, so this never serves stale data within a session.
- **Directive declared in `SharedModule`.** Matches the precedent of `HasPermissionDirective` (FN-771) — making the directive available to lazy-loaded feature modules without each one importing the standalone directive directly.
- **Trigger styles in `styles.css` (not the panel SCSS).** A `[appAiExplainable]` element should look interactive even on routes that haven't yet rendered the panel component (e.g. before first hydration). Putting the rule in component SCSS would have required `::ng-deep`, which is deprecated.
- **No mock fallback for the API.** FN-1176 and FN-1177 ship in the same integration PR — the panel doesn't need to fake a response. Until any caller wires `[appAiExplainable]` onto a real value, the panel is dormant.

## Deployment Handoff

**Services to redeploy:**
- `fleetneuron-ai-service` (FN-1176) — adds explainability store + `GET /api/ai/explain/:token` route; modifies `briefing-handler` (per-section token mint) and `score-alert-handler` (severity token mint). Existing endpoints unchanged.
- `fleetneuron-logistics-gateway` (FN-1177) — adds `services/explain-forwarder.js` + new `/api/ai/explain/:token` route; previous catch-all `/api/ai` proxy still handles other AI paths unchanged.
- `fleetneuron-logistics-ui` (FN-1179) — frontend bundle. Adds the global `<app-explain-panel>` and `[appAiExplainable]` directive. No-op for end users until subsequent stories attach the directive to actual AI values.

**Jobs/workers affected:** None.

**DB migrations:** None — token store is in-memory only.

**Environment/config changes:** None — ai-service uses no new env vars; gateway reuses existing `AI_SERVICE_URL` and `JWT_SECRET`; frontend reads `environment.apiUrl` (existing).

**Cache invalidation:** None server-side. Browser users will get the new bundle from the existing cache-busting interceptor on next reload. Restart of the ai-service drops the in-memory token store; clients holding stale tokens will see 404 from `GET /explain/:token` and the panel renders "this explanation expired" — no migration needed.

**Feature flags:** None.

**Deploy order:** ai-service (FN-1176) **before** gateway (FN-1177) so the upstream endpoint exists when the gateway route goes live; frontend (FN-1179) can deploy any time afterward (it gracefully handles 5xx/404 with a panel error). When shipped from this integration branch in a single PR, deploy order is enforced automatically by Render's per-service triggers.

**Post-deploy verification:**
1. `curl -H "Authorization: Bearer <jwt>" https://fleetneuron-logistics-gateway-dev.onrender.com/api/ai/explain/<bogus-token>` → expect 404.
2. `curl` with no `Authorization` header → expect 401.
3. Trigger a daily briefing (`POST /api/ai/briefing/refresh` or wait for scheduler), grab any section's `explainabilityToken` from the response, then `GET /api/ai/explain/<token>` → expect 200 with rationale JSON whose `data.kind === 'briefing-section'`.
4. Trigger a smart-alert score (`POST /api/ai/score-alert` via the alerts pipeline), grab `meta.explainabilityToken` from the response, then resolve via gateway → expect 200 with `data.kind === 'severity'` and `data.scores.{baseScore,boost,finalSeverity}` populated.
5. **Frontend smoke (after a future story wires up a trigger):** click any `[appAiExplainable]` element → panel slides in with rationale. Esc / backdrop close it. "View source row" link routes to the record and closes the panel.
