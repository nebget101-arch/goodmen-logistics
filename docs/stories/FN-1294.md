# FN-1294: [Bug] Stop card — State select never displays persisted/AI-extracted value (drawer + rate-con auto-approve)

## Requirement

The State `<select>` in `StopCardComponent` (used by both the side-nav Load Edit drawer's Stops tab and the rate-confirmation auto-approve flow) never renders the bound model value — it always shows the `--` placeholder, even when `stop.state` is populated. Same root cause silently breaks the Stop Type select for DELIVERY stops (PICKUP looks fine only because it's the first option, the implicit default).

## Root cause (already diagnosed)

`stop-card.component.html` lines 79–85 (State) and 51–55 (Stop Type) use `<select [value]="...">` whose `<option>` children come from `*ngFor`. Angular sets `select.value` BEFORE the `*ngFor` options are inserted into the DOM, so the browser silently rejects the assignment and the select stays on its first option. City/ZIP work because they're `<input>` elements, which don't have this DOM-ordering dependency.

Both consuming modules (`shared.module.ts`, `loads-dashboard.module.ts`) already import `FormsModule`, so `[ngModel]` is available without any module change.

## Implementation Summary

Swapped the State and Stop Type `<select>` bindings in `stop-card.component.html` from `[value]` (one-way property binding) to `[ngModel]` (FormsModule directive). The `[value]` binding silently fails on a `<select>` whose `<option>` children are produced by `*ngFor` because Angular property bindings settle before the structural directive materializes children, so the browser rejects `select.value = 'GA'` and the control stays on its first option. `[ngModel]` defers the value sync to NgModel's `OnInit` / `OnChanges` lifecycle, after the options exist.

The State select keeps the FN-1089 normalization plumbing — `[ngModel]="normalizedState"` and `(ngModelChange)="onStateChange($event)"` — so full-name AI extractions still resolve to the 2-letter code. The Stop Type select uses `[ngModel]="stop.stop_type"` and `(ngModelChange)="onStopTypeChange($event)"`, no handler signature change.

## Key Decisions

1. **Preserved FN-1089 `normalizedState` getter and `onStateChange` handler** — the Jira ticket described an older baseline (`[value]="stop.state || ''"` + `onFieldChange('state', ...)`), but FN-1089 has since landed on `dev`, replacing those with `normalizedState` / `onStateChange` for full-name → 2-letter coercion. The fix is purely the binding mechanism, so the FN-1089 normalization is preserved verbatim — the swap is `[value]→[ngModel]` and `(change)→(ngModelChange)` only.
2. **No `name=` attribute on either select** — there is no parent `ngForm` in this template; adding `name` would attach the controls to a phantom form group and trigger `NG01203: No value accessor for form control with name`.
3. **Did not refactor to `ReactiveForms`** — explicitly out of scope per the ticket.
4. **Updated existing `TestBed` configs to import `FormsModule`** — required because `[ngModel]` is now in the template that all describe blocks render. Without this, every test in the file would fail to compile the template.
