# FN-1317: reporting-service deploy fails on dev — Cannot find module 'pdfkit' (FN-1167 regression)

## Requirement

Hotfix the `fleetneuron-reporting-service` Render deploy failure on `dev`. The service crash-loops at startup with `MODULE_NOT_FOUND` for `pdfkit`, introduced when FN-1167 added `goodmen-shared/services/pdf-template.js`. Render's `npm install` in the microservice rootDir does not populate `goodmen-shared/node_modules/`, so `require('pdfkit')` from inside the shared package cannot resolve.

## Implementation Summary

Added `pdfkit ^0.16.0` directly to `backend/microservices/reporting-service/package.json` so Node can resolve it at runtime when `goodmen-shared/services/pdf-template.js` (introduced in FN-1167) issues `require('pdfkit')`. `package-lock.json` regenerated with npm 10.

### FN-1318 (subtask)

**Audit of `goodmen-shared/routes/reports.js` require chain:**

| Module required | Source | In reporting-service deps? |
|---|---|---|
| `express` | direct | Yes |
| `pdfkit` (via `services/pdf-template`) | transitive | **No → added in this fix** |
| `axios` (via `services/ai-narrative-client`) | transitive | Yes |
| `jsonwebtoken` (via `middleware/auth-middleware`) | transitive | Yes |
| `knex` (via `internal/db`) | transitive | Yes |

No other transitive package requires from `routes/reports.js` are missing. Common suspects (`@aws-sdk/client-s3`, `pdf-lib`, `pdf-parse`) are declared by `goodmen-shared` but not reachable from `routes/reports.js` — left out of `reporting-service` per the "do not blanket-copy" guidance.

`render.yaml` build commands intentionally untouched (systemic fix tracked separately per ticket notes).

## Files Changed

- `backend/microservices/reporting-service/package.json` — add `pdfkit ^0.16.0` to `dependencies`.
- `backend/microservices/reporting-service/package-lock.json` — regenerated with npm 10. Net diff is +201 lines / 0 deletions: pdfkit + 17 transitive packages (`fontkit`, `linebreak`, `png-js`, `crypto-js`, `pako`, `restructure`, `tiny-inflate`, `unicode-properties`, `unicode-trie`, `dfa`, `brotli`, `clone`, `base64-js`, `jpeg-exif`, `fast-deep-equal`, `tslib`, `@swc/helpers`), plus a stale-catch-up sync of `@goodmen/shared`'s declared deps in the workspace metadata block (otel/cheerio/jimp/stripe/javascript-barcode-reader — already present in `goodmen-shared/package.json`, just not previously reflected here). No top-level reporting-service deps changed.

## Key Decisions

- **Add `pdfkit` to the consumer microservice, not to `goodmen-shared`.** Render's per-service `npm install` runs in the microservice rootDir; `goodmen-shared/node_modules/` is not populated, so requires from inside the shared package fall through to the consumer's `node_modules/`. Declaring at the consumer is the runtime-correct fix.
- **Do not blanket-add other goodmen-shared deps.** Audited the actual require chain reachable from `goodmen-shared/routes/reports.js`; only `pdfkit` was missing. `pdf-lib` / `pdf-parse` / `@aws-sdk/*` are not on this chain.
- **Lockfile regen with npm 10 (lockfileVersion 3).** This service's lockfile was the only one in the repo still on lockfileVersion 1; regenerating brings it in line with all sibling microservices and produces a clean +201/-0 diff. Regenerating with the existing local npm 6 would have downgraded other services' format and produced a 6500-line churn.
- **Do not modify `render.yaml`.** Per ticket notes, the systemic build-command fix is a separate follow-up Story.
