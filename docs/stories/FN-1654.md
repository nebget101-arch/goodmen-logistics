## API Contract (wire shape — FN-1665 must implement; bridges FN-1666 UI ↔ FN-1664 storage)
The frontend consumes a storage-agnostic `/api/geofences` contract. The DB stores
GeoJSON in `jsonb` (FN-1664) with `[lng, lat]` axis order; the API layer (FN-1665)
translates between the `{ lat, lng }` wire objects below and the stored GeoJSON.

**Geofence object (wire)**
```ts
{
  id: string;
  name: string;
  kind: 'circle' | 'polygon';
  center?:  { lat: number; lng: number } | null;  // circle only
  radiusMeters?: number | null;                   // circle only
  vertices?: { lat: number; lng: number }[] | null; // polygon only, ≤ 40
  active?: boolean;
  triggers?: GeofenceTrigger[];
}

GeofenceTrigger {
  id?: string;
  vehicleId?: string | null;   // null/empty = all vehicles
  eventKind: 'enter' | 'exit' | 'dwell';
  dwellMinutes?: number | null; // only for 'dwell'
  action: 'notify' | 'update_load_status' | 'webhook';
  targetUrl?: string | null;    // required when action === 'webhook'
}
```

**Stored geometry (FN-1664, `geofences.geometry` jsonb)**
- circle → `{ "type": "Circle", "center": [lng, lat], "radius_m": <number> }`
- polygon → `{ "type": "Polygon", "coordinates": [[[lng, lat], ...]] }`

**Endpoints**
| Method | Path | Notes |
|--------|------|-------|
| GET | `/api/geofences` | List. Query filters: `active`, `ownedBy`, `near=lng,lat`, `nearRadiusMeters`. Returns `{ data, meta }`. |
| GET | `/api/geofences/:id` | Single geofence. |
| POST | `/api/geofences` | Create. Body = Geofence payload (no `id`). |
| PUT | `/api/geofences/:id` | Update. |
| DELETE | `/api/geofences/:id` | Delete. |

> Note for FN-1665: `near` is encoded `lng,lat` (GeoJSON axis order). Polygon
> vertices are an ordered open ring (UI does not duplicate the first point as the
> closing vertex). The ≤40-vertex limit is enforced client-side and should be
> re-validated server-side. Point-in-polygon / point-in-circle is app-side (no
> PostGIS — see FN-1664 decision).

its shape onto the map for in-place editing.

  save guard.

## Key Decisions
- **Geometry storage = GeoJSON in `jsonb` (NOT PostGIS).** The PostGIS extension is **not** enabled on prod — the only extensions present across the entire migration history are `uuid-ossp` and `pgcrypto` (no `CREATE EXTENSION postgis`, no `geometry()`/`geography()` columns, no GIST indexes anywhere in `goodmen-database`). Introducing PostGIS would add a deploy-coupled extension prerequisite to every environment. Instead the `geometry` column is `jsonb` holding GeoJSON.
  - **`geometry` jsonb shape:**
    - circle → `{ "type": "Circle", "center": [lng, lat], "radius_m": <number> }` (GeoJSON has no native circle, so this is a documented custom shape)
    - polygon → `{ "type": "Polygon", "coordinates": [[[lng, lat], ...]] }` (standard GeoJSON; `[lng, lat]` order per spec)
  - **⚠️ Story C (FN-1655) impact:** point-in-polygon and point-in-circle tests MUST be done **app-side** (e.g. ray-casting for polygons, haversine/great-circle distance ≤ `radius_m` for circles). There is no DB-side `ST_Contains`/`ST_DWithin` available.
- **FK targets:** `geofences.address_id` → `locations.id` (`ON DELETE SET NULL`); there is no `addresses` table — `locations` is the addressable-place table (UUID PK). The FK is wired conditionally (only if `locations` exists) so the migration is portable. `geofence_triggers.geofence_id` → `geofences.id` (`CASCADE`); `geofence_triggers.vehicle_id` → `vehicles.id` (`CASCADE`, NULL = applies to all vehicles).
- **Added columns beyond the literal subtask list:** `is_active BOOLEAN DEFAULT true` (required to satisfy the parent AC `active` CRUD filter) and `updated_at` (conventional). Composite index `(tenant_id, is_active)` supports tenant-scoped active listing; `(created_by)` supports the `owned-by` filter. `near-point` filtering is app-side (no PostGIS).
- **Integrity CHECKs:** `kind ∈ (circle,polygon)`; `event_kind ∈ (enter,exit,dwell)`; `action ∈ (notify,update_load_status,webhook)`; dwell events require `dwell_minutes > 0`; webhook actions require `target_url`.
- **Draw library: `leaflet-draw`** (over leaflet-geoman) — already pairs with the existing `leaflet` dependency, has mature `@types/leaflet-draw`, and ships its own control CSS. Isolated in the lazy `GeofencesModule` to keep it out of the initial bundle.
- **Wire contract uses `{ lat, lng }` objects** (storage-agnostic); FN-1665 translates to/from FN-1664's `[lng, lat]` GeoJSON jsonb so the storage decision stays server-side.
