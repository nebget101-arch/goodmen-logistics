# FN-1455 — FMCSA census + authority imports fail with HTTP 400 from data.transportation.gov

**Type:** Bug · **Severity:** High · **Epic:** [FN-1411](../../docs/stories/FN-1411.md) (FMCSA reference dataset)

## Symptom

Manual + cron runs of the `census` and `authority` files recorded `status='error'` /
`error_message='Request failed with status code 400'` in `fmcsa.import_runs` between
2026-05-07T00:32 and 00:37 UTC. `dryRun: true` runs continued to succeed with 0 rows,
which isolated the failure to the live HTTP fetch (queue, registry, ledger paths
were healthy).

## Root cause

The legacy Socrata "NBE" CSV export endpoint —

* `https://data.transportation.gov/api/views/4a2k-zf79/rows.csv?accessType=DOWNLOAD`
* `https://data.transportation.gov/api/views/6qg9-x4f8/rows.csv?accessType=DOWNLOAD`

— is being progressively deprecated and now returns **HTTP 400** for unauthenticated
bulk pulls. The supported successor is `/resource/{id}.csv` with `$limit/$offset`
paging, and a Socrata **app token** sent as `X-App-Token` to avoid throttling /
rejection.

FN-1413 / FN-1420 were tested locally against manually-downloaded CSVs (`source.filePath`),
so the live URL was never exercised on Render. The 400 only surfaced once the cron
fired in production.

## Implementation Summary

* **`runner.js`** — added a `socrataDataset` source descriptor and an async generator
  `socrataPagedChunks()` that fetches `${baseUrl}/resource/${datasetId}.csv?$limit=N&$offset=K`
  pages in series and concatenates the byte streams into a single `Readable`. The CSV
  header line that Socrata re-emits on every page is stripped from pages 1..N so the
  existing CSV consumer (`createCsvStream` → row Writable) sees one contiguous CSV.
  Termination: a page that returns fewer than `pageSize` rows is the last; a hard
  `SOCRATA_MAX_PAGES = 400` ceiling guards against runaway loops.
* **`buildSocrataHeaders()`** — central helper attaches `User-Agent` to every outbound
  request and `X-App-Token: $FMCSA_SOCRATA_APP_TOKEN` when the env var is set. Used by
  both the new paged-Socrata path and the legacy single-`source.url` path, so any
  caller still passing a `url` automatically picks up the token.
* **`census.js` / `authority.js`** — `DEFAULT_CENSUS_URL` / `DEFAULT_AUTHORITY_URL`
  now resolve to `https://data.transportation.gov/resource/{id}.csv` (the bare resource
  endpoint, useful for token-verification fetches and the new live test). The default
  `source` for `runCensusImport` / `runAuthorityImport` switched to a
  `{ socrataDataset: { baseUrl, datasetId } }` descriptor so the runner takes the
  paged path automatically.
* **`register-importers.js`** — `censusImporterAdapter` / `authorityImporterAdapter`
  drop the explicit `source: { url: DEFAULT_*_URL }` and let the driver default kick
  in. `DEFAULT_CENSUS_URL` / `DEFAULT_AUTHORITY_URL` are no longer required at module
  load time here.
* **`__tests__/live-source.test.js`** (new) — gated behind `FMCSA_LIVE_TEST=true`, hits
  `?$limit=5` against each dataset and asserts HTTP 200 + ≥2 lines (header + ≥1 data
  row). Skipped on default test runs and in CI so the public Socrata endpoint is not
  hammered on every PR.

The parsers (`parsers/census.v1.js`, `parsers/authority.v1.js`) needed no changes —
their header lookup is case-insensitive and already lists every snake_case alias the
Socrata `/resource/{id}.csv` endpoint emits (`DOT_NUMBER`, `LEGAL_NAME`, `MC_NUMBER`,
`AUTHORITY_TYPE`, …).

## Files Changed

* `backend/packages/goodmen-shared/services/fmcsa-importer/runner.js` — paged Socrata source + token-aware headers
* `backend/packages/goodmen-shared/services/fmcsa-importer/census.js` — default to paged Socrata `/resource/{id}.csv`
* `backend/packages/goodmen-shared/services/fmcsa-importer/authority.js` — default to paged Socrata `/resource/{id}.csv`
* `backend/packages/goodmen-shared/services/fmcsa-importer/register-importers.js` — drop explicit URL source from adapters
* `backend/packages/goodmen-shared/services/fmcsa-importer/__tests__/live-source.test.js` — new, `FMCSA_LIVE_TEST`-gated

## Key Decisions

* **Switched unconditionally to `/resource/{id}.csv`, not just on legacy 400.** The AC
  permits a fallback when the legacy URL still 400s with a token. Even with a token
  the legacy `rows.csv?accessType=DOWNLOAD` path is in active deprecation; making the
  choice deterministic (always-paged) avoids a fragile probe-and-fall-back that would
  drift the production code path away from anything tested locally. The legacy URL
  path is still supported via an explicit `source: { url: ... }`, so an operator
  workaround is still possible.
* **Page size 50 000 with hard 400-page cap.** Census is ~2.5M rows, authority ~5M.
  At 50k/page that's ~50 / ~100 page fetches per run. The 400-page cap gives 4× headroom.
* **Header detection by raw byte index, not utf-8 string scan.** Socrata CSV chunks are
  pure ASCII (UTF-8 with no multi-byte in headers), so finding `0x0a` is safe and
  avoids a per-chunk `Buffer.toString()`.
* **Token is opt-in.** If `FMCSA_SOCRATA_APP_TOKEN` is unset the importer still runs;
  it just falls back to anonymous rate limits (which Socrata documents as throttled
  but not blocked outright). Production must set it; local dev can run unauthenticated
  for small samples. Header is omitted entirely when unset rather than sent empty.
* **Live test gated by env, not file location.** Putting the live test in a separate
  `__live__` dir would break the existing `node --test __tests__/` glob. Gating with
  `describe.skip` keeps the file in the normal test directory but no-ops the body.

## Deployment Handoff

* **Service to redeploy:** `fleetneuron-integrations-service` (this is where the FMCSA
  import queue and runners live; see `.agent/docs/render_services.md`).
* **Migrations to run:** none. Schema (`fmcsa.carriers`, `fmcsa.authorities`,
  `fmcsa.import_runs`) is unchanged from FN-1412.
* **Env vars to set on `fleetneuron-integrations-service`:**
  * **`FMCSA_SOCRATA_APP_TOKEN`** *(new, required for production)* — Socrata app token.
    Register a free token at <https://data.transportation.gov/profile/edit/developer_settings>.
    When unset, the importer still runs but is subject to anonymous rate limits and
    may fail on bulk pulls.
  * `REDIS_URL` — unchanged; required by the import queue.
  * `FMCSA_IMPORT_CRON` / `FMCSA_IMPORT_CRON_ENABLED` — unchanged; FN-1424.
* **Cache invalidation:** none.
* **Feature flags:** none.
* **Deploy order:** standalone redeploy of `fleetneuron-integrations-service` once the
  env var is set. No frontend or DB changes.
* **Post-deploy verification:**
  1. Confirm `FMCSA_SOCRATA_APP_TOKEN` is set on Render → fleetneuron-integrations-service → Environment.
  2. As a `super_admin` on the FleetNeuron-internal tenant:
     `POST /api/fmcsa/imports/run { "files": ["census","authority"] }` → expect `202` + `runIds`.
  3. Wait for the runs to complete (5–30 min total depending on dataset volume),
     then `GET /api/fmcsa/imports?limit=5` → confirm both rows show `status='success'`
     with `rows_inserted > 0`.
  4. (Optional) Run the live-source integration test against the dev environment:
     ```
     FMCSA_LIVE_TEST=true FMCSA_SOCRATA_APP_TOKEN=<token> \
       npm test --prefix backend/packages/goodmen-shared -- services/fmcsa-importer/__tests__/live-source.test.js
     ```
* **Rollback:** revert the PR; the previous behavior (legacy `/api/views/...` URL)
  will resume failing with 400 but won't otherwise affect the rest of the service.

## Out of scope

* The `inspections` / `crashes` / `sms` 400 path — those importers correctly gate
  behind `FMCSA_INSPECTION_URL` / `FMCSA_CRASH_URL` / `FMCSA_SMS_URL` env vars, which
  are intentionally unset until snapshot URLs are sourced. Tracked in FN-1422.
* Auto-detection of dataset ID changes — if FMCSA renames `4a2k-zf79` or `6qg9-x4f8`,
  the cron will fail loudly, which is acceptable.
