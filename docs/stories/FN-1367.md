- [x] Each date field (Request Date, Scheduled Date, Start Date, Completion Date) shows a visible label identifying it.
- [x] No regressions in vehicle/customer auto-fill, credit lookup, or VIN decode flows.

## Implementation Summary

Single frontend pass bundling all 5 Basics-tab fixes (matches subtask scope; same files would conflict if split).

- **FN-1369 — FMCSA → Quick Create Customer:** Updated `fetchFmcsadot()` and `lookupFmcsaForInlineCustomer()` to map the actual FMCSA proxy response (`{ name, dot_number, address, city, state, zip, phone, email }` per `backend/packages/goodmen-shared/routes/shop-clients.js:63`). Previous code only read `company.legal_name || company.name` and dropped address/city/state/zip. Added Address / City / State / ZIP / Email inputs to the inline Quick Create Customer form so the populated values are visible & editable. New keys flow into `apiService.createCustomer(this.newCustomer)` payload unchanged.
- **FN-1370 — Server-side customer typeahead:** Replaced in-memory `customers.filter(...)` with an RxJS `Subject<string>` debounced 250ms via `switchMap(query => apiService.getCustomers({ query, pageSize: 50 }))`. Cancels in-flight requests on new input. Quick-create option still rendered at the bottom of the dropdown (template guard `*ngIf="canCreateCustomerInline()"`). Subscription torn down in `ngOnDestroy` via a `destroy$` subject. Display lookup for an already-saved customer (`populateCustomerDisplay`) still uses the parent-loaded `customers[]` for instant render.
- **FN-1372 — Vehicle search scoped to customer:** `getVehicleSearchPool()` now reads `v.customer_id ?? v.customerId` so payloads using either casing match. `loadVehicles()` in `work-order.component.ts` normalizes each vehicle to expose `customer_id` regardless of incoming casing, ensuring the filter works.
- **FN-1374 — Shop Location dropdown empty:** `loadLocations()` was assigning the raw response object to `this.locations`; the legacy `/locations` route returns an array but defensive shape unwrapping (`data?.data || data?.rows || data || []`) brings it in line with the audited loaders (`loadCustomers`, `loadParts`, `loadTechnicians`).
- **FN-1376 — Date field labels:** Chose option (b) — render an external `<label for="…">` above each `<app-ai-date-picker>` and pass `ariaLabel` instead of the internal `label` input. The picker’s template already supports the “no label” mode via the `ai-date-picker-field--external-label` host class (`shared/ai-date-picker/ai-date-picker.component.html:4`), so this changes only the parent template — no SCSS sweep required and no risk to other usages of `<app-ai-date-picker label="…">`.

## Key Decisions
- **Date label fix: external `<label>` (option b), not SCSS.** Scoped to one component, matches every other field on the tab, and avoids a global mat-label color audit. Picker already supports a label-less mode.
- **FMCSA proxy response mapping uses `name` first.** The shop-clients FMCSA proxy returns `name` (mapped from `legalName || dbaName`), not `legal_name`. Kept `legal_name` as a defensive fallback in case any caller still hits a legacy proxy.
- **Customer typeahead leaves `customers[]` (parent-loaded) intact for display.** The 5000-row preload still drives `populateCustomerDisplay()` on existing-WO load — only the *search* path is now server-side. This keeps load-time WO display instant and preserves the `vehicle.customer_id → customer name` lookup in `onVehicleSelect()`.
- **Vehicle field normalization happens at load time, not in the filter.** Doing it once in `loadVehicles()` means downstream consumers (filter, `onVehicleSelect`, `applyCustomerVehicleFilter`) keep reading the canonical `customer_id` only.
