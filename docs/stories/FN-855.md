# FN-855: Loads grid Actions/hover-toolbar missing on some rows

## Requirement
As a dispatcher, every row on the loads dashboard should reveal the hover toolbar (View, Edit, Clone, Map, Dispatch, Delete-if-draft) on row hover. The FN-787 implementation introduced the toolbar but it is missing on some rows, leaving those loads without quick actions.

## Acceptance Criteria
- [ ] Hover toolbar fades in within 100ms on every row regardless of status (draft, new, dispatched, in_transit, delivered).
- [ ] View, Edit, Map, Dispatch present on every row.
- [ ] Clone gated by `loads.create` permission; Delete gated to draft-only.
- [ ] Toolbar does not visually conflict with Status/Billing inline dropdowns (FN-854).
- [ ] No console errors when hovering across all rows.

## Implementation Summary

**Root cause.** `frontend/src/app/components/loads-dashboard/loads-dashboard.component.scss` revealed the toolbar with a single `tr:hover .row-toolbar` rule. Three unrelated features silently broke this trigger:

1. **Portaled hover-popovers steal the pointer (FN-806).** Driver/Broker/Notes/Attachments cells wrap in `<app-hover-popover>`, whose popover content is portaled to `document.body`. Once the user's cursor enters the popover, it is no longer a descendant of the `<tr>`, so `tr:hover` becomes `false` and the toolbar fades back to `opacity: 0`.
2. **Detail drawer active row has no hover (FN-808).** With the drawer open, the user's pointer typically sits in the drawer panel; `tr.row-drawer-active` already gets a teal left-border but had no rule to keep its toolbar visible, so quick actions vanished for the active row.
3. **Keyboard focus without hover.** `:focus-within` was scoped only to `.row-toolbar` itself, so focusing any other cell in the row (status trigger, notes editor) left the toolbar hidden.

The reported "CONF pill only, no other actions" matches scenarios (1) and (2) — the status cell's content remained while the actions cell faded.

**Fix.** Broadened the reveal selectors and reserved cell space so permission-gated buttons don't shrink the cell:

```scss
.table--body tr:hover .row-toolbar,
.table--body tr:focus-within .row-toolbar,
.table--body tr.row-drawer-active .row-toolbar,
.row-toolbar:hover,
.row-toolbar:focus-within {
  opacity: 1;
  transform: translateY(0);
  pointer-events: auto;
}
```

Permission guards (`*appHasPermission="'loads.create'"` on Clone, `*ngIf="canDeleteLoad(load)"` on Delete) are intentional per AC and were not changed.

## Files Changed

- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.scss` — broadened row-toolbar reveal triggers (added `:focus-within`, `.row-drawer-active`, `.row-toolbar:hover`); reserved cell height with `min-height: 28px` / `min-width: 0`.

## Key Decisions

- **CSS-only fix.** No JS/state change — the symptoms are entirely in the reveal trigger. Adding a JS mouseenter listener would add change-detection churn on every scroll for no extra reliability.
- **Kept opacity-based fade.** The 100ms fade is part of the AC and matches the FN-804 hover-popover timing; switching to `display: none` / `visibility` would break the transition.
- **Did not delete dead FN-773/FN-775 `.actions-dropdown` SCSS.** That class is unused since FN-805 replaced the ⋮ menu with the hover toolbar, but removing it is out of scope for this bug; flagged separately in tech-debt thinking.
