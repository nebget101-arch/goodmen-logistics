# FN-1124: S1 — Daily AI Briefing on Control Center

## Requirement
On `/dashboard` load, surface a Daily AI Briefing card summarizing fleet state for the current day: load throughput, exception count, top driver risk, top vehicle risk, and one recommended next action.

## Acceptance Criteria
- [ ] Briefing renders within 2s on dashboard load (cached) or 6s (cold)
- [ ] Includes 5 sections: throughput, exceptions, driver risk, vehicle risk, recommended action
- [ ] AI service caches briefing per tenant per day; manual refresh button invalidates cache
- [ ] Backend endpoint `GET /api/ai/briefing` aggregates upstream data and forwards to ai-service
- [ ] Frontend card has loading skeleton, error fallback, and a11y labels

## Implementation Summary
- **FN-1139 (ai)**: New `POST /api/ai/briefing/generate` route in ai-service. Validates upstream metrics, calls Claude Sonnet 4.6 with a markdown-defined system prompt, validates the model's five-section response, and caches the result per (tenantId, date) for 24h. Returns typed 502 codes on AI failures. 16 unit tests, mocked Anthropic client.
- **FN-1141 (backend)**: Gateway-local `GET /api/ai/briefing` aggregator that fans out to logistics/drivers/vehicles services in parallel, forwards aggregated `fleetState` to ai-service, and caches the response per tenant per day with `?refresh=true` invalidation.

## Key Decisions
- **Markdown prompt file vs. inline string**: System prompt lives in `src/prompts/briefing.md` (loaded once per process). Lets non-engineers iterate on prompt wording in PRs without touching JS, and matches the path the original ticket description called out. The other handlers inline their prompts as JS strings — those are simpler, but the briefing prompt is significantly longer and more likely to be tuned post-launch.
- **Model**: `claude-sonnet-4-6` for the generator (Sonnet handles structured 5-section output reliably). Made overridable via `ANTHROPIC_BRIEFING_MODEL` so we can pin a snapshot in prod or fall back to Haiku for cost reduction without a code change.
- **Strict schema validation, fail-loud with 502**: When the model returns malformed JSON or misses a required section, return 502 with a typed code (`AI_PARSE_ERROR`, `AI_SCHEMA_ERROR`) rather than papering over with a partial response. The frontend (FN-1143) will surface a typed error state — better UX than rendering "Unknown" headlines that look like real briefing content.
- **24h TTL on the ai-service cache**: Briefing is per-day so the upper bound is the calendar day. We chose 24h rather than aligning to midnight UTC to avoid mass-invalidation storms; the gateway invalidates per-tenant when the user explicitly refreshes.
- **In-memory cache, not Redis**: ai-service is single-instance on Render dev; for multi-instance prod we may need Redis, but doubling the storage backend just for a per-tenant per-day briefing felt premature. Documented in deployment handoff.

## Subtask Implementation Notes

### FN-1139 (subtask)
- New handler `backend/microservices/ai-service/src/handlers/briefing-handler.js` exposes `POST /api/ai/briefing/generate`. The handler:
  - Validates the request shape (`tenantId` required string; `date` required `YYYY-MM-DD`; `metrics` optional object; `forceRefresh` optional boolean) and short-circuits the cache before calling Anthropic.
  - Loads the system prompt from `backend/microservices/ai-service/src/prompts/briefing.md` (cached in-process after first read) so prompt edits do not require a code change.
  - Calls Anthropic Claude Sonnet 4.6 (`claude-sonnet-4-6`, configurable via `ANTHROPIC_BRIEFING_MODEL`) with `temperature: 0.2`, `max_tokens: 1024`, system + a single user message that JSON-stringifies `{ tenantId, date, metrics }`.
  - Validates the model output against a strict five-section schema (`throughput`, `exceptions`, `driverRisk`, `vehicleRisk`, `recommendedAction`, each `{ headline ≤ 60, detail ≤ 200, metric ≤ 30 }`); returns 502 with typed codes (`AI_PARSE_ERROR`, `AI_SCHEMA_ERROR`, `AI_UNAVAILABLE`) on failure so the frontend can render typed fallbacks.
- Cache module `backend/microservices/ai-service/src/cache/briefing-cache.js` is a simple in-memory `Map` keyed on `tenantId::date` with a 24h TTL. Exposes `get/set/invalidate/clearAll/size`. The gateway (FN-1141) decides when to bypass via `forceRefresh: true`; the cache also lazily expires on `get` if the TTL passed.
- Tests `backend/microservices/ai-service/src/handlers/__tests__/briefing-handler.test.js` follow the loads-nlq pattern — plain `node`, no jest. The Anthropic client is mocked via `deps.anthropic`. 16 cases covering: happy path, cache hit on second call, `forceRefresh` bypass, missing tenantId / missing date / bad date format / non-object metrics → 400, unparseable model output / missing required section / upstream error → 502, validator behavior (trim, empty-metric rule, fenced JSON), and TTL expiry triggering regeneration.
- Anthropic SDK was already a dependency (used by `loads-nlq`, `fmcsa-driver-match`, `toll-invoice-vision`, `psp-report-vision`); no new dependencies. `package.json` `test` script chained to run both handler test files.

### FN-1141 (subtask)
- New module `backend/gateway/services/briefing-aggregator.js` does the parallel fan-out via `Promise.allSettled` (so a single upstream failure does not break the briefing) and forwards aggregated `fleetState` to `${AI_SERVICE_URL}/api/ai/briefing/generate`.
- New router `backend/gateway/routes/ai.js` exposes `GET /briefing`, mounted at `/api/ai` **before** the catch-all `/api/ai` proxy in `index.js`. JWT is verified locally on the gateway (using `JWT_SECRET`) so the briefing route can extract `tenant_id` and apply per-tenant caching without a downstream round-trip.
- Caching is gateway-side, in-memory, 12h TTL, keyed `tenantId:YYYY-MM-DD`. `?refresh=true` invalidates the gateway cache **and** sets `forceRefresh: true` in the body sent to ai-service so its own per-day cache is bypassed too.
- Tests run with plain Node (`node backend/gateway/__tests__/briefing.test.js`) — same convention as `ai-service/src/handlers/__tests__/loads-nlq-handler.test.js`. Cases: unauthorized (no/invalid/missing-tenant token), happy path, refresh invalidation, partial upstream failure (sources rejected with `Promise.allSettled` are tracked in `upstreamErrors[]` and forwarded as `null` in `fleetState`).
