# FN-894: Scaffold Karate Maven multi-module project (karate-core + karate-gateway)

## Requirement
Create the Maven multi-module skeleton at `tests/karate/` so future microservice modules can be added additively with zero rework.

## Scope
- Parent `pom.xml` at `tests/karate/` declaring shared dependency versions (Karate, JUnit5, Allure, etc.)
- Module `karate-core/` (shared config, hooks, helpers — empty `src/main/java` skeleton, runner test class)
- Module `karate-gateway/` (Phase 1 features will live here — empty `src/test/java/gateway/` skeleton)
- Top-level `tests/karate/README.md` documenting the additive-module pattern (e.g. `karate-auth-users/`, `karate-logistics/`) without scaffolding them
- `.gitignore` for `target/`, `karate-reports/`, IDE folders

## Acceptance Criteria
- [ ] `mvn -pl karate-gateway -am test-compile` succeeds locally
- [ ] Module dependency: `karate-gateway` depends on `karate-core` test-jar
- [ ] README documents how to add a new module (copy-paste pattern)
- [ ] `.gitignore` excludes `target/` and report directories
- [ ] No code in `karate-gateway/src/test/java/gateway/` yet beyond a placeholder runner

### FN-911 (subtask)
**Branch**: `qa/FN-911/readme-gitignore`

**What shipped**
- `tests/karate/README.md` — top-level orientation doc covering:
  - Multi-module layout diagram showing `karate-core/` (shared, test-jar producer) and `karate-gateway/` (Phase 1 consumer), with named placeholder folders for `_smoke / _contract / routing / security / resiliency / negative` so the reader sees exactly where new features land.
  - **Adding a new module** copy-paste recipe (uses `karate-auth-users` as a worked example): create the directory, copy `karate-gateway/pom.xml`, swap `<artifactId>` + `<name>`, append the module to the parent `<modules>` list, drop features under `src/test/java/<module>/`, run `mvn -pl <new-module> -am test-compile` to verify.
  - **Running tests** — `mvn -pl karate-gateway -am test-compile` (sanity), `mvn -pl … -am test`, single-feature via `-Dkarate.options="classpath:gateway/_smoke/health.feature"`, tag selection (`--tags @smoke`), full-suite `mvn test`. Explains the `-am` flag.
  - **Environment switching** — `karate.env` system property (`local` default, `dev`, `ci`); the parent pom already wires it through Surefire so the README documents the contract that `karate-config.js` (FN-909 scope) needs to honor.
  - **Reports** — `target/karate-reports/` (HTML), `target/surefire-reports/` (JUnit XML for CI), `target/allure-results/`; includes the `allure serve …` command for local rendering.
  - **Conventions** — feature placement next to runners, suggested tag taxonomy (`@smoke`, `@contract`, `@negative`), keep env-specific values in `karate-config.js`, shared helpers belong in `karate-core`.
  - **Requirements** — Java 17, Maven 3.9+, optional Allure CLI.
- `tests/karate/.gitignore` — excludes Maven build output (`target/`, `**/target/`), all four flavors of report directories (`karate-reports`, `allure-results`, `allure-report`, `surefire-reports`, `cucumber-html-reports` — both top-level and nested), IDE artifacts (`.idea/`, `*.iml`, IntelliJ project files, `.vscode/*` while allowlisting `settings.json` / `extensions.json` / `launch.json` so a shared workspace config can ship later, plus Eclipse `.project` / `.classpath` / `.settings/`), OS junk (`.DS_Store`, `Thumbs.db`), and local env overrides (`*.local`, `.env.local`).
- Scoped to docs + ignore rules only — no Maven changes (those belong to FN-908 / FN-909 / FN-910). The README references the `karate.env` / `karate.options` system properties that are already wired up in the FN-908 parent pom.
