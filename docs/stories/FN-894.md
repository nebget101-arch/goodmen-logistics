# FN-894 — Scaffold Karate Maven multi-module project (karate-core + karate-gateway)

**Epic**: [FN-893 — Enterprise Karate API testing framework — gateway phase](https://nebget101.atlassian.net/browse/FN-893)
**Type**: Story
**Agent**: qa
**Labels**: `agent:qa`, `framework:karate`, `service:gateway`

## Requirement

Create the Maven multi-module skeleton at `tests/karate/` so future microservice modules can be added additively with zero rework.

## Scope

- Parent `pom.xml` at `tests/karate/` declaring shared dependency versions (Karate, JUnit5, Allure, etc.)
- Module `karate-core/` — shared config, hooks, helpers (empty `src/main/java` skeleton + runner test class)
- Module `karate-gateway/` — Phase 1 features will live here (empty `src/test/java/gateway/` skeleton)
- Top-level `tests/karate/README.md` documenting the additive-module pattern (e.g. `karate-auth-users/`, `karate-logistics/`) without scaffolding them
- `.gitignore` for `target/`, `karate-reports/`, IDE folders

## Acceptance Criteria

- [ ] `mvn -pl karate-gateway -am test-compile` succeeds locally
- [ ] Module dependency: `karate-gateway` depends on `karate-core` test-jar
- [ ] README documents how to add a new module (copy-paste pattern)
- [ ] `.gitignore` excludes `target/` and report directories
- [ ] No code in `karate-gateway/src/test/java/gateway/` yet beyond a placeholder runner

## Implementation Notes

### FN-908 (subtask) — Parent pom.xml

**Files created**

- `tests/karate/pom.xml` — Maven parent POM (`<packaging>pom</packaging>`)

**Decisions**

- **Group/artifact**: `ai.fleetneuron.tests:karate-parent:1.0.0-SNAPSHOT`. Group prefix matches the FleetNeuron domain so future modules namespace cleanly.
- **Java version**: 17 (matches Karate 1.4.x baseline and FleetNeuron Render runtime).
- **Modules listed**: `karate-core`, `karate-gateway`. The module directories themselves are scaffolded by FN-909/FN-910 — until those subtasks land, `mvn` invocations against this parent will fail to resolve children. That's expected during scaffolding; the story-level acceptance check (`mvn -pl karate-gateway -am test-compile`) is verified by FN-912 after all sibling subtasks complete.
- **Pinned versions** (declared in `<dependencyManagement>` so child modules inherit without restating):
  - `karate-junit5` — `1.4.1` (current stable Karate line, JUnit5 runner)
  - `junit-jupiter-engine` / `junit-jupiter-api` — `5.10.2`
  - `allure-cucumber7-jvm` — `2.27.0` (Karate uses Cucumber 7 internals; this is the matching Allure adapter)
  - `jackson-databind` — `2.17.1`
- **Plugin management**:
  - `maven-compiler-plugin` `3.13.0` — Java 17 source/target
  - `maven-surefire-plugin` `3.2.5` — supports JUnit Jupiter natively, includes `*Runner.java` patterns so Karate runners are picked up; threads `argLine` of `-Xmx1024m`; passes `karate.options` and `karate.env` system properties through
  - `maven-resources-plugin` `3.3.1`
- **Default profile** — declares empty `karate.options` and `karate.env` so child modules can override via CLI without `unresolved property` errors during inheritance.
- All Karate/JUnit/Allure dependencies declared at `<scope>test</scope>`. `jackson-databind` left at default scope for shared helpers in `karate-core`'s main source set.

**Out of scope for this subtask**: child module POMs (FN-909/FN-910), `karate-config.js` (covered by separate story), CI workflow (separate story), README and `.gitignore` (FN-911).

### FN-909 (subtask) — karate-core module skeleton + pom

**Files created**

- `tests/karate/karate-core/pom.xml` — child POM, packaging `jar`, parent `karate-parent`
- `tests/karate/karate-core/src/main/java/.gitkeep` — placeholder so the empty source set is committed
- `tests/karate/karate-core/src/test/java/core/CoreRunner.java` — placeholder class in package `core`

**Decisions**

- **Packaging**: `jar`. Per the story's acceptance criterion that `karate-gateway` depends on `karate-core`'s test-jar, the `maven-jar-plugin` is wired with a `test-jar` execution so a separate `karate-core-1.0.0-SNAPSHOT-tests.jar` is produced alongside the main jar. Karate-gateway will declare a `<type>test-jar</type>` dependency on this in FN-910.
- **Inherited deps**: `karate-junit5`, `junit-jupiter-engine`/`api`, `allure-cucumber7-jvm`, `jackson-databind` are referenced WITHOUT versions — versions resolve from the parent's `<dependencyManagement>`. This keeps version pinning single-sourced.
- **CoreRunner**: Concrete `public class CoreRunner` in package `core` (matches the subtask's directory spec `src/test/java/core/`). No `@Karate.Test` annotation yet — there are no feature files to drive. Surefire will scan it (per the parent's `*Runner.java` include) but find no test methods, which is a no-op rather than a failure under JUnit Jupiter.
- **`.gitkeep` in `src/main/java`**: empty file so the otherwise-empty directory is tracked. Future shared helpers (e.g. base spec classes, JSON utilities) will land here.
- **No `src/test/resources/`**: deliberately deferred. `karate-config.js` is owned by a later story in this epic.

**Out of scope**: `karate-gateway` module (FN-910), `karate-config.js`, README/`.gitignore` (FN-911), validation run (FN-912).

### FN-910 (subtask) — karate-gateway module skeleton

**Files created**

- `tests/karate/karate-gateway/pom.xml` — child module POM (`<packaging>jar</packaging>`)
- `tests/karate/karate-gateway/src/test/java/gateway/GatewayParallelTest.java` — placeholder Karate parallel runner
- Six package folders under `src/test/java/gateway/` with `.gitkeep` placeholders so git tracks them while empty:
  - `_smoke/`, `_contract/`, `routing/`, `security/`, `resiliency/`, `negative/`

**Decisions**

- **Parent inheritance**: `<parent>` points to `karate-parent` via `relativePath=../pom.xml`. All shared versions for `karate-junit5`, `junit-jupiter-*`, `allure-cucumber7-jvm`, and `jackson-databind` are inherited from the parent's `<dependencyManagement>` — the child only declares `<groupId>/<artifactId>` so version drift is impossible.
- **karate-core test-jar dependency**: declared as `<type>test-jar</type>` `<scope>test</scope>`. This is the gateway module's only intra-project dependency and is exactly what the story acceptance criterion calls out. The corresponding `test-jar` is produced by FN-909's `karate-core` module via the `maven-jar-plugin` test-jar goal (FN-909 work).
- **Runner stub** `gateway.GatewayParallelTest` uses `Runner.path("classpath:gateway").tags("~@ignore").parallel(5)`. With no feature files present, Karate returns zero features and zero failures, so `assertEquals(0, results.getFailCount())` holds — the runner serves as a smoke check that the module wires up and surefire picks up `*Test.java` patterns. Threads default to 5 to match the Phase 1 epic's parallel target; `@ignore` is the only tag exclusion so `@smoke`, `@contract`, etc. all execute by default until tag-based runners arrive in later stories.
- **Test resources**: declared `src/test/java` and `src/test/resources` as `<testResource>` directories so `.feature` files placed alongside `.java` runners (Karate's recommended layout) are copied to `target/test-classes/`. `.java` files are explicitly excluded from the resource copy so they aren't double-handled by the resources plugin.
- **No source main folder** — gateway module is test-only; runtime helpers live in `karate-core`. Maven defaults `src/main/java` and `src/main/resources` are simply absent.
- **Folder naming preserved**: `_smoke` and `_contract` use the leading-underscore convention from the epic (FN-893) so they sort to the top of file listings and read as cross-cutting layers separate from the by-owner `routing/`, `security/`, `resiliency/`, `negative/` buckets.

**Out of scope for this subtask**: actual `.feature` files (later FN-893 stories), `karate-config.js` (separate story), per-folder/per-tag runners, CI workflow, README/`.gitignore` (FN-911), test-compile validation (FN-912).
