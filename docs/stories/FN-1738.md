## Implementation Summary
Built the Branding settings page as a single NgModule-declared component (`BrandingSettingsComponent`) routed at `admin/branding`. It renders two groups â€” Carriers (operating entities) and Shop locations â€” each row showing the current logo preview with upload / replace / remove. Upload accepts file-picker or drag-drop, validates type/size client-side, shows per-row loading + error states, and refreshes the preview from the GET endpoint after every upload/delete. The component is built against the documented FN-1737 API contract; backend endpoints land in FN-1737/FN-1742.

### FN-1744 (subtask)
- **API methods** added to `frontend/src/app/services/api.service.ts` honoring the FN-1737 contract â€” `get/upload/deleteOperatingEntityLogo` and `get/upload/deleteLocationLogo`. Upload uses the same `FormData().append('file', file)` pattern as `uploadDQFDocument`.
- **Lists** sourced from existing endpoints: carriers via `listOperatingEntities()` (`{ data: [...] }`, fields `dba_name/legal_name/name`, `mc_number`, `dot_number`); locations via `getLocations()` (`{ data: [...] }` or array, fields `name`, `code`, `city`, `state`).
- **Per-row state model** (`BrandingRow`): each MC/location carries its own `logoUrl/mimeType/uploadedAt/loading/error/dragOver`, so rows upload and fail independently.
- **Validation**: MIME must be one of `image/png|jpeg|webp`, size â‰¤ 2MB; rejects render an inline error on the row without hitting the network. The same constraints feed the `<input accept>` attr and the hint text.
- **Preview refresh**: after a successful POST or DELETE, the row re-fetches via the GET endpoint so the preview reflects the canonical (signed) URL rather than trusting the mutation response.
- **Empty state**: dashed placeholder tile + "Upload logo" CTA when no logo is set; the button label flips to "Replace" when one exists, and "Remove" is guarded by a confirm.
- **Styling**: `.scss` using the app's AI dark-theme tokens (`--ai-surface-*`, `--ai-text-*`, `--ai-accent-*`, `--sev-critical-*`); responsive stacking under 560px.
- **Registration**: declared in `app.module.ts`; route `admin/branding` added to `app-routing.module.ts` guarded by `AuthGuard + PermissionGuard` (`ACCESS_ADMIN | ROLES_MANAGE | LOCATIONS_MANAGE`).

## Key Decisions
- **Path**: used the ticket-specified `frontend/src/app/settings/branding/` location (a new `settings/` area) rather than `components/admin/`, per the subtask's explicit file list.
- **Single component, no child component**: both lists share one reusable `<ng-template #logoRow>` instantiated per row â€” avoids a second declared component for identical markup.
- **GET-after-mutation**: trust the GET for preview state (matches the contract note "preview refreshes from GET after success") instead of optimistically using the POST response, so signed-URL TTL behavior is consistent.
- **Permissions**: reused existing admin permissions; no new permission constant introduced.
