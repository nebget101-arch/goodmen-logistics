# FN-1570: [Locations Admin] Bins tab returns 404 — location-bins router never mounted in logistics-service

## Requirement
The Edit Location modal's **Bins** tab fails to load on dev with "Failed to load bins." The XHR `GET /api/locations/:locationId/bins` returns a 404 from the logistics microservice. The router and database table both exist (delivered in FN-687 + FN-692) but the router was never wired into `logistics-service/server.js`. Mount it.

## Implementation Summary

### FN-1572 (subtask)
Mounted the existing `location-bins` router in the logistics microservice. Added the `require()` alongside the other shared-route imports (after `locationsRouter`) and added `app.use('/api/locations/:locationId/bins', authMiddleware, tenantContextMiddleware, locationBinsRouter)` next to the other `/api/locations` mount, so the bins routes inherit the same auth + tenant-context middleware chain. `mergeParams: true` is already set on the router, so handlers see `req.params.locationId`.

## Files Changed
- `backend/microservices/logistics-service/server.js` (+2 lines: one require, one mount)

## Key Decisions
- **Mount path is `/api/locations/:locationId/bins`** (not nested into `locationsRouter`). This matches what the frontend already calls (`api.service.ts`) and what the gateway forwards (`backend/gateway/index.js:489`). It also avoids modifying `locations.js` (out of scope).
- **Mount order is after the broader `/api/locations` mount.** Express routes both as `app.use`, but `locationsRouter` only defines `/`, `/:id`, etc. — none match `/:locationId/bins`, so requests fall through to the bins mount. Verified by inspection of `routes/locations.js`.
- **Same middleware chain as `locationsRouter`** (`authMiddleware` + `tenantContextMiddleware`) — bins are tenant-scoped and the router already reads `req.context?.tenantId`.
