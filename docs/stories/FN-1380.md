## Implementation Summary

Single-component refactor of `VehiclesComponent` (the shared truck/trailer list). Highlights:

- **Ownership classification** is derived client-side from existing API fields. A new `getOwnership(vehicle)` helper returns one of `'company' | 'oo' | 'leased'`:
  - `leased` if `trailer_details.ownership === 'leased'` (covers leased trailers per FN-1351's trailer schema)
  - `oo` if `company_owned === false` (covers owner-operator trucks/trailers; `equipment_owner_name` shown as a sub-line)
  - `company` otherwise (default — pre-existing behavior was to hide non-company rows entirely)
- **Filter chip-set** added above the table: `All | Company | OO | Leased`. Buttons follow the FN-798 `smart-filter-chip` pattern (rounded pill, teal active state) but are scoped to `.vehicles-dashboard` to avoid global style leakage. Chips have `aria-pressed` + `role="group"`.
- **Ownership column** inserted between Vehicle Details and VIN. Cell renders a `chip-ownership chip-ownership--{class}` badge plus an optional `equipment_owner_name` sub-line for OO units.
- **Removed the `vehicle.company_owned !== false` filter in `applyFiltersAndSort`** — that one-liner was hiding all OO/leased rows from both lists. Now controlled by the ownership chip-set.
- **Trailer-row data parity** in the Vehicle Details cell: when `vehicleType === 'trailer'`, the cell shows `{code} — {label}` from `trailer_details` (e.g. `R — Reefer`), the existing license-plate (state) line, and an italic "Driver: {first} {last}" sub-line if a driver is currently linked via `drivers.trailer_id`. Trucks keep the original `{year} {make} {model}` line.
- **Driver name lookup**: `loadVehicles()` now uses `forkJoin({ vehicles, drivers })` to load both endpoints in parallel and builds an in-memory `Map<trailerId, "First Last">` indexed once on load. No backend change required.
- **N/A → em-dash**: Mileage / Inspection / Next PM cells show `—` (with column units preserved) when the field is blank, matching FN-1351 conventions.
- **Density polish + sticky header**: `td` padding tightened from 14px → 10px and `line-height: 1.35`. `<thead>` is now `position: sticky; top: 0` inside a bounded `.table-responsive` (`max-height: calc(100vh - 360px)`); the mobile card layout (`@media max-width: 768px`) overrides this back to natural scroll.
- **No new hex** — LEASED chip reuses the existing blue (`rgba(59, 130, 246, *)`) already used for cards/buttons; COMPANY uses the teal (`rgba(20, 184, 166, *)`) and OO uses the amber (`rgba(245, 158, 11, *)`) already in the file.

## Key Decisions
- **No backend changes.** `equipment_owner_name`, `equipment_owner_id`, `company_owned`, and `trailer_details` (JSONB) are already in the GET `/vehicles` response (`SELECT av.*`). Verified by reading `backend/packages/goodmen-shared/routes/vehicles.js` and the `20260329100000_settlement_v2_schema.js` migration.
- **Driver lookup via forkJoin (extra GET /drivers call)** rather than adding a server-side join. Keeps the change scoped to the frontend and matches the `dispatch-drivers.component.ts` pattern that does similar client-side equipment indexing. The drivers endpoint is already cached and small.
- **LEASED uses blue, not purple.** The design called for purple but the AI dark-theme palette has no purple token; introducing one would violate the "no new hex" constraint. Blue is visually distinct from teal (COMPANY) and amber (OO) and is consistent with leasing/financial accents elsewhere in the UI.
- **Sticky header bound by viewport-relative max-height** (`calc(100vh - 360px)`) rather than a fixed pixel value, so it adapts to varying header/controls heights and stays usable on tall screens. The mobile card layout disables it.
- **Status filter uses the existing `app-ai-select` dropdown** (kept). Ownership uses chips (added) because the four options benefit from one-click toggle access — chips also keep the redesign visually distinct from the legacy controls row.
