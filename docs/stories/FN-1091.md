# FN-1091: Manufacturers & Vendors master tables (foundation)

## Requirement
Replace free-text `parts.manufacturer` and `parts.preferred_vendor_name` with normalized `manufacturers` / `vendors` master tables so dedup is enforceable. Foundation for the Quick Add Part epic (FN-1090) — Stories 2–5 depend on this.

## Acceptance Criteria
- [ ] `manufacturers` and `vendors` tables created with unique normalized names + indexes.
- [ ] Nullable `manufacturer_id` / `vendor_id` FKs added to `parts`.
- [ ] Backfill populates master tables and updates parts FKs.
- [ ] Legacy text columns (`parts.manufacturer`, `parts.preferred_vendor_name`) preserved and kept in sync on write.
- [ ] CRUD + autocomplete endpoints: `/api/manufacturers`, `/api/vendors`.
- [ ] Add/Edit Part form replaces free-text inputs with typeaheads + inline "Create new".
- [ ] QA validates dedup, autocomplete latency, and backfill.

## Implementation Summary

### FN-1092 (subtask)
Foundation schema for the Quick Add Part epic. Two Knex migrations land on `integration/FN-1091`:

1. `20260505100000_add_manufacturers_vendors_tables.js` — creates `manufacturers` and `vendors` master tables with `bigserial id`, `name UNIQUE`, `normalized_name UNIQUE`, and timestamps. `vendors` adds optional `contact_email` / `contact_phone`. Adds nullable `manufacturer_id` / `vendor_id` BIGINT FKs to `parts` (`ON DELETE SET NULL`) plus btree indexes on each FK column.
2. `20260505100100_backfill_manufacturers_vendors.js` — inserts distinct, case-insensitive, whitespace-collapsed values from `parts.manufacturer` and `parts.preferred_vendor_name` into the new master tables, then updates each part's FK to its matching master row. Uses `LOWER(BTRIM(REGEXP_REPLACE(value, '\s+', ' ', 'g')))` for normalization — the backend autocomplete (FN-1093) must use the identical expression. NULL/empty source values are skipped, leaving the FK NULL.

Free-text columns `parts.manufacturer` and `parts.preferred_vendor_name` are intentionally preserved; FN-1093 will keep them in sync on write until callers fully migrate.

End-to-end verified locally: `migrate:latest` → schema correct, dedup collapses three case/whitespace variants of "ACME Corp" to one row; `migrate:rollback --all` → tables and FK columns dropped, free-text columns intact; idempotent re-run after rollback succeeds.

### FN-1093 (subtask)
Express routes + services for manufacturers and vendors CRUD and autocomplete, plus parts write-path FK/text sync.

- **Shared normalization helper** `backend/packages/goodmen-shared/utils/normalize-name.js`: lowercases, trims, collapses whitespace. Output is byte-for-byte identical to the SQL `LOWER(BTRIM(REGEXP_REPLACE(v, '\s+', ' ', 'g')))` used by the FN-1092 backfill, so autocomplete lookups against `normalized_name` find rows the backfill wrote.
- **Services** (`manufacturers.service.js`, `vendors.service.js`): `list` / `getById` / `search` / `create` / `findOrCreate` / `update` / `remove`. `findOrCreate` is the dedup primitive — used by the parts write path. `create` is also dedup-safe (returns existing row if `normalized_name` matches). All inserts handle 23505 (unique violation) by re-reading.
- **Search endpoint** (`GET /api/{manufacturers,vendors}/search?q=&limit=`): SQL-only similarity scoring (no `pg_trgm` dependency) — prefix matches score 1.0, substring matches score `0.5 + length_ratio`. Default limit 10, max 50. Returns `[{ id, name, normalized_name, similarity }]` ordered by similarity desc.
- **Routes** (`routes/manufacturers.js`, `routes/vendors.js`): standard `GET`/`POST`/`PATCH`/`DELETE` with `authMiddleware` everywhere and `requireRole(['admin', 'parts_manager'])` on writes — same shape as `routes/parts.js`. `/search` is registered before `/:id(\d+)` so the autocomplete path doesn't collide with id lookup.
- **Parts write-path sync** (`services/parts.service.js`): new `resolveManufacturerVendor()` helper. If `manufacturer_id` is provided, the master row is loaded and `parts.manufacturer` is forced to `master.name` (FK is authoritative). If only `manufacturer` text is provided, `findOrCreate` is invoked and both the FK and the canonicalized text are written. `null` FK explicitly clears it. Same logic for vendor / `preferred_vendor_name`. Bug fix: `createPart` previously dropped the `manufacturer` column entirely on insert — now it's threaded through and kept in sync.
- **Mounting**: `vehicles-maintenance-service/server.js` mounts both routers with the same `authMiddleware → tenantContextMiddleware → requirePartsPlan` stack as `/api/parts`. Gateway proxies `/api/manufacturers` and `/api/vendors` to vehicles-maintenance.
- **Tests** (`test/manufacturers-vendors-dedup.test.js`, 15 cases under `node:test`): normalization parity, `findOrCreate` dedup across case/whitespace for both tables, `resolveManufacturerVendor` sync rules (text→FK find-or-create, FK→text canonicalization, explicit null clearing, omitted-field passthrough). Hermetic — uses an in-memory mock knex injected via `setDatabase`.

## Key Decisions

### FN-1092 (subtask)
- **`bigserial` PKs (BIGINT FKs) for masters, even though `parts.id` is `uuid`.** The Jira AC explicitly specifies BIGINT FKs; bigserial is also cheaper to index and join for high-cardinality lookup tables that live entirely behind the application. The mixed-type FK on `parts` is acceptable because `manufacturer_id` / `vendor_id` only point at the new tables, never at `parts`.
- **`normalized_name` is UNIQUE, not just indexed.** AC asks for an index for autocomplete + dedup. Promoting it to UNIQUE pushes dedup enforcement down to the database so two case/whitespace variants cannot create separate rows even under a race — exactly what the epic needs. `name` is also UNIQUE per AC.
- **`ON DELETE SET NULL`** on both FKs. We never want a master deletion to cascade and erase part rows; nulling the FK preserves the part and (until cutover) the free-text fallback.
- **`MIN(name)` for the canonical display value** during backfill. Deterministic and good enough; the backend service (FN-1093) can offer a "rename" endpoint later if curators want to fix the casing.

### FN-1093 (subtask)
- **No `pg_trgm` dependency for autocomplete.** Pure-SQL `CASE` similarity (prefix=1.0, substring=`0.5 + length_ratio`) is monotonic, predictable, and cheap on the existing `normalized_name` btree. We can promote to `pg_trgm` later if curators ask for true fuzzy match (typo tolerance) — at that point a separate migration adds the extension and a GIN index.
- **`requirePartsPlan` gate on the new routes.** Manufacturers and vendors are functionally a sub-feature of parts; mounting them under the same plan gate means tenants without parts access can't enumerate the master tables either. Easy to relax later if other features need them.
- **Bug fix folded into FN-1093**: `createPart` was silently discarding `manufacturer` (it was never inserted). The fix is in scope because the AC requires text↔FK sync — without the fix, every new part had `manufacturer = NULL` and the FK sync had nothing to canonicalize.
- **`findOrCreate` returns `null` on empty/whitespace input** rather than throwing. Lets the parts write path treat "no manufacturer provided" symmetrically with "manufacturer = ''" without a special case at the call site.
- **`requireRole(['admin', 'parts_manager'])` on writes**, mirroring `parts.js` exactly. Read paths only require auth so any signed-in user can drive the typeahead.
- **Hermetic tests via mock knex.** Pulling in a real PG (and seeding migrations) for unit tests would be heavy; the mock enforces the unique-on-`normalized_name` constraint with PG error code `23505`, which is exactly what `findOrCreate`'s race-handler expects. End-to-end coverage on a real DB is FN-1095's job.
