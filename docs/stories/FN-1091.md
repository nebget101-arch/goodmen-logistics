# FN-1091: Manufacturers & Vendors master tables (foundation)

- [ ] Legacy text columns (`parts.manufacturer`, `parts.preferred_vendor_name`) preserved and kept in sync on write.
- [ ] CRUD + autocomplete endpoints: `/api/manufacturers`, `/api/vendors`.
- [ ] Add/Edit Part form replaces free-text inputs with typeaheads + inline "Create new".
- [ ] QA validates dedup, autocomplete latency, and backfill.

## Implementation Summary

### FN-1092 (subtask)
Foundation schema for the Quick Add Part epic. Two Knex migrations land on `integration/FN-1091`:

1. `20260505100000_add_manufacturers_vendors_tables.js` — creates `manufacturers` and `vendors` master tables with `bigserial id`, `name UNIQUE`, `normalized_name UNIQUE`, and timestamps. `vendors` adds optional `contact_email` / `contact_phone`. Adds nullable `manufacturer_id` / `vendor_id` BIGINT FKs to `parts` (`ON DELETE SET NULL`) plus btree indexes on each FK column.
2. `20260505100100_backfill_manufacturers_vendors.js` — inserts distinct, case-insensitive, whitespace-collapsed values from `parts.manufacturer` and `parts.preferred_vendor_name` into the new master tables, then updates each part's FK to its matching master row. Uses `LOWER(BTRIM(REGEXP_REPLACE(value, '\s+', ' ', 'g')))` for normalization — the backend autocomplete (FN-1093) must use the identical expression. NULL/empty source values are skipped, leaving the FK NULL.

Free-text columns `parts.manufacturer` and `parts.preferred_vendor_name` are intentionally preserved; FN-1093 will keep them in sync on write until callers fully migrate.

End-to-end verified locally: `migrate:latest` → schema correct, dedup collapses three case/whitespace variants of "ACME Corp" to one row; `migrate:rollback --all` → tables and FK columns dropped, free-text columns intact; idempotent re-run after rollback succeeds.

## Key Decisions

### FN-1092 (subtask)
- **`bigserial` PKs (BIGINT FKs) for masters, even though `parts.id` is `uuid`.** The Jira AC explicitly specifies BIGINT FKs; bigserial is also cheaper to index and join for high-cardinality lookup tables that live entirely behind the application. The mixed-type FK on `parts` is acceptable because `manufacturer_id` / `vendor_id` only point at the new tables, never at `parts`.
- **`normalized_name` is UNIQUE, not just indexed.** AC asks for an index for autocomplete + dedup. Promoting it to UNIQUE pushes dedup enforcement down to the database so two case/whitespace variants cannot create separate rows even under a race — exactly what the epic needs. `name` is also UNIQUE per AC.
- **`ON DELETE SET NULL`** on both FKs. We never want a master deletion to cascade and erase part rows; nulling the FK preserves the part and (until cutover) the free-text fallback.
- **`MIN(name)` for the canonical display value** during backfill. Deterministic and good enough; the backend service (FN-1093) can offer a "rename" endpoint later if curators want to fix the casing.

## Deployment Handoff

### FN-1092 (subtask)
- **DB migrations to run** (in order, on `dev` after PR merge):
  - `20260505100000_add_manufacturers_vendors_tables.js`
  - `20260505100100_backfill_manufacturers_vendors.js`
- **Service to redeploy:** `goodmen-database-migrations` (Render migration job — see `.agent/docs/render_services.md`). Migrations are additive and reversible; no downtime.
- **Backfill cost:** scans `parts` twice. On dev/staging this is cheap (low row count); on prod, expect a single full table scan + grouped insert + indexed update. Run during low-traffic window if `parts` exceeds ~1M rows (current row count: well below this — safe to run anytime).
- **Service restarts after migration:** none required for FN-1092 itself. The new columns are nullable and unread until FN-1093 ships, so existing services continue to function. Once FN-1093 lands, `vehicles-maintenance-service` (or whichever service owns `/api/manufacturers` and `/api/vendors`) must be redeployed.
- **Env / config / feature flags:** none.
- **Cache invalidation:** none.
- **Post-deploy verification:**
  - `SELECT to_regclass('manufacturers'), to_regclass('vendors');` returns two non-null OIDs.
  - `\d parts` shows `manufacturer_id` and `vendor_id` columns with FK constraints.
  - `SELECT COUNT(*) FROM manufacturers` and `SELECT COUNT(*) FROM vendors` are non-zero (assuming `parts` has data).
  - `SELECT COUNT(*) FROM parts WHERE manufacturer IS NOT NULL AND BTRIM(manufacturer) <> '' AND manufacturer_id IS NULL;` returns 0 (every non-empty source row got an FK).
  - Same check for `preferred_vendor_name` / `vendor_id`.
