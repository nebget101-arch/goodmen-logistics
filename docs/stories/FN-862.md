## Implementation Summary

### FN-873 (subtask)
Added `LoadWizardComponent` (selector `app-load-wizard-v2`) at `frontend/src/app/components/load-wizard/`. Standalone Angular 17 component, built on `<app-wizard-shell>` from FN-871. Step bodies are placeholders — real controls land in S3–S6 (FN-863..FN-866). Exists alongside the legacy `<app-load-wizard>` modal until S10 (FN-869) removes it.

**Component API**
- Inputs: `mode: 'create' | 'edit' | 'view' | 'ai-extract'` (default `create`), `loadId: string | null`.
- Outputs: `created: EventEmitter<LoadDetail>`, `updated: EventEmitter<LoadDetail>`, `closed: EventEmitter<void>`.
- Nested `FormGroup` matching the shape specified on FN-862:
  - `basics` — `loadNumber`, `status`, `billingStatus`, `brokerId`, `poNumber`, `rate`, `dispatcherId`, `notes`.
  - `stops` (`FormArray<FormGroup>`) seeded with a PICKUP + DELIVERY pair.
  - `driverEquipment` — `driverId`, `truckId`, `trailerId`, `showAllTrucks`.
  - `attachments.queued` — `FormArray<FormGroup>` of `{ file, type, notes, uploading, uploaded, error }`.

**Step navigation**
- `currentStepId` drives `<app-wizard-shell>`; `canProceed` is derived from step-level validity:
  - `basics` — group valid (status/billingStatus/rate required).
  - `stops` — at least 2 stops and all valid.
  - `driver` — always valid (no required fields at this stage).
  - `attachments` — valid by default, disabled while submitting.
- Mode mapping: `ai-extract` → shell mode `create`; `view` and `edit` pass through.

**Create submit**
- On the final step's Submit, builds a payload matching the legacy `createLoad` contract (same keys as `loads-dashboard.component.ts:createLoad`) and calls `LoadsService.createLoad`.
- On success, `forkJoin`s `LoadsService.uploadAttachment` over every queued file. Individual upload failures are tolerated — error count is surfaced in `errorMessage` but `created` still emits the load so the dashboard can refresh.
- On `createLoad` failure, surfaces `error.error` (server msg) inline via `errorMessage`, keeps `submitting=false`, and never resets the form — user can correct and retry.

**Dashboard trigger**
- Added a small "Try new wizard (beta)" pill next to the hero CTA in `loads-dashboard.component.html`, gated by `appHasPermission="'loads.create'"` and hidden for driver role.
- New state: `showLoadWizardV2` + handlers `openLoadWizardV2`, `closeLoadWizardV2`, `onLoadWizardV2Created`.
- Integrated the standalone component into `LoadsDashboardModule.imports` so it can be used in the dashboard template without duplicate declaration.
- Added `showLoadWizardV2` to the page-drop guard so dragging files over the page doesn't hijack the new modal.

**Tests**
- `load-wizard.component.spec.ts` covers: form build & defaults, per-step `canProceed` gating (basics required-rate, stops length ≥ 2, driver always valid, attachments submitting-gate), navigation no-op when blocked, create-mode submit calls `createLoad` with the correct payload and emits `created`, queued attachments trigger `uploadAttachment` for every queued file, partial upload failure emits `created` but surfaces an error, `createLoad` failure preserves form values, `closed` emission, and mode→shell-mode mapping.

## Key Decisions

- **New directory at `components/load-wizard/` rather than reusing `components/loads-dashboard/load-wizard/`.** The legacy folder owns the current `app-load-wizard` selector and `LoadWizardModule`. Co-locating the new component under `loads-dashboard/` risked shadowing imports, and the story explicitly says the new path is `components/load-wizard/`. The new component is standalone so it is imported into `LoadsDashboardModule.imports` — no new feature module needed yet.
- **Distinct selector `app-load-wizard-v2`.** Avoids colliding with the legacy `app-load-wizard` selector, which is still declared by `LoadWizardModule`. Rename to `app-load-wizard` happens in FN-869 (S10) when the legacy modal is deleted.
- **Placeholder step bodies are real enough to create a load.** S3–S6 replace them with rich UIs; for S2 they just surface the fields the payload needs (status / billing / rate / sequenced stops) so the create-submit path can be exercised end-to-end without depending on later subtasks.
- **`forkJoin` with per-upload `catchError` for attachments.** Matches the dashboard's legacy behavior (load is created even if some attachments fail) and keeps the wizard deterministic — it doesn't stall waiting on a single failing upload and never silently drops the `created` emission.
- **`ai-extract` mode folds into the shell's `create` mode.** The shared `WizardShellComponent` only models `create | edit | view`; the extra mode here is a loader-origin flag for later stories (FN-868) and doesn't change footer semantics.
