# FN-1788: Fill fields, send secure e-sign link, capture signature, generate signed PDF

> Epic: **FN-1786**. Blocked by **FN-1787** (needs the finalized template + field map).

## Requirement
From a finalized template, the internal user fills `internal`-assigned fields, then the system sends a **secure tokenized e-sign link** to the **signer** (customer/driver). The signer opens a **web interface** showing all details, fills their `signer`-assigned fields, and applies an **in-house e-signature** (typed name = signature + IP/UA capture, reusing `consent-service` semantics). On submit, a **signed PDF** is generated (values + signature overlaid on the source doc), stored in R2, and the request is marked `signed`/`completed`.

## Contract (fixed up front ‚Äî backend/frontend build in parallel)
- Internal fill: `POST /api/agreements/:templateId/requests` (internal field values + signer contact) ‚Üí `{ requestId, signerLink }`.
- Public (token-gated, rate-limited; mirror `public-employer-investigations.js`):
  - `GET /public/agreements/sign/:token` ‚Üí `{ document, fields (signer-assigned; internal values read-only), status }`
  - `POST /public/agreements/sign/:token` ‚Üí `{ fieldValues, signerName, signatureValue, consent }` ‚Üí records signature (name/IP/UA/ts), generates signed PDF ‚Üí `{ status: "signed", signedPdfUrl }`
- Statuses: `draft ‚Üí sent ‚Üí viewed ‚Üí signed ‚Üí completed` (+ `expired`, `voided`).

## Acceptance Criteria
- [ ] Schema: `signature_requests`, `signature_request_fields`, `signatures` (token_hash, expiry, signer name/IP/UA, signed_pdf_key, status).
- [ ] Backend: internal-fill endpoint; tokenized link (reuse `token-service`); public GET/POST signer routes; signed-PDF (`pdf.service`) + R2; status transitions; send link via Twilio/SendGrid.
- [ ] Frontend: internal fill screen; public signer web interface; reusable signature-capture component (typed; optional drawn).
- [ ] Signed PDF downloadable by internal user + signer; token reuse/expiry/void handled; signing idempotent.
