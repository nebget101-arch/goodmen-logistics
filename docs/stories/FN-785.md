### FN-797 (subtask)

Added `GET /api/loads/smart-filter-counts` returning `{ ai_drafts, overdue, high_value, from_email, missing_docs, my_drafts }` in one round trip, and extended `GET /api/loads` with a `smart_filter=<csv>` query param so chip clicks apply server-side (multiple chips AND'd). Tenant- and operating-entity-scoped; driver role limited to own loads.

Predicates (mapped to existing schema — no `loads.source`/`loads.created_by` columns exist):
- `ai_drafts` → `needs_review = true AND status = 'DRAFT'` (FN-746 convention)
- `overdue` → last-delivery `stop_date` (or `completed_date`) `< CURRENT_DATE` AND status not in DELIVERED/COMPLETED/CANCELLED/CANCELED/TONU
- `high_value` → `rate > COALESCE(percentile_cont(0.75) WITHIN GROUP (ORDER BY rate))` over tenant+OE loads created in last 30 days with `rate > 0`
- `from_email` → `EXISTS (inbound_emails WHERE load_id = l.id)` (FN-759 table)
- `missing_docs` → delivered/completed without `PROOF_OF_DELIVERY`/`POD` attachment OR picked-up/in-transit without `BOL`
- `my_drafts` → `status = 'DRAFT' AND dispatcher_user_id = req.user.id` (no `created_by` column; `dispatcher_user_id` is set from `req.user.id` on create)

### FN-798 (subtask)

New `<app-smart-filters>` component renders the 6 chips above the loads table with count badges. Each chip click toggles membership in a `smartFilterKeys: SmartFilterKey[]` array on the dashboard component; any size > 0 produces a `smart_filter=<csv>` query param on the next `listLoads()` call, so multiple active chips AND together server-side (the backend helper from FN-797 does the heavy lifting). "Clear smart filters" appears whenever at least one chip is active and resets the array in one click. The existing `Clear all` button (FN-746 / FN-762 active-filter strip) also clears the chip array.

Counts come from a second request to `GET /api/loads/smart-filter-counts` fired alongside each list reload so badges stay in sync with the underlying data. Count requests fail silently — if the backend 404s (FN-797 not deployed yet) the chips still render with their last-known counts and don't block the list.

Visual: active chip uses a teal fill (`rgba(20, 184, 166, 0.18)` bg / `#2dd4bf` text) to distinguish from the yellow active state used by the pre-existing `Needs review` / `Source: Email` chips. Hover state matches the teal palette.

## Key Decisions
### FN-797
- Mapped the spec's `source='ai_extraction'`/`source='email'`/`created_by` references to existing columns (see predicate table above) — no schema change in this subtask.
- `/smart-filter-counts` computes all six counts in a single SQL statement via `COUNT(*) FILTER (WHERE …)` so the page makes one round trip per chip refresh.
- The 75th-percentile threshold is computed inline as a scalar subquery so the endpoint stays self-contained and the list endpoint reuses the exact same predicate helper.
- Driver role is scoped to own loads in both the counts endpoint and the list endpoint's smart_filter branch (mirrors existing list-endpoint behavior).
- Route ordering: `/smart-filter-counts` is registered before `/:id` to avoid Express matching it as a load id.
- Unknown chip values passed to `smart_filter` are silently ignored (no 400) — consistent with the forgiving style of the existing `needsReview` / `status` filters.

### FN-798
- Kept the existing FN-746 "Needs review" and FN-762 "Source: Email" chips in place (they survive as standalone toggles because they hook into `filters.needsReview` / `filters.source`, not the `smart_filter` pipe). The new smart-filter row sits directly above them so the two systems read as adjacent chip strips rather than competing controls. A later consolidation (fold `needsReview` into `ai_drafts`, `source='email'` into `from_email`) is intentionally deferred — handling it here would have expanded scope beyond the subtask AC and risked breaking in-flight saved-view work (FN-767).
- Counts are fetched as a separate request (not embedded in the list response) to match the backend contract from FN-797 and to avoid coupling chip badges to the paginated list payload.
- Silent degradation: if `/smart-filter-counts` fails (backend down / not yet deployed / 500), chips still render with zero or stale counts and the list continues to work — the failure is logged-and-dropped, not shown to the user. Rationale: the chips are a progressive-enhancement over the list, not a blocking control.
- Active chip palette: teal (`#2dd4bf` family) — picked to match the existing AI accent (`#2dd4bf` / `#5eead4` already used elsewhere in `styles.css`) and to be visually distinct from the yellow `quick-filter-chip--active` used by the FN-746 / FN-762 chips.
- `smartFilterKeys` is stored as an ordered array rather than a `Set` so the `active` `@Input()` can pass a plain, Angular-change-detection-friendly reference without extra spread conversions in the template.
