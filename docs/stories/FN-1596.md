# FN-1596 — [P1] Loads Import: AI Analysis renders empty mapping (FE/BE contract drift)

## Bug
After uploading a CSV/XLSX, the wizard advances to step 2 (AI Analysis) and the "Suggested Column Mapping" table is empty. The AI service runs and returns a valid mapping; the frontend can't see it because of multiple field-name mismatches between the FN-1585 wire contract (used by the backend) and the FE model that diverged in FN-1594.

## Implementation Summary

The bug write-up correctly identified five FE field-name mismatches against the FN-1585 wire contract. While fixing those, I also found a sixth contract issue not in the AC: the loads-import routes (`/preview`, `/stage`, `/commit`, `/batches*`) all wrap responses in `{ success: true, data: <payload> }`, but `loads-import.service.ts` was treating that envelope as the payload. Renaming `aiMapping → columnMapping` alone wouldn't have fixed the empty table — the FE would still be reading off the envelope, where neither field name exists. The service now `pipe(map(r => r.data))` for every endpoint.

`/stage` was also calling the wrong protocol: it sent multipart with the file, while the BE route expects JSON `{ batchId, columnMapping, ... }`. The wizard now persists `batchId` from the preview response and posts the staged mapping (translated from the FE's `field → header` map back into the BE's `field → { sourceHeader, confidence }` shape) as JSON.

The mapping-step's pattern toggle was a binary `Single / Multi` UI; it now exposes the four BE enum values (`single`, `multi_row`, `extra_columns`, `free_text`) so the user can override the AI's pattern detection without the value being silently coerced server-side.

`LOADS_IMPORT_FIELDS` gained `po_number`, `broker_mc`, `broker_dot`, `status`, `billing_status` (the AC explicitly named the first three; the last two are required for the AC's smoke-test bullet, which checks Status / Billing Status confidence) and the address keys were aligned to `pickup_address1` / `delivery_address1`.
