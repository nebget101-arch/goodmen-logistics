# FN-1454: FMCSA importer fails on Render: csv-parse missing from integrations-service package.json

## Requirement

After FN-1452 (registerImporter wiring) + FN-1453 (clearer 503 messages) shipped, `POST /api/fmcsa/imports/run` on Render dev returned the captured boot error: `Cannot find module 'csv-parse'`. `csv-parse` is declared in `goodmen-shared/package.json` but not in the consuming `integrations-service/package.json`. Render builds each service from its own `rootDir`, and npm's `file:` protocol does not run a nested install on linked packages, so `goodmen-shared/node_modules/csv-parse` never gets created. Mirror the dep into `integrations-service` (the repo's existing convention for every other shared runtime dep) and regenerate the lockfile.

## Acceptance Criteria

- [x] `backend/microservices/integrations-service/package.json` lists `csv-parse: ^5.5.0` (matches goodmen-shared pin).
- [x] `backend/microservices/integrations-service/package-lock.json` regenerated; `csv-parse@5.6.0` resolved under `node_modules/csv-parse`.
- [x] Audit confirms only `integrations-service` consumes `fmcsa-import-queue` / `register-importers`. `ai-service/src/handlers/fmcsa-driver-match-handler.js` does not pull the bulk-import chain. No other service needs csv-parse.
- [x] Sanity-check confirms `csv-parse` is the only undeclared parser dep — `parsers/{inspection,crash,sms}.v1.js` require `csv-parse`; `parsers/{census,authority}.v1.js` have no extra requires.
- [ ] After Render redeploy of `fleetneuron-integrations-service`, `POST /api/fmcsa/imports/run {"files":["census","authority"]}` no longer returns the MODULE_NOT_FOUND boot error (verified post-merge).

## Files Changed

- `backend/microservices/integrations-service/package.json` — added `"csv-parse": "^5.5.0"` to `dependencies`.

## Key Decisions

- **Mirror, don't restructure.** The bug class is solved long-term by FN-1320 (chained shared install in Render build) but the per-service `npm install` on Render is the current contract. Adding csv-parse to the consumer matches the existing pattern (`axios`, `cheerio`, `ioredis`, `pdf-parse`, `jimp`, `multer`, `pdfkit`/FN-1318, `@aws-sdk/*`/FN-1358 all do this). Out-of-scope: monorepo restructure, CI guard for missing deps.
- **Lockfile regenerated, not hand-edited.** Render runs `npm install` against the lockfile; the lockfile must contain a resolved entry for csv-parse or the install falls back to package.json semver-resolve and may diverge across rebuilds. The diff is large (~4k lines) because the previous lockfile had drifted: `packages."".dependencies` only listed ~9 of 23 deps, and many AWS-SDK / cheerio / ioredis / pg transitive trees had no entries at all. `npm install --package-lock-only` syncs all of that in one pass — same pattern as FN-1358 (ai-service +7577/-1103 lockfile sync alongside a 2-line package.json change). Resolved versions are pulled from npm registry "latest matching semver"; the only intentional pin is csv-parse@^5.5.0, which resolved to 5.6.0.
- **Used `--package-lock-only --ignore-scripts`.** Avoids the cross-package install collision (`enoent` rename across `file:` linked packages on macOS) that the full `npm install` triggers locally; produces the same lockfile that Render's `npm install` would write on a clean machine.

## Deployment Handoff

- **Service to redeploy:** `fleetneuron-integrations-service` only.
- **Migrations:** none.
- **Env vars:** none.
- **Cache invalidation:** none.
- **Feature flags:** none.
- **Deploy order:** standalone — no cross-service coordination.
- **Post-deploy verification:**
  1. Watch Render build logs — confirm `npm install` step lists `csv-parse@5.6.0` under added packages and exits 0.
  2. After boot, check `/api/fmcsa/imports/status` (or any of the route-gated endpoints) — expect 200 instead of FN-1453's 503-with-MODULE_NOT_FOUND body.
  3. `POST /api/fmcsa/imports/run` with `{"files":["census","authority"]}` as a `super_admin` of the internal tenant. Expect 202 + `runIds`. (If Redis is up and importers were registered by FN-1452, ledger rows in `fmcsa.import_runs` will progress past the boot-fail state.)
  4. If the import URL is unreachable from Render, the run will land on `status='error'` with a network/HTTP message rather than the previous boot-time `MODULE_NOT_FOUND`. That's the correct downstream-failure surface, not this bug.
