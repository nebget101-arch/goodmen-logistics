# FN-1414: FMCSA Inspection + Crash + SMS Input bulk importer

## Requirement

Replace the per-page CompleteProfile.aspx and SMS BASIC detail scrapers with three bulk-file importers:

1. **MCMIS Inspection File** — populates `fmcsa.inspections` and `fmcsa.violations`. Daily refresh, 3 years of history.
2. **MCMIS Crash File** — populates `fmcsa.crashes`.
3. **SMS Input Files** — populates `fmcsa.basic_scores` (keeps history per measurement period).

Source: `ai.fmcsa.dot.gov/SMS/Tools/Downloads.aspx`. May require captcha/login — confirm during implementation.

## Implementation Summary

### FN-1422 (subtask)

Stood up `backend/packages/goodmen-shared/services/fmcsa-importer/` as a new module separate from the existing `fmcsa-scrape-queue.js`. Three Bull-queue processors (`import-fmcsa-inspections`, `import-fmcsa-crashes`, `import-fmcsa-sms`) backed by per-file driver modules and a versioned parser layer (`parsers/{inspection,crash,sms}.v1.js`). All DB I/O goes through `getFmcsaKnex()` so the Phase 2 swap to a separate FMCSA DB stays a single-file change.

Streaming: `csv-parse` is added as a dependency. Parsers are async iterables; drivers buffer 500-row batches and upsert in a single transaction. Inspection rows in MCMIS files are one-row-per-violation; the inspection parser groups by `REPORT_NUMBER` (single-pass with a "current report" buffer that flushes on key change) and aggregates `vehicle_count`/`driver_count`/`hazmat_count` and the matching OOS counts from the row collection.

Idempotency:
- `fmcsa.inspections` upserts on `inspection_report_number` via Knex `onConflict().merge(...)`.
- `fmcsa.violations` upserts on `(inspection_report_number, violation_code, sequence)`.
- `fmcsa.crashes` upserts on `crash_report_number`.
- `fmcsa.basic_scores` uses `ON CONFLICT (dot, basic, computed_at) DO NOTHING` — history is **never overwritten**, satisfying the BASIC trend-line requirement. E2E test `__tests__/sms.test.js` re-runs the same fixture and asserts row count is unchanged.

Audit trail: every run inserts a `fmcsa.import_runs` row up front (status `running`) so a process crash still leaves a trace, then updates to `success`/`error` with row counts on completion.

No service wiring or cron: `index.js` exports `createImportQueue(knex, redisUrl)` but `integrations-service/server.js` is intentionally untouched. FN-1415 owns the wiring + manual-trigger UI + cron.

## Key Decisions

### FN-1422 (subtask)
1. **New module, not extension of `fmcsa-scrape-queue.js`** — the scraper writes to legacy `fmcsa_*` (snake-case, public schema) tables; the importer writes to the new `fmcsa.*` (schema-qualified) tables. Different DB targets, different lifecycles (scraper retires per FN-1416). A separate `fmcsa-importer/index.js` keeps the boundary clean and lets FN-1416 delete the scraper without disturbing import code.
2. **Versioned parsers (`v1` filename suffix)** — FMCSA changes its file column shapes occasionally. A `v2` can be added side-by-side and selected by importer config without rewriting drivers.
3. **Source-resolution boundary** — the queue layer translates `{ type: 'url' | 'path', value, auth? }` into a Readable stream, drivers only see the stream. Tests inject a `fs.createReadStream(fixture)` directly. The Open Item on captcha is contained to the URL branch.
4. **Insert-vs-update accounting is approximate** for inspections/crashes (compare `created_at == updated_at` from RETURNING). Exact counts would require a per-row pre-check that tanks throughput on 5–7M rows. The audit-row totals are accurate; the split is a heuristic. SMS does not need this — `DO NOTHING` is unambiguous.
5. **Batch size 500, 4hr Bull timeout** — chosen to fit a single Postgres write window cleanly while keeping the largest historical inspection backfill under the timeout on a Render standard plan.

### Initial 3-yr Inspection Backfill (one-time runbook)

This is intentionally **not** wired into the cron path because the heavy file (~5–7M rows) needs operator oversight and may need an out-of-hours window.

1. Confirm DB has space (~2 GB free PG) and `fmcsa.inspections` is empty or backfill-acceptable.
2. Download the 3-yr MCMIS Inspection file from `https://ai.fmcsa.dot.gov/SMS/Tools/Downloads.aspx` to a local path (manual fetch — the captcha question is open at time of writing).
3. From a Render shell on `fleetneuron-integrations-service`:
   ```js
   const { createImportQueue } = require('@goodmen/shared/services/fmcsa-importer');
   const q = createImportQueue(knex, process.env.REDIS_URL);
   await q.enqueueInspectionImport({
     source: { type: 'path', value: '/tmp/Inspection.csv' },
     triggeredBy: 'manual',
   });
   ```
4. Watch logs (`[fmcsa-import]` prefix) — progress logs every 10000 rows.
5. After completion, validate by aggregating inspections for one known DOT (e.g. 3761784) and comparing vehicle/driver/hazmat counts and OOS rates against the existing scraper's "US Inspection table" snapshot. Tolerance: ±1 row (per acceptance criteria).
6. If aggregation differs > 1 row, check parser column-name assumptions in `parsers/inspection.v1.js` against the file header — FMCSA occasionally renames columns.

After the initial backfill, FN-1415's cron handles incremental imports; the upsert is idempotent so a daily re-import of the daily-refresh file just touches changed rows.
