# FN-289: Invite driver endpoint — create pending driver, generate onboarding packet, send email

## Requirement
Allow admin/safety users to invite a driver by email. Creates a minimal "pending" driver record, generates an onboarding packet with a secure token link, and sends the invite via email. When the driver completes onboarding and finalizes, their pending record is populated with application data.

## Acceptance Criteria
- [ ] POST /api/onboarding/invite creates pending driver + packet + sends email
- [ ] Duplicate email detection under same operating entity
- [ ] POST /api/onboarding/packets/:id/resend refreshes token and re-sends
- [ ] Finalization populates pending driver fields from application data
- [ ] Auth: admin/safety roles only

### FN-291 (subtask)
**Endpoints implemented:**

1. **POST `/api/onboarding/invite`** — Creates pending driver, generates onboarding packet, sends email
   - Validates firstName, lastName, email (required), phone (optional)
   - Checks for existing driver by email + operating entity
   - Creates driver with `status='pending'`, `dqf_completeness=0`
   - Generates 7-day token, creates packet with `status='sent'`
   - Initializes 3 sections (employment_application, mvr_authorization, uploads)
   - Sends email via `sendOnboardingLink()`
   - Returns `{ driverId, packetId, publicUrl, sentTo, isNewDriver }`

2. **POST `/api/onboarding/packets/:id/resend`** — Refreshes token and re-sends invite
   - Rejects if packet already submitted (409)
   - Generates new token, extends expiry 7 days
   - Optionally accepts new email in body
   - Returns `{ packetId, sentTo, publicUrl, emailSent, emailError }`

3. **Enhanced finalization** (`POST /public/onboarding/:packetId/finalize`) — Populates pending driver
   - After marking packet submitted, checks if driver status is 'pending'
   - Loads employment_application section data (JSONB)
   - Updates driver with: phone, dateOfBirth, address fields, CDL fields
   - Transitions driver status from 'pending' to 'applicant'
   - Gracefully fails without blocking finalization

**Files changed:**

## Implementation Summary
Added invite-driven onboarding flow where admin/safety can invite drivers by email without requiring CDL info upfront. The pending driver record gets populated when the driver completes and finalizes their onboarding application.

## Files Changed
- `backend/packages/goodmen-shared/routes/onboarding.js`
- `backend/packages/goodmen-shared/routes/public-onboarding.js`

## Key Decisions
- Driver status 'pending' is a new state — distinct from 'applicant' (has CDL info) and 'active'
- Invite creates driver without CDL fields (cdl_number is NOT NULL in schema, but pending drivers may need schema relaxation — see Open Items)
- Finalization reads from employment_application section JSONB data to populate driver fields
- Email is fire-and-forget to avoid blocking the invite response
