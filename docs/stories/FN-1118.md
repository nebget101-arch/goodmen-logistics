## Implementation Summary

### FN-1173 (subtask) — Decision: ADD `?variant=long`

**Outcome:** existing endpoint (FN-1123) is suitable as a base, but the 2–3 sentence default is too short for a 1+ page branded PDF. Added an opt-in `?variant=long` query parameter that doubles the token budget (400 → 900) and extends the narrative to 5–8 sentences across 1–2 paragraphs. The on-screen panel (FN-1146) keeps the original short variant by omitting the query param.

Also fixed a real server-to-server gap: `reports-narrative-handler.js#tryRequireJwt` falls back to `null` when `jsonwebtoken` isn't installed, returning 403 to any caller without `req.user` (which is exactly what reporting-service forwards). `jsonwebtoken@^9.0.3` was missing from `ai-service/package.json` even though every other microservice carries it. Added it as a direct dependency so the JWT verification path works end-to-end.

**Variant mechanism:**
- `resolveVariant(req)` reads `req.query.variant`, normalises (case-insensitive, trims, accepts first element of an array), and falls back to `short` for any unknown value. Never errors.
- The variant directive is delivered via the **per-call user message** (`form: "long"|"short"`) — the cached system blocks stay byte-identical between variants, so the prompt cache hit-rate is preserved across both call sites.
- `SYSTEM_PROMPT` describes both variants once; the runtime decision lives in the user payload's `form` key.
- `max_tokens` is the only Anthropic-call argument that varies per variant (400 vs 900).
- Response now includes `meta.variant` so callers can verify which length was applied.

**Server-to-server validation:**
- Reporting-service forwards the user's `Authorization: Bearer <jwt>` header verbatim. `req.user` is NOT pre-attached because the request bypasses the gateway.
- `resolveUser()` already supports this — it reads `req.headers.authorization`, verifies via `JWT_SECRET`, and surfaces the payload as `req.user`. Same `JWT_SECRET` env var must be set on both services (already true on Render).
- No IP allowlist / mTLS needed: AUTH is via the user's own JWT, which is shorter-lived and scoped to the same user that triggered the PDF export. No service-to-service shared secret to leak.
- Test: `server-to-server (Bearer JWT, no req.user) reaches handler with variant=long` signs a token with `JWT_SECRET`, sends it via `Authorization: Bearer …`, and verifies the handler 200s and reaches Anthropic with `variant=long`.

**Prompt-caching end-to-end:**
- The two cached system blocks (system prompt + per-report schema) are byte-identical between short/long calls — the `form` key only changes the per-call user message which is uncached.
- Therefore the second PDF export for the same `reportKey` will report `meta.cacheReadTokens > 0`, satisfying the AC.
- Test: `?variant=long uses 900 max_tokens + form="long" in user msg, cached system unchanged` asserts `system[0].text === SYSTEM_PROMPT` and that both blocks carry `cache_control: ephemeral` regardless of variant.

### FN-1160 (subtask)
- `ReportsService.exportReport(reportKey, 'pdf', filters)` now sets `includeNarrative=true` on the same `GET /reports/v2/export/:reportKey` endpoint. CSV requests are byte-for-byte identical to before.

The component's existing error-display channel is used rather than introducing a new global toast service that doesn't yet exist in this app.

## Key Decisions

### FN-1173 (subtask)
- **`?variant=long` over splitting endpoints**: a query-string variant is the smallest possible API surface change; reporting-service appends `?variant=long` to its existing call. A new `/narrative-long` endpoint would have duplicated 60 lines of validation/auth/caching wiring.
- **Variant lives in the user message, not the system block**: keeping `SYSTEM_PROMPT` and the schema block byte-identical between variants preserves prompt-cache hits across both call sites. If we'd built a per-variant cached block, every call would split the cache by variant and the first PDF after a panel call would be a cold miss.
- **Token budget doubled (400 → 900), not tripled**: 5–8 sentences across 1–2 paragraphs comfortably fits in ~700 output tokens; 900 leaves headroom for occasional verbose generations without the model being truncated mid-sentence. Going larger would invite the model to start recommending actions, which is FN-1146's territory.
- **Unknown variants silently degrade to `short`** instead of returning 400: keeps the endpoint forward-compatible with future variants and avoids a contract break if reporting-service ever sends a typo. Caller can verify the applied variant via `meta.variant` if it matters.
- **Added `jsonwebtoken` to ai-service**: the handler relied on it for the server-to-server path but the dep was missing — every other microservice already carries `^9.0.3`. Without this fix the FN-1167 backend implementation would 403 in production.
- **No IP allowlist or shared-secret scheme** for the server-to-server call: the user's own JWT is forwarded and verified with the same `JWT_SECRET` both services already share. Adding a service-to-service secret would create another credential to rotate without improving security (the user JWT is already the source of truth for the authorisation decision).

### FN-1160 (subtask)
- **Same endpoint, new flag** rather than a new branded URL. Keeps the contract surface small, preserves backwards compatibility, and avoids inventing a route name before the BE subtask (FN-1167) has chosen one. BE only needs to read one query param.
- **No global toast service introduced.** This component already exposes errors via an inline `<p class="error">` element rendered in the template; reusing it is consistent with the rest of the page and avoids cross-cutting changes outside the FN-1160 scope.
- **Per-format export flags** (`{ csv, pdf }`) instead of a single `isExporting` boolean — the two formats can race independently and we want each button to show its own loading state.

## Deployment Handoff

### FN-1173 (ai-service)
- **Service**: `ai-service` redeploy. The new `jsonwebtoken` dep needs to land via `npm install` during the Render build, so a redeploy (not a hot restart) is required.
- **Migrations**: none.
- **Env vars**:
  - `JWT_SECRET` must be identical on `ai-service` and `reporting-service`. Both already read it (`process.env.JWT_SECRET || 'dev_secret'`); confirm Render has the same value set on both — this was already the case before FN-1173, but worth re-checking before the FN-1167 / FN-1160 deploy.
  - No new env vars introduced.
- **Cache**: the cached system block content changed (the variant guidance was appended to `SYSTEM_PROMPT`). Anthropic will rebuild the cache on the first call after deploy — one-time cold call per `reportKey`. Subsequent calls warm normally and report `cacheReadTokens > 0`.
- **Feature flags**: none.
- **Verification**:
  1. `curl -X POST $AI_SERVICE/api/ai/reports/revenue-by-driver/narrative -H "Authorization: Bearer <user-jwt>" -d '{"cards":[],"data":[]}'` → expect `meta.variant: "short"`, narrative is 2–3 sentences, max ~400 tokens.
  2. Repeat with `?variant=long` → expect `meta.variant: "long"`, narrative is 5–8 sentences across 1–2 paragraphs.
  3. Run the same `?variant=long` call twice → second call's `meta.cacheReadTokens > 0`.
  4. From inside the reporting-service container (or any non-gateway origin), call the same endpoint with the user's bearer token → expect 200 (no `req.user` pre-attached, JWT verified locally).
  5. Swagger UI at `$AI_SERVICE/api-docs` shows the `variant` query parameter on `/api/ai/reports/{reportKey}/narrative`.

### FN-1160 (frontend)
- **Services to redeploy:** `frontend` (only). No env vars, migrations, jobs, feature flags, or cache changes.
- **Coordinated deploy:** the new `?includeNarrative=true` flag is **forwards-compatible**: until FN-1167 ships, BE will simply ignore the extra query param and continue serving the legacy PDF — there is no breakage from FE shipping first. Full branded output appears once FN-1167 is also deployed.
- **Post-deploy verification:** open any report page → click "Export PDF" → confirm the button shows the spinner + "Generating PDF…" label, the request URL contains both `format=pdf` and `includeNarrative=true`, and the downloaded file is named `<reportKey>-<dateRange>.pdf`. Click "Export CSV" → confirm CSV path is unchanged (no `includeNarrative` param) and downloads `<reportKey>-<dateRange>.csv`.
