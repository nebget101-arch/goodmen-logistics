# FN-384: Approve draft overrides dropdown field values (Status, Billing Status, Driver, Truck, Trailer)

## Requirement
Fix the approve-draft flow so dropdown field values (billingStatus, driverId, truckId, trailerId) are correctly preserved when a user clicks Approve on an AI-extracted draft load.

## Acceptance Criteria
- [x] Approve preserves Billing Status value selected in the form
- [x] Approve preserves Driver selection
- [x] Approve preserves Truck selection
- [x] Approve preserves Trailer selection
- [x] Values match exactly what was displayed in the form before clicking Approve
- [x] Status is always set to DISPATCHED by the backend (by design)

## Implementation Summary

**Root cause:** The `approveDraftLoad()` frontend method was missing `billingStatus` from the request body. The backend approve-draft endpoint's `fieldMap` also lacked `billingStatus`, so even if sent it would have been ignored.

**Fixes applied:**

1. **Frontend** (`loads-dashboard.component.ts`): Added `billingStatus: formValue.billingStatus || null` to the approve request body
2. **Backend** (`loads.js` approve-draft endpoint): Added `billingStatus: 'billing_status'` to the fieldMap, with proper `normalizeEnum()` normalization and `BILLING_STATUSES` validation (same pattern as the PUT endpoint)

Note: `driverId`, `truckId`, and `trailerId` were already included in both the frontend payload and backend fieldMap. The form is populated correctly by `populateFormFromDetail()` with the current values, and `|| null` correctly converts empty strings to null. These fields were working — `billingStatus` was the missing piece.

`status` is intentionally excluded — the approve-draft endpoint always forces `status = 'DISPATCHED'`.

## Files Changed
- `frontend/src/app/components/loads-dashboard/loads-dashboard.component.ts` — added billingStatus to approve body
- `backend/packages/goodmen-shared/routes/loads.js` — added billingStatus to approve-draft fieldMap with validation
