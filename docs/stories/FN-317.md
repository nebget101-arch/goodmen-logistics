# FN-317: [CRITICAL] Fix app-ai-select infinite change detection loop

## Requirement
PR #340 mass-converted raw `<select>` elements to `<app-ai-select>` but used getter-based `[options]` bindings that return new array references on every Angular change detection cycle. This causes infinite change detection loops that freeze the site at 100% CPU. Production was rolled back to abc15a7.

## Acceptance Criteria
- [x] All 15 getters converted to cached properties
- [ ] Site loads without freezing
- [ ] All select dropdowns still function correctly
- [ ] No infinite change detection loops

## Dependencies
- None

## Agent
frontend

## Implementation Summary
Converted all 15 getter-based `[options]` bindings across 7 files to cached properties that only rebuild when source data changes:

**Pattern used:**
- **Static source data** (readonly arrays like `drugTestTypes`, `expenseKeys`): Converted getter to `readonly` property initialized at class declaration time
- **Dynamic source data** (API-loaded arrays like `drivers`, `trailers`, `cards`): Converted getter to a plain property, added rebuild call at every point where the source data is assigned

**Key insight:** The existing `truckSelectOptions` in `dispatch-drivers.component.ts` already used the correct cached pattern — the fix applies the same approach to all 15 remaining getters.

## Key Decisions
- Used `readonly` for static-source getters (simplest, zero overhead)
- Used explicit rebuild calls at assignment sites for dynamic-source getters (follows existing `truckSelectOptions` pattern)
- Inlined `deductionExpenseTypeOptions` values into `sourceTypeOptions` to fix TS2729 (used-before-initialization) — the values are identical readonly literals
- Removed `filteredDrivers` getter from loads component (was only used by `driverSelectOptions`)
- Made `rebuildDriverSelectOptions()` public so the template can call it via `(ngModelChange)`
