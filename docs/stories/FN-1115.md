## API Contract
**Request** `POST /api/ai/reports/:reportKey/anomalies`
```json
{ "data": [...], "filters": {...}, "priorPeriod": {...} }
```
**Response**
```json
{ "anomalies": [
  { "metric": "revenue", "value": 12345, "deltaPct": -0.32, "severity": "warning", "context": "Below 90-day mean by 1.8σ" }
] }
```

## Implementation Summary

### FN-1134 (subtask) — AI anomaly endpoint
- `POST /api/ai/reports/:reportKey/anomalies` registered in `backend/microservices/ai-service/src/ai-router.js` and documented in Swagger.
- Handler builds the Claude system as **two cached blocks** (system prompt + per-report schema) so warm calls report `cache_read_input_tokens > 0`. Cold/warm token counts are surfaced on the response in `meta.cacheReadTokens` / `meta.cacheWriteTokens` for QA.
- Output is validated against the contract: bad entries are dropped, the array is capped to 6, and any unparseable / schema-mismatched / upstream-error response collapses to `{ anomalies: [] }` with a `meta.scoredBy` tag (never a 500).
- RBAC: a lightweight middleware (`services/auth-context.js`) trust-decodes the gateway-forwarded JWT (no signature re-verification — gateway is the trust boundary) and surfaces `req.user.{ id, role, tenantId, permissions }`. The handler returns 403 unless the caller is `super_admin` or holds `reports.view`. The middleware also honors a `x-user-permissions: <csv>` header so the gateway can override the static role map without a code change.
- 24 unit tests for the handler (happy path, malformed → empty, schema mismatch → empty, upstream error → empty, RBAC denial in three shapes, super_admin bypass, cache-read surfacing, validators) and 14 unit tests for `auth-context` — all passing under `node`.

### FN-1131 (subtask) — Frontend anomaly chip row
Added a token-driven anomaly chip row above the report table in `ReportViewComponent`. The component is purely presentational: the parent fetches `getAnomalies(reportKey, { data, filters })` after each successful report load and passes the response into `<app-report-anomalies>` via an `@Input`.

Severity → icon + color is a single `--chip-color` CSS custom property mapped to existing global tokens (`--primary-color`, `--warning-color`, `--danger-color`); chip surface and border use `color-mix()` against that token, so retuning a severity is a one-token change. No hardcoded hex was introduced.

Per-session dismissal lives in a component-local `Set<string>` keyed by `severity|metric|value|deltaPct`. Reload restores all chips, matching the AC's session scope.

The HTTP call quietly degrades: on `getAnomalies` error or empty `data` rows, the component receives `[]` and the chip row is hidden (no error UI). This avoids creating noise when the AI endpoint isn't available yet (e.g. before FN-1134 lands on the integration branch).

## Key Decisions

### FN-1134
- **Prompt caching shape:** `system` is sent as an array of two `cache_control: { type: 'ephemeral' }` text blocks. Block 1 is the static analyst prompt; block 2 is a per-`reportKey` JSON schema preamble. Per-call user content is just the JSON envelope — keeps the cache hit-rate high across reports of the same kind.
- **No 500s from a malformed model:** parse / schema mismatches / upstream errors all return 200 with `{ anomalies: [] }` and a `meta.scoredBy` tag. The chip row's empty-state handles this naturally and we never blow up the parent report view because the AI sidecar misbehaved.
- **RBAC location:** the score-alert / briefing handlers don't enforce RBAC themselves — auth lives at the gateway. The AC for FN-1115 explicitly required `reports.view` enforcement on this endpoint, so the handler enforces it directly. The new `auth-context` middleware is intentionally narrow (no DB calls, no `jsonwebtoken` dependency) — it trust-decodes the gateway-forwarded JWT and looks up role permissions in a static map mirrored from `01_rbac_seed.js`. For custom-role tenants the gateway can supply `x-user-permissions` and we prefer that over the static map.
- **Model:** defaults to `claude-sonnet-4-6` (`ANTHROPIC_REPORTS_MODEL` overrides). Temperature `0.1` for stable structured output.

### FN-1131
- **Color tokens via `color-mix()`**: lets the chip use `var(--warning-color)` directly without hardcoding the rgba background fade. Modern browsers (Angular 17 baseline) ship support; no polyfill needed.
- **Parent fetches, child presents**: keeps the component testable and lets the parent batch-coordinate the call with the report load (filters in scope, no duplicate state).
- **Silent degradation when AI endpoint is unreachable**: the chip row simply hides instead of surfacing an error, since anomalies are an optional augmentation of the report.
- **Dismissal key includes value + deltaPct**: prevents collisions when two anomalies share a metric but differ in magnitude across reloads in the same session.

## Deployment Handoff

### FN-1134

**Service to redeploy:** `ai-service` only (per `.agent/docs/render_services.md`).

- **Migrations:** none.
- **New env vars (optional):** `ANTHROPIC_REPORTS_MODEL` — defaults to `claude-sonnet-4-6` if unset. Existing `ANTHROPIC_API_KEY` is reused.
- **Cache invalidation:** none — Claude prompt caching is server-side at Anthropic and warms naturally on the second call.
- **Feature flags:** none.
- **Deploy order:** ship `ai-service` before FN-1131 (frontend chip row) lands so the FE has a backend to call. The FE will fall back to "no chips" gracefully if it ever calls before deploy.
- **Post-deploy verification:**
  1. `POST /api/ai/reports/dispatcher_summary/anomalies` (Bearer JWT for a `dispatch_manager`) with a small `{ data, priorPeriod }` body — expect `200 { anomalies: [...], meta: { cacheWriteTokens > 0 } }`.
  2. Repeat the same request — expect `meta.cacheReadTokens > 0` (proves prompt caching).
  3. Repeat with a JWT for a role lacking `reports.view` (e.g. `driver`) — expect `403 { code: 'AI_FORBIDDEN' }`.
  4. Hit `POST /api/ai/reports/'../etc/passwd'/anomalies` — expect `400`.
  5. Confirm `/api-docs` on `ai-service` shows the new operation under the AI tag.
