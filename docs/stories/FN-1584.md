# FN-1584: Spreadsheet import — backend + DB foundation

## Requirement
Build the server-side foundation for spreadsheet → loads import: a parse endpoint that returns headers + sample rows, a stage endpoint that validates rows against the AI-supplied column mapping, and a commit endpoint that creates loads atomically with idempotency and entity fuzzy-matching.

This story owns the API contract for the whole epic. Story B (FN-1585, AI handler) and Story C (FN-1586, frontend wizard) consume this contract — finalize the request/response shapes here before B and C cut implementation branches.

## Acceptance Criteria
- [x] New `load_import_batches` and `load_import_rows` tables (see FN-1589)
- [x] `POST /api/loads/import/preview` — multipart upload, hashes file, R2 stores, returns headers + 20 sample rows + cacheHit flag
- [x] `POST /api/loads/import/stage` — applies AI column mapping deterministically, writes per-row outcomes
- [x] `POST /api/loads/import/commit/:batchId` — transactional commit with duplicate detection, fuzzy entity matching, threshold-based DRAFT/NEW routing, status/billing enum mapping, multi-stop construction
- [x] `GET /api/loads/import/batches` and `GET /api/loads/import/batches/:id`
- [x] Tenant + operating_entity scoping on every endpoint; RBAC: Admin/Dispatch only
- [x] Phase 1 row cap: 500 rows; above → 413 with Phase 2 pointer
- [x] Idempotent commit (re-committing a batch is a no-op)
- [x] All inserts in a single transaction

## Key Decisions

### FN-1589 (subtask)
- **Timestamp `20260508150000`** chosen instead of the original `…130000`
  placeholder to avoid colliding with `20260508130000_receiving_ticket_invoice_columns.js`
  already on `dev`.
- **Status / validation_status modeled as `text` + `CHECK` constraint** rather
  than Postgres enum types — matches the existing convention in this codebase
  (see `loads.status`, `loads.billing_status`) and keeps future enum extensions
  to a one-line `ALTER … DROP/ADD CONSTRAINT` instead of a `CREATE TYPE` dance.
- **`confidence_score NUMERIC(4,3)`** (range 0.000 – 9.999) — the spec only
  needs 0.000–1.000 but `(4,3)` leaves headroom and matches Knex `decimal(4,3)`
  cleanly.
- **Partial index on `resulting_load_id`** (`WHERE resulting_load_id IS NOT NULL`)
  — most rows will be `null` until commit; the partial index keeps the lookup
  cheap without indexing the long pre-commit tail.
- **Index strategy**: tenant_id and (tenant_id, file_hash) cover both the
  cache-hit lookup that Story FN-1584 spec depends on and tenant-scoped batch
  listings; status drives the worker-style "find pending/failed" queries.

### FN-1590 (subtask)
- **Routes live in `loads.js`, not a new router file** — matches the
  fuel-import precedent (fuel.js holds all `/api/fuel/import/*` handlers) and
  avoids touching `logistics-service/server.js`. Pre-existing
  `app.use('/api/loads', authMiddleware, tenantContextMiddleware, loadsRouter)`
  already covers auth + tenant context for the new endpoints.
- **AI failure is non-fatal at preview time.** A timeout/unreachable AI service
  returns `aiUnavailable: true` with empty `columnMapping` instead of 5xx, so the
  wizard can fall back to manual mapping. Mirrors `loads-nlq-handler.js` policy.
- **Fuzzy match is pure SQL, no AI.** Three layers: exact identifier (MC#, DOT#,
  VIN, unit, email, plate) → exact name → token-set Levenshtein similarity.
  Threshold is per-call but reads `LOADS_IMPORT_AUTO_THRESHOLD` (default 0.85) at
  commit time. Brokers are tenant-global (FN-205); drivers/vehicles are scoped to
  `tenant_id` and softly to `operating_entity_id` (matches IS NULL OR =).
- **Idempotency uses batch.status as the gate.** A re-commit with status='committed'
  returns the cached `result_summary` JSONB. The transaction itself handles
  partial-failure rollback via per-row try/catch that converts known errors to
  row-level `validation_status='error'` records but lets unexpected throws
  propagate to ROLLBACK.
- **Cache shares the existing `load_ai_extractions` table.** The legacy
  column name `pdf_hash` is re-used to store the spreadsheet SHA-256 hex; the
  `extraction_method` column distinguishes `'spreadsheet-import'` from PDF
  rows. The `(tenant_id, pdf_hash)` UNIQUE plus a 7-day window in the SELECT
  give us the spec's TTL for free, and an `INSERT … ON CONFLICT DO UPDATE`
  refreshes stale entries.
- **Pure helpers extracted to `loads-import-mapper.js`** so they can be
  unit-tested without pulling in `@aws-sdk/client-s3` and `pg` transitively.
  Service still re-exports them for callers that want one import path.
- **Multer config is import-specific.** Reused the loads.js multer pattern but
  with allowed types {csv, xls, xlsx, txt} and 15 MB limit, leaving the
  existing PDF-only multer untouched.

## Deployment Handoff

### FN-1589 (subtask)
- **DB migration to run on deploy**: `20260508150000_create_load_import_tables.js`
  via `npm run migrate:latest` from `backend/packages/goodmen-database` (or
  the per-env aliases `migrate:dev` / `migrate:prod`).
- **No service restart required for this subtask alone** — tables are unused
  until FN-1590 ships the routes. When the full story (FN-1584) deploys, restart
  any service that imports `@goodmen/shared` and reads `load_import_*` (logistics-service,
  gateway). See `.agent/docs/render_services.md` for service names.
- **No env vars introduced.** FN-1590 will add `LOADS_IMPORT_AUTO_THRESHOLD`.
- **No data backfill** — both tables start empty.
- **Rollback**: `knex migrate:rollback` is safe; `down()` drops both tables in
  child-then-parent order. Verified locally.

### FN-1590 (subtask)
- **Service to redeploy**: `fleetneuron-logistics-service` (loads new
  `loads-import-service`, `loads-import-mapper`, and `fuzzy-match-service`
  modules; routes are mounted off the existing loads router at startup).
  No gateway redeploy is required — `app.use('/api/loads', …)` already
  covers `/api/loads/import/*`.
- **No DB migrations introduced by this subtask.** Depends on the FN-1589
  migration `20260508150000_create_load_import_tables.js` already shipped on
  the integration branch.
- **New env vars (logistics-service)**:
  - `LOADS_IMPORT_AUTO_THRESHOLD` (optional, default `0.85`) — fuzzy-match
    confidence floor at which a row commits to status='NEW'; below the
    threshold the row commits as DRAFT + `needs_review=true`.
  - `AI_SERVICE_URL` (already required elsewhere; preview calls
    `${AI_SERVICE_URL}/api/ai/loads/spreadsheet-import` with a 15 s timeout).
  - R2 vars (`R2_ACCOUNT_ID`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`,
    `R2_BUCKET`) — already required by the existing fuel-import / load-PDF
    flows; uploads use prefix `loads/import/{tenantId}/`.
- **No frontend or feature flags.** UI is a separate sub-task.
- **Cache invalidation**: nothing to evict. `load_ai_extractions` rows for
  the new `extraction_method='spreadsheet-import'` accumulate organically
  with the same 7-day TTL the PDF cache uses.
- **Post-deploy verification**:
  1. `curl -X POST -F file=@sample.csv $GATEWAY/api/loads/import/preview`
     with a valid bearer token returns `{ success: true, data: { batchId, … } }`.
  2. `SELECT id, status, row_count FROM load_import_batches ORDER BY
     created_at DESC LIMIT 5` shows the new batch row in `'pending'` status.
  3. POSTing `/api/loads/import/stage` with the AI-suggested mapping
     transitions the batch to `'staged'` and populates `load_import_rows`.
  4. POSTing `/api/loads/import/commit/:batchId` once inserts loads + stops;
     a second POST returns `{ idempotent: true }` without inserting.

## Open Items
- The `loads` schema has no customer FK — only `broker_id`. Confirmed during
  FN-1590 implementation: still no customer table, no follow-up needed.
- Multi-stop pattern `multi_row` is partially implemented (groupByValue is
  recorded per-row in `load_import_rows.normalized_values._stops_hint` but
  commit treats every row as a separate load). True multi-row grouping
  (where N rows compose one load with N stops) is deferred — most spreadsheet
  formats observed in intake use the `extra_columns` or `single` pattern.
  Track follow-up if a customer file requires it.
- `pickup2_*`/`delivery2_*` `extra_columns` support handles up to two pickups
  + two deliveries. Files with >2 of either type fall through to manual
  review. Extending is a one-loop change in `loads-import-mapper.buildStopsFromRow`.
