# FN-1730 — Trial signup: password set at signup fails at login ("Invalid credentials")

**Type:** Bug
**Agent:** backend
**Branch:** `backend/FN-1730/trial-signup-login-fix`
**Affected service:** `fleetneuron-auth-users-service` (+ minor `fleetneuron-logistics-ui`)

## Root Cause (confirmed)

The credential write/read path (bcrypt hash → `users.password_hash` → `bcrypt.compare`) is correct. The
failure is in **login identifier resolution**, in `internal/user.js:getUserByUsername()`:

```sql
-- before
SELECT * FROM users WHERE username = $1 OR LOWER(email) = LOWER($1) LIMIT 1
```

Trial-signup usernames are normalized to **lowercase** and may be **uniqueness-mutated** (a numeric
suffix appended on collision — `generateUniqueUsername`). Two defects in the lookup made the password
appear wrong:

1. **Case-sensitive username match.** `username = $1` is exact. The stored username is always lowercase,
   so a user typing their username with any other casing matched **no row** → `getUserByUsername` returns
   `undefined` → `401 Invalid credentials`. (Reproduced: test "matches the username case-insensitively".)
2. **Non-deterministic OR-match.** With `username = $1 OR LOWER(email) = LOWER($1)` + `LIMIT 1` and no
   ordering, when the typed value matched **one row by username and a different row by email**, Postgres
   could return the wrong row. `bcrypt.compare` then ran against the wrong hash → `401 Invalid credentials`.
   (Reproduced: test "prefers the username match over a different row that collides by email".)

`is_active` was **not** the cause here (its column default is already `true`), but it was being relied on
implicitly — AC4 requires it set explicitly so a future default change can't silently lock out new trials.

## Fix

1. **`backend/packages/goodmen-shared/internal/user.js`** — `getUserByUsername` now matches the username
   **case-insensitively** (`LOWER(username) = LOWER($1)`) and, when more than one row matches, resolves the
   row in JS with a stable preference order: **exact username → case-insensitive username → email**. This
   makes the typed identifier deterministically land on the intended account. `LIMIT 1` was removed so all
   candidate rows are visible to the resolver; a blank identifier short-circuits to `undefined`.
2. **`backend/packages/goodmen-shared/services/trial-request-service.js`** — the trial-admin `users` insert
   now sets `is_active: true` explicitly (AC4).
3. **Frontend `public-trial-signup` success screen** — the created username (already returned by the API and
   shown in a chip) is now clearly labelled "Your username" with copy telling the user to sign in with the
   username **or** their email plus the password they set (AC3).
