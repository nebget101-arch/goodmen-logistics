# FN-1068: Loads Dashboard — Load # and Actions columns don't resize after FN-1059

## Type
Bug (under Epic FN-723: Loads Page Redesign)

## Requirement
After FN-1060 (PR #635, commit `fb5c9b5`) shipped resizable column widths on the Loads Dashboard, manual smoke shows resize works on every column EXCEPT `Load #` (`load_number`) and `Actions`. Both columns have no drag handle and stay pinned to their default width. Restore drag-handle parity for these two columns.

## Background
- **Component**: `frontend/src/app/components/loads-dashboard/loads-dashboard.component.{html,ts}`
- **Source of bug**: FN-1059 spec at `docs/stories/FN-1059.md:15` stated that `load_number` and `actions` are "alwaysVisible and not resizable", conflating column-visibility lock (`alwaysVisible` / FN-767) with resizability (FN-1059). The implementer correctly followed the spec.
- **Code state (verified)**:
  - `loads-dashboard.component.html` lines 728, 741, 871, 884: `<col>` elements for both columns already bind `[style.width.px]="getColWidth('load_number'|'actions')"` — width plumbing is wired for both the sticky header and the virtualized body table.
  - Lines 754 and 767: the `<th>` elements for `load_number` and `actions` deliberately omit the `[appResizableColumn]` directive plus `currentWidth` / `widthChange` bindings. An inline comment confirms this was intentional.
  - `loads-dashboard.component.ts` already seeds `columnWidthDefaultsPx` for both keys (`load_number: 110`, `actions: 90`) and exposes `COL_WIDTH_MIN_PX` / `COL_WIDTH_MAX_PX` and `onColumnWidthChange()`.
  - `resizable-column.directive.ts` `stopPropagation()`s mousedown/click on the divider only — the existing `(click)="toggleSort('load_number')"` continues to fire on header-text clicks.

## Implementation Summary
The two `<th>` elements for `load_number` and `actions` did not carry the `appResizableColumn` directive — they had been omitted in FN-1060 because the FN-1059 spec wrongly conflated `alwaysVisible` (visibility-lock in the column picker) with resizability. The fix adds the same five bindings already present on the other 12 column headers: `[appResizableColumn]`, `[currentWidth]="getColWidth(...)"`, `[minWidth]="COL_WIDTH_MIN_PX"`, `[maxWidth]="COL_WIDTH_MAX_PX"`, and `(widthChange)="onColumnWidthChange(..., $event)"`. The existing `(click)="toggleSort('load_number')"` is preserved on the Load # header — the directive `stopPropagation()`s mousedown/click on the divider only, so header-text clicks still fire the sort handler. No `.ts` or `.scss` changes were needed: width getters, min/max constants, the `onColumnWidthChange` handler, and the `<col>` `[style.width.px]` plumbing for both keys are already in place from FN-1060.

The FN-1059 spec was also corrected so the same defect can't be re-introduced: Background note now states explicitly that `alwaysVisible` does not gate resizability, and AC1 was rewritten as "Every column header (including pinned columns) exposes a draggable resize handle".

## Key Decisions
- **Reuse the existing directive verbatim, no signature change.** The directive already exposes everything needed (`colKey`, `currentWidth`, `minWidth`, `maxWidth`, `widthChange`); the bug was purely a missing usage in the template, not a directive limitation.
- **Keep `(click)="toggleSort('load_number')"` on the Load # `<th>`.** The directive injects the resize handle as a child element and stops propagation on the divider's mousedown/click only — the host `<th>` sort click handler continues to fire on header-text clicks, matching the behavior already shipped on `pickup_date`, `rate`, and `completed_date`.
- **`actions` does not get a sort click handler.** The Actions column is action-only (no field to sort on); preserves parity with the prior template.
- **No backend / persistence change.** `LoadsDashboardPrefs.columnWidths` is keyed by column name as a free-form `Record<string, number>` — `load_number` and `actions` round-trip without any schema or route change. Defaults come from `columnWidthDefaultsPx` which already seeds both keys (`load_number: 110`, `actions: 90`).
- **Single-agent flow off `origin/dev` (no integration branch).** Per the parent-bug instructions: this is a one-file frontend fix plus two doc edits, with the QA cypress work tracked separately under FN-1070 against `origin/dev` after this merges.
