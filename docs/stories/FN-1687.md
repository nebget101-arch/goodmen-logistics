## Requirement

Trial/payment state is tracked on `tenants` but never **enforced**: an
expired-no-card tenant, or a tenant past `payment_grace_period_end`, is not
actually blocked. Add enforcement middleware, ship the deferred payment-failure
email (the `FN-76` TODO in the Stripe webhook), and repair the trial-reminder
script.

## Implementation Summary

### FN-1694 (subtask)

**1. Enforcement middleware** â€” `goodmen-shared/middleware/trial-enforcement-middleware.js` (new).
A factory mirroring `plan-access-middleware`: a 60s read-through cache of each
tenant's `{trial_status, payment_grace_period_end, stripe_payment_method_id}`, so
the hot path is one cached lookup. Blocks with **402 Payment Required** and a
machine-readable `code`:
- `PAYMENT_PAST_DUE` â€” `now > payment_grace_period_end`.
- `TRIAL_EXPIRED_NO_PAYMENT_METHOD` â€” `trial_status='expired'` and no card.

Response body is only `{ success, error, code, action }` â€” no dates/plan/Stripe
ids (no info leak). Exemptions: super_admin/platform_admin (`req.context.isGlobalAdmin`,
role fallback), billing/stripe path prefixes (so a blocked tenant can still pay),
and unauthenticated/public requests (no tenant context). **Fails open** on a
lookup error â€” a transient DB blip must not 402 every paying tenant. Unit-tested
(`trial-enforcement-middleware.test.js`, 12 cases).

**Wiring** â€” inserted as `requireActiveSubscription` right after
`tenantContextMiddleware` in every authenticated route group of the five services
(after tenant context, before plan checks). Public routes (`/api/track`,
`/public/*`, the unauthenticated `/api/address`) deliberately keep no enforcement.

**2. Payment-failure email** â€” `goodmen-shared/services/billing-email-service.js`
(new) sends via `@sendgrid/mail` using the same inline-HTML + graceful-no-op +
never-throw conventions as `trial-request-email-service.js`. The
`invoice.payment_failed` handler in `auth-users-service/routes/stripe.js` now calls
`sendPaymentFailureEmail` with the 3-day grace deadline (one send per failed
invoice), replacing the `FN-76` log stub.

**3. Reminder script** â€” `backend/scripts/sendTrialReminders.js` rewritten. The old
version was unrunnable (JOINed a non-existent `trial_status` *table*, MySQL
`INTERVAL` syntax, columns/tables that were never created, and required modules
that don't exist). Now a one-shot CLI on the shared knex client querying the real
`tenants` trial columns: 7/3/1-day "ending soon" reminders (active trials with no
card) + a "trial ended" notice, through `billing-email-service`. `--dry-run`
supported.

## Key Decisions

- **402 Payment Required** (not 403) â€” the block is a billing state, and the AC
  allows 402/403; 402 is the most semantic and lets the frontend branch cleanly.
- **Fail open** on lookup error â€” availability over strict enforcement; matches
  `plan-access-middleware`'s tolerance and avoids a DB hiccup locking out paying
  customers.
- **New `billing-email-service` instead of the legacy `backend/emails/trialEmailService.js`** â€”
  the legacy module can't load (its `./emailService` dependency exists nowhere in
  the repo) and is wired into nothing. The new service uses the established
  SendGrid pattern. The legacy email stack was left untouched (out of scope).
- **Reminder dedup without schema change** â€” exact 7/3/1-day buckets mean a daily
  run notifies each tenant at most once per threshold; no migration, honoring "no
  new tables". Data-expiry reminders were dropped (no schema backing).
- **Per-group wiring** (not a global `/api` gate) â€” a global gate would have
  re-run auth on public routes (e.g. drivers-compliance `/api/address`); per-group
  insertion is zero-regression and matches the plan-access pattern.

## Deployment Handoff

- **No DB migration. No new tables.** Operates entirely on existing `tenants`

  `node backend/scripts/sendTrialReminders.js --dry-run` first.
- **No feature flags. No cache invalidation** (the 60s middleware cache self-expires;
