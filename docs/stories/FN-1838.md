# FN-1838 — Fillable (AcroForm) PDF: AI field detection returns 0 fields

**Type:** Bug · **Parent epic:** FN-1786 (Customer-uploaded agreements: AI field detection + in-house e-signature) · **Agent:** ai · **Branch:** `ai/FN-1838/acroform-field-detection`

## Problem

`POST /api/ai/agreements/detect-fields` (`agreement-fields-handler.js`) is vision-only.
A genuine **fillable PDF** (AcroForm) has empty widget boxes with no visible ink, so
Claude vision sees a blank page and returns `fields: []` / `pageCount: 0` → the Review
Detected Fields screen shows "All 0 detected fields". The epic premise ("typically
scanned, no form layer") doesn't hold for user-supplied fillable docs.

## Fix

Add a deterministic **AcroForm fast-path** ahead of the vision call:

1. **Obtain PDF bytes** — base64 input is decoded locally; `fileUrl` input (the
   agreement service passes a signed R2 URL with **no** `contentType`) is downloaded
   with a capped, timed `axios` GET. A `%PDF-` magic-byte check gates everything; known
   image content-types skip the work entirely.
2. **Extract the form layer** — `src/lib/acroform-extractor.js` walks each page's widget
   annotations via `pdf-lib`, reading every form field's fully-qualified name, type
   (`FT` + name heuristics), 1-based page, and widget rectangle.
3. **Map to the contract** — `{ key, label, type, page, bbox, suggestedRole,
   suggestedValue, confidence }` with `confidence: 1.0` (directly extracted). Role is
   inferred internal-vs-signer from the field name/label (lessor/company → internal;
   lessee/driver/signer → signer; signature/initials default to signer).
4. **Vision fallback** — if there's no form layer (scanned/flat doc) the existing
   Claude-vision path runs unchanged.

### Coordinate convention

AcroForm bboxes are emitted in **top-left PDF points** per
`docs/design/agreements-bbox-coordinates.md` — the canonical convention the placement
editor (FN-1807) draws in and the signed-PDF overlay (FN-1797) consumes. PDF widget
rects are bottom-left origin, so the extractor flips the y-axis: `y_top = pageHeight −
y_bottom − height`. `normalizeDetection` gained a `clampBbox` option: the vision path
still clamps to `[0,1]` (it emits normalized coords), the AcroForm path clamps only the
lower bound to 0 so points survive.

> Note: the vision prompt asks the model for *normalized 0..1* bboxes, which contradicts
> the canonical *points* convention the rest of the agreements stack uses. That is a
> pre-existing inconsistency in the vision path (FN-1791) and is **out of scope** for
> this bug (AC: "no regression" to scanned-doc vision). The AcroForm path follows the
> documented points convention, which is what AC #5 requires.

## Key Decisions

- **pdf-lib over pdfjs** — already a repo-standard dependency (`^1.17.1` in 6 services,
  including the `goodmen-shared` package this service depends on); pure-JS AcroForm read.
- **Low-level annotation walk** — iterating each page's `Annots` (rather than
  `form.getFields()`) yields the correct page index per widget for free and handles
  per-page repeated fields (initials), inherited `FT`/`Ff`, and fully-qualified names.
- **Download bytes for the `fileUrl` path** — the service sends `{ fileUrl }` with no
  contentType, so the handler must fetch + sniff the bytes itself to know it's a PDF.
  Capped at `MAX_FILE_BYTES`, 8s timeout, failures swallowed (degrade to vision).
- **`confidence: 1.0`** per the ticket; the hallucination guard leaves it untouched
  because extracted type/role are always valid enum values.
