# FN-1605: [Recommend Driver] Suggest-driver always returns "AI service unavailable" with empty candidates on dev

## Requirement
The Suggest-driver feature on the load drawer (FN-1431) is unusable on dev — every call to `POST /api/loads/:id/recommend-driver` returns `{ success: true, candidates: [], reasoning: "AI service unavailable" }`. We need to (a) diagnose the actual failure mode on dev infra, (b) fix it, and (c) make the response distinguish real failure modes so users get accurate, actionable reasoning instead of a single misleading string.

## Reproduction
- URL: `POST https://fleetneuron-logistics-gateway-dev.onrender.com/api/loads/3230a7ef-395e-4dc2-953a-48e4c74c4c46/recommend-driver`
- Status 200, body:
  ```json
  { "success": true, "candidates": [], "reasoning": "AI service unavailable" }
  ```
- Triggered from the recommend-driver modal on the load detail drawer (Driver tab → Suggest driver).

## Root-cause hypotheses
The string `"AI service unavailable"` is hard-coded in two branches of `backend/packages/goodmen-shared/routes/loads.js` `/:id/recommend-driver`:
- ~line 4598 — non-2xx response from AI service (any 4xx or 5xx)
- ~line 4611 — `fetch` throws (timeout/abort/network)

Possible real causes that all collapse to that one string:
1. AI 4xx because the load is missing `equipmentClass` in `ai_metadata` or has no PICKUP stop / no `zip_codes` row → AI handler validation rejects.
2. AI 5xx because `ANTHROPIC_API_KEY` missing/invalid on `fleetneuron-ai-service-dev`.
3. Cold-start timeout on the dev AI service (Render starter plan spin-down) exceeding `RECOMMEND_DRIVER_AI_TIMEOUT_MS`.
4. `AI_SERVICE_URL` on `fleetneuron-logistics-service-dev` unset (→ falls back to `http://localhost:4100`) or wrong (→ pointed at prod).

External probe of `https://fleetneuron-ai-service-dev.onrender.com/health` returns 200 and `/api/ai/loads/recommend-driver` returns proper 400 on empty body, so the service is reachable from outside Render.

## Implementation Summary

### FN-1606 (subtask, devops)

**Verdict: this is NOT an infrastructure problem.** Dev wiring is correct; the bug is logistics-side (sending `originLat: null` to the AI service).

#### 1. Render env audit — dev (no drift found, no `render.yaml` change required)

| Variable | Service | Observed value | Expected | Status |
|---|---|---|---|---|
| `AI_SERVICE_URL` | `fleetneuron-logistics-service-dev` | `https://fleetneuron-ai-service-dev.onrender.com` | dev AI URL | OK — matches `render-dev.yaml:39` |
| `ANTHROPIC_API_KEY` | `fleetneuron-ai-service-dev` | _set_ (108 chars, prefix `sk-a…`) | non-empty Anthropic key | OK — present |
| `ANTHROPIC_VISION_MODEL` | `fleetneuron-ai-service-dev` | `claude-sonnet-4-20250514` | any valid model | OK |
| `ANTHROPIC_LOAD_MATCH_MODEL` | `fleetneuron-ai-service-dev` | _unset_ | optional override | OK — falls back to `claude-sonnet-4-20250514` (handler line 268) |
| `RECOMMEND_DRIVER_AI_TIMEOUT_MS` | `fleetneuron-logistics-service-dev` | _N/A_ | _N/A_ | NOTE — constant, not env-overridable. Hard-coded `15000` at `loads.js:4371`. Recommend FN-1607 make this overridable + raise default to 30000 ms for cold-start headroom. |

The `render.yaml` file in the repo only configures the **prod** stack and correctly hardcodes the prod AI URL there. The **dev** stack is described in `render-dev.yaml`, which already points logistics-dev at the dev AI service. The dashboard env on `fleetneuron-logistics-service-dev` matches `render-dev.yaml`, so there is no drift to codify.

Both dev services are on `plan: starter` in `region: oregon` (same region — server-to-server latency ms-scale).

- Durations 69–88 ms ≪ 15 s timeout — **cold start was not a factor** in the observed failures.

#### 3. Failure mode (definitive)

**Type: AI 4xx (HTTP 400) from `fleetneuron-ai-service-dev`.**
The AI handler `validateRequestBody` at `backend/microservices/ai-service/src/handlers/load-driver-match-handler.js:84-86` rejects requests where `load.originLat` / `load.originLng` are not numbers.

The logistics-side query at `backend/packages/goodmen-shared/routes/loads.js` resolves origin lat/lng by joining the load's first PICKUP stop's `zip` against the `zip_codes` table. For load `3230a7ef-395e-4dc2-953a-48e4c74c4c46`, that join yields NULL — either no PICKUP stop, no `zip` on the stop, or no `zip_codes` row for that ZIP — and the route still calls AI with `originLat: null`. AI returns 400, logistics masks it as `"AI service unavailable"`.

#### 4. Handoff to FN-1605.backend (FN-1607)

This subtask is unblocking. The required code-side fix:

1. **Pre-flight check** before the AI fetch (around `loads.js:4563`): if `originLat == null || originLng == null`, short-circuit with a specific reasoning string such as `"Load is missing pickup location coordinates — add a PICKUP stop with a known ZIP."` Same pattern for missing `equipmentClass`.
2. **Distinguish failure modes** in the existing two error branches:
    - At line 4591 (AI non-2xx): emit reasoning `"AI service rejected the request (HTTP <status>): <error>"` parsed from the AI response body's `error` field, instead of `"AI service unavailable"`. (The AI service already returns a structured `{ error, code }` JSON — surface it.)
    - At line 4605 (fetch threw): distinguish `AbortError` / `TimeoutError` (→ `"AI service timed out — please retry."`) from other network errors (→ `"Could not reach the AI service — check connectivity."`).
3. **Make timeout env-overridable**: change `const RECOMMEND_DRIVER_AI_TIMEOUT_MS = 15000` to read `parseInt(process.env.RECOMMEND_DRIVER_AI_TIMEOUT_MS, 10) || 30000`. 30 s default leaves headroom for a Render starter-plan cold start.

No env-var change is needed on Render dashboards as part of FN-1607 — keep the new env override unset on dev/prod and let it fall through to the new 30 s default.

### FN-1607 (subtask, backend)

Implemented the three-part fix the FN-1606 audit handed off:

#### Pre-flight short-circuits (`loads.js:4443`–`~4475`)
Before the candidate-pool query and the AI roundtrip, the route now returns `{ success: true, candidates: [], reasoning: <specific> }` and emits a `loads_recommend_driver_skipped` info log when:
- `loadEquipmentClass` is null/empty → `"Load has no equipment class set. Set the equipment class on the load to enable driver suggestions."` (`reason: no_equipment_class`).
- `originLat == null || originLng == null` → `"Pickup location is not yet geocoded. Add a PICKUP stop with a US zip code to enable driver suggestions."` (`reason: no_origin_geocode`). This is the one that fires on the originally-reported load `3230a7ef-395e-4dc2-953a-48e4c74c4c46`.

Both checks run before the candidate-pool SQL — no wasted DB roundtrip.

#### AI failure-mode distinction (`loads.js:~4595`–`~4660`)
The catch + non-2xx branches now emit four distinct reasoning strings:

| Trigger | Reasoning | Log key |
|---|---|---|
| AI 4xx with `code` starting `AI_BAD_REQUEST*` | `Suggestion request was malformed: <error from AI body>` | `bug:loads_recommend_driver_ai_http` (loud — pre-flight should make this unreachable) |
| AI 4xx without that code (auth/rate-limit/etc.) | `AI service rejected the request (status <code>). Try again in a moment.` | `loads_recommend_driver_ai_http` |
| AI 5xx | `AI service is currently failing (status <code>). Try again in a moment.` | `loads_recommend_driver_ai_http` |
| Fetch throws `AbortError` / `TimeoutError` | `AI service is taking too long to respond. Try again in a moment.` | `loads_recommend_driver_ai_fetch_failed` |
| Fetch throws other (DNS / network) | `AI service is unreachable from logistics-service. Check AI_SERVICE_URL configuration.` | `loads_recommend_driver_ai_fetch_failed` |

The string `"AI service unavailable"` no longer appears anywhere in the route — `grep` confirms.

#### Log payload upgrades
- `loads_recommend_driver_ai_http`: snippet bumped to 500 chars (was 200), added `aiErrorCode` parsed from the AI response body when present. Field renamed to `aiPayloadSnippet` to match the description spec.
- `loads_recommend_driver_ai_fetch_failed`: added `errorName: fetchErr.name` so dashboards can split timeouts from real network errors without grep-matching messages.

#### Timeout env-overridable (`loads.js:4371`)
`RECOMMEND_DRIVER_AI_TIMEOUT_MS` is now read from `process.env.RECOMMEND_DRIVER_AI_TIMEOUT_MS` (positive number) with a 30 000 ms default — adopting FN-1606's recommendation to leave headroom for a Render starter-plan cold start.

#### Tests (`loads-recommend-driver.test.js`)
- All 7 pre-existing test cases still pass; the two that asserted `'AI service unavailable'` now assert the new mode-specific strings.
- Added 5 FN-1607 cases:
  1. Pre-flight on missing `equipmentClass` (verifies AI not called + candidate-pool query not run).
  2. Pre-flight on null `origin_lat` / `origin_lng` (same).
  3. AI 4xx with `code: AI_BAD_REQUEST_INVALID_PAYLOAD` → `"Suggestion request was malformed: …"`.
  4. AI 5xx (502) → `"AI service is currently failing (status 502). Try again in a moment."`
  5. `fetch` throws `AbortError` → `"AI service is taking too long to respond. Try again in a moment."`

Run: `node --test routes/loads-recommend-driver.test.js` → 12/12 pass.

## Key Decisions

- **No `render.yaml` patch.** The dev IaC (`render-dev.yaml`) is already correct; the deployed env matches it. Patching `render.yaml` would have been wrong (that file is the prod blueprint and correctly points at the prod AI service).
- **Recommend env-overridable timeout in FN-1607.** Constant rename + default raised to 30 s. Investigation showed the current 15 s default has not been the cause of dev failures, but a starter-plan cold start could plausibly exceed 15 s in a dispatcher-blocking moment, so we treat this as a low-risk hardening.
- **FN-1607: pre-flight runs before the candidate-pool SQL, not after.** The natural place to put pre-flight checks is alongside the existing `candidatePool.length === 0` guard (just before the AI fetch), but that wastes a candidate-pool query that we know we won't use. Moved both checks to immediately after the load row is fetched and decoded, before the candidate-pool SQL.
- **FN-1607: 4xx without `AI_BAD_REQUEST*` gets its own bucket.** The handoff named three failure modes (4xx malformed / 5xx / timeout / network), but the AI service can also produce non-`AI_BAD_REQUEST` 4xx (auth, rate-limit, unknown route). Treating those as 5xx-style ("try again") rather than as malformed requests avoids misleading dispatchers when the cause is an upstream auth issue, not a bad payload.
- **FN-1607: `bug:` prefix on the route key only when `code` matches `AI_BAD_REQUEST*`.** The `bug:` prefix is reserved for the case the description called out — the pre-flight should have caught this, so seeing it in logs means the pre-flight has a gap. Other 4xx and 5xx use the plain log key so dashboards continue to aggregate routine upstream blips.
