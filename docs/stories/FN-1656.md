### FN-1672 (subtask) — Vehicle-positions read endpoint + WebSocket broadcast
Backend half of the live map: the read API that seeds the marker layer and breadcrumb trails, plus the live `vehicle:position` broadcast that keeps markers moving.

**Endpoints** (`backend/packages/goodmen-shared/routes/vehicle-positions.js`, mounted by `vehicles-maintenance-service` at `/api/vehicle-positions`, gateway-proxied):
- `GET /api/vehicle-positions` — latest ping per vehicle (the marker layer). One row per vehicle with a ping in the last `LOOKBACK_HOURS` (24h). `{ data, meta }` envelope. Filters: `status`, `driverId`, `vehicleIds` (comma list), `bbox` (`minLng,minLat,maxLng,maxLat`), `geofenceId` (only vehicles currently inside that tenant geofence). Each row: `vehicleId, unitNumber, make, model, year, vehicleType, status, driverId, lat, lng, speedMph, headingDeg, ts, lastPingAgeSeconds`.
- `GET /api/vehicle-positions/:vehicleId/breadcrumbs?hours=4` — chronological trail for one vehicle (hover layer). Default 4h, capped 24h, row-capped. Each row: `lat, lng, speedMph, headingDeg, ts`.

**Live broadcast**:
- `websocket.service.emitVehiclePosition({ tenantId, position })` — new domain helper wrapping `emitToTenant` with the `vehicle:position` event and a normalized lean payload (`vehicleId, lat, lng, speedMph, headingDeg, ts`).
- Wired into `integrations-service/services/telematics-ingest-service.js`: after a ping is persisted, `broadcastPing` resolves the vehicle's tenant (memoized via `vehicles.tenant_id`) and fires the emit **fire-and-forget** — never awaited by, and never able to fail, ingestion.

**Key design choices**:
- **Tenant boundary is `vehicles`, not the ping table.** `vehicle_position_pings` (FN-1660) carries no `tenant_id` (write-optimized ingest), so the endpoint resolves the tenant's vehicles first and constrains the ping read to those ids.
- **Latest-per-vehicle via `DISTINCT ON (vehicle_id)` bounded to the last 24h** — touches only the newest day-partitions and rides the `(vehicle_id, ts DESC)` index, keeping the marker read cheap at 500 vehicles (relevant to FN-1673's perf check). A vehicle silent >24h drops off the live map.
- **Breadcrumbs are per-vehicle on hover**, not fetched for the whole fleet, so the marker read never pulls the high-volume trail rows.
- **`driverId` filter matches `vehicles.leased_driver_id`** (the only vehicle→driver column available today). The side panel's richer driver/current-load enrichment is the frontend's job (FN-1671), which already has those services — the endpoint stays lean and fast.
- **Geofence filter reuses `geofence-service.geofenceContainsPoint`** (app-side, no PostGIS) against the stored GeoJSON row, fetched directly because the wire geofence from `getGeofence()` drops the geometry field.

### FN-1671 (subtask) — Live map page
Built the `/tracking` live map as a lazy-loaded feature module (`TrackingMapModule`), mirroring the sibling `GeofencesModule` (FN-1654) for Leaflet isolation and AI dark-theme tokens. Wired against the **FN-1672** contract above (verified against the merged backend code, not assumptions).

**What shipped**
- **Map**: Leaflet with `preferCanvas` + `leaflet.markercluster` clustered vehicle layer (`addLayers` bulk path + `chunkedLoading`) so 500 vehicles pan/zoom smoothly.
- **Markers**: status-coloured `divIcon` (moving = green, idle = amber, offline = slate), rotated by `headingDeg` when present.
- **Movement status is derived client-side** (`moving` / `idle` / `offline`) from `speedMph` + ping age — the backend `status` field is the *vehicle* status, not movement, so the status filter and marker colour compute locally.
- **Live updates**: subscribes to the `vehicle:position` WS event (lean `{ vehicleId, lat, lng, speedMph, headingDeg, ts }` payload) and *merges* each ping into the existing marker (keeps metadata, updates geometry + recomputes movement status). A header dot reflects `WebsocketService.status$`; `pollTick$` triggers a full refetch when the socket falls back to polling.
- **Hover breadcrumbs**: marker `mouseover` fetches the last-4h trail (`GET /vehicle-positions/:id/breadcrumbs?hours=4`) and draws a dashed polyline; cleared on `mouseout`.
- **Geofence overlay**: reuses `GeofenceService.list()` (FN-1654) to render circles/polygons as a toggleable overlay layer.
- **Side panel**: click → vehicle (unit number), driver (resolved from the active-driver list since the payload carries only `driverId`), current load (best-effort lookup via `listLoads({ driverId })` for an active-status load, + deep link to `/loads?loadId=`), last-ping age, speed, heading, position. Collapses to a bottom sheet under 1120px (map fills the viewport).
- **Filter bar**: driver / status / geofence via `app-ai-select`. Filtering is client-side (driver + movement-status equality; geofence containment via haversine for circles, ray-cast for polygons), so live pings re-filter without a refetch. The backend `list()` filters are still wired for future use.

**FN-317 compliance**: all three `[options]` bindings (`statusOptions`, `driverOptions`, `geofenceOptions`) are plain class fields — `driverOptions`/`geofenceOptions` are reassigned once on data load (stable references between CD passes), never getters.

## Deployment Handoff

### FN-1672
- **Services to redeploy** (Render):
  - `goodmen-vehicles-maintenance` — hosts the new `/api/vehicle-positions` read endpoints.
  - `goodmen-gateway` — new `/api/vehicle-positions` proxy route.
  - `goodmen-integrations` — telematics ingest now emits the live `vehicle:position` broadcast.
  - `goodmen-shared` is a library (no own service) — its consumers above pick up the change on redeploy.
- **DB migrations**: none new. Depends on FN-1660's `vehicle_position_pings` (+ `telematics_devices`, `vehicles`), already on `dev`.
- **Env/config**: no new vars. The live broadcast uses the existing WS bridge (`INTERNAL_WS_EMIT_URL`, `INTERNAL_WS_SECRET`) on the **integrations** service — if unset there, the broadcast silently no-ops (markers still update on the next manual refresh / poll). Confirm both are set on `goodmen-integrations` for live updates.
- **Deploy order**: shared consumers first (vehicles-maintenance, integrations), then gateway. No strict ordering required — routes are additive and backward-compatible.
- **Post-deploy verification**:
  1. `GET /api/vehicle-positions` (authed) returns `{ data, meta }`; `data` has one row per recently-pinged vehicle, all tenant-scoped.
  2. `GET /api/vehicle-positions/:vehicleId/breadcrumbs?hours=4` returns an ascending trail for a vehicle in the tenant; `404` for a foreign vehicle.
  3. With the WS bridge configured, ingesting a ping (webhook/poll) pushes a `vehicle:position` event to the tenant room.
