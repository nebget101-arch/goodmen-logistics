### FN-1672 (subtask) ‚Äî Vehicle-positions read endpoint + WebSocket broadcast
Backend half of the live map: the read API that seeds the marker layer and breadcrumb trails, plus the live `vehicle:position` broadcast that keeps markers moving.

**Endpoints** (`backend/packages/goodmen-shared/routes/vehicle-positions.js`, mounted by `vehicles-maintenance-service` at `/api/vehicle-positions`, gateway-proxied):
- `GET /api/vehicle-positions` ‚Äî latest ping per vehicle (the marker layer). One row per vehicle with a ping in the last `LOOKBACK_HOURS` (24h). `{ data, meta }` envelope. Filters: `status`, `driverId`, `vehicleIds` (comma list), `bbox` (`minLng,minLat,maxLng,maxLat`), `geofenceId` (only vehicles currently inside that tenant geofence).
- `GET /api/vehicle-positions/:vehicleId/breadcrumbs?hours=4` ‚Äî chronological trail for one vehicle (hover layer). Default 4h, capped 24h, row-capped.

**Live broadcast**:
- `websocket.service.emitVehiclePosition({ tenantId, position })` ‚Äî new domain helper wrapping `emitToTenant` with the `vehicle:position` event and a normalized lean payload (`vehicleId, lat, lng, speedMph, headingDeg, ts`).
- Wired into `integrations-service/services/telematics-ingest-service.js`: after a ping is persisted, `broadcastPing` resolves the vehicle's tenant (memoized via `vehicles.tenant_id`) and fires the emit **fire-and-forget** ‚Äî never awaited by, and never able to fail, ingestion.

**Key design choices**:
- **Tenant boundary is `vehicles`, not the ping table.** `vehicle_position_pings` (FN-1660) carries no `tenant_id` (write-optimized ingest), so the endpoint resolves the tenant's vehicles first and constrains the ping read to those ids.
- **Latest-per-vehicle via `DISTINCT ON (vehicle_id)` bounded to the last 24h** ‚Äî touches only the newest day-partitions and rides the `(vehicle_id, ts DESC)` index, keeping the marker read cheap at 500 vehicles (relevant to FN-1673's perf check). A vehicle silent >24h drops off the live map.
- **Breadcrumbs are per-vehicle on hover**, not fetched for the whole fleet, so the marker read never pulls the high-volume trail rows.
- **`driverId` filter matches `vehicles.leased_driver_id`** (the only vehicle‚Üídriver column available today). The side panel's richer driver/current-load enrichment is the frontend's job (FN-1671), which already has those services ‚Äî the endpoint stays lean and fast.
- **Geofence filter reuses `geofence-service.geofenceContainsPoint`** (app-side, no PostGIS) against the stored GeoJSON row, fetched directly because the wire geofence from `getGeofence()` drops the geometry field.

## Deployment Handoff

### FN-1672
- **Services to redeploy** (Render):
  - `goodmen-vehicles-maintenance` ‚Äî hosts the new `/api/vehicle-positions` read endpoints.
  - `goodmen-gateway` ‚Äî new `/api/vehicle-positions` proxy route.
  - `goodmen-integrations` ‚Äî telematics ingest now emits the live `vehicle:position` broadcast.
  - `goodmen-shared` is a library (no own service) ‚Äî its consumers above pick up the change on redeploy.
- **DB migrations**: none new. Depends on FN-1660's `vehicle_position_pings` (+ `telematics_devices`, `vehicles`), already on `dev`.
- **Env/config**: no new vars. The live broadcast uses the existing WS bridge (`INTERNAL_WS_EMIT_URL`, `INTERNAL_WS_SECRET`) on the **integrations** service ‚Äî if unset there, the broadcast silently no-ops (markers still update on the next manual refresh / poll). Confirm both are set on `goodmen-integrations` for live updates.
- **Deploy order**: shared consumers first (vehicles-maintenance, integrations), then gateway. No strict ordering required ‚Äî routes are additive and backward-compatible.
- **Post-deploy verification**:
  1. `GET /api/vehicle-positions` (authed) returns `{ data, meta }`; `data` has one row per recently-pinged vehicle, all tenant-scoped.
  2. `GET /api/vehicle-positions/:vehicleId/breadcrumbs?hours=4` returns an ascending trail for a vehicle in the tenant; `404` for a foreign vehicle.
  3. With the WS bridge configured, ingesting a ping (webhook/poll) pushes a `vehicle:position` event to the tenant room.
