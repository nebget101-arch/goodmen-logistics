# FN-1129: S5 — Contextual Quick Actions on Control Center

## Requirement
Each Smart Alert and each Predictive Insight card surfaces 1–3 contextual quick actions ("Reassign load", "Schedule maintenance", "Notify driver"). Actions deep-link to the appropriate flow with prefilled context.

## Acceptance Criteria
- [ ] Each alert/insight card shows up to 3 quick action buttons
- [ ] Buttons emit `(action)` event consumed by router/service
- [ ] Deep-link routes prefill context (load id, driver id, vehicle id) via query params
- [ ] Quick action visible only if user role has permission

### FN-1169 (subtask)

- Built `QuickActionsComponent` (`<app-quick-actions>`) as a standalone Angular 17 component. Inputs: `actions: QuickActionDef[]`, `context: Record<string, primitive>`, `ariaLabel`, `max` (default 3). Output: `(action) = QuickActionEvent`.
- Permission gating uses `AccessControlService.hasPermission` / `hasAnyPermission`. `requiredPermission` accepts a string or string-array; missing perms drop the action from the list before render. Cap is enforced AFTER filtering, so over-provisioned action arrays still render at most 3.
- Three render modes per action: `routerLink` → anchor + Angular `[routerLink]+[queryParams]`; `href` → external anchor; otherwise plain `<button>` (for emit-only actions consumed by services).
- Card-level `context` (e.g. `{ loadId: 'abc' }`) is merged into each action's `queryParams` so deep-links pre-fill IDs.
- Smart Alerts wiring (`smart-alerts.component.ts`): per-alert-type action sets — `late_load_risk` → Reassign load + Notify driver; `hos_imminent`/`fatigue` → Notify driver + Reassign load; `inspection_overdue` → Schedule maintenance. `subjectKind`+`subjectId` resolved into `loadId`/`driverId`/`vehicleId` context.
- Predictive Insights wiring (`predictive-insights.component.ts`): per-trend action sets — `maintenance` → Schedule maintenance + Open vehicles; `loadVolume`/`onTimePct` → Open loads + Create load; `fuelCost` → Open fuel.
- Real permission codes used: `loads.edit`, `loads.view`, `loads.create`, `drivers.edit`/`drivers.manage` (any-of), `work_orders.create`, `vehicles.view`, `fuel.view`. All exist in `frontend/src/app/models/access-control.model.ts`.
- Spec coverage (`quick-actions.component.spec.ts`): empty list, permission filtering (single + any-of), 3-action cap, click emits with merged context, unconditional action renders without permission. Note: this repo has no Karma test target wired in `angular.json` — specs compile during `ng build` but there is no `npm test` runner.

## Key Decisions
- **Permission gating client-side only is presentation, not security.** The component drops actions the user can't see, but the routes and downstream services must still enforce the same permissions. This is a contract for QA (FN-1170) to verify each deep-link returns `403`/redirects when accessed directly.
- **`requiredPermission` accepts a string OR `string[]`** — the array form is OR-semantics (`hasAnyPermission`). This matches existing patterns in `drivers.component.ts` and avoids forcing callers to invent compound permission strings.
- **Action emit fires alongside `routerLink` navigation** — the component lets the router handle navigation declaratively while still emitting `(action)` so future telemetry/service hooks don't need a refactor. Consuming components currently bind a no-op handler.
- **Card-level `context` merges INTO action `queryParams`** (action wins on key collision). This makes the typical "loadId from row + action='reassign'" deep-link composable without callers having to clone+spread per row.
- **3-action cap is enforced post-filter.** Callers can declare 5 actions with mixed permissions and the user sees at most 3 from the subset they qualify for — keeps the action list stable as roles change without callers re-prioritizing.
