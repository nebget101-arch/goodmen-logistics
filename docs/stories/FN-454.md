# FN-454: [Bug] Fuel AI preprocess fetch crashes with "fetch failed" when AI service is unreachable

## Requirement
Fix the unhandled `TypeError: fetch failed` in the fuel AI preprocess route when the AI service is unreachable (cold start, down, DNS failure). Add a 15-second timeout and return structured error responses so the frontend can gracefully fall back to manual mapping.

## Acceptance Criteria
- [x] `fetch()` has a 15-second `AbortController` timeout
- [x] Timeout returns 504 with code `AI_TIMEOUT`
- [x] Connection failure returns 502 with code `AI_UNREACHABLE`
- [x] Structured log messages distinguish timeout vs unreachable vs HTTP error
- [x] No regression: successful AI preprocess still works end-to-end
- [x] Manual mapping fallback still works when AI fails

## Agent
backend

## Implementation Summary
Wrapped the `fetch()` call to the AI service in an inner try/catch with an `AbortController` timeout (15s). The inner catch differentiates:
- `AbortError` → 504 + log `fuel_ai_preprocess_timeout`
- `TypeError` / other network errors → 502 + log `fuel_ai_preprocess_unreachable`
- HTTP errors from AI service → existing 400/502 handling preserved
- Unexpected errors → outer catch returns 500 (unchanged)

The `finally` block always clears the timeout to prevent leaks.

## Files Changed
- `backend/packages/goodmen-shared/routes/fuel.js` — ai-preprocess route (~line 265-296)

## Key Decisions
- Used `AbortController` + `setTimeout` instead of `fetch` `timeout` option (broader Node.js compatibility)
- 15-second timeout matches the AC in FN-413 QA spec ("AI step completes in < 15 seconds")
- Kept the outer try/catch for any non-fetch errors (e.g., JSON parsing of request body)
