## Implementation Summary

### FN-877 (subtask) — Stops sub-component + Trip Metrics panel

New standalone component `<app-load-wizard-stops>` renders controls bound to the parent wizard's `stops` FormArray and owns no form state of its own. Mirrors the legacy `StepStopsComponent` shape but upgraded for reactive FormArray binding and the v2 wizard's standalone-component architecture.

**Stops list**
- Per-row inline form (type, company, zip, city, state, address, date, time) bound directly to each `FormGroup` in `stops`.
- CDK `cdkDropList` / `cdkDrag` with `cdkDragHandle` — reordering calls `moveItemInArray` and re-syncs the FormArray via `setControl` so `valueChanges` fires with the new order, then renumbers `sequence`.
- `+ Add Stop` pushes a new `DELIVERY` FormGroup (default) at the end.
- Delete (`×`) button per row is disabled when removing would leave zero of that stop type (guards the ≥ 1 pickup + ≥ 1 delivery invariant). The parent wizard additionally gates `canProceed` on `stops.length >= 2 && stops.valid`.
- `mode='view'` calls `stops.disable({ emitEvent: false })` and flips `cdkDropListDisabled` / `cdkDragDisabled`, hiding the handle and delete button.

**Zip lookup**
- On `blur` of the zip input, calls `LoadsService.lookupZip(zip)` and `patchValue`s `city`/`state` on that row's FormGroup (emitting so metrics pick up the change). Per-row loading flag (`WeakMap<FormGroup, boolean>`) drives a small "Looking up…" hint.
- Skipped in view mode.

**Trip Metrics panel**
- Subscribes to `stops.valueChanges` (debounced 350ms) and recomputes on any change; also runs on `ngOnInit` and when the parent passes a new `rate`.
- Pipeline: collect non-empty zips (≥ 2 required) → `forkJoin(lookupZip)` to resolve lat/lon → `getRouteGeometry` via the backend OSRM proxy.
- **Total miles** — sum of Haversine segment distances across the route geometry polyline (accurate road-following total); falls back to waypoint-to-waypoint great-circle when the proxy is unavailable.
- **Loaded vs empty** — computed from waypoint sequence + stop types: a segment starting at a PICKUP is "loaded" until the next DELIVERY; everything else is "empty" (handles multi-pickup and multi-delivery loads). Scaled proportionally to the road-following total so the three numbers reconcile.
- **Rate/mile** — `rate / totalMiles`, rounded to two decimals. Fed from the parent via a numeric `ratePerMileInput` getter so OnPush CD keeps it in sync as the user edits the basics `rate` field.
- In-flight requests are cancelable via an internal `Subject` so rapid edits collapse to one backend call.

**Wizard wiring (`load-wizard.component.{ts,html}`)**
- Imports `LoadWizardStopsComponent` into the standalone wizard's imports array.
- Replaces the `*ngSwitchCase="'stops'"` placeholder `<section>` with `<app-load-wizard-stops [stops]="stops" [rate]="ratePerMileInput" [mode]="mode">`.
- Adds a `ratePerMileInput` getter that normalizes the basics `rate` FormControl value to `number | null`.

## Key Decisions

1. **Sub-component owns no form state.** The parent LoadWizardComponent owns the `stops` FormArray; the child renders it by reference. Avoids double-source-of-truth and keeps `canProceed` / submit payload logic in one place.
2. **Drag reorder reaches into `FormArray.controls`** and uses `moveItemInArray` followed by `setControl(previousIndex, …)` + `setControl(currentIndex, …)` to force the FormArray to re-emit. Directly mutating `_controls` would not fire `valueChanges`, which metrics recompute depends on.
3. **Route geometry over pure Haversine** for total miles — follows roads rather than straight-line distance. Falls back to Haversine only when the backend OSRM proxy returns no coordinates, so the panel still shows *something* in degraded mode.
4. **Loaded/empty from stop-type sequencing, not driver home base.** In the create flow the driver is often unassigned, so "empty" is the deadhead between a delivery and the next pickup (and any segment before the first pickup). Matches how dispatchers think about multi-leg loads.
5. **Debounce on metrics recompute (350 ms)** — same order of magnitude as the backend `getRouteGeometry` latency; avoids spamming OSRM while the user types zips.
