# FN-1042: Dispatcher name is blank on Basics tab of load detail drawer

## Type
Bug — Severity: Medium

## Symptom
Opening any existing load in the side drawer shows an empty, read-only Dispatcher field on the Basics tab.

## Confirmed Root Cause
- DB stores `loads.dispatcher_user_id` (FK to users).
- `getLoadDetail()` in `backend/packages/goodmen-shared/routes/loads.js` (lines 651–660) joins `drivers` and `brokers` but does NOT join `users`.
- `populateFromDetail()` in `frontend/src/app/components/loads-dashboard/load-detail-drawer/load-detail-drawer.component.ts:169` reads `(d as any).dispatcher_name` — undefined.
- `step-basics.component.ts:95–98` localStorage fallback uses *current* viewer's username, wrong for loads assigned to someone else.

## Acceptance Criteria
- [ ] Existing load assigned to user A, viewed by user B → drawer shows user A
- [ ] Brand-new wizard create → shows current user (existing behavior preserved)
- [ ] Loads list view also shows correct dispatcher name (no regression)
- [ ] Saving the load does not blank the dispatcher

## Implementation Summary

### FN-1047 (subtask)
Frontend follow-up. Three small changes:

1. **`LoadListItem.dispatcher_name`** added to `frontend/src/app/models/load-dashboard.model.ts`. `LoadDetail extends LoadListItem`, so the drawer/list both gain the typed field. Optional (`string | null`) since older payloads (and the current backend until FN-1046 ships) won't include it.
2. **Drawer cast removed.** `load-detail-drawer.component.ts:169` now reads `d.dispatcher_name` directly off the typed `LoadDetail`. No behavior change — when backend hasn't shipped, the field is still undefined and the existing `|| ''` fallback kicks in.
3. **`step-basics` mode input.** Added `@Input() mode: 'create' | 'edit' = 'create'`. The `localStorage` fallback in `ngOnInit` now only runs when `mode === 'create'`. The drawer passes `mode="edit"` so a load assigned to user A, opened by user B, will no longer be silently overwritten with B's username when the dispatcher field is empty. The new-load wizard call site (`loads-dashboard.component.html:2511`) is unchanged; `'create'` is the default.

## Files Changed
- `frontend/src/app/models/load-dashboard.model.ts` — `dispatcher_name?: string | null` on `LoadListItem` (inherited by `LoadDetail`).
- `frontend/src/app/components/loads-dashboard/load-detail-drawer/load-detail-drawer.component.ts` — drop `as any` cast.
- `frontend/src/app/components/loads-dashboard/load-detail-drawer/load-detail-drawer.component.html` — pass `mode="edit"` to `<app-step-basics>`.
- `frontend/src/app/components/loads-dashboard/load-wizard/step-basics/step-basics.component.ts` — new `mode` input; gate the localStorage fallback on `mode === 'create'`.

## Key Decisions
- **`mode` input on `step-basics` (default `'create'`).** Picked over a heuristic ("loadNumber was just generated") because the heuristic conflates two unrelated states. An explicit input also stays correct if the wizard ever pre-fills a load number for create mode. Default `'create'` keeps every existing call site behavior-identical.
- **Dispatcher remains read-only.** Out of scope for this bug — only fixing the read path so the display reflects backend truth.
- **`LoadListItem` typed too.** The bug AC mentions the list view; backend FN-1046 also adds the column to the list query. Typing it now means future list-view consumers (e.g. column add) won't need a cast.
