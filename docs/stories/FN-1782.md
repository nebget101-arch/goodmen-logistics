# FN-1782: Enforce DOT document readiness before a truck/trailer can be made Active

## Requirement
On the Equipment (Vehicles) page a user can flip any truck or trailer to **In Service** with no checks. There is no validation that DOT-required documents are uploaded or unexpired (`vehicle-form.component.html:339-342`, `setInactive()` ~471; backend forced `status='in-service'` on create in `vehicles.js` ~944). A unit can therefore be dispatched while missing or expired registration, insurance, or its annual DOT inspection — a compliance risk.

## Goal
A truck/trailer may only become **ACTIVE** (`in-service`) when ALL required DOT documents are (a) uploaded and (b) not expired. If any required document is missing or expired, activation is blocked in **both the UI and the API**, with a clear readiness checklist of what is missing.

## Required-document ruleset (server-side single source of truth, keyed by `vehicle_type`)
- **Truck (power unit):** registration, insurance, inspection, ifta
- **Trailer:** registration, inspection

A requirement is **satisfied** when EITHER a `vehicle_documents` row of that `document_type` has a today-or-future `expiry_date`, OR the matching vehicle column (`registration_expiry` / `insurance_expiry` / `inspection_expiry`) is today-or-future. `ifta` has no column, so it is satisfied only by a `vehicle_documents` row of type `ifta` (free-string `document_type` — no schema change).

## Readiness contract
`GET /api/vehicles/:id/readiness` → 200
```json
{
  "vehicleId": "uuid",
  "vehicleType": "truck",
  "ready": false,
  "requiredDocuments": [
    { "type": "registration", "label": "Registration", "state": "valid|missing|expired", "expiryDate": "2026-01-01|null", "documentId": "uuid|null" }
  ],
  "missing": ["insurance"],
  "expired": ["inspection"]
}
```

- `requiredDocuments` is server-driven — the client never hardcodes the set.
- `missing` and `expired` are subsets of `requiredDocuments`. A document not in either is valid.

Activation guard on `POST /api/vehicles` and `PUT /api/vehicles/:id`: when the target status is `in-service`/`active` and the unit is not ready → **HTTP 422** `{ message, code: "VEHICLE_NOT_READY", missing: [...], expired: [...] }`.

On create with documents absent, the backend persists `status='out-of-service'` rather than defaulting to `in-service`.

`GET /api/equipment` exposes a `ready` flag (or `readiness` object) per row so dispatch sees eligible units. The vehicles list/detail badge also reads an optional `readiness`/`ready` field when present.

## Key decision
A brand-new vehicle cannot have documents before it has an id, so **create defaults to** `out-of-service` (pending) when activation is not explicitly requested; the user uploads docs then activates via a subsequent update that passes the guard. (A trailer whose `registration_expiry` + `inspection_expiry` are provided and current on create CAN be born ready, since both its requirements are column-backed.) An already-active unit whose document later lapses is **surfaced** (readiness flips to expired) but is **not** auto-deactivated in this story.

### FN-1783 (subtask) — Backend

Implemented all backend pieces in one pass.

- **Rule engine + evaluator** — new `services/vehicle-readiness.service.js` is the single source of truth. `READINESS_RULES` maps `truck → [registration, insurance, inspection, ifta]`, `trailer → [registration, inspection]`. `evaluateReadiness(vehicle, documents, now)` returns `{ vehicleType, ready, requiredDocuments[], missing[], expired[] }`. Each requirement resolves to `valid` / `missing` / `expired` from a dual source: matching `vehicle_documents` rows (preferred so the UI can link to a `documentId`) and the matching `*_expiry` column. `ifta` is document-only (no column). Date comparison is day-granular; a candidate satisfies when its expiry is today-or-future, matching the existing `inspection_expiry < NOW()::date` overdue semantics.
- **`GET /api/vehicles/:id/readiness`** — `vehicles.js` returns the contract; 404 when the id is not a fleet vehicle.
- **Activation guard** — `evaluateActivationGuard()` shared by POST and PUT. On `POST`, status defaults to `out-of-service` unless the caller requests `in-service`/`active` AND the prospective row (its create-time expiry columns) is ready; otherwise 422 `VEHICLE_NOT_READY`. The previously hard-coded `'in-service'` literal in the INSERT is now a bound parameter. On `PUT`, when the body sets status to `in-service`/`active`, the current vehicle + documents are loaded, any expiry/type fields in the same request are merged in, and the guard runs → 422 when not ready. Non-fleet `customer_vehicles` are not loaded and therefore not gated.
- **`GET /api/equipment`** — selects the three `*_expiry` columns from `all_vehicles` and runs `deriveReadinessForRows()` to attach `ready` (boolean) + `readiness: { missing, expired }` per row. Documents are batch-loaded once for all ids; a DB error (e.g. unmigrated schema) falls back to column-only evaluation rather than failing the list.
- **Document-type matching** — `document_type` is a free string, so matching normalizes separators and accepts variants (`annual_inspection`, `dot-inspection`, `ifta_decal`) by substring against the requirement key.
- **Unit tests** — `test/vehicle-readiness.test.js` (node:test, no DB/express needed): 22 cases covering the rule set per `vehicle_type`, missing/expired/valid states, dual-source expiry precedence, ifta document-only rule, free-string variants, non-DOT units, and the activation guard (422 paths + happy path). All pass.

## Key Decisions
- **No schema change.** Readiness is derived at read time from existing `vehicles.*_expiry` columns and free-string `vehicle_documents.document_type`, per the story's "no migration" constraint.
- **Dual-source, document-preferred.** When both a document row and a column are valid, the document is reported (carries a `documentId`) so the frontend checklist can deep-link to the file.
- **Present-but-undated document = expired (not valid).** A document with no `expiry_date` is no proof of currency, so the unit stays gated until a current, dated document is provided.
- **Default-to-`out-of-service` on create**, only flipping to `in-service` when explicitly requested and ready — implements the story's "pending until docs uploaded" decision while still allowing a fully-documented trailer to be born active.
- **Equipment list is resilient.** A document-load failure degrades to column-only readiness instead of 500-ing the dispatch list, consistent with the endpoint's existing schema-tolerance.
- **Server-driven required set on the client.** The frontend never hardcodes the required-document list; backend readiness / 422 is authoritative, with an expiry-column fallback only for resilience before the backend lands.
