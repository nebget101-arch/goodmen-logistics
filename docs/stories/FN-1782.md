# FN-1782: Enforce DOT document readiness before a truck/trailer can be made Active

## Requirement
On the Equipment (Vehicles) page a user can flip any truck or trailer to **In Service** with no checks. There is no validation that DOT-required documents are uploaded or unexpired (`vehicle-form.component.html:339-342`, `setInactive()` ~471; backend forces `status='in-service'` on create in `vehicles.js` ~944). A unit can be dispatched while missing or expired registration, insurance, or its annual DOT inspection — a compliance risk.

A truck/trailer may only become **ACTIVE** (`in-service` / "ready to work") when ALL required DOT documents are (a) uploaded and (b) not expired. Activation is blocked in **both the UI and the API**, with a clear readiness checklist of what is missing.

## Acceptance Criteria
- [ ] Server-side readiness rule set + evaluator returns `{ ready, requiredDocuments[], missing[], expired[] }` per `vehicle_type`.
- [ ] `GET /api/vehicles/:id/readiness` returns the contract below.
- [ ] `POST` / `PUT /api/vehicles` reject setting `status=in-service` when not ready (HTTP 422 with missing/expired lists).
- [ ] `GET /api/equipment` exposes/derives readiness so dispatch can see eligible units.
- [ ] Vehicle form disables the "In Service" option (or blocks save) until ready, and shows a document-readiness checklist (each required doc: missing / expired / valid) linking to its upload control.
- [ ] Readiness badge surfaced on the vehicles list/detail; an active unit with a lapsed doc is flagged.
- [ ] Backend unit tests (evaluator + activation guard); frontend specs (gating + checklist); QA evidence.

## Required-document ruleset (server-side single source of truth, keyed by `vehicle_type`)
| vehicle_type | Required documents | DOT basis |
|--------------|--------------------|-----------|
| `truck` (power unit) | registration, insurance, inspection (annual), ifta | IRP plate; 49 CFR 387 / MCS-90; 49 CFR 396.17; IFTA license/decal |
| `trailer` | registration, inspection (annual) | registration; 49 CFR 396.17 |

A requirement is **satisfied** when EITHER a `vehicle_documents` row of that `document_type` has a future `expiry_date`, OR the matching vehicle column (`registration_expiry` / `insurance_expiry` / `inspection_expiry`) is in the future. `ifta` has no column today, so it is satisfied only by a `vehicle_documents` row of `document_type='ifta'` (free-string column — **no schema change / migration needed**).

## API contract (fixed up front — backend & frontend build against this in parallel)
`GET /api/vehicles/:id/readiness` → `200`
```json
{
  "vehicleId": "uuid",
  "vehicleType": "truck",
  "ready": false,
  "requiredDocuments": [
    { "type": "registration", "label": "Registration", "state": "valid", "expiryDate": "2026-09-01", "documentId": "uuid" },
    { "type": "insurance",    "label": "Insurance",    "state": "missing", "expiryDate": null, "documentId": null },
    { "type": "inspection",   "label": "Annual Inspection", "state": "expired", "expiryDate": "2026-01-01", "documentId": "uuid" },
    { "type": "ifta",         "label": "IFTA License/Decal", "state": "valid", "expiryDate": "2026-12-31", "documentId": "uuid" }
  ],
  "missing": ["insurance"],
  "expired": ["inspection"]
}
```
Activation guard on `POST /api/vehicles` and `PUT /api/vehicles/:id`: when target `status=in-service` and not ready →
`422` `{ "message": "...", "code": "VEHICLE_NOT_READY", "missing": ["insurance"], "expired": ["inspection"] }`.

## Key Decisions
- **New vehicle defaults to `out-of-service`** when required docs are absent (a brand-new unit has no id yet, so docs can't be attached at create time). The user uploads docs, then activates via a subsequent update that passes the guard. This replaces the current hardcoded `status='in-service'` on create (`vehicles.js` ~944).
- **Lapse is surfaced, not auto-enforced**: an already-active unit whose document later expires flips its readiness badge to "expired" on list/detail, but is **not** auto-deactivated in this story (out of scope; can be a follow-up automation).
- **Ruleset lives server-side** as the single source of truth; the UI checklist is driven by the `/readiness` response, never a hardcoded list.
