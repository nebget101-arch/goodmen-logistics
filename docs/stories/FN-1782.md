# FN-1782 — Enforce DOT document readiness before a truck/trailer can be made Active

> Story doc reconstructed during FN-1784 implementation (the original was missing
> from the repo). Captures the readiness contract the subtasks share so backend
> (FN-1783) and frontend (FN-1784) stay aligned.

## Summary

A truck/trailer may only be set **In Service** (active) once all DOT documents
required for its type are present and unexpired. The rule engine is server-side
(single source of truth); the frontend consumes a readiness endpoint to gate
activation, render a per-document checklist, and surface a readiness badge.

## Required document set (by vehicle type)

| vehicle_type | required documents |
|---|---|
| `truck` | `registration`, `insurance`, `inspection`, `ifta` |
| `trailer` | `registration`, `inspection` |

A requirement is satisfied when **either** a `vehicle_documents` row of that type
has a future `expiry_date`, **or** the matching vehicle column
(`registration_expiry` / `insurance_expiry` / `inspection_expiry`) is in the
future. `ifta` is satisfied **only** by a `vehicle_documents` row of type `ifta`.

## API contract

### `GET /api/vehicles/:id/readiness`

```json
{
  "vehicleId": "…",
  "vehicleType": "truck",
  "ready": false,
  "requiredDocuments": ["registration", "insurance", "inspection", "ifta"],
  "missing": ["ifta"],
  "expired": ["inspection"]
}
```

- `requiredDocuments` is server-driven — the client never hardcodes the set.
- `missing` and `expired` are subsets of `requiredDocuments`. A document not in
  either is valid.

### Activation guard (422)

`POST /api/vehicles` and `PUT /api/vehicles/:id`, when the target status is
`in-service` and the unit is not ready, respond **HTTP 422**:

```json
{
  "message": "…",
  "code": "VEHICLE_NOT_READY",
  "missing": ["ifta"],
  "expired": ["inspection"]
}
```

On create with documents absent, the backend persists `status='out-of-service'`
rather than defaulting to `in-service`.

### Equipment list

`GET /api/equipment` exposes a `ready` flag (or `readiness` object) per row so
dispatch sees eligible units. The vehicles list/detail badge also reads an
optional `readiness`/`ready` field when present.

### FN-1784 (subtask) — Frontend

Implemented against the contract above; the readiness response is consumed as-is
(no hardcoded required set), with graceful fallbacks while the backend endpoint
lands.

**Service** (`services/vehicle.service.ts`)
- Added `getReadiness(vehicleId)` → `GET /api/vehicles/:id/readiness`, plus the
  `VehicleReadiness` / `ReadinessDocState` types.

**Vehicle form** (`vehicle-form/vehicle-form.component.*`)
- Added `ifta` to `documentTypes` (and a `docIconMap` entry).
- Loads readiness for the edited unit; renders a **Document Readiness** section —
  a checklist of each required document with its state (valid / missing /
  expired). Each non-valid row links to the matching upload control via
  `<label for="file-…">`.
- Activation gate: the **In Service** option is disabled and save is blocked with
  a clear message while the unit is known-not-ready. The backend **422
  `VEHICLE_NOT_READY`** is handled as the authoritative backstop — its
  `missing`/`expired` are surfaced into the checklist and the gate message.
- Brand-new units (no id yet) rely on the 422 path since readiness can't be
  fetched pre-create.

**Vehicles list + detail** (`vehicles.component.*`)
- Added a **DOT Readiness** badge (Ready / Not Ready) to the list Status cell and
  the detail drawer, with a tooltip listing missing/expired docs.
- Readiness resolves from a server-provided `readiness`/`ready` field when
  present, otherwise a client-side fallback derived from the expiry columns
  (registration/insurance/inspection). The fallback can't see IFTA documents, so
  it never reports IFTA as missing — backend readiness/422 stays authoritative.
- An **active** unit that is no longer ready flags with a `danger` badge
  (`isActiveButNotReady`), satisfying the lapse-surfacing requirement without
  auto-deactivating the unit.

**Specs** (`vehicle-form/vehicle-form.component.spec.ts`)
- Cover the gate (block In Service when not ready; allow Out of Service; allow In
  Service when ready), per-document state derivation, the 422 handling path, and
  IFTA presence in `documentTypes`.
