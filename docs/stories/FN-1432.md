## Implementation Summary

### FN-1441 (subtask)

`work-order-triage-handler.js` now emits parts in the normalized shape `{ partName, manufacturerHint, suggestedSku, qty, confidence }`. The system prompt was rewritten to instruct the LLM to return this shape directly, with `suggestedSku` set only when confident; the handler additionally normalizes legacy aliases (`name`, `query`, `description`, `sku`) so older model responses still parse cleanly.

The legacy `{ query, qty }` shape is still emitted as `partsLegacy` for one release so existing frontend clients keep working until FN-1443 ships. Both fields are populated from the same canonical part list, so they cannot drift.

The static system prompt was hoisted to a module-level constant (`TRIAGE_SYSTEM_PROMPT`) and an explicit `prompt_cache_key: 'work-order-triage-v2'` was passed to the OpenAI call so the prefix stays cache-warm across requests; per-request data lives only in the user message.

### FN-1442 (subtask)

Added `POST /api/work-orders/triage-enriched` to `work-orders-hub.js`. The route forwards the request body (`description`, `vehicleId?`, `customerId?`, `locationId?`) to the AI service's existing `/api/ai/work-order/triage` endpoint, then enriches each suggested part against the tenant's parts catalog and inventory before returning the full triage payload.

Enrichment lives in a separate, pure-function-friendly module: `services/triage-enrichment.service.js`. The route stays thin; the service is unit-tested with no DB.

For each suggested part the response now includes `{ partId, suggestedSku, onHand, binLocation, reorderPoint, isLowStock, inventoryStatus }` where `inventoryStatus âˆˆ { in_stock, low_stock, out_of_stock, not_found }`. Suggested SKUs that don't match any catalog row are returned with `inventoryStatus: 'not_found'` rather than dropped (per AC).

## Key Decisions

- **Backward-compat via parallel field, not in-place mutation.** `partsLegacy` is mapped from the same canonical `parts` array, so the two cannot diverge. Removing it later is a one-commit change once FN-1443 lands.
- **Aliases on the way in, not the way out.** The handler accepts `name`, `query`, `description`, `sku` from the LLM (since prompt drift is real), but emits only the documented shape â€” clients see one schema even when the model returns a stale one.
- **Cache key bumped from default to `work-order-triage-v2`** to invalidate any cached responses from the old prompt format.
- **(FN-1442) Dedicated `/triage-enriched` route on `work-orders-hub`** instead of mutating the existing AI-service triage endpoint. The AI service stays a pure inference layer; tenant-scoped DB joins live with the rest of the work-order routes (where `tenantContextMiddleware` is already wired and where parts/inventory queries naturally belong).
- **(FN-1442) Two single SQL `IN`-clause queries, no per-SKU round-trips.** First query resolves candidate parts via `UPPER(sku) = ANY($skus) OR LOWER(name) = ANY($names)` scoped to `parts.tenant_id = $tenantId OR tenant_id IS NULL` (legacy rows). Second query joins `inventory â†’ locations` and filters by `locations.tenant_id` so a tenant can never see another tenant's inventory even if a SKU collides.
- **(FN-1442) Aggregate on-hand across locations when no `locationId` is given**, single-location lookup when it is. The triage UI typically has a location in scope; tests cover both modes.
- **(FN-1442) Defensive on AI shape.** Reads both the new `{ partName, suggestedSku, qty, confidence }` shape (FN-1441) and the legacy `{ query, qty }` shape, so the route stays correct if a stale ai-service deploy slips out.
