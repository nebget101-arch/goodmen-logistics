# FN-1432: T1 — Triage + Live Parts Availability

## Requirement

Extend the existing work-order triage flow so each AI-suggested SKU is enriched with live on-hand qty + bin location, with a one-click "Create reorder" action when out of stock.

## Acceptance Criteria

- [x] `work-order-triage-handler.js` returns suggested SKUs in a normalized shape (`partName, suggestedSku?, qty, confidence`) without breaking existing clients.
- [x] Backend enrichment route attaches `{ onHand, binLocation, reorderPoint, isLowStock }` per SKU; tenant-scoped; SKU-not-found case surfaced explicitly.
- [x] Frontend triage panel renders availability inline; "Create reorder" CTA when out of stock; toast on reorder success.
- [ ] Cypress E2E covers in-stock, out-of-stock, and not-found cases.

## Implementation Summary

### FN-1441 (subtask)

`work-order-triage-handler.js` now emits parts in the normalized shape `{ partName, manufacturerHint, suggestedSku, qty, confidence }`. The system prompt was rewritten to instruct the LLM to return this shape directly, with `suggestedSku` set only when confident; the handler additionally normalizes legacy aliases (`name`, `query`, `description`, `sku`) so older model responses still parse cleanly.

The legacy `{ query, qty }` shape is still emitted as `partsLegacy` for one release so existing frontend clients keep working until FN-1443 ships. Both fields are populated from the same canonical part list, so they cannot drift.

The static system prompt was hoisted to a module-level constant (`TRIAGE_SYSTEM_PROMPT`) and an explicit `prompt_cache_key: 'work-order-triage-v2'` was passed to the OpenAI call so the prefix stays cache-warm across requests; per-request data lives only in the user message.

### FN-1442 (subtask)

Added `POST /api/work-orders/triage-enriched` to `work-orders-hub.js`. The route forwards the request body (`description`, `vehicleId?`, `customerId?`, `locationId?`) to the AI service's existing `/api/ai/work-order/triage` endpoint, then enriches each suggested part against the tenant's parts catalog and inventory before returning the full triage payload.

Enrichment lives in a separate, pure-function-friendly module: `services/triage-enrichment.service.js`. The route stays thin; the service is unit-tested with no DB.

For each suggested part the response now includes `{ partId, suggestedSku, onHand, binLocation, reorderPoint, isLowStock, inventoryStatus }` where `inventoryStatus ∈ { in_stock, low_stock, out_of_stock, not_found }`. Suggested SKUs that don't match any catalog row are returned with `inventoryStatus: 'not_found'` rather than dropped (per AC).

### FN-1443 (subtask)

UI extends the existing AI Triage panel inside the Service Details tab. `runAiTriage()` now calls the new `POST /api/work-orders/triage-enriched` endpoint (FN-1442) instead of the AI-only triage route, so a single round-trip yields parts already enriched with `{ partId, suggestedSku, onHand, binLocation, reorderPoint, isLowStock, inventoryStatus }`.

Each suggested part renders one of four states (driven by `inventoryStatus`):

- **`in_stock`** — green pill: bin location + `qty N`.
- **`low_stock`** — amber pill: `N of <reorderPoint>`. Shows "Create reorder" CTA.
- **`out_of_stock`** — red pill: "Out of stock". Shows "Create reorder" CTA.
- **`not_found`** — cyan link to `/parts-catalog?search=<sku|partName>` so the technician can look up the part manually instead of having the row silently dropped.

The "Create reorder" button POSTs to `/api/parts/reorder` (still pending on the backend — see Open Items). On success, the row flips to "Reorder queued" and a bottom-right toast says "Reorder created". Errors surface inline + as an error toast. Toasts auto-clear after 3.5 s. The component uses local toast state (matching the existing `tolls-accounts.component.ts` pattern) — there is no shared `ToastService` to introduce.

## Key Decisions

- **Backward-compat via parallel field, not in-place mutation.** `partsLegacy` is mapped from the same canonical `parts` array, so the two cannot diverge. Removing it later is a one-commit change once FN-1443 lands.
- **Aliases on the way in, not the way out.** The handler accepts `name`, `query`, `description`, `sku` from the LLM (since prompt drift is real), but emits only the documented shape — clients see one schema even when the model returns a stale one.
- **Cache key bumped from default to `work-order-triage-v2`** to invalidate any cached responses from the old prompt format.
- **(FN-1442) Dedicated `/triage-enriched` route on `work-orders-hub`** instead of mutating the existing AI-service triage endpoint. The AI service stays a pure inference layer; tenant-scoped DB joins live with the rest of the work-order routes (where `tenantContextMiddleware` is already wired and where parts/inventory queries naturally belong).
- **(FN-1442) Two single SQL `IN`-clause queries, no per-SKU round-trips.** First query resolves candidate parts via `UPPER(sku) = ANY($skus) OR LOWER(name) = ANY($names)` scoped to `parts.tenant_id = $tenantId OR tenant_id IS NULL` (legacy rows). Second query joins `inventory → locations` and filters by `locations.tenant_id` so a tenant can never see another tenant's inventory even if a SKU collides.
- **(FN-1442) Aggregate on-hand across locations when no `locationId` is given**, single-location lookup when it is. The triage UI typically has a location in scope; tests cover both modes.
- **(FN-1442) Defensive on AI shape.** Reads both the new `{ partName, suggestedSku, qty, confidence }` shape (FN-1441) and the legacy `{ query, qty }` shape, so the route stays correct if a stale ai-service deploy slips out.
- **(FN-1443) Single round-trip from the FE.** Calls `/api/work-orders/triage-enriched` once instead of running AI triage and a separate enrichment query — the BE already does both, so duplicating that on the FE would just add latency and a coordination bug.
- **(FN-1443) Drive UI off the BE's `inventoryStatus` enum**, not a recomputed flag. Adding a separate `notFound` boolean would force the FE to keep its own classification logic in sync with the BE's; one enum value is the single source of truth.
- **(FN-1443) Local toast, not a new service.** FleetNeuron has no shared `ToastService`; the matching pattern in `tolls-accounts.component.ts` is per-component toast state. Reused that — introducing a new shared service is out of scope for this subtask.
- **(FN-1443) No `app-ai-select` getter `[options]` bindings** — confirmed by grep; project memory: this caused infinite CD loops in FN-317.
- **(FN-1443) AI dark theme tokens** (`--ai-accent-success`, `--ai-accent-danger`, `--ai-accent-warning`, `--ai-accent-cyan`, `--ai-text-strong/body/muted`) — pulled from `styles.css`, no raw hex values introduced.
