## Requirement
Three regressions surfaced while exercising the Quick Add Part flow on dev:

1. **UI theming** ŌĆö `quick-add-invoice-modal` (Review Invoice Lines) renders in a light theme (white card, slate text, light borders) while the rest of the page is AI dark themed.
2. **DB constraint crash on Confirm and Create** ŌĆö bulk-create from invoice review fails with `null value in column "category" of relation "parts" violates not-null constraint`. The AI invoice handler does not extract per-line category, so the request body has no `category` and the DB rejects the insert. Manufacturer is already nullable but treated as required in the manual form ŌĆö clarify policy.
3. **Photo intake prefill empty** ŌĆö `/api/ai/parts/identify-from-photo` returns a valid payload (manufacturer, partNumber, category, descriptionGuess + confidence), but the Create New Part modal opens with all fields blank. Likely a response-shape unwrapping bug (`aiResult.data` not unwrapped).

## Implementation Summary

### FN-1363 (subtask) ŌĆö DB: drop NOT NULL on parts.category
New Knex migration `20260505220000_relax_parts_category_notnull.js` issues `ALTER TABLE parts ALTER COLUMN category DROP NOT NULL`. The reverse migration backfills any rows where `category IS NULL` to `'Uncategorized'` before re-applying `SET NOT NULL`, so `down` is safe to run on data created during the relaxed window. Both branches are guarded with `hasTable('parts')` / `hasColumn('parts', 'category')` to stay idempotent across environments. No default value is introduced ŌĆö empty stays empty so the BE/FE layers decide how to render and whether to coerce.

Verified directly against the dev database with `psql`: starting nullable=YES ŌåÆ forced down() body ŌåÆ nullable=NO and `INSERT ŌĆ” category=NULL` raised `not_null_violation` as expected ŌåÆ forced up() body ŌåÆ nullable=YES and the same insert succeeded, returning a row with NULL category. Probe row was cleaned up.

### FN-1364 (subtask) ŌĆö BE: tolerate null/missing category in parts create + bulk-create
Added a `normalizeCategory(value)` helper at the top of `backend/packages/goodmen-shared/services/parts.service.js` that returns `'Uncategorized'` when the input is missing, non-string, empty, or whitespace-only, and otherwise returns the trimmed input. Both `createPart` (single) and `bulkCreateParts` (Quick Add Invoice OCR path) now route the inbound `category` field through this helper at row-assembly time. Validation is unchanged ŌĆö `sku` and `name` remain required and still throw / get skipped with `missing_sku_or_name`.

The helper lives in the service layer (not a DB default) per FN-1363's product policy: the column stays nullable so reporting queries can still distinguish "intentionally classified" rows from gaps if the policy ever flips, but the live write path coerces so list/filter UI never sees NULL category. Manufacturer behavior is intentionally untouched ŌĆö `resolveManufacturerVendor` continues to leave both `manufacturer_id` and `manufacturer` unset on the insert row when the caller supplies neither, so the DB stores NULL and the existing FN-1093 typeahead behavior is preserved.

The route layer (`backend/packages/goodmen-shared/routes/parts.js`) passes `req.body` straight into the service for both single-create (`POST /api/parts`) and bulk-create (`POST /api/parts/bulk`) ŌĆö no field stripping happens upstream, so the service-layer default is the single source of truth. This was verified by reading the route handler.

Three new test cases added to `parts.service.bulk.test.js` (mock-knex, hermetic, run with `node --test services/parts.service.bulk.test.js` from the package root):

* **`bulkCreateParts: missing/empty category`** ŌĆö five-item batch covering no-category-key, empty-string, whitespace-only, explicit-with-trim, and missing-manufacturer rows. Asserts `category='Uncategorized'` on the first three, `category='Brakes'` on the trim case, and `manufacturer/manufacturer_id` left unset (NULL) when not supplied.
* **`createPart: missing category`** ŌĆö three single-create calls covering no-key / empty / explicit; same defaulting expectations.

## Key Decisions

### FN-1363 (subtask)
- **No default value on the column.** Product policy is "AI couldn't classify it; let the UI decide." A DB-level `DEFAULT 'Uncategorized'` would silently rewrite NULLs and hide gaps from any reporting query that filters on category. The BE subtask FN-1364 owns request-shape contracts; this migration just removes the constraint.
- **Raw SQL over Knex `.string('category').nullable().alter()`.** The `.alter()` form rebuilds the column type and can drop other column attributes on Postgres (collation, default, length). The repo's existing nullability changes (`20260325150000_make_test_date_nullable.js`, `20260325020000_allow_null_result_add_result_received_date.js`) all use raw `ALTER COLUMN ŌĆ” DROP NOT NULL`; this migration follows that convention.

### FN-1364 (subtask)
- **Default in the service layer, not the DB.** A column-level `DEFAULT 'Uncategorized'` would coerce on every insert path (including ad-hoc SQL and seeds) and hide future regressions of "AI didn't return a category" ŌĆö which is the signal product wants to keep visible at the wire boundary. Putting the default in `createPart`/`bulkCreateParts` keeps the column NULL-able for reporting and gives a single, testable seam.
- **Don't relax `sku`/`name` validation.** The Acceptance Criteria explicitly call this out: a malformed item is still skipped (in bulk) or rejected (single create). Only the category gap is forgiven.
- **Manufacturer untouched.** The existing FK-and-text behavior in `resolveManufacturerVendor` already leaves both `manufacturer_id` and `manufacturer` undefined when the caller doesn't supply either, which inserts as NULL ŌĆö exactly what the AC requires. No code change needed there; tests now lock that contract in.
- **Helper at module scope rather than inline `c.category || 'Uncategorized'`.** Inline `||` would also overwrite `0`-like falsy edge cases (not a real risk for category strings, but a footgun for future maintainers); the helper makes the trim + empty-check explicit and reusable if a third call site appears.

### FN-1365 (subtask)
- **Tolerant unwrap, not a contract change.** `normalize` accepts both `aiResult: { data: {...} }` and a flat `aiResult: {...}`. The BE may eventually choose one shape; the FE survives either, which prevents another "modal opens empty" regression if the contract is touched again.
- **`duplicateCheckParts` for SKU lookup, not a new `getPartBySku` HTTP route.** The BE service has a `getPartBySku` helper but no exposed route. Adding one is BE scope and unnecessary: duplicate-check returns id/name/sku/manufacturer with similarity scoring, so an exact-SKU exact-match filter on the candidates is sufficient and reuses an already-tested code path. Lookup errors fall through to Create-prefilled to preserve the "never block Add Part" guarantee.
- **Reuse FN-1111's getPartById ŌåÆ cache ŌåÆ candidate fallback.** Same pattern that `onEditExistingDuplicate` already uses, so the Edit modal opens with the richest record available regardless of which layer is degraded.
- **Category cell between Description and Qty.** Spec offered two placement options; this position keeps the natural reading order Description ŌåÆ Category ŌåÆ Qty without truncating the Manufacturer typeahead.
