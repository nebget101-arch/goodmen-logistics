## Requirement
Three regressions surfaced while exercising the Quick Add Part flow on dev:

1. **UI theming** — `quick-add-invoice-modal` (Review Invoice Lines) renders in a light theme (white card, slate text, light borders) while the rest of the page is AI dark themed.
2. **DB constraint crash on Confirm and Create** — bulk-create from invoice review fails with `null value in column "category" of relation "parts" violates not-null constraint`. The AI invoice handler does not extract per-line category, so the request body has no `category` and the DB rejects the insert. Manufacturer is already nullable but treated as required in the manual form — clarify policy.
3. **Photo intake prefill empty** — `/api/ai/parts/identify-from-photo` returns a valid payload (manufacturer, partNumber, category, descriptionGuess + confidence), but the Create New Part modal opens with all fields blank. Likely a response-shape unwrapping bug (`aiResult.data` not unwrapped).

## Implementation Summary

### FN-1363 (subtask) — DB: drop NOT NULL on parts.category
New Knex migration `20260505220000_relax_parts_category_notnull.js` issues `ALTER TABLE parts ALTER COLUMN category DROP NOT NULL`. The reverse migration backfills any rows where `category IS NULL` to `'Uncategorized'` before re-applying `SET NOT NULL`, so `down` is safe to run on data created during the relaxed window. Both branches are guarded with `hasTable('parts')` / `hasColumn('parts', 'category')` to stay idempotent across environments. No default value is introduced — empty stays empty so the BE/FE layers decide how to render and whether to coerce.

Verified directly against the dev database with `psql`: starting nullable=YES → forced down() body → nullable=NO and `INSERT … category=NULL` raised `not_null_violation` as expected → forced up() body → nullable=YES and the same insert succeeded, returning a row with NULL category. Probe row was cleaned up.

## Key Decisions

### FN-1363 (subtask)
- **No default value on the column.** Product policy is "AI couldn't classify it; let the UI decide." A DB-level `DEFAULT 'Uncategorized'` would silently rewrite NULLs and hide gaps from any reporting query that filters on category. The BE subtask FN-1364 owns request-shape contracts; this migration just removes the constraint.
- **Raw SQL over Knex `.string('category').nullable().alter()`.** The `.alter()` form rebuilds the column type and can drop other column attributes on Postgres (collation, default, length). The repo's existing nullability changes (`20260325150000_make_test_date_nullable.js`, `20260325020000_allow_null_result_add_result_received_date.js`) all use raw `ALTER COLUMN … DROP NOT NULL`; this migration follows that convention.
