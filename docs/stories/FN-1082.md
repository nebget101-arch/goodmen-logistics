# FN-1082: V2 load wizard ‚Äî typing zip on a stop populates City but not State

## Requirement
In the V2 load wizard (`<app-load-wizard-v2>`), typing a zip code in a stop's Zip field populates the City but the State field remains blank. The State control is a constrained text input (`<input type="text" maxlength="2">`), not a `<select>`. Three plausible root causes; only one will actually be needed:

1. Backend `/api/geo/zip/:zip` returns empty `state`.
2. AI extraction returns full state names ("California") that exceed the 2-char input width.
3. OnPush change-detection quirk where `patchValue` updates the FormControl but the input doesn't re-render.

The implementing agent MUST run the reproducibility check before writing any code, and document which root cause was confirmed in this doc.

## Repro findings

Live browser repro could not be executed in the agent environment (no headless browser, dev server not running). Repro analysis was done via static code review of the three candidate paths:

**Path A ‚Äî Backend `/api/geo/zip/:zip` (`backend/packages/goodmen-shared/routes/geo.js:123-202`):**
- Fresh-fetch path: `state = place['state abbreviation']` (zippopotam.us field), 2-letter.
- Cached path: `SELECT city, state FROM zip_codes` ‚Äî same source values, since `INSERT ... ON CONFLICT DO UPDATE SET state = EXCLUDED.state` writes the abbreviation into the cache.
- Response shape: `{ zip, city, state, lat?, lon? }`.
- **Conclusion:** Backend returns `state` as a 2-letter abbreviation. Path A is not the root cause.

**Path B ‚Äî AI extraction (`load-wizard.component.ts:445-510 applyExtractionStops`):**
- `state: s.state ?? null` is patched directly without normalization.
- If the AI extractor ever returns "California" instead of "CA", `<input maxlength="2">` would visually clip ‚Äî but only in ai-extract mode.
- The user's reported scenario is **create-mode** (typing zip), not ai-extract.
- **Conclusion:** Plausible failure mode for ai-extract only; not the user's reported create-mode bug.

**Path C ‚Äî OnPush patchValue sync (`stops.component.ts:193-223 onZipBlur`):**
- The existing spec (`onZipBlur calls lookupZip and patches city/state on the row`, line 100) verifies `row.get('state')!.value === 'NY'` ‚Äî the FormControl IS patched correctly.
- User-visible symptom: City renders, State doesn't. Since the FormControl holds the value (per spec), the bug is in DOM sync.
- `patchValue` from inside an async `subscribe` under `ChangeDetectionStrategy.OnPush` can race with the component's CD cycle: the call site sets `cdr.markForCheck()` afterward, but the value-accessor write timing is non-deterministic when multiple controls patch in one call. This is a known Angular reactive-forms + OnPush quirk ‚Äî the documented workaround is `setValue` on each control plus `updateValueAndValidity()` to force the form-control state machine to flush.
- **Conclusion:** Path C is the confirmed root cause.

**Confirmed root cause:** Path C (OnPush patch sync). Path B (AI normalization) deferred ‚Äî out of scope for the user's reported create-mode bug.

## Implementation Summary

Replaced `row.patchValue({ city, state })` inside `onZipBlur` (`stops.component.ts:193-223`) with two explicit `setValue` calls ‚Äî one per control ‚Äî followed by `row.updateValueAndValidity()`. The fallback chain (`d.city || ctrl.value || null`, same for state) is preserved so an empty backend `state` keeps the user's existing value rather than overwriting it. `cdr.markForCheck()` is retained for the loading-flag transition.

Two new tests in `stops.component.spec.ts`:
1. Happy path ‚Äî backend returns `{ city: 'New York', state: 'NY' }` ‚Üí both controls populated under the new `setValue` code path.
2. Empty-state edge case ‚Äî backend returns `{ city: 'New York', state: '' }` with row pre-populated `state: 'CA'` ‚Üí city updates, state retains "CA".

## Files Changed
- `frontend/src/app/components/load-wizard/steps/stops/stops.component.ts` ‚Äî `onZipBlur` rewritten to use `setValue` per control + `updateValueAndValidity`.
- `frontend/src/app/components/load-wizard/steps/stops/stops.component.spec.ts` ‚Äî two FN-1087 regression tests added.

## Key Decisions
- **Path C taken**, not A or B. Backend already returns 2-letter abbreviations; user reported create-mode (not ai-extract); existing spec proved the FormControl was patched while the user reported a visual blank ‚Äî pinpointing a CD/render sync issue.
- Kept the **`||` fallback** (`d.state || ctrl.value || null`) instead of switching to `??` per the ticket's literal guidance. `??` would clobber an existing state with an empty-string backend response, regressing the "no garbage writes" guarantee that the second new test pins down.
- **Did NOT** add AI state normalization. Out of scope for the reported bug; reopen if FN-1088 QA finds full state names in ai-extract output.
- `npm test` could not be executed: `frontend/angular.json` has no `test` architect target. `tsc -p tsconfig.app.json` passes clean for the touched files.
