# FN-1609: [P1] Loads Import — commit 500s on load_stops insert (JS Date.toString() form unparsed)

## Requirement
Commit step at `POST /api/loads/import/commit/:sessionId` returns HTTP 500 with PostgreSQL error:

```
invalid input syntax for type date: "Thu May 07 2026 00:00:00 GMT+0000 (Coordinated Universal Time)"
```

The whole batch transaction aborts; no loads or stops are created.

## Root Cause
`buildStopsFromRow()` in `backend/packages/goodmen-shared/services/loads-import-mapper.js` calls `trimOrNull(normalized[`${prefix}_date`])` for `stopDate`. That JS `Date.toString()` string flows straight into `INSERT INTO load_stops (..., stop_date, ...)` and PostgreSQL rejects the value.

The top-level load INSERT already calls `parseImportDate` on `pickup_date / delivery_date / completed_date`, but the stops insert is missing this coercion.

## Acceptance Criteria
- [ ] Commit returns 200 when staged `normalized_values` contain JS `Date.toString()` form dates
- [ ] `load_stops.stop_date` is always ISO `YYYY-MM-DD` or `NULL` — never a raw JS Date string
- [ ] Unparseable stop date inserts the stop with `stop_date = NULL` (no whole-row failure)
- [ ] Existing `loads.pickup_date / delivery_date / completed_date` persistence unchanged (regression)
- [ ] Unit tests added in `loads-import-mapper.test.js` and `loads-import-service.test.js`

## Implementation Summary
`buildStopsFromRow()` in `loads-import-mapper.js` now coerces the per-stop date through the existing `parseImportDate()` helper instead of `trimOrNull()`. The helper already understands every input shape `applyColumnMapping` can produce — `Date` instances, ISO strings, JS `Date.toString()` form, `M/D/YYYY` — so `stops[].stopDate` is now always either ISO `YYYY-MM-DD` or `null`.

The downstream `INSERT INTO load_stops (..., stop_date, ...)` is unchanged: it already binds `stop.stopDate` directly to the DATE column, so coercing at the seam (mapper) keeps the parse logic in one place and leaves the service untouched. When a stop date is unparseable, the stop is still emitted with `stop_date = NULL` rather than failing the row — matching how the load-level `pickup_date / delivery_date / completed_date` already behave.

## Files Changed
- `backend/packages/goodmen-shared/services/loads-import-mapper.js` — `buildStopsFromRow.pushStop` now calls `parseImportDate(normalized[`${prefix}_date`])` instead of `trimOrNull(...)`.
- `backend/packages/goodmen-shared/services/loads-import-mapper.test.js` — new `describe('buildStopsFromRow — stop date coercion (FN-1609)')` block: `Date` instance, JS `Date.toString()` form, ISO, `M/D/YYYY`, and unparseable/null inputs all produce ISO-or-null `stopDate`.
- `backend/packages/goodmen-shared/services/loads-import-service.test.js` — two new `commitBatch` tests captured from `INSERT INTO load_stops`: one asserts pickup/delivery `stop_date` arrive at the DB as ISO when normalized values are JS `Date.toString()` form / `M/D/YYYY`; the other asserts a gibberish stop date inserts the stop with `stop_date = NULL` rather than aborting the row.

## Key Decisions
- **Coerce at the consumer (mapper), not the producer (`applyColumnMapping`).** That helper is field-type agnostic and `String(raw).trim()` is correct for non-date columns (load_number, po_number, notes, etc). Pushing date coercion into it would either require a per-field type registry or coerce values that should stay raw. Keeping the seam at `buildStopsFromRow` mirrors how `commitBatch` already coerces the load-level dates.
- **Unparseable date → `NULL`, not an error.** `parseImportDate` already returns `null` for gibberish, and the `load_stops.stop_date` column is nullable. This satisfies AC #3 (unparseable stop date should not fail the whole row) without any extra branching.
- **Service-level test asserts on captured `INSERT INTO load_stops` params, not just `INSERT INTO loads`.** The bug is in the stops insert; the loads insert already had `parseImportDate` and was passing pre-fix. Capturing the stops binding makes the regression specific.
