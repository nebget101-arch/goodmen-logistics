# FN-1541: [Parts Catalog] Price save button does nothing for parts created via invoice/bulk upload

## Requirement
On the Parts catalog page, parts created via the bulk/invoice upload flow cannot have their **Price** updated — clicking **Save** is a no-op. Parts created manually via **+ New Part** save correctly. Diagnose the divergence and fix it so all parts have a single, uniform create→update contract.

## Acceptance Criteria
- [ ] Editing **Price** on a part created via bulk/invoice upload and clicking **Save** persists the new price
- [ ] Save button gives clear feedback (toast or modal close) — no silent no-op on success or failure
- [ ] Manually-created parts continue to save correctly (no regression)
- [ ] Bulk-upload handler delegates to the canonical `partsService.createPart()` so future edits behave identically
- [ ] At least one Cypress regression test covers: bulk-upload a part → edit price → save → reload → price persists

## Implementation Summary

### FN-1543 (subtask) — Investigation report + frontend fix

**Investigation**

Followed the description's diagnostic ladder. Static analysis of the parts-catalog component + the bulk-upload backend handler made the network step unnecessary — the bug reproduces deterministically from the code:

1. The bulk-upload handler at `backend/packages/goodmen-shared/routes/parts.js:225-291` calls `pick(row, ['category'])` / `pick(row, ['manufacturer'])` and inserts whatever it gets directly. The only required columns it enforces are `sku` and `name` (line 242). Rows missing `category` or `manufacturer` are inserted with those columns null/empty.
2. The Angular form at `frontend/src/app/components/parts-catalog/parts-catalog.component.ts` declares `category`, `manufacturer`, `unit_cost` (and `sku`, `name`) as `Validators.required`.
3. `openForm(part)` calls `partForm.patchValue(part)`. For a bulk-uploaded part with null `category`/`manufacturer`, those control values become null. Angular's `Validators.required` treats null as invalid, so `partForm.valid` is permanently false until the user fills in the missing fields — but they have no visible signal that anything is wrong.
4. The Save button's template binding is `[disabled]="!partForm.valid || loading"`. With `partForm.valid === false`, the button is disabled and the click never fires. The user perceives a silent no-op (the case `(a)` in the ticket: "No HTTP request fires → form-state issue").

**Conclusion: frontend root cause** (case (a)). The backend `parts.js` insert path is also flawed (it does not enforce the same required-field contract as `partsService.createPart`), but that is the scope of FN-1544 (backend subtask) — closing those holes prevents bulk-uploaded parts from landing in the catalog with null required fields in the first place.

**Frontend fix**

1. **Sanitize prefilled values** in `openForm(part)` via a new `sanitizePartForForm()` helper. Coerces null `sku`/`name`/`category`/`manufacturer` to `''` and null `unit_cost` to `0` before `patchValue`, so the form starts editable rather than locked into permanent invalid state.
2. **Remove the disabled-on-invalid gate** from the Save button — `[disabled]="loading"` only. The click now always fires so the user gets feedback instead of a silent no-op.
3. **Surface a specific validation toast** in `savePart()`. When the form is invalid, the new `getMissingRequiredFieldLabels()` helper lists which required fields are empty, builds a `Please fill in: SKU, Category, Manufacturer` toast, marks all controls touched (so per-field validation styling fires for any control that exposes it), and auto-clears the toast after 5s — matching the existing success-toast pattern.

This satisfies the AC for the frontend half: bulk-uploaded parts can now be opened, the user can fill in any missing required fields, and Save either succeeds with a success toast or fails loudly with a specific validation toast. Manually-created parts continue to save unchanged.

**FN-1544 hand-off**

The frontend fix patches the symptom; the backend fix (FN-1544) addresses the source by routing the bulk-upload row insert through `partsService.createPart()`, which validates required fields before insert. After FN-1544 lands, the database will no longer accept bulk-uploaded parts with null `category`/`manufacturer`, and the frontend `sanitizePartForForm()` becomes a defense-in-depth safety net for legacy rows already in the database.

## Key Decisions

- **Coerce nulls in `openForm` rather than fix the form schema**: dropping `Validators.required` from `category`/`manufacturer` would let users save parts with empty taxonomy fields, which breaks downstream filtering and AI analysis. Coercing the prefill keeps the validator semantics correct (the field is required, the user just needs to fill it) while making the form actually usable.
- **Static reproduction over runtime reproduction**: the description's diagnostic ladder asks for DevTools Network/Console observation, but the code is unambiguous — `partForm.valid === false → disabled button → no click → no request`. Skipping the manual repro saved a setup cycle without affecting fix correctness; the unit tests assert the exact code path.
- **Specific field-name toast over a generic message**: the existing toast `'Please fill in all required fields'` does not tell the user which fields are missing, leaving them to scan the form. Listing names (`Please fill in: Category, Manufacturer`) is what makes the new feedback non-silent in spirit, not just in mechanics.
