# FN-1597 — [P1] [Loads Import] BE drops AI columnMapping (envelope unwrap bug) and poisons load_ai_extractions cache

**Type:** Bug
**Priority:** Highest (P1)
**Parent epic:** FN-1583 — AI-powered spreadsheet → Loads importer
**Relates to:** FN-1596 (frontend field-name alignment — both must merge for the AI Analysis step to render)
**Branch:** `backend/FN-1597/fix-ai-envelope-unwrap`

## Symptom

`POST /api/loads/import/preview` returned a body where every AI field was null even though the AI service was reachable and (after the first call) cached. Subsequent re-uploads with the same file kept returning `cacheHit: true, columnMapping: null` because the cached value was poisoned.

## Root cause

The AI handler at `backend/microservices/ai-service/src/handlers/loads-spreadsheet-handler.js:476-481` returns:

```json
{ "success": true, "fallback": false, "cacheHit": false, "data": { /* columnMapping ... */ }, "meta": { ... } }
```

`callAiColumnMapping` in `loads-import-service.js` returned the **whole envelope** instead of `body.data`. As a result:

1. `previewImport` read `aiResult.columnMapping` → undefined → returned null to the FE.
2. `writeAiCache` stored the **envelope** in `load_ai_extractions.extracted_data`.
3. Subsequent previews hit the cache, found a row, returned the envelope as `aiResult` → still no `columnMapping` → null mappings for 7 days (TTL window).

## Fix

`backend/packages/goodmen-shared/services/loads-import-service.js`:

1. **`callAiColumnMapping`** now unwraps `body.data` on a success envelope (`body.success === true && body.fallback !== true`). On any failure mode (non-2xx, parse error, fallback, success !== true) it returns `null`. The result is the validated mapping payload — no envelope keys leak through.
2. **`previewImport`** simplified the post-call branch: `aiResult` is already either the unwrapped data or null, so the fallback check is redundant. Cache write and return-shape construction now operate on the unwrapped data.
3. **`lookupAiCache`** added a shape-guard: if `extracted_data` exists but lacks a top-level `columnMapping`, treat it as a cache miss. This auto-heals existing poisoned rows on the next preview without a manual `DELETE` (chose option (a) from the AC over (b) cache-key versioning — self-healing, no migration, no orphaned rows).
4. Exported `callAiColumnMapping`, `lookupAiCache`, `writeAiCache` for unit testing.

## Key Decisions

- **Shape-guard over cache-key versioning** — option (a) repairs existing poisoned rows on next preview rather than orphaning them for 7 days. No DB migration, no version-prefix string to maintain.
- **Strict success gate in `callAiColumnMapping`** — also returns null when `success !== true`, in addition to the explicit `fallback === true` case the AC mentioned. Defensive against any future handler variant that flips success without setting fallback.
- **No envelope leak into cache** — the cache now only ever holds the validated data shape, so the shape-guard's `columnMapping` check is the canonical "is this a usable entry" predicate.
- **Tests stub via `require.cache` + `dbBridge.setDatabase`** — same pattern as `parts.service.bulk.test.js`, no new test infra. The service destructures `{ query, getClient }` at module load (project-wide pattern), so the test helper re-loads the service after each `setDatabase` call.
