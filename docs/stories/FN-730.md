## Implementation Notes

### FN-766 (subtask)
**Branch**: `frontend/FN-766/loads-virtual-scroll`

Introduced `@angular/cdk/scrolling` virtual scrolling around the loads table body so render cost stays flat regardless of `filteredLoads.length`. The existing server-side pagination (`pageSize = 25`) is untouched — CDK virtualization makes the component safe if pagination is later relaxed or if aggressive header-filter narrowing produces large local result sets.

**Shape of the change**
- Registered `ScrollingModule` in `LoadsDashboardModule` (`loads-dashboard.module.ts`).
- Split the single dashboard table into two `<table>` elements sharing identical `<colgroup>` percentage widths and `table-layout: fixed`:
  - **`.table--header`** — colgroup + thead (column labels + header-filter row). Stays outside the viewport so filter inputs remain interactive while scrolling rows below.
  - **`.table--body`** — colgroup + tbody, placed **inside** `cdk-virtual-scroll-viewport` with `*cdkVirtualFor` on the data `<tr>` and `trackBy: trackByLoadId` for stable row recycling.
- Viewport sized via `.loads-viewport { height: calc(100vh - 420px); min-height: 320px; }` — the container needs a definite height for virtualization to engage.
- Empty state moved out of the tbody into a sibling `.empty-state` div (the old `<tr *ngIf="loads.length === 0">` rendered a `colspan="14"` cell which was stale; current grid has 13 columns).
- `itemSize="48"` — matches current td padding (`12px 10px`) + 13px font-size (line-height ~1.25) + top border. `minBufferPx="400"` / `maxBufferPx="800"` give ~8 rows above/below the visible window so fast scroll stays smooth without over-rendering.
- Added `trackByLoadId(_i, load) => load.id` on the component. `LoadListItem.id` is required/non-null in the model.

**Why two tables instead of one**
`*cdkVirtualFor` renders virtualized items as direct children of the viewport's internal `cdk-virtual-scroll-content-wrapper`, which applies `transform: translateY(...)` for scroll offset. Wrapping the whole native `<table>` (thead + tbody) inside the viewport would push the filter inputs out of view as the user scrolled. Splitting into a static header table + a virtualized body table keeps filters pinned while body rows virtualize. Both tables share the same percentage colgroup and `table-layout: fixed`, so columns line up pixel-for-pixel.

**What was NOT changed**
- `filteredLoads` getter, `sortBy`/`sortDir`, header filters (`headerFilters`), row dropdown actions, status/billing pills, attachment chips — all unchanged.
- Pagination controls and `pageSize = 25` — unchanged. Ticket scope was "introduce virtual scroll", not "eliminate pagination".
- No new backend routes, no new DB work, no new services.

## Key Decisions

### FN-766
- **Keep pagination.** Virtual scroll and server pagination are orthogonal — one protects render cost, the other protects API/DB cost. Removing pagination would be out of scope and would also affect the `summaryTotals` + status-bar segments that operate on the current-page slice.
- **Two-table split over `display: contents` hack.** `display: contents` on the CDK content wrapper would disable the `transform: translateY()` that drives virtualization. A header/body table split is boring, safe, and keeps column alignment via `table-layout: fixed` + identical colgroup percentages.
- **Fixed `itemSize`.** All loads rows have the same `<td>` padding/font, so fixed-size virtualization (cheaper, simpler) beats `AutoSizeVirtualScrollStrategy`.
- **`trackBy: trackByLoadId`.** Without it, virtual recycling + Angular's default identity tracking re-creates `<app-status-pill>`, `<app-attachment-chip>`, and the actions dropdown on every scroll tick, which negates the perf win.
- **Viewport height via `calc(100vh - 420px)`.** Matches the space left over after the status-summary bar + filters + header. `min-height: 320px` prevents the viewport from collapsing on short screens.
