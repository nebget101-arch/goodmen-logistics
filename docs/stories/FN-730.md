## Implementation Summary

### FN-767 (subtask) — Column visibility + saved filter views + inline status change
Branch: `frontend/FN-767/loads-saved-views`

- **Persistence layer.** Added a JSONB `preferences` column on `users` (migration `20260419300000_add_preferences_to_users.js`) instead of a separate table — matches the existing pattern of incremental column adds on `users` (first_name, last_name, email, is_active, driver_id). Avoids a join on every dashboard load.
- **Backend route.** New shared route `backend/packages/goodmen-shared/routes/user-preferences.js` exposes `GET /api/user-preferences` and `PUT /api/user-preferences`. PUT shallow-merges the body at the top level into `users.preferences`, so scoping by feature key (e.g. `loadsDashboard`) keeps unrelated prefs untouched. Payload capped at 32 KB. Mounted in the auth-users-service and proxied through the gateway.
- **Frontend service.** `user-preferences.service.ts` provides a `BehaviorSubject`-backed cache with `load()`, `patch()`, and namespace helpers `getLoadsDashboardPrefs()` / `patchLoadsDashboard()`. Optimistic local updates on network failure so UI stays responsive.
- **Column visibility.** Loads-dashboard now carries a `columnDefs` metadata array (13 columns, two flagged `alwaysVisible`: Load # and Actions). A "Columns" menu toolbar above the table toggles checkboxes and persists to `loadsDashboard.columnVisibility`. Every `<col>`, both header rows, and every `<td>` gates on `isColVisible(key)`; empty-state `colspan` is computed from `visibleColumnCount`.
- **Saved filter views.** A "Views" menu above the table lists persisted views and applies them by replacing `filters`, `sortBy`, `sortDir` and reloading. Users can delete any view or save the current filter combo under a name via inline input. On first use we seed three starter views in-memory (My Drafts, This Week, Unpaid) — they are not persisted until the user saves/edits, so the list stays empty server-side until used.
- **Inline status change.** The status cell replaces the read-only `app-status-pill` with a button-wrapped pill + chevron. Clicking opens a dropdown of `statusOptions`; selection calls `LoadsService.updateLoad(id, { status })` (existing `PUT /api/loads/:id`) with an optimistic UI update and rollback on error. No modal is opened.

## Key Decisions

### FN-767
- **Extend `users.preferences` JSONB vs. separate `user_preferences` table.** Chose the JSONB column. Single-user-scoped prefs are a classic 1:1 relationship — adding a separate table would add a join on every dashboard mount and complicate RLS/tenant scoping. If prefs ever grow to feature-level rows (shared dashboards, team views), we can migrate later.
- **Top-level shallow merge in PUT.** Keeps the contract simple and lets callers namespace by feature key. Nested objects like `loadsDashboard` replace wholesale per call, which means the service sends the entire `loadsDashboard` sub-object on each update — that's fine given the small size and avoids server-side deep-merge complexity.
- **Seed default views in-memory only.** Showing "My Drafts", "This Week", "Unpaid" as starter options feels polished, but we don't write them to prefs until the user interacts. Keeps `savedViews` empty by default so we can tell used-vs-unused from the backend.
- **`alwaysVisible` for Load # and Actions.** Hiding Load # would leave rows unidentifiable; hiding Actions would remove the only way to open the edit modal. Checkboxes for these are disabled in the UI and the persisted payload omits them entirely.
- **Optimistic inline status update with rollback.** Revert `load.status` on error and surface a transient error message. No re-fetch of the full list — matches the inline-edit UX goal (no modal, no full reload).
