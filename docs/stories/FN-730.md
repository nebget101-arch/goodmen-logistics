## Implementation Summary

### FN-768 (subtask)
Multi-select bulk operations on the loads list, backed by two new transactional endpoints.

- **Backend** (`backend/packages/goodmen-shared/routes/loads.js`)
  - `POST /api/loads/bulk-update` — whitelisted to `status`, `billingStatus`, `driverId`, `truckId`. Validates every id belongs to the caller's `tenant_id` + `operating_entity_id` *before* writing; any stranger ids roll back the whole transaction and are returned in `notFound`. Mirrors single-PUT behaviour by auto-filling `completed_date` when a bulk transition to DELIVERED/COMPLETED is applied. Hard cap 500 ids per request.
  - `POST /api/loads/bulk-delete-drafts` — same tenant scoping + transactional semantics, but additionally refuses to delete unless *every* selected load is currently `DRAFT`. Non-drafts are returned in `nonDraft`.
  - Both endpoints gated by `requireRole(['admin', 'dispatch'])`.
- **Frontend service** (`frontend/src/app/services/loads.service.ts`)
  - `bulkUpdate(ids, changes)` and `bulkDeleteDrafts(ids)` wrap the two endpoints.
- **Frontend UI** (`frontend/src/app/components/loads-dashboard/loads-dashboard.component.{ts,html,scss}`)
  - Per-row checkbox + header select-all (with indeterminate state for partial selection). Row visuals highlight when selected.
  - Bulk actions toolbar renders above the table when `selectedCount > 0`:
    - Change Status (dropdown → POST bulk-update)
    - Assign Driver (dropdown → POST bulk-update)
    - Assign Truck (dropdown → POST bulk-update)
    - Approve drafts (filters to DRAFT, POST bulk-update `{status: 'NEW'}`)
    - Delete drafts (filters to DRAFT, POST bulk-delete-drafts)
    - Export CSV (client-side — no server round-trip)
  - Toolbar is hidden entirely in driver-role views, which never render the checkbox column.
  - Action buttons disable while a bulk request is in flight; the "Approve drafts" and "Delete drafts" buttons further disable when the selection contains no drafts.
  - CSV generated from the already-loaded rows so exported columns match the UI (no extra server load, and it works while the user is offline of the rest of the API).

### Key Decisions (FN-768)
- **Whitelist, don't passthrough.** `bulk-update` explicitly enumerates the four columns it accepts (`status`, `billingStatus`, `driverId`, `truckId`). This keeps the transactional endpoint out of business-logic it doesn't understand (stops, rates, broker overrides) and avoids accidentally exposing columns that single-PUT validates individually.
- **Pre-flight SELECT then UPDATE inside the same transaction.** The endpoint first SELECTs the ids to confirm tenant membership, and aborts before any write if any id is unknown. This is slightly more conservative than `UPDATE … RETURNING id` with a mismatch check, and makes the error message deterministic for the UI.
- **Two endpoints instead of a single multiplexed one.** A single `bulk-update` that accepts a magic `action: 'delete'` flag would blur the difference between "update some fields" and "destroy rows" — a bug in that code path could silently wipe data. Splitting keeps the delete path obviously scoped to drafts.
- **Delete/approve apply only to drafts in the selection.** Instead of erroring when the selection mixes drafts and non-drafts, the UI filters to drafts on the client so the user never has to curate the selection twice. The backend still enforces "drafts only" as a safety net.
- **CSV export is client-side.** The user only exports what they've already loaded, so a server endpoint would duplicate existing data for no gain. Client CSV also avoids permission checks that would need to re-run tenant scoping — the rows are already tenant-scoped because they came from `GET /loads`.
- **Hard cap at 500 ids.** Keeps worst-case response time bounded and protects Postgres from a rare misuse that would build a massive ANY($1::uuid[]) param.

- **Services to redeploy**: `fleetneuron-frontend` + every backend service that mounts `loads.js` (typically `fleetneuron-backend` / the API gateway — follow the same set of services redeployed for prior loads-route changes).
- **Database migrations**: none. No schema changes.
- **Environment / config changes**: none.
- **Cache invalidation**: none.
- **Feature flags**: none.
- **Deploy order**: backend first (so the new endpoints exist before the frontend calls them), then frontend.
- **Post-deploy verification**:
  1. Loads list → tick a row checkbox → bulk actions toolbar appears. Header checkbox toggles select-all for visible rows; indeterminate state shows when only some rows are selected.
  2. Change Status dropdown → choose `DISPATCHED` → network tab shows `POST /api/loads/bulk-update` with `{ids, changes: {status: 'DISPATCHED'}}`; UI refreshes and rows show the new status.
  3. Assign Driver / Assign Truck → same pattern.
  4. Select a mix of drafts and non-drafts → click Delete drafts → only drafts are deleted (backend returns `{deleted: N}`); non-drafts remain. Try the same with a non-draft-only selection → button is disabled.
  5. Click Export CSV → a `loads-YYYY-MM-DD.csv` file downloads. Open it and confirm headers + every selected row.
  6. Driver role login → checkbox column + toolbar must NOT render (ACL guard).
