## Implementation Summary

### FN-765 (subtask) â€” Keyboard shortcuts service + help modal
Global keyboard shortcut system with a help modal that lists every shortcut active on the current view.

- **`KeyboardShortcutsService`** (`frontend/src/app/shared/services/keyboard-shortcuts.service.ts`)
  - A single `window`-level `keydown` listener dispatches to bindings registered via `register()` / `registerAll()`.
  - Input-field guard: bindings are skipped when focus is inside `<input>`, `<textarea>`, `<select>`, or a `contentEditable` element unless the binding opts in with `allowInInput: true`.
  - Listener runs outside Angular's zone to avoid change-detection churn on unrelated keystrokes; matched handlers are re-entered into the zone.
  - Exposes `bindings$` and `helpOpen$` observables so the help modal updates reactively.
  - Static `formatKey(binding)` helper renders mac glyphs (âŒ˜/â‡§/âŒ¥) on macOS and `Ctrl+â€¦` / `Shift+â€¦` elsewhere.
- **`KeyboardShortcutsHelpComponent`** â€” rendered once at app root (`app.component.html`), groups bindings by `group` and shows formatted keys + descriptions, closes on Esc/âœ•/backdrop.
- **Loads list wiring** â€” registers `N` (new load), `Cmd/Ctrl+K` (quick search), `/` (focus search) on mount; unregisters on destroy.
- **Load wizard migration** â€” inline `@HostListener` replaced with service registration for `Esc`, `Cmd/Ctrl+S`, `Cmd/Ctrl+Shift+S`, `Enter` so they appear in the help modal only while the wizard is mounted.

### FN-766 (subtask) â€” Virtual scrolling for loads list (Angular CDK)
Introduced `@angular/cdk/scrolling` virtual scrolling around the loads table body so render cost stays flat regardless of `filteredLoads.length`. Existing server-side pagination (`pageSize = 25`) is untouched.

- Registered `ScrollingModule` in `LoadsDashboardModule`.
- Split the dashboard table into two `<table>` elements sharing identical `<colgroup>` percentage widths and `table-layout: fixed`:
  - `.table--header` â€” colgroup + thead (stays outside viewport so filter inputs remain interactive).
  - `.table--body` â€” colgroup + tbody inside `cdk-virtual-scroll-viewport` with `*cdkVirtualFor` + `trackBy: trackByLoadId`.
- Viewport sized via `.loads-viewport { height: calc(100vh - 420px); min-height: 320px; }`.
- Empty state moved out of tbody into a sibling `.empty-state` div.
- `itemSize="48"`, `minBufferPx="400"`, `maxBufferPx="800"`.

### FN-767 (subtask) â€” Column visibility + saved filter views + inline status change
Branch: `frontend/FN-767/loads-saved-views`

- **Persistence layer.** Added a JSONB `preferences` column on `users` (migration `20260419300000_add_preferences_to_users.js`) instead of a separate table â€” matches the existing pattern of incremental column adds on `users` and avoids a join on every dashboard load.
- **Backend route.** New shared route `backend/packages/goodmen-shared/routes/user-preferences.js` exposes `GET /api/user-preferences` and `PUT /api/user-preferences`. PUT shallow-merges the body at the top level into `users.preferences`, so scoping by feature key (e.g. `loadsDashboard`) keeps unrelated prefs untouched. Payload capped at 32 KB. Mounted in the auth-users-service and proxied through the gateway.
- **Frontend service.** `user-preferences.service.ts` provides a `BehaviorSubject`-backed cache with `load()`, `patch()`, and namespace helpers `getLoadsDashboardPrefs()` / `patchLoadsDashboard()`. Optimistic local updates on network failure so UI stays responsive.
- **Column visibility.** Loads-dashboard now carries a `columnDefs` metadata array (13 columns, two flagged `alwaysVisible`: Load # and Actions). A "Columns" menu toolbar above the table toggles checkboxes and persists to `loadsDashboard.columnVisibility`. Every `<col>`, both header rows, and every `<td>` gates on `isColVisible(key)`; empty-state `colspan` is computed from `visibleColumnCount`.
- **Saved filter views.** A "Views" menu above the table lists persisted views and applies them by replacing `filters`, `sortBy`, `sortDir` and reloading. Users can delete any view or save the current filter combo under a name via inline input. On first use we seed three starter views in-memory (My Drafts, This Week, Unpaid) â€” they are not persisted until the user saves/edits.
- **Inline status change.** The status cell replaces the read-only `app-status-pill` with a button-wrapped pill + chevron. Clicking opens a dropdown of `statusOptions`; selection calls `LoadsService.updateLoad(id, { status })` (existing `PUT /api/loads/:id`) with an optimistic UI update and rollback on error.

## Key Decisions

### FN-765
- **Window-level listener, not document-level.**
- **No-bind-collision rule** â€” exact-modifier matching so `Cmd+S` and `Cmd+Shift+S` coexist.
- **Wizard owns its shortcuts** â€” scoped lifecycle registration.
- **`allowInInput` opt-in** â€” `Cmd/Ctrl+S` works in inputs; `N`/`/`/`?` do not.
- **Outside Angular zone** â€” avoids change-detection churn on unrelated keystrokes.

### FN-766
- **Keep pagination.** Virtual scroll and server pagination are orthogonal.
- **Two-table split** â€” static header + virtualized body keeps filter inputs pinned, columns aligned via `table-layout: fixed`.
- **Fixed `itemSize`** â€” all rows share padding/font, so fixed-size virtualization beats auto-size.
- **`trackBy: trackByLoadId`** â€” required to avoid re-creating child components on every scroll tick.
- **Viewport `calc(100vh - 420px)`** with `min-height: 320px` â€” keeps layout stable on short screens.

### FN-767
- **Extend `users.preferences` JSONB vs. separate `user_preferences` table.** Chose the JSONB column. Single-user-scoped prefs are a classic 1:1 relationship â€” a separate table would add a join on every dashboard mount and complicate RLS/tenant scoping.
- **Top-level shallow merge in PUT.** Keeps the contract simple and lets callers namespace by feature key. Nested objects like `loadsDashboard` replace wholesale per call.
- **Seed default views in-memory only.** "My Drafts", "This Week", "Unpaid" show as starter options but aren't written to prefs until the user interacts.
- **`alwaysVisible` for Load # and Actions.** Hiding Load # would leave rows unidentifiable; hiding Actions would remove the edit entry point.
- **Optimistic inline status update with rollback.** Revert `load.status` on error and surface a transient error message. No full-list re-fetch.
