# FN-1682: Story H — Demo position-ping simulator + seed loads (client demo data)

## Requirement

Provide synthetic vehicle-position data that drives the real Phase 1 tracking UI (FN-1652) for client demos and internal dogfooding, **without** requiring a live Samsara/Motive feed. Writes interpolated pings into the real `vehicle_position_pings` table (from Story A, FN-1653) so the existing WS broadcast (from Story D, FN-1656) pushes them to the live map UI unchanged — clients see the actual product behavior, not a throwaway mock.

This is **demo-only infrastructure**: lives under `backend/scripts/`, is opt-in (not part of regular boot), and is fully reversible via a teardown script that deletes the demo loads and their pings.

## Why this, not a frontend-only mock

The original ask was a throwaway frontend mock. After scoping, FN-1652 was found to already deliver the real schema, WS broadcast, and live map page (most stories already in Code Review). Layering synthetic data on top of the real plumbing means the demo is identical to production behavior, the work survives past the demo as seed/staging data, and there is no parallel code path to maintain.

## Acceptance Criteria

## Implementation Summary

Three idempotent CLI scripts under `backend/scripts/`, plus a README. They layer synthetic data onto the real Story A/D plumbing — no new WS code, no schema changes.

- **Seed** upserts the route zip codes into `zip_codes` (the table ships empty on a fresh DB), resolves a target tenant, picks in-service trucks + active drivers, and creates up to 5 `DEMO-` `IN_TRANSIT` loads with PICKUP/DELIVERY stops.
- **Simulator** runs a `setInterval` loop that linearly interpolates each demo load's position pickup→delivery, inserting one `vehicle_position_pings` row per tick (`source_event_id='demo-sim'`, `payload={demo:true,…}`, random `speed_mph` 50–65, segment-bearing `heading_deg`). Marks `DELIVERED` on arrival; in `loop` mode pauses 30s then restarts.
- **Teardown** deletes the demo pings (scoped by `source_event_id='demo-sim'` on the demo trucks) and the `DEMO-` loads + stops; leaves `zip_codes` and all real data alone.

### FN-1683 (subtask)

## Key Decisions

- **Linear lat/lng interpolation** (not Vincenty/haversine geodesic) for position — sanctioned by the ticket as fine for a demo; documented in the simulator header. Haversine is used only for the cheap "within ~1km of delivery" arrival check and the bearing is the initial great-circle bearing.
- **Fixed-duration trips (~5 min/leg)** regardless of real distance — each load advances a fixed fraction per tick (`interval / TRIP_DURATION_MS`), so the demo is watchable and predictable.
- **Tenant resolution**: `--tenant` / `DEMO_TENANT_ID`, else auto-pick the tenant with the most in-service trucks, else fall back to any in-service truck with a loud warning (markers won't show on a tenant-scoped map until trucks have a tenant_id).
- **Teardown scopes ping deletion by `source_event_id='demo-sim'`** so a demo truck's real telematics pings are never deleted. `zip_codes` is intentionally left untouched (reference data).

## Deployment Handoff

- **No deploy required.** No migrations, no new tables, no Render service or env changes. These are opt-in, run-on-demand CLI scripts.
- **Run location**: a developer laptop or a one-off Render shell, pointed at the target DB via `DATABASE_URL` (or `PG_*` / `DB_*` vars).
- **Operational order for a demo**: seed → start simulator (leave running) → open `/tracking` logged in as the seeded tenant → teardown afterward. See the "Demo prep checklist" in `backend/scripts/README.md`.
- **Note for QA (FN-1684)**: run against the dev DB the demo browser uses; confirm markers move on `/tracking` and capture evidence under `docs/stories/evidence/FN-1682/`.
