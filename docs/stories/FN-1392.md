# FN-1392: Dispatch drivers list redesign + extracted Edit Driver routed page (UX)

## Requirement

Redesign the **Dispatch → Drivers** view (`/dispatch/drivers`) to raise visual hierarchy and ease-of-navigation while keeping the AI dark theme. Convert the inline **Edit Driver** modal into a dedicated routed page so the editing flow has more room and a clearer information architecture (Pay rates / Recurring deductions / Equipment Owner / Notes tabs become first-class sections on the page). Also remove the **+ New Driver** button from the dispatch view — driver creation should only happen from the DQF (Driver Qualification File) page.

Today the dispatch drivers view is monolithic: `frontend/src/app/components/dispatch-drivers/dispatch-drivers.component.ts` is 1,857 lines and its template is 993 lines, with the Edit Driver UX crammed into an inline modal. Users have asked for an Edit page with sticky save controls and roomier sections so they can manage Pay routing, Recurring deductions, Equipment Owner, and Notes without modal scroll lock.

## Key Decisions

1. **Route path is `/drivers/:id/edit`**, not `/dispatch/drivers/:id/edit`. The actual `dispatch-drivers` list routes from `/drivers`, so `/dispatch/drivers/...` would create an inconsistent URL hierarchy. The story’s `/dispatch/drivers` references were navigation-breadcrumb shorthand, not literal URLs.
2. **Saving uses `alert()` for failures and silent navigation back to `/drivers` on success** — the pre-existing dispatch-drivers component never used a toast service. The story doc said “use whatever toast service the dispatch-drivers component already uses — don’t introduce a new one,” so saving simply navigates back, matching the prior modal behavior.
3. **`DriverEditComponent` owns state directly rather than extracting micro-components for Pay routing / Deductions / Equipment Owner / Notes.** Because the modal is fully removed from `dispatch-drivers`, there is no duplication left to factor out — the new page is the only consumer. Pulling these out into separate sub-components would be a future refactor (worth doing if a third surface ever needs the same pieces) and is out of scope here.
4. **Table preserved at >=1024px viewports** with no new breakpoints; only padding and column padding tighten at 1024px. Sticky footer uses `position: sticky` so it doesn’t obscure content on short viewports.
