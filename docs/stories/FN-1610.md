# FN-1610: [P2] Control Center — daily briefing + 7-day trends use UTC for "today"

## Requirement
Daily Briefing card shows the wrong date (and "no data" / "investigate missing feed" copy) for late-evening US users because the gateway and reporting service compute `today` in UTC. 7-day Trends `actualDays` window shifts a day forward at UTC midnight and pushes the user's actual current local-day's loads into a previous slot.

Switch both endpoints to honor a user-supplied local date, falling back to today-UTC when the param is absent (preserves cron / server-internal callers). Cache key gets the resolved date so cross-tz users in the same tenant don't share a slot.

## Acceptance Criteria
- [ ] At 23:30 local on dev, Daily Briefing card header shows the user's local date (not UTC tomorrow).
- [ ] 7-day Trends `actualDays` ends on user's local today.
- [ ] `?localDate=YYYY-MM-DD` honored on `/api/ai/briefing` and `/api/insights/trends`; malformed → 400; missing → falls back to today-UTC (no regression).
- [ ] Cache key for both briefing tiers and trends includes the resolved date; cross-tz callers don't share entries.
- [ ] `?refresh=true` still bypasses both gateway and AI-service caches.
- [ ] Briefing's recommended-action no longer trips the "missing metrics feed" copy purely because the user is past UTC midnight.

## API Contract (locked here so FE/BE can run in parallel)

**Request**
- `GET /api/ai/briefing?localDate=YYYY-MM-DD&refresh=true`
- `GET /api/insights/trends?range=7d&localDate=YYYY-MM-DD&refresh=true`

`localDate` optional. Format: `^\d{4}-\d{2}-\d{2}$` strict. Malformed → `400 { error: "Invalid localDate" }`. Absent → server falls back to `new Date().toISOString().slice(0,10)` (existing behavior).

**Response**
- Briefing payload's top-level `date` field MUST equal the resolved `localDate` (or UTC fallback).
- Trends `window.actualDays` MUST end on the resolved date; `window.futureDays` starts the day after.
- Cache key includes the resolved date; `cached: true` only when the same `(tenantId, localDate)` was cached within TTL.

## Caching reminder (no behavior change in this ticket)

- Briefing gateway tier: 12 h in-memory `Map` (`briefing-aggregator.js`).
- Briefing AI-service tier: 24 h in-memory `Map` (`briefing-cache.js`).
- Trends: 10 min in-memory `Map` (`trend-cache.js`).
- All per-process; restart clears. `?refresh=true` bypasses.

## Implementation Summary

### FN-1612 (subtask, frontend)
Added a small `localDateIso()` helper that formats a `Date` using its **local** Y/M/D components (never UTC) and threaded it through both Control Center HTTP services. Every `getBriefing()` and `getTrends()` request now sends `?localDate=YYYY-MM-DD` derived from the user's browser wall clock — including the `refresh=true` path. No change to component code or to the `getBriefing` / `getTrends` argument signatures, so existing component specs continue to pass against the spy-based mock service.

## Key Decisions

### FN-1612 (subtask, frontend)
- **Local components, not UTC, not `toISOString().slice(0,10)`**: the whole point of the bug is that `toISOString()` is UTC. The helper builds the string from `getFullYear/getMonth/getDate` so US/Central at 23:30 emits the local day.
- **Service-level injection, not per-call**: keeping the change inside the services means the two Control Center components and any future caller automatically get correct behavior. No component-level changes were needed (verified against `daily-briefing.component.spec.ts` and `predictive-insights.component.spec.ts`, which mock the services with SpyObj and assert on the `{ refresh }` arg, not the URL).
- **Always send the param** (even when `refresh=false`): the API contract treats absence as "fall back to UTC", which is exactly the bug we're fixing. The browser always knows its local date, so the FE always sends it.
- **Tests freeze `Date` via `jasmine.clock().mockDate`** so the assertion `localDate === '2026-05-08'` is deterministic regardless of the runner's wall clock.
