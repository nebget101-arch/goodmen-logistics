# FN-1750: Vehicle Tracking page (Equipment) — live 2D map + telemetry (stubbed)

## Requirement
Add an in-app **Vehicle Tracking** page under the **Equipment** section: a live 2D map plus a telemetry panel, filtered by truck (unit number). Telemetry data is **STUBBED/MOCK** for now, behind a provider-agnostic API contract so a real telematics integration can swap in later without changing the contract.

This is a **standalone, narrow** feature — NOT the full telematics platform. Geofencing, public share-links, the unauthenticated `/track/:token` page, ETA prediction, and SMS milestones are **out of scope** (see the closed-but-unbuilt Epic FN-1652 and `docs/design/telematics-tracking-research-2026-06-02.md` for future context only).

## Acceptance Criteria
- [ ] "Vehicle Tracking" entry point appears under Equipment in the side nav and routes to `/vehicle-tracking` (same guards as `/vehicles`).
- [ ] Truck filter dropdown (`<app-ai-select>`) lists trucks by `unit_number`; selecting one focuses the map and loads its telemetry.
- [ ] Plain Leaflet 2D map with a streets/satellite (optionally terrain) view switcher via `L.control.layers` — no 3D, no Cesium/Mapbox/deck.gl. Marker(s) at last-known position.
- [ ] Telemetry panel shows fuel level, active fault codes (code + description + severity), last-moved time (relative + absolute), and current position/speed.
- [ ] Backend returns deterministic mock telemetry in the contract shape, clearly commented as MOCK pending real integration.
- [ ] AI dark-theme tokens only; reuse `<app-ai-select>`, `<app-kpi-card>`, `<app-ai-skeleton>`; all `[options]` bindings are readonly class fields (FN-317).
- [ ] QA validates flows + responsiveness at 1920/1280/768/375 with screenshot evidence.

## Mock telemetry API contract
```
GET /api/vehicles/:id/telemetry
GET /api/fleet/telemetry?type=truck   // array, for the all-trucks map layer
->
{
  vehicle_id, latitude, longitude, speed_mph, heading_deg,
  fuel_level_pct, odometer, engine_status, last_moved_at,
  fault_codes: [{ code, description, severity }],
  updated_at, source: 'mock'
}
```

## Key Decisions
- **No database subtask.** In-route deterministic mock is sufficient for the stub; a `vehicle_telemetry` table is deferred to when the real integration lands (swap = writer change only). Keeps FE/BE fully parallel with no DB dependency.
- **No 3D map to remove.** The repo has only Leaflet 2D (`loads-dashboard.component.ts`, already switching OSM ↔ ArcGIS satellite). "Remove 3D / use regular map w/ satellite" = build plain Leaflet 2D reusing that tile-layer + `L.control.layers` pattern.
- **Lane reuse, not new epic.** Tagged `epic:fleet-equipment-redesign` (Equipment-section feature) rather than spinning up a new lane/epic for one stubbed story.

## Implementation Summary

### FN-1752 (backend subtask) — Mock telemetry endpoints
- New shared router `backend/packages/goodmen-shared/routes/vehicle-telemetry.js` exposing the FN-1750 contract:
  - `GET /api/vehicles/:id/telemetry` → single mock telemetry payload; `:id` resolved against the existing `all_vehicles` view (tenant-scoped). Returns 404 if the vehicle isn't found (or if the view is unmigrated).
  - `GET /api/fleet/telemetry?type=truck` → array of mock payloads for the all-trucks map layer. `type=truck` filters to "not a trailer" to match `equipment.js` (so the layer stays in sync with the FN-1751 truck dropdown); `type=trailer` and other explicit types also supported. Degrades to `[]` if `all_vehicles` is unmigrated.
- **MOCK data, clearly flagged.** Values are fabricated and deterministic per vehicle — seeded (FNV-1a hash → mulberry32 PRNG) from `unit_number` (fallback `id`) so position/fuel/odometer/fault-codes/engine-state stay identical across reloads; only the relative timestamps advance so it reads as "live". Header comment references `docs/design/telematics-tracking-research-2026-06-02.md` and the no-DB decision.
- **Contract shape:** `{ vehicle_id, latitude, longitude, speed_mph, heading_deg, fuel_level_pct, odometer, engine_status, last_moved_at, fault_codes:[{code,description,severity}], updated_at, source:'mock' }`. Responses are wrapped `{ success, data }` (matches `equipment.js`).
- **Guard chain per-route:** `auth → tenant context → active subscription` (same gating as the sibling `/api/equipment` + `/api/vehicles` routes). Applied per-route (not at mount) so unknown `/api/*` paths still fall through to the service 404 instead of returning 401.
- **No DB work** (per FN-1750 decision — no `vehicle_telemetry` table). The provider-agnostic shape means a real telematics adapter can later replace `buildMockTelemetry` with a reader against a positions table without changing the contract or the FE.
- **Tests:** `vehicle-telemetry.test.js` (`node --test`) — 8 tests covering contract shape, determinism across clock changes, unit_number-seeding, value-range invariants (US bbox / fuel 0-100 / speed 0 unless driving), fault-code well-formedness + severity enum, position scatter, and that both routes are guarded (blocked, never served, without a valid auth/tenant context). All pass under Node 20.

## Deployment Handoff

### FN-1752 (backend)
- **Service to redeploy:** `fleetneuron-vehicles-maintenance-service` (mounts the new shared telemetry router) **and** `fleetneuron-logistics-gateway` (new `/api/fleet` proxy entry). Both consume `backend/packages/goodmen-shared`, so the package must be published/bundled to both on deploy.
- **Gateway paths exposed:** `GET /api/vehicles/:id/telemetry` (existing `/api/vehicles` proxy) and `GET /api/fleet/telemetry` (new `/api/fleet` proxy) → `fleetneuron-vehicles-maintenance-service`.
- **DB migrations:** none (in-route mock; no schema change).
- **Env/config changes:** none. Relies on existing `VEHICLES_MAINTENANCE_SERVICE_URL` on the gateway.
- **Feature flags / cache:** none.
- **Deploy order:** deploy `fleetneuron-vehicles-maintenance-service` first (so the routes exist), then `fleetneuron-logistics-gateway` (so the new `/api/fleet` proxy has a live target). No downtime concerns — additive only.
- **Post-deploy verification:** with a valid session, `GET /api/fleet/telemetry?type=truck` returns a `{ success, data:[…] }` array of mock payloads; `GET /api/vehicles/<known-id>/telemetry` returns one payload with `source: 'mock'`; an unknown id returns 404.
