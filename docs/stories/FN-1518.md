# FN-1518: [Work Order] Service Details tab fields not populating on open

## Requirement

All Service Details tab fields (Service Category, Service Description, Problem Reported, Safety Issue, Downtime Reason, Road Call, Breakdown Location, Estimated Duration) are blank on open and silently lost on save. The `work_orders` table has no backing columns for these fields. Add columns, wire API + UI, restore round-trip.

This story's frontend subtask (FN-1530) also covers the customer auto-select fix from FN-1519 — same `loadWorkOrder` codepath, intentionally collapsed to avoid merge conflicts.

## Implementation Summary

### FN-1528 (subtask)

Knex migration `20260507120000_add_workorder_service_details.js` adds 8 columns to `work_orders` to back the Service Details tab. Pattern follows existing `20260228152000_add_work_orders_financial_columns.js` — uses `hasTable` + per-column `hasColumn` guards so the migration is safe on both fresh DBs (no-op when `work_orders` is missing — backfill migration creates it later) and existing dev DBs (each column added only if missing). Down drops columns symmetrically, also guarded.

Schema delivered exactly as specified in the ticket:
- `service_category` `varchar(64)` nullable
- `service_description` `text` nullable
- `problem_reported` `text` nullable
- `safety_issue` `varchar(32)` nullable (semantic values: `none|low|medium|high`)
- `downtime_reason` `varchar(64)` nullable
- `road_call` `boolean` NOT NULL default `false`
- `breakdown_location` `varchar(255)` nullable
- `estimated_duration_hours` `decimal(6,2)` nullable

No indexes (none of these columns are query targets per ticket).

### FN-1529 (subtask)

Wired the 8 Service Details columns through the work-order service and route layer so the API now accepts, persists, and returns them.

- **`work-orders.service.js`** — added field-level normalizers (`normalizeNullableText`, `normalizeNullableNumber`, `normalizeRoadCall`) and two helpers (`buildServiceDetailsInsert`, `buildServiceDetailsPatch`). Insert uses the full eight-key form on `createWorkOrder`. Update uses the patch form (only includes keys that appear in the payload) so omitted fields keep their existing value — important because the frontend may PATCH a single field at a time. `roadCall` is coerced via accepted truthy strings (`true`/`1`/`yes`) so HTML form submissions of `"false"` don't end up `true`. `getWorkOrderById` already uses `SELECT *` so the new columns return automatically — no SELECT change required.
- **`work-orders-hub.js`** — extended the OpenAPI request-body schemas for `POST /api/work-orders` and `PUT /api/work-orders/{id}` so consumers can discover the eight fields. The handlers themselves needed no logic change — they pass `req.body` straight to the service, which is now responsible for the normalization.
- **`routes/work-orders-service-details.test.js`** (new) — hermetic round-trip integration test using the in-memory mock-knex pattern from `services/parts.service.bulk.test.js`. Four cases: create-with-all-8-fields, empty-string-to-null + roadCall default, update-and-fetch round-trip, and partial-update preserves omitted fields.

### FN-1530 (subtask)

Wired the 8 Service Details fields through the WO load/save codepath and fixed the customer auto-select OnPush refresh bug. Both fixes touched `loadWorkOrder` / `populateCustomerDisplay`, which is why FN-1530 covers both FN-1518 (Service Details) and FN-1519 (customer auto-select) — splitting them would have created a guaranteed merge conflict on the same `setTimeout` block.

- **`work-order.component.ts → loadWorkOrder`** — extends the `Object.assign` call onto `this.workOrder` to pull all 8 Service Details fields out of the API payload (snake_case `service_category` etc., with camelCase fallback for safety). `roadCall` is normalized to a strict boolean so the `[ngValue]="true|false"` select binds correctly even when the backend echoes `null` for legacy rows.
- **`work-order.component.ts → submitWorkOrder`** — adds the same 8 fields to the save payload using the camelCase keys the FN-1529 backend accepts (`buildServiceDetailsInsert` / `buildServiceDetailsPatch`). Sends `roadCall === true` so a falsy local value lands as `false` on insert (matches the column default) rather than as the truthy string `"false"`.
- **`service-details-tab.component.html`** — flipped the Road Call select from string-valued options (`value="No"` / `value="Yes"`) to `[ngValue]="false"` / `[ngValue]="true"` so the model holds a boolean directly. Renamed the Estimated Duration `[(ngModel)]` from `workOrder.estimatedDuration` to `workOrder.estimatedDurationHours` to align with the API field name (`estimated_duration_hours`) — the prior name silently dropped on save because the backend ignored the unknown key.
- **`basics-tab.component.ts → populateCustomerDisplay`** — added `cdr.markForCheck()` after setting `customerSearch` / `selectedCustomer`. Root cause of FN-1519: the method is invoked externally by the parent (`this.basicsTab.populateCustomerDisplay()` from a `setTimeout` in `loadWorkOrder`), but the basics-tab uses `ChangeDetectionStrategy.OnPush`. External method calls don't mark OnPush components dirty, so the input binding stayed stale even though the model was correct (the bug report saw the "Selected: …" status text appear because some later event woke up CD, but the input itself stayed empty until the user re-typed). Also restructured the if/else so the empty-customer-cache branch falls through to the API fetch instead of being mutually exclusive with the in-cache branch — defends against the load-order race where `loadWorkOrder` resolves before `loadCustomers`.

No backend or DB work in this subtask — backend already accepts the camelCase keys and DB already has the columns from the upstream subtasks.

## Key Decisions

### FN-1528 (subtask)
- **Idempotent helper pattern** (`hasColumn` guard per column) chosen over a single `alterTable` block to match the convention established by `add_work_orders_financial_columns.js`. Lets reruns / out-of-order migration recovery succeed without errors.
- **`safety_issue` stored as `varchar(32)`** (not a CHECK / enum) so the API/UI layer can validate the allowed set without requiring a schema migration if the vocabulary expands.
- **No backfill data**: existing rows get `NULL` for nullable columns and `false` for `road_call` via the column default — matches AC ("Existing rows get NULL / FALSE defaults").

### FN-1529 (subtask)
- **Patch vs insert helper split**. `buildServiceDetailsPatch` only emits keys present in the payload (uses `'key' in payload`), so a partial PUT that omits a Service Details field won't overwrite it with `null`. The insert helper always emits all eight keys so new rows get explicit values (and `road_call` gets its `false` default for free). Same idea as the existing `cost_type` patch handling on the line above.
- **No SELECT change in `getWorkOrderById`**. The query is already `SELECT *` from `work_orders`, so the new columns appear in the response without code changes. Verified by the round-trip test, which fetches and asserts all 8 fields.
- **Test placement under `routes/`, not `routes/__tests__/`**. The Jira description suggested `routes/__tests__/work-orders-service-details.test.js`, but every other test in this package lives as a sibling `*.test.js`. Followed repo convention for discoverability and to match the `node --test` working pattern.
- **Hermetic mock knex** rather than a real Postgres harness. Same lightweight pattern as `services/parts.service.bulk.test.js`. Sufficient to assert the round-trip contract without standing up a DB in CI.

### FN-1530 (subtask)
- **`markForCheck` on the OnPush child instead of refactoring to `@Input` flow**. The parent already mutates `this.workOrder` in place (`Object.assign(this.workOrder, …)`), so swapping to `ngOnChanges` would have required immutable input replacement everywhere the workOrder is touched (basics, service-details, work, financials, notes — five tabs). Adding `cdr.markForCheck()` at the existing extension points keeps the change surface tiny and matches the pattern already used in this same file (`fetchFmcsadot`, `createCustomer`, `cancelInlineCustomerCreate`, etc.).
- **Boolean `roadCall` end-to-end**. Backend stores boolean and FN-1529 normalizes `roadCall` per its rules; the cleanest UI binding is `[ngValue]="true|false"` so the model never holds a `"Yes"`/`"No"` string that would round-trip as truthy. The `(wo.road_call ?? wo.roadCall) === true` guard on load makes legacy rows (where `road_call` is null) display "No" rather than blank.
- **Renamed `estimatedDuration` → `estimatedDurationHours` in the template** instead of mapping it in TS on save/load. The HTML field name was a pre-existing typo; the column and the FN-1529 API both use `estimated_duration_hours` / `estimatedDurationHours`. A rename is one diff line and the right canonical form; mapping in TS would have left the typo lurking for the next person.
- **Did not touch the financials/work tabs** even though several of them also exhibit the OnPush stale-binding pattern. Out of scope for FN-1530 (whose acceptance criteria are limited to Service Details + customer auto-select). Logged as a follow-up via this note rather than scope-creeping the bug-fix subtask.

- **No env/config changes**, **no new env vars**, **no cache invalidation**, **no feature flag**.
- **Post-deploy verification**: `POST /api/work-orders` with the eight Service Details fields, then `GET /api/work-orders/{id}` and confirm `data.workOrder.service_category`, `service_description`, `problem_reported`, `safety_issue`, `downtime_reason`, `road_call`, `breakdown_location`, `estimated_duration_hours` are all returned and persisted.

### Frontend (FN-1530)
- **Service to redeploy**: the Angular frontend Render service (`fleetneuron-frontend` — see `.agent/docs/render_services.md`).
- **Deploy order**: FN-1530 frontend deploys **after** FN-1529 backend. Deploying the frontend first would 400 every WO save with `serviceCategory unknown` because the older backend strips unknown keys (or worse, persists nothing while reporting success).
- **No env/config changes**, **no new env vars**, **no migration**, **no feature flag**, **no cache invalidation needed** (Angular bundle filename is hash-busted on each build).
- **Post-deploy verification (manual)**: open an existing WO that has `shop_client_id` set → confirm the customer name input is pre-filled (FN-1519 fix). Switch to Service Details, fill all 8 fields, save, navigate away, reopen → confirm all 8 values reappear (FN-1518 round-trip). Then open a WO with no `shop_client_id` → confirm the customer input stays empty and DOT search still works (regression check).
- **Automated verification**: covered by FN-1531 Cypress round-trip spec.
