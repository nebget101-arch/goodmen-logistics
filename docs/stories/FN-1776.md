# FN-1776: Vehicle Tracking — show city & state in the position section

## Requirement
The Vehicle Tracking Position section shows only raw `lat, lng`. Show a human-readable **City, State** for the vehicle's current position.

The telemetry contract (`GET /api/vehicles/:id/telemetry`, FN-1750/FN-1752) returns coordinates only, and the mock generator scatters vehicles at random lat/lng inside a US bounding box — those points aren't real cities. Fix: seed each vehicle's position from a catalog of real US cities (deterministic per `unit_number`/`id`) and add `city`/`state` to the contract, so coordinates and city/state stay consistent. When real telematics lands, the provider/ingest populates the same fields (reverse-geocoded at ingest) — no frontend rework.

## Contract change
Telemetry payload (single + fleet) gains:
```
city:  string   // e.g. "Dallas"
state: string   // 2-letter, e.g. "TX"
```

## Acceptance Criteria
- [ ] Telemetry payload includes `city` + `state` for every vehicle; deterministic and consistent with `latitude`/`longitude`.
- [ ] Position section shows "City, ST"; raw coordinates + Google Maps link (FN-1768) remain.
- [ ] Graceful fallback when city/state absent — coords only, no "undefined, ".
- [ ] AI dark-theme tokens only; no new hex; builds pass.

### FN-1777 (subtask) — Real-city catalog + city/state in mock telemetry
- Added a `CITY_CATALOG` constant (42 real US cities, `{ city, state, lat, lng }`, 2-letter states) to `vehicle-telemetry.js`, all chosen to sit inside the historical continental-US box (lat 31–44 N, lng -118 to -68 W).
- `buildMockTelemetry()` now deterministically picks a catalog city via the existing seeded `rand()` (so a vehicle always lands on the same city), applies a ±0.03° (~2 mi) jitter to the city's lat/lng, and sets `latitude`/`longitude` from the jittered coords. `city` and `state` are added to the returned payload — consistent with the coordinates.
- Both endpoints (`GET /api/vehicles/:id/telemetry` single + `GET /api/fleet/telemetry`) inherit the change since both go through `buildMockTelemetry`.
- Updated the contract doc comment block and both `@openapi` descriptions to list `city`/`state`.
- Determinism preserved: same `unit_number`/`id` ⇒ same city/state/coords across reloads.
- Tests (`vehicle-telemetry.test.js`): shape test asserts `city` (non-empty) + `state` (2-letter); determinism test asserts city/state stability; new test asserts every vehicle maps to a real catalog city with coords inside the US box and within the jitter bound. All 9 tests pass under `node --test` (Node 20).

## Key Decisions
- City catalog restricted to the existing lat/lng bounding box so the FN-1752 coordinate-range contract (and its tests) still hold after the change.
- Jitter is derived from the seeded `rand()` sequence (not `Math.random()`), so positions stay deterministic per vehicle while looking "live".
- No external reverse-geocoding — the mock owns both the coords and the label, guaranteeing they always agree.
