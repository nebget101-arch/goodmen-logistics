# FN-1460: DQF document upload writes to legacy `dqf_documents` table; FK violation on requirement update

## Requirement

Re-point the DQF document upload flow at the canonical `driver_documents` table so the FK on `dqf_driver_status.evidence_document_id` is satisfied. Backfill any existing `dqf_documents` rows into `driver_documents` to preserve customer-uploaded files.

## Acceptance Criteria

- [ ] `POST /api/dqf-documents/upload` writes to `driver_documents` (not `dqf_documents`).
- [ ] Existing `dqf_documents` rows backfilled into `driver_documents` with same UUIDs (idempotent).
- [ ] All 8 references in `routes/dqf-documents.js` switched to canonical table.
- [ ] End-to-end UI flow: upload driver license → requirement status persists → file downloadable.
- [ ] No regression in `routes/dqf.js`, `routes/public-onboarding.js`, drug-alcohol, or employer-investigation features.
- [ ] `dqf_documents` left in place for one release cycle; follow-up ticket drops it.

## Implementation Summary

### FN-1461 (subtask)

Switched all 8 `dqf_documents` references in `routes/dqf-documents.js` to `driver_documents`. Reads alias canonical column names back to the legacy keys so the JSON response shape is unchanged for the frontend (`doc_type AS document_type`, `storage_key AS file_path`, `size_bytes AS file_size`). The `POST /upload` insert now writes `(driver_id, doc_type, file_name, storage_key, size_bytes, mime_type, storage_mode='r2')`, returning a UUID that satisfies `dqf_driver_status_evidence_document_id_foreign`.

Column-mapping decisions:
- `document_type` → `doc_type`
- `file_path` → `storage_key`
- `file_size` → `size_bytes`
- `storage_mode` set to `'r2'` (matches `routes/drug-alcohol.js` pattern for R2-backed driver_documents rows)
- `blob_id` left NULL (R2-only; no in-DB blob)
- `uploaded_by` dropped from INSERT — column does not exist on `driver_documents` and the parent AC says "prefer reusing existing columns" over adding a new column. SELECTs return `NULL AS uploaded_by` so the response shape stays identical.

### FN-1462 (subtask)

Added a Knex migration that backfills `dqf_documents` rows into `driver_documents`, preserving UUIDs so existing `dqf_driver_status.evidence_document_id` references remain valid post-rollout. Implemented as a single `INSERT ... SELECT` with `LEFT JOIN drivers` for tenant derivation and `ON CONFLICT (id) DO NOTHING` for idempotency. Logs `inserted / skipped / source` counts on every run.

Column-mapping decisions:
- `dqf_documents.document_type` → `driver_documents.doc_type`
- `dqf_documents.file_path` → `driver_documents.storage_key`
- `dqf_documents.file_size` → `driver_documents.size_bytes` (NULL legacy → `0` since canonical column is NOT NULL; legacy file_size was nullable)
- `dqf_documents.mime_type` → `driver_documents.mime_type` (NULL legacy → `'application/octet-stream'`; canonical NOT NULL)
- `storage_mode` set to `'legacy_dqf'` sentinel — clearly distinguishes backfilled rows from new uploads (which FN-1461 writes with `storage_mode='r2'`). Download code can branch on this if legacy storage paths need different handling.
- `tenant_id` derived from `drivers.tenant_id` via JOIN (matches the rule in `20260316220000_add_tenant_id_to_unscoped_root_tables.js`); column remains nullable so rows with an orphaned `driver_id` still copy.
- `packet_id`, `blob_id`, `deleted_at`, and the legacy `uploaded_by` / `updated_at` columns are dropped — no semantic source in `dqf_documents` and the canonical schema either has no equivalent column or the column accepts NULL with a sensible default.

`down()` is intentionally a no-op: once the backend ships, these rows are FK targets for `dqf_driver_status.evidence_document_id`, and reverting would re-introduce the original FK-violation bug.

Tested locally against the dev DB schema: SQL applies cleanly, conflict path skips correctly on re-run, NULL coalescing produces expected canonical values, tenant_id derives correctly. Validation done in a `BEGIN ... ROLLBACK` so dev data was untouched.

## Key Decisions

- **Option 1 chosen** (re-point upload to canonical table) over (a) switching the FK to `dqf_documents` — would break four other features that already use `driver_documents` — or (b) hybrid dual-write — adds complexity without solving the underlying duplication.
- **Drop deferred** — `dqf_documents` stays as a read-only legacy shim for one release cycle; a follow-up ticket drops it after prod verification.
- **Migration is idempotent and additive** — backfill uses `ON CONFLICT (id) DO NOTHING` so it can be re-run and ordering with the backend deploy doesn't matter.
