# FN-541: GET /api/equipment returns empty array — truck and trailer dropdowns blank in load edit modal

## Problem
`GET /api/equipment?type=truck&status=active` and `GET /api/equipment?type=trailer&status=active` both return `{ "success": true, "data": [] }`. The load edit modal truck and trailer dropdowns are completely empty as a result.

Trucks and trailers are visible in the Dispatch → Driver view (e.g. truck `3918`, trailer `TR-2244` assigned to driver Rishawn Williams), confirming the vehicles exist in the DB.

## Root Cause
Two contributing issues were addressed:

### Issue 1 — OE filter excludes fleet vehicles with NULL `operating_entity_id` (Candidate 2)
When `req.context.operatingEntityId` is set (always true for authenticated requests), the truck filter was:
```sql
AND (
  av.operating_entity_id = $N           -- must match OE exactly
  OR av.vehicle_source = 'shop_client'
  OR EXISTS (SELECT 1 FROM drivers d WHERE ...)
)
```
Fleet vehicles created before multi-entity setup have `operating_entity_id = NULL` and fail all three conditions → excluded. Similarly for trailers.

**Fix**: Added `OR av.operating_entity_id IS NULL` as the first condition for both truck and trailer OE filters. Fleet-wide vehicles (NULL OE) are now visible to all operating entities within the same tenant.

### Issue 2 — Silent catch block masks DB errors (Candidate 1 visibility)
The catch block silently returned `[]` when the `all_vehicles` view was missing or had a stale schema (`42P01` / "does not exist" errors) — with no server log. If the migration hasn't been applied, the endpoint appears to return valid empty data with no indication of the real failure.

**Fix**: Added `console.error(...)` before the silent return so DB-level failures surface in Render logs immediately.

## Acceptance Criteria
- [x] `GET /api/equipment?type=truck&status=active` returns fleet trucks with `operating_entity_id = NULL`
- [x] `GET /api/equipment?type=trailer&status=active` returns fleet trailers with `operating_entity_id = NULL`
- [x] Vehicles explicitly scoped to an OE (`operating_entity_id = <id>`) continue to be returned correctly
- [x] If `all_vehicles` view is missing or has a schema error, the error is logged to the console before returning `[]`

## Key Decisions
- `OR av.operating_entity_id IS NULL` is prepended as the first condition in both truck and trailer OE filters — fleet vehicles without an OE assignment are always visible to all OEs within the same tenant.
- The silent `[]` return for missing-view errors is preserved (avoids UI hard-fail) but now always emits a `console.error` first so Render logs will show the true cause.
- If Candidate 1 (missing migration) is confirmed via Render logs post-deploy, the database agent must run `20260317110000_update_all_vehicles_view_vehicle_source_and_is_deleted` migration on the dev/prod DB.
