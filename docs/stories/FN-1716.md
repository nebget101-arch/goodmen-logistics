## Implementation Summary

### FN-1718 (backend subtask)
Scaled the FN-1683 demo position-ping toolkit from ~3‚Äì5 borrowed trucks to a
self-contained 50-truck fleet.

- **Seed (`seed-demo-tracking-loads.js`)** now *mints its own* demo trucks rather
  than borrowing existing in-service ones, so the demo no longer depends on the DB
  already having 50 real trucks. It upserts `DEMO_TRUCK_COUNT` (default 50)
  `vehicles` (`unit_number` `DEMO-TRUCK-001`‚Ä¶, synthetic 17-char `DEMOVIN‚Ä¶` VINs,
  scoped to the resolved tenant) and one `DEMO-TRUCK-NNN` `IN_TRANSIT` load each.
  Fleet size is a single knob: `--count` ‚Ä∫ `DEMO_TRUCK_COUNT` env ‚Ä∫ 50. Routes are
  generated deterministically from a 41-metro table spanning the country; the
  generator guarantees **50 distinct, non-self pickup‚Üídelivery corridors** (no two
  trucks superimposed). Still idempotent (upsert on `unit_number` / `load_number`).
  Tenant resolution gained a "first tenant in `tenants`" fallback so a fresh DB
  still scopes the trucks to a real tenant.
- **Simulator (`demo-tracking-simulator.js`)** already did one bulk insert per tick;
  FN-1718 made the rest scale: `loadDemoRoutes()` is now **3 queries total**
  (loads, all stops, all zips) instead of `1 + 2¬∑N` (‚âà3 vs 100+ round-trips/tick at
  50 trucks), arrival handling no longer churns the DB every tick (loop mode keeps
  the load `IN_TRANSIT` in-memory; once mode batches the `DELIVERED` update), and
  each tick logs its duration (`inserted N ping(s) in Mms`) so perf is observable.
- **Teardown (`teardown-demo-tracking.js`)** now also removes the minted
  `DEMO-TRUCK-` trucks, all in one transaction (pings ‚Üí stops ‚Üí loads ‚Üí trucks, in
  FK-safe order) for a no-orphans guarantee. Pre-FN-1718 demo loads that borrowed
  real trucks leave those real trucks untouched.

Acceptance criteria: all five met (50 trucks/loads ‚úì, single `DEMO_TRUCK_COUNT`
knob ‚úì, all interpolated each tick with bearing heading + 50‚Äì65 mph ‚úì, one bulk
insert/tick with sub-interval timing ‚úì, clean teardown incl. trucks ‚úì).

## Key Decisions
- **Mint dedicated demo trucks** (vs. borrowing in-service trucks). The AC says the
  seed *creates* 50 trucks and teardown *removes* them; minting makes the demo
  self-contained and teardown exact (`unit_number LIKE 'DEMO-TRUCK-%'`).
- **Loop mode keeps loads `IN_TRANSIT`** instead of the original
  DELIVERED‚ÜíIN_TRANSIT round-trip on every arrival ‚Äî avoids a 100-write burst when
  all 50 trucks arrive on the same tick; the visible behaviour (sit 30s, restart)
  is unchanged.
- **Linear interpolation kept** (demo, not navigation) per the FN-1683 decision.

## Deployment Handoff
- **Type:** operational CLI scripts run manually ‚Äî **no service deploy, no
  migration, no env-var change** required to ship this subtask.
- **Services to restart/redeploy:** none. (The pings the simulator writes are
  broadcast by the existing Story D WS path on `fleetneuron-logistics-service` /
  gateway ‚Äî no code change there.)
- **DB migrations:** none. Uses existing `vehicles`, `loads`, `load_stops`,
  `zip_codes`, `vehicle_position_pings` tables. Note `vehicle_position_pings` is
  partitioned by day (maintained by the `fleetneuron-telematics-partition-cron`);
  the simulator's inserts land in the current day partition automatically.
- **How to run for a client demo** (against the target DB via `DATABASE_URL`):
  1. `node backend/scripts/seed-demo-tracking-loads.js` (note the tenant id it prints)
  2. Log in as a user of that tenant
  3. `node backend/scripts/demo-tracking-simulator.js` (leave running)
  4. Open `/tracking` ‚Üí ~50 markers moving
  5. After: `Ctrl+C`, then `node backend/scripts/teardown-demo-tracking.js`
- **Cleanup:** teardown removes all demo trucks/loads/stops/pings (no orphans);
  `zip_codes` reference rows are intentionally left.
