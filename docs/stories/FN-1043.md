# FN-1043: State dropdown does not pre-populate on Stops tab

## Type
Bug — Severity: Medium

## Symptom
Expanding any stop card on the Stops tab shows an empty State dropdown, even when DB or AI-extracted data clearly contains a state.

## Confirmed Root Cause
- `frontend/src/app/shared/components/stop-card/stop-card.component.ts:71–76` hard-codes 51 uppercase 2-letter codes.
- Template binds `<select [value]="stop.state || ''">` — only matches exact uppercase codes.
- `backend/packages/goodmen-shared/services/load-ai-extractor.js` accepts raw `string | null` from LLM with no normalization, so `"California"`, `"ca"`, `"Ca."` flow into `load_stops.state`.
- Legacy rows have the same mixed-format problem.

- [x] No regression for stops already stored as "TX", "CA", etc. *(FN-1049 + FN-1050 — both pass-through valid 2-letter codes unchanged)*

## Implementation Summary

### FN-1049 (subtask) — Frontend stop-card normalization

The stop-card dropdown previously bound `<select [value]="stop.state || ''">` against a
hard-coded list of 51 uppercase 2-letter codes, so any non-conforming value (full state
name, lowercase, trailing punctuation) failed to match an option and the dropdown
rendered empty even though `stop.state` had a value.

Added a name → 2-letter code map covering the 50 states + DC (key = uppercased name) and
a `normalizeStateCode(input)` instance method that:
- Trims, strips trailing `.` / `,`, collapses internal whitespace, uppercases.
- Returns the input unchanged when it is already a valid 2-letter code from `stateList`.
- Looks the cleaned key up in `STATE_NAME_TO_CODE`.
- Returns `''` for `null`, `undefined`, blank, or unknown input — so the "--" option
  remains selected and we never write `"California"` back to the DB.

Template change: `<select [value]="normalizedState" (change)="onStateChange(...)">`
where `normalizedState` is a getter that calls `normalizeStateCode(stop.state)`. The
`onStateChange` handler normalizes the dropdown value and emits the stop with `state`
set to the 2-letter code (or `null` when blank/unknown).

`cityStateLabel` was also updated so the collapsed-row display shows `City, CA` instead
of `City, California` once a legacy load is loaded.

### FN-1050 (subtask) — Backend persistence + extractor + backfill

Backend data-normalization counterpart. Added a `normalizeStateCode(input)` util and applied it everywhere that writes `load_stops.state`:

- `backend/packages/goodmen-shared/utils/state-code.js` — exports `normalizeStateCode` and `STATE_NAME_TO_CODE`. Trims/uppercases, strips `.`/`,`, collapses internal whitespace, maps full state names (50 + DC) to 2-letter codes, returns `null` for anything unrecognized (no silent coercion).
- `backend/packages/goodmen-shared/utils/state-code.test.js` — 7 `node --test` cases covering 2-letter pass-through, mixed case + punctuation, full-name mapping, all 50+DC, unknown input, null/non-string. All pass.
- `backend/packages/goodmen-shared/services/load-ai-extractor.js` — normalizes `state` on every emitted stop and on top-level `pickup`/`delivery`, in both the OpenAI text path and the Claude vision fallback path.
- `backend/packages/goodmen-shared/routes/loads.js` — normalizes `state` on every `INSERT INTO load_stops` and on the PATCH `/api/loads/:id/stops/:stopId` update path. Five INSERT sites + one PATCH site covered. This is defense-in-depth; the frontend is expected to normalize too (FN-1049).
- `backend/packages/goodmen-database/migrations/20260502120000_normalize_load_stops_state.js` — idempotent backfill of legacy `load_stops.state` rows. Selects only rows where `state` is non-null AND not already an uppercase 2-letter code, then updates them to the normalized value (or `null` if unrecognized). Re-running on a clean DB is a no-op. Mapping is inlined so the migration stays self-contained.

## Key Decisions

**FN-1049 (frontend):**
- **Normalization lives in the component, not in a shared util.** Per the subtask
  acceptance criteria the change is scoped to stop-card; promoting it to a shared
  service would force a sibling-subtask coordination this ticket does not require.
  If a second consumer appears, hoist the static map and the function then.
- **`normalizedState` is a getter**, not a memoised field. It returns a primitive
  string so Angular's dirty-check compares by value — no infinite-CD loop risk
  (the FN-317 incident only applies to `[options]` array bindings on
  `app-ai-select`, not native `<select [value]>` with primitives).
- **Unknown input collapses to `null`**, never to a partial value. This matches the
  AC "no regression for stops already stored as `TX`, `CA`" — a typo like
  `Californa` will simply clear the field rather than silently persist garbage.

**FN-1050 (backend):**
- `normalizeStateCode(garbage)` returns `null`, not the original. Falsy garbage shouldn't survive the persistence layer — better to drop it than to write `"California "` to a column the dropdown can't match.
- Mapping inlined in the migration rather than `require()`-ing the shared util. Migrations run with `goodmen-database` as their cwd and shouldn't depend on `goodmen-shared` package resolution at migrate time.
- Replaced `normalizeNullable(stop.state)` with `normalizeStateCode(stop.state)` at INSERT sites — the latter already handles null/empty and gives the trim-and-uppercase semantics for free.

## Deployment Handoff

**FN-1049 (frontend):**
- Render service to redeploy: `fleetneuron-logistics-ui` (per `.agent/docs/render_services.md`).
- No DB migrations / env-var changes / cache-invalidation needs.
- The frontend fix is independent of FN-1050 — even with legacy `state="California"` rows still in the DB, the dropdown will display `CA` correctly. FN-1050 + the backfill migration just ensure newly-extracted stops persist 2-letter codes from the start.
- Post-deploy verification: open any existing load whose stops have full state names → confirm Stops tab dropdown shows the 2-letter code; edit → save → confirm outgoing payload contains the 2-letter code.

**FN-1050 (backend):**
- Render services to redeploy: any service that loads `@goodmen/shared` with the `loads` route or AI extractor — `fleetneuron-logistics-service`, `fleetneuron-ai-service`, and `fleetneuron-logistics-gateway` (per `.agent/docs/render_services.md`). Code-only changes; no env vars, no flags, no cache invalidation.
- DB migration: `20260502120000_normalize_load_stops_state.js`. Optional but recommended — without it, legacy rows still hold full state names and the FN-1049 frontend fallback will have to keep handling them. Idempotent and small (only touches rows where `state` is non-null and not already a 2-letter uppercase code). Run via the standard `knex migrate:latest` step on dev/prod (`fleetneuron-db-migrations` service).
- Post-deploy verification: open any pre-existing load drawer; `load_stops.state` for newly saved stops should be 2-letter uppercase. AI-extracted loads (POST a rate-con PDF that has "California" on it) should land in DB as "CA".

**Combined deploy order (recommended):**
1. `fleetneuron-db-migrations` — run the state backfill migration.
2. Backend services — `fleetneuron-logistics-service`, `fleetneuron-ai-service`, `fleetneuron-logistics-gateway`.
3. `fleetneuron-logistics-ui` — frontend.

Order isn't load-bearing (each layer tolerates the others' old/new state) but
front-loading the migration gives QA a clean dataset to validate FN-1051 against.
