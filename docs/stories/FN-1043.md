# FN-1043: State dropdown does not pre-populate on Stops tab

## Type
Bug — Severity: Medium

## Symptom
Expanding any stop card on the Stops tab shows an empty State dropdown, even when DB or AI-extracted data clearly contains a state.

## Confirmed Root Cause
- `frontend/src/app/shared/components/stop-card/stop-card.component.ts:71–76` hard-codes 51 uppercase 2-letter codes.
- Template binds `<select [value]="stop.state || ''">` — only matches exact uppercase codes.
- `backend/packages/goodmen-shared/services/load-ai-extractor.js` accepts raw `string | null` from LLM with no normalization, so `"California"`, `"ca"`, `"Ca."` flow into `load_stops.state`.
- Legacy rows have the same mixed-format problem.

## Acceptance Criteria
- [ ] Legacy load with `state="California"` → dropdown shows "CA" selected
- [ ] PDF-extracted load with full state name → dropdown selected
- [x] Newly saved stop persists 2-letter uppercase code in DB *(FN-1050)*
- [x] No regression for stops already stored as "TX", "CA", etc. *(FN-1050)*

### FN-1050 (subtask)
**Implementation**

Backend data-normalization counterpart. Added a `normalizeStateCode(input)` util and applied it everywhere that writes `load_stops.state`:

- `backend/packages/goodmen-shared/utils/state-code.js` — exports `normalizeStateCode` and `STATE_NAME_TO_CODE`. Trims/uppercases, strips `.`/`,`, collapses internal whitespace, maps full state names (50 + DC) to 2-letter codes, returns `null` for anything unrecognized (no silent coercion).
- `backend/packages/goodmen-shared/utils/state-code.test.js` — 7 `node --test` cases covering 2-letter pass-through, mixed case + punctuation, full-name mapping, all 50+DC, unknown input, null/non-string. All pass.
- `backend/packages/goodmen-shared/services/load-ai-extractor.js` — normalizes `state` on every emitted stop and on top-level `pickup`/`delivery`, in both the OpenAI text path and the Claude vision fallback path.
- `backend/packages/goodmen-shared/routes/loads.js` — normalizes `state` on every `INSERT INTO load_stops` and on the PATCH `/api/loads/:id/stops/:stopId` update path. Five INSERT sites + one PATCH site covered. This is defense-in-depth; the frontend is expected to normalize too (FN-1049).
- `backend/packages/goodmen-database/migrations/20260502120000_normalize_load_stops_state.js` — idempotent backfill of legacy `load_stops.state` rows. Selects only rows where `state` is non-null AND not already an uppercase 2-letter code, then updates them to the normalized value (or `null` if unrecognized). Re-running on a clean DB is a no-op. Mapping is inlined so the migration stays self-contained.

**Files Changed**
- `backend/packages/goodmen-shared/utils/state-code.js` (new)
- `backend/packages/goodmen-shared/utils/state-code.test.js` (new)
- `backend/packages/goodmen-shared/services/load-ai-extractor.js` (state normalization on stops + pickup/delivery in text + vision paths)
- `backend/packages/goodmen-shared/routes/loads.js` (state normalization at every persistence site)
- `backend/packages/goodmen-database/migrations/20260502120000_normalize_load_stops_state.js` (new, optional one-shot backfill)
- `docs/stories/FN-1043.md` (this update)

**Key Decisions**
- `normalizeStateCode(garbage)` returns `null`, not the original. Falsy garbage shouldn't survive the persistence layer — better to drop it than to write `"California "` to a column the dropdown can't match.
- Mapping inlined in the migration rather than `require()`-ing the shared util. Migrations run with `goodmen-database` as their cwd and shouldn't depend on `goodmen-shared` package resolution at migrate time.
- Replaced `normalizeNullable(stop.state)` with `normalizeStateCode(stop.state)` at INSERT sites — the latter already handles null/empty and gives the trim-and-uppercase semantics for free.

**Verification**
- `node --test utils/state-code.test.js` → 7/7 pass.
- `node -c routes/loads.js` and `node -c services/load-ai-extractor.js` → both clean.
- Pre-existing test failures in this package are unrelated: missing `node_modules` for `express`, `@goodmen/database`, etc. inside the worktree — none touch state code or load_stops persistence.

## Deployment Handoff

**FN-1050 (backend):**
- Render services to redeploy: any service that loads `@goodmen/shared` with the `loads` route or AI extractor — `logistics-service`, `ai-service`, and the `gateway` (per `.agent/docs/render_services.md`). Code-only changes; no env vars, no flags, no cache invalidation.

Optional but recommended — without it, legacy rows still hold full state names and the FN-1049 frontend fallback will have to keep handling them. Idempotent and small (only touches rows where `state` is non-null and not already a 2-letter uppercase code). Run via the standard `knex migrate:latest` step on dev/prod.
- Deploy order: migration → services. Services tolerate either DB state — but running the migration first means the backfill completes before any FN-1049 frontend deploy and a clean dataset is what QA (FN-1051) will validate against.
- Post-deploy verification: open any pre-existing load drawer; `load_stops.state` for newly saved stops should be 2-letter uppercase. AI-extracted loads (POST a rate-con PDF that has "California" on it) should land in DB as "CA".
