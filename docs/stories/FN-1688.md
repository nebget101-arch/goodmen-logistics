# FN-1688: Story C — Subscription management API + Customer Portal

## Requirement
Customers can only add a card and buy seats — no way to view their subscription, change plans, see invoices, or cancel. Add the subscription-management API surface and a Stripe Customer Portal hand-off. **API request/response shapes are defined below so Story D (frontend) can build in parallel.**

## Acceptance Criteria
- [x] `POST /api/billing/portal-session` → `stripe.billingPortal.sessions.create`, returns redirect URL (tenant-scoped, billing-admin only).
- [x] `POST /api/billing/change-plan` → `stripe.subscriptions.update` with proration; validate target against `plans.js`; update `tenants.subscription_plan`.
- [x] `POST /api/billing/cancel` → cancel at period end (reuse `stripeService.cancelSubscription`).
- [x] `customer.subscription.deleted` webhook → sync local state (clear/flag `stripe_subscription_id`, audit log).
- [x] `GET /api/billing/subscription` → plan, status, current period start/end, next renewal, line items, cancel-at-period-end flag.
- [x] `GET /api/billing/invoices` → Stripe invoices list (id, date, amount, status, hosted_invoice_url/pdf).
- [x] All endpoints billing-admin gated; no info leak in errors; idempotency keys on Stripe writes.

## API contract (AUTHORITATIVE — frontend builds against this)
> Finalized by FN-1696. All endpoints are mounted under `/api/billing`, require a Bearer token, and are
> gated to billing-admin roles (`super_admin`, `admin`, `company_admin`). **Every response uses the
> `{ success, data }` envelope** already used by the rest of `billing.js`; the shapes below describe `data`.
> Error responses are `{ success: false, error: <generic message> }` — no Stripe internals are leaked.
>
> Stripe write endpoints (`change-plan`, `cancel`) accept an optional `Idempotency-Key` request header;
> if omitted the backend mints one per request so SDK/network retries are safe.

- `POST /api/billing/portal-session` → `data: { url: string }`
  - 400 if the tenant has no Stripe customer yet.
- `GET /api/billing/subscription` → `data: { planId, status, currentPeriodStart, currentPeriodEnd, nextRenewal, cancelAtPeriodEnd, lineItems[] }`
  - `status` is the Stripe subscription status (`active`, `trialing`, `past_due`, `canceled`, …) or `"none"` when the tenant has no active subscription.
  - `currentPeriodStart` / `currentPeriodEnd` / `nextRenewal` are ISO-8601 strings (or `null`). `nextRenewal` is `null` when `cancelAtPeriodEnd` is true.
  - `lineItems[]` = `{ priceId, nickname, quantity, unitAmount, currency, interval }` (`unitAmount` in cents).
- `POST /api/billing/change-plan` body `{ planId: 'basic'|'multi_mc'|'end_to_end'|'enterprise' }` → `data: { …subscription }` (same shape as `GET /subscription` `data`).
  - 400 if `planId` is invalid, the plan has no configured Stripe price, or the tenant has no active subscription.
- `POST /api/billing/cancel` → `data: { cancelAtPeriodEnd: true, currentPeriodEnd: ISO|null }`
  - 400 if there is no active subscription.
- `GET /api/billing/invoices` → `data: { invoices: [{ id, created, amountDue, amountPaid, currency, status, hostedInvoiceUrl, pdfUrl }] }`
  - `created` is ISO-8601. `amountDue` / `amountPaid` are in the smallest currency unit (cents). Returns `{ invoices: [] }` when the tenant has no Stripe customer or billing is disabled.

## Implementation Summary
Added the subscription-management API surface to `auth-users-service` and a `customer.subscription.deleted`
webhook sync. All five endpoints are billing-admin gated (existing `requireBillingAdmin` middleware),
return sanitized errors via a shared `sendBillingError` helper (Stripe detail is logged server-side, never
returned), and the two Stripe write endpoints pass an idempotency key (client `Idempotency-Key` header or a
freshly minted one) through to Stripe.

### FN-1696 (subtask)
**Branch:** `backend/FN-1696/subscription-management-api`

`stripeService.js` — added `createBillingPortalSession`, `getSubscription`, `changePlan`, `listInvoices`,
and an optional `idempotencyKey` arg on `cancelSubscription`. `changePlan` retrieves the subscription,
finds the **base** plan item (the item whose price is not the extra-seat add-on price), and updates only
that item's price with `proration_behavior: 'create_prorations'` so any extra-seat line is left untouched.

`routes/billing.js` — added `POST /portal-session`, `GET /subscription`, `POST /change-plan`,
`POST /cancel`, `GET /invoices`. Plan validation reuses `normalizePlanId` + `VALID_PLAN_IDS` from
`plans.js`; price IDs come from the `STRIPE_PRICE_*` env vars via the same `PLAN_PRICE_MAP` pattern as
`jobs/processTrialConversions.js`. `change-plan` updates `tenants.subscription_plan` only after the Stripe
update succeeds. Period timestamps are read from the subscription (falling back to the first item) to stay
compatible with newer Stripe API versions that moved `current_period_*` onto items.

`routes/stripe.js` — added `handleSubscriptionDeleted` wired into `routeEvent` for
`customer.subscription.deleted`: looks the tenant up by `stripe_subscription_id`, clears it, flags
`trial_status = 'canceled'`, and writes a `subscription_deleted` audit log. Best-effort and never throws,
matching the other async webhook handlers.

## Key Decisions
- **`{ success, data }` envelope** kept for all new endpoints to match the rest of `billing.js`; the documented contract shapes describe `data` (frontend reads from `data`).
- **No-info-leak**: a single `sendBillingError` helper logs the underlying Stripe/internal message server-side and returns only a generic public message; the existing `stripeService` already wraps Stripe errors in generic messages.
- **Idempotency**: write endpoints honor a client `Idempotency-Key` header and otherwise mint a UUID-based key per request; the key is threaded into the Stripe SDK call options.
- **`change-plan` targets the base item only** so an existing extra-seat line item is never overwritten.
- **`subscription.deleted` flag**: set `trial_status = 'canceled'` and clear `stripe_subscription_id` (no schema change — `trial_status` is `varchar(20)`, no CHECK constraint).
- **Tests** for proration / cancel / portal / sync / no-leak are owned by the dedicated QA subtask **FN-1697** (requires Stripe mocking); none added here.
