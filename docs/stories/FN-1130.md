## Role Defaults

**Dispatcher**: Daily Briefing → Smart Alerts (load) → Predictive Insights (load volume + on-time %) → Quick Actions
**Safety**: Daily Briefing → Smart Alerts (HOS + fatigue) → Predictive Insights (compliance) → Quick Actions
**Maintenance**: Daily Briefing → Smart Alerts (inspection + maintenance) → Predictive Insights (maintenance windows + fuel) → Quick Actions
**Owner**: Daily Briefing → Predictive Insights (all 4) → Smart Alerts (top severity)

## Implementation Summary

### FN-1172 (subtask)
Added the `auth-users-service` REST surface for the Control Center layout: `GET/PUT/DELETE /api/users/me/dashboard-layout`, backed by `user_dashboard_layouts` (FN-1174). The store falls back to a role-keyed default layout when no row exists, so a freshly-onboarded user immediately sees a sensible Control Center without the frontend having to seed anything. DELETE provides the reset-to-default action consumed by FN-1171.

- `services/layout-store.js` exports `createLayoutStore({ knex })` plus pure helpers (`getRoleDefault`, `normalizeRoleKey`, `ROLE_DEFAULTS`, `DEFAULT_ROLE_KEY`) so the frontend/QA can mirror the role mapping without a network round-trip if needed.
- Role aliases collapse legacy JWT codes to four canonical keys: `dispatcher` (`dispatch`, `dispatcher`), `safety` (`safety`, `safety_manager`), `maintenance` (`maintenance`, `mechanic`, `technician`), `owner` (`owner`, `admin`, `super_admin`, `platform_admin`, `accounting`). Unknown roles fall back to `dispatcher`.
- Upsert uses `INSERT … ON CONFLICT (user_id) DO UPDATE` so PUT is idempotent and matches the FN-1174 PK.
- Payload size ceiling: 16 KB (smaller than `user-preferences`' 32 KB because the layout is just a card-order list, not a full preference graph).
- Tests use `node:test` and an in-memory store stub — no Postgres needed in CI. Mirrors the convention from `packages/goodmen-shared/routes/user-preferences.test.js`.

### FN-1174 (subtask)
Added knex migration creating `user_dashboard_layouts` to back the per-user layout persistence consumed by FN-1172 (layout endpoints) and FN-1171 (Control Center drag-reorder + reset).

- One row per user (`user_id` PK), so reads/writes are point lookups, not range scans.
- `tenant_id` is a NOT NULL FK + secondary index for tenant-scoped reads and ON DELETE CASCADE cleanup paths.
- `layout_json jsonb` defaults to `'{}'` so GETs before the user has reordered anything return an empty object instead of NULL — backend can fall back to the role default cleanly.
- FKs cascade on user/tenant delete, matching the surrounding `users` / `tenants` cleanup contract used by `user_alert_dismissals` (FN-1165) and `user_notifications` (FN-507).

## Key Decisions

### FN-1172 (subtask)
- **Service-local route, not shared router** — the ticket scopes the files to `auth-users-service/routes/dashboard-layout.js`. Mounting it directly on `auth-users-service/server.js` (rather than adding `/me/dashboard-layout` to `packages/goodmen-shared/routes/users.js`) keeps the layout feature contained to the one microservice that owns user profile state, and matches the pattern set by `routes/billing.js` for service-specific concerns.
- **Mount BEFORE `/api/users`** — Express forwards `/api/users/me/dashboard-layout` to `usersRouter` first if mounted later; the shared router doesn't define that path so it would fall through, but mounting the more-specific path first avoids any chance of regression if the shared router later gains a wildcard `/me/*` route.
- **Role key normalization in the store, not the route** — the store is the single source of truth for the role→default-layout mapping. The route just reads `req.user.role` from the JWT and forwards it. Frontend (FN-1171) re-implements the same mapping for offline / first-paint use, but the backend default is authoritative.
- **`DELETE` (not `POST /reset`) for reset-to-default** — REST-clean and matches the table semantics: removing the row is exactly what "reset" means. The endpoint returns the role default in the response so the frontend doesn't have to issue a follow-up GET.
- **16 KB payload cap** — sized for the actual data (a small `cards: []` list); rejects accidental blob uploads loudly without becoming a constraint to design around.

### FN-1174 (subtask)
- **`user_id` as PK, not a surrogate `id` UUID** — FN-1172 will upsert per user (one layout per user). A composite/unique constraint on `user_id` would force a separate PK; using `user_id` as PK is simpler and what the ticket specifies.
- **Skipped the README append** — the ticket lists `backend/migrations/README.md` as a touched file, but the actual migrations directory is `backend/packages/goodmen-database/migrations/` and there is no per-folder migrations README in this codebase (recent migrations like FN-1165, FN-507 don't update one). Following established convention.
- **`layout_json` default `'{}'`, not NULL** — keeps the FN-1172 endpoint contract trivial (always returns a JSON object).

## Deployment Handoff

### FN-1172 (subtask)
- **Services to redeploy:** `goodmen-auth-users-service` (Render). Reference `.agent/docs/render_services.md` for the canonical service name.
- **Order:** Deploy *after* FN-1174's migration has run on dev (the route reads/writes `user_dashboard_layouts`). On the integration branch the migration commit (4c47e07) precedes this one, so a single `migrate:latest` + service restart on dev is sufficient.
- **Migrations:** None added by this subtask. The endpoint depends on `20260504200000_create_user_dashboard_layouts.js` (FN-1174).
- **Env / config / flags / cache:** None changed. No new env vars.
- **Post-deploy verification:**
  1. `curl -H "Authorization: Bearer <jwt>" https://<auth-users-service>/api/users/me/dashboard-layout` → 200, `data.is_default=true`, `data.role` matches the JWT role, `data.layout.cards` is the role default.
  2. `curl -X PUT -H "Authorization: Bearer <jwt>" -H "Content-Type: application/json" -d '{"cards":["daily-briefing"]}' …/dashboard-layout` → 200, `data.is_default=false`.
  3. Subsequent GET returns the persisted layout (not the role default).
  4. `curl -X DELETE …/dashboard-layout` → 200, response shape returns role default.
- **Rollback:** Roll back the service to the prior image; the new endpoint is additive — no existing route paths are altered. Existing data in `user_dashboard_layouts` is harmless once the consuming endpoint is gone (FN-1174 migration can stay).
