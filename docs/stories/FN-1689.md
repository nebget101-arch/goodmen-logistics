## Implementation Summary
The `/billing` page was rewritten from the legacy light theme to the AI dark theme and now provides full self-service subscription management:

- **Dark-theme rewrite** — header (kicker/title/subtitle), panels (navy glass-morphism via documented rgba), and all status/plan/invoice surfaces re-tokenized. Loading uses the FN-1636 `<app-ai-skeleton>` primitive; the KPI summary row uses `<app-kpi-card>`; the contextual status banner uses the `.ai-panel-flat` utility with `data-severity`. No new hex — every color is a `--ai-*` / `--success/--warning/--danger-color` CSS variable or a documented palette `rgba()`.
- **Single plan source** — `marketing.config.ts` is the one source for plan name/price/features. Added `PLAN_ORDER`, `getMarketingPlan()`, `getPlanMonthlyPriceUsd()`, and `getPlanRank()` helpers (consumed by both the marketing site and the in-app plan comparison). The plans grid highlights the current plan and offers Upgrade / Switch / Contact actions based on tier rank.
- **Payment method** — reuses the existing `PaymentMethodFormComponent` dialog (add/update/remove), unchanged.
- **Banner states** — `<app-trial-banner>` gained a 4th `state-d` (past-due / unpaid) that overrides trial states and prompts a card update; the billing page also renders an in-page contextual status banner for active / trial / card-saved / expired / past-due / cancelling.
- **Invoice history** — `GET /api/billing/invoices` rendered in a dark-theme table with status pills and hosted-invoice + PDF links (Stripe cents normalized to currency; unix-seconds timestamps normalized to ms).
- **Customer Portal** — "Manage in portal" button calls `POST /api/billing/portal-session` and redirects to the returned URL (redirect isolated in a `protected redirect()` method for testability).
- **Change-plan / cancel** — upgrade/downgrade and cancel run through an in-page confirmation modal with a client-side proration preview derived from the plan price delta. Cancel is a "cancel at period end" flow shown in a danger zone, hidden once already scheduled to cancel.
- **Upgrade-to-unlock** — a seat-limit prompt renders when `seat-usage` reports active users ≥ effective seat limit, linking to the portal to add seats/upgrade.
- **Resilience** — `trial-state.service` now also fetches `GET /api/billing/subscription` (status, cancelAtPeriodEnd, currentPeriodEnd, nextRenewal) but wraps it in `catchError(() => of(null))`, and the billing page's invoice/seat loads degrade quietly, so the page works before FN-1688 (Story C) deploys.

## Key Decisions
- **No backend changes in this subtask.** Built strictly against the FN-1688 API contract; all new API calls degrade gracefully so the page is usable before Story C merges to `dev`.
- **Client-side proration preview.** The FN-1688 contract has no preview endpoint, so the confirmation modal computes an indicative proration note from the plan price delta and states that Stripe applies the exact proration. If Story C later exposes a preview endpoint, swap the note for the server value.
- **Invoice amounts treated as Stripe cents** (`/100`) per Stripe's native invoice shape; timestamps normalized from unix seconds.
- **FN-317 compliance:** `plans` is a `readonly` class field, not a getter; no `app-ai-select` `[options]` getter bindings introduced.
