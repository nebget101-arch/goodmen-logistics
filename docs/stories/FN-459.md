# FN-459: Implement Fuel Card-to-Driver Assignment with Smart Matching

## Requirement
Link fuel cards to specific drivers so that imported transactions can be auto-matched by card, and mismatches between the card's assigned driver and the CSV driver name generate exceptions.

## Acceptance Criteria
- [ ] Fuel cards can have a driver assigned/reassigned/revoked
- [ ] Assignment history is tracked (assigned_at, revoked_at, status)
- [ ] Imported transactions auto-match the assigned driver when card matches
- [ ] driver_card_mismatch exception generated when CSV driver differs from assigned
- [ ] Fuel Exceptions tab shows driver_card_mismatch exception type with resolve flow

frontend + backend + database

## Implementation Summary

### FN-460 (database)
- Created migration `20260328300000_create_fuel_card_driver_assignments.js`
- Table: `fuel_card_driver_assignments` with UUID PK, tenant_id, fuel_card_account_id FK, driver_id FK, card_number_last4, assigned_at, revoked_at, status (active/revoked), notes
- Unique partial index on active assignments per card (only one active assignment per card at a time)

### FN-461 (backend)
- Added 4 endpoints to `fuel.js` routes:
  - `GET /cards/:cardId/assignments` — list all assignments for a card
  - `POST /cards/:cardId/assign-driver` — assign or reassign a driver to a card
  - `POST /cards/:cardId/revoke-driver` — revoke active assignment
  - `GET /driver-assignments` — all active assignments for tenant
- Updated `commitBatch()` in `fuel-service.js`: checks active assignment for card, uses assigned driver_id override, creates `driver_card_mismatch` exception when CSV driver differs

### FN-462 (frontend)
- Added `CardDriverAssignment` interface to `fuel.model.ts`
- Added `getCardAssignments`, `assignDriver`, `revokeDriver` methods to `FuelService`
- Rewrote `FuelCardsComponent` with full assignment state management:
  - `assignedDriverMap` built via forkJoin across all cards
  - Assign/Reassign/Revoke action buttons per row
  - Assignment dialog modal with driver select + notes
  - Assignment History modal with full history table and status pills
  - Assigned Driver column in cards table

## Key Decisions
- One active assignment per card enforced by unique partial index in DB
- Assignment uses card_number_masked for matching if card_number_last4 present
- driver_card_mismatch exception created during commitBatch when assigned driver differs from CSV driver
- Frontend uses plain `<select>` (not app-ai-select) for driver picker to avoid infinite CD loop

## Deployment Handoff
- **Services**: Backend (goodmen-shared), Frontend (goodmen-frontend)
- **Migrations**: `20260328300000_create_fuel_card_driver_assignments.js` — run first
- **Env/config**: None
- **Deploy order**: Run DB migration first, then deploy backend, then frontend
- **Post-deploy verification**: Assign a driver to a fuel card, import a CSV with that card number — verify transaction auto-matches the assigned driver
