# FN-1413: FMCSA Census + Operating Authority bulk importer

## Requirement

Replace the SAFER snapshot scraper and the `li-public.fmcsa.dot.gov` Licensing & Insurance scraper with two bulk-file importers:

1. **Motor Carrier Registrations Census** (`data.transportation.gov` dataset `4a2k-zf79`) — populates `fmcsa.carriers` with USDOT, MC/MX/FF, names, addresses, **phone, fax, email**, power units, drivers, mileage, hazmat/passenger flags
2. **Carrier - All With History** (`data.transportation.gov` dataset `6qg9-x4f8`) — populates `fmcsa.authorities` for carriers, **brokers**, and freight forwarders with authority types and insurance presence

### FN-1420 (subtask)

**Branch:** `backend/FN-1420/fmcsa-census-authority-import` (off `origin/integration/FN-1413`)

**What shipped:**
- Streaming RFC-4180 CSV parser (`utils/csv-stream.js`) — Transform stream with O(longest-row) memory; smoke-tested on 50k synthetic rows, designed to handle 2M+ row FMCSA bulk files.
- Versioned row parsers — `parsers/census.v1.js` and `parsers/authority.v1.js`. Both are header-name-driven (case-insensitive, with synonym lists) so small FMCSA column-name drifts don't break the importer; renames trigger a v2 instead of editing v1 in place.
- Job runner (`runner.js`) — owns the `fmcsa.import_runs` audit row, the streaming pipeline (HTTP→gunzip→CSV→batch upserter), and counts (`inserted`, `updated`, `skipped`).
- Census job (`census.js`) — batched `INSERT … ON CONFLICT (dot) DO UPDATE … WHERE … IS DISTINCT FROM …` so unchanged rows never touch `fmcsa_synced_at`. Returns counts via `RETURNING (xmax = 0)`.
- Authority job (`authority.js`) — same upsert shape against `(dot, mc_number, authority_type)` plus a freshness guard (`EXCLUDED.authority_status_changed_at >= existing`) and an in-batch deduper for the "All With History" file (multiple historical rows per key would otherwise trip the "ON CONFLICT cannot affect row a second time" Postgres error).
- Queue (`fmcsa-import-queue.js`) — separate Bull queue prefixed `fmcsa-import`, registers the two job processors. Cron wiring is intentionally deferred to FN-1415.

**Verification on dev DB:**
- Census fixture import → 3 inserted, 0 updated, 2 skipped (empty + DOT=0).
- Authority fixture import → 3 inserted, 0 updated, 4 skipped (in-batch dedupe + invalid). Three canonical authority types present (Carrier / Broker / Freight Forwarder).
- Re-run is a no-op: census re-run yields 0 inserted, 0 updated, 5 skipped — and `fmcsa_synced_at` does not advance for unchanged rows.
- Out-of-band mutation (set `phone='000…'`) followed by re-run yields exactly 1 updated row and the phone is restored from the CSV.
- `fmcsa.import_runs` row created per run with `status='success'`, `triggered_by='manual'`, and finished/started timestamps.

## Key Decisions

- **New `fmcsa-import` Bull queue, not extending `fmcsa-scrape`.** The two workloads have very different shapes — scrape jobs are short, throttled HTTP requests against SAFER and want `limiter: { max: 1, duration: 3000 }`; import jobs are minute-to-hour streaming downloads of multi-hundred-MB CSVs that batch-upsert. Sharing a queue would couple their concurrency/retry policies in ways neither workload wants, and a stuck import would block scrape jobs. **Resolves the first Open Item.**
- **Header-name-driven parsers with synonym lists.** FMCSA has historically renamed columns between dataset revisions; positional parsing would be brittle. Each canonical field maps to an alias list and we look up case-insensitively.
- **`v1` filenames + `_internals` exports.** The parser layer is versioned. If FMCSA breaks compatibility, we add a v2 and switch the queue handler over rather than editing v1 in place — preserves an audit trail of what shape we accepted at any historical run.
- **Conditional `ON CONFLICT … DO UPDATE … WHERE …` for idempotency.** The `IS DISTINCT FROM` predicate makes unchanged rows a no-op (not even RETURNING'd), so re-runs neither bump `fmcsa_synced_at` nor count as updates. The freshness guard on authorities prevents a stale snapshot from clobbering a fresher record.
- **In-batch dedupe for authorities.** The "All With History" file has multiple rows per `(dot, mc, type)` key. We collapse to the newest before sending to Postgres because a single statement can't touch the same conflict tuple twice.
- **Streaming with bounded buffers.** `runner.js` flushes in 500-row batches; `csv-stream.js` releases each row immediately after emit, so the parser is O(longest-row) memory.
