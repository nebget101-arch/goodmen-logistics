# FN-1517: [Work Order] Basics tab — Scheduled/Start/Completion dates and Vehicle Make/Model/Year not populating on open

## Requirement

When opening an existing Work Order, the Basics tab Work Order Info dates (Scheduled / Start / Completion) and Vehicle Info Make / Model / Year fields are blank. Vehicle fields are present in the API payload but not bound; the three date fields have no backing DB columns at all.

## Acceptance Criteria

- [ ] Migration adds `scheduled_date / start_date / completion_date DATE` columns to `work_orders`
- [ ] Backend GET / PUT / POST read & write the three new dates
- [ ] Frontend binds Make/Model/Year + the three dates on load
- [ ] Round-trip: enter values, save, refresh → values persist
- [ ] No regression in totals, FN-1367 wiring

## Implementation Summary

### FN-1523 (subtask)
Added Knex migration `20260507220000_add_workorder_planned_dates.js` that adds three nullable `DATE` columns to `work_orders`: `scheduled_date`, `start_date`, `completion_date`. The migration is idempotent — each column is added only if missing, and the down step drops each only if present. The existing `completed_at TIMESTAMPTZ` audit column is untouched.

Verified locally:
- Up on a fresh state adds all three columns as `date`, nullable.
- Down drops all three cleanly.
- Up/down cycle is repeatable.
- All 1000 existing `work_orders` rows have `NULL` in the three new columns (no backfill needed).

### FN-1525 (subtask)
Wired the Work Order Basics tab loader and save payload so vehicle Make/Model/Year and the three new planned-date columns round-trip cleanly.

- `loadWorkOrder()` now copies `vehicle.make`, `vehicle.model`, `vehicle.year` (also `license_plate` and `type`) from the API payload onto the form-bound `workOrder` model. Previously these were only set as a side effect of `onVehicleSelect` (the dropdown handler), so they were blank when reopening an existing WO.
- `loadWorkOrder()` reads the new `wo.scheduled_date / start_date / completion_date` fields returned by FN-1524 and binds them to `workOrder.scheduledDate / startDate / completionDate`. The legacy `wo.completed_at` is kept as a fallback for `completionDate` so existing rows that pre-date FN-1523 still pre-fill the field.
- `submitWorkOrder()` now includes `scheduledDate`, `startDate`, `completionDate` in the PUT/POST payload (snake_case → camelCase as accepted by the FN-1524 service). Empty values are coerced to `null`; the backend's `normalizeDateString` then writes `NULL`.
- Added `toDateInputValue()` private helper that normalizes the API response (ISO timestamp, `YYYY-MM-DD` string, or `Date`) to the `YYYY-MM-DD` string format the `<app-ai-date-picker>` form-control value expects, and gracefully returns `''` for null/empty/invalid.
- No HTML changes: the existing `[(ngModel)]` bindings for `workOrder.scheduledDate / startDate / completionDate / make / model / year` were already in `basics-tab.component.html` from prior work — only the parent loader was missing the field copies.
- TypeScript compiles cleanly under `tsc --noEmit -p tsconfig.app.json`.

### FN-1524 (subtask)
Wired the work-order service to read and write the three new date columns added in FN-1523.

- `createWorkOrder` insert now includes `scheduled_date`, `start_date`, `completion_date`, normalized via a new `normalizeDateString` helper that maps `''`, whitespace, `null`, and `undefined` to SQL `NULL` and passes ISO date strings through unchanged.
- `updateWorkOrder` update conditionally adds each date column only when the payload key is present (`!== undefined`) — so omitted fields preserve the existing row value, and explicit empty strings are persisted as `NULL`.
- `getWorkOrderById` already returns the full `work_orders` row via `.first()` (no explicit projection), so the three new columns flow into the GET response automatically once the migration runs.
- Routes layer (`work-orders-hub.js`) needed no changes — payload field names (`scheduledDate`, `startDate`, `completionDate`) pass straight through to the service. Empty-string sentinel handling lives in the service so list/bulk-upload paths get the same behavior.
- No effect on totals computation: `recomputeWorkOrderTotals` does not read the date columns, and the date columns are not part of any total/discount/tax expression.
- Added `routes/__tests__/work-orders-dates.test.js` (9 tests, all passing) covering: helper-level normalization, create-side persistence (ISO date passthrough, empty-string→NULL, omitted→NULL), and update-side semantics (provided→persisted, empty-string→NULL, omitted→column absent so existing row value is preserved). Test stubs the `internal/db` module via `require.cache` and asserts captured insert/update payloads — no live DB needed.

## Key Decisions

### FN-1523 (subtask)
- Used `addColumnIfMissing` / `dropColumnIfExists` pattern (matches `20260228152000_add_work_orders_financial_columns.js`) so the migration is safe to re-run.
- Did **not** add an index — none of the three columns are query predicates today (per ticket guidance).
- Kept `completed_at TIMESTAMPTZ` separate from the new `completion_date DATE`: the former is system-set on completion (audit), the latter is user-entered planned/actual completion date on the form.

### FN-1524 (subtask)
- Update path uses conditional spreads (`...(payload.scheduledDate !== undefined ? { scheduled_date: ... } : {})`) instead of the `payload.X ?? workOrder.X` pattern used for other columns. Reason: the AC requires that an explicit empty string be **stored as NULL** rather than treated as a "not provided, preserve existing" signal. The conditional spread distinguishes "key absent from payload" (preserve) from "key present but empty" (NULL).
- Exported `normalizeDateString` from the service module so the test can assert the contract directly without re-implementing the normalization.
- No changes to `work-orders-hub.js` route handlers: the date payload fields are plain strings (no UUID coercion needed), and putting normalization in the service keeps a single code path for both REST and bulk-upload flows.

### FN-1525 (subtask)
- Field-copy lives in the parent `WorkOrderComponent.loadWorkOrder()` rather than inside `WoBasicsTabComponent`. The basics tab uses `OnPush` change detection and consumes `workOrder` as an `@Input` reference; placing the assignments at the top of the loader (alongside the other field-copy block) keeps the data flow uni-directional and avoids a redundant `populate*` method on the child.
- Kept legacy `wo.completed_at` as a fallback for `completionDate` (`wo.completion_date ?? wo.completionDate ?? wo.completed_at`). Reason: rows created before FN-1523 only have `completed_at` (the system-set audit timestamp on COMPLETED transitions). Without the fallback those WOs would lose their visible completion date until they're saved again. New saves persist into `completion_date`; that takes precedence on the next load.
- Used the picker's documented contract: `<app-ai-date-picker>` form value is `YYYY-MM-DD | null`. The new `toDateInputValue` helper slices ISO timestamps to the first 10 chars (which is what the picker also does on `parseToDate`) so we feed the picker a value it can round-trip without re-parsing.
- Added `vehicle.license_plate` and `vehicle.type` to the loader copy as well — both already had `[(ngModel)]` bindings on the readonly inputs but were also blank on reopen for the same reason. This is in-scope under "no regression in existing fields" and avoids leaving the bug only half-fixed.
