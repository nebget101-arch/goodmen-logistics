# FN-1599: [P1] [Loads Import] Commit drops mapped fields — saved loads have null dates/driver_name and status="DRAFT"

## Requirement
The spreadsheet import preview correctly maps Driver, Pickup Date, Delivery Date, Completed Date, Status, and Notes from source columns at ≥0.9 confidence and emits a `statusEnumMapping` (e.g. "Delivered" → "DELIVERED"). After commit, persisted load rows show `pickup_date=null`, `delivery_date=null`, `completed_date=null`, `driver_name=""`, and `status="DRAFT"` for **every** row, regardless of source values.

`commitBatch()` in `backend/packages/goodmen-shared/services/loads-import-service.js` writes only 16 columns and never persists dates, never falls back to textual `driver_name` when fuzzy match fails, and never consults `ai_metadata.statusEnumMapping` (force-DRAFTs low-confidence rows). `billing_status` enum already works — `status` does not — so the fix is targeted, not a wholesale rewrite.

## Acceptance Criteria
- [ ] Committed loads persist `pickup_date`, `delivery_date`, `completed_date` from AI-mapped source columns (parsed to ISO date / timestamptz; nullable).
- [ ] When fuzzy driver match returns no FK, `driver_name` retains the raw mapped string (not empty).
- [ ] When `ai_metadata.statusEnumMapping` is present and source value has a mapped enum, `loads.status` uses that mapped enum (subject to FN allowed values; otherwise fall back to DRAFT and log warning).
- [ ] `billing_status` enum behavior unchanged — regression test added.
- [ ] `ai_metadata.confidences.{pickup_date,delivery_date,completed_date,driver_name,status}` recorded.
- [ ] Unit tests cover dates persisted, status enum applied, driver_name fallback, billing_status regression.
- [ ] No change to PDF rate-confirmation flow (`POST /api/loads/bulk-rate-confirmations`).

### FN-1601 (subtask)
**Branch:** `backend/FN-1601/commit-persist-fields` (off `integration/FN-1599`)

Fixed `commitBatch()` to persist all the AI-mapped fields the preview produces, plus a small migration for the column the spec required. Seven-part change:

1. **Date parser** — added `parseImportDate(value)` to `loads-import-mapper.js`. Handles `Date` objects, ISO date / timestamp strings, US `MM/DD/YYYY` (and 2-digit-year), and the JS `Date.toString()` form spreadsheets sometimes round-trip through (`Thu May 07 2026 00:00:00 GMT+0000 (UTC)`). Uses UTC accessors throughout so the calendar day doesn't drift in host TZ. Returns `null` on empty / unparseable input so the `DATE` column stays valid.
2. **INSERT extension** — `commitBatch()` INSERT now writes 4 additional columns: `driver_name`, `pickup_date`, `delivery_date`, `completed_date`. Param order documented in the test fixture (`IDX` constant).
3. **Status decoupled from FK matching** — pre-fix, low broker/driver fuzzy score force-DRAFTed the row even when the AI had a high-confidence `_status: "DELIVERED"` from `statusEnumMapping`. Status now comes from the source data + AI mapping; `needs_review` flips on FK miss / no stops / free-text pattern but no longer overrides status.
4. **Defensive statusEnumMapping at commit** — when stage couldn't pre-set `_status` (e.g. row demoted to `needs_review` for an unrelated reason), commit re-applies `batch.ai_metadata.statusEnumMapping` and validates against `LOAD_STATUSES`. Unmapped values fall back to `DRAFT` and append a warning to the row's `ai_metadata.warnings`.
5. **Driver name fallback** — when fuzzy `driverMatch?.id` is null, the raw `normalized.driver_name` text is written to a new `loads.driver_name` column (added by migration `20260509120000_add_loads_driver_name_text.js`). Mirrors the existing `loads.broker_name` pattern.
6. **List/detail SELECT projections** updated in `routes/loads.js` to `COALESCE(NULLIF(concat_ws(' ', d.first_name, d.last_name), ''), l.driver_name)` so the textual fallback is visible to the FE without API contract changes.
7. **Per-row confidences** — `ai_metadata.confidences` on the inserted load now carries `pickup_date / delivery_date / completed_date / driver_name / status` from `batch.ai_metadata.finalColumnMapping`, alongside the existing fuzzy-match scores.

**Out of scope (intentionally not touched):**
- The fuzzy matchers read `normalized.driver` / `normalized.truck` / `normalized.trailer`, but the AI emits canonical `driver_name` / `truck_unit` / `trailer_unit`. That key mismatch means FK matching is largely ineffective regardless of underlying data — but it's adjacent to FN-1601's ACs and would balloon scope. The `driver_name` text fallback masks the symptom for FN-1599's repro; the key alignment is a follow-up.
- `notes` was reported missing on some rows in FN-1599 repro. The INSERT already writes `trimOrNull(normalized.notes)`. If the AI didn't map a `notes` source column, `applyColumnMapping` produces no key and `normalized.notes` is undefined — that's a mapping-quality issue, not a commit-path bug.

## Implementation Summary
`commitBatch()` was the bug epicenter — it dropped 4 mapped fields on INSERT and force-DRAFTed any row whose broker/driver fuzzy match was below threshold even when the AI had a confident status. Fix: extended the INSERT, added a date parser, decoupled status resolution from FK confidence, added a one-column migration for the textual driver_name fallback, and surfaced the fallback in the list/detail projections.

## Key Decisions
- **Added a DB migration despite this being a backend-labeled subtask.** Spec said "write to `loads.driver_name`" but no such column existed and no DB sibling subtask was scoped. The backend INSERT can't satisfy the AC without the column; the migration is the smallest possible addition (one nullable text column, mirrors `loads.broker_name`).
- **Status no longer demotes on low FK confidence.** This changes a pre-existing behavior: previously, a row with `_status: "DELIVERED"` and broker_score=0 would land as DRAFT+needs_review. Now it lands as DELIVERED+needs_review. AC #3 plus the bug repro made this the right call — status carries the user's intent, FK matching is operational metadata.
- **driver_name written ONLY when no FK match** (rather than always, like broker_name). Matches the literal AC wording. Trade-off: when a driver IS matched, we lose the imported text spelling. The audit confidences in `ai_metadata` partially compensate.
