# FN-1322: Action Queue — unified, severity-ranked, grouped alerts feed

## Requirement
Merge Smart Alerts and Compliance Alerts into one severity-ranked, grouped, action-queue feed in the Control Center main column. Group near-identical alerts so 12 "preventive maintenance overdue" rows collapse into one expandable row, and expose bulk actions.

## Acceptance Criteria
- [ ] One unified Action Queue component renders in the Control Center main column; the right-rail Compliance Alerts section is removed (or replaced with narrower "upcoming deadlines" context).
- [ ] Near-identical alerts are grouped ("N vehicles overdue for preventive maintenance — view list").
- [ ] Alerts ranked by severity (Critical → High → Medium → Low) using tokens from FN-1326, then by recency.
- [ ] Each row exposes at least one primary action.
- [ ] Bulk actions: select-all-in-group + acknowledge / dismiss group; persists via existing dismissals table.
- [ ] Empty state replaces the current "No compliance alerts match your filters" string with action-oriented copy.
- [ ] No regression in alert counts vs. legacy feeds.

## API Contract (defined here so frontend + backend can run in parallel)
```
GET /api/dashboard/action-queue?window=today|7d|30d&severity=critical|high|medium|low|all
→ 200 { groups: [...], total, window, severity, generatedAt, upstreamErrors: [...] }

  group shape:
  {
    id: 'compliance:maintenance:pm_overdue' | 'smart:hos_imminent' | ...,
    source: 'smart' | 'compliance',
    severity: 'critical' | 'high' | 'medium' | 'low',
    category: 'driver' | 'vehicle' | 'load' | 'maintenance' | 'compliance',
    message: '3 vehicles — Preventive maintenance overdue',
    count: 3,
    latest_at: '2026-05-05T...',
    targets: [{ id, label, route, raw_alert_id }],
    primary_action: { label, action_id, payload }
  }

POST /api/dashboard/action-queue/dismiss
body { group_id?: string, target_ids?: string[] }
→ 200 { dismissed_count, group_id }
```

## Implementation Summary

### FN-1330 (subtask — backend)
Implemented the grouped-alerts aggregation endpoint at the **gateway** rather than `backend/microservices/dashboard-stats-service` as originally noted in the Jira description (that service does not exist in this codebase; smart-alerts and the dismissals store already live in `backend/gateway/services/`, so co-locating the action-queue endpoint there reuses the same singletons and avoids a new microservice).

Endpoint pipeline (`backend/gateway/routes/action-queue.js`):
1. `GET /api/dashboard/action-queue` runs the existing `smart-alerts-aggregator` and the new `compliance-alerts-client` (which calls reporting-service `GET /api/dashboard/alerts`) in parallel.
2. The new `alert-grouper` service normalizes both feeds to a common shape, dedupes by `(source, category, message_template)`, applies window + severity filters, and rolls per-target dismissals.
3. `POST /api/dashboard/action-queue/dismiss` writes to the same `dismissals-store` used by `/api/alerts/smart/:id/dismiss`. Dismissing by `group_id` hides the whole rolled-up group for the dismissal TTL; passing `target_ids` hides only those rows.

Mounted on the gateway BEFORE the `/api/dashboard` proxy block so the route resolves locally instead of being forwarded to reporting-service. The shared `smartAlertsAggregator`, `dismissalsStore`, and `alertsBroadcaster` were moved up in `index.js` to be constructed before the proxy block; `/api/alerts/smart` (FN-1161) continues to consume the same singletons.

## Key Decisions

### FN-1330 (subtask — backend)
- **Location: gateway, not a new microservice.** The Jira description references `dashboard-stats-service` which does not exist. Smart Alerts already live in the gateway and the dismissals store is per-process; building a new microservice would force cross-process state for dismissals. Co-locating in the gateway keeps the dismissal TTL behavior identical to `/api/alerts/smart`.
- **Group key strategy.** Compliance alerts are templatized by message regex (`pm_overdue`, `medical_cert_expired`, etc.) so per-target detail is stripped without losing the group identity. Smart alerts use `smart:${type}` since the aggregator already classifies them.
- **Group-level dismissals reuse alert dismissals.** The existing `MemoryDismissalsStore` keys by `(tenantId, userId, alertId)` and is content-agnostic. We pass the `group_id` as the `alertId` for group-wide dismissal and `target_ids` for individual rows; both paths share the same TTL and tenant isolation contract.
- **Lenient upstream failures.** Mirroring `smart-alerts-aggregator`, the route returns 200 with whatever upstream succeeded and surfaces errors in `upstreamErrors[]`. A reporting-service outage degrades to "smart-alerts only" rather than 5xx.
- **Severity buckets.** Smart alert numeric scores (0-100) are bucketed `>=80 critical / >=60 high / >=40 medium / <40 low`; compliance string `critical → critical, warning → high, other → medium`. Buckets aligned to the Story E severity tokens (FN-1326) so the frontend can switch on `group.severity` directly.
