## Implementation Summary

### FN-1533 (subtask)

Backend slice landed on integration branch `integration/FN-1520`.

1. **Selling-price source fix** тАФ `reservePart()` in
   `backend/packages/goodmen-shared/services/work-orders.service.js` now defaults
   the WO line `unit_price` to `parts.default_retail_price` (selling price)
   instead of the non-existent `part.unit_price` column. An explicit
   `payload.unitPrice` still overrides; falls through to `0` only when both are
   missing.

2. **New `PATCH /api/work-orders/:id/parts/:partLineId`** тАФ accepts
   `{ qtyRequested?, qtyReserved?, qtyIssued?, unitPrice?, taxable? }`. The
   service computes the final state from the merge of current line + payload
   and enforces:
   - `qty_reserved <= qty_requested`
   - `qty_issued <= qty_reserved`
   - non-negative quantities and `unitPrice`
   Inventory side effects:
   - **Decrement `qty_reserved`** тЖТ `inventory.reserved_qty -= delta` (release).
   - **Increment `qty_reserved`** тЖТ checks `on_hand - reserved` availability
     before incrementing `inventory.reserved_qty`.
   - **Decrement `qty_issued`** тЖТ `inventory.on_hand_qty += delta` (mirrors
     `returnPart()` semantics тАФ back to available stock).
   - **Increment `qty_issued`** тЖТ `inventory.reserved_qty -= delta` and
     `inventory.on_hand_qty -= delta` (mirrors `issuePart()`).
   Each delta logs an `inventory_transactions` row (RESERVE / ISSUE / RETURN).
   Status is re-derived from final qtys (BACKORDERED / RESERVED / ISSUED) and
   `recomputeWorkOrderTotals()` runs in the same transaction. Response shape:
   `{ success: true, data: { line, workOrder } }` so the frontend can re-render
   totals from the same call.

3. **Tests** тАФ
   `backend/packages/goodmen-shared/routes/__tests__/work-orders-parts-patch.test.js`
   covers happy-path multi-field PATCH, status derivation, all four invariant
   violations, all six inventory side-effect paths, and the three branches of
   the selling-price source change. 16/16 passing under `node --test`.

## Key Decisions

- **PATCH supports both increment and decrement of qty_reserved / qty_issued**, not just decrement. The story description only called out "decrement releases inventory", but inline editing in the UI naturally goes both directions; rejecting increments would force users to delete and re-add lines, which would lose history. Increment paths reuse the existing `reservePart` / `issuePart` semantics for inventory updates and availability checking.
- **Status derivation is deterministic from the final qtys**, not from the diff. `derivePartLineStatus` returns `ISSUED` when `qty_issued >= qty_requested` (and requested > 0), `RESERVED` when fully reserved, otherwise `BACKORDERED`. Frontend doesn't need to send `status`.
- **Response shape is `{ line, workOrder }`** so a single PATCH lets the frontend re-render both the row and the totals strip without a follow-up GET.
- **Decrementing qty_issued mirrors `returnPart` semantics** (back to on-hand stock, not back to qty_reserved). The line's `qty_reserved` is left alone so the user retains their reservation; if they want to drop reservation too, they patch `qtyReserved` separately.
- _UX direction for inline edit (editable Requested column + тЖУ promote links vs. three editable cells) decided at frontend implementation time and recorded here_
