# FN-1520: [Work Order] Parts table ‚Äî populate selling price (not cost) and add inline quantity edit

## Requirement

On the Work tab parts table:
1. When a part is added, populate Unit Price from the part's selling/retail price (`parts.default_retail_price`), not its cost.
2. Make the quantity columns inline-editable; debounced PATCH updates the line and re-renders WO totals.

## Acceptance Criteria

- [ ] Add Part dialog defaults Unit Price to `default_retail_price`
- [ ] New `PATCH /work-orders/:id/parts/:lineId` validates qty invariants and recomputes totals
- [ ] Inline qty edit triggers debounced PATCH with saving indicator
- [ ] Validation errors display inline (no global toast)
- [ ] No regression in Issue All Reserved / per-row Issue / Remove buttons

## Implementation Summary

### FN-1533 (subtask)

Backend slice landed on integration branch `integration/FN-1520`.

1. **Selling-price source fix** ‚Äî `reservePart()` in
   `backend/packages/goodmen-shared/services/work-orders.service.js` now defaults
   the WO line `unit_price` to `parts.default_retail_price` (selling price)
   instead of the non-existent `part.unit_price` column. An explicit
   `payload.unitPrice` still overrides; falls through to `0` only when both are
   missing.

2. **New `PATCH /api/work-orders/:id/parts/:partLineId`** ‚Äî accepts
   `{ qtyRequested?, qtyReserved?, qtyIssued?, unitPrice?, taxable? }`. The
   service computes the final state from the merge of current line + payload
   and enforces:
   - `qty_reserved <= qty_requested`
   - `qty_issued <= qty_reserved`
   - non-negative quantities and `unitPrice`
   Inventory side effects:
   - **Decrement `qty_reserved`** ‚Üí `inventory.reserved_qty -= delta` (release).
   - **Increment `qty_reserved`** ‚Üí checks `on_hand - reserved` availability
     before incrementing `inventory.reserved_qty`.
   - **Decrement `qty_issued`** ‚Üí `inventory.on_hand_qty += delta` (mirrors
     `returnPart()` semantics ‚Äî back to available stock).
   - **Increment `qty_issued`** ‚Üí `inventory.reserved_qty -= delta` and
     `inventory.on_hand_qty -= delta` (mirrors `issuePart()`).
   Each delta logs an `inventory_transactions` row (RESERVE / ISSUE / RETURN).
   Status is re-derived from final qtys (BACKORDERED / RESERVED / ISSUED) and
   `recomputeWorkOrderTotals()` runs in the same transaction. Response shape:
   `{ success: true, data: { line, workOrder } }` so the frontend can re-render
   totals from the same call.

3. **Tests** ‚Äî
   `backend/packages/goodmen-shared/routes/__tests__/work-orders-parts-patch.test.js`
   covers happy-path multi-field PATCH, status derivation, all four invariant
   violations, all six inventory side-effect paths, and the three branches of
   the selling-price source change. 16/16 passing under `node --test`.

### FN-1534 (subtask)

Frontend slice landed on integration branch `integration/FN-1520`.

1. **Add Part dialog ‚Äî selling-price default**. New helper
   `resolveSellingPrice(part)` resolves the unit-price candidate in priority
   order `default_retail_price ‚Üí unit_price ‚Üí unit_cost`. Catalog selection,
   barcode add, and the catalog-search dropdown now all read through this
   helper, so the Unit Price field defaults to the part's selling/retail price
   instead of cost. The Unit Price label in catalog mode shows a small
   **(retail)** affordance once a part is selected, so the user can tell at a
   glance which source is in play.

2. **Inline qty edit on the parts table**. Replaced the read-only
   Requested / Reserved / Issued cells with `type="number"` inputs bound
   one-way to `line.qty_*`. Each edit fires `onQtyEdit(line, field, value)`
   which:
   - optimistically updates the local row so the input never flickers,
   - accumulates a per-line `pendingPatch` payload (so multi-field edits
     within the debounce window fire as a **single** PATCH, keeping the
     backend invariants `qty_reserved ‚â§ qty_requested` and
     `qty_issued ‚â§ qty_reserved` consistent),
   - debounces 500 ms via a per-line RxJS `Subject` + `debounceTime`,
   - calls the new `apiService.patchWorkOrderPart(id, lineId, payload)`,
   - on success, merges `data.line` back into the row and copies
     `data.workOrder` totals (`labor_subtotal`, `parts_subtotal`,
     `fees_subtotal`, `tax_amount`, `total_amount`) into the parent
     `workOrder` object ‚Äî no follow-up GET, no full `reloadWorkOrder()`.
   - on error, sets `line._error = { field, message }`, the offending
     `<input>` gets `.has-error` (red border + tooltip), and the message
     also shows below the row's status badge.
   While a PATCH is in-flight the row gets `[class.row-saving]` (subtle
   opacity dim) and a "Saving‚Ä¶" inline indicator with a spinning sync icon
   next to the part name. All three pre-existing flows (Issue All Reserved,
   per-row Issue / Reserve / Return / Remove buttons, dialog Add Part) are
   untouched.

3. **UX direction (per the parent story's open question)**: I went with
   "all three columns editable" rather than the alternative "editable
   Requested + ‚Üì Reserve / ‚Üì Issue promote links". Reasons:
   - the AC says "Editing Quantity in the table triggers a PATCH" with no
     restriction on which qty,
   - users editing finished work orders sometimes need to correct an
     accidental over-issue or release an idle reservation; promote-only
     links would force them to use the existing per-row buttons or contact
     an admin,
   - the existing per-row Issue / Reserve / Return buttons are kept in
     parallel ‚Äî they remain the primary "guided" path; inline edit is the
     "expert" override for corrections.

## Key Decisions

- **PATCH supports both increment and decrement of qty_reserved / qty_issued**, not just decrement. The story description only called out "decrement releases inventory", but inline editing in the UI naturally goes both directions; rejecting increments would force users to delete and re-add lines, which would lose history. Increment paths reuse the existing `reservePart` / `issuePart` semantics for inventory updates and availability checking.
- **Status derivation is deterministic from the final qtys**, not from the diff. `derivePartLineStatus` returns `ISSUED` when `qty_issued >= qty_requested` (and requested > 0), `RESERVED` when fully reserved, otherwise `BACKORDERED`. Frontend doesn't need to send `status`.
- **Response shape is `{ line, workOrder }`** so a single PATCH lets the frontend re-render both the row and the totals strip without a follow-up GET.
- **Decrementing qty_issued mirrors `returnPart` semantics** (back to on-hand stock, not back to qty_reserved). The line's `qty_reserved` is left alone so the user retains their reservation; if they want to drop reservation too, they patch `qtyReserved` separately.
- **UX direction for inline edit**: chose three editable cells (Requested / Reserved / Issued) over "editable Requested + ‚Üì promote links" ‚Äî see FN-1534 implementation summary above for rationale. Existing per-row Issue / Reserve / Return / Remove buttons are kept for the guided path.
- **Multi-field debouncing per line**: pending payload is a `Partial<{qtyRequested, qtyReserved, qtyIssued}>` accumulated over the 500 ms window, so a user editing Requested ‚Üí Reserved within ‚â§500 ms triggers ONE PATCH with both fields. Sending them as separate sequential PATCHes would briefly violate `qty_reserved ‚â§ qty_requested` and 400 the second call.
- **Direct mutation of `@Input() workOrder` for totals**: the alternative ‚Äî emitting `reloadWorkOrder` after every PATCH ‚Äî would re-fetch the WO + parts list and dismiss any dropdowns/foci the user has open mid-edit. The parent component uses OnPush; the same object reference is mutated and `cdr.markForCheck()` is called from this component, which propagates through the input binding.
