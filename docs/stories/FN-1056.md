# FN-1056: "Upload Rate Confirmation" click flow only accepts a single PDF (Auto-Create modal blocks bulk select)

## Requirement
Restore parity between the click and drag-drop entry points for "Upload Rate Confirmation" on the Loads page. The hero CTA already supports 1–10 PDFs end-to-end via drag-drop; the click path is broken because the Auto-Create modal (1) is missing the `multiple` attribute on its file inputs, (2) collapses the FileList to `files[0]` in `onAutoFileSelected`, and (3) lacks drag/drop handlers on its dropzone. Fix all three so 1–10 PDFs route correctly: 1 → Auto-Create flow, 2–10 → existing bulk flow, 11+ → cap at 10 with a non-blocking message.

## Parent
Epic **FN-723** — Loads Page Redesign — AI-First Workflow, Wizard, Inline Stops, Templates, Cloning & Email Ingestion

## Acceptance Criteria
- [ ] Clicking "Upload Rate Confirmation" → browsing → selecting 2–10 PDFs opens the Bulk Extraction Grid (or Bulk Upload modal), not the single-file Auto-Create modal.
- [ ] Selecting 1 PDF still opens the existing Auto-Create modal (no regression).
- [ ] Inside the Auto-Create modal, clicking "browse" with 2–10 PDFs OR dropping 2–10 PDFs onto the dropzone closes the modal and transitions to the bulk flow with all files preserved.
- [ ] Selecting >10 PDFs caps at 10 and shows a non-blocking message; the flow proceeds with the first 10.
- [ ] Non-PDF files are rejected with the existing "PDF only" message.
- [ ] Empty-state "Import from PDF" CTA (`onEmptyStateImportFromPdf`) still works for 1–10 PDFs.
- [ ] No console errors. No broken existing tests.

## Implementation Summary

### FN-1057 (subtask)

Extracted `selectPdfs()` into `frontend/src/app/utils/pdf-upload.util.ts` and routed both entry points through it:

- **Hero CTA** (`loads-hero-cta.component.ts`) — replaced inline `_handleFiles` filter/cap with the shared util; added `pdfsCapped: EventEmitter<void>` so the parent can surface a non-blocking message when 11+ PDFs are selected. Behavior for 1 / 2–10 PDFs is byte-for-byte identical.
- **Auto-Create modal** (`loads-dashboard.component.html` + `…component.ts`) —
  - Added `multiple` to both `<input type="file">` elements (lines 2322, 2328) and reset `value` after change so re-selecting the same file works.
  - Added `(dragover)` / `(dragleave)` / `(drop)` bindings on `.ai-dropzone` plus an `ai-dropzone--drag-over` SCSS modifier for visual feedback.
  - Replaced the `files[0]`-only `onAutoFileSelected` with `_handleAutoModalFiles(files)`:
    - 0 PDFs → `autoError = 'Please select PDF files only.'` (or `'Please select a PDF file.'` when input was empty/all-PDF mistype).
    - 1 PDF → keeps existing Auto-Create flow (sets `autoPdfFile`).
    - 2–10 PDFs → sets `showAutoModal = false` and routes to `onHeroBulkPdfs(pdfs)` to match the hero CTA's existing `bulkPdfsSelected` handler.
    - 11+ PDFs → caps to 10 via the util and surfaces "Maximum 10 PDFs. Extra files were not added." through the existing `errorMessage` snackbar (auto-clears after 4s).
  - Sibling `onAutoFileDropped(event)` handles drop on the modal dropzone with the same pipeline.
- **Hero CTA wiring** in `loads-dashboard.component.html` — added `(pdfsCapped)="onPdfsCapped()"` so the hero CTA's cap event surfaces the same snackbar.
- **Util coverage**: added `pdf-upload.util.spec.ts` with 7 cases (null/undefined input, no PDFs, mixed list ordering, exact-max, over-max with cap flag, custom max, single-PDF).

## Key Decisions

- **Shared util chosen** over duplicated filter+cap logic. Both entry points now call `selectPdfs(files, max=10) → { pdfs, capped }`.
- **2–10 PDF route from Auto-Create modal → `onHeroBulkPdfs(pdfs)`** (Bulk Upload modal), to match the hero CTA's `bulkPdfsSelected` wiring exactly. The page-level full-screen drop continues to route to the FN-745 Bulk Extraction Grid (unchanged) — this PR does not introduce a third route.
- **Cap message uses the existing `errorMessage` snackbar** (auto-clears after 4s) instead of a new toast component — non-blocking, surfaces above whichever modal the user lands in.
- **Non-PDF rejection** lives in `_handleAutoModalFiles` (Auto-Create modal context) since the existing hero CTA `_handleFiles` silently returned for non-PDFs and that "byte-for-byte" behavior is preserved for the hero path.
